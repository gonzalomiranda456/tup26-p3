# `IServiceProvider` en .NET — Inyección de dependencias

## El problema con `new`

Cuando una clase crea sus dependencias con `new`:

```csharp
public class ProcesadorPedidos {
    private readonly EnviadorEmails emails = new EnviadorEmails(); // acoplado
    private readonly RepositorioPedidosSql pedidos = new RepositorioPedidosSql();
}
```

- No se puede testear sin enviar emails reales
- Cambiar el proveedor obliga a modificar la clase de negocio
- Las dependencias son invisibles desde afuera
- No hay control sobre cuántas instancias existen

## La solución: inyección por constructor

```csharp
public class ProcesadorPedidos(IEnviadorEmails emails, IRepositorioPedidos pedidos, IGatewayCobros cobros) {
    public void Procesar(Pedido p) {
        cobros.Cobrar(p);
        pedidos.Guardar(p);
        emails.EnviarConfirmacion(p);
    }
}
```

Las dependencias son explícitas, intercambiables y no las construye la clase. El problema se traslada: ¿quién las construye? → El contenedor de DI.

## IServiceCollection + IServiceProvider

```csharp
// Fase de configuración (antes del Build)
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddScoped<IRepositorioPedidos, RepositorioPedidosSql>();
builder.Services.AddSingleton<IEnviadorEmails, EnviadorEmailsSmtp>();
builder.Services.AddTransient<CalculadoraImpuestos>();

// Forma con factory (cuando hace falta configuración)
builder.Services.AddSingleton<IEnviadorEmails>(sp => {
    var config = sp.GetRequiredService<IConfiguration>();
    return new EnviadorEmailsSmtp(config["Smtp:Host"]!);
});

// Fase de ejecución — el contenedor resuelve el grafo automáticamente
var app = builder.Build();
```

## Los tres lifetimes

| Lifetime | Cuándo crear | Cuándo usar |
|---|---|---|
| `AddTransient` | Nueva instancia cada vez | Livianos, sin estado, sin recursos |
| `AddScoped` | Una por scope (request HTTP) | Repositorios, DbContext, servicios de aplicación |
| `AddSingleton` | Una para toda la app | Caches, loggers, clientes HTTP, configuración |

**Regla de dependencias:**
- Singleton solo puede depender de Singleton
- Scoped puede depender de Singleton y Scoped
- Transient puede depender de cualquiera

**Captive dependency** (bug clásico): un Singleton captura un Scoped en su constructor → el Scoped queda vivo para siempre, compartido entre requests. Activar `ValidateScopes = true` en desarrollo para detectarlo.

## Uso correcto en código

El 95% del uso es simplemente declarar dependencias en el constructor:

```csharp
public class ProcesadorPedidos(
    IRepositorioPedidos pedidos,
    IGatewayCobros cobros,
    ILogger<ProcesadorPedidos> log) {
    
    public async Task ProcesarAsync(Pedido p) {
        log.LogInformation("Procesando {Id}", p.Id);
        await cobros.CobrarAsync(p);
        await pedidos.GuardarAsync(p);
    }
}
```

## Factory cuando el tipo depende del runtime

Único caso legítimo de inyectar `IServiceProvider` directamente:

```csharp
public class FactoryEstrategiaPago(IServiceProvider sp) : IFactoryEstrategiaPago {
    public IEstrategiaPago Para(string metodo) => metodo switch {
        "tarjeta"  => sp.GetRequiredService<EstrategiaTarjeta>(),
        "efectivo" => sp.GetRequiredService<EstrategiaEfectivo>(),
        _          => throw new ArgumentException($"Método desconocido: {metodo}")
    };
}
```

## BackgroundService con servicios scoped

Un `BackgroundService` es singleton; no puede inyectar servicios scoped directamente. Solución: `IServiceScopeFactory`:

```csharp
public class WorkerMails(IServiceScopeFactory scopeFactory) : BackgroundService {
    protected override async Task ExecuteAsync(CancellationToken ct) {
        while (!ct.IsCancellationRequested) {
            using var scope = scopeFactory.CreateScope();
            var repo = scope.ServiceProvider.GetRequiredService<IRepositorioPedidos>();
            await repo.ProcesarPendientesAsync(ct);
            await Task.Delay(TimeSpan.FromMinutes(1), ct);
        }
    }
}
```

```csharp
builder.Services.Configure<ConfigSmtp>(builder.Configuration.GetSection("Smtp"));

public class EnviadorEmailsSmtp(IOptions<ConfigSmtp> opt) : IEnviadorEmails {
    private readonly ConfigSmtp cfg = opt.Value;

}
```

## Anti-patrones

**Service Locator** — `IServiceProvider` en código de negocio oculta las dependencias y dificulta los tests:

```csharp
// ✗ Mal
public class ProcesadorPedidos(IServiceProvider sp) {
    public void Procesar(Pedido p) {
        var repo = sp.GetRequiredService<IRepositorioPedidos>(); // dependencias ocultas
    }
}
```

**Provider raíz como global estático** — rompe los scopes, imposible de testear.

## Inicialización al arrancar

```csharp
var app = builder.Build();
using (var scope = app.Services.CreateScope()) {
    var db = scope.ServiceProvider.GetRequiredService<MiDbContext>();
    db.Database.EnsureCreated();
}
app.Run();
```

## Resumen

```
El contenedor responde dos preguntas:
  1. ¿Cómo se construye cada dependencia?  → IServiceCollection
  2. ¿Cuánto tiempo vive?                  → Transient / Scoped / Singleton

Regla práctica:
  Toca BD o infraestructura → Scoped
  Sin estado, sin recursos   → Transient
  Global, thread-safe        → Singleton
  IServiceProvider en negocio → ✗ (solo en factories e infraestructura)
```
