## 1. ¿Qué problema resuelve Dapper?

Cuando trabajamos con ADO.NET puro, escribir la parte de la consulta SQL es sencillo. El problema es lo que viene después: convertir las filas que devuelve la base de datos en objetos C#.

Con ADO.NET crudo, ese proceso se ve así:

```csharp
var contactos = new List<Contacto>();

using var reader = command.ExecuteReader();
while (reader.Read())
{
    contactos.Add(new Contacto
    {
        Id       = reader.GetInt32(0),
        Nombre   = reader.GetString(1),
        Apellido = reader.GetString(2),
        Telefono = reader.GetString(3),
        Email    = reader.GetString(4),
    });
}
```

Esto funciona, pero:

- Es verboso y repetitivo.
- Si cambia el orden de las columnas, hay que actualizar índices manualmente.
- Cualquier tabla nueva requiere escribir el mismo código de conversión.

**Dapper** resuelve exactamente ese problema: sigue usando el SQL que nosotros escribimos, pero se encarga de **mapear automáticamente** las filas del resultado a objetos C#.

---

## 2. Dapper no es un ORM completo

Antes de continuar conviene ubicar a Dapper en el espectro de herramientas:

| Herramienta | Nivel | Qué hace |
|---|---|---|
| ADO.NET | Bajo | Conexión, comandos, parámetros, readers, transacciones |
| **Dapper** | **Medio** | **Usa SQL escrito por nosotros y mapea filas a objetos** |
| EF Core | Alto | ORM completo: LINQ, tracking, relaciones, migraciones |

Dapper no genera SQL. No sabe cuándo un objeto cambió. No tiene migraciones. No controla relaciones entre tablas.

Lo que sí hace, y muy bien: toma una consulta SQL y un tipo C#, y devuelve los objetos ya construidos.

## 4. Cómo funciona el mapping automático

Dapper compara los **nombres de las columnas** del resultado SQL con los **nombres de las propiedades** de la clase C#.

Si la columna se llama `Nombre` y la propiedad también se llama `Nombre`, Dapper los conecta automáticamente.

```csharp
class Contacto {
    public int    Id       { get; set; }
    public string Nombre   { get; set; } = "";
    public string Apellido { get; set; } = "";
    public string Telefono { get; set; } = "";
    public string Email    { get; set; } = "";
}
```

La tabla correspondiente:

```sql
CREATE TABLE Contactos (
    Id       INTEGER PRIMARY KEY AUTOINCREMENT,
    Nombre   TEXT NOT NULL,
    Apellido TEXT NOT NULL,
    Telefono TEXT NOT NULL,
    Email    TEXT NOT NULL
);
```

Los nombres coinciden → Dapper hace el trabajo solo.

## 5. Métodos principales

### `Query<T>` — consultas que devuelven múltiples filas

```csharp
IEnumerable<Contacto> contactos = db.Query<Contacto>(
    "SELECT * FROM Contactos"
);
```

El tipo genérico `T` indica en qué clase mapear cada fila.

### `QuerySingle<T>` — cuando esperamos exactamente un resultado

```csharp
int total = db.QuerySingle<int>("SELECT COUNT(*) FROM Contactos");
```

Lanza excepción si no hay resultados o si hay más de uno.

### `QuerySingleOrDefault<T>` — cuando puede haber cero o uno

```csharp
Contacto? c = db.QuerySingleOrDefault<Contacto>(
    "SELECT * FROM Contactos WHERE Id = @id",
    new { id = 5 }
);
```

Devuelve `null` si no existe.

### `Execute` — para INSERT, UPDATE, DELETE

```csharp
db.Execute(
    "INSERT INTO Contactos (Nombre, Apellido, Telefono, Email) VALUES (@Nombre, @Apellido, @Telefono, @Email)",
    new { Nombre = "Ana", Apellido = "Arce", Telefono = "555-0001", Email = "ana@example.com" }
);
```

## 6. Parámetros

Dapper usa parámetros con `@nombre`. Eso evita **SQL injection** y mejora la performance porque la base puede reutilizar el plan de ejecución.

Se pueden pasar como objeto anónimo:

```csharp
db.Query<Contacto>(
    "SELECT * FROM Contactos WHERE Apellido = @apellido",
    new { apellido = "Díaz" }
);
```

O como diccionario:

```csharp
db.Execute(
    "UPDATE Contactos SET Email = @email WHERE Id = @id",
    new Dictionary<string, object> { ["email"] = "nuevo@mail.com", ["id"] = 3 }
);
```

O directamente con el objeto a insertar:

```csharp
var contacto = new Contacto { Nombre = "Laura", Apellido = "López", Telefono = "555-9999", Email = "laura@test.com" };
db.Execute(
    "INSERT INTO Contactos (Nombre, Apellido, Telefono, Email) VALUES (@Nombre, @Apellido, @Telefono, @Email)",
    contacto
);
```

## 7. Dapper.Contrib — operaciones CRUD sin SQL

`Dapper.Contrib` es una extensión opcional que agrega métodos CRUD completos **sin necesidad de escribir SQL** para operaciones básicas.

```bash
dotnet add package Dapper.Contrib
```

### 7.1. Métodos disponibles

| Método | Qué hace |
|---|---|
| `db.Insert(obj)` | INSERT, devuelve el `Id` generado |
| `db.Insert(lista)` | INSERT de múltiples objetos de una vez |
| `db.Get<T>(id)` | SELECT por clave primaria |
| `db.GetAll<T>()` | SELECT de todos |
| `db.Update(obj)` | UPDATE de un objeto |
| `db.Update(lista)` | UPDATE de múltiples objetos |
| `db.Delete(obj)` | DELETE de un objeto |
| `db.Delete(lista)` | DELETE de múltiples objetos |
| `db.DeleteAll<T>()` | DELETE de toda la tabla |

### 7.2. Convenciones de nombre

Dapper.Contrib sigue convenciones para deducir tabla y clave primaria sin configuración extra:

| Qué deduce | Cómo lo hace |
|---|---|
| Nombre de tabla | Pluraliza el nombre de la clase: `Contacto` → `Contactos` |
| Clave primaria | Busca una propiedad llamada `Id` |
| Columnas ignoradas | Propiedades marcadas con `[Computed]` |

Si las convenciones no aplican, se pueden sobreescribir con atributos.

### 7.3. Atributos de configuración

```csharp
using Dapper.Contrib.Extensions;

[Table("mis_contactos")]          // nombre de tabla distinto
class Contacto
{
    [Key]                         // PK generada por la base (AUTOINCREMENT)
    public int Id { get; set; }

    public string Nombre   { get; set; } = "";
    public string Apellido { get; set; } = "";

    [Computed]                    // columna calculada, Dapper.Contrib la ignora en INSERT/UPDATE
    public string NombreCompleto => $"{Nombre} {Apellido}";
}
```

| Atributo | Cuándo usarlo |
|---|---|
| `[Table("nombre")]` | El nombre de la tabla no coincide con el de la clase |
| `[Key]` | La PK es autogenerada por la base (AUTOINCREMENT / IDENTITY) |
| `[ExplicitKey]` | La PK la asignamos nosotros (no se autogenera) |
| `[Computed]` | Propiedad calculada o columna que no debe incluirse en INSERT/UPDATE |
| `[Write(false)]` | Propiedad que solo se lee pero nunca se escribe en la base |

### 7.4. `[Key]` vs `[ExplicitKey]`

```csharp
// La base asigna el Id (AUTOINCREMENT) — usar [Key]
class Pedido
{
    [Key]
    public int Id { get; set; }
    public string Descripcion { get; set; } = "";
}

// Nosotros asignamos el Id (ej: GUID o código propio) — usar [ExplicitKey]
class Producto
{
    [ExplicitKey]
    public string Codigo { get; set; } = "";   // "P-001", "P-002"...
    public string Nombre { get; set; } = "";
}
```

Con `[Key]`, Dapper.Contrib omite el campo en el INSERT y deja que la base lo genere.  
Con `[ExplicitKey]`, incluye el campo en el INSERT tal como viene del objeto.

### 7.5. Ejemplo completo

```csharp
using Dapper.Contrib.Extensions;

// Insertar uno — devuelve el Id generado
long id = db.Insert(new Contacto { Nombre = "Elena", Apellido = "Estrada",
                                    Telefono = "555-0011", Email = "elena@example.com" });

// Insertar varios de una vez
db.Insert(new List<Contacto>
{
    new() { Nombre = "Fabio",   Apellido = "Ferreyra", Telefono = "555-0012", Email = "fabio@example.com" },
    new() { Nombre = "Graciela", Apellido = "Godoy",   Telefono = "555-0013", Email = "graciela@example.com" },
});

// Obtener por id
Contacto? elena = db.Get<Contacto>(id);

// Actualizar
elena!.Email = "elena.nueva@example.com";
db.Update(elena);

// Eliminar uno
db.Delete(elena);

// Obtener todos
IEnumerable<Contacto> todos = db.GetAll<Contacto>();
foreach (var c in todos)
    Console.WriteLine($"{c.Id}: {c.Nombre} {c.Apellido}");

// Eliminar todos
db.DeleteAll<Contacto>();
```

### 7.6. Cuándo usar Dapper.Contrib vs Dapper puro

| Situación | Qué usar |
|---|---|
| CRUD simple sobre una sola tabla | `Dapper.Contrib` |
| Consultas con JOINs, filtros o agregaciones | `Dapper` puro (`Query<T>`) |
| INSERT con lógica especial (ON CONFLICT, etc.) | `Dapper` puro (`Execute`) |
| Necesito saber el SQL exacto que se ejecuta | `Dapper` puro |

Ambas pueden convivir en el mismo proyecto. Lo habitual es usar Dapper.Contrib para el CRUD básico y Dapper puro para las consultas más complejas.

## 8. Filtrado: en el servidor vs en el cliente

Esta es una de las decisiones más importantes cuando usamos Dapper.

### Filtrado en el servidor (correcto)

```csharp
var resultado = db.Query<Contacto>(
    "SELECT * FROM Contactos WHERE Nombre > @nombre",
    new { nombre = "Carlos" }
);
```

La base de datos aplica el filtro. Solo viajan por la red (o por el proceso) las filas que coinciden.

### Filtrado en el cliente (ineficiente)

```csharp
var resultado = db.GetAll<Contacto>()
    .Where(c => string.Compare(c.Nombre, "Carlos") > 0);
```

`GetAll<Contacto>()` trae **todos** los registros a memoria. Después LINQ filtra en C#. Si la tabla tiene millones de filas, cargamos todo solo para quedarnos con una parte.

**Regla:** siempre que sea posible, filtrar con `WHERE` en SQL, no con `.Where()` en LINQ después de traer todo.

## 9. El flujo completo

```text
Aplicación C#
    ↓ escribe SQL con parámetros
Dapper
    ↓ envía la consulta
ADO.NET / Provider (ej: Microsoft.Data.Sqlite)
    ↓
SQLite (u otro motor)
    ↑ devuelve filas
ADO.NET / Provider
    ↑ DataReader
Dapper
    ↑ mapea columnas → propiedades
Objeto C# listo para usar
```

Dapper vive entre ADO.NET y nuestro código. No reemplaza la conexión ni el provider: los usa.

## 10. Ejemplo de clase completa

Código del archivo `21.1-dapper.cs`:

```csharp
#!/usr/bin/env dotnet
#:package Microsoft.Data.Sqlite@*
#:package Dapper@*
#:package Dapper.Contrib@*

using Microsoft.Data.Sqlite;
using Dapper;
using System.Data.Common;
using Dapper.Contrib.Extensions;

using DbConnection db = new SqliteConnection("Data Source=contactos.db");
db.Open();

// Crear tabla con ADO.NET puro
using DbCommand cmd = db.CreateCommand();
cmd.CommandText = """
    CREATE TABLE IF NOT EXISTS Contactos (
        Id       INTEGER PRIMARY KEY AUTOINCREMENT,
        Nombre   TEXT NOT NULL,
        Apellido TEXT NOT NULL,
        Telefono TEXT NOT NULL,
        Email    TEXT NOT NULL
    );
    """;
cmd.ExecuteNonQuery();

// Insertar con Dapper.Contrib
db.Insert(new Contacto { Nombre = "Adrián",  Apellido = "Andrade",  Telefono = "555-0001", Email = "adrian@example.com"  });
db.Insert(new Contacto { Nombre = "Betina",  Apellido = "Benites",  Telefono = "555-0002", Email = "betina@example.com"  });
db.Insert(new Contacto { Nombre = "Carlos",  Apellido = "Cabrera",  Telefono = "555-0003", Email = "carlos@example.com"  });

// Listar todos
foreach (Contacto c in db.GetAll<Contacto>())
    Console.WriteLine($"{c.Id}: {c.Nombre} {c.Apellido} - {c.Email}");

// Eliminar por id
db.Delete(db.Get<Contacto>(3));

// Contar con Dapper
int total = db.QuerySingle<int>("SELECT COUNT(*) FROM Contactos");
Console.WriteLine($"Quedan {total} contactos.");

// Filtrar en el servidor (recomendado)
var filtrados = db.Query<Contacto>(
    "SELECT * FROM Contactos WHERE Nombre > @nombre",
    new { nombre = "Carlos" }
);

foreach (Contacto c in filtrados)
    Console.WriteLine($"{c.Id}: {c.Nombre} {c.Apellido}");


class Contacto {
    public int    Id       { get; set; }
    public string Nombre   { get; set; } = "";
    public string Apellido { get; set; } = "";
    public string Telefono { get; set; } = "";
    public string Email    { get; set; } = "";
}
```

## 11. Cuándo usar Dapper vs las alternativas

| Situación | Herramienta recomendada |
|---|---|
| Necesito control total sobre el SQL | ADO.NET o Dapper |
| Consultas complejas con JOINs propios | Dapper |
| CRUD simple sobre entidades relacionadas | EF Core |
| Necesito migraciones y tracking automático | EF Core |
| Proyecto chico, SQL visible, sin ORM completo | Dapper |
| Quiero entender qué pasa por debajo | ADO.NET → Dapper → EF Core |

Dapper es la opción del **punto medio**: no escribe SQL por nosotros como EF Core, pero tampoco nos obliga a leer columna por columna como ADO.NET.

---

## 12. Resumen

- **Dapper** extiende `DbConnection` con métodos como `Query<T>`, `Execute` y `QuerySingle<T>`.
- El mapping es por **nombre de columna ↔ nombre de propiedad**.
- Los parámetros van con `@nombre` y se pasan como objeto anónimo o diccionario.
- **Dapper.Contrib** agrega `Insert`, `Get`, `GetAll`, `Update` y `Delete` sin SQL manual.
- Siempre es mejor filtrar en la base de datos (`WHERE` en SQL) que traer todo y filtrar en memoria con LINQ.
- Dapper no reemplaza ADO.NET: lo usa internamente. Saber ADO.NET ayuda a entender qué hace Dapper por abajo.
