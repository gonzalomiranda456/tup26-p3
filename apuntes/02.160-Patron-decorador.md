# Patrón Decorador en C#

El Decorador agrega comportamiento a un objeto **envolviéndolo** con otro que implementa la **misma interfaz**:

```csharp
decorador3(decorador2(decorador1(objetoOriginal)))
```

## El problema

Una clase que empieza simple termina acumulando logging, validación, retry, cache, métricas... si todo eso se mete adentro, viola SRP y se vuelve imposible de mantener.

## La solución

```csharp
public interface INotificador { void Enviar(string mensaje); }

public class NotificadorEmail : INotificador {
    public void Enviar(string mensaje) => Console.WriteLine($"Email: {mensaje}");
}

// Decorador: misma interfaz, delega al inner
public class NotificadorConLogging : INotificador {
    private readonly INotificador inner;
    public NotificadorConLogging(INotificador inner) => this.inner = inner;

    public void Enviar(string mensaje) {
        Console.WriteLine($"[LOG] Enviando: {mensaje}");
        inner.Enviar(mensaje);
        Console.WriteLine("[LOG] Enviado");
    }
}

// Uso:
INotificador n = new NotificadorConLogging(new NotificadorEmail());
n.Enviar("Hola");
```

El consumidor no sabe si habla con el original o un decorador — solo ve `INotificador`.

## Por qué no herencia

Con herencia necesitarías `EmailConLogging`, `EmailConRetry`, `EmailConLoggingYRetry`, `SmsConLogging`... — explosión combinatoria. Con decoradores tenés piezas combinables:

```csharp
INotificador n = new LoggingNotificador(
    new RetryNotificador(
        new ValidacionNotificador(
            new NotificadorEmail())));
```

## Casos de uso típicos

**Logging:**
```csharp
public class ProcesadorConLogging : IProcesadorPedidos {
    private readonly IProcesadorPedidos inner;
    public ProcesadorConLogging(IProcesadorPedidos inner) => this.inner = inner;
    public void Procesar(int id) {
        Console.WriteLine($"[LOG] Inicio {id}");
        inner.Procesar(id);
        Console.WriteLine($"[LOG] Fin {id}");
    }
}
```

**Cache:**
```csharp
public class BuscadorConCache : IBuscadorProductos {
    private readonly IBuscadorProductos inner;
    private readonly Dictionary<int, string> cache = new();

    public BuscadorConCache(IBuscadorProductos inner) => this.inner = inner;
    public string BuscarNombre(int id) {
        if (cache.TryGetValue(id, out var nombre)) return nombre;
        return cache[id] = inner.BuscarNombre(id);
    }
}
```

**Retry:**
```csharp
public class ServicioConRetry : IServicio {
    private readonly IServicio inner;
    private readonly int intentos;
    public ServicioConRetry(IServicio inner, int intentos = 3) {
        this.inner = inner; this.intentos = intentos;
    }
    public void Ejecutar() {
        for (int i = 1; i <= intentos; i++) {
            try { inner.Ejecutar(); return; }
            catch when (i < intentos) { Console.WriteLine($"Intento {i} falló"); }
        }
    }
}
```

## Con Dependency Injection

```csharp
builder.Services.AddScoped<SmtpEmailSender>();
builder.Services.AddScoped<IEmailSender>(sp => {
    IEmailSender inner = sp.GetRequiredService<SmtpEmailSender>();
    inner = new RetryEmailSender(inner);
    inner = new LoggingEmailSender(inner);
    return inner;
});
```

## El orden importa

`Logging(Validacion(Servicio))` → primero loguea, después valida.  
`Validacion(Logging(Servicio))` → si falla la validación, ni llega al logging.  
Ambas pueden ser correctas dependiendo de la intención.

## Ejemplo en .NET: Stream

```csharp
using var archivo = new FileStream("datos.gz", FileMode.Open);
using var gzip    = new GZipStream(archivo, CompressionMode.Decompress);
using var lector  = new StreamReader(gzip, Encoding.UTF8);
```

`FileStream` → bytes; `GZipStream` → descompresión; `StreamReader` → texto. Cada capa agrega una responsabilidad sin modificar la anterior.

## Comparación

| | Decorador | Herencia |
|---|---|---|
| Combinable | Sí | No fácilmente |
| Desacoplado | Sí (usa interfaz) | No (acoplado a clase base) |
| Aplicable en runtime | Sí | No |
| SRP | Respeta | Suele violar |

## Cuándo usarlo

Cuando una clase acumula: `Validar(); LogInicio(); MedirTiempo(); VerificarPermisos(); EjecutarReal(); LogFin();` — es señal de que esas capas deberían ser decoradores.

Usalo para: logging, cache, retry, validación, autorización, métricas, compresión, cifrado.  
No lo uses si la clase es trivial, si el orden de capas se vuelve confuso, o si estás decorando todo "porque patrones" sin necesidad real.
