## 1. El problema de la clase 2

Tenemos una API que funciona. Acepta requests, persiste datos, devuelve JSON. Pero el "cliente" sigue siendo REST Client — una herramienta para desarrolladores, no algo que pueda usar cualquier persona.

Los usuarios reales necesitan una **interfaz visual**: botones, listas, formularios. Algo que corra en un browser.

En esta clase construimos esa interfaz con **Blazor**. Al terminar, vamos a tener una página web que lista las tareas, permite crear nuevas y marcarlas como completadas — todo consumiendo la API de las clases anteriores.

## 2. Cómo funciona una página web

Antes de Blazor, necesitamos entender qué es lo que el browser realmente muestra.

Un browser solo entiende tres lenguajes:

- **HTML** — define la *estructura* del contenido: qué elementos hay y cómo se organizan.
- **CSS** — define la *presentación*: colores, tamaños, disposición.
- **JavaScript** — define el *comportamiento*: qué pasa cuando el usuario interactúa.

Cuando entrás a una URL, el servidor devuelve HTML. El browser lo interpreta y construye el **DOM** (*Document Object Model*): una representación en árbol de todos los elementos de la página.

```
┌─ document ──────────────────────────────┐
│  ┌─ html ────────────────────────────┐  │
│  │  ┌─ body ──────────────────────┐  │  │
│  │  │  ┌─ h1 ─┐  ┌─ ul ────────┐ │  │  │
│  │  │  │Tareas│  │ ┌─ li ─────┐ │ │  │  │
│  │  │  └──────┘  │ │Estudiar  │ │ │  │  │
│  │  │            │ └──────────┘ │ │  │  │
│  │  │            │ ┌─ li ─────┐ │ │  │  │
│  │  │            │ │Practicar │ │ │  │  │
│  │  │            │ └──────────┘ │ │  │  │
│  │  │            └─────────────┘ │  │  │
│  │  └─────────────────────────────┘  │  │
│  └────────────────────────────────────┘  │
└──────────────────────────────────────────┘
```

El DOM es dinámico: JavaScript puede modificarlo en tiempo real sin recargar la página. Cuando marcás una tarea como completada y el tachado aparece instantáneamente, es JavaScript modificando el DOM.

El problema: JavaScript es un lenguaje que hay que aprender por separado, con una sintaxis diferente a C# y muchas inconsistencias históricas.

**Blazor permite escribir ese código de interactividad en C# en lugar de JavaScript.**

## 3. El problema de construir UI con HTML puro

Imaginá que querés mostrar una lista de tareas. En HTML puro sería:

```html
<ul>
  <li>Estudiar HTTP</li>
  <li>Practicar Minimal API</li>
  <li>Hacer el parcial</li>
</ul>
```

Esto funciona si la lista es estática y siempre son esas tres tareas. Pero en la práctica:

- La lista viene de una base de datos — no la conocés al momento de escribir el HTML.
- El usuario puede agregar y eliminar tareas — el HTML tiene que actualizarse.
- Cada tarea tiene estado (completada o no) que afecta cómo se muestra.

Para manejar datos dinámicos, el HTML tiene que **generarse en función del estado actual**. Si tenés diez tareas, generás diez `<li>`. Si el usuario agrega una, generás once. Si filtra por completadas, generás solo las completadas.

La forma natural de pensar esto es como una **función**:

```
render(estado) → HTML
```

Dado el estado actual, la función produce el HTML correspondiente. Cada vez que el estado cambia, llamás la función de nuevo y actualizás la pantalla.

Eso es exactamente lo que hace un **componente**.

## 4. La solución: un componente

Un componente es una pieza de UI que combina tres cosas:

1. **Estado** — los datos que necesita para mostrarse (`List<Tarea>`, un booleano de carga, etc.).
2. **Plantilla** — el HTML que describe cómo se ve en función del estado.
3. **Comportamiento** — los métodos que modifican el estado cuando el usuario interactúa.

Cuando el estado cambia, el componente se **re-renderiza**: recalcula su plantilla con el nuevo estado y actualiza el DOM.

El ciclo es:

```
1. Componente se inicializa con estado inicial
2. Se renderiza la plantilla con ese estado
3. El usuario interactúa (click, tipeo, etc.)
4. El evento llama a un método del componente
5. El método modifica el estado
6. El componente se re-renderiza con el nuevo estado
7. Volver a 3
```

Comparalo con el modelo de Minimal API:

```
Minimal API:  HTTP request  → función → JSON response
Blazor:       evento del DOM → método  → re-render del DOM
```

La estructura es la misma. En ambos casos hay un evento que dispara una función que produce un resultado. La diferencia es el medio: HTTP vs DOM, JSON vs HTML.

## 5. Blazor y el modelo de componentes

Blazor implementa este modelo de componentes en C#. Un componente Blazor es un archivo `.razor` que mezcla HTML con C#:

```razor
@* Parte HTML: la plantilla *@
<h1>Mis Tareas</h1>
<ul>
    @foreach (var tarea in tareas)
    {
        <li>@tarea.Texto</li>
    }
</ul>

@* Parte C#: el estado y el comportamiento *@
@code {
    List<Tarea> tareas = new() {
        new(1, "Estudiar HTTP",        false),
        new(2, "Practicar Minimal API", false),
    };
}
```

El `@` es el conector entre HTML y C#:

- `@foreach` — ejecuta C# dentro del HTML.
- `@tarea.Texto` — inserta el valor de una expresión C# en el HTML.
- `@code { }` — bloque donde van las propiedades, campos y métodos del componente.

## 6. Blazor Server vs Blazor WASM

Blazor tiene dos modos de ejecución con diferencias importantes:

### Blazor Server

El componente **corre en el servidor**. El browser solo recibe HTML y una conexión persistente (SignalR). Cuando el usuario interactúa, el evento viaja al servidor por esa conexión, el servidor ejecuta el método C#, calcula el nuevo HTML y envía solo las diferencias de vuelta al browser.

```
Browser                     Servidor
  │── click en botón ──────►│
  │                          │ ejecuta método C#
  │                          │ calcula nuevo HTML
  │◄── diff del DOM ─────────│
  │ actualiza la pantalla    │
```

**Ventajas:** simple de entender, el código C# accede directamente a la base de datos, no hay CORS.
**Desventaja:** cada interacción necesita un viaje de red al servidor.

### Blazor WASM

El componente **corre en el browser** mediante WebAssembly — una tecnología que permite ejecutar código compilado (no solo JavaScript) dentro del browser. El runtime de .NET completo se descarga al browser la primera vez.

```
Browser                     Servidor
  │                          │
  │ ejecuta C# localmente    │
  │ actualiza el DOM         │
  │── fetch a la API ────────► (cuando necesita datos)
  │◄── JSON ─────────────────│
```

**Ventajas:** interacciones instantáneas (sin viaje de red), puede funcionar offline.
**Desventaja:** descarga inicial grande, no puede acceder directamente a la base de datos.

### ¿Cuál usamos?

**Blazor Server** para esta clase. El modelo es más simple: el componente puede acceder directamente a la base de datos sin pasar por HTTP. Eso nos permite enfocarnos en entender los componentes sin agregar la complejidad de fetch + CORS.

En la clase 4 hacemos la transición: el componente pasa a consumir la API via HTTP, que es la arquitectura correcta para producción.

## 8. Anatomía de un componente

Antes de escribir el componente de tareas, veamos la anatomía completa con un ejemplo mínimo:

```razor
@page "/contador"
@inject ILogger<Contador> Logger

<h1>Contador: @cuenta</h1>
<button @onclick="Incrementar">+1</button>

@code {
    int cuenta = 0;

    void Incrementar() {
        cuenta++;
        Logger.LogInformation("Cuenta incrementada a {Cuenta}", cuenta);
    }
}
```

Línea por línea:

**`@page "/contador"`** — declara que este componente es una página accesible en la URL `/contador`. Sin esta directiva, el componente existe pero no tiene URL propia.

**`@inject ILogger<Contador> Logger`** — inyección de dependencias. Pide un `ILogger<Contador>` al contenedor de DI y lo asigna a la variable `Logger`. Funciona igual que en Minimal API, pero con la sintaxis `@inject`.

**`<h1>Contador: @cuenta</h1>`** — `@cuenta` es una expresión C#. Blazor evalúa `cuenta` e inserta su valor en el HTML. Cada vez que `cuenta` cambia y el componente se re-renderiza, este texto se actualiza.

**`@onclick="Incrementar"`** — conecta el evento `click` del botón al método `Incrementar`. Cuando el usuario hace click, Blazor llama `Incrementar()`, que modifica el estado. Blazor detecta el cambio y re-renderiza el componente.

**`@code { }`** — todo lo que va acá es C# puro: campos, propiedades, métodos, ciclo de vida.

### Ciclo de vida básico

Blazor llama métodos especiales en momentos clave del ciclo de vida del componente. El más útil para nosotros:

```razor
@code {
    protected override async Task OnInitializedAsync() {
        // Se ejecuta una vez cuando el componente se monta por primera vez
        // Ideal para cargar datos iniciales
        tareas = await CargarTareas();
    }
}
```

`OnInitializedAsync` es el equivalente del "constructor" para la lógica de inicialización. Se llama automáticamente cuando el componente aparece en la página.

## 9. Estado y re-render

El re-render en Blazor es automático en la mayoría de los casos: después de cada event handler (`@onclick`, `@oninput`, etc.), Blazor re-renderiza el componente.

Pero hay un caso que requiere atención: cuando modificás el estado **desde fuera de un event handler** (por ejemplo, desde un callback asincrónico), necesitás avisarle a Blazor que el estado cambió:

```razor
@code {
    List<Tarea> tareas = new();
    bool cargando = true;

    protected override async Task OnInitializedAsync() {
        // Esto corre en un contexto asincrónico
        // Blazor re-renderiza automáticamente al terminar
        tareas   = await ObtenerTareasDeLaBD();
        cargando = false;
        // No necesitamos llamar StateHasChanged() — OnInitializedAsync lo hace por nosotros
    }

    async Task AgregarTarea() {
        // Event handler — Blazor re-renderiza automáticamente al terminar
        await GuardarEnBD(nueva);
        tareas = await ObtenerTareasDeLaBD();
    }
}
```

La regla simple: **en event handlers y en `OnInitializedAsync`, el re-render es automático**. En cualquier otro contexto (timers, callbacks de librerías externas), llamás `StateHasChanged()` manualmente.

### Mostrar estado de carga

Como `OnInitializedAsync` es asincrónico, hay un momento entre que el componente aparece y que los datos están listos. Durante ese tiempo podemos mostrar un indicador:

```razor
@if (cargando) {
    <p>Cargando...</p>
} else {
    <ul>
        @foreach (var tarea in tareas)
        {
            <li>@tarea.Texto</li>
        }
    </ul>
}

@code {
    List<Tarea> tareas = new();
    bool cargando = true;

    protected override async Task OnInitializedAsync() {
        tareas   = await db.Tareas.ToListAsync();
        cargando = false;
    }
}
```

## 10. Construir la UI de tareas paso a paso

Vamos a construir el componente `Pages/Index.razor` de a pasos. Cada paso agrega una funcionalidad sobre la anterior.

### Paso 1 — listar las tareas

```razor
@page "/"
@inject TareasContext db

<h1>Mis Tareas</h1>

@if (cargando) {
    <p>Cargando...</p>
} else {
    <ul>
        @foreach (var tarea in tareas)
        {
            <li>@tarea.Texto</li>
        }
    </ul>
}

@code {
    List<Tarea> tareas = new();
    bool cargando = true;

    protected override async Task OnInitializedAsync() {
        tareas   = await db.Tareas.ToListAsync();
        cargando = false;
    }
}
```

`@inject TareasContext db` pide el contexto de base de datos. El contenedor de DI lo entrega porque lo registramos en `Program.cs`.

Ejecutá con `dotnet run`. Vas a ver la lista de tareas que tenés en la base de datos.

### Paso 2 — mostrar el estado de completada

```razor
<ul>
    @foreach (var tarea in tareas)
    {
        <li style="text-decoration: @(tarea.Completada ? "line-through" : "none")">
            @tarea.Texto
        </li>
    }
</ul>
```

`@(expresión)` evalúa cualquier expresión C# dentro del HTML. Acá usamos un operador ternario para decidir si mostrar el texto tachado o no.

### Paso 3 — marcar como completada con un checkbox

```razor
<ul>
    @foreach (var tarea in tareas)
    {
        <li>
            <input type="checkbox"
                   checked="@tarea.Completada"
                   @onchange="() => ToggleCompletada(tarea)" />
            <span style="text-decoration: @(tarea.Completada ? "line-through" : "none")">
                @tarea.Texto
            </span>
        </li>
    }
</ul>

@code {
    // ... OnInitializedAsync igual que antes ...

    async Task ToggleCompletada(Tarea tarea) {
        var actualizada = tarea with { Completada = !tarea.Completada };
        db.Entry(tarea).CurrentValues.SetValues(actualizada);
        await db.SaveChangesAsync();
        tareas = await db.Tareas.ToListAsync();
    }
}
```

`@onchange="() => ToggleCompletada(tarea)"` — la lambda captura la `tarea` del `foreach` actual. Cada checkbox está conectado a su propia tarea.

### Paso 4 — agregar una tarea nueva

```razor
<div>
    <input @bind="textoNueva" placeholder="Nueva tarea..." />
    <button @onclick="AgregarTarea">Agregar</button>
</div>

@code {
    string textoNueva = "";

    async Task AgregarTarea() {
        if (string.IsNullOrWhiteSpace(textoNueva)) return;

        db.Tareas.Add(new Tarea(0, textoNueva.Trim(), false));
        await db.SaveChangesAsync();

        textoNueva = "";                             // limpiar el campo
        tareas     = await db.Tareas.ToListAsync();  // recargar la lista
    }
}
```

**`@bind="textoNueva"`** — vinculación bidireccional entre el campo de texto y la variable `textoNueva`. Cuando el usuario escribe, `textoNueva` se actualiza. Cuando el código modifica `textoNueva`, el campo de texto se actualiza. Un `@bind` equivale a dos líneas: `value="@textoNueva"` y `@onchange="e => textoNueva = e.Value.ToString()"`.

### Paso 5 — eliminar una tarea

```razor
<li>
    <input type="checkbox" ... />
    <span ...>@tarea.Texto</span>
    <button @onclick="() => EliminarTarea(tarea)">✕</button>
</li>

@code {
    async Task EliminarTarea(Tarea tarea) {
        db.Tareas.Remove(tarea);
        await db.SaveChangesAsync();
        tareas = await db.Tareas.ToListAsync();
    }
}
```

## 11. El componente completo

```razor
@page "/"
@inject TareasContext db

<h1>Mis Tareas</h1>

<div>
    <input @bind="textoNueva" placeholder="Nueva tarea..." />
    <button @onclick="AgregarTarea">Agregar</button>
</div>

@if (cargando) {
    <p>Cargando...</p>
} else if (!tareas.Any()) {
    <p>No hay tareas. ¡Agregá una!</p>
} else {
    <ul>
        @foreach (var tarea in tareas)
        {
            <li>
                <input type="checkbox"
                       checked="@tarea.Completada"
                       @onchange="() => ToggleCompletada(tarea)" />
                <span style="text-decoration: @(tarea.Completada ? "line-through" : "none")">
                    @tarea.Texto
                </span>
                <button @onclick="() => EliminarTarea(tarea)">✕</button>
            </li>
        }
    </ul>
}

@code {
    List<Tarea> tareas    = new();
    string      textoNueva = "";
    bool        cargando   = true;

    protected override async Task OnInitializedAsync() {
        tareas   = await db.Tareas.ToListAsync();
        cargando = false;
    }

    async Task AgregarTarea() {
        if (string.IsNullOrWhiteSpace(textoNueva)) return;

        db.Tareas.Add(new Tarea(0, textoNueva.Trim(), false));
        await db.SaveChangesAsync();

        textoNueva = "";
        tareas     = await db.Tareas.ToListAsync();
    }

    async Task ToggleCompletada(Tarea tarea) {
        var actualizada = tarea with { Completada = !tarea.Completada };
        db.Entry(tarea).CurrentValues.SetValues(actualizada);
        await db.SaveChangesAsync();
        tareas = await db.Tareas.ToListAsync();
    }

    async Task EliminarTarea(Tarea tarea) {
        db.Tareas.Remove(tarea);
        await db.SaveChangesAsync();
        tareas = await db.Tareas.ToListAsync();
    }
}
```

Abrí `http://localhost:5000` en el browser. Vas a ver la lista de tareas, podés agregar nuevas, marcarlas como completadas y eliminarlas. Los cambios persisten — si recargás la página, todo sigue ahí.

## 12. Resumen

```
Una página web = HTML + CSS + JavaScript
  HTML   → estructura (qué elementos hay)
  CSS    → presentación (cómo se ven)
  JS     → comportamiento (qué pasa al interactuar)

El DOM es el árbol de elementos que el browser mantiene en memoria.
Modificar el DOM = actualizar lo que se ve en pantalla sin recargar.

Un componente Blazor combina tres cosas:
  Estado      → campos y propiedades en @code { }
  Plantilla   → HTML con expresiones @C# intercaladas
  Comportamiento → métodos llamados por eventos (@onclick, @onchange)

Sintaxis clave:
  @variable            → insertar valor en el HTML
  @(expresión)         → insertar expresión C# en el HTML
  @if / @foreach       → estructuras de control en el HTML
  @onclick="Método"    → conectar evento a método
  @bind="variable"     → vinculación bidireccional campo ↔ variable
  @inject Tipo nombre  → inyección de dependencias
  @page "/ruta"        → declarar URL del componente

Ciclo de vida:
  OnInitializedAsync() → se llama una vez al montar el componente
                         ideal para cargar datos iniciales

Blazor Server vs WASM:
  Server → el componente corre en el servidor, browser recibe diffs
  WASM   → el componente corre en el browser vía WebAssembly

Lo que cambia respecto a la clase 2:
  Antes: el cliente era REST Client (herramienta de desarrollador)
  Ahora: el cliente es un componente Blazor (interfaz para usuarios)
  La base de datos y los endpoints son los mismos
```

**Lo que le falta a esta arquitectura:**

El componente Blazor accede **directamente** a la base de datos con `@inject TareasContext`. Eso funciona en Blazor Server, pero no es la arquitectura correcta: acopla el frontend al backend. En la próxima clase separamos los dos proyectos y hacemos que el componente consuma la API via HTTP.
