# Programación Asincrónica en C# — Guía Práctica

La programación asincrónica es fundamental en C# para desarrollar aplicaciones que no bloqueen el hilo de ejecución principal (por ejemplo, la interfaz de usuario o el hilo que atiende peticiones en un servidor web) mientras se realizan operaciones de entrada/salida (I/O), como consultar una base de datos, descargar un archivo de internet o leer del disco.

---

## 1. La Idea Central: I/O-Bound vs. CPU-Bound

La asincronía con `async` y `await` en C# está diseñada principalmente para **operaciones de entrada/salida (I/O-Bound)**:
* **Sincrónico (Bloqueante):** Tu programa le pide datos a la base de datos y el hilo de ejecución se queda cruzado de brazos esperando a que la red y el disco de la base de datos respondan. El hilo no puede hacer otra cosa mientras tanto.
* **Asincrónico (No bloqueante):** Tu programa inicia la petición a la base de datos y **libera el hilo** para que pueda hacer otras tareas (como responder a clics de UI o atender otras peticiones web). Cuando la base de datos responde, se retoma la ejecución en el punto donde se quedó.

> [!NOTE]
> No creamos hilos nuevos manualmente. Delegamos la espera al sistema operativo o al hardware de red, optimizando drásticamente el uso de recursos.

---

## 2. Las Palabras Clave: `async` y `await`

* **`async`**: Es un modificador de método. Su única función es habilitar el uso de la palabra clave `await` dentro del método y permitir que el compilador transforme el método en una máquina de estados.
* **`await`**: Se coloca antes de una tarea (`Task`). Suspende la ejecución del método actual de manera no bloqueante hasta que la tarea finalice. Mientras espera, el hilo actual queda libre.

```csharp
// Método sincrónico
void Procesar() {
    string datos = LeerArchivo("datos.txt"); // Bloquea el hilo aquí
    Console.WriteLine(datos);
}

// Método asincrónico equivalente
async Task ProcesarAsync() {
    string datos = await LeerArchivoAsync("datos.txt"); // Libera el hilo aquí mientras lee
    Console.WriteLine(datos);
}
```

---

## 3. Tipos de Retorno Asincrónicos

Al escribir un método `async`, el tipo de retorno debe ser uno de los siguientes:

| Tipo de Retorno    | Cuándo usarlo                                                                                                                                               | Ejemplo de firma                                    |
| :----------------- | :---------------------------------------------------------------------------------------------------------------------------------------------------------- | :-------------------------------------------------- |
| **`Task`**         | El método hace un trabajo asincrónico pero **no devuelve ningún valor** (equivalente asincrónico a `void`).                                                 | `async Task GuardarDatosAsync(...)`                 |
| **`Task<T>`**      | El método hace un trabajo asincrónico y **devuelve un valor de tipo `T`** al terminar.                                                                      | `async Task<Contacto> ObtenerContactoAsync(int id)` |
| **`ValueTask<T>`** | Optimización de memoria para métodos que en la mayoría de los casos resuelven su resultado sincrónicamente y no necesitan instanciar una `Task` en el heap. | `async ValueTask<int> LeerCacheAsync(...)`          |
| **`void`**         | **Únicamente** para manejadores de eventos (*event handlers* en interfaces visuales).                                                                       | `async void OnClick(object sender, EventArgs e)`    |

> [!WARNING]
> **Evitá `async void` en métodos comunes.** Si ocurre una excepción dentro de un método `async void`, esta no puede ser capturada por un bloque `try-catch` externo y tirará abajo la aplicación entera de forma abrupta.

## 4. Uso Práctico con Librerías Comunes

En el día a día, casi nunca crearás `Task` desde cero. En su lugar, **consumirás métodos asincrónicos** provistos por .NET o por librerías externas. La convención en C# es que estos métodos terminen con el sufijo `Async`.

### A. Consumo de APIs Web (`HttpClient`)
```csharp
using System.Net.Http;
using System.Net.Http.Json;

var cliente = new HttpClient();

// Obtener contenido de texto de forma asincrónica
string html = await cliente.GetStringAsync("https://api.github.com/users/octocat");

// Obtener y deserializar JSON directamente a un objeto
Contacto? c = await cliente.GetFromJsonAsync<Contacto>("https://api.ejemplo.com/contactos/1");
```

### B. Lectura y Escritura de Archivos (`System.IO`)
```csharp
using System.IO;

string ruta = "archivo.txt";

// Escribir texto de forma asincrónica
await File.WriteAllTextAsync(ruta, "Hola, mundo asincrónico!");

// Leer texto de forma asincrónica
string contenido = await File.ReadAllTextAsync(ruta);
```

### C. Persistencia con Dapper (`Dapper`)
Dapper provee extensiones asincrónicas equivalentes para todos sus métodos sobre `DbConnection`.
```csharp
using System.Data;
using Microsoft.Data.Sqlite;
using Dapper;

using IDbConnection db = new SqliteConnection("Data Source=agenda.db");

// Consulta asincrónica de múltiples registros
var contactos = await db.QueryAsync<Contacto>("SELECT * FROM Contactos");

// Consulta asincrónica de un único registro
var contacto = await db.QuerySingleOrDefaultAsync<Contacto>(
    "SELECT * FROM Contactos WHERE Id = @id", new { id = 1 }
);

// Ejecutar comando asincrónico (INSERT/UPDATE/DELETE)
int filasAfectadas = await db.ExecuteAsync(
    "UPDATE Contactos SET Favorito = @fav WHERE Id = @id",
    new { fav = true, id = 5 }
);
```

## 5. Reglas de Oro y Buenas Prácticas

### 1. "Async All the Way" (Asincrónico de punta a punta)
La asincronía es contagiosa. Si un método es asincrónico, el método que lo llama también debería ser `async` y usar `await`, propagándose hasta el punto de entrada de la aplicación (`Main`, un endpoint de API o un controlador).

### 2. Nunca bloquees usando `.Result` o `.Wait()`
Si necesitás esperar un método asincrónico desde un contexto sincrónico, **no uses** las propiedades bloqueantes de `Task`:

```csharp
// ✗ MAL — Bloquea el hilo actual y puede causar Deadlocks (bloqueos mutuos)
var datos = ObtenerDatosAsync().Result; 
ObtenerDatosAsync().Wait();

// ✓ BIEN — Espera de forma no bloqueante usando await
var datos = await ObtenerDatosAsync();
```

Si estás obligado a llamar código asincrónico desde un método sincrónico (por ejemplo, al implementar una interfaz sincrónica de una librería externa que no podés modificar), podés usar descartes o correrlo en segundo plano, pero la regla general es siempre propagar el `await`.

### 3. El manejo de excepciones es transparente
Con `await`, el manejo de excepciones asincrónicas funciona exactamente igual que el código sincrónico mediante bloques `try-catch`:

```csharp
try {
    string datos = await cliente.GetStringAsync("https://url-invalida.com");
} catch (HttpRequestException ex) {
    Console.WriteLine($"Error de red: {ex.Message}");
}
```

## 6. Ejecución en Paralelo Simultánea

A veces querés disparar múltiples tareas al mismo tiempo en lugar de esperar a que termine una para empezar la otra. Podés lograr esto guardando las tareas en variables (sin usar `await` inmediatamente) y luego esperándolas juntas con `Task.WhenAll`.

```csharp
// Dispara las tres peticiones HTTP al mismo tiempo
Task<string> tarea1 = cliente.GetStringAsync("https://api.com/datos1");
Task<string> tarea2 = cliente.GetStringAsync("https://api.com/datos2");
Task<string> tarea3 = cliente.GetStringAsync("https://api.com/datos3");

// Espera a que las tres terminen (las operaciones corren en paralelo en la red)
string[] resultados = await Task.WhenAll(tarea1, tarea2, tarea3);

string resultado1 = resultados[0];
string resultado2 = resultados[1];
string resultado3 = resultados[2];
```

## 7. Cancelación de Operaciones (`CancellationToken`)

Las librerías asincrónicas suelen aceptar un parámetro opcional llamado `CancellationToken`. Esto permite que tu aplicación aborte una operación larga (como una descarga o consulta pesada) si el usuario hace clic en "Cancelar" o si expira un tiempo límite (*timeout*).

```csharp
using System.Threading;

// Crear una fuente de tokens de cancelación con un tiempo límite de 3 segundos
using var cts = new CancellationTokenSource(TimeSpan.FromSeconds(3));
CancellationToken token = cts.Token;

try {
    // Si la API tarda más de 3 segundos, GetStringAsync lanzará OperationCanceledException
    string html = await cliente.GetStringAsync("https://api.lenta.com/datos", token);
} catch (OperationCanceledException) {
    Console.WriteLine("La operación fue cancelada por timeout.");
}
```
