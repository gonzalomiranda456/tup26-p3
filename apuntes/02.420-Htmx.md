## 1. El punto de partida: cómo funciona la web clásica

Antes de cualquier framework, la web funcionaba con un mecanismo simple y robusto. El navegador muestra una página HTML. Dentro de esa página hay dos cosas que pueden iniciar una nueva petición al servidor:

- Un **enlace** (`<a href="...">`): al hacer clic, el navegador pide esa URL por GET.
- Un **formulario** (`<form action="..." method="post">`): al enviarlo, el navegador manda sus campos a esa URL.

En ambos casos pasa lo mismo: el navegador hace la petición, el servidor responde con **una página HTML completa y nueva**, y el navegador **descarta la página actual y dibuja la nueva** de cero.

A este modelo se lo llama **hipertexto**: documentos HTML enlazados entre sí, donde navegar es reemplazar un documento por otro. Es el modelo para el que se diseñó la web, y tiene virtudes enormes: es simple, no requiere JavaScript, funciona en cualquier navegador, y el estado vive en un solo lugar (el servidor). El primer principio a retener es ese: **en la web clásica, solo dos elementos —enlaces y formularios— pueden hablar con el servidor, solo responden a dos eventos —clic y envío—, y la respuesta siempre reemplaza la página entera.**

---

## 2. El problema: recargar toda la página es un desperdicio

Ese modelo tiene una incomodidad concreta. Imaginá una pantalla con una lista a la izquierda y un panel de detalle a la derecha. Hacés clic en un elemento de la lista para ver su detalle. En la web clásica, eso pide una página nueva y **redibuja todo**: la lista de la izquierda —que no cambió— se descarta y se vuelve a descargar y pintar, igual que el encabezado, el menú y todo lo demás.

Las consecuencias se sienten:

- **Parpadeo**: la pantalla se pone en blanco un instante mientras llega y se dibuja la página nueva.
- **Se pierde el estado visual**: la posición del scroll, el foco, lo que el usuario tenía a medio escribir.
- **Se transfiere de más**: el servidor manda HTML de zonas que ya estaban iguales.

La pregunta natural es: *si lo único que cambió es el panel de la derecha, ¿por qué no actualizar solo ese panel y dejar el resto intacto?* Esa es exactamente la mejora que queremos. El problema es que las herramientas de la web clásica no la permiten: un enlace o un formulario reemplazan **todo**, no saben hacer un reemplazo parcial.

---

## 3. La solución de las SPA (y su costo)

La industria resolvió esto con las **Single Page Applications** (SPA): aplicaciones donde el navegador carga una vez una base, y a partir de ahí JavaScript se encarga de todo. La mecánica es:

1. El usuario hace algo (un clic). Código JavaScript lo intercepta.
2. Ese código hace una petición al servidor con `fetch`, y el servidor responde con **datos crudos**, normalmente en formato **JSON**.
3. Más código JavaScript toma esos datos, **construye el HTML** correspondiente en el navegador, y lo inserta en el lugar justo del documento.

Esto resuelve el problema: solo se actualiza la parte que cambió, sin recargar nada. Pero tiene un costo grande, y entenderlo es clave para entender por qué existe HTMX:

- **La lógica de presentación se duplica.** El servidor sabe cómo son los datos; el navegador sabe cómo convertirlos en HTML. Las reglas de "cómo se ve un contacto" viven ahora en *dos* lados.
- **Hay estado en dos lugares.** El servidor tiene la verdad de los datos, pero el navegador mantiene su propia copia para dibujar, y mantener ambas sincronizadas es una fuente constante de bugs.
- **Hace falta mucho JavaScript** y un andamiaje considerable (frameworks, herramientas de build, etc.) para algo que conceptualmente era simple.

> El principio: **la SPA gana interactividad parcial a cambio de duplicar la presentación y el estado.** Ese intercambio vale la pena para apps muy interactivas (editores colaborativos, juegos), pero es carísimo para la enorme mayoría de pantallas, que son formularios, listas y detalles.

## 4. La idea central de HTMX: generalizar el hipertexto

Acá entra HTMX, con una observación tan simple que parece tramposa. Recordá las tres limitaciones de la web clásica (sección 1):

1. Solo **enlaces y formularios** pueden hablar con el servidor.
2. Solo responden a **clic y envío**.
3. La respuesta siempre **reemplaza la página entera**.

HTMX pregunta: *¿y si quitamos esas tres limitaciones arbitrarias?* ¿Qué pasaría si **cualquier** elemento pudiera hacer una petición, ante **cualquier** evento, y la respuesta pudiera reemplazar **cualquier** parte de la página? Eso es, literalmente, todo lo que hace HTMX. No inventa un paradigma nuevo: **generaliza el que la web ya tenía.**

HTMX es una librería de JavaScript chica (unos 14 kb) que leés en el `<head>` de tu página. Una vez cargada, lee **atributos HTML** que vos escribís —los que empiezan con `hx-`— y, por debajo, hace el `fetch` y el reemplazo del DOM que vos habrías tenido que programar a mano en una SPA. La diferencia es que vos no escribís ese JavaScript: lo declarás con atributos.

```html
<!-- web clásica: solo un enlace, reemplaza todo -->
<a href="/contactos/5">Ver contacto</a>

<!-- HTMX: cualquier elemento, cualquier evento, reemplazo parcial -->
<button hx-get="/contactos/5" hx-target="#detalle">Ver contacto</button>
```

El botón de HTMX hace una petición GET a `/contactos/5`, recibe HTML del servidor, y lo mete dentro del elemento `#detalle` —sin tocar el resto de la página—. Mismo espíritu que el enlace, pero sin las tres limitaciones.

> El principio: **HTMX es hipertexto sin restricciones.** Si entendés cómo funcionaba un `<a href>`, ya entendés HTMX; solo que ahora el "enlace" puede ser cualquier elemento, dispararse con cualquier evento, y actualizar solo un pedazo de la pantalla.

## 5. Los cuatro atributos que lo explican casi todo

Toda interacción de HTMX se arma combinando, como mucho, cuatro decisiones. Cada una corresponde a un atributo, y juntas responden a las cuatro preguntas de cualquier interacción:

| Pregunta            | Atributo                | Responde...                                  |
| ------------------- | ----------------------- | -------------------------------------------- |
| ¿**Qué** pido?      | `hx-get` / `hx-post`    | a qué URL, con qué verbo HTTP                |
| ¿**Cuándo** lo pido?| `hx-trigger`            | qué evento dispara la petición               |
| ¿**Dónde** lo pongo?| `hx-target`             | en qué elemento del DOM va la respuesta      |
| ¿**Cómo** lo pongo? | `hx-swap`               | de qué forma se inserta el HTML recibido     |

Esa es la columna vertebral. Casi todo lo demás de HTMX son variaciones o comodidades sobre estas cuatro. Las vemos una por una.

---

## 6. `hx-get` y `hx-post`: qué pedir

Estos atributos dicen **a qué URL** hacer la petición y **con qué verbo HTTP**. Hay uno por cada verbo: `hx-get`, `hx-post`, `hx-put`, `hx-patch`, `hx-delete`.

```html
<button hx-get="/contactos/5">Ver</button>
<button hx-post="/contactos">Crear</button>
<button hx-delete="/contactos/5">Borrar</button>
```

La elección del verbo sigue la semántica de siempre: `GET` para *leer* sin cambiar nada, `POST` (o `PUT`/`PATCH`/`DELETE`) para *modificar* el estado del servidor. Tener ese atributo en un elemento es lo que lo convierte en "activo": le dice a HTMX "este elemento puede hacer peticiones". El valor es la URL, igual que el `href` de un enlace o el `action` de un formulario.

> El principio: **`hx-get`/`hx-post` reemplazan al `href` y al `action`.** Es la misma idea de siempre —una URL y un verbo— pero ahora aplicable a cualquier elemento, no solo a enlaces y formularios.

---

## 7. `hx-trigger`: cuándo pedir

En la web clásica, los eventos estaban fijos: un enlace dispara con el clic, un formulario con el envío. `hx-trigger` rompe esa fijeza: vos elegís **qué evento** del DOM dispara la petición.

```html
<!-- al hacer clic (es el default para un botón, podés omitirlo) -->
<button hx-get="/datos" hx-trigger="click">Cargar</button>

<!-- al cargarse el elemento en la página -->
<div hx-get="/lista" hx-trigger="load">Cargando...</div>

<!-- al cambiar el valor de un input -->
<input hx-get="/buscar" hx-trigger="change" />
```

Cada elemento tiene un evento por defecto razonable (clic para botones, `submit` para formularios, `change` para inputs), así que muchas veces ni hace falta escribir `hx-trigger`. Pero cuando lo escribís, podés afinarlo con **modificadores**, que son la parte realmente potente:

```html
<input
    hx-get="/buscar"
    hx-trigger="input changed delay:300ms" />
```

Esto se lee: dispará cuando llegue el evento `input` (cada tecla), pero **solo si el valor cambió** (`changed`, evita disparar si tocaste una flecha sin cambiar texto), y **esperá 300 ms desde la última tecla** antes de mandar la petición (`delay:300ms`). Ese `delay` es el clásico *debounce*: en vez de pedir al servidor en cada tecla, espera a que el usuario haga una pausa. Con un atributo conseguís un comportamiento que a mano sería bastante código.

Otros modificadores útiles: `once` (dispara una sola vez), `from:<selector>` (escuchar el evento en *otro* elemento, no en este), y `every <tiempo>` (para *polling*: pedir periódicamente).

> El principio: **`hx-trigger` libera el "cuándo".** La web clásica te daba un evento fijo por elemento; HTMX te deja elegirlo y refinarlo (debounce, una sola vez, escuchar a otro elemento, repetir en intervalos), todo de forma declarativa.

---

## 8. `hx-target`: dónde poner la respuesta

Por defecto, HTMX inserta la respuesta **en el mismo elemento que disparó la petición**. Pero casi siempre querés ponerla en otro lado: hacés clic en un botón de la lista y querés que el resultado aparezca en el panel de detalle. `hx-target` dice **en qué elemento** va la respuesta, usando un selector CSS:

```html
<button hx-get="/contactos/5" hx-target="#detalle">Ver</button>
<div id="detalle"></div>
```

El valor es un selector CSS (`#detalle`, `.panel`, etc.), pero HTMX agrega palabras clave relativas muy cómodas para no depender de IDs fijos:

- **`this`** — el propio elemento que dispara (es el default).
- **`closest <selector>`** — el ancestro más cercano que coincide (ej. `closest tr` apunta a la fila de tabla que contiene al botón).
- **`find <selector>`** — el primer descendiente que coincide.
- **`next <selector>` / `previous <selector>`** — el siguiente/anterior elemento que coincide en el DOM.

```html
<!-- pone la respuesta en la fila de tabla que contiene este botón -->
<button hx-get="/refrescar" hx-target="closest tr">Refrescar fila</button>
```

> El principio: **`hx-target` separa "quién dispara" de "qué se actualiza".** En la web clásica eran lo mismo (el enlace que clickeás recarga todo). HTMX desacopla las dos cosas: un elemento puede disparar un cambio en cualquier otro.

---

## 9. `hx-swap`: cómo insertarla

Sabiendo *dónde* va la respuesta (`hx-target`), falta decidir *cómo* se inserta respecto a ese target. Eso es `hx-swap`. Las opciones se basan en nombres estándar del DOM:

| Valor         | Qué hace                                                        |
| ------------- | --------------------------------------------------------------- |
| `innerHTML`   | Reemplaza el **contenido** del target (es el default).          |
| `outerHTML`   | Reemplaza el **target entero**, incluido él mismo.              |
| `beforeend`   | Agrega la respuesta **al final** del contenido del target.      |
| `afterbegin`  | Agrega la respuesta **al principio** del contenido del target.  |
| `beforebegin` | Inserta **antes** del target (como hermano previo).             |
| `afterend`    | Inserta **después** del target (como hermano siguiente).        |
| `delete`      | Elimina el target (ignora la respuesta).                        |
| `none`        | No inserta nada (útil si solo te interesan los efectos colaterales). |

La distinción más usada es `innerHTML` vs `outerHTML`: el primero deja el contenedor y cambia su interior; el segundo reemplaza el contenedor mismo. Y `beforeend` es típico para "agregar a una lista" sin tocar lo que ya estaba (pensá en cargar más resultados al final).

```html
<!-- agrega cada comentario nuevo al final de la lista, sin recargar los anteriores -->
<form hx-post="/comentarios" hx-target="#lista" hx-swap="beforeend">
    <textarea name="texto"></textarea>
    <button type="submit">Comentar</button>
</form>
<div id="lista"></div>
```

`hx-swap` también acepta modificadores de tiempo, como `swap:200ms` (esperar antes de reemplazar) o `transition:true` (usar la View Transitions API del navegador para animar el cambio), útiles para sincronizar con animaciones CSS.

> El principio: **`hx-target` y `hx-swap` son una pareja:** uno elige el elemento de referencia, el otro la operación del DOM respecto a ese elemento. Entre los dos cubren cualquier forma de meter HTML en la página.

## 10. Enviar datos: parámetros, formularios y `hx-include`

Pedir HTML está bien, pero muchas veces necesitás **mandar datos** al servidor. HTMX recoge datos de varias formas, sin que vos los serialices a mano:

- **Desde un formulario**: si el elemento con `hx-post` es un `<form>` (o está adentro), HTMX recoge automáticamente todos sus campos y los manda, igual que un formulario clásico.
- **En la URL**: podés poner parámetros directamente en el atributo, como `hx-get="/buscar?activo=true"`.
- **Con `hx-include`**: este es el atributo que da flexibilidad. Le indicás, con un selector CSS, **qué otros campos incluir** en la petición aunque no estén en el mismo formulario.

```html
<input id="busqueda" name="q" />

<!-- este botón no está en el form de búsqueda, pero incluye su campo -->
<button hx-get="/buscar" hx-include="#busqueda" hx-target="#resultados">
    Buscar
</button>
```

`hx-include` también acepta las palabras relativas que vimos (`closest form` incluye los campos del formulario más cercano, por ejemplo). Es la herramienta para juntar, en una petición, datos que viven en distintas partes de la pantalla.

> El principio: **HTMX reusa el mecanismo de envío de los formularios** (campos con `name`, recogidos automáticamente) y lo extiende con `hx-include` para componer la petición con campos de donde haga falta. No hay serialización manual: nombrás los campos y HTMX arma el envío.

---

## 11. El servidor manda HTML, no JSON

Acá está la diferencia filosófica con las SPA, y conviene detenerse. En una SPA, el servidor responde **datos** (JSON) y el navegador los convierte en HTML. En HTMX, el servidor responde **HTML ya armado**, y HTMX simplemente lo coloca donde indica `hx-target`.

```html
<!-- Lo que el servidor devuelve a una petición HTMX NO es esto: -->
{ "id": 5, "nombre": "Ana", "telefono": "381..." }

<!-- ...sino directamente esto: -->
<div class="detalle">
    <h2>Ana Gómez</h2>
    <p>381...</p>
</div>
```

Esto tiene una consecuencia profunda que resuelve el costo de las SPA (sección 3): **la lógica de presentación vive en un solo lugar, el servidor.** No hay plantillas duplicadas en el navegador, no hay estado de UI que mantener sincronizado. El servidor es la única fuente de verdad, tanto de los datos como de cómo se ven. Por eso con HTMX se escribe poco o nada de JavaScript propio: la pantalla la arma el servidor con sus plantillas habituales (las de tu framework de backend), y HTMX solo transporta esos fragmentos y los inserta.

Esto también explica por qué HTMX es **agnóstico del backend**: como solo necesita que el servidor devuelva fragmentos de HTML, funciona igual con cualquier lenguaje o framework que sepa generar HTML.

> El principio: **HTMX devuelve la presentación al servidor.** La SPA repartía el "cómo se ve" entre cliente y servidor; HTMX lo concentra del lado del servidor y deja al navegador un rol mínimo: pedir un fragmento y pegarlo. Una sola fuente de verdad, cero JavaScript que mantener.

## 12. Comunicación de vuelta: los headers `HX-*` y los eventos

Hasta acá la conversación fue en una dirección: el cliente pide, el servidor manda HTML. Pero a veces el servidor necesita decirle algo *extra* al cliente más allá del fragmento: "además de esto, refrescá tal otra zona" o "redirigí a tal página". Para eso, HTMX usa **headers HTTP** en la respuesta.

El más característico es **`HX-Trigger`**: con él, el servidor puede **disparar un evento** en el navegador. Esto resuelve un problema típico: una petición HTMX apunta a *un solo* `hx-target`, pero a veces una acción debería actualizar *dos* zonas. La técnica es:

**1. Un elemento escucha un evento personalizado** usando `hx-trigger` con el modificador `from:body`:

```html
<div id="lista"
     hx-get="/lista"
     hx-trigger="load, datosCambiaron from:body">
</div>
```

Esto dice: cargá la lista al inicio (`load`) y cada vez que en el `<body>` se dispare un evento llamado `datosCambiaron`.

**2. El servidor dispara ese evento** devolviendo un header en la respuesta de *otra* petición (por ejemplo, al guardar):

```
HX-Trigger: datosCambiaron
```

Cuando HTMX recibe una respuesta con ese header, dispara el evento `datosCambiaron` en el navegador. El `#lista`, que lo está escuchando, se refresca solo. Así, una acción de guardar que devuelve el detalle puede *además* avisar "los datos cambiaron", y la lista se actualiza sin que las dos plantillas se conozcan entre sí.

> El modificador `from:body` es importante acá: el evento del header se dispara a nivel del `<body>`, así que el elemento que quiere reaccionar tiene que escuchar desde ahí, no desde sí mismo.

Hay otros headers de respuesta útiles: `HX-Redirect` (forzar una redirección del navegador), `HX-Refresh` (recargar la página entera), `HX-Retarget` (cambiar a último momento dónde va la respuesta). Y en la *petición*, HTMX manda headers como `HX-Request: true`, que le permiten al servidor saber que la pidió HTMX (y devolver un fragmento) en vez de un navegador normal (y devolver la página completa).

> El principio: **los headers `HX-*` son el canal de vuelta.** El cuerpo de la respuesta lleva el HTML; los headers llevan instrucciones de control. Con `HX-Trigger`, el servidor puede orquestar actualizaciones de varias zonas disparando eventos, sin acoplar las plantillas entre sí.

## 13. El recorrido completo de una interacción

Cerremos atando todo. El usuario está en una pantalla con una lista y un panel de detalle, y hace clic en un contacto:

```
   El usuario hace clic en un contacto de la lista
        │  el elemento tiene: hx-get="/contactos/5"
        │                     hx-trigger="click"   (¿CUÁNDO? al clic)
        │                     hx-target="#detalle" (¿DÓNDE? en #detalle)
        │                     hx-swap="innerHTML"  (¿CÓMO? reemplaza su interior)
        ▼
   HTMX hace GET /contactos/5
        │  (manda también el header HX-Request: true)
        ▼
   El servidor ejecuta su lógica y renderiza un FRAGMENTO de HTML
   (no una página entera, no JSON): el detalle del contacto 5
        │  opcionalmente agrega headers, ej. HX-Trigger: datosCambiaron
        ▼
   HTMX recibe el HTML
        │  lo inserta DENTRO de #detalle (innerHTML)
        │  el resto de la página (la lista, el menú) NO se toca
        │  si vino HX-Trigger, dispara ese evento → otras zonas que lo
        │  escuchan se refrescan solas
        ▼
   La pantalla se actualizó parcialmente, sin recargar, sin parpadeo,
   sin que escribiéramos una línea de JavaScript
```

Visto así, HTMX no es un framework nuevo que haya que aprender de cero: es la web clásica con tres restricciones levantadas. **Cualquier** elemento puede pedir (`hx-get`/`hx-post`), ante **cualquier** evento (`hx-trigger`), poniendo la respuesta en **cualquier** lugar (`hx-target`) de **cualquier** forma (`hx-swap`). El servidor sigue mandando HTML como siempre, y por eso la presentación y el estado siguen viviendo en un solo lado. Toda la complejidad de las SPA —el JSON, las plantillas duplicadas, el estado sincronizado— desaparece, y a cambio recuperás la interactividad parcial que la web clásica no tenía.

## Resumen de los principios

| Concepto       | Regla en una frase                                                              |
| -------------- | ------------------------------------------------------------------------------- |
| Hipertexto     | La web clásica: solo enlaces/formularios piden, y la respuesta reemplaza todo.  |
| Idea de HTMX   | Levantar tres restricciones: cualquier elemento, cualquier evento, reemplazo parcial. |
| `hx-get`/`hx-post` | Qué pedir: la URL y el verbo HTTP, en cualquier elemento.                   |
| `hx-trigger`   | Cuándo pedir: qué evento dispara, con modificadores como `delay` (debounce).    |
| `hx-target`    | Dónde poner la respuesta: un selector CSS, desacoplado de quién dispara.        |
| `hx-swap`      | Cómo insertarla: reemplazar el interior, el elemento, agregar al final, etc.    |
| HTML, no JSON  | El servidor manda presentación ya armada; una sola fuente de verdad.            |
| Headers `HX-*` | El canal de vuelta: `HX-Trigger` dispara eventos para orquestar varias zonas.   |
