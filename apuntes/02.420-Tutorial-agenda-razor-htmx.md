## 1. Qué vamos a construir

Una agenda de contactos con las operaciones de un CRUD: listar, buscar, ver detalle, crear, editar y borrar. La pantalla es de tipo **maestro-detalle**: a la izquierda la lista de contactos, a la derecha un panel que cambia según lo que estés haciendo (ver, crear o editar).

Lo que define esta arquitectura es que **no escribimos JavaScript propio para la interacción**. HTMX es una pequeña librería que lee atributos HTML —`hx-get`, `hx-post`, `hx-trigger`, `hx-target`, `hx-swap`— para decidir qué evento dispara una petición, qué método HTTP usa y qué parte del HTML reemplaza. El servidor responde con fragmentos de HTML ya renderizados por Razor, y HTMX los inserta en su lugar. No hay API JSON, no hay framework de frontend, no hay estado duplicado en el navegador.

Las técnicas que vas a aprender y poder reutilizar en otros proyectos:

- **Renderizar HTML parcial en el servidor** y reemplazar regiones de la página con HTMX, sin escribir JavaScript.
- **Separar el dominio de la entrada** con un modelo y un DTO distintos.
- **Estructurar el acceso a datos en capas** (repositorio + servicio) con interfaces e inyección de dependencias.
- **Persistir con EF Core + SQLite** usando contextos efímeros por operación.
- **Mantener la protección antiforgery activa** mientras las peticiones viajan por HTMX.
- **Refrescar una región de la pantalla desde el servidor** emitiendo un evento por cabecera HTTP (`HX-Trigger`), la técnica central del tutorial.
- **Construir una lista seleccionable accesible** con radios navegables por teclado.

La estructura de carpetas queda así:

```txt
AgendaRazorHtmx/
├── Program.cs                         <- arranque y registro de servicios
├── Data/
│   ├── AgendaDbContext.cs             <- DbContext de EF Core
│   └── AgendaDbSeeder.cs              <- semilla de datos
├── Models/

├── Dtos/
│   └── ContactInput.cs                <- contrato de entrada de formularios
├── Repositories/

├── Services/

├── ViewModels/

├── Pages/

│   ├── _ContactList.cshtml            <- lista (panel maestro)
```

> **Principio: una carpeta por responsabilidad.** Cada tipo vive cerca de sus pares y tiene una sola razón para cambiar. El dominio en `Models/`, el contrato de entrada en `Dtos/`, el acceso a datos en `Repositories/`, la lógica de negocio en `Services/` y las clases de presentación en `ViewModels/`. Esta organización escala: cuando el proyecto crece, sabés exactamente dónde va cada pieza nueva.

## 2. Cómo fluye una interacción

Antes de escribir código conviene fijar el modelo mental. Cada vez que el usuario hace algo —un clic, escribir en el buscador— el recorrido es el mismo:

```txt
Navegador
   │  HTMX dispara un GET o POST
   ▼
Index.cshtml  (un handler: OnGetDetail, OnPostSave, …)
   │
   ▼
ContactService   (lógica de negocio)
   │
   ▼
ContactRepository  →  EF Core  →  SQLite (agenda.db)
```

Y la respuesta vuelve convertida en HTML:

```txt
SQLite → Repository → Service → Handler → Partial Razor → HTML
                                                            │
                                                            ▼
                          HTMX inserta ese HTML en una región de la página
```

> **La idea central de HTMX (hipermedia).** El servidor no devuelve datos crudos (JSON) que el navegador deba interpretar y convertir en pantalla. Devuelve **HTML ya armado**, y HTMX lo coloca en el lugar indicado. La consecuencia es profunda: la lógica de "qué se ve" vive en un solo lado, el servidor. No hay plantillas duplicadas ni estado sincronizado entre cliente y servidor. Esto es lo que hace que no necesites escribir JavaScript.

## 4. Razor Pages y los handlers inline

Razor Pages genera HTML en el servidor. Una página es un archivo `.cshtml` que mezcla HTML con C#, y la directiva `@page` al inicio la convierte en una URL accesible:

```razor
@page

<h1>Hola Razor</h1>
```

La lógica de una página puede vivir en un archivo aparte (`Index.cshtml.cs`, el *code-behind*) o, como acá, en **handlers inline** dentro del mismo `.cshtml`, en un bloque `@functions`:

```razor
@functions {
    public IActionResult OnGetDetail(int id) {
        // Se ejecuta con: GET /?handler=Detail&id=1
    }
}
```

La convención de Razor traduce la URL al nombre del método combinando el verbo HTTP con el parámetro `handler`:

| Petición HTTP                  | Método que se ejecuta      |
|--------------------------------|----------------------------|
| `GET /`                        | `OnGet()`                  |
| `GET /?handler=Detail&id=1`    | `OnGetDetail(int id)`      |
| `POST /?handler=Save`          | `OnPostSave(...)`          |

Los parámetros restantes de la URL (`id=1`) se asignan a los argumentos del método por nombre: eso es **model binding**, y funciona igual con tipos complejos (un `record` o una clase) armándolos a partir de los campos de un formulario.

> **Cuándo usar handlers inline.** Concentrar los handlers en el `.cshtml` deja toda la lógica de una pantalla en un archivo, fácil de leer de corrido. Es ideal cuando la página es el punto de orquestación y delega el trabajo pesado a servicios (como acá). Si la lógica de la página creciera mucho, el code-behind separado vuelve a ser preferible. La regla práctica: inline mientras la página solo orqueste; code-behind cuando empiece a contener reglas de negocio.

## 5. CSS propio en vez de un framework

Escribimos un `wwwroot/css/site.css` propio en lugar de usar Bootstrap, Tailwind o Pico. La pantalla maestro-detalle tiene requisitos concretos —dos paneles con scroll independiente, lista seleccionable con radios, altura completa de la ventana— que resultan más limpios con un puñado de reglas a medida que adaptando los defaults de un framework.

El CSS se apoya en tres ideas que conviene conocer porque se reutilizan en cualquier hoja de estilos:

```css
:root {
    --color-surface:  #ffffff;
    --color-selected: #dbeafe;
    --color-focus:    #2563eb;
    --radius: 8px;
    --space-4: 1rem;
    /* ... */
}
```

- **Variables CSS** (`--color-*`, `--space-*`): un solo lugar para definir el tema y el espaciado. Cambiás una variable y se actualiza toda la app.
- **Layout con Grid** para el maestro-detalle (`grid-template-columns: 320px 1fr`), con scroll contenido en cada panel.
- **Accesibilidad de fábrica:** estilos de `:focus-visible`, una utilidad `.visually-hidden`, y bloques `@media (prefers-reduced-motion)` y `@media (forced-colors)` que respetan las preferencias del sistema operativo.

El archivo completo está en [wwwroot/css/site.css](wwwroot/css/site.css). No hace falta memorizarlo; alcanza con reconocer qué hace cada clase cuando aparezca en los partials (`.panel`, `.contact-item`, `.detail-grid`, `.actions`, `.primary`, `.danger`, `.muted`, `.visually-hidden`).

> **Cuándo escribir CSS propio.** Un framework sin clases es ideal para formularios y páginas convencionales. Cuando necesitás control fino sobre el layout —un grid de dos columnas con scroll independiente, el resaltado preciso de una fila seleccionada— suele salir más limpio escribir las reglas vos. El costo es mantenerlo; la ventaja es que no hay clases ajenas que pelean entre sí ni dependencias externas.

## 6. HTMX: el modelo de hipermedia

HTMX amplía el HTML de forma declarativa. En vez de programar clics, fetch y reemplazo del DOM, agregás atributos al markup que describen el comportamiento:

```html
<button
    hx-get="/?handler=Detail&id=2"
    hx-target="#detail"
    hx-swap="innerHTML">
    Ver detalle
</button>
```

Al hacer clic, HTMX hace tres cosas: pide por `GET` a esa URL, recibe el HTML que devuelve el servidor, y lo inserta dentro de `#detail`. Toda la lógica está declarada en atributos HTML.

Estos son los atributos y técnicas que usamos:

| Atributo / técnica               | Para qué sirve                                                              |
|----------------------------------|----------------------------------------------------------------------------|
| `hx-get` / `hx-post`            | Hacer una petición GET o POST                                              |
| `hx-target`                      | A qué elemento aplicar la respuesta                                        |
| `hx-swap`                        | Cómo insertar el HTML (`innerHTML` reemplaza el contenido)                 |
| `hx-trigger`                     | Qué evento dispara la petición, con modificadores como `delay:250ms`       |
| `hx-include`                     | Enviar campos extra junto con la petición (un token, el buscador)          |
| `hx-vals`                        | Adjuntar pares clave-valor fijos a la petición                            |
| `hx-confirm`                     | Mostrar un cuadro de confirmación antes de enviar                         |
| `hx-trigger="load"`             | Disparar una petición apenas el elemento entra al DOM                      |
| `hx-trigger="evento from:body"`  | Escuchar en el `<body>` un evento personalizado                           |
| Cabecera `HX-Trigger-After-Swap` | El servidor ordena a HTMX disparar un evento tras insertar el HTML        |

Las dos últimas filas son la base de la técnica central (sección 14): el servidor puede emitir un evento desde una respuesta HTTP, y cualquier elemento de la página que escuche ese evento reacciona. Eso permite que una mutación actualice una región distinta de la que apuntó la petición.

> **Por qué HTML parcial y no JSON.** En una SPA tradicional, el servidor manda JSON y el navegador tiene código que lo convierte en HTML; la lógica de presentación queda duplicada en dos lados. Con HTMX, el servidor manda el HTML ya armado por Razor y el navegador solo lo coloca: una sola fuente de verdad para la presentación, cero JavaScript que mantener. Esta es la decisión arquitectónica que sostiene todo el proyecto.

## 7. El modelo y el DTO de entrada

Empecemos por los dos tipos que representan un contacto: uno hacia adentro (el dominio) y otro hacia afuera (la entrada).

El **modelo** vive en `Models/Contact.cs`. Representa cómo existe un contacto dentro de la aplicación y es también la entidad que EF Core mapea a una tabla:

```csharp
// Models/Contact.cs
namespace AgendaRazorHtmx.Models;

public class Contact {
    public int Id { get; set; }
    public string Nombre   { get; set; } = "";
    public string Apellido { get; set; } = "";
    public string Telefono { get; set; } = "";
    public string Email    { get; set; } = "";
    public string Notas    { get; set; } = "";
}
```

El **DTO de entrada** vive en `Dtos/ContactInput.cs`. Define qué datos acepta un formulario. Es un `record` porque es inmutable y se declara en pocas líneas:

```csharp
// Dtos/ContactInput.cs
namespace AgendaRazorHtmx.Dtos;

public record ContactInput(
    int Id,
    string Nombre,
    string Apellido,
    string Telefono,
    string Email,
    string Notas
);
```

> **Principio: separar el dominio de la entrada.** El modelo es lo que la app persiste y manipula; el DTO es la frontera de lo que aceptamos de afuera. Mantenerlos separados impide que un formulario toque propiedades que no debería y deja explícito qué campos son válidos como entrada. Es un patrón que se aplica en cualquier capa de borde: lo que entra no es lo mismo que lo que guardás.
>
> **Un solo DTO para crear y editar.** Usamos el `Id` como señal: si vale `0`, es un contacto nuevo; si trae un valor real, es una edición. El formulario es el mismo en ambos casos y manda un campo oculto `Id`. Esto colapsa dos pantallas en una sola: un único `_ContactForm` y un único handler `OnPostSave`. El servicio decide internamente si inserta o actualiza.

Los **view models** viven en `ViewModels/` y existen para pasarle a cada vista exactamente lo que necesita, ni más ni menos:

```csharp
// ViewModels/ContactListViewModel.cs
public record ContactListViewModel(
    List<Contact> Contacts,
    int? SelectedId,
    int? DetailIdToLoad = null,
    bool ClearDetail = false
);

// ViewModels/ContactListItemViewModel.cs
public record ContactListItemViewModel(Contact Contact, bool Selected);
```

`ContactListViewModel` combina los contactos con cuál está seleccionado y dos señales (`DetailIdToLoad`, `ClearDetail`) que veremos en acción cuando una búsqueda cambia la selección. `ContactListItemViewModel` le entrega a cada ítem solo su contacto y si está seleccionado.

> **Por qué un view model y no el modelo a secas.** Una vista que necesita "el contacto más si está seleccionado" no debería recibir el modelo crudo y calcular la selección por su cuenta. El view model empaqueta exactamente lo que la vista consume. Mantiene las vistas tontas (solo muestran) y la decisión de qué mostrar en el handler.

## 8. El acceso a datos: repositorio y servicio

El acceso a datos está en dos capas, cada una con su interfaz. El **repositorio** habla con EF Core; el **servicio** contiene la lógica de negocio. La página depende del servicio, el servicio depende del repositorio, y ninguno conoce los detalles del otro.

### El DbContext

El puente entre las clases C# y SQLite, en `Data/AgendaDbContext.cs`:

```csharp
// Data/AgendaDbContext.cs
using AgendaRazorHtmx.Models;
using Microsoft.EntityFrameworkCore;

namespace AgendaRazorHtmx.Data;

public class AgendaDbContext(DbContextOptions<AgendaDbContext> options) : DbContext(options) {
    public DbSet<Contact> Contacts => Set<Contact>();
}
```

### El repositorio

`Repositories/ContactRepository.cs` implementa `IContactRepository` con operaciones atómicas sobre la base. Cada método crea y cierra su propio `DbContext`:

```csharp
// Repositories/IContactRepository.cs
public interface IContactRepository {
    List<Contact> GetAll(string? search = null);
    Contact? GetById(int id);
    Contact Add(Contact contact);
    Contact Update(Contact contact);
    void Delete(int id);
}
```

```csharp
// Repositories/ContactRepository.cs
public class ContactRepository(IDbContextFactory<AgendaDbContext> dbContextFactory) : IContactRepository {
    public List<Contact> GetAll(string? search = null) {
        using AgendaDbContext dbContext = dbContextFactory.CreateDbContext();
        IQueryable<Contact> query = dbContext.Contacts
            .OrderBy(contact => contact.Apellido)
            .ThenBy(contact => contact.Nombre);

        if (!string.IsNullOrWhiteSpace(search)) {
            string term = search.Trim();

            query = query.Where(contact =>
                EF.Functions.Like(contact.Nombre, $"%{term}%")
                || EF.Functions.Like(contact.Apellido, $"%{term}%")
                || EF.Functions.Like(contact.Telefono, $"%{term}%")
                || EF.Functions.Like(contact.Email, $"%{term}%")
                || EF.Functions.Like(contact.Notas, $"%{term}%")
            );
        }

        return query.ToList();
    }

    public Contact? GetById(int id) {
        using AgendaDbContext dbContext = dbContextFactory.CreateDbContext();
        return dbContext.Contacts.SingleOrDefault(contact => contact.Id == id);
    }

    public Contact Add(Contact contact) {
        using AgendaDbContext dbContext = dbContextFactory.CreateDbContext();
        dbContext.Contacts.Add(contact);
        dbContext.SaveChanges();
        return contact;
    }

    public Contact Update(Contact contact) {
        using AgendaDbContext dbContext = dbContextFactory.CreateDbContext();
        dbContext.Contacts.Update(contact);
        dbContext.SaveChanges();
        return contact;
    }

    public void Delete(int id) {
        using AgendaDbContext dbContext = dbContextFactory.CreateDbContext();
        Contact? contact = dbContext.Contacts.SingleOrDefault(c => c.Id == id);
        if (contact is null) { return; }
        dbContext.Contacts.Remove(contact);
        dbContext.SaveChanges();
    }
}
```

`GetAll` unifica listar y buscar: sin `search` devuelve todo ordenado por apellido; con texto, filtra con `EF.Functions.Like`, que EF Core traduce al `LIKE` de SQL. Así hay un solo camino para poblar la lista, con o sin filtro.

### El servicio

`Services/ContactService.cs` implementa `IContactService` e interpreta el DTO: si el `Id` es `0` agrega; si no, busca el contacto, le aplica los cambios y lo actualiza.

```csharp
// Services/IContactService.cs
public interface IContactService {
    List<Contact> GetAll(string? search = null);
    Contact? GetById(int id);
    Contact Save(ContactInput input);
    void Delete(int id);
}
```

```csharp
// Services/ContactService.cs
public class ContactService(IContactRepository contactRepository) : IContactService {
    public List<Contact> GetAll(string? search = null) =>
        contactRepository.GetAll(search);

    public Contact? GetById(int id) =>
        contactRepository.GetById(id);

    public Contact Save(ContactInput input) {
        if (input.Id == 0) {
            Contact newContact = FromInput(input);
            return contactRepository.Add(newContact);
        }

        Contact contact = contactRepository.GetById(input.Id)
            ?? throw new InvalidOperationException($"No existe el contacto {input.Id}.");

        contact.Nombre   = input.Nombre;
        contact.Apellido = input.Apellido;
        contact.Telefono = input.Telefono;
        contact.Email    = input.Email;
        contact.Notas    = input.Notas;

        return contactRepository.Update(contact);
    }

    public void Delete(int id) =>
        contactRepository.Delete(id);

    private static Contact FromInput(ContactInput input) => new() {
        Nombre   = input.Nombre,
        Apellido = input.Apellido,
        Telefono = input.Telefono,
        Email    = input.Email,
        Notas    = input.Notas
    };
}
```

> **Principio: separar acceso a datos de lógica de negocio.** El repositorio solo sabe leer y escribir filas —no interpreta el DTO ni decide qué operación corresponde. El servicio sabe qué hacer con un `ContactInput` —no sabe nada de EF Core. Cada capa tiene una razón de cambio independiente: si cambiás el ORM, tocás el repositorio; si cambia una regla de negocio, tocás el servicio. Las interfaces (`IContactRepository`, `IContactService`) hacen explícito el contrato y permiten reemplazar cualquier capa por un doble en tests.
>
> **`IDbContextFactory` y un contexto por operación.** En vez de compartir un `DbContext`, inyectamos una *fábrica* y creamos un contexto efímero en cada método (`using` lo libera al terminar). Un `DbContext` acumula estado y no es seguro entre hilos; uno de vida corta por operación evita problemas de concurrencia y de datos rancios, y encaja con peticiones breves como las de HTMX.

## 9. La semilla de datos

Para no arrancar con la agenda vacía, sembramos contactos al iniciar, **solo si la tabla está vacía**. Para no repetir `new Contact { Nombre = …, Apellido = … }` dieciocho veces, guardamos los datos como un array de **tuplas con nombre** y los proyectamos a `Contact` con un `Select`:

El `Select` final convierte cada tupla en la entidad real. Es un patrón cómodo para cualquier dato de ejemplo o de configuración que tenga forma de tabla.

> **La guarda `if (Any()) return` hace la siembra idempotente.** Podés arrancar la app muchas veces sin duplicar datos. Para volver al estado inicial, borrá `agenda.db` y volvé a correr: se recrea y se siembra. Los datos persisten entre ejecuciones porque viven en el archivo SQLite.

## 10. Program.cs: dependencias y arranque

`Program.cs` registra los servicios, configura el pipeline y crea/siembra la base al iniciar.

```csharp
using AgendaRazorHtmx.Data;
using AgendaRazorHtmx.Repositories;
using AgendaRazorHtmx.Services;
using Microsoft.EntityFrameworkCore;

var builder = WebApplication.CreateBuilder(args);
string connectionString = "Data Source=./agenda.db";

builder.Services.AddRazorPages();

builder.Services.AddScoped<IContactService, ContactService>();

var app = builder.Build();

using (IServiceScope scope = app.Services.CreateScope()) {
    IDbContextFactory<AgendaDbContext> dbContextFactory =
        scope.ServiceProvider.GetRequiredService<IDbContextFactory<AgendaDbContext>>();

    using AgendaDbContext dbContext = dbContextFactory.CreateDbContext();
    dbContext.Database.EnsureCreated();
    AgendaDbSeeder.Seed(dbContext);
}

app.UseHttpsRedirection();
app.UseStaticFiles();
app.UseRouting();

app.MapRazorPages();

app.Run();
```

> **Registrar interfaces, no implementaciones.** `AddScoped<IContactService, ContactService>()` le dice al contenedor: cuando alguien pida `IContactService`, entregá un `ContactService`. La página pide `IContactService` y nunca menciona la clase concreta. Esto es **inversión de dependencias**: el código de alto nivel depende de abstracciones, lo que te deja cambiar la implementación o inyectar un doble en tests sin tocar a quien la consume.
>
> **Antiforgery queda activado.** `AddRazorPages()` exige un token en cada `POST` para protegerse de ataques CSRF. No lo desactivamos: HTMX puede mandar el token. Cada formulario incluye `@Html.AntiForgeryToken()` (un input oculto); en el guardado, `hx-post` está en el `<form>` y HTMX envía sus campos solo; en el borrado, el botón usa `hx-include="closest form"`. Tenés la protección sin renunciar a HTMX.
>
> **`AddDbContextFactory` y `EnsureCreated()`.** Registramos una *fábrica* porque el repositorio crea contextos efímeros (sección 8). Al arrancar, `EnsureCreated()` crea el archivo y la tabla si no existen, y `Seed()` carga los datos. `EnsureCreated` alcanza para un tutorial; en producción con un esquema que evoluciona se usan **migraciones** de EF Core.

```razor
            hx-include="#search-form, #current-selection"
            hx-vals='{"autoSelect":"false"}'
            hx-swap="innerHTML">
        </div>

        @* Carga inicial: autoselecciona el primer contacto visible. *@
        <div
            class="visually-hidden"
            hx-get="/?handler=List"
            hx-trigger="load"
            hx-target="#master"
            hx-include="#search-form, #current-selection"
            hx-vals='{"autoSelect":"true"}'
            hx-swap="innerHTML">
        </div>
    </section>

    <section class="panel">
        <div id="detail" aria-live="polite" aria-atomic="true">
            <div id="current-selection"></div>

        </div>

@functions {

    public IActionResult OnGetList(string? search, int? selectedId, bool autoSelect) {
        List<Contact> contacts = ContactService.GetAll(search);
        int? effectiveSelectedId = contacts.Any(c => c.Id == selectedId) ? selectedId : null;
        int? detailIdToLoad = null;
        bool clearDetail = false;

        if (autoSelect && effectiveSelectedId is null) {
            Contact? firstContact = contacts.FirstOrDefault();

            if (firstContact is null) {
                clearDetail = true;
            } else {
                effectiveSelectedId = firstContact.Id;
                detailIdToLoad = firstContact.Id;
            }
        }

        return Partial("_ContactList", new ContactListViewModel(contacts, effectiveSelectedId, detailIdToLoad, clearDetail));
    }

    public IActionResult OnGetNew() {
        TriggerMasterRefresh();
        return Partial("_ContactForm", new ContactInput(0, "", "", "", "", ""));
    }

    public IActionResult OnGetEmpty() {
        return Content("""<div id="current-selection"></div> <p>Seleccioná un contacto.</p>""", "text/html");
    }

    public IActionResult OnGetDetail(int id) {
        Contact? contact = ContactService.GetById(id);
        if (contact is null) { return NotFound(); }
        return Partial("_ContactDetail", contact);
    }

    public IActionResult OnGetEdit(int id) {
        Contact? contact = ContactService.GetById(id);
        if (contact is null) { return NotFound(); }
        return Partial("_ContactForm", ToInput(contact));
    }

    public IActionResult OnPostSave(ContactInput input) {
        Contact savedContact = ContactService.Save(input);
        TriggerMasterRefresh();
        return Partial("_ContactDetail", savedContact);
    }

    public IActionResult OnPostDelete(int id) {
        ContactService.Delete(id);
        TriggerMasterRefresh();
        return Content("""
            <div id="current-selection"></div>
            <p>Contacto eliminado.</p>
            """, "text/html");
    }

    private void TriggerMasterRefresh() {
        // HTMX dispara este evento en el body tras actualizar #detail; #master lo escucha.
        Response.Headers["HX-Trigger-After-Swap"] = "refresh-master";
    }

    private static ContactInput ToInput(Contact contact) => new(
        contact.Id,
        contact.Nombre,
        contact.Apellido,
        contact.Telefono,
        contact.Email,
        contact.Notas
    );
}
```

La página concentra el HTML de la pantalla, la inyección del servicio con `@inject` y los handlers. No contiene el modelo, ni el DTO, ni el acceso a datos: todo eso vive en sus carpetas (secciones 7 y 8).

Dos regiones tienen `id` para que HTMX sepa a cuál apuntar: **`#master`** (la lista) y **`#detail`** (ver/crear/editar). Dentro de `#detail` vive siempre `#current-selection`, un contenedor con el input oculto que guarda el `selectedId` del contacto visible.

> **El buscador, en detalle.** `hx-trigger="input changed delay:250ms, search"` dispara cuando el valor *cambia* (`input changed`), pero espera 250 ms desde la última tecla (`delay:250ms`, un *debounce* que evita una petición por pulsación), y también ante el evento `search` (cuando el usuario limpia el campo con la "x"). `hx-include="#search-form, #current-selection"` manda el texto y, si hay un contacto a la vista, su id. `hx-vals='{"autoSelect":"true"}'` le avisa al servidor que esta petición puede seleccionar el primer resultado.
>
> **`OnGetList` resuelve listar, filtrar y auto-seleccionar.** Trae los contactos (con o sin filtro). Si el seleccionado sigue visible, lo conserva. Si `autoSelect` es `true` y no hay selección válida, elige el primero (`DetailIdToLoad`) o, si la búsqueda no devolvió nada, pide limpiar el detalle (`ClearDetail`). Toda la decisión de selección vive en un solo lugar.
>
> **`TriggerMasterRefresh()` centraliza la sincronización de la lista.** Los handlers que *mutan* datos —`OnGetNew` (abre el formulario de alta), `OnPostSave`, `OnPostDelete`— lo llaman antes de devolver su respuesta. Los que solo *leen* —`OnGetDetail`, `OnGetEdit`— no, porque no cambian la lista. La sección 14 explica cómo ese método refresca `#master`.

## 13. Los partials

Los partials son fragmentos de HTML reutilizables. Cada handler devuelve uno (o un `Content` simple) y HTMX lo inserta.

### Lista (el panel maestro)

`Pages/_ContactList.cshtml`:

```razor
@using AgendaRazorHtmx.Models
@using AgendaRazorHtmx.ViewModels
@model ContactListViewModel

<form class="contact-list">
    <fieldset class="contact-group">
        <legend class="visually-hidden">Contactos</legend>

        @if (Model.Contacts.Count == 0) {
            <p class="muted">No hay contactos para esa búsqueda.</p>
        } else {
            foreach (Contact contact in Model.Contacts) {
                @await Html.PartialAsync(
                    "_ContactListItem",
                    new ContactListItemViewModel(contact, Model.SelectedId == contact.Id)
                )
            }
        }
    </fieldset>
</form>

@if (Model.DetailIdToLoad is not null) {
    <div
        class="visually-hidden"
        hx-get="/?handler=Detail&id=@Model.DetailIdToLoad"
        hx-trigger="load"
        hx-target="#detail"
        hx-include="this"
        hx-swap="innerHTML">
    </div>
} else if (Model.ClearDetail) {
    <div
        class="visually-hidden"
        hx-get="/?handler=Empty"
        hx-trigger="load"
        hx-target="#detail"
        hx-include="this"
        hx-swap="innerHTML">
    </div>
}
```

> **La lista delega cada ítem a un partial.** En vez de escribir el markup del radio inline, llama a `Html.PartialAsync` con un `ContactListItemViewModel`. Cada partial tiene una sola responsabilidad y se compone con los demás. Es el equivalente Razor de descomponer una UI en componentes pequeños.
>
> **Los disparadores al final cargan el detalle cuando la búsqueda mueve la selección.** Marcar un radio como `checked` desde el servidor no dispara su evento `change` en el navegador. Por eso, cuando `OnGetList` eligió un contacto nuevo (`DetailIdToLoad`) o vació la búsqueda (`ClearDetail`), la lista incluye un elemento oculto con `hx-trigger="load"` que pide el detalle correspondiente apenas entra al DOM. Solo aparece en respuesta a búsquedas que cambian la selección.

### Ítem de la lista

`Pages/_ContactListItem.cshtml`:

```razor
@using AgendaRazorHtmx.ViewModels

@{
    string inputId = $"contact-{Model.Contact.Id}";
}

<div

        class="contact-radio"
        type="radio"
        name="selectedId"
        value="@Model.Contact.Id"
        checked="@Model.Selected"
        hx-get="/?handler=Detail&id=@Model.Contact.Id"
        hx-trigger="change"
        hx-target="#detail"
        hx-include="this"
```

> **Por qué radios y no enlaces.** Cada contacto es un `<input type="radio">` (oculto con CSS) con su `<label>` estilado como tarjeta. Un grupo de radios da, sin código extra, **navegación por teclado**: con las flechas te movés entre opciones, y al cambiar la selección (`hx-trigger="change"`) HTMX pide el detalle. El `<fieldset>` con `<legend>` oculto los agrupa de forma accesible.
>
> El radio puede sentirse forzado para una lista, pero es una elección deliberada: el **radio (o checkbox) es el único control nativo de HTML que da selección única más navegación con flechas sin JavaScript**. Las alternativas no encajan con la premisa de cero JS de este proyecto: una lista de `<a>`/`<button>` con `aria-current` tiene markup más natural pero pierde el movimiento con flechas (quedás con Tab + Enter); el patrón ARIA `role="listbox"` es semánticamente ideal, pero su navegación por teclado **exige JavaScript** (roving tabindex o `aria-activedescendant`). Cuando necesites una lista seleccionable, accesible y navegable por teclado sin escribir JS, los radios te dan la semántica y el teclado gratis.
>
> **`hx-include="this"`** limita lo que se envía al valor del propio radio, sin arrastrar otros campos de la página.

## 14. La técnica clave: eventos del servidor para refrescar otra zona

Acá está el corazón del tutorial. Cuando **guardás** o **borrás**, cambian *dos* regiones a la vez:

- Al guardar: `#detail` muestra el contacto, y `#master` puede haber cambiado (nombre nuevo, orden distinto, un contacto más).
- Al borrar: `#detail` muestra "Contacto eliminado", y el contacto debe desaparecer de `#master`.

Pero una petición HTMX apunta a **un solo** elemento (`hx-target`). ¿Cómo actualizamos el otro?

### El mecanismo: el servidor emite un evento

HTMX define un canal para esto. Si una respuesta HTTP trae la cabecera `HX-Trigger-After-Swap: nombre-del-evento`, HTMX —después de insertar el HTML en el DOM— **dispara un evento del DOM con ese nombre** en el `<body>`. Cualquier elemento que escuche ese evento reacciona.

Son dos piezas que encajan:

**1. El servidor emite el evento** desde cualquier handler que mute datos:

```csharp
private void TriggerMasterRefresh() {
    Response.Headers["HX-Trigger-After-Swap"] = "refresh-master";
}
```

**2. El elemento que debe refrescarse escucha el evento** con `hx-trigger="evento from:body"`:

```html
<div
    id="master"
    hx-get="/?handler=List"
    hx-trigger="refresh-master from:body"
    hx-include="#search-form, #current-selection"
    hx-vals='{"autoSelect":"false"}'
    hx-swap="innerHTML">
</div>
```

El `from:body` indica que el listener escuche el evento en el `<body>` (donde HTMX lo emite), no solo en el propio elemento. Cuando llega `refresh-master`, `#master` hace su `GET /?handler=List` y se reconstruye, incluyendo el filtro actual y el `selectedId` de `#current-selection`. Usa `autoSelect: false` porque la selección ya la determina el id que viaja en `#current-selection`.

### Por qué esta técnica es valiosa

> **Desacopla quién responde de qué se actualiza.** El partial que devuelve un handler (`_ContactDetail`, el mensaje de borrado) no necesita saber que existe `#master`. Solo el handler, que tiene la vista de conjunto, decide "esto fue una mutación, refrescá la lista". Si mañana renombrás `#master` o cambiás su URL, tocás un solo lugar.
>
> **Centraliza la regla de sincronización.** "Cuándo refrescar la lista" vive en un único método, `TriggerMasterRefresh()`. Agregar una nueva mutación es una línea: llamarlo. La lógica no se dispersa por las plantillas.
>
> **Es un patrón general de HTMX.** Cualquier app donde una acción deba actualizar regiones distintas a la apuntada puede usar `HX-Trigger` (o `HX-Trigger-After-Swap`) para emitir eventos y dejar que cada región se suscriba. Un evento, varios oyentes: el mismo modelo de los eventos del navegador, manejado desde el servidor.

### La carga inicial

`#master` escucha un evento, así que no se llena solo al cargar la página. De eso se encarga un `<div>` aparte con `hx-trigger="load"`, que se ejecuta una vez al entrar al DOM y manda `autoSelect: true` para seleccionar el primer contacto y mostrar su detalle:

```html
@* Carga inicial: autoselecciona el primer contacto visible. *@
<div
    class="visually-hidden"
    hx-get="/?handler=List"
    hx-trigger="load"
    hx-target="#master"
    hx-include="#search-form, #current-selection"
    hx-vals='{"autoSelect":"true"}'
    hx-swap="innerHTML">
</div>
```

> **Requisito de la técnica.** El elemento que escucha (`#master`) debe existir en el DOM cuando el evento se dispara. Acá siempre existe porque es parte del HTML estático de `Index.cshtml`. Si un elemento que escucha eventos pudiera ser reemplazado por otra operación, perdería su listener: tenelo presente al elegir qué región escucha.

## 15. Los dos flujos de mutación

Con la técnica clara, los dos flujos que cambian datos se leen de un vistazo. En ambos, la respuesta actualiza `#detail` y el evento `refresh-master` actualiza `#master`.

### Guardar (alta o edición)

```txt
Enviás el formulario  →  hx-post /?handler=Save
   │
   ▼
OnPostSave(input)
   ├─ ContactService.Save(input)   guarda en SQLite (Id==0 inserta; si no, actualiza)
   ├─ TriggerMasterRefresh()       cabecera HX-Trigger-After-Swap: refresh-master
   └─ devuelve _ContactDetail
   │
   ▼
HTMX inserta el detalle en #detail   (#current-selection queda con el id guardado)
   y dispara refresh-master
   │
   ▼
#master pide la lista y queda con el contacto recién guardado seleccionado
```

### Borrar

```txt
Clic en "Eliminar"  →  hx-confirm  →  hx-post /?handler=Delete&id=…
   │
   ▼
OnPostDelete(id)
   ├─ ContactService.Delete(id)
   ├─ TriggerMasterRefresh()       cabecera HX-Trigger-After-Swap: refresh-master
   └─ devuelve el mensaje + #current-selection vacío
   │
   ▼
HTMX inserta el mensaje en #detail  y dispara refresh-master
   │
   ▼
#master pide la lista; el contacto ya no está
```

> **Abrir el formulario de alta** sigue el mismo patrón: `OnGetNew()` llama a `TriggerMasterRefresh()` antes de devolver `_ContactForm`, así la lista se refresca y queda sin selección mientras creás un contacto.

## 16. Probar la aplicación

Levantá la app:

```bash
dotnet run
```

Abrí la URL que muestre la consola (por ejemplo `https://localhost:7xxx`) y recorré todo:

1. **Listar.** Al cargar, ves los contactos ordenados por apellido, con el primero seleccionado y su detalle visible.
2. **Buscar.** Escribí "ana" o "gómez": la lista se filtra tras una pausa breve, sin recargar la página.
3. **Navegar con teclado.** Hacé foco en la lista y movete con las flechas: el detalle cambia con la selección.
4. **Ver detalle.** Clic en un contacto: el panel derecho muestra sus datos.
5. **Crear.** "Agregar", completá el formulario, "Guardar": el detalle muestra el contacto nuevo **y** aparece en la lista.
6. **Editar.** Desde un detalle, "Editar", cambiá algo, "Guardar": vuelve al detalle actualizado y la lista se refresca con ese contacto seleccionado.
7. **Borrar.** Desde un detalle, "Eliminar", confirmá: el contacto desaparece de la lista.

> **Si la página aparece sin estilos:** verificá que `Index.cshtml` tenga `Layout = "_Layout";` y que exista `wwwroot/css/site.css`. Sin el layout no se cargan ni los estilos ni el script de HTMX.
>
> **Si un `POST` (guardar, borrar) devuelve 400:** casi seguro falta el token antiforgery. Confirmá que el formulario tenga `@Html.AntiForgeryToken()`. En el guardado, `hx-post` está en el `<form>`; en el borrado, el botón usa `hx-include="closest form"`.
>
> **Si la lista no se refresca al guardar o borrar:** verificá que `TriggerMasterRefresh()` se llame en ese handler y que `#master` tenga `hx-trigger="refresh-master from:body"`. El nombre del evento en la cabecera y en el `hx-trigger` debe coincidir exactamente.
>
> **Los datos persisten.** Al frenar y volver a correr `dotnet run`, los contactos que creaste siguen ahí (viven en `agenda.db`). Para volver al estado inicial, borrá `agenda.db` y arrancá de nuevo.
