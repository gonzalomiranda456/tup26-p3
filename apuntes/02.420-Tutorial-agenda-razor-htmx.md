## 1. La idea: por qué Razor primero y HTMX después

Casi todo lo que sabés del navegador moderno gira alrededor de JavaScript: frameworks que arman la pantalla en el cliente, llamadas a una API que devuelve JSON, estado duplicado entre servidor y navegador. Razor Pages propone algo más viejo y más simple: **el HTML se arma en el servidor** y el navegador solo lo muestra. Cada vez que hacés clic en un enlace o enviás un formulario, el servidor produce una página nueva y completa.

Ese modelo —el de la web "de toda la vida"— es el que vamos a entender primero, **sin HTMX y sin JavaScript propio**. Es importante porque es la base sobre la que todo lo demás se apoya. Si entendés cómo un `GET` o un `POST` terminan ejecutando un método de C# y devolviendo HTML, ya entendés Razor Pages. HTMX no cambia eso: lo *aprovecha*.

El plan es entonces:

1. Construimos la agenda completa con **Razor puro**: formularios HTML normales, navegación por URL, la página se recarga entera en cada acción. Funciona perfecto y no tiene una sola línea de JavaScript.
2. Detectamos qué es incómodo de esa versión (recargar toda la pantalla para cambiar un panel).
3. Incorporamos **HTMX** para resolver exactamente eso, reusando los *mismos handlers* de Razor. La app pasa a sentirse como una SPA, pero el servidor sigue mandando HTML y nosotros seguimos sin escribir JavaScript.

> **Por qué este orden importa.** Si arrancás directamente con HTMX, los atributos `hx-get` y `hx-target` parecen magia. Cuando primero ves la versión sin HTMX, queda clarísimo que HTMX **no inventa nada nuevo del lado del servidor**: usa los mismos handlers, las mismas URLs, los mismos partials. Lo único que cambia es *cuánto* de la página se reemplaza.

## 2. Qué es Razor Pages, mecánicamente

Una **página Razor** es un archivo `.cshtml` que mezcla HTML con C#. La directiva `@page` al inicio la convierte en una URL accesible:

```razor
@page

<h1>Hola Razor</h1>
```

Eso solo ya responde en `/` con ese `<h1>`. Pero una página estática sirve de poco; lo interesante es que la página tiene **handlers**: métodos de C# que se ejecutan según el verbo HTTP y un parámetro `handler` en la URL.

Normalmente esos métodos viven en un archivo aparte, el *code-behind* (`Index.cshtml.cs`). En este tutorial usamos **handlers inline**: los escribimos dentro del mismo `.cshtml`, en un bloque `@functions`. Así toda la lógica de la pantalla se lee de corrido, en un solo lugar.

```razor
@functions {
    public IActionResult OnGetDetail(int id) {
        // Se ejecuta con: GET /?handler=Detail&id=1
    }
}
```

La convención de nombres de Razor traduce sola la petición HTTP al método:

| Petición HTTP                 | Método que se ejecuta   |
| ----------------------------- | ----------------------- |
| `GET /`                       | `OnGet()`               |
| `GET /?handler=Detail&id=1`   | `OnGetDetail(int id)`   |
| `POST /?handler=Save`         | `OnPostSave(...)`       |

La fórmula es siempre la misma: `On` + verbo (`Get`/`Post`) + nombre del `handler`. Y los demás parámetros de la URL o del formulario (`id=1`, los campos de un `<form>`) se asignan **por nombre** a los argumentos del método. Eso último es el **model binding**: Razor mira el nombre del argumento (`int id`) y busca un valor que se llame igual en la query string, la ruta o el cuerpo del formulario.

> **Esto es todo el "motor" de Razor Pages.** Una URL llega, Razor arma el nombre del método, completa sus argumentos con model binding, ejecuta el método y usa lo que devuelve para producir HTML. Tenelo presente: cuando lleguemos a HTMX, lo único que va a hacer HTMX es **disparar estas mismas peticiones** desde atributos HTML, en vez de desde enlaces y formularios.

> **Por qué handlers inline acá.** Mantener los handlers en el `.cshtml` deja toda la pantalla (su HTML y su lógica) en un archivo, fácil de leer entero. En un proyecto grande conviene el code-behind separado; es una decisión pedagógica, no una recomendación general.

Abrí una terminal y ejecutá:

```bash
dotnet new webapp -n AgendaRazor
cd AgendaRazor
```

> **`dotnet new webapp`** crea un proyecto **Razor Pages** (distinto de `dotnet new web`, que es el mínimo para Minimal API). Trae la carpeta `Pages/` y la configuración base que necesitamos.

La estructura final del proyecto va a quedar así. La primera mitad del tutorial usa solo los archivos marcados con `[Razor puro]`; HTMX y el CSS agregan los marcados con `[HTMX]`.

## 4. El modelo, el DTO y el acceso a datos

Todo este código va en `Program.cs`, **debajo** de las instrucciones de arranque (que vemos en la sección 5), dentro de `namespace AgendaRazor`.

### El modelo

El **modelo** representa cómo vive un contacto **dentro** de la app. Es también la entidad que EF Core mapea a una tabla de SQLite:

```csharp
public class Contact {
    public int    Id       { get; set; }
    public string Nombre   { get; set; } = "";
    public string Apellido { get; set; } = "";
    public string Telefono { get; set; } = "";
    public string Email    { get; set; } = "";
    public string Notas    { get; set; } = "";
}
```

### El DTO de entrada

El **DTO** define qué datos llegan desde un formulario. Usamos un `record` porque es inmutable y se declara en una línea:

```csharp
public record ContactInput(
    int    Id,
    string Nombre,
    string Apellido,
    string Telefono,
    string Email,
    string Notas
);
```

> **Por qué un solo DTO para crear y editar.** Usamos el `Id` como señal: si `Id == 0`, es un contacto nuevo; si trae un `Id` real, es una edición. El formulario es el mismo en ambos casos y manda un campo oculto `Id` (`0` al crear). Un único `_ContactForm`, un único handler `OnPostSave`. El `ContactStore.Save` decide adentro si inserta o actualiza.

> **Por qué no usar `Contact` directamente en el formulario.** El modelo es lo que vive adentro y lo que EF Core persiste; el DTO es la frontera de lo que aceptamos de afuera. Separarlos deja explícito qué campos son válidos como entrada.

Y un pequeño **view model** para la lista, que combina los contactos con cuál está seleccionado (para resaltarlo):

```csharp
public record ContactListViewModel(List<Contact> Contacts, int? SelectedId);
```

### El acceso a datos: EF Core, SQLite y `ContactStore`

El `DbContext` es el puente entre las clases de C# y la base:

```csharp
public class AgendaDbContext(DbContextOptions<AgendaDbContext> options) : DbContext(options) {
    public DbSet<Contact> Contacts => Set<Contact>();
}
```

Y el `ContactStore` concentra todas las operaciones que la pantalla necesita:

```csharp
public class ContactStore(IDbContextFactory<AgendaDbContext> dbContextFactory) {
    public List<Contact> GetAll(string? search = null) {
        using AgendaDbContext dbContext = dbContextFactory.CreateDbContext();
        IQueryable<Contact> query = dbContext.Contacts
            .OrderBy(contact => contact.Apellido)
            .ThenBy(contact => contact.Nombre);

        if (!string.IsNullOrWhiteSpace(search)) {
            string term = search.Trim();
            query = query.Where(contact =>
                EF.Functions.Like(contact.Nombre, $"%{term}%")
                || EF.Functions.Like(contact.Apellido, $"%{term}%")
                || EF.Functions.Like(contact.Telefono, $"%{term}%")
                || EF.Functions.Like(contact.Email, $"%{term}%")
                || EF.Functions.Like(contact.Notas, $"%{term}%")
            );
        }

        return query.ToList();
    }

    public Contact? GetById(int id) {
        using AgendaDbContext dbContext = dbContextFactory.CreateDbContext();
        return dbContext.Contacts.SingleOrDefault(contact => contact.Id == id);
    }

    public Contact Save(ContactInput input) {
        using AgendaDbContext dbContext = dbContextFactory.CreateDbContext();
        Contact contact;

        if (input.Id == 0) {
            contact = new Contact();
            dbContext.Contacts.Add(contact);
        }
        else {
            contact = dbContext.Contacts.Single(c => c.Id == input.Id);
        }

        contact.Nombre   = input.Nombre;
        contact.Apellido = input.Apellido;
        contact.Telefono = input.Telefono;
        contact.Email    = input.Email;
        contact.Notas    = input.Notas;

        dbContext.SaveChanges();
        return contact;
    }

    public void Delete(int id) {
        using AgendaDbContext dbContext = dbContextFactory.CreateDbContext();
        Contact? contact = dbContext.Contacts.SingleOrDefault(c => c.Id == id);

        if (contact is null) {
            return;
        }

        dbContext.Contacts.Remove(contact);
        dbContext.SaveChanges();
    }
}
```

> **`GetAll` unifica listar y buscar.** Sin `search` devuelve todo ordenado por apellido; con texto filtra con `EF.Functions.Like` (la traducción del `LIKE` de SQL a EF Core). Un solo camino para poblar la lista, con o sin filtro.

> **`Save` decide insertar o actualizar según el `Id`.** Con `Id == 0` crea y agrega; con un `Id` real busca y copia los campos. Una operación para las dos intenciones, en línea con el DTO único.

> **Por qué `IDbContextFactory` y un `using` por operación.** En vez de un `DbContext` compartido, inyectamos una *fábrica* y creamos un contexto efímero en cada método (`using` lo libera al terminar). Esto evita problemas de concurrencia y de estado acumulado, y encaja con peticiones cortas como las que vamos a tener.

## 5. Program.cs: armar y arrancar la app

Acá registramos los servicios, configuramos el pipeline y creamos/sembramos la base. Estas instrucciones van **arriba de todo** en `Program.cs`; las clases de la sección 4 van debajo, dentro de `namespace AgendaRazor`.

```csharp
using AgendaRazor;
using Microsoft.EntityFrameworkCore;

var builder = WebApplication.CreateBuilder(args);
string connectionString = "Data Source=./agenda.db";

builder.Services.AddRazorPages();
builder.Services.AddDbContextFactory<AgendaDbContext>(options => options.UseSqlite(connectionString));
builder.Services.AddScoped<ContactStore>();

var app = builder.Build();

using (IServiceScope scope = app.Services.CreateScope()) {
    IDbContextFactory<AgendaDbContext> dbContextFactory =
        scope.ServiceProvider.GetRequiredService<IDbContextFactory<AgendaDbContext>>();

    using AgendaDbContext dbContext = dbContextFactory.CreateDbContext();
    dbContext.Database.EnsureCreated();
    AgendaDbSeeder.Seed(dbContext);
}

app.UseHttpsRedirection();
app.UseStaticFiles();
app.UseRouting();

app.MapRazorPages();

app.Run();
```

> **`AddRazorPages()` tal cual.** No desactivamos nada. Esto deja **activada** la protección antiforgery, que exige un token en cada `POST` para evitar ataques CSRF. En la versión Razor pura, los `<form>` con `method="post"` ya incluyen ese token solos. Cuando lleguemos a HTMX vamos a ver cómo hacer que el token viaje igual (sección 14).

> **`AddDbContextFactory` en vez de `AddDbContext`.** Registramos una *fábrica* porque el `ContactStore` crea un contexto efímero por operación (sección 4). Es el patrón recomendado cuando querés contextos de vida corta y bien acotada en vez de uno por request.

> **`AddScoped<ContactStore>`:** una instancia por petición HTTP. El store no guarda estado propio, así que un objeto fresco por request es lo correcto.

> **`EnsureCreated()` + `Seed(...)` al arrancar:** crea el archivo SQLite y su tabla si no existen, y siembra los datos. `EnsureCreated` alcanza para un tutorial; un proyecto real con esquema cambiante usa **migraciones** de EF Core.

### El layout

El layout es la plantilla que envuelve a todas las páginas. Por ahora —en la versión Razor pura— **no carga HTMX**: solo el `<head>`, los estilos y la estructura base. Más adelante le agregamos una línea para HTMX.

Creá `Pages/Shared/_Layout.cshtml`:

```razor
<!DOCTYPE html>
<html lang="es">
<head>
    <meta charset="utf-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1.0" />
    <title>@ViewData["Title"] - Agenda Razor</title>
    <link rel="icon" href="data:," />
    <link rel="stylesheet" href="/css/site.css" />
</head>
<body>
    <main class="container">
        @RenderBody()
    </main>
</body>
</html>
```

> **`@RenderBody()`** es donde se inserta el contenido de cada página. El `<link rel="icon" href="data:," />` evita el pedido automático de `/favicon.ico` (y su 404 en consola) sin tener que crear un ícono.

El `wwwroot/css/site.css` lo escribimos nosotros (grid maestro-detalle, lista, formularios). No es el foco del tutorial; alcanza con saber que define clases como `.agenda`, `.panel`, `.contact-item`, `.detail-grid`, `.actions`, `.primary`, `.danger`, `.muted` y `.visually-hidden`. Lo importante es entender **qué hace cada clase cuando aparece**, no memorizar el CSS.

## 6. Primera versión: Razor puro, sin nada de JavaScript

Acá está el corazón del tutorial. Vamos a hacer la agenda **entera** con Razor Pages clásico: enlaces que navegan, formularios que hacen `POST`, y la página que se **recarga completa** en cada acción. Cero JavaScript, cero HTMX.

La clave de esta versión es que `OnGet` recibe en la URL **qué vista mostrar** y **qué contacto está seleccionado**, y arma la página completa con esa información. Toda la pantalla se reconstruye en cada paso.

Creá `Pages/Index.cshtml`:

```razor
@page
@using AgendaRazor
@using Microsoft.AspNetCore.Mvc
@inject ContactStore ContactStore

@{
    ViewData["Title"] = "Agenda";
    Layout = "_Layout";

    // Estado de la pantalla, leído de la query string:
    string? buscar     = Request.Query["buscar"];
    string  vista      = Request.Query["vista"].ToString();   // "", "detalle", "nuevo", "editar"
    int     seleccion  = int.TryParse(Request.Query["id"], out int parsed) ? parsed : 0;

    List<Contact> contactos = ContactStore.GetAll(buscar);
    Contact? seleccionado   = seleccion > 0 ? ContactStore.GetById(seleccion) : null;
}

<h1>Agenda</h1>

<div class="agenda">

    <section class="panel master-panel">
        <a class="primary button" href="/?vista=nuevo&buscar=@buscar">Nuevo contacto</a>

        <hr />

        <form method="get" action="/">
            <label class="visually-hidden" for="buscar">Buscar contactos</label>
            <input id="buscar" name="buscar" type="search" value="@buscar"
                   placeholder="Buscar por nombre, teléfono o email" autocomplete="off" />
            <button type="submit">Buscar</button>
        </form>

        <div class="list-scroll">
            <partial name="_ContactList" model="new ContactListViewModel(contactos, seleccion)" />
        </div>
    </section>

    <section class="panel">
        <div id="detail">
            @if (vista == "nuevo") {
                <partial name="_ContactForm" model="new ContactInput(0, "", "", "", "", "")" />
            }
            else if (vista == "editar" && seleccionado is not null) {
                <partial name="_ContactForm" model="ToInput(seleccionado)" />
            }
            else if (vista == "detalle" && seleccionado is not null) {
                <partial name="_ContactDetail" model="seleccionado" />
            }
            else {
                <p class="muted">Seleccioná un contacto.</p>
            }
        </div>
    </section>

</div>

@functions {
    public IActionResult OnPostSave(ContactInput input) {
        Contact savedContact = ContactStore.Save(input);
        // Después de guardar, redirigimos al detalle del contacto guardado.
        return RedirectToPage("Index", new { vista = "detalle", id = savedContact.Id });
    }

    public IActionResult OnPostDelete(int id, string? buscar) {
        ContactStore.Delete(id);
        // Después de borrar, volvemos a la pantalla sin selección.
        return RedirectToPage("Index", new { buscar });
    }

    private static ContactInput ToInput(Contact contact) {
        return new ContactInput(
            contact.Id, contact.Nombre, contact.Apellido,
            contact.Telefono, contact.Email, contact.Notas
        );
    }
}
```

Fijate en lo simple que es el modelo mental:

- **`OnGet` es implícito.** No lo escribimos: Razor renderiza la página al pedir `GET /`. El bloque `@{ ... }` de arriba lee la query string (`buscar`, `vista`, `id`), consulta el `ContactStore` y arma el estado. La pantalla entera es función de esos tres parámetros de la URL.
- **Navegar es cambiar la URL.** "Nuevo contacto" es un `<a href="/?vista=nuevo">`. Ver un contacto va a ser un enlace `/?vista=detalle&id=5`. Editar, `/?vista=editar&id=5`. Cada enlace recarga la página completa con un estado distinto.
- **Guardar y borrar son `POST` que redirigen.** `OnPostSave` y `OnPostDelete` hacen su trabajo y devuelven un `RedirectToPage`: el navegador vuelve a pedir la página por `GET` con la URL nueva. Esto es el patrón **POST-Redirect-GET**, que evita que al refrescar el navegador se reenvíe el formulario.

Ahora los partials. **Los mismos archivos van a servir para la versión HTMX**, con un par de ajustes que haremos después; por ahora, los enlaces y formularios son HTML normal.

Creá `Pages/_ContactList.cshtml`:

```razor

@model ContactListViewModel

<ul class="contact-list">
    @if (Model.Contacts.Count == 0) {
        <li><p class="muted">No hay contactos para esa búsqueda.</p></li>
    } else {
        foreach (Contact contact in Model.Contacts) {
            string clase = Model.SelectedId == contact.Id ? "contact-item selected" : "contact-item";
            <li>
                <a class="@clase" href="/?vista=detalle&id=@contact.Id">
                    <strong>@contact.Apellido, @contact.Nombre</strong>
                    <br />
                    <span class="muted">@contact.Telefono</span>
                </a>
            </li>

    }

```

```razor
        <div class="detail-notes">

            <dd>@(string.IsNullOrWhiteSpace(contact.Notas) ? "Sin notas" : contact.Notas)</dd>
        </div>
    </dl>

    <div class="actions">
        <a class="primary button" href="/?vista=editar&id=@contact.Id">Editar</a>

            @Html.AntiForgeryToken()

            <button class="danger" type="submit"
                    onclick="return confirm('¿Eliminar este contacto?');">
                Eliminar
            </button>
        </form>
    </div>
</div>
```

> Acá aparece el único pedacito de JavaScript de la versión "pura": el `onclick="return confirm(...)"` para confirmar el borrado. Es nativo del navegador, no requiere ninguna librería. Cuando pasemos a HTMX, lo reemplazamos por el atributo `hx-confirm`, que hace lo mismo de forma declarativa.

## 7. Leer el flujo de la versión Razor pura

Sigamos un par de acciones de punta a punta. Es importante porque es *exactamente* el mismo flujo que va a quedar con HTMX, solo que recargando la página entera.

**Ver un contacto:**

```
Clic en un contacto de la lista
   │  <a href="/?vista=detalle&id=5">
   ▼
GET /?vista=detalle&id=5
   │  el bloque @{ } lee vista="detalle", id=5
   │  ContactStore.GetById(5) trae el contacto
   ▼
Razor arma la PÁGINA COMPLETA: lista a la izquierda (con el 5 resaltado),
detalle a la derecha
   ▼
El navegador reemplaza toda la pantalla
```

**Guardar un contacto nuevo:**

```
Clic en "Nuevo contacto"  →  GET /?vista=nuevo  →  página con el formulario (Id=0)
   ▼
El usuario completa y envía
   │  POST /?handler=Save  (el form manda los campos + el token antiforgery)
   ▼
OnPostSave(input)  →  ContactStore.Save(input)  (Id==0 ⇒ inserta)
   │  return RedirectToPage("Index", new { vista="detalle", id=nuevoId })
   ▼
El navegador recibe un redirect y hace GET /?vista=detalle&id=nuevoId
   ▼
Razor arma la página completa: lista actualizada + detalle del nuevo contacto
```

> **Notá el patrón POST-Redirect-GET.** El `POST` no devuelve HTML directamente: redirige a una URL que el navegador pide por `GET`. Así, si el usuario refresca, repite un `GET` inofensivo en vez de reenviar el formulario (lo que duplicaría el contacto). Es la forma correcta de cerrar un `POST` en la web clásica.

## 8. El problema de la versión pura (y qué nos gustaría)

La versión Razor pura está bien y es perfectamente usable. Pero tiene una incomodidad concreta:

**Cada acción recarga la pantalla entera.** Si estás mirando el contacto 5 y hacés clic en el 6, el navegador descarta toda la página y pide una nueva, aunque lo único que cambió es el panel de la derecha. La lista de la izquierda —que es idéntica— se vuelve a descargar y redibujar. Hay un parpadeo, se pierde la posición del scroll, y se transfiere más HTML del necesario.

Lo que nos gustaría es: **cuando hago clic en un contacto, que se actualice solo el panel del detalle**, dejando la lista intacta. Y cuando guardo o borro, que se actualicen *solo* las partes que cambiaron.

En la web clásica, lograr eso significaba escribir JavaScript: interceptar los clics, hacer un `fetch`, recibir datos, construir el HTML en el navegador y meterlo en el DOM. Mucho código, y la lógica de presentación duplicada entre servidor y cliente.

**HTMX nos da exactamente eso sin escribir JavaScript.** Y lo mejor: reusando los handlers de Razor que *ya tenemos*. Solo necesitamos que esos handlers devuelvan **fragmentos** (partials) en vez de páginas completas, y agregar unos atributos al HTML.

---

## 9. Entra HTMX: la idea en una página

HTMX es una librería JavaScript chica que amplía el HTML de forma **declarativa**. En vez de programar clics y `fetch` a mano, agregás atributos al markup que le dicen al navegador: *qué evento* dispara una petición, *qué método HTTP* usar, *qué URL* pedir y *qué parte de la página* reemplazar con la respuesta.

```html
<button
    hx-get="/?handler=Detail&id=5"
    hx-target="#detail"
    hx-swap="innerHTML">
    Ver detalle
</button>
```

Al hacer clic, HTMX hace tres cosas: pide por `GET` a esa URL, recibe el HTML que devuelve el servidor, y lo mete dentro de `#detail`. Nada más. El resto de la página no se toca.

Compará con el enlace de la versión pura:

```html
<!-- Razor puro: recarga toda la página -->
<a href="/?vista=detalle&id=5">Ver detalle</a>

<!-- HTMX: reemplaza solo #detail -->
<button hx-get="/?handler=Detail&id=5" hx-target="#detail" hx-swap="innerHTML">Ver detalle</button>
```

Misma URL del servidor, misma idea. La diferencia es que el `<a>` reemplaza **todo** y el `hx-get` reemplaza **solo lo que indica `hx-target`**.

Estos son los atributos que vamos a usar:

| Atributo / técnica     | Para qué sirve                                                              |
| ---------------------- | --------------------------------------------------------------------------- |
| `hx-get` / `hx-post`   | Hacer una petición GET o POST                                               |
| `hx-target`            | A qué elemento del DOM aplicar la respuesta                                 |
| `hx-swap`              | Cómo insertar el HTML (`innerHTML` reemplaza el contenido del target)       |
| `hx-trigger`           | Qué evento dispara la petición, con modificadores como `delay:250ms`        |
| `hx-include`           | Enviar campos extra junto con la petición (el token antiforgery, el buscador)|
| `hx-confirm`           | Mostrar un cuadro de confirmación antes de enviar                           |
| Header `HX-Trigger`    | El servidor dispara un evento del lado del cliente devolviendo un header    |

> **Por qué HTML parcial y no JSON.** En una SPA tradicional, el servidor manda JSON y el navegador tiene código JavaScript que lo convierte en HTML; eso duplica la lógica de presentación en dos lados. Con HTMX, el servidor manda el HTML ya armado por Razor y el navegador solo lo coloca. **Una sola fuente de verdad para la presentación, cero JavaScript que mantener.** Y como ese HTML lo arman *los mismos partials* que ya escribimos, no duplicamos nada.

Primero, cargá HTMX como archivo local (así la app funciona sin internet):

```bash
curl -fsSL https://unpkg.com/htmx.org@2.0.4/dist/htmx.min.js -o wwwroot/js/htmx.min.js
```

Y agregá una línea al `<head>` del layout (`Pages/Shared/_Layout.cshtml`):

```razor
    <script src="/js/htmx.min.js"></script>
    <link rel="stylesheet" href="/css/site.css" />
```

> **Por qué HTMX local y no CDN.** Lo sirve `app.UseStaticFiles()` (sección 5). Frente a una CDN, la app funciona sin internet, no depende de un servicio externo, y carga siempre la misma versión. El archivo queda versionado en el repo.

## 10. Reescribir la pantalla con HTMX

Ahora viene la conversión. La estrategia es:

1. Los handlers de `Index.cshtml` pasan a devolver **partials** (`Partial(...)`), no páginas completas ni redirects.
2. El HTML de la pantalla usa atributos `hx-*` en vez de enlaces y formularios que navegan.
3. Los dos paneles tienen un `id` para que HTMX sepa a cuál apuntar: **`#master`** (la lista) y **`#detail`** (ver/crear/editar).

Reemplazá `Pages/Index.cshtml` por esta versión:

```razor
@page
@using AgendaRazor
@using Microsoft.AspNetCore.Mvc
@inject ContactStore ContactStore

@{
    ViewData["Title"] = "Agenda";
    Layout = "_Layout";
}

<h1 id="agenda-title">Agenda</h1>

<div class="agenda">
    <section class="panel master-panel" aria-labelledby="agenda-title">
        <button class="primary"
                hx-get="/?handler=New"
                hx-target="#detail"
                hx-swap="innerHTML">
            Nuevo contacto
        </button>

        <hr />

        <form id="search-form">
            <label class="visually-hidden" for="search">Buscar contactos</label>
            <input id="search" name="search" type="search"
                   placeholder="Buscar por nombre, teléfono o email" autocomplete="off"
                   hx-get="/?handler=List"
                   hx-trigger="input changed delay:250ms, search"
                   hx-target="#master"
                   hx-swap="innerHTML" />
        </form>

        <div id="master" class="list-scroll"
             hx-get="/?handler=List"
             hx-trigger="load, contactsChanged from:body"
             hx-include="#search-form"
             hx-swap="innerHTML">
        </div>
    </section>

    <section class="panel">
        <div id="detail" aria-live="polite" aria-atomic="true">
            <p class="muted">Seleccioná un contacto.</p>
        </div>
    </section>
</div>

@functions {
    public IActionResult OnGetList(string? search, int? selectedId) {
        return Partial("_ContactList", new ContactListViewModel(ContactStore.GetAll(search), selectedId));
    }

    public IActionResult OnGetNew() {
        Response.Headers["HX-Trigger"] = "contactsChanged";
        return Partial("_ContactForm", new ContactInput(0, "", "", "", "", ""));
    }

    public IActionResult OnGetEmpty() {
        return Content("<p class=\"muted\">Seleccioná un contacto.</p>", "text/html");
    }

    public IActionResult OnGetDetail(int id) {
        Contact? contact = ContactStore.GetById(id);
        if (contact is null) {
            return NotFound();
        }
        return Partial("_ContactDetail", contact);
    }

    public IActionResult OnGetEdit(int id) {
        Contact? contact = ContactStore.GetById(id);
        if (contact is null) {
            return NotFound();
        }
        Response.Headers["HX-Trigger"] = "contactsChanged";
        return Partial("_ContactForm", ToInput(contact));
    }

    public IActionResult OnPostSave(ContactInput input) {
        Contact savedContact = ContactStore.Save(input);
        Response.Headers["HX-Trigger"] = "contactsChanged";
        return Partial("_ContactDetail", savedContact);
    }

    public IActionResult OnPostDelete(int id) {
        ContactStore.Delete(id);
        Response.Headers["HX-Trigger"] = "contactsChanged";
        return Content("<p class=\"muted\">Contacto eliminado.</p>", "text/html");
    }

    private static ContactInput ToInput(Contact contact) {
        return new ContactInput(
            contact.Id, contact.Nombre, contact.Apellido,
            contact.Telefono, contact.Email, contact.Notas
        );
    }
}
```

Compará con la versión pura para ver qué cambió y qué no:

- **Los handlers son casi los mismos**, pero ahora devuelven `Partial(...)` (un fragmento) en lugar de armar la página entera o redirigir. `OnPostSave` ya no hace `RedirectToPage`: devuelve directamente el partial del detalle, porque HTMX va a insertar ese fragmento en `#detail` sin recargar.
- **Aparecieron handlers nuevos** (`OnGetList`, `OnGetNew`, `OnGetDetail`, `OnGetEdit`, `OnGetEmpty`) que en la versión pura eran ramas del `@if` dentro del `@{ }`. Ahora cada vista es un handler que devuelve su partial, porque HTMX las pide por separado.
- **El `#master` arranca vacío** y se llena con `hx-trigger="load, ..."`: se pide la lista al cargar la página. En la versión pura, la lista venía ya renderizada en el HTML inicial.

> **El buscador ahora filtra en vivo.** `hx-trigger="input changed delay:250ms, search"` significa: dispará cuando el valor *cambie*, pero esperá 250 ms desde la última tecla (`delay:250ms`, el clásico *debounce*), y también ante el evento `search` (cuando se limpia el campo con la "x"). El `hx-include="#search-form"` del `#master` hace que el texto del buscador viaje también en los refrescos automáticos de la lista. En la versión pura, buscar era un `<form method="get">` con un botón; acá filtra solo mientras escribís.

> **Por qué no hay `OnGet` que arme la lista.** La lista la trae `OnGetList` (con `search` nulo devuelve todo), igual que cualquier búsqueda. Un solo handler para el primer render y para los filtros.

> **`OnGetEmpty` y `OnPostDelete` devuelven `Content(...)` y no un partial.** Son fragmentos triviales (un `<p>`); se devuelven como HTML directo en vez de crear un archivo de partial para una sola línea.

## 11. Los partials con HTMX

Los partials son los **mismos archivos** de la versión pura, con los enlaces y formularios convertidos a atributos `hx-*`. Notá cuánto se parecen: solo cambia cómo se dispara la petición.

### Lista (panel maestro)

En la versión pura, cada contacto era un `<a href>`. Acá lo convertimos en un `<input type="radio">` con su `<label>`, lo que da, gratis, **navegación por teclado** con las flechas.

Reemplazá `Pages/_ContactList.cshtml`:

```razor
@using AgendaRazor
@model ContactListViewModel

<form class="contact-list">
    <fieldset class="contact-group">
        <legend class="visually-hidden">Contactos</legend>

        @if (Model.Contacts.Count == 0) {
            <p class="muted">No hay contactos para esa búsqueda.</p>
        } else {
            foreach (Contact contact in Model.Contacts) {
                string inputId = $"contact-{contact.Id}";

                <div class="contact-option">
                    <input id="@inputId" class="contact-radio" type="radio"
                           name="selectedContact" value="@contact.Id"
                           checked="@(Model.SelectedId == contact.Id)"
                           hx-get="/?handler=Detail&id=@contact.Id"
                           hx-trigger="change"
                           hx-target="#detail"
                           hx-swap="innerHTML" />

                    <label class="contact-item" for="@inputId">
                        <strong>@contact.Apellido, @contact.Nombre</strong>
                        <br />
                        <span class="muted">@contact.Telefono</span>
                    </label>
                </div>
            }
        }
    </fieldset>
</form>
```

> **Por qué radios y no enlaces.** Cada contacto es un `<input type="radio">` (visualmente oculto) con su `<label>` estilado como tarjeta. Esto da navegación por teclado: con las flechas te movés entre contactos como en cualquier grupo de radios, y al cambiar la selección (`hx-trigger="change"`) HTMX pide el detalle. El `checked="..."` resalta el activo. El `<fieldset>`/`<legend>` (oculto) agrupa los radios de forma accesible.

### Detalle (solo lectura)

Reemplazá `Pages/_ContactDetail.cshtml`:

```razor

<div class="detail-view">
```

## 12. Refrescar dos paneles con un solo evento

Acá está la única técnica de HTMX que va más allá de lo básico, y vale la pena entenderla bien.

Hay un problema clásico en pantallas maestro-detalle: cuando **guardás** o **borrás**, cambian *dos* zonas a la vez. Al guardar, el `#detail` muestra el contacto **y** la lista de `#master` puede haber cambiado (nombre nuevo, otro orden, un contacto más). Al borrar, el `#detail` muestra "Contacto eliminado" **y** el contacto debe desaparecer de `#master`.

Pero una petición HTMX normal apunta a **un solo** elemento (`hx-target`). ¿Cómo actualizamos el otro panel?

La solución son los **eventos personalizados disparados desde el servidor**. Funciona en dos partes.

**1. El `#master` escucha un evento.** En `Index.cshtml`, el `<div id="master">` tiene:

```razor
hx-trigger="load, contactsChanged from:body"
```

Esto significa: pedí la lista al cargar la página (`load`) **y** cada vez que en el `<body>` se dispare un evento llamado `contactsChanged`.

**2. El servidor dispara ese evento** devolviendo un header en la respuesta:

```csharp
public IActionResult OnPostSave(ContactInput input) {
    Contact savedContact = ContactStore.Save(input);
    Response.Headers["HX-Trigger"] = "contactsChanged";   // ← avisa "la lista cambió"
    return Partial("_ContactDetail", savedContact);
}
```

Cuando HTMX recibe una respuesta con el header `HX-Trigger: contactsChanged`, **dispara ese evento en el navegador**. El `#master`, que lo está escuchando, vuelve a pedir la lista solo. Así, con una respuesta que apunta a `#detail`, también logramos actualizar `#master` —sin escribir JavaScript y sin que las plantillas se conozcan entre sí.

Los cuatro handlers que tocan la pantalla principal usan el mismo recurso:

| Handler        | Por qué dispara `contactsChanged`                                  |
| -------------- | ------------------------------------------------------------------ |
| `OnPostSave`   | El contacto pudo cambiar de nombre, de orden, o ser nuevo          |
| `OnPostDelete` | El contacto desaparece de la lista                                 |
| `OnGetNew`     | Limpia el resaltado de selección al abrir el formulario de alta    |
| `OnGetEdit`    | Limpia el resaltado de selección al abrir el formulario de edición |

> **Por qué esto es más limpio que incrustar la lista en cada respuesta.** Una alternativa es el *out-of-band swap* (`hx-swap-oob`): devolver, en la misma respuesta, un segundo fragmento marcado para reemplazar `#master`. Funciona, pero acopla las plantillas (cada respuesta de guardar/borrar tendría que incluir la lista). Con `HX-Trigger`, cada handler devuelve solo lo suyo y "avisa" del cambio; el `#master` se encarga de refrescarse. El costo es una petición HTTP extra (la acción + el refresco de la lista), imperceptible en esta app.

> **Comparación con la versión pura.** En Razor puro este problema *ni existía*: como toda la página se recargaba, la lista y el detalle se rearmaban juntos siempre. HTMX nos da la actualización parcial, y este mecanismo de evento es el precio (pequeño y elegante) de actualizar dos zonas con una sola acción.

## 13. Los flujos completos, ahora con HTMX

Compará estos flujos con los de la sección 7 (la versión pura). El recorrido del servidor es casi el mismo; lo que cambia es que ahora se reemplazan fragmentos, no la página entera.

### Flujo de alta

```
Clic en "Nuevo contacto"
   │  hx-get /?handler=New
   ▼
OnGetNew()  →  setea HX-Trigger: contactsChanged  +  devuelve _ContactForm (Id=0)
   │  HTMX mete el formulario en #detail  Y  dispara contactsChanged
   │  (#master se refresca y se limpia el resaltado de selección)
   ▼
El usuario completa el formulario y envía
   │  hx-post /?handler=Save  (model binding arma ContactInput; hx-include manda el token)
   ▼
OnPostSave(input)  →  ContactStore.Save(input)  (Id==0 ⇒ inserta)
   │  setea HX-Trigger: contactsChanged  +  devuelve _ContactDetail del nuevo contacto
   ▼
HTMX mete el detalle en #detail  Y  dispara contactsChanged
   ▼
#master (que escucha contactsChanged) vuelve a pedir la lista → aparece el nuevo contacto
```

### Flujo de borrado

```
El usuario ve un detalle y hace clic en "Eliminar"
   │  hx-confirm pide confirmación
   │  hx-post /?handler=Delete&id=2  (hx-include manda el token)
   ▼
OnPostDelete(2)  →  ContactStore.Delete(2)
   │  setea HX-Trigger: contactsChanged  +  devuelve Content("<p>Contacto eliminado.</p>")
   ▼
HTMX pone ese mensaje en #detail  Y  dispara contactsChanged
   ▼
#master vuelve a pedir la lista → el contacto ya no está
```

## 14. Antiforgery: la seguridad que no apagamos

Una tentación común al meter HTMX es desactivar la protección antiforgery para que los `POST` "anden". **No lo hacemos.** ASP.NET exige un token en cada `POST` para protegerse de ataques **CSRF** (que un sitio malicioso envíe peticiones en tu nombre). Eso es lo correcto y lo dejamos activado con `AddRazorPages()` (sección 5).

La clave es que **HTMX puede mandar ese token igual que un formulario normal**:

- Cada formulario que hace `POST` incluye `@Html.AntiForgeryToken()`, que genera un `<input type="hidden">` con el token.
- Los botones que hacen `hx-post` usan `hx-include="closest form"`, que le dice a HTMX: "incluí los campos del formulario más cercano en la petición". Entre esos campos viaja el token oculto.

Así el `POST` pasa la validación antiforgery **sin desactivar nada**. En la versión Razor pura ni nos preocupamos por esto, porque un `<form method="post">` manda el token automáticamente; con HTMX solo hace falta el `hx-include` para reproducir ese comportamiento.

> **Si los `POST` devuelven 400**, casi seguro falta el token: confirmá que el formulario tenga `@Html.AntiForgeryToken()` y que el botón `hx-post` use `hx-include="closest form"`.

## 16. Resumen y próximos pasos

Recorrimos el camino completo: primero una agenda con **Razor Pages puro**, donde cada acción recarga la página entera y no hay JavaScript; después la **misma agenda con HTMX**, donde solo se reemplazan fragmentos y la app se siente como una SPA, reusando exactamente los mismos handlers y partials.

### La idea central

Razor Pages arma HTML en el servidor: una URL llega, Razor elige un handler por su nombre, completa sus argumentos con model binding, ejecuta el método y produce HTML. **HTMX no cambia nada de eso.** Solo dispara esas mismas peticiones desde atributos HTML y reemplaza una parte de la página con la respuesta. Por eso pudimos pasar de una versión a la otra tocando casi solo la *forma de disparar* las peticiones, no la lógica del servidor.

### Qué resuelve cada pieza

| Pieza                  | Responsabilidad                                           |
| ---------------------- | --------------------------------------------------------- |
| `_Layout.cshtml`       | Plantilla base: HTMX (archivo local) y `site.css`         |
| `site.css`             | Estilos propios: grid maestro-detalle, lista, formularios |
| `Index.cshtml`         | Pantalla principal y los handlers inline                  |
| `_ContactList`         | Lista del panel maestro (radios navegables)               |
| `_ContactDetail`       | Detalle de solo lectura con acciones                      |
| `_ContactForm`         | Alta y edición en un único formulario                     |
| `Contact`              | Modelo interno / entidad de EF Core                       |
| `ContactInput`         | Contrato de entrada de los formularios (DTO único)        |
| `ContactListViewModel` | Contactos + cuál está seleccionado                        |
| `AgendaDbContext`      | Puente EF Core ↔ SQLite                                   |
| `ContactStore`         | Acceso a datos (listar/buscar, ver, guardar, borrar)      |
| `AgendaDbSeeder`       | Siembra los datos de ejemplo solo si la base está vacía   |

### Los handlers de un vistazo (versión HTMX)

```csharp
public IActionResult OnGetList(string? search, int? selectedId); // listar y filtrar
public IActionResult OnGetNew();                                 // formulario de alta
public IActionResult OnGetEmpty();                               // panel vacío (cancelar alta)
public IActionResult OnGetDetail(int id);                        // ver
public IActionResult OnGetEdit(int id);                          // formulario de edición
public IActionResult OnPostSave(ContactInput input);             // crear o actualizar
public IActionResult OnPostDelete(int id);                       // borrar
```

### Conceptos de Razor que practicamos

- La directiva `@page` y el mapeo URL → handler (`On` + verbo + nombre).
- **Model binding**: cómo los parámetros de la URL y los campos del formulario se asignan a los argumentos del método por nombre.
- **Partials** (`<partial>` y `Partial(...)`) para componer y reutilizar fragmentos de pantalla.
- El patrón **POST-Redirect-GET** en la versión pura, y cómo HTMX lo reemplaza devolviendo un partial directamente.
- **Antiforgery** activado y cómo el token viaja en cada `POST`.

### Características de HTMX que usamos

- `hx-get` / `hx-post`, `hx-target`, `hx-swap`.
- `hx-trigger` con modificadores (`delay:250ms`) y eventos (`load`, `change`, `search`).
- **Eventos personalizados** (`contactsChanged from:body`) disparados desde el servidor con el header **`HX-Trigger`**.
- `hx-include` para enviar el token antiforgery (`closest form`) y el buscador (`#search-form`).
- `hx-confirm` para confirmar el borrado.
