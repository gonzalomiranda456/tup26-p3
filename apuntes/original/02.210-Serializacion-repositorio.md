# Persistencia con serialización: del archivo plano al repositorio genérico

## 1. ¿Por qué persistir?

Hasta acá nuestros programas viven y mueren con el proceso. Cargamos personas en una `List<Persona>`, las recorremos, las modificamos, mostramos algo en pantalla, y cuando termina `Main` todo desaparece. Para ejercicios chicos eso alcanza, pero ninguna aplicación real funciona así. Una agenda que olvida los contactos al cerrar no es una agenda. Un sistema de stock que pierde los productos al apagar la computadora no es un sistema de stock.

Necesitamos **persistencia**: guardar el estado en algún lado donde sobreviva al programa, y volver a cargarlo en la próxima ejecución. La pregunta es dónde.

Las opciones, en orden creciente de complejidad, son:

- Un archivo plano de texto, escrito a mano con el formato que se nos ocurra.
- Un archivo con un **formato estándar** de serialización (JSON, XML, CSV).
- Una **base de datos**, que es una pieza de software dedicada a guardar y consultar datos.

Cada una tiene su lugar. En este tutorial vamos a recorrer las dos primeras desde cero, y al final vamos a ver qué problemas dejan abiertos —los que justifican pasar a una base de datos, que es el tema del próximo tutorial.

---

## 2. El primer impulso: un archivo de texto

Imaginemos que tenemos una lista de personas y queremos guardarlas en un archivo. La idea más directa es: una persona por línea, con los campos separados por comas.

```csharp
var personas = new List<Persona> {
    new() { Id = 1, Nombre = "Ana",  Apellido = "Araujo", Edad = 25 },
    new() { Id = 2, Nombre = "Beto", Apellido = "Benites", Edad = 40 },
};

using var writer = new StreamWriter("personas.txt");
foreach (var p in personas) {
    writer.WriteLine($"{p.Id},{p.Nombre},{p.Apellido},{p.Edad}");
}
```

Para leer:

```csharp
var personas = new List<Persona>();
foreach (var linea in File.ReadAllLines("personas.txt")) {
    var partes = linea.Split(',');
    personas.Add(new Persona {
        Id       = int.Parse(partes[0]),
        Nombre   = partes[1],
        Apellido = partes[2],
        Edad     = int.Parse(partes[3])
    });
}
```

Funciona. Hasta que pasa algo de esto:

- Alguien tiene apellido **"García, López"**. La coma del apellido rompe el `Split`.
- Alguien tiene un nombre con un **salto de línea** o una comilla.
- Decidimos agregar un nuevo campo (teléfono). Todos los archivos viejos quedan inválidos: tienen 4 columnas y el código nuevo espera 5.
- Queremos que un campo pueda ser **null**. ¿Cómo lo representamos? ¿Como string vacío? ¿Como la palabra "null"? ¿Y si el dato real es la palabra "null"?
- Tenemos un campo `List<string> Telefonos`. ¿Cómo metemos una lista adentro de una columna?

Cada uno de esos problemas es resoluble. Podemos inventar reglas de escape, separadores más exóticos, marcadores de versión, formatos jerárquicos. Pero a esta altura ya estamos inventando un formato. Y resulta que **otra gente ya inventó formatos para esto**, los inventó bien, y vienen soportados de fábrica en .NET.

---

## 3. Serialización: el concepto

Llamamos **serialización** al proceso de convertir un objeto en memoria a una secuencia de bytes (típicamente texto) que podemos guardar o transmitir. Y **deserialización** al proceso inverso: a partir de los bytes, reconstruir el objeto.

Los formatos estándar de serialización resuelven los problemas que mencionamos arriba:

- Tienen reglas claras de **escape**: los caracteres especiales dentro de un valor no rompen la estructura.
- Soportan **estructuras anidadas**: listas dentro de objetos, objetos dentro de objetos, sin límite.
- Distinguen explícitamente **null**, strings vacíos, números, booleanos.
- Existen librerías maduras para leerlos y escribirlos en prácticamente todos los lenguajes.

Los dos formatos que vamos a ver son **JSON** y **XML**. Hay otros (YAML, TOML, Protobuf, MessagePack), pero estos dos son los que .NET trae listos en la BCL.

JSON, por ejemplo, expresa la lista de antes así:

```json
[
  { "id": 1, "nombre": "Ana",  "apellido": "Araujo",  "edad": 25 },
  { "id": 2, "nombre": "Beto", "apellido": "Benites", "edad": 40 }
]
```

Es legible, jerárquico, escapa correctamente las comillas y los saltos de línea, y la librería se encarga de todos los detalles. No tenemos que tocar el formato a mano.

## 4. JSON con `System.Text.Json`

.NET incluye `System.Text.Json` como serializador JSON oficial. La API que vamos a usar es mínima: dos métodos estáticos en `JsonSerializer`.

**Serializar** (objeto → texto):

```csharp
using System.Text.Json;

var personas = new List<Persona> {
    new() { Id = 1, Nombre = "Ana",  Apellido = "Araujo", Edad = 25 },
    new() { Id = 2, Nombre = "Beto", Apellido = "Benites", Edad = 40 },
};

string json = JsonSerializer.Serialize(personas);
Console.WriteLine(json);
```

Imprime algo como:

```json
[{"Id":1,"Nombre":"Ana","Apellido":"Araujo","Edad":25},{"Id":2,"Nombre":"Beto","Apellido":"Benites","Edad":40}]
```

Todo en una línea, sin formato. Para leerlo más cómodo, le pasamos opciones:

```csharp
var options = new JsonSerializerOptions {
    WriteIndented = true,
    PropertyNamingPolicy = JsonNamingPolicy.CamelCase
};
string json = JsonSerializer.Serialize(personas, options);
```

Ahora sale indentado y con los nombres en `camelCase` (`nombre` en vez de `Nombre`). Las opciones más útiles son:

| Opción                          | Para qué sirve                                                           |
|---------------------------------|-----|
| `WriteIndented`                 | Indenta la salida, mucho más legible cuando se abre el archivo a mano.   |
| `PropertyNamingPolicy`          | Convierte los nombres de propiedades (`CamelCase`, `SnakeCase`, etc.).   |
| `PropertyNameCaseInsensitive`   | Al **leer**, ignora mayúsculas y minúsculas en los nombres del JSON.     |
| `Encoder`                       | Controla qué caracteres se escapan en los strings.                       |

**Deserializar** (texto → objeto):

```csharp
string json = File.ReadAllText("personas.json");
List<Persona>? personas = JsonSerializer.Deserialize<List<Persona>>(json, options);
```

Notar dos cosas:

- Hay que decirle el tipo esperado, en el parámetro genérico (`<List<Persona>>`).
- El retorno es **nullable**: si el JSON está vacío o es la cadena `"null"`, devuelve `null`. Por eso típicamente combinamos con `??`:

```csharp
var personas = JsonSerializer.Deserialize<List<Persona>>(json, options) ?? [];
```

Para que la deserialización funcione, la clase debe tener:

- Un **constructor sin argumentos** (las clases con `{ get; set; }` lo tienen por defecto).
- Propiedades **públicas con setter**, o un constructor cuyos parámetros coincidan con las propiedades.

Vamos a usar la primera variante porque nos deja modificar la entidad después de crearla, y porque vamos a necesitar **escribir el `Id`** desde el repositorio.

## 5. Leer y escribir directo a un archivo

Hasta acá serializamos a una `string` y después escribimos. Funciona, pero hay una versión más directa que opera contra un `Stream`:

```csharp
using var stream = File.Create("personas.json");
JsonSerializer.Serialize(stream, personas, options);
```

Y para leer:

```csharp
using var stream = File.OpenRead("personas.json");
var personas = JsonSerializer.Deserialize<List<Persona>>(stream, options) ?? [];
```

La diferencia con la versión `string` es:

- **Memoria.** No mantenemos en RAM toda la representación textual además del objeto. Para archivos chicos no importa; para archivos grandes sí.
- **Encoding.** El `Stream` deja la responsabilidad del encoding (UTF-8 por defecto) en la librería, sin que tengamos que pensar en `Encoding.UTF8`.

El `using` antes de `var stream` cierra el archivo al salir del bloque, sin importar si hubo excepción. Si no cerramos el archivo después de escribir, los datos pueden quedar en el buffer del sistema operativo y el archivo aparece corrupto o vacío. **Siempre `using`.**

Una sutileza importante: si el archivo no existe al momento de leer, `File.OpenRead` lanza una excepción. Tenemos que chequear primero:

```csharp
if (!File.Exists("personas.json")) return [];

using var stream = File.OpenRead("personas.json");
return JsonSerializer.Deserialize<List<Persona>>(stream, options) ?? [];
```

Devolver una lista vacía cuando el archivo todavía no existe es lo más razonable: la primera vez que corre el programa, no hay nada guardado, así que arrancamos vacíos.

## 6. Una primera versión que persiste

Con esto ya podemos armar un programa que mantiene una lista de personas en un archivo JSON entre ejecuciones:

```csharp
var options = new JsonSerializerOptions { WriteIndented = true };
var personas = new List<Persona>();

// Leer, si hay algo guardado.
if (File.Exists("personas.json")) {
    using var stream = File.OpenRead("personas.json");
    personas = JsonSerializer.Deserialize<List<Persona>>(stream, options) ?? [];
}

// Modificar.
personas.Add(new() { Id = personas.Count + 1, Nombre = "Caro", Apellido = "Cabrera", Edad = 31 });

// Escribir.
using var output = File.Create("personas.json");
JsonSerializer.Serialize(output, personas, options);
```

Funciona. Cada ejecución suma una persona y la guarda. Pero como código, hay tres problemas que duelen:

**El procedimiento es repetitivo.** Cada vez que queremos agregar, modificar o borrar una persona, tenemos que repetir el ciclo "leer del archivo, modificar la lista, escribir al archivo". Si tenemos diez puntos en el programa que tocan datos, tenemos diez copias del ciclo.

**El cálculo del `Id` está embebido.** `personas.Count + 1` parece razonable, pero si borramos una persona y agregamos otra, generamos un `Id` repetido. La lógica de "asignar un Id único" es responsabilidad de quien gestiona la colección, no del código de aplicación.

**Está pegado a `Persona`.** Si mañana necesitamos lo mismo para `Producto` o `Pedido`, hay que copiar y pegar todo.

Estos tres problemas tienen un nombre: **falta una abstracción**. La abstracción que estamos buscando se llama **repositorio**.

## 7. CRUD: las cuatro operaciones que siempre aparecen

Sobre cualquier colección persistente terminamos haciendo cuatro cosas:

- **Create** — agregar un elemento nuevo.
- **Read** — leer uno (por id) o todos.
- **Update** — modificar un elemento existente.
- **Delete** — borrar un elemento (lo vamos a llamar `Remove` para no chocar con la palabra reservada `delete`).

A este conjunto de operaciones se le dice **CRUD**. Aparece en aplicaciones web, en aplicaciones de escritorio, en APIs, en bases de datos. Es tan común que tener una clase dedicada que las exponga es prácticamente automático.

Para que el CRUD funcione necesitamos algo: cada elemento tiene que ser **identificable de forma única**. Si quiero modificar "la persona con id 3", el repositorio tiene que poder encontrarla. Y si quiero crear una nueva, el repositorio tiene que poder asignarle un id que no choque con los existentes.

La identificación va a ser un campo `int Id` en cada entidad. La regla de asignación, la más simple posible: **el siguiente id es el máximo actual más uno** (y si la colección está vacía, empezamos en 1).

---

## 8. `IEntity`: el contrato mínimo

Para que el repositorio pueda leer y asignar el `Id` de cualquier entidad, necesita un punto de acoplamiento: una propiedad llamada `Id` de tipo `int`, con setter y getter públicos. Lo expresamos como una interfaz:

```csharp
interface IEntity {
    int Id { get; set; }
}
```

Esa es toda la interfaz. Una sola propiedad. Y las clases que vayan al repositorio la implementan:

```csharp
public class Persona : IEntity {
    public int    Id       { get; set; }
    public string Nombre   { get; set; } = "";
    public string Apellido { get; set; } = "";
    public int    Edad     { get; set; }
}
```

¿Por qué una interfaz, y no una clase base abstracta? Porque la entidad no necesita heredar comportamiento del repositorio: solo necesita exponer un campo con un nombre y tipo conocidos. Las interfaces son la herramienta correcta cuando lo único que queremos es un **contrato estructural**, sin código compartido.

Esto deja a `Persona` libre de heredar de otra clase si más adelante hace falta —algo que no podríamos garantizar si `IEntity` fuera una clase abstracta, dado que C# no soporta herencia múltiple.

## 9. El repositorio: primera versión, atada a JSON

Antes de hacerlo genérico, escribamos un repositorio concreto para `Persona`, que serializa a JSON. Después lo generalizamos.

```csharp
class PersonaRepository {
    private string filePath;
    private Dictionary<int, Persona> datos = new();

    private static readonly JsonSerializerOptions Options = new() {
        WriteIndented = true,
        PropertyNamingPolicy = JsonNamingPolicy.CamelCase,
        PropertyNameCaseInsensitive = true
    };

    public PersonaRepository(string filePath) {
        this.filePath = filePath;
        if (File.Exists(filePath)) {
            using var stream = File.OpenRead(filePath);
            var lista = JsonSerializer.Deserialize<List<Persona>>(stream, Options) ?? [];
            foreach (var p in lista) {
                datos[p.Id] = p;
            }
        }
    }

    public IEnumerable<Persona> ReadAll() => datos.Values;

    public Persona? ReadOne(int id) => datos.GetValueOrDefault(id);

    public void Create(Persona p) {
        p.Id = datos.Count > 0 ? datos.Values.Max(x => x.Id) + 1 : 1;
        datos[p.Id] = p;
        Save();
    }

    public void Update(Persona p) {
        if (datos.ContainsKey(p.Id)) {
            datos[p.Id] = p;
            Save();
        }
    }

    public void Remove(int id) {
        if (datos.Remove(id)) {
            Save();
        }
    }

    private void Save() {
        using var stream = File.Create(filePath);
        JsonSerializer.Serialize(stream, datos.Values, Options);
    }
}
```

Tres decisiones para señalar:

**El `Dictionary<int, Persona>` interno.** Cargamos todo el archivo en memoria una sola vez, en el constructor, y a partir de ahí trabajamos con un diccionario indexado por `Id`. Esto hace que `ReadOne(id)` sea O(1) en vez de tener que recorrer la lista entera. Es una de las ventajas de tener todo en memoria: las búsquedas por clave son inmediatas.

**`Save()` reescribe el archivo entero.** Cada operación de escritura (Create, Update, Remove) llama a `Save()`, que serializa la colección completa de nuevo. No hay forma de "modificar parcialmente" un archivo JSON: si cambia un solo registro, hay que rescribir todo. Es uno de los costos de este enfoque, y vamos a volver a este punto al final.

**El cálculo del `Id`.** `datos.Values.Max(x => x.Id) + 1`, con un caso especial para la lista vacía. No es la única estrategia (podríamos usar GUIDs, o un contador persistente en otro archivo), pero es la más simple y suficiente para nuestros propósitos.

Esta clase resuelve los problemas del programa de la sección 6: ya no hay ciclos de "leer-modificar-escribir" repetidos en la aplicación, el `Id` se asigna automáticamente, y la lógica de persistencia está en un solo lugar. Pero queda un problema: está atada a `Persona`. Para `Producto` necesitaríamos un `ProductoRepository` idéntico salvo el tipo. Eso es lo que vamos a resolver con genéricos.

## 10. Generalizar: `Repository<T>`

La observación clave es que el código del repositorio nunca usa **nada específico de `Persona`**. No accede a `p.Nombre`, no consulta `p.Edad`. Lo único que necesita es leer y escribir el `Id`, que ya tenemos en `IEntity`. Todo lo demás —serializar, deserializar, almacenar en el diccionario— funciona igual para cualquier tipo.

Esto es la señal canónica de que estamos frente a un **tipo genérico**: queremos el mismo código, pero parametrizado por un tipo `T`.

```csharp
class Repository<T> where T : class, IEntity {
    // ... mismo código que antes, reemplazando Persona por T ...
}
```

Las dos restricciones (`where T : class, IEntity`) tienen propósitos distintos:

- **`class`** dice que `T` tiene que ser un tipo por referencia. Lo necesitamos porque `JsonSerializer.Deserialize<List<T>>` lo exige (no se pueden serializar/deserializar tipos por valor de la misma forma sin más cuidado), y porque queremos que `ReadOne` pueda devolver `T?` con la semántica de "podría ser null".

- **`IEntity`** dice que `T` tiene que tener la propiedad `Id`. Sin esa restricción, el compilador no nos dejaría escribir `item.Id` adentro del repositorio, porque desde su punto de vista `T` podría ser cualquier cosa.

Con eso, la firma queda:

```csharp
interface IRepository<T> where T : IEntity {
    IEnumerable<T> ReadAll();
    T? ReadOne(int id);
    void Create(T item);
    void Update(T item);
    void Remove(int id);
}
```

Sí, agregamos también una **interfaz** `IRepository<T>`. ¿Por qué? Porque puede haber implementaciones distintas (en memoria, en archivo, contra una base de datos, contra un servicio web), y el código que consume el repositorio se beneficia de no atarse a ninguna.

Ahora, cuando alguien quiere persistir productos en JSON, escribe:

```csharp
public class Producto : IEntity {
    public int    Id          { get; set; }
    public string Descripcion { get; set; } = "";
    public decimal Precio     { get; set; }
}

var repo = new Repository<Producto>(...);  // ya veremos qué le pasamos
```

Y obtiene un repositorio funcional, sin una sola línea de código nueva. Eso es lo que ganamos al generalizar.

## 11. ¿Y si quiero XML en vez de JSON?

JSON es nuestro formato favorito para datos modernos, pero hay contextos donde hace falta XML: integraciones con sistemas viejos, archivos de configuración heredados, formatos de intercambio empresariales. La buena noticia es que .NET también incluye un serializador XML listo para usar, en `System.Xml.Serialization`:

```csharp
using System.Xml.Serialization;

var serializer = new XmlSerializer(typeof(List<Persona>));

using (var stream = File.Create("personas.xml")) {
    serializer.Serialize(stream, personas);
}

using (var stream = File.OpenRead("personas.xml")) {
    var leidas = (List<Persona>?)serializer.Deserialize(stream) ?? [];
}
```

La API es parecida pero con dos diferencias notorias:

- `XmlSerializer` se construye con el tipo pasado como parámetro (`typeof(List<Persona>)`), no como genérico. El motivo es histórico: la clase es de las primeras de .NET, anterior a los genéricos.
- `Deserialize` devuelve `object?`, así que hay que castear al tipo concreto.

Si tomamos la versión genérica de la sección anterior y la duplicamos para usar XML en vez de JSON, terminamos con dos clases casi idénticas. Eso es tufo a abstracción faltante: la lógica del repositorio (cargar al diccionario, asignar ids, exponer CRUD) no cambia entre formatos. Lo único que cambia es **cómo se traducen los datos a bytes y de vuelta**.

## 12. Aislar el formato: `IFileSerializer<T>`

Vamos a separar el "cómo persisto" del "qué hago con los datos persistidos". Ese "cómo persisto" se condensa en dos operaciones:

```csharp
interface IFileSerializer<T> where T : class {
    IEnumerable<T> Read();
    void Write(IEnumerable<T> items);
}
```

`Read()` devuelve la colección actual del archivo (vacía si el archivo no existe). `Write(items)` la sobrescribe con lo que reciba. Eso es todo. La interfaz no sabe ni le importa si por debajo hay JSON, XML, un binario propio, o un archivo de texto plano.

Las implementaciones concretas se ocupan cada una de su formato:

```csharp
class JsonFileSerializer<T> : IFileSerializer<T> where T : class {
    private string filePath;

    private static readonly JsonSerializerOptions Options = new() {
        WriteIndented = true,
        PropertyNamingPolicy = JsonNamingPolicy.CamelCase,
        Encoder = System.Text.Encodings.Web.JavaScriptEncoder.UnsafeRelaxedJsonEscaping,
        PropertyNameCaseInsensitive = true
    };

    public JsonFileSerializer(string filePath) {
        this.filePath = filePath;
    }

    public IEnumerable<T> Read() {
        if (!File.Exists(filePath)) return [];

        using var stream = File.OpenRead(filePath);
        return JsonSerializer.Deserialize<List<T>>(stream, Options) ?? [];
    }

    public void Write(IEnumerable<T> items) {
        using var stream = File.Create(filePath);
        JsonSerializer.Serialize(stream, items, Options);
    }
}

class XmlFileSerializer<T> : IFileSerializer<T> where T : class {
    private string filePath;
    private static readonly XmlSerializer Serializer = new(typeof(List<T>));

    public XmlFileSerializer(string filePath) {
        this.filePath = filePath;
    }

    public IEnumerable<T> Read() {
        if (!File.Exists(filePath)) return [];

        using var stream = File.OpenRead(filePath);
        return (IEnumerable<T>?)Serializer.Deserialize(stream) ?? [];
    }

    public void Write(IEnumerable<T> items) {
        using var stream = File.Create(filePath);
        Serializer.Serialize(stream, items.ToList());
    }
}
```

Sobre `UnsafeRelaxedJsonEscaping`: por defecto, `System.Text.Json` escapa caracteres no-ASCII (las tildes, las eñes, los signos de pregunta de apertura) como secuencias `\u00e1`. El nombre suena alarmante, pero la opción "relajada" simplemente deja pasar esos caracteres como UTF-8 normal, que es lo que queremos cuando los archivos los van a leer humanos. Para datos que van a viajar por contextos donde el escape estricto importa (HTML, URLs), conviene la versión por defecto.

Sobre el `.ToList()` en el serializador XML: `XmlSerializer` necesita un tipo concreto, no un `IEnumerable<T>` genérico. Materializar la colección a una lista cumple con eso.

Ahora el repositorio recibe un `IFileSerializer<T>` por constructor, y trabaja contra él sin saber cuál es:

```csharp
class Repository<T> : IRepository<T> where T : class, IEntity {
    private readonly IFileSerializer<T> serializer;
    private Dictionary<int, T> datos = new();

    public Repository(IFileSerializer<T> serializer) {
        this.serializer = serializer;
        foreach (var item in serializer.Read()) {
            datos[item.Id] = item;
        }
    }

    public IEnumerable<T> ReadAll() => datos.Values;

    public T? ReadOne(int id) => datos.GetValueOrDefault(id);

    public void Create(T item) {
        item.Id = datos.Count > 0 ? datos.Values.Max(i => i.Id) + 1 : 1;
        datos[item.Id] = item;
        Save();
    }

    public void Update(T item) {
        if (datos.ContainsKey(item.Id)) {
            datos[item.Id] = item;
            Save();
        }
    }

    public void Remove(int id) {
        if (datos.Remove(id)) {
            Save();
        }
    }

    private void Save() {
        serializer.Write(datos.Values);
    }
}
```

Esto es **inyección de dependencias** en su forma más simple: el repositorio declara que necesita "algo que sepa serializar a archivo" y deja que quien lo construya decida cuál concreto. El repositorio se vuelve agnóstico al formato.

## 13. La fábrica: elegir el serializador por extensión

Ahora hay una pregunta práctica: en el `Main` del programa, ¿quién decide si crear un `JsonFileSerializer<Persona>` o un `XmlFileSerializer<Persona>`?

Una respuesta posible es: que lo decida el código de aplicación, a mano. Funciona, pero es ruidoso —cada uso del repositorio repite el if/switch. Una respuesta mejor es: **que lo decida la extensión del archivo**. Si paso `personas.json`, quiero JSON; si paso `personas.xml`, quiero XML.

Esa lógica de "elegir cuál implementación crear según un parámetro" es lo que se conoce como una **fábrica** (factory). La nuestra es estática y muy chica:

```csharp
static class FileSerializerFactory {
    public static IFileSerializer<T> Create<T>(string filePath) where T : class {
        var ext = Path.GetExtension(filePath).ToLowerInvariant();
        return ext switch {
            ".json" => new JsonFileSerializer<T>(filePath),
            ".xml"  => new XmlFileSerializer<T>(filePath),
            _       => throw new NotSupportedException($"No hay serializador para la extensión '{ext}'.")
        };
    }
}
```

Tres detalles a notar:

**`Path.GetExtension(filePath).ToLowerInvariant()`.** `GetExtension` devuelve la extensión incluyendo el punto (`.json`, no `json`). `ToLowerInvariant` evita que `personas.JSON` o `personas.Xml` —que son archivos perfectamente válidos— rompan la fábrica.

**El `switch` expression.** Reemplaza al `switch` clásico cuando lo que queremos es retornar un valor según un caso. Es más compacto y no necesita `break`. Cada brazo es una expresión, y el patrón `_` cubre los casos no manejados.

**El `throw` en el caso default.** Si alguien pasa `personas.csv`, no tenemos serializador, y lanzamos una excepción explicando el problema. Es preferible fallar ruidosamente acá, en el momento de la creación, que devolver `null` y que el programa falle más tarde con un `NullReferenceException` confuso.

Agregar un nuevo formato (digamos CSV) es ahora un cambio localizado: implementar `CsvFileSerializer<T>` y agregar un brazo al switch. Nada más se toca.

## 14. El programa completo

Juntando todas las piezas, este es el programa de uso:

```csharp
var serializer = FileSerializerFactory.Create<Persona>("datos/contactos.json");
var db = new Repository<Persona>(serializer);

db.Create(new() { Nombre = "Ana",  Apellido = "Araujo",   Edad = 25 });
db.Create(new() { Nombre = "Beto", Apellido = "Benites",  Edad = 40 });
db.Create(new() { Nombre = "Caro", Apellido = "Cabrera",  Edad = 31 });
db.Create(new() { Nombre = "Dani", Apellido = "Díaz",     Edad = 28 });
db.Create(new() { Nombre = "Ema",  Apellido = "Espinoza", Edad = 22 });

Console.WriteLine("\n === Contactos cargados desde JSON ===");
foreach (var p in db.ReadAll()) {
    Console.WriteLine($"- Id: {p.Id,2}, Nombre: {p.Nombre,-20}, Apellido: {p.Apellido,-20}, Edad: {p.Edad}");
}

var c = db.ReadOne(3);
if (c != null) {
    c.Edad = 32;
    db.Update(c);
}

db.Remove(2);
```

Mirá lo que **no** aparece en este código: ni `JsonSerializer`, ni `XmlSerializer`, ni `File.Open*`, ni `using var stream`. La aplicación pide un repositorio para `Persona` apuntando a un archivo, y trabaja con él como si fuera una colección en memoria. Toda la maquinaria de persistencia quedó del otro lado de la abstracción.

Y la promesa que prometimos en la sección 11 se cumple: para cambiar de JSON a XML, hay **una sola línea** que tocar:

```csharp
// Antes:
var serializer = FileSerializerFactory.Create<Persona>("datos/contactos.json");

// Después:
var serializer = FileSerializerFactory.Create<Persona>("datos/contactos.xml");
```

El resto del programa no se entera. Esa es la prueba de que la abstracción está bien hecha: cambiar la implementación concreta no propaga cambios al código de aplicación.

Como referencia rápida, las piezas y sus responsabilidades:

| Pieza                       | Responsabilidad                                                       |
|-----------------------------|-----------------------------------------------------------------------|
| `IEntity`                   | Contrato mínimo: tener un `Id` int.                                   |
| `IRepository<T>`            | Contrato CRUD genérico.                                               |
| `Repository<T>`             | Implementación in-memory + escritura/lectura vía un serializador.     |
| `IFileSerializer<T>`        | Contrato para leer/escribir una colección desde/hacia un archivo.     |
| `JsonFileSerializer<T>`     | Implementación concreta usando `System.Text.Json`.                    |
| `XmlFileSerializer<T>`      | Implementación concreta usando `System.Xml.Serialization`.            |
| `FileSerializerFactory`     | Decide la implementación concreta según la extensión del archivo.     |

## 15. Los límites de este enfoque

El repositorio que armamos es perfectamente serviciable para ejercicios, prototipos y aplicaciones chicas con un único usuario. Pero no escala. En cuanto los datos crecen o el escenario se complica, varios problemas empiezan a aparecer.

**Toda la colección vive en memoria.** El constructor lee el archivo entero al diccionario interno. Para 100 personas no se nota; para 100.000 se nota; para 100.000.000 directamente no entra. No tenemos una forma de cargar parcialmente: la abstracción asume "todo o nada".

**Cualquier cambio reescribe el archivo completo.** Modificar una sola persona implica volcar al disco a las otras 99.999. Es ineficiente y, para archivos grandes, lento.

**No hay búsquedas eficientes salvo por `Id`.** Si quiero "todas las personas con apellido que empieza con G", tengo que recorrer toda la colección con LINQ. No hay índices que aceleren la búsqueda. Para una colección chica está bien; para una grande, cada consulta se vuelve cara.

**No hay atomicidad ni control de errores en mitad de escritura.** Si el programa muere mientras se está ejecutando `Save()`, el archivo queda corrupto: el JSON puede haberse escrito a la mitad y ya no es deserializable. Perdemos los datos. La forma usual de mitigarlo es escribir primero a un archivo temporal y luego renombrar atómicamente, pero eso es complejidad adicional que tendríamos que escribir nosotros.

**No hay acceso concurrente.** Si dos procesos abren el mismo archivo y lo modifican, el último en escribir pisa al primero, sin ninguna advertencia. Y si una sola aplicación tiene varios hilos que tocan el repositorio al mismo tiempo, podemos corromper el diccionario interno o el archivo. Tampoco hay locks.

**No hay validación estructural fuerte.** Si manualmente editamos el archivo y le ponemos `"edad": "veinte"` (un string donde se espera un int), la deserialización explota en la próxima carga. La estructura solo está garantizada si todos los que tocan el archivo son nuestro propio programa; en cuanto otro proceso o un humano lo manipulan, las garantías se debilitan.

Cada uno de estos problemas tiene solución dentro de este modelo, pero la solución implica reinventar progresivamente lo que una **base de datos** ya hace bien: índices, escritura granular, atomicidad por transacciones, locking, validación de esquema, control de concurrencia.

Cuando se llega a ese punto, la respuesta correcta deja de ser "mejorar mi sistema de archivos" y pasa a ser "usar la herramienta diseñada para este problema". Ese es el tema del próximo tutorial.

## 16. Cierre y siguientes pasos

Lo que cubrimos en este tutorial:

- Por qué los formatos serializados estándar (JSON, XML) son mejores que un archivo de texto inventado a mano.
- Cómo usar `JsonSerializer` y `XmlSerializer` de la BCL para leer y escribir colecciones.
- El patrón CRUD y por qué cualquier colección persistente termina exponiéndolo.
- La interfaz `IEntity` como contrato mínimo para que un repositorio pueda gestionar identidades.
- Cómo generalizar un repositorio con `Repository<T> where T : class, IEntity`.
- Cómo separar la lógica del repositorio del formato de persistencia con `IFileSerializer<T>`.
- El patrón fábrica para elegir la implementación concreta según un parámetro (en nuestro caso, la extensión del archivo).
- Inyección de dependencias por constructor: el repositorio declara lo que necesita y deja que el código que lo arma decida cómo satisfacerlo.
- Los límites del enfoque "archivo serializado" y por qué motivan saltar a una base de datos.

Lo que queda pendiente y van a ver más adelante:

- **Bases de datos relacionales**: SQLite, SQL Server, PostgreSQL. Resuelven todo lo que el archivo serializado no resuelve, a cambio de aprender un lenguaje nuevo (SQL).
- **ADO.NET**: la API estándar de .NET para hablar con bases de datos relacionales. La misma idea de "contrato abstracto + proveedor concreto" que vimos con `IFileSerializer<T>` aparece allí en `DbConnection` / `DbCommand`.
- **ORMs (Entity Framework Core)**: una capa que mapea filas de la base a objetos de C# automáticamente, y que conceptualmente cumple el rol de un repositorio glorificado, pero conectado a una base de datos en lugar de a un archivo.

La regla práctica es la misma de siempre: empezar simple, llegar al límite del enfoque actual entendiendo qué problema concreto duele, y solo entonces saltar al siguiente nivel de complejidad. El repositorio sobre archivo serializado no es un paso falso ni una pérdida de tiempo: es la forma más clara de entender qué problemas resuelve una base de datos antes de usarla.
