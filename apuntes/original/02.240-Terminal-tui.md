## 1. ¿Por qué una interfaz de terminal?

Hasta ahora todos los programas de la cátedra fueron de **línea de comandos (CLI)** puros: el usuario escribe argumentos al ejecutar el programa, o el programa escribe preguntas y lee respuestas con `Console.ReadLine()`. Esto es perfecto para tareas chicas, pero se vuelve incómodo cuando la aplicación tiene **muchas opciones**, **estado que cambia** o **datos que conviene ver al mismo tiempo que se editan**.

La alternativa familiar son las **interfaces gráficas** (WinForms, WPF, Avalonia, una página web). Resuelven el problema, pero traen costo: requieren un entorno gráfico, dependencias pesadas, y para tareas administrativas o herramientas de desarrollador suelen ser exageradas.

Hay una tercera vía, antigua pero que sigue teniendo sentido: las **interfaces de terminal (TUI)**. Son aplicaciones que corren dentro de una consola pero usan toda la pantalla, dibujan ventanas, menús, listas, formularios — todo con caracteres. Si alguna vez usaste **Midnight Commander**, **htop** o **vim**, ya viste TUIs.

Las TUIs ganan en tres situaciones concretas:

- **Aplicaciones administrativas** donde lo importante es la velocidad de teclado y no el aspecto gráfico (gestión de bases, herramientas de operaciones, agendas, configuradores).
- **Acceso remoto** vía SSH, donde una GUI no es opción pero una TUI sí.
- **Casos donde una CLI plana se queda corta** pero un entorno gráfico completo es desproporcionado.

Para nuestro contexto académico, las TUIs tienen una ventaja didáctica adicional: nos permiten construir aplicaciones interactivas reales sin tener que aprender un framework gráfico completo encima del lenguaje. Toda la inteligencia que pongamos sigue siendo C# puro, ejecutándose en una consola.

---

## 2. ¿Qué es Terminal.Gui?

Terminal.Gui es una librería de .NET que provee el motor para construir TUIs. Resuelve los problemas técnicos que hacen que escribir una TUI desde cero sea complicado:

- **Renderizado.** Saber qué carácter dibujar en cada celda de la consola, manejar el doble buffer para evitar parpadeo, redibujar solo lo que cambió.
- **Layout.** Calcular dónde van los componentes cuando se redimensiona la ventana del terminal.
- **Input.** Capturar teclas, combinaciones, mouse, y dirigirlas al componente correcto.
- **Compatibilidad.** Hacer que lo mismo funcione en Windows, macOS y Linux, con sus diferentes drivers de consola.

Por arriba de todo eso, expone una **biblioteca de componentes** (botones, etiquetas, campos de texto, listas, tablas, menús, diálogos) que se combinan al estilo de cualquier framework de UI: se crean objetos, se agregan a un contenedor, se configuran sus propiedades y se conectan eventos.

Si conocés WinForms o WPF, el modelo mental va a resultarte familiar. La diferencia es que en lugar de píxeles, todo se mide y se dibuja en **celdas de carácter**.

## 3. Configuración del proyecto

Empecemos creando un proyecto de consola y agregando la librería:

```bash
dotnet new console -n AgendaTui
cd AgendaTui
dotnet add package Terminal.Gui --version "2.0.0-alpha.*"
```

El parámetro `--version "2.0.0-alpha.*"` es importante: sin él, NuGet podría instalar la versión 1.x estable, que tiene una API distinta. Estamos pidiendo explícitamente la última alpha de la línea v2.

Los namespaces que vamos a usar repetidamente:

```csharp
using Terminal.Gui.App;        // Application, IApplication, IRunnable
using Terminal.Gui.ViewBase;   // View, Pos, Dim
using Terminal.Gui.Views;      // Window, Label, Button, MenuBar, ListView, etc.
```

En v1 todo vivía bajo `Terminal.Gui` directamente. En v2 los nombres se reorganizaron en sub-namespaces para clarificar qué cosa es qué. Si copiás código de tutoriales viejos y los `using` no compilan, esa es la primera cosa que conviene revisar.

## 4. El primer programa: una ventana vacía

El programa más simple posible que use Terminal.Gui se ve así:

```csharp
using Terminal.Gui.App;
using Terminal.Gui.ViewBase;
using Terminal.Gui.Views;

using IApplication app = Application.Create();
app.Init();

using Window window = new() {
    Title = "Hola Terminal.Gui (Esc para salir)"
};

Label label = new() {
    Text = "Hola, mundo!",
    X = Pos.Center(),
    Y = Pos.Center()
};

window.Add(label);
app.Run(window);
```

Ejecutalo con `dotnet run`. La consola entera se reemplaza por una ventana con un borde, un título arriba y "Hola, mundo!" centrado. Esc cierra el programa y devuelve la consola al estado original.

Este ejemplo, aunque mínimo, muestra los **cinco conceptos** que estructuran todo programa con Terminal.Gui:

1. **`Application`** — el motor que gestiona el ciclo de vida (driver de consola, bucle de eventos, foco). Se crea una vez con `Application.Create()` y se inicializa con `Init()`. El `using` garantiza que se libere correctamente al cerrar.

2. **`Window`** — el contenedor de tope nivel: una caja con borde y título que ocupa la pantalla. Es lo que el usuario percibe como "la ventana del programa".

3. **`Label`** (y, en general, los demás *Views*) — los componentes visibles individuales. `Label` es solo texto; hay `Button`, `TextField`, `ListView`, `CheckBox`, etc.

4. **El sistema de layout (`Pos` y `Dim`)** — cómo se posicionan y dimensionan los componentes. `Pos.Center()` significa "centrado dentro del padre". Veremos este sistema en detalle.

5. **`app.Run(window)`** — arranca el bucle principal: dibuja, espera input del usuario, lo despacha al componente apropiado, y vuelve a dibujar. Se queda bloqueado ahí hasta que la ventana se cierra.

## 5. La jerarquía de Views

En Terminal.Gui, **todo lo visible es un `View`** — directa o indirectamente. `Window`, `Label`, `Button`, `ListView`, `Dialog`, `MenuBar` — todos heredan de `View`. Esto significa que comparten el mismo conjunto básico de propiedades y comportamientos: posición, tamaño, capacidad de tener foco, capacidad de contener otros views.

La jerarquía típica de una aplicación se ve así:

```
Application
└── Window  (el "toplevel" que ocupa la pantalla)
    ├── MenuBar  (barra de menú arriba)
    ├── Label    (un texto suelto)
    ├── ListView (una lista)
    └── ...
```

Un `View` es un **contenedor potencial**: puede tener subviews adentro mediante `view.Add(...)`. Una `Window` es solo un `View` con borde y título; un `Dialog` es un `View` modal que se abre encima. Todos siguen la misma lógica.

Esta uniformidad tiene una consecuencia muy útil: lo que aprendamos para un componente (cómo posicionarlo, cómo conectarle un evento, cómo darle foco) se aplica igual a todos los demás.

## 6. Layout: `Pos` y `Dim`

Una de las decisiones de diseño más importantes de Terminal.Gui es cómo se especifican posición y tamaño. Cada `View` tiene cuatro propiedades de layout:

- `X` — posición horizontal (de tipo `Pos`)
- `Y` — posición vertical (de tipo `Pos`)
- `Width` — ancho (de tipo `Dim`)
- `Height` — alto (de tipo `Dim`)

Notar que **no son `int`**. Son tipos especiales que representan **expresiones de layout**: en vez de "X = 5", podés decir "X = a la derecha del label anterior, más 1 columna". Estas expresiones se evalúan cuando se dibuja, así que el layout se adapta automáticamente cuando la ventana cambia de tamaño.

Las formas más comunes son:

```csharp
// Posiciones absolutas (como int)
X = 0,
Y = 1,

// Centrado en el padre
X = Pos.Center(),
Y = Pos.Center(),

// Relativo a otro view
X = Pos.Right(otroView) + 1,     // a la derecha de otroView, más 1
Y = Pos.Bottom(otroView) + 1,    // debajo de otroView, más 1
X = Pos.Left(otroView),          // mismo X que otroView (alinear izquierdas)

// Tamaños fijos o que se adaptan
Width = 20,                      // 20 columnas exactas
Width = Dim.Fill(),              // todo el ancho disponible del padre
Width = Dim.Fill() - 2,          // todo menos 2 columnas
Width = Dim.Percent(50),         // la mitad del padre
```

El patrón típico para un formulario es: posicionar el primer label en `(0, 0)`, los siguientes labels uno debajo del otro con `Pos.Bottom(anterior) + 1`, y los campos de texto a la derecha de su label con `Pos.Right(label) + 1`. Si el usuario redimensiona el terminal, todo se reacomoda solo.

Un ejemplo concreto, un formulario de login:

```csharp
Label lblUsuario = new() {
    Text = "Usuario:",
    X = 1,
    Y = 1
};

TextField txtUsuario = new() {
    X = Pos.Right(lblUsuario) + 1,
    Y = Pos.Top(lblUsuario),
    Width = Dim.Fill() - 1
};

Label lblClave = new() {
    Text = "Clave:",
    X = Pos.Left(lblUsuario),
    Y = Pos.Bottom(lblUsuario) + 1
};

TextField txtClave = new() {
    X = Pos.Left(txtUsuario),
    Y = Pos.Top(lblClave),
    Width = Dim.Fill() - 1,
    Secret = true
};

window.Add(lblUsuario, txtUsuario, lblClave, txtClave);
```

Notar `window.Add(...)` con varios argumentos: la mayoría de los métodos `Add` aceptan un array y los agregan en orden.

## 7. Eventos: el patrón `Accepting`

Las TUIs son inherentemente reactivas: el usuario hace algo (presiona una tecla, hace clic), y el programa responde. En Terminal.Gui esto se expresa con **eventos**.

El evento más importante es `Accepting`. Se dispara cuando el usuario "acepta" un componente: presiona Enter sobre un botón, presiona Enter en un campo de texto, hace clic, etc. Es el equivalente a `Click` en WinForms, pero generalizado.

```csharp
Button btnSaludar = new() {
    Text = "Saludar",
    X = Pos.Center(),
    Y = Pos.Center()
};

btnSaludar.Accepting += (sender, e) => {
    MessageBox.Query("Saludo", "Hola desde el botón", "Ok");
    e.Handled = true;
};

window.Add(btnSaludar);
```

Dos detalles importantes:

**`e.Handled = true`** le dice al framework "ya manejé este evento, no sigas propagándolo". Si no lo seteamos, el evento puede llegar al padre del componente y disparar comportamientos no deseados (por ejemplo, cerrar el diálogo). Es una buena práctica setearlo cuando manejamos un `Accepting`.

**`MessageBox.Query`** es un helper para mostrar un cuadro de mensaje modal con botones. Devuelve el índice del botón presionado (0 para "Ok" en este caso). Es la forma más rápida de pedirle una confirmación al usuario.

Otros eventos útiles:

- `KeyDown` / `KeyUp` — para interceptar teclas específicas (por ejemplo, F2 para editar, Delete para borrar).
- `SelectedItemChanged` — en listas y combos, cuando cambia el item seleccionado.
- `OpenSelectedItem` — en listas, cuando el usuario hace doble clic o presiona Enter sobre un item.

## 8. El menú principal

Un menú principal es típicamente lo primero que ve el usuario en una aplicación TUI. En Terminal.Gui se construye con tres clases:

- **`MenuBar`** — la barra horizontal que ocupa la fila superior.
- **`MenuBarItem`** — cada entrada de la barra ("Archivo", "Editar", "Ayuda"). Tiene un menú desplegable adentro.
- **`MenuItem`** — cada opción dentro de un menú desplegable ("Nuevo", "Abrir", "Salir").

La construcción se hace con arrays anidados:

```csharp
MenuBar menu = new(new MenuBarItem[] {
    new MenuBarItem("_Archivo", new MenuItem[] {
        new MenuItem("_Nuevo",  "Crear un contacto", () => NuevoContacto()),
        new MenuItem("_Abrir",  "Abrir agenda",      () => AbrirAgenda()),
        null!,   // separador
        new MenuItem("_Salir",  "Salir del programa", () => app.RequestStop())
    }),
    new MenuBarItem("_Editar", new MenuItem[] {
        new MenuItem("_Buscar", "Buscar contacto", () => Buscar()),
        new MenuItem("_Eliminar", "Eliminar contacto", () => Eliminar())
    }),
    new MenuBarItem("Ay_uda", new MenuItem[] {
        new MenuItem("_Acerca de", "", () => AcercaDe())
    })
});

window.Add(menu);
```

Tres convenciones a explicar:

**El guión bajo (`_`)** marca la **letra de acceso rápido**. `_Archivo` significa que la 'A' aparece subrayada y, con Alt+A, se abre ese menú. Es la forma estándar de dar accesos por teclado, y los usuarios de TUIs la esperan.

**`null!`** intercalado entre `MenuItem`s representa un **separador horizontal** dentro del menú. Sirve para agrupar opciones relacionadas.

**`app.RequestStop()`** termina la aplicación de forma ordenada. Es la forma idiomática de implementar "Salir".

Cada `MenuItem` recibe tres parámetros: la etiqueta visible, un texto de ayuda opcional (que aparece en la barra de estado si la hubiera), y una **acción** (un `Action`, normalmente una lambda). Cuando el usuario selecciona el item, se ejecuta esa acción.

Una vez agregado el menú a la ventana, queda accesible vía mouse, vía Alt+letra, o vía las flechas del teclado.

## 9. Mostrando una lista con `ListView`

`ListView` es uno de los componentes más útiles para aplicaciones de gestión: muestra una lista de elementos, permite seleccionar uno con las flechas, y dispara eventos al seleccionar o "abrir" un item.

Modelo del dominio (consistente con tutoriales anteriores):

```csharp
public record Contacto(int Id, string Nombre, string Telefono, string Email);
```

Versión simple, mostrando solo nombres:

```csharp
List<Contacto> contactos = new() {
    new Contacto(1, "Ana López", "381-111", "ana@mail.com"),
    new Contacto(2, "Beto Pérez", "381-222", "beto@mail.com"),
    new Contacto(3, "Carla Ríos", "381-333", "carla@mail.com")
};

ListView listView = new() {
    X = 0,
    Y = 1,                    // dejar fila 0 para el menú
    Width = Dim.Fill(),
    Height = Dim.Fill()
};

listView.SetSource(contactos.Select(c => c.Nombre).ToList());

window.Add(listView);
```

`SetSource` recibe una colección de **strings** (o más precisamente, un `IList` que la lista sabe convertir a strings). Para mostrar más de un campo de cada contacto, lo más simple es preformatear:

```csharp
var lineas = contactos
    .Select(c => $"{c.Nombre,-30} {c.Telefono,-15} {c.Email}")
    .ToList();
listView.SetSource(lineas);
```

El `,-30` en el formato significa "alineado a la izquierda, mínimo 30 caracteres". Eso da una tabla prolija con columnas alineadas.

`ListView` ya soporta sin código adicional:

- Flechas arriba/abajo para mover la selección.
- Page Up / Page Down para saltar.
- Home / End para ir al principio/fin.
- Scroll automático si la lista es más larga que la altura disponible.
- Mouse para hacer clic y seleccionar.

## 10. Búsqueda en caliente sobre la lista

"Búsqueda en caliente" significa: el usuario escribe en un campo de texto y la lista se filtra en tiempo real, mostrando solo los elementos que coinciden con lo escrito.

El truco es combinar tres ideas:

1. Un `TextField` arriba de la lista para escribir el filtro.
2. Una **lista original** que nunca cambia y una **lista filtrada** que es lo que el `ListView` muestra.
3. Conectarse al evento `TextChanged` del campo: cada vez que el texto cambia, recalcular la lista filtrada y reasignarla al `ListView`.

```csharp
public class PantallaContactos : View {
    private readonly List<Contacto> contactosOriginales;
    private readonly TextField txtBuscar;
    private readonly ListView listView;

    public PantallaContactos(List<Contacto> contactos) {
        this.contactosOriginales = contactos;

        Label lblBuscar = new() {
            Text = "Buscar:",
            X = 0,
            Y = 0
        };

        this.txtBuscar = new TextField {
            X = Pos.Right(lblBuscar) + 1,
            Y = 0,
            Width = Dim.Fill()
        };

        this.listView = new ListView {
            X = 0,
            Y = Pos.Bottom(this.txtBuscar) + 1,
            Width = Dim.Fill(),
            Height = Dim.Fill()
        };

        this.Add(lblBuscar, this.txtBuscar, this.listView);

        this.txtBuscar.TextChanged += (sender, e) => this.RefrescarLista();
        this.RefrescarLista();
    }

    private void RefrescarLista() {
        string filtro = this.txtBuscar.Text?.ToString() ?? "";
        var filtrados = this.contactosOriginales
            .Where(c => string.IsNullOrEmpty(filtro)
                     || c.Nombre.Contains(filtro, StringComparison.OrdinalIgnoreCase))
            .Select(c => $"{c.Nombre,-30} {c.Telefono,-15} {c.Email}")
            .ToList();
        this.listView.SetSource(filtrados);
    }
}
```

Tres cosas para señalar:

**Heredamos de `View`.** Esto es lo idiomático en Terminal.Gui cuando una pantalla tiene varios componentes que coordinar entre sí. La clase derivada encapsula los componentes (campos privados) y la lógica que los conecta (métodos privados como `RefrescarLista`). Después se la agrega a la ventana como cualquier otro view: `window.Add(new PantallaContactos(...))`.

**`StringComparison.OrdinalIgnoreCase`** hace que la búsqueda sea insensible a mayúsculas y minúsculas. Para una **bold text**agenda en español es lo que se espera.

**Llamamos `RefrescarLista()` una vez en el constructor** para que la lista aparezca poblada desde el inicio, sin requerir que el usuario tipee algo primero.

Una sutileza importante: cuando filtramos, la lista del `ListView` ya no contiene los `Contacto` originales sino strings. Si queremos saber qué contacto seleccionó el usuario, tenemos dos opciones: volver a buscar el contacto por el texto seleccionado (frágil), o **mantener una lista paralela** de los contactos filtrados, indexada por la misma posición. Esta segunda opción es la robusta:

```csharp
private List<Contacto> contactosFiltrados = new();

private void RefrescarLista() {
    string filtro = this.txtBuscar.Text?.ToString() ?? "";
    this.contactosFiltrados = this.contactosOriginales
        .Where(c => string.IsNullOrEmpty(filtro)
                 || c.Nombre.Contains(filtro, StringComparison.OrdinalIgnoreCase))
        .ToList();

    var lineas = this.contactosFiltrados
        .Select(c => $"{c.Nombre,-30} {c.Telefono,-15} {c.Email}")
        .ToList();
    this.listView.SetSource(lineas);
}

private Contacto? ContactoSeleccionado() {
    int indice = this.listView.SelectedItem;
    if (indice < 0 || indice >= this.contactosFiltrados.Count) return null;
    return this.contactosFiltrados[indice];
}
```

Ahora `ContactoSeleccionado()` devuelve siempre el `Contacto` correcto correspondiente a lo que está resaltado en la lista, sin importar cuántas veces se filtró.

## 11. Diálogo para editar un contacto

Un diálogo es una ventana modal: aparece encima de la principal, captura todo el input hasta que el usuario la cierra, y devuelve un resultado. Para editar un contacto, necesitamos un diálogo con campos para cada propiedad y dos botones (Aceptar / Cancelar).

La forma idiomática en v2 es heredar de `Dialog` y exponer el resultado vía propiedades:

```csharp
public class EditorContacto : Dialog {
    private readonly TextField txtNombre;
    private readonly TextField txtTelefono;
    private readonly TextField txtEmail;

    public Contacto? Resultado { get; private set; }
    public bool Cancelado { get; private set; } = true;

    public EditorContacto(Contacto contacto) {
        this.Title = "Editar contacto";
        this.Width = 60;
        this.Height = 12;

        Label lblNombre = new() { Text = "Nombre:",   X = 1, Y = 1 };
        Label lblTel    = new() { Text = "Teléfono:", X = 1, Y = 3 };
        Label lblEmail  = new() { Text = "Email:",    X = 1, Y = 5 };

        this.txtNombre = new TextField {
            Text = contacto.Nombre,
            X = 12, Y = 1, Width = Dim.Fill() - 2
        };
        this.txtTelefono = new TextField {
            Text = contacto.Telefono,
            X = 12, Y = 3, Width = Dim.Fill() - 2
        };
        this.txtEmail = new TextField {
            Text = contacto.Email,
            X = 12, Y = 5, Width = Dim.Fill() - 2
        };

        Button btnOk     = new() { Text = "Aceptar", IsDefault = true };
        Button btnCancel = new() { Text = "Cancelar" };

        btnOk.Accepting += (s, e) => {
            this.Resultado = contacto with {
                Nombre   = this.txtNombre.Text?.ToString() ?? "",
                Telefono = this.txtTelefono.Text?.ToString() ?? "",
                Email    = this.txtEmail.Text?.ToString() ?? ""
            };
            this.Cancelado = false;
            Application.RequestStop(this);
            e.Handled = true;
        };

        btnCancel.Accepting += (s, e) => {
            Application.RequestStop(this);
            e.Handled = true;
        };

        this.Add(lblNombre, lblTel, lblEmail,
                 this.txtNombre, this.txtTelefono, this.txtEmail);
        this.AddButton(btnOk);
        this.AddButton(btnCancel);
    }
}
```

Notar varias cosas:

**`Dialog` es un `View` modal con borde y título preconstruido.** Tiene un método especial `AddButton()` que coloca los botones en la fila inferior, centrados y separados — la convención de cualquier diálogo. Los demás views se agregan con `Add()` como siempre.

**`IsDefault = true`** en el botón OK lo marca como el botón "por defecto": presionar Enter en cualquier parte del diálogo (que no sea otro botón) lo activa. Es la convención que esperan los usuarios.

**`contacto with { ... }`** es la sintaxis de records de C# para crear una copia con algunos campos modificados. Esto permite que `Contacto` siga siendo inmutable: el editor no muta el contacto recibido, devuelve uno nuevo con los cambios.

**`Application.RequestStop(this)`** cierra solo este diálogo (no la aplicación entera). El bucle de eventos que estaba corriendo el diálogo termina, y el control vuelve al código que lanzó el diálogo.

**Cómo se usa el diálogo** desde la pantalla principal:

```csharp
private void EditarSeleccionado() {
    var contacto = this.ContactoSeleccionado();
    if (contacto == null) return;

    var editor = new EditorContacto(contacto);
    Application.Run(editor);

    if (!editor.Cancelado && editor.Resultado != null) {
        // reemplazar el contacto editado en la lista original
        int idx = this.contactosOriginales.FindIndex(c => c.Id == contacto.Id);
        if (idx >= 0) this.contactosOriginales[idx] = editor.Resultado;
        this.RefrescarLista();
    }
}
```

`Application.Run(editor)` lanza el bucle del diálogo. El programa queda bloqueado en esa línea hasta que el diálogo se cierra. Al volver, leemos las propiedades `Cancelado` y `Resultado` para saber qué hacer.

## 12. Editar al pulsar Enter

Una de las características más cómodas de las TUIs es que **toda la interacción se puede hacer sin sacar las manos del teclado**. El usuario no debería tener que ir al menú "Editar" cada vez que quiere modificar un contacto: estando con el cursor sobre la fila, debería bastar con Enter.

`ListView` provee exactamente el evento que necesitamos: `OpenSelectedItem`. Se dispara cuando el usuario hace doble clic o presiona Enter sobre un elemento de la lista.

Agregamos al constructor de `PantallaContactos`:

```csharp
this.listView.OpenSelectedItem += (sender, e) => {
    this.EditarSeleccionado();
};
```

Y listo: con flechas para moverse y Enter para abrir el editor, la pantalla queda completamente operable por teclado.

Si quisiéramos asignar otras teclas (por ejemplo, F2 para editar, Delete para borrar, F3 para nuevo), se hace en el evento `KeyDown` del `ListView`:

```csharp
this.listView.KeyDown += (sender, e) => {
    if (e.KeyCode == KeyCode.F2) {
        this.EditarSeleccionado();
        e.Handled = true;
    } else if (e.KeyCode == KeyCode.Delete) {
        this.EliminarSeleccionado();
        e.Handled = true;
    }
};
```

`KeyCode` es un enum con todas las teclas. Lo importante es setear `e.Handled = true` cuando manejamos la tecla, para que el `ListView` no haga su comportamiento por defecto encima.

Una recomendación pedagógica: cuando enseñes esto, mostrá primero `OpenSelectedItem` y solo después `KeyDown`. Los eventos especializados (`OpenSelectedItem`, `SelectedItemChanged`) son más legibles que interceptar teclas a mano, y conviene preferirlos cuando existen.

Y la clase `PantallaContactos` que cubre el filtrado, la edición por Enter, y los métodos públicos que el menú llama:

```csharp
public class PantallaContactos : View {
    private readonly List<Contacto> contactosOriginales;
    private List<Contacto> contactosFiltrados = new();
    private readonly TextField txtBuscar;
    private readonly ListView listView;

    public PantallaContactos(List<Contacto> contactos) {
        this.contactosOriginales = contactos;

        Label lblBuscar = new() { Text = "Buscar:", X = 0, Y = 0 };
        this.txtBuscar = new TextField {
            X = Pos.Right(lblBuscar) + 1, Y = 0, Width = Dim.Fill()
        };
        this.listView = new ListView {
            X = 0, Y = Pos.Bottom(this.txtBuscar) + 1,
            Width = Dim.Fill(), Height = Dim.Fill()
        };

        this.Add(lblBuscar, this.txtBuscar, this.listView);

        this.txtBuscar.TextChanged += (s, e) => this.RefrescarLista();
        this.listView.OpenSelectedItem += (s, e) => this.EditarSeleccionado();
        this.listView.KeyDown += (s, e) => {
            if (e.KeyCode == KeyCode.Delete) {
                this.EliminarSeleccionado();
                e.Handled = true;
            }
        };

        this.RefrescarLista();
    }

    private void RefrescarLista() {
        string filtro = this.txtBuscar.Text?.ToString() ?? "";
        this.contactosFiltrados = this.contactosOriginales
            .Where(c => string.IsNullOrEmpty(filtro)
                     || c.Nombre.Contains(filtro, StringComparison.OrdinalIgnoreCase))
            .ToList();

        var lineas = this.contactosFiltrados
            .Select(c => $"{c.Nombre,-30} {c.Telefono,-15} {c.Email}")
            .ToList();
        this.listView.SetSource(lineas);
    }

    private Contacto? ContactoSeleccionado() {
        int idx = this.listView.SelectedItem;
        if (idx < 0 || idx >= this.contactosFiltrados.Count) return null;
        return this.contactosFiltrados[idx];
    }

    public void Nuevo() {
        int nuevoId = (this.contactosOriginales.Max(c => (int?)c.Id) ?? 0) + 1;
        var vacio = new Contacto(nuevoId, "", "", "");
        var editor = new EditorContacto(vacio);
        Application.Run(editor);
        if (!editor.Cancelado && editor.Resultado != null) {
            this.contactosOriginales.Add(editor.Resultado);
            this.RefrescarLista();
        }
    }

    public void EditarSeleccionado() {
        var c = this.ContactoSeleccionado();
        if (c == null) return;

        var editor = new EditorContacto(c);
        Application.Run(editor);
        if (!editor.Cancelado && editor.Resultado != null) {
            int idx = this.contactosOriginales.FindIndex(x => x.Id == c.Id);
            if (idx >= 0) this.contactosOriginales[idx] = editor.Resultado;
            this.RefrescarLista();
        }
    }

    public void EliminarSeleccionado() {
        var c = this.ContactoSeleccionado();
        if (c == null) return;

        int respuesta = MessageBox.Query(
            "Confirmar",
            $"¿Eliminar a {c.Nombre}?",
            "Sí", "No"
        );
        if (respuesta == 0) {
            this.contactosOriginales.RemoveAll(x => x.Id == c.Id);
            this.RefrescarLista();
        }
    }
}
```

- Diálogo modal para editar todos los campos.

Lo que queda para explorar:

- **Más componentes:** `TableView` (para mostrar datos tabulares con columnas reales), `TreeView`, `TextView` (para edición multilinea), `CheckBox`, `RadioGroup`.
- **`StatusBar`:** una barra inferior que muestra atajos de teclado contextuales. Combinada con MenuBar, da el estilo clásico de aplicación de terminal profesional (F2 Editar, F3 Buscar, F10 Salir).
- **Themes y colores:** Terminal.Gui v2 soporta temas configurables y true color en terminales modernos.
- **TerminalGuiDesigner:** una herramienta gratuita que permite diseñar pantallas visualmente y genera el código C# correspondiente. Puede ser útil para alumnos que prefieren empezar visualmente y leer el código generado.
- **UI Catalog:** el proyecto oficial de ejemplo, con demos de cada componente. `dotnet run --project Examples/UICatalog` (clonando el repo) o explorarlo en GitHub.

Una recomendación final: cuando estés explorando, abrí el UI Catalog y dejalo correr en una terminal aparte. Es la forma más rápida de descubrir qué hace cada componente y cómo se ve en la práctica antes de codearlo.
