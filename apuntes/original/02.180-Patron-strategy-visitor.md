# Patrón Strategy y patrón Visitor en C#

Los dos patrones resuelven problemas de **variación**, pero en ejes distintos.

**Strategy** responde:

> Tengo varias formas de hacer una operación. ¿Cómo elijo una sin llenar el código de `if`, `switch` o herencia rígida?

**Visitor** responde:

> Tengo una estructura de objetos con varios tipos. ¿Cómo agrego nuevas operaciones sobre esos objetos sin modificar sus clases?

Dicho corto:

```text
Strategy = variar el algoritmo.
Visitor  = agregar operaciones sobre una jerarquía de objetos.
```

---

# Parte 1: Patrón Strategy

## 1. El problema que resuelve

Supongamos que tenés que calcular descuentos.

```csharp
public class CalculadoraDescuento
{
    public decimal Calcular(string tipoCliente, decimal total)
    {
        if (tipoCliente == "Normal")
            return total;

        if (tipoCliente == "Premium")
            return total * 0.90m;

        if (tipoCliente == "Vip")
            return total * 0.80m;

        throw new ArgumentException("Tipo de cliente inválido.");
    }
}
```

Funciona, sí.

Pero el problema aparece cuando el sistema crece:

```csharp
if tipoCliente == "Normal"
if tipoCliente == "Premium"
if tipoCliente == "Vip"
if tipoCliente == "Empleado"
if tipoCliente == "Mayorista"
if tipoCliente == "BlackFriday"
if tipoCliente == "CyberMonday"
```

La clase `CalculadoraDescuento` empieza a conocer demasiadas variantes.

Cada nueva política de descuento obliga a modificarla.

Eso viola una idea básica de diseño:

> Si algo cambia por motivos distintos, probablemente debe estar separado.

---

## 2. Primer principio: separar la decisión del algoritmo

Acá hay dos responsabilidades distintas:

```text
1. Saber cuándo aplicar una política.
2. Saber cómo calcular esa política.
```

El cálculo del descuento para cliente VIP no debería estar mezclado con el cálculo del descuento para Black Friday.

Cada política puede ser una clase independiente.

---

## 3. La idea de Strategy

Definimos una interfaz común:

```csharp
public interface IEstrategiaDescuento
{
    decimal Aplicar(decimal total);
}
```

Cada algoritmo concreto implementa esa interfaz.

```csharp
public class DescuentoNormal : IEstrategiaDescuento
{
    public decimal Aplicar(decimal total)
    {
        return total;
    }
}
```

```csharp
public class DescuentoPremium : IEstrategiaDescuento
{
    public decimal Aplicar(decimal total)
    {
        return total * 0.90m;
    }
}
```

```csharp
public class DescuentoVip : IEstrategiaDescuento
{
    public decimal Aplicar(decimal total)
    {
        return total * 0.80m;
    }
}
```

Ahora la calculadora no necesita saber la lógica de cada descuento.

```csharp
public class CalculadoraDescuento
{
    private readonly IEstrategiaDescuento _estrategia;

    public CalculadoraDescuento(IEstrategiaDescuento estrategia)
    {
        _estrategia = estrategia;
    }

    public decimal Calcular(decimal total)
    {
        return _estrategia.Aplicar(total);
    }
}
```

Uso:

```csharp
IEstrategiaDescuento estrategia = new DescuentoPremium();

var calculadora = new CalculadoraDescuento(estrategia);

decimal final = calculadora.Calcular(1000);

Console.WriteLine(final); // 900
```

---

## 4. Filosofía de diseño de Strategy

La filosofía de Strategy es:

> Encapsular algoritmos intercambiables detrás de una abstracción común.

En vez de esto:

```csharp
if (tipo == "A")
{
    // algoritmo A
}
else if (tipo == "B")
{
    // algoritmo B
}
else if (tipo == "C")
{
    // algoritmo C
}
```

hacés esto:

```csharp
IEstrategia estrategia = ObtenerEstrategia();

estrategia.Ejecutar();
```

La clase cliente deja de decidir con `if`.

Delega el comportamiento variable en otro objeto.

---

## 5. Estructura general de Strategy

Interfaz:

```csharp
public interface IEstrategia
{
    void Ejecutar();
}
```

Estrategia concreta A:

```csharp
public class EstrategiaA : IEstrategia
{
    public void Ejecutar()
    {
        Console.WriteLine("Ejecutando estrategia A");
    }
}
```

Estrategia concreta B:

```csharp
public class EstrategiaB : IEstrategia
{
    public void Ejecutar()
    {
        Console.WriteLine("Ejecutando estrategia B");
    }
}
```

Contexto:

```csharp
public class Contexto
{
    private readonly IEstrategia _estrategia;

    public Contexto(IEstrategia estrategia)
    {
        _estrategia = estrategia;
    }

    public void EjecutarOperacion()
    {
        _estrategia.Ejecutar();
    }
}
```

Uso:

```csharp
var contexto = new Contexto(new EstrategiaA());

contexto.EjecutarOperacion();
```

## 6. Strategy en una frase

Strategy es reemplazar este diseño:

```text
Una clase con muchos if/switch que eligen algoritmos.
```

```text
Muchas clases pequeñas, cada una con un algoritmo, usadas a través de una interfaz común.
```

## 7. Caso de uso 1: ordenamiento

Supongamos que querés ordenar productos de distintas maneras.

Modelo:

```csharp
public class Producto
{
    public string Nombre { get; set; } = "";
    public decimal Precio { get; set; }
    public int Stock { get; set; }
}
```

Interfaz de estrategia:

```csharp
public interface IEstrategiaOrdenamientoProductos
{
    IEnumerable<Producto> Ordenar(IEnumerable<Producto> productos);
}
```

Ordenar por precio:

```csharp
public class OrdenarPorPrecio : IEstrategiaOrdenamientoProductos
{
    public IEnumerable<Producto> Ordenar(IEnumerable<Producto> productos)
    {
        return productos.OrderBy(p => p.Precio);
    }
}
```

Ordenar por nombre:

```csharp
public class OrdenarPorNombre : IEstrategiaOrdenamientoProductos
{
    public IEnumerable<Producto> Ordenar(IEnumerable<Producto> productos)
    {
        return productos.OrderBy(p => p.Nombre);
    }
}
```

Ordenar por stock descendente:

```csharp
public class OrdenarPorStockDescendente : IEstrategiaOrdenamientoProductos
{
    public IEnumerable<Producto> Ordenar(IEnumerable<Producto> productos)
    {
        return productos.OrderByDescending(p => p.Stock);
    }
}
```

Contexto:

```csharp
public class Catalogo
{
    private readonly IEstrategiaOrdenamientoProductos _estrategia;

    public Catalogo(IEstrategiaOrdenamientoProductos estrategia)
    {
        _estrategia = estrategia;
    }

    public IEnumerable<Producto> ObtenerOrdenados(IEnumerable<Producto> productos)
    {
        return _estrategia.Ordenar(productos);
    }
}
```

Uso:

```csharp
var productos = new List<Producto>
{
    new() { Nombre = "Mouse", Precio = 15000, Stock = 20 },
    new() { Nombre = "Teclado", Precio = 30000, Stock = 10 },
    new() { Nombre = "Monitor", Precio = 200000, Stock = 5 }
};

var catalogo = new Catalogo(new OrdenarPorPrecio());

foreach (var producto in catalogo.ObtenerOrdenados(productos))
{
    Console.WriteLine($"{producto.Nombre} - {producto.Precio}");
}
```

## 8. Caso de uso 2: exportadores

Tenés que exportar un reporte en distintos formatos.

Mala solución:

```csharp
public class ReporteService
{
    public void Exportar(string formato)
    {
        if (formato == "PDF")
        {
            Console.WriteLine("Exportando PDF...");

        else if (formato == "Excel")
        {
            Console.WriteLine("Exportando Excel...");
        }
        else if (formato == "CSV")

            Console.WriteLine("Exportando CSV...");

}
```

```csharp
public interface IExportadorReporte
{

}
```

```csharp
public class ExportadorPdf : IExportadorReporte
{
    public void Exportar(Reporte reporte)
    {
        Console.WriteLine($"Exportando '{reporte.Titulo}' a PDF");
    }
}
```

```csharp
public class ExportadorExcel : IExportadorReporte
{
    public void Exportar(Reporte reporte)
    {
        Console.WriteLine($"Exportando '{reporte.Titulo}' a Excel");
    }
}
```

```csharp
public class ExportadorCsv : IExportadorReporte
{
    public void Exportar(Reporte reporte)
    {
        Console.WriteLine($"Exportando '{reporte.Titulo}' a CSV");
    }
}
```

Modelo:

```csharp
public class Reporte
{
    public string Titulo { get; set; } = "";
}
```

Servicio:

```csharp
public class ReporteService
{
    private readonly IExportadorReporte _exportador;

    public ReporteService(IExportadorReporte exportador)
    {
        _exportador = exportador;
    }

    public void Exportar(Reporte reporte)
    {
        _exportador.Exportar(reporte);

```

```csharp
var reporte = new Reporte { Titulo = "Ventas 2026" };
```

## 11. Caso de uso 5: validaciones intercambiables

```csharp
public interface IValidadorUsuario
{
    bool EsValido(Usuario usuario);
}
```

```csharp
public class ValidadorUsuarioBasico : IValidadorUsuario
{
```

```csharp
public class ValidadorUsuarioEstricto : IValidadorUsuario
```

## 12. Strategy con delegates

En C#, no siempre necesitás crear una clase por estrategia.

Para casos simples, podés usar funciones.

```csharp
public class CalculadoraDescuento
{
    private readonly Func<decimal, decimal> _estrategia;

    public CalculadoraDescuento(Func<decimal, decimal> estrategia)
    {
        _estrategia = estrategia;
    }

    public decimal Calcular(decimal total)
    {
        return _estrategia(total);
    }
}
```

Uso:

```csharp
var calculadoraPremium = new CalculadoraDescuento(
    total => total * 0.90m
);

Console.WriteLine(calculadoraPremium.Calcular(1000)); // 900
```

Otra:

```csharp
var calculadoraVip = new CalculadoraDescuento(
    total => total * 0.80m
);
```

Esto es Strategy también, pero expresado con funciones.

Para lógica simple, una clase entera puede ser demasiado ceremonial.

No hace falta traer un tanque para matar un mosquito.

## 13. Strategy con Dependency Injection

En una app real, las estrategias suelen registrarse en el contenedor.

```csharp
builder.Services.AddScoped<ExportadorPdf>();
builder.Services.AddScoped<ExportadorExcel>();
builder.Services.AddScoped<ExportadorCsv>();
```

Después podés resolver según una clave.

Una forma simple:

```csharp
public class ExportadorFactory
{
    private readonly IServiceProvider _serviceProvider;

    public ExportadorFactory(IServiceProvider serviceProvider)
    {
        _serviceProvider = serviceProvider;
    }

    public IExportadorReporte Crear(string formato)
    {
        return formato.ToUpperInvariant() switch
        {
            "PDF" => _serviceProvider.GetRequiredService<ExportadorPdf>(),
            "EXCEL" => _serviceProvider.GetRequiredService<ExportadorExcel>(),
            "CSV" => _serviceProvider.GetRequiredService<ExportadorCsv>(),
            _ => throw new ArgumentException("Formato inválido.")
        };
    }
}
```

Uso:

```csharp
IExportadorReporte exportador = factory.Crear("PDF");
exportador.Exportar(reporte);
```

Ojo: el `switch` no desaparece mágicamente. Muchas veces se mueve a un lugar más adecuado.

Eso está bien.

El problema no es que exista un `switch`.

El problema es tener `switch` desperdigados por toda la aplicación como migas de pan malditas.

---

## 14. Cuándo usar Strategy

Usalo cuando:

```text
tenés varios algoritmos intercambiables
querés evitar if/switch repetidos
querés cambiar comportamiento en runtime
querés testear cada algoritmo por separado
querés cumplir Open/Closed Principle
tenés políticas, reglas, validaciones, cálculos o formatos variables
```

---

## 15. Cuándo no usar Strategy

No lo uses cuando:

```text
hay una sola variante
el algoritmo no cambia
el if es simple y local
crear clases empeora la legibilidad
la variación no tiene entidad conceptual
```

Ejemplo absurdo:

```csharp
public interface ISumador
{
    int Sumar(int a, int b);
}
```

```csharp
public class SumadorNormal : ISumador
{
    public int Sumar(int a, int b) => a + b;
}
```

Eso no es diseño. Es burocracia premium.

# Parte 2: Patrón Visitor

Visitor es más difícil de entender porque resuelve un problema menos obvio.

---

## 16. El problema que resuelve Visitor

Supongamos que tenés una jerarquía de elementos de un documento.

```csharp
public abstract class ElementoDocumento
{
}
```

```csharp
public class Parrafo : ElementoDocumento
{
    public string Texto { get; set; } = "";
}
```

```csharp
public class Imagen : ElementoDocumento
{
    public string Url { get; set; } = "";
    public string TextoAlternativo { get; set; } = "";
}
```

```csharp
public class Tabla : ElementoDocumento
{
    public int Filas { get; set; }
    public int Columnas { get; set; }
}
```

Ahora querés hacer operaciones sobre esos elementos.

Por ejemplo:

```text
exportar a HTML
exportar a Markdown
contar palabras
validar accesibilidad
generar resumen
calcular estadísticas
```

Una opción es meter métodos en cada clase:

```csharp
public class Parrafo : ElementoDocumento
{
    public string Texto { get; set; } = "";

    public string ToHtml()
    {
        return $"<p>{Texto}</p>";
    }

    public string ToMarkdown()
    {
        return Texto;
    }

    public int ContarPalabras()
    {
        return Texto.Split(' ').Length;
    }
}
```

Pero rápidamente cada clase se llena de operaciones.

`Parrafo`, `Imagen` y `Tabla` empiezan a saber de HTML, Markdown, accesibilidad, estadísticas, validación, etc.

Eso mezcla responsabilidades.

---

## 17. Primer principio: tipos estables, operaciones variables

Visitor conviene cuando tenés esta situación:

```text
La jerarquía de clases cambia poco.
Las operaciones sobre esa jerarquía cambian mucho.
```

Ejemplo:

```text
Tipos estables:
- Parrafo
- Imagen
- Tabla
- Titulo
- Lista

Operaciones variables:
- Exportar HTML
- Exportar Markdown
- Validar accesibilidad
- Contar palabras
- Generar índice
- Extraer imágenes
```

Si cada operación vive dentro de las clases del modelo, las clases se contaminan.

Visitor mueve esas operaciones afuera.

---

## 18. Filosofía de diseño de Visitor

La filosofía de Visitor es:

> Separar operaciones de la estructura de objetos sobre la que operan.

El objeto visitado representa una entidad.

El visitante representa una operación.

```text
Parrafo  = elemento del documento
Imagen   = elemento del documento
Tabla    = elemento del documento

HtmlVisitor          = sabe exportar elementos a HTML
MarkdownVisitor      = sabe exportar elementos a Markdown
WordCountVisitor     = sabe contar palabras
AccessibilityVisitor = sabe validar accesibilidad
```

El elemento no hace la operación.

El elemento acepta un visitante.

---

## 19. Estructura general de Visitor

Interfaz del visitante:

```csharp
public interface IElementoDocumentoVisitor
{
    void Visit(Parrafo parrafo);
    void Visit(Imagen imagen);
    void Visit(Tabla tabla);
}
```

Elemento base:

```csharp
public abstract class ElementoDocumento
{
    public abstract void Accept(IElementoDocumentoVisitor visitor);
}
```

Elemento concreto:

```csharp
public class Parrafo : ElementoDocumento
{
    public string Texto { get; set; } = "";

    public override void Accept(IElementoDocumentoVisitor visitor)
    {
        visitor.Visit(this);
    }
}
```

```csharp
public class Imagen : ElementoDocumento
{
    public string Url { get; set; } = "";
    public string TextoAlternativo { get; set; } = "";

    public override void Accept(IElementoDocumentoVisitor visitor)
    {
        visitor.Visit(this);
    }
}
```

```csharp
public class Tabla : ElementoDocumento
{
    public int Filas { get; set; }
    public int Columnas { get; set; }

    public override void Accept(IElementoDocumentoVisitor visitor)
    {
        visitor.Visit(this);
    }
}
```

Visitor concreto:

```csharp
public class HtmlVisitor : IElementoDocumentoVisitor
{
    public void Visit(Parrafo parrafo)
    {
        Console.WriteLine($"<p>{parrafo.Texto}</p>");
    }

    public void Visit(Imagen imagen)
    {
        Console.WriteLine($"<img src=\"{imagen.Url}\" alt=\"{imagen.TextoAlternativo}\" />");
    }

    public void Visit(Tabla tabla)
    {
        Console.WriteLine($"<table data-filas=\"{tabla.Filas}\" data-columnas=\"{tabla.Columnas}\"></table>");
    }
}
```

Uso:

```csharp
var elementos = new List<ElementoDocumento>
{
    new Parrafo { Texto = "Hola mundo" },
    new Imagen { Url = "foto.png", TextoAlternativo = "Foto de perfil" },
    new Tabla { Filas = 3, Columnas = 4 }
};

var visitor = new HtmlVisitor();

foreach (var elemento in elementos)
{
    elemento.Accept(visitor);
}
```

---

## 20. ¿Por qué hace falta `Accept`?

Esta es la parte clave.

Podrías intentar esto:

```csharp
foreach (var elemento in elementos)
{
    visitor.Visit(elemento);
}
```

Pero `elemento` está declarado como:

```csharp
ElementoDocumento elemento
```

El compilador solo conoce el tipo base.

No sabe si en ejecución es `Parrafo`, `Imagen` o `Tabla`.

El método correcto sería:

```csharp
Visit(Parrafo)
Visit(Imagen)
Visit(Tabla)
```

Pero en tiempo de compilación solo tiene:

```csharp
ElementoDocumento
```

Entonces usamos:

```csharp
elemento.Accept(visitor);
```

Y dentro de cada clase concreta:

```csharp
visitor.Visit(this);
```

Ahí `this` sí tiene el tipo concreto.

En `Parrafo`:

```csharp
visitor.Visit(this); // this es Parrafo
```

En `Imagen`:

```csharp
visitor.Visit(this); // this es Imagen
```

En `Tabla`:

```csharp
visitor.Visit(this); // this es Tabla
```

Esto se llama **double dispatch**.

---

## 21. Double dispatch explicado sin humo

En una llamada normal, C# decide el método usando un receptor:

```csharp
elemento.Accept(visitor);
```

Primer despacho:

```text
¿Qué implementación de Accept uso?
Depende del tipo real de elemento.
```

Después, dentro del elemento:

```csharp
visitor.Visit(this);
```

Segundo despacho:

```text
¿Qué overload de Visit uso?
Depende del tipo concreto de this.
```

Por eso se llama **doble despacho**.

No es magia negra. Es magia con contrato.


```csharp
public class Titulo : ElementoDocumento

    public int Nivel { get; set; } = 1;

    public override void Accept(IElementoDocumentoVisitor visitor)
    {
        visitor.Visit(this);
    }
}
```

```csharp
public class Parrafo : ElementoDocumento
{
    public string Texto { get; set; } = "";

    public override void Accept(IElementoDocumentoVisitor visitor)
    {
        visitor.Visit(this);

```


Visitor HTML:

```csharp
public class HtmlExportVisitor : IElementoDocumentoVisitor
{
    private readonly List<string> _html = new();

    public void Visit(Titulo titulo)
    {
        _html.Add($"<h{titulo.Nivel}>{titulo.Texto}</h{titulo.Nivel}>");
    }

    public void Visit(Parrafo parrafo)
    {
        _html.Add($"<p>{parrafo.Texto}</p>");
    }

    public void Visit(Imagen imagen)
    {
        _html.Add($"<img src=\"{imagen.Url}\" alt=\"{imagen.Alt}\" />");
    }

    public string ObtenerHtml()
    {
        return string.Join(Environment.NewLine, _html);
    }
}
```

Uso:

```csharp
var documento = new List<ElementoDocumento>
{
    new Titulo { Texto = "Mi documento", Nivel = 1 },
    new Parrafo { Texto = "Este es el primer párrafo." },
    new Imagen { Url = "foto.png", Alt = "Una foto" }
};

var visitor = new HtmlExportVisitor();

foreach (var elemento in documento)
{
    elemento.Accept(visitor);
}

Console.WriteLine(visitor.ObtenerHtml());
```

Salida:

```html
<h1>Mi documento</h1>
<p>Este es el primer párrafo.</p>
<img src="foto.png" alt="Una foto" />
```

## 23. Caso de uso 2: exportar a Markdown

Sin tocar `Titulo`, `Parrafo` ni `Imagen`, agregás otra operación.

```csharp
public class MarkdownExportVisitor : IElementoDocumentoVisitor
```

```csharp
var visitor = new MarkdownExportVisitor();
```

## 25. Caso de uso 4: validar accesibilidad

```csharp
public class AccessibilityVisitor : IElementoDocumentoVisitor
{
    private readonly List<string> _errores = new();

    public void Visit(Titulo titulo)
    {
        if (titulo.Nivel < 1 || titulo.Nivel > 6)
        {
            _errores.Add($"Título inválido: nivel {titulo.Nivel}");
        }
    }

        if (string.IsNullOrWhiteSpace(parrafo.Texto))

    public void Visit(Imagen imagen)
    {

            _errores.Add($"La imagen '{imagen.Url}' no tiene texto alternativo.");
        }
    }

    public IReadOnlyList<string> ObtenerErrores()
    {
        return _errores;
    }
}
```

```csharp
var visitor = new AccessibilityVisitor();

foreach (var elemento in documento)
{

}
```

## 26. Caso de uso 5: AST / compiladores / intérpretes

Visitor se usa muchísimo en árboles de sintaxis abstracta.

Supongamos expresiones matemáticas:

```text
1 + 2 * 3
```

Podrías representarlo así:

```text
Suma
 ├── Numero(1)
 └── Multiplicacion
      ├── Numero(2)
      └── Numero(3)
```

Modelo:

```csharp
public abstract class Expresion
{
    public abstract T Accept<T>(IExpresionVisitor<T> visitor);
}
```

Visitor genérico:

```csharp
public interface IExpresionVisitor<T>
{
    T VisitNumero(Numero numero);
    T VisitSuma(Suma suma);
    T VisitMultiplicacion(Multiplicacion multiplicacion);
}
```

Expresiones:

```csharp
public class Numero : Expresion
{
    public int Valor { get; }

    public Numero(int valor)
    {
        Valor = valor;
    }

    public override T Accept<T>(IExpresionVisitor<T> visitor)
    {
        return visitor.VisitNumero(this);
    }
}
```

```csharp
public class Suma : Expresion
{
    public Expresion Izquierda { get; }
    public Expresion Derecha { get; }

    public Suma(Expresion izquierda, Expresion derecha)
    {
        Izquierda = izquierda;
        Derecha = derecha;
    }

    public override T Accept<T>(IExpresionVisitor<T> visitor)
    {
        return visitor.VisitSuma(this);
    }
}
```

```csharp
public class Multiplicacion : Expresion
{
    public Expresion Izquierda { get; }
    public Expresion Derecha { get; }

    public Multiplicacion(Expresion izquierda, Expresion derecha)
    {
        Izquierda = izquierda;
        Derecha = derecha;
    }

    public override T Accept<T>(IExpresionVisitor<T> visitor)
    {
        return visitor.VisitMultiplicacion(this);
    }
}
```

Visitor evaluador:

```csharp
public class EvaluadorVisitor : IExpresionVisitor<int>
{
    public int VisitNumero(Numero numero)
    {
        return numero.Valor;
    }

    public int VisitSuma(Suma suma)
    {
        return suma.Izquierda.Accept(this) + suma.Derecha.Accept(this);
    }

    public int VisitMultiplicacion(Multiplicacion multiplicacion)
    {
        return multiplicacion.Izquierda.Accept(this) * multiplicacion.Derecha.Accept(this);
    }
}
```

Visitor impresor:

```csharp
public class ImpresorVisitor : IExpresionVisitor<string>
{
    public string VisitNumero(Numero numero)
    {
        return numero.Valor.ToString();
    }

    public string VisitSuma(Suma suma)
    {
        return $"({suma.Izquierda.Accept(this)} + {suma.Derecha.Accept(this)})";
    }

    public string VisitMultiplicacion(Multiplicacion multiplicacion)
    {
        return $"({multiplicacion.Izquierda.Accept(this)} * {multiplicacion.Derecha.Accept(this)})";
    }
}
```

Uso:

```csharp
Expresion expresion =
    new Suma(
        new Numero(1),
        new Multiplicacion(
            new Numero(2),
            new Numero(3)
        )
    );

var evaluador = new EvaluadorVisitor();
var impresor = new ImpresorVisitor();

Console.WriteLine(expresion.Accept(evaluador)); // 7
Console.WriteLine(expresion.Accept(impresor));  // (1 + (2 * 3))
```

Este es un ejemplo muy bueno porque muestra cuándo Visitor brilla.

El árbol de expresiones suele ser relativamente estable.

Las operaciones pueden crecer:

```text
evaluar
imprimir
optimizar
traducir a SQL
traducir a JavaScript
validar tipos
generar bytecode
```

## 27. Visitor genérico con resultado

La versión anterior devuelve un resultado:


Eso es más flexible que un visitor con `void`.


```csharp
string Visit(Parrafo parrafo);
```

```csharp

public abstract class ElementoDocumento
```

```csharp
public class Parrafo : ElementoDocumento
```

## 28. Visitor y Open/Closed Principle

Visitor tiene una particularidad interesante.

Es muy bueno para agregar **nuevas operaciones**.

Por ejemplo, ya tenés:

```text
Parrafo
Imagen
Tabla
```

Podés agregar:

```text
HtmlVisitor
MarkdownVisitor
WordCountVisitor
AccessibilityVisitor
```

sin tocar las clases existentes.

Pero es malo para agregar **nuevos tipos de elementos**.

Si agregás:

```csharp
public class Video : ElementoDocumento
{
}
```

tenés que modificar la interfaz:

```csharp
void Visit(Video video);
```

Y entonces tenés que actualizar todos los visitors existentes.

Ese es el precio.

Visitor conviene cuando los tipos son estables y las operaciones cambian.

## 29. Strategy vs Visitor

Aunque ambos usan interfaces y polimorfismo, resuelven problemas distintos.

### Strategy

Tengo una operación y muchas formas de hacerla.

```text
CalcularDescuento
    - descuento normal
    - descuento premium
    - descuento VIP
```

El cliente elige una estrategia.

```csharp
var calculadora = new CalculadoraDescuento(new DescuentoVip());
```

### Visitor

Tengo muchos tipos de objetos y quiero aplicarles nuevas operaciones.

```text
Elementos:
    - Parrafo
    - Imagen
    - Tabla

Operaciones:
    - exportar HTML
    - exportar Markdown
    - contar palabras
```

El objeto acepta un visitante.

```csharp
elemento.Accept(visitor);
```

---

## 30. Diferencia conceptual fuerte

Strategy encapsula una variación horizontal de comportamiento:

```text
Mismo tipo de operación, distintas implementaciones.
```

Visitor encapsula operaciones transversales sobre muchos tipos:

```text
Mismos tipos de datos, distintas operaciones externas.
```

Ejemplo:

```text
Strategy:
    ¿Cómo calculo el envío?
    - Por distancia
    - Por peso
    - Por zona
    - Gratis por promoción

Visitor:
    ¿Qué hago con cada elemento de un documento?
    - Lo exporto a HTML
    - Lo exporto a Markdown
    - Lo valido
    - Lo cuento
```

---

## 31. Comparación directa

| Aspecto | Strategy | Visitor |
|---|---|---|
| Problema central | Cambiar algoritmo | Agregar operaciones sobre una jerarquía |
| Pregunta que responde | ¿Cómo hago esto? | ¿Qué operación aplico a estos tipos? |
| Variación principal | Algoritmos | Operaciones |
| Estructura típica | `IEstrategia` | `IVisitor` + `Accept` |
| Cliente llama | `estrategia.Ejecutar()` | `elemento.Accept(visitor)` |
| Bueno para agregar | Nuevas estrategias | Nuevas operaciones |
| Malo para agregar | Muchas dependencias entre estrategias | Nuevos tipos visitables |
| Ejemplos | descuentos, pagos, ordenamiento | AST, documentos, UI tree, reportes |

---

## 32. Strategy y Visitor juntos

Pueden combinarse.

Supongamos que tenés un sistema de documentos.

Visitor puede recorrer elementos:

```text
Parrafo
Imagen
Tabla
```

Y Strategy puede decidir cómo renderizar ciertas partes.

Ejemplo conceptual:

```csharp
public class HtmlExportVisitor : IElementoDocumentoVisitor
{
    private readonly IEstrategiaEscapeHtml _escape;

    public HtmlExportVisitor(IEstrategiaEscapeHtml escape)
    {
        _escape = escape;
    }

    public void Visit(Parrafo parrafo)
    {
        string textoSeguro = _escape.Escapar(parrafo.Texto);
        Console.WriteLine($"<p>{textoSeguro}</p>");
    }

    public void Visit(Imagen imagen)
    {
        string altSeguro = _escape.Escapar(imagen.Alt);
        Console.WriteLine($"<img src=\"{imagen.Url}\" alt=\"{altSeguro}\" />");
    }

    public void Visit(Tabla tabla)
    {
        Console.WriteLine("<table></table>");
    }
}
```

Acá:

```text
Visitor = operación de exportar HTML sobre muchos elementos.
Strategy = forma de escapar texto HTML.
```

## 33. Visitor vs Pattern Matching moderno en C#

En C# moderno, muchas veces podés reemplazar Visitor con pattern matching.

Ejemplo:

```csharp
public static string ToMarkdown(ElementoDocumento elemento)
{
    return elemento switch
    {
        Titulo t => $"{new string('#', t.Nivel)} {t.Texto}",
        Parrafo p => p.Texto,
        Imagen i => $"![{i.Alt}]({i.Url})",
        _ => throw new NotSupportedException()
    };
}
```

Esto es mucho más simple.

Entonces, ¿Visitor sigue teniendo sentido?

Sí, cuando:

```text
la operación es grande
hay muchas operaciones
querés objetos visitor reutilizables
necesitás acumular estado durante el recorrido
querés evitar switch repetidos
querés doble despacho explícito
trabajás con árboles complejos
```

Para casos chicos, `switch` con pattern matching puede ser más limpio.

No hay que usar Visitor para saludar a tres clases. Eso es mandar una comitiva diplomática a comprar pan.

## 34. Visitor vs Composite

Visitor se combina mucho con Composite.

Composite permite representar árboles:

```text
Carpeta
    Archivo
    Carpeta
        Archivo
        Archivo
```

Visitor permite operar sobre ese árbol:

```text
calcular tamaño
listar nombres
buscar archivos
exportar estructura
```

Ejemplo simple:

```csharp
public abstract class NodoArchivo
{
    public string Nombre { get; set; } = "";
    public abstract void Accept(INodoArchivoVisitor visitor);
}
```

```csharp
public class Archivo : NodoArchivo
{
    public long Tamano { get; set; }

    public override void Accept(INodoArchivoVisitor visitor)
    {
        visitor.Visit(this);
    }
}
```

```csharp
public class Carpeta : NodoArchivo
{
    public List<NodoArchivo> Hijos { get; } = new();

    public override void Accept(INodoArchivoVisitor visitor)
    {
        visitor.Visit(this);

        foreach (var hijo in Hijos)
        {
            hijo.Accept(visitor);
        }
    }
}
```

Visitor:

```csharp
public interface INodoArchivoVisitor
{
    void Visit(Archivo archivo);
    void Visit(Carpeta carpeta);
}
```

Calcular tamaño:

```csharp
public class CalcularTamanoVisitor : INodoArchivoVisitor
{
    public long Total { get; private set; }

    public void Visit(Archivo archivo)
    {
        Total += archivo.Tamano;
    }

    public void Visit(Carpeta carpeta)
    {
        // La carpeta no suma tamaño propio en este ejemplo.
    }
}
```

Uso:

```csharp
var raiz = new Carpeta { Nombre = "root" };

raiz.Hijos.Add(new Archivo { Nombre = "a.txt", Tamano = 100 });
raiz.Hijos.Add(new Archivo { Nombre = "b.txt", Tamano = 300 });

var visitor = new CalcularTamanoVisitor();

raiz.Accept(visitor);

Console.WriteLine(visitor.Total); // 400
```

## 35. Cuándo usar Visitor

Usalo cuando:

```text
tenés una jerarquía de tipos relativamente estable
querés agregar muchas operaciones externas
no querés llenar las clases del modelo de lógica secundaria
necesitás recorrer árboles o estructuras compuestas
querés acumular estado durante una operación
trabajás con AST, documentos, nodos, expresiones, estructuras de UI
```

---

## 36. Cuándo no usar Visitor

No lo uses cuando:

```text
la jerarquía cambia todo el tiempo
hay pocos tipos y pocas operaciones
un switch con pattern matching alcanza
el costo ceremonial es mayor que el beneficio
las operaciones pertenecen naturalmente al objeto
```

Ejemplo donde Visitor sobra:

```csharp
public class Persona
{
    public string Nombre { get; set; } = "";

    public void Saludar()
    {
        Console.WriteLine($"Hola, soy {Nombre}");
    }
}
```

No hagas:

```csharp
PersonaAcceptSaludoVisitorFactoryService
```

Ahí no hay arquitectura; hay un trámite municipal.

## 37. Relación con SOLID

### Strategy y SRP

Cada estrategia tiene una responsabilidad clara:

```text
DescuentoVip       → calcular descuento VIP
DescuentoPremium   → calcular descuento premium
DescuentoMayorista → calcular descuento mayorista
```

### Strategy y OCP

Podés agregar una nueva estrategia sin modificar la clase cliente.

```csharp
public class DescuentoBlackFriday : IEstrategiaDescuento
{
    public decimal Aplicar(decimal total)
    {
        return total * 0.50m;
    }
}
```

### Visitor y SRP

Los elementos del modelo no se llenan de operaciones externas.

```text
Parrafo no necesita saber exportar a HTML, Markdown, PDF y validar accesibilidad.
```

### Visitor y OCP

Visitor es abierto para nuevas operaciones, pero cerrado de manera imperfecta para nuevos elementos.

Esto es importante: ningún patrón te regala todo.

Visitor elige un eje de extensibilidad:

```text
Fácil agregar operaciones.
Difícil agregar nuevos tipos.
```

## 40. Regla práctica para elegir

Usá **Strategy** cuando tu frase mental sea:

> “Necesito distintas formas de hacer la misma cosa.”

Ejemplos:

```text
distintas formas de pagar
distintas formas de ordenar
distintas formas de validar
distintas formas de calcular descuentos
distintas formas de exportar un reporte
```

Usá **Visitor** cuando tu frase mental sea:

> “Necesito aplicar distintas operaciones sobre una familia de objetos.”

Ejemplos:

```text
documento con títulos, párrafos e imágenes
árbol de archivos con carpetas y archivos
AST con números, sumas y multiplicaciones
UI tree con botones, paneles y etiquetas
```

## 41. Cierre brutalmente útil

### Strategy

```text
Problema:
    muchos algoritmos intercambiables.

Solución:
    encapsular cada algoritmo en una clase o función.

Resultado:
    el contexto usa una abstracción y no sabe qué algoritmo concreto está usando.
```

Código mental:

```csharp
contexto = new Contexto(new EstrategiaConcreta());
contexto.Ejecutar();
```

---

### Visitor

```text
Problema:
    muchas operaciones sobre una jerarquía de objetos.

Solución:
    mover cada operación a un visitor.

Resultado:
    agregás operaciones nuevas sin tocar las clases visitadas.
```

Código mental:

```csharp
elemento.Accept(visitor);
```

---

La diferencia final:

```text
Strategy cambia el “cómo”.
Visitor cambia el “qué operación aplico”.
```

Uno elige comportamiento.  
El otro visita estructuras.

Los dos sirven para evitar que una clase termine siendo un placard lleno de lógica mezclada, que al abrirlo se te cae encima el sistema completo.
