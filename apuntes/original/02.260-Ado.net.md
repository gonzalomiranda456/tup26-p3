## 1. ¿Qué problema resuelve ADO.NET?

Toda aplicación que maneja información necesita resolver una pregunta básica:

> ¿Cómo guardo y recupero datos de manera confiable?

Al principio, la respuesta suele ser un archivo: `.txt`, `.csv`, `.json`, `.xml`. Para muchos casos pequeños alcanza. Pero cuando la aplicación empieza a crecer aparecen problemas:

- **Búsquedas.** Buscar registros por condiciones requiere leer todo el archivo y filtrar en memoria.
- **Actualizaciones parciales.** Modificar un dato puede obligar a reescribir el archivo completo.
- **Concurrencia.** Dos partes del programa pueden intentar escribir al mismo tiempo.
- **Consistencia.** Una operación incompleta puede dejar los datos a medias.
- **Relaciones.** Representar entidades conectadas, como factura y detalle, empieza a ser incómodo.
- **Consultas agregadas.** Contar, sumar, agrupar y calcular saldos se vuelve código manual repetitivo.

Las bases de datos existen para resolver esos problemas. Pero desde C# necesitamos una forma de hablar con ellas. Ahí entra **ADO.NET**.

**ADO.NET** es el conjunto de clases base de .NET para conectarse a fuentes de datos, ejecutar comandos, leer resultados y controlar transacciones.

La idea general es esta:

```text
Aplicación C#
    ↓
ADO.NET
    ↓
Provider concreto
    ↓
SQLite / SQL Server / PostgreSQL / MySQL / Oracle / etc.
```

ADO.NET no es una base de datos. Tampoco es un ORM. Es una **capa de acceso a datos de bajo nivel**.

Su trabajo es dar una estructura común para operaciones como:

```text
abrir conexión
preparar comando
enviar parámetros
ejecutar SQL
leer filas
confirmar o cancelar transacciones
cerrar recursos
```

## 2. La filosofía de diseño de ADO.NET

ADO.NET tiene una filosofía bastante directa:

> Darle al programador control explícito sobre la comunicación con la base de datos.

No intenta ocultar SQL. No intenta convertir automáticamente todas las tablas en objetos. No intenta decidir por nosotros cuándo guardar los cambios.

ADO.NET nos da piezas simples:

- Una **conexión**.
- Un **comando**.
- Una colección de **parámetros**.
- Un **lector** de resultados.
- Una **transacción**.

Con esas piezas podemos construir operaciones CRUD, reportes, consultas agregadas, repositorios, Unit of Work e incluso un mini ORM.

La ventaja es el control. La desventaja es que hay más código manual.

Comparación rápida:

```text
ADO.NET:
    escribo SQL
    ejecuto comandos
    leo columnas
    convierto filas a objetos

Dapper:
    escribo SQL
    la librería mapea filas a objetos

EF Core:
    trabajo con objetos
    el ORM genera SQL y administra cambios
```

ADO.NET está más cerca de la base. Es menos cómodo que un ORM, pero enseña mejor qué está pasando. Es como ver el motor sin la tapa plástica de diseño.

## 3. Providers: ADO.NET no habla solo

ADO.NET define el modelo general, pero cada base necesita una implementación concreta. Esa implementación se llama **provider**.

Ejemplos:

| Base de datos | Provider habitual | Namespace |
|---|---|---|
| SQLite | `Microsoft.Data.Sqlite` | `Microsoft.Data.Sqlite` |
| SQL Server | `Microsoft.Data.SqlClient` | `Microsoft.Data.SqlClient` |
| PostgreSQL | `Npgsql` | `Npgsql` |
| MySQL / MariaDB | `MySqlConnector` | `MySqlConnector` |
| Oracle | `Oracle.ManagedDataAccess` | `Oracle.ManagedDataAccess.Client` |

Cada provider implementa clases concretas:

```text
SQLite:
    SqliteConnection
    SqliteCommand
    SqliteDataReader
    SqliteTransaction

SQL Server:
    SqlConnection
    SqlCommand
    SqlDataReader
    SqlTransaction

PostgreSQL:
    NpgsqlConnection
    NpgsqlCommand
    NpgsqlDataReader
    NpgsqlTransaction
```

Pero todas siguen la misma estructura conceptual.

También existen clases base comunes en `System.Data.Common`:

```csharp
DbConnection
DbCommand
DbDataReader
DbTransaction
DbParameter
```

Eso permite escribir código más abstracto, aunque en los primeros ejemplos conviene usar las clases concretas del provider para no agregar ruido.

## 5. Connection: abrir una comunicación con la base

Una **conexión** representa el vínculo entre nuestro programa y la base de datos.

En SQLite, la base es un archivo:

```csharp
using var connection = new SqliteConnection("Data Source=agenda.db");
connection.Open();
```

El texto `"Data Source=agenda.db"` es el **connection string**. En SQLite indica el archivo de base de datos.

Si el archivo no existe, SQLite lo crea.

Tres ideas importantes:

**Crear el objeto no abre la conexión.**

```csharp
var connection = new SqliteConnection("Data Source=agenda.db");
```

Eso solo crea el objeto C#. Para conectarse realmente hay que llamar:

```csharp
connection.Open();
```

**La conexión debe cerrarse.**

Para eso usamos `using var`:

```csharp
using var connection = new SqliteConnection("Data Source=agenda.db");
connection.Open();
```

Cuando termina el bloque, C# llama a `Dispose()` y la conexión se libera.

**Regla práctica:**

```text
abrir tarde
usar rápido
liberar siempre
```

No conviene mantener una conexión abierta toda la vida de la aplicación salvo que haya una razón concreta.

## 6. Command: enviar una orden SQL

Una vez abierta la conexión, necesitamos enviarle comandos a la base.

Un **command** representa una instrucción SQL lista para ejecutarse.

```csharp
using var connection = new SqliteConnection("Data Source=agenda.db");
connection.Open();

using var command = connection.CreateCommand();
command.CommandText = "CREATE TABLE IF NOT EXISTS Contactos (Id INTEGER PRIMARY KEY, Nombre TEXT NOT NULL);";
command.ExecuteNonQuery();
```

La propiedad central es:

```csharp
command.CommandText
```

Ahí colocamos el SQL.

El comando puede ser:

```sql
CREATE TABLE
INSERT
SELECT
UPDATE
DELETE
DROP TABLE
```

Para cada tipo de comando usamos un método de ejecución distinto. Los tres principales son:

```csharp
ExecuteNonQuery();
ExecuteScalar();
ExecuteReader();
```

Los veremos uno por uno.

## 7. ExecuteNonQuery: ejecutar sin leer filas

`ExecuteNonQuery()` se usa cuando el comando no devuelve una tabla de resultados.

Ejemplos típicos:

```sql
CREATE TABLE
INSERT
UPDATE
DELETE
```

Crear tabla:

```csharp
using var connection = new SqliteConnection("Data Source=agenda.db");
connection.Open();

using var command = connection.CreateCommand();
command.CommandText = """
    CREATE TABLE IF NOT EXISTS Contactos (
        Id INTEGER PRIMARY KEY AUTOINCREMENT,
        Nombre TEXT NOT NULL,
        Email TEXT
    );
""";

command.ExecuteNonQuery();
```

Insertar una fila:

```csharp
using var command = connection.CreateCommand();
command.CommandText = """
    INSERT INTO Contactos (Nombre, Email)
    VALUES ('Ana', 'ana@mail.com');
""";

int affectedRows = command.ExecuteNonQuery();
```

`ExecuteNonQuery()` devuelve la cantidad de filas afectadas. Para `INSERT`, `UPDATE` y `DELETE` suele ser útil.

## 8. Parámetros: nunca concatenar SQL

Una de las reglas más importantes de ADO.NET es esta:

> Si un valor viene de una variable, va como parámetro.

No hacemos esto:

```csharp
// Mal
command.CommandText = $"SELECT * FROM Contactos WHERE Nombre = '{nombre}'";
```

Esto puede romperse con nombres como `O'Brien` y también abre la puerta a inyección SQL.

La forma correcta:

```csharp
using var command = connection.CreateCommand();
command.CommandText = """
    SELECT Id, Nombre, Email
    FROM Contactos
    WHERE Nombre = $nombre
""";

command.Parameters.AddWithValue("$nombre", nombre);
```

El SQL y los valores viajan separados. La base recibe una instrucción parametrizada y el provider se encarga de pasar los datos correctamente.

Insert con parámetros:

```csharp
public int Create(Contacto contacto)
{
    using var connection = new SqliteConnection("Data Source=agenda.db");
    connection.Open();

    using var command = connection.CreateCommand();
    command.CommandText = """
        INSERT INTO Contactos (Nombre, Email)
        VALUES ($nombre, $email);
    """;

    command.Parameters.AddWithValue("$nombre", contacto.Nombre);
    command.Parameters.AddWithValue("$email", contacto.Email ?? (object)DBNull.Value);

    return command.ExecuteNonQuery();
}
```

En SQLite se suelen usar parámetros con `$`, aunque también existen otras formas como `@` y `:`.

## 9. ExecuteScalar: leer un solo valor

`ExecuteScalar()` se usa cuando esperamos una sola celda como resultado: primera fila, primera columna.

Ejemplos:

```sql
SELECT COUNT(*) FROM Contactos;
SELECT MAX(Id) FROM Contactos;
SELECT SUM(TotalCentavos) FROM Ventas;
```

Contar contactos:

```csharp
using var connection = new SqliteConnection("Data Source=agenda.db");
connection.Open();

using var command = connection.CreateCommand();
command.CommandText = "SELECT COUNT(*) FROM Contactos;";

object? result = command.ExecuteScalar();
int count = Convert.ToInt32(result);

Console.WriteLine($"Cantidad: {count}");
```

`ExecuteScalar()` devuelve `object?` porque ADO.NET no sabe de antemano si la consulta devolverá un `int`, `long`, `string`, `decimal`, `DateTime`, etc.

Por eso convertimos explícitamente:

```csharp
int count = Convert.ToInt32(result);
```

Ejemplo con condición:

```csharp
public int CountByMinimumAge(int minimumAge)
{
    using var connection = new SqliteConnection("Data Source=agenda.db");
    connection.Open();

    using var command = connection.CreateCommand();
    command.CommandText = """
        SELECT COUNT(*)
        FROM Contactos
        WHERE Edad >= $minimumAge
    """;

    command.Parameters.AddWithValue("$minimumAge", minimumAge);

    return Convert.ToInt32(command.ExecuteScalar());
}
```

## 10. ExecuteReader: leer muchas filas

Cuando un `SELECT` devuelve varias filas, usamos `ExecuteReader()`.

```csharp
using var command = connection.CreateCommand();
command.CommandText = """
    SELECT Id, Nombre, Email
    FROM Contactos
    ORDER BY Nombre
""";

using var reader = command.ExecuteReader();

while (reader.Read())
{
    int id = reader.GetInt32(0);
    string nombre = reader.GetString(1);
    string? email = reader.IsDBNull(2) ? null : reader.GetString(2);

    Console.WriteLine($"{id} - {nombre} - {email}");
}
```

El `DataReader` funciona como un **cursor**.

Al principio está antes de la primera fila. Cada llamada a `Read()` avanza una posición.

```text
antes de la primera fila
Read() → fila 1
Read() → fila 2
Read() → fila 3
Read() → false
```

Por eso el patrón típico es:

```csharp
while (reader.Read())
{
    // leer la fila actual
}
```

## 11. Cómo acceder a las columnas de una fila

Dentro del `while`, el reader está parado sobre una fila concreta.

Si el SQL es:

```sql
SELECT Id, Nombre, Email
FROM Contactos;
```

Los índices son:

```text
0 → Id
1 → Nombre
2 → Email
```

Entonces podemos leer:

```csharp
int id = reader.GetInt32(0);
string nombre = reader.GetString(1);
string? email = reader.IsDBNull(2) ? null : reader.GetString(2);
```

También podemos obtener el índice por nombre:

```csharp
int idIndex = reader.GetOrdinal("Id");
int nombreIndex = reader.GetOrdinal("Nombre");
int emailIndex = reader.GetOrdinal("Email");

while (reader.Read())
{
    int id = reader.GetInt32(idIndex);
    string nombre = reader.GetString(nombreIndex);
    string? email = reader.IsDBNull(emailIndex)
        ? null
        : reader.GetString(emailIndex);
}
```

Esta forma es más robusta, porque no depende tanto de recordar que `Nombre` está en la posición 1.

También existe el indexador:

```csharp
object value = reader["Nombre"];
```

Pero devuelve `object`, así que suele requerir conversión:

```csharp
string nombre = Convert.ToString(reader["Nombre"])!;
```

## 12. NULL, DBNull y tipos

En SQL existe `NULL`, que significa ausencia de valor.

En ADO.NET, un `NULL` de base de datos aparece como:

```csharp
DBNull.Value
```

Por eso no conviene hacer directamente:

```csharp
string email = reader.GetString(emailIndex);
```

si la columna puede venir con `NULL`.

La forma correcta:

```csharp
string? email = reader.IsDBNull(emailIndex)
    ? null
    : reader.GetString(emailIndex);
```

Cuando enviamos parámetros, también hay que convertir `null` de C# a `DBNull.Value`:

```csharp
command.Parameters.AddWithValue("$email", contacto.Email ?? (object)DBNull.Value);
```

Son dos mundos distintos:

```text
C#      → null
SQL     → NULL
ADO.NET → DBNull.Value
```

## 13. ¿Por qué algunos métodos devuelven object?

Métodos como `ExecuteScalar()`, `GetValue()` o el indexador `reader["Nombre"]` devuelven `object`.

Eso ocurre porque una consulta SQL puede devolver cualquier tipo de valor:

```sql
SELECT COUNT(*) FROM Contactos;       -- número
SELECT Nombre FROM Contactos;         -- texto
SELECT FechaAlta FROM Contactos;      -- fecha como texto o DateTime según provider
SELECT Total FROM Ventas;             -- número decimal, entero, real, etc.
```

ADO.NET necesita un tipo común para todos los posibles valores. En C#, ese tipo común es `object`.

Después nosotros convertimos:

```csharp
int count = Convert.ToInt32(command.ExecuteScalar());
```

O usamos métodos tipados del reader:

```csharp
int id = reader.GetInt32(idIndex);
string nombre = reader.GetString(nombreIndex);
```

También existe una forma genérica por columna:

```csharp
int id = reader.GetFieldValue<int>(idIndex);
string nombre = reader.GetFieldValue<string>(nombreIndex);
```

Pero igual tenemos que saber qué tipo esperamos.

## 14. Convertir filas en objetos

ADO.NET no convierte automáticamente filas en objetos de dominio. Eso lo hacemos nosotros.

Modelo:

```csharp
public class Contacto
{
    public int Id { get; set; }
    public string Nombre { get; set; } = "";
    public string? Email { get; set; }
}
```

Lectura:

```csharp
public List<Contacto> ReadAll()
{
    var contactos = new List<Contacto>();

    using var connection = new SqliteConnection("Data Source=agenda.db");
    connection.Open();

    using var command = connection.CreateCommand();
    command.CommandText = """
        SELECT Id, Nombre, Email
        FROM Contactos
        ORDER BY Nombre
    """;

    using var reader = command.ExecuteReader();

    int idIndex = reader.GetOrdinal("Id");
    int nombreIndex = reader.GetOrdinal("Nombre");
    int emailIndex = reader.GetOrdinal("Email");

    while (reader.Read())
    {
        var contacto = new Contacto
        {
            Id = reader.GetInt32(idIndex),
            Nombre = reader.GetString(nombreIndex),
            Email = reader.IsDBNull(emailIndex)
                ? null
                : reader.GetString(emailIndex)
        };

        contactos.Add(contacto);
    }

    return contactos;
}
```

Este proceso se llama informalmente **mapeo**:

```text
fila SQL → objeto C#
```

ADO.NET se queda en el nivel bajo. Si queremos mapeo automático, usamos librerías como Dapper o EF Core, o escribimos nuestro propio mini mapper con reflection.

## 15. CRUD completo con ADO.NET

Supongamos esta tabla:

```sql
CREATE TABLE IF NOT EXISTS Contactos (
    Id INTEGER PRIMARY KEY AUTOINCREMENT,
    Nombre TEXT NOT NULL,
    Edad INTEGER NOT NULL,
    Email TEXT
);
```

Y esta clase:

```csharp
public class Contacto
{
    public int Id { get; set; }
    public string Nombre { get; set; } = "";
    public int Edad { get; set; }
    public string? Email { get; set; }
}
```

### Crear

```csharp
public int Create(Contacto contacto)
{
    using var connection = new SqliteConnection("Data Source=agenda.db");
    connection.Open();

    using var command = connection.CreateCommand();
    command.CommandText = """
        INSERT INTO Contactos (Nombre, Edad, Email)
        VALUES ($nombre, $edad, $email);

        SELECT last_insert_rowid();
    """;

    command.Parameters.AddWithValue("$nombre", contacto.Nombre);
    command.Parameters.AddWithValue("$edad", contacto.Edad);
    command.Parameters.AddWithValue("$email", contacto.Email ?? (object)DBNull.Value);

    int id = Convert.ToInt32(command.ExecuteScalar());
    contacto.Id = id;

    return id;
}
```

### Leer uno

```csharp
public Contacto? ReadOne(int id)
{
    using var connection = new SqliteConnection("Data Source=agenda.db");
    connection.Open();

    using var command = connection.CreateCommand();
    command.CommandText = """
        SELECT Id, Nombre, Edad, Email
        FROM Contactos
        WHERE Id = $id
    """;

    command.Parameters.AddWithValue("$id", id);

    using var reader = command.ExecuteReader();

    if (!reader.Read())
        return null;

    return new Contacto
    {
        Id = reader.GetInt32(0),
        Nombre = reader.GetString(1),
        Edad = reader.GetInt32(2),
        Email = reader.IsDBNull(3) ? null : reader.GetString(3)
    };
}
```

### Leer todos

```csharp
public List<Contacto> ReadAll()
{
    var contactos = new List<Contacto>();

    using var connection = new SqliteConnection("Data Source=agenda.db");
    connection.Open();

    using var command = connection.CreateCommand();
    command.CommandText = """
        SELECT Id, Nombre, Edad, Email
        FROM Contactos
        ORDER BY Nombre
    """;

    using var reader = command.ExecuteReader();

    while (reader.Read())
    {
        contactos.Add(new Contacto
        {
            Id = reader.GetInt32(0),
            Nombre = reader.GetString(1),
            Edad = reader.GetInt32(2),
            Email = reader.IsDBNull(3) ? null : reader.GetString(3)
        });
    }

    return contactos;
}
```

### Actualizar

```csharp
public bool Update(Contacto contacto)
{
    using var connection = new SqliteConnection("Data Source=agenda.db");
    connection.Open();

    using var command = connection.CreateCommand();
    command.CommandText = """
        UPDATE Contactos
        SET Nombre = $nombre,
            Edad = $edad,
            Email = $email
        WHERE Id = $id
    """;

    command.Parameters.AddWithValue("$id", contacto.Id);
    command.Parameters.AddWithValue("$nombre", contacto.Nombre);
    command.Parameters.AddWithValue("$edad", contacto.Edad);
    command.Parameters.AddWithValue("$email", contacto.Email ?? (object)DBNull.Value);

    int affectedRows = command.ExecuteNonQuery();

    return affectedRows > 0;
}
```

### Eliminar

```csharp
public bool Delete(int id)
{
    using var connection = new SqliteConnection("Data Source=agenda.db");
    connection.Open();

    using var command = connection.CreateCommand();
    command.CommandText = """
        DELETE FROM Contactos
        WHERE Id = $id
    """;

    command.Parameters.AddWithValue("$id", id);

    int affectedRows = command.ExecuteNonQuery();

    return affectedRows > 0;
}
```

Cuidado con `UPDATE` y `DELETE`: si olvidamos el `WHERE`, modificamos o borramos toda la tabla. La base ejecuta lo que le pedimos, no lo que quisimos pedir. Muy obediente, peligrosamente obediente.

## 16. Consultas agregadas: contar, sumar, agrupar

No siempre queremos traer filas. Muchas veces queremos pedirle a la base que calcule.

### Contar

```sql
SELECT COUNT(*)
FROM Contactos
WHERE Edad >= $minimumAge;
```

```csharp
public int CountAdults(int minimumAge)
{
    using var connection = new SqliteConnection("Data Source=agenda.db");
    connection.Open();

    using var command = connection.CreateCommand();
    command.CommandText = """
        SELECT COUNT(*)
        FROM Contactos
        WHERE Edad >= $minimumAge
    """;

    command.Parameters.AddWithValue("$minimumAge", minimumAge);

    return Convert.ToInt32(command.ExecuteScalar());
}
```

### Calcular saldo con debe y haber

```sql
SELECT COALESCE(SUM(DebeCentavos), 0) - COALESCE(SUM(HaberCentavos), 0)
FROM Movimientos
WHERE CuentaId = $cuentaId;
```

```csharp
public long GetBalanceInCents(int cuentaId)
{
    using var connection = new SqliteConnection("Data Source=contabilidad.db");
    connection.Open();

    using var command = connection.CreateCommand();
    command.CommandText = """
        SELECT COALESCE(SUM(DebeCentavos), 0) - COALESCE(SUM(HaberCentavos), 0)
        FROM Movimientos
        WHERE CuentaId = $cuentaId
    """;

    command.Parameters.AddWithValue("$cuentaId", cuentaId);

    return Convert.ToInt64(command.ExecuteScalar());
}
```

`COALESCE` evita que `SUM` devuelva `NULL` cuando no hay filas.

### Ventas mensuales

```sql
SELECT
    strftime('%Y-%m', Fecha) AS Mes,
    SUM(TotalCentavos) AS TotalCentavos
FROM Ventas
GROUP BY strftime('%Y-%m', Fecha)
ORDER BY Mes;
```

Como devuelve varias filas, usamos `ExecuteReader()`:

```csharp
public List<VentaMensual> GetMonthlySales()
{
    var result = new List<VentaMensual>();

    using var connection = new SqliteConnection("Data Source=ventas.db");
    connection.Open();

    using var command = connection.CreateCommand();
    command.CommandText = """
        SELECT
            strftime('%Y-%m', Fecha) AS Mes,
            SUM(TotalCentavos) AS TotalCentavos
        FROM Ventas
        GROUP BY strftime('%Y-%m', Fecha)
        ORDER BY Mes
    """;

    using var reader = command.ExecuteReader();

    int mesIndex = reader.GetOrdinal("Mes");
    int totalIndex = reader.GetOrdinal("TotalCentavos");

    while (reader.Read())
    {
        result.Add(new VentaMensual
        {
            Mes = reader.GetString(mesIndex),
            TotalCentavos = reader.GetInt64(totalIndex)
        });
    }

    return result;
}
```

```csharp
public class VentaMensual
{
    public string Mes { get; set; } = "";
    public long TotalCentavos { get; set; }
    public decimal Total => TotalCentavos / 100m;
}
```

## 17. Relaciones 1 a N: factura y detalle

Una relación 1 a N significa:

```text
Una factura tiene muchos detalles.
Un detalle pertenece a una sola factura.
```

En SQL se modela con una clave foránea:

```sql
CREATE TABLE IF NOT EXISTS Facturas (
    Id INTEGER PRIMARY KEY AUTOINCREMENT,
    Fecha TEXT NOT NULL,
    Cliente TEXT NOT NULL
);

CREATE TABLE IF NOT EXISTS DetallesFactura (
    Id INTEGER PRIMARY KEY AUTOINCREMENT,
    FacturaId INTEGER NOT NULL,
    Producto TEXT NOT NULL,
    Cantidad INTEGER NOT NULL,
    PrecioUnitarioCentavos INTEGER NOT NULL,

    FOREIGN KEY (FacturaId)
        REFERENCES Facturas(Id)
        ON DELETE CASCADE
);
```

En C# se modela con una lista:

```csharp
public class Factura
{
    public int Id { get; set; }
    public DateOnly Fecha { get; set; }
    public string Cliente { get; set; } = "";
    public List<DetalleFactura> Detalles { get; set; } = [];
}
```

```csharp
public class DetalleFactura
{
    public int Id { get; set; }
    public int FacturaId { get; set; }
    public string Producto { get; set; } = "";
    public int Cantidad { get; set; }
    public long PrecioUnitarioCentavos { get; set; }
}
```

La relación aparece en dos mundos:

```text
Base de datos:
    DetallesFactura.FacturaId → Facturas.Id

C#:
    Factura.Detalles → List<DetalleFactura>
```

ADO.NET no reconstruye esta relación automáticamente. Hay que hacer consultas y mapear.

## 18. Transacciones: todo o nada

Cuando una operación involucra varias tablas, necesitamos una transacción.

Ejemplo: crear una factura con detalles.

```csharp
public int CreateInvoice(Factura factura)
{
    using var connection = new SqliteConnection("Data Source=ventas.db");
    connection.Open();

    using var pragma = connection.CreateCommand();
    pragma.CommandText = "PRAGMA foreign_keys = ON;";
    pragma.ExecuteNonQuery();

    using var transaction = connection.BeginTransaction();

    try
    {
        using var command = connection.CreateCommand();
        command.Transaction = transaction;

        command.CommandText = """
            INSERT INTO Facturas (Fecha, Cliente)
            VALUES ($fecha, $cliente);

            SELECT last_insert_rowid();
        """;

        command.Parameters.AddWithValue("$fecha", factura.Fecha.ToString("yyyy-MM-dd"));
        command.Parameters.AddWithValue("$cliente", factura.Cliente);

        int facturaId = Convert.ToInt32(command.ExecuteScalar());
        factura.Id = facturaId;

        foreach (var detalle in factura.Detalles)
        {
            command.Parameters.Clear();

            command.CommandText = """
                INSERT INTO DetallesFactura
                    (FacturaId, Producto, Cantidad, PrecioUnitarioCentavos)
                VALUES
                    ($facturaId, $producto, $cantidad, $precioUnitarioCentavos)
            """;

            command.Parameters.AddWithValue("$facturaId", facturaId);
            command.Parameters.AddWithValue("$producto", detalle.Producto);
            command.Parameters.AddWithValue("$cantidad", detalle.Cantidad);
            command.Parameters.AddWithValue("$precioUnitarioCentavos", detalle.PrecioUnitarioCentavos);

            command.ExecuteNonQuery();
        }

        transaction.Commit();

        return facturaId;
    }
    catch
    {
        transaction.Rollback();
        throw;
    }
}
```

La idea:

```text
BEGIN TRANSACTION
    INSERT Factura
    INSERT Detalle 1
    INSERT Detalle 2
    INSERT Detalle 3
COMMIT
```

Si algo falla:

```text
ROLLBACK
```

La transacción garantiza atomicidad: se guarda todo o no se guarda nada.

## 19. Repository Pattern sobre ADO.NET

ADO.NET es explícito, pero repetir conexión, comando, reader y mapeo en todos lados vuelve el código pesado.

Por eso solemos encapsularlo en un **repositorio**.

Un repositorio representa una colección persistente de objetos del dominio.

```csharp
public interface IRepository<T>
{
    List<T> ReadAll();
    T? ReadOne(int id);
    int Create(T entity);
    bool Update(T entity);
    bool Delete(int id);
}
```

Desde la lógica de negocio usamos:

```csharp
var contacto = contactos.ReadOne(10);

if (contacto is not null)
{
    contacto.Email = "nuevo@mail.com";
    contactos.Update(contacto);
}
```

La lógica no necesita saber de:

```text
SqliteConnection
SqliteCommand
SqliteDataReader
DBNull
SQL parametrizado
```

Eso queda dentro del repositorio.

La frontera queda así:

```text
Caso de uso / servicio
    ↓
Repositorio
    ↓
ADO.NET
    ↓
SQLite
```

## 20. Unit of Work

El patrón **Unit of Work** coordina varios repositorios para que trabajen bajo una misma conexión y transacción.

El problema aparece cuando una operación de negocio toca varias entidades:

```text
crear factura
crear detalles
descontar stock
registrar movimiento contable
```

No queremos que cada repositorio confirme por separado. Queremos una única operación atómica.

Uso ideal:

```csharp
using var unitOfWork = new UnitOfWork("Data Source=ventas.db");

try
{
    unitOfWork.Facturas.Create(factura);
    unitOfWork.Detalles.Create(detalle);
    unitOfWork.Stock.Descontar(productoId, cantidad);

    unitOfWork.Commit();
}
catch
{
    unitOfWork.Rollback();
    throw;
}
```

Conceptualmente:

```text
Repository:
    sabe cómo guardar una entidad

Unit of Work:
    decide cuándo confirmar todos los cambios juntos
```

En EF Core, `DbContext` cumple un rol muy parecido a Unit of Work, y `DbSet<T>` se parece a un repositorio.

## 21. ADO.NET, Dapper y EF Core

Conviene ubicar cada herramienta:

| Herramienta | Nivel | Qué hace |
|---|---|---|
| ADO.NET | Bajo | Conexión, comandos, parámetros, readers, transacciones |
| Dapper | Medio | Usa SQL escrito por nosotros y mapea filas a objetos |
| EF Core | Alto | ORM completo: LINQ, tracking, relaciones, migraciones |

Con ADO.NET:

```csharp
using var reader = command.ExecuteReader();
while (reader.Read())
{
    // mapear manualmente
}
```

Con Dapper:

```csharp
var contactos = connection.Query<Contacto>(
    "SELECT Id, Nombre, Email FROM Contactos"
);
```

Con EF Core:

```csharp
var contactos = db.Contactos
    .Where(c => c.Edad > 30)
    .ToList();
```

ADO.NET es la base. Dapper y EF Core están construidos sobre ideas que ADO.NET ya expone: conexión, comandos, parámetros, lectores y transacciones.

## 23. Cierre

ADO.NET resuelve el problema fundamental de conectar C# con una base de datos de manera explícita y controlada.

Lo esencial que hay que dominar:

- Abrir y liberar conexiones con `using`.
- Ejecutar SQL con `Command`.
- Pasar valores con parámetros, nunca concatenando.
- Usar `ExecuteNonQuery()` para comandos sin filas.
- Usar `ExecuteScalar()` para un único valor.
- Usar `ExecuteReader()` para leer filas.
- Mapear manualmente columnas a objetos.
- Controlar `NULL` con `IsDBNull()` y `DBNull.Value`.
- Usar transacciones cuando varias operaciones deben confirmarse juntas.
- Encapsular acceso a datos en repositorios cuando el código empieza a repetirse.

La frase final:

> ADO.NET es la capa donde C# habla con la base sin intermediarios elegantes.

No es la herramienta más cómoda, pero sí una de las más importantes para entender qué hacen por debajo Dapper, EF Core y cualquier ORM serio.

Aprender ADO.NET primero evita creer en magia. Y en programación, como en electricidad, la magia suele ser un corto mal explicado.
