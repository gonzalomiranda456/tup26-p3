# Construir un mini CRM TUI en C#

---

## 1. ¿Qué app estamos construyendo?

La aplicación es un **mini CRM** (*Customer Relationship Management*). En vez de pensarla como “una lista de contactos con esteroides”, conviene verla como una herramienta para responder tres preguntas:

1. **¿Con quién estamos hablando?** → contactos.
2. **¿Qué negocio posible tenemos con esa persona o empresa?** → oportunidades.
3. **¿Qué seguimiento hay que hacer y cuándo?** → actividades.

La interfaz corre íntegramente en terminal, pero no es una CLI tradicional de “un comando y listo”. Es una **TUI** (*Terminal User Interface*): ocupa toda la pantalla, muestra paneles, listas, formularios, menús y diálogos.

En términos de usuario, la aplicación permite:

- buscar contactos,
- crear, editar y eliminar contactos,
- ver el detalle del contacto seleccionado,
- asociar oportunidades a un contacto,
- asociar actividades a un contacto,
- marcar actividades como realizadas,
- persistir todo en una base SQLite local.

Es una app pequeña, pero muy buena como ejercicio porque combina varias ideas importantes a la vez:

- interfaz interactiva,
- diseño de dominio,
- persistencia,
- separación por capas,
- manejo de eventos,
- validaciones,
- navegación por teclado.

## 2. Funcionalidad requerida

Antes de programar, conviene escribir qué debería poder hacer la app. Esto evita el clásico problema de empezar por la UI y descubrir después que faltaba media lógica.

### Requerimientos funcionales mínimos

#### Gestión de contactos

- Alta de contacto.
- Edición de contacto existente.
- Baja de contacto.
- Visualización de datos del contacto seleccionado.
- Búsqueda por nombre, empresa, email, teléfono o estado.

#### Gestión de oportunidades

- Cada oportunidad pertenece a un contacto.
- Se puede crear una oportunidad para el contacto actual.
- Se puede editar una oportunidad existente.
- Se puede eliminar una oportunidad.
- Cada oportunidad tiene al menos:
  - título,
  - etapa,
  - valor,
  - fecha estimada de cierre,
  - notas.

#### Gestión de actividades

- Cada actividad pertenece a un contacto.
- Se puede crear una actividad.
- Se puede eliminar una actividad.
- Se puede marcar una actividad como hecha.
- Cada actividad tiene al menos:
  - tipo,
  - asunto,
  - fecha,
  - estado de realización,
  - notas.

#### Persistencia

- Los datos deben sobrevivir al cierre del programa.
- La base debe crearse automáticamente si no existe.
- La app debe poder arrancar con datos de ejemplo si la base está vacía.

### Requerimientos no funcionales

- Debe poder usarse por teclado.
- Debe funcionar en macOS, Linux y Windows.
- Debe ser simple de ejecutar con `dotnet run crm.cs`.
- Debe mantener una estructura comprensible aunque esté implementada en un archivo único.

## 3. La idea central del diseño

Si uno mira la app terminada, puede parecer que “todo es UI”. Pero en realidad hay **cuatro capas conceptuales** distintas. Aunque el archivo esté unificado, mentalmente conviene separarlas.

```text
Presentación (TUI)
    ↓
Aplicación / Orquestación
    ↓
Persistencia / Acceso a datos
    ↓
Dominio
```

Ese orden importa.

- La **presentación** muestra cosas y captura acciones del usuario.
- La **aplicación** decide qué hacer cuando el usuario pide algo.
- La **persistencia** sabe guardar y recuperar datos.
- El **dominio** define qué cosas existen en el problema.

Si mezclamos todo, terminamos con botones que ejecutan SQL, consultas que formatean texto para la UI, y modelos de datos que conocen demasiado de la pantalla. Un festival del acoplamiento; muy entretenido para sufrir, poco para mantener.

## 4. Separando claramente las capas

### 4.1. Capa de dominio

El dominio responde: **¿qué entidades existen en el CRM?**

En esta app hay tres principales:

- `Contact`
- `Opportunity`
- `Activity`

Estas clases representan los datos del negocio.

#### `Contact`

Un contacto tiene identidad y datos de relación:

- `Id`
- `Name`
- `Company`
- `Email`
- `Phone`
- `Status`
- `Notes`
- `CreatedAt`

#### `Opportunity`

Una oportunidad representa una venta potencial asociada a un contacto:

- `Id`
- `ContactId`
- `Title`
- `Stage`
- `Value`
- `ExpectedCloseDate`
- `Notes`
- `CreatedAt`

#### `Activity`

Una actividad representa una acción de seguimiento:

- `Id`
- `ContactId`
- `Type`
- `Subject`
- `DueDate`
- `IsDone`
- `Notes`
- `CreatedAt`

### Decisión de diseño en esta capa

Las entidades son **simples**, casi anémicas. No cargan comportamiento complejo; funcionan como modelos de datos.

¿Por qué está bien en este ejercicio?

Porque el foco pedagógico acá no está en DDD avanzado ni en invariantes sofisticadas, sino en:

- organizar una app interactiva,
- mover datos entre UI y base,
- mantener separación suficiente entre responsabilidades.

Más adelante podría evolucionarse hacia entidades con más comportamiento, pero para esta app una representación simple es razonable.

---

### 4.2. Capa de persistencia

La persistencia responde: **¿dónde se guardan los datos y cómo se recuperan?**

En esta app esa responsabilidad vive en `CrmDatabase`.

Sus tareas principales son:

- abrir conexiones SQLite,
- crear el esquema si no existe,
- insertar datos semilla,
- buscar contactos,
- insertar, actualizar y eliminar contactos,
- insertar, actualizar y eliminar oportunidades,
- insertar, completar y eliminar actividades.

### ¿Por qué SQLite?

Porque ofrece varias ventajas didácticas y prácticas:

- no requiere servidor externo,
- guarda todo en un archivo,
- es ideal para demos, ejercicios y herramientas pequeñas,
- permite introducir SQL real sin complejidad operativa extra.

### ¿Por qué Dapper?

Porque resuelve el mapping entre filas SQL y objetos C# sin meter un ORM completo.

Eso lo hace perfecto para enseñar el punto medio entre dos extremos:

- **ADO.NET crudo**, muy explícito pero verboso,
- **EF Core**, muy potente pero con mayor abstracción.

Dapper deja ver claramente el SQL, pero reduce el ruido de lectura y materialización.

### Decisiones de diseño en persistencia

#### a. Una clase única de acceso a datos

Para una app chica, centralizar la persistencia en `CrmDatabase` simplifica el recorrido del código. No hace falta introducir repositorios separados si eso solo agrega capas ceremoniales.

#### b. Relaciones explícitas con claves foráneas

`Opportunity` y `Activity` dependen de `Contact` mediante `ContactId`. Esto expresa correctamente la relación del problema y permite que SQLite haga parte del trabajo de integridad.

#### c. `ON DELETE CASCADE`

Cuando se elimina un contacto, también se eliminan sus oportunidades y actividades.

Esto tiene dos ventajas:

- evita datos huérfanos,
- simplifica la lógica de borrado.

Es una decisión de negocio razonable: si el contacto desaparece del CRM, también desaparece su historial asociado dentro de esta versión de la aplicación.

---

### 4.3. Capa de aplicación u orquestación

Esta capa responde: **¿cómo se coordinan las acciones del usuario con la persistencia y la UI?**

Acá el actor principal es `CrmWindow`.

`CrmWindow` no es solo una pantalla: también es el **coordinador** de la experiencia.

Hace cosas como:

- leer el texto de búsqueda,
- pedir contactos a la base,
- refrescar las listas,
- decidir cuál es el contacto seleccionado,
- abrir diálogos de alta/edición,
- validar que haya un contacto antes de crear oportunidades,
- actualizar la barra de estado.

### Esta capa no debería hacer

- SQL directo embebido en eventos de botones,
- lógica de dibujado dentro de `CrmDatabase`,
- conocimiento de detalles de layout dentro de las entidades.

Es, en otras palabras, la capa que **conecta** todo sin convertirse en todo.

### Decisión importante: método de refresco centralizado

La app usa métodos como:

- `RefreshContacts()`
- `RefreshSelectedContactArea()`
- `RefreshOpportunities()`
- `RefreshActivities()`

Esto es una buena decisión de diseño porque evita duplicar lógica después de cada operación.

En vez de repartir `SetSource(...)` y reasignaciones por veinte lugares distintos, se concentra la actualización de cada sector en métodos específicos.

---

### 4.4. Capa de presentación

La presentación responde: **¿qué ve el usuario y cómo interactúa con la app?**

Acá entran:

- `MenuBar`
- `FrameView`
- `ListView`
- `Label`
- `TextField`
- `Dialog`
- `Button`
- `MessageBox`

La ventana principal está organizada en zonas:

- arriba: menú y búsqueda,
- izquierda: listado de contactos,
- derecha superior: detalle del contacto,
- derecha media: oportunidades,
- derecha inferior: actividades,
- abajo: barra de estado.

Esta distribución refleja bastante bien el flujo mental del usuario:

1. busca o selecciona un contacto,
2. mira el detalle,
3. trabaja sobre lo relacionado con ese contacto.

### Decisiones de diseño en la UI

#### a. La lista manda

El contacto seleccionado gobierna el resto de la pantalla. Es decir:

- el detalle mostrado depende del contacto actual,
- las oportunidades mostradas dependen del contacto actual,
- las actividades mostradas dependen del contacto actual.

Eso evita pantallas separadas y mantiene el contexto visible.

#### b. Diálogos modales para edición

Las operaciones de alta/edición se hacen con `Dialog`.

¿Por qué es una buena elección?

- aíslan la tarea actual,
- simplifican la validación,
- evitan que el usuario edite varias cosas a la vez sin contexto,
- siguen una convención fuerte en TUIs.

#### c. Botones visibles + menú principal

La app ofrece acciones de dos formas:

- desde el menú,
- desde botones visibles junto a cada panel.

Esto mejora descubribilidad. Un usuario nuevo puede “ver” qué se puede hacer, mientras que uno más experto puede apoyarse en teclado y menú.

## 5. Recorrido progresivo del desarrollo

La mejor forma de construir esta app no es abrir Terminal.Gui y empezar a tirar `Button`s por la pantalla como confeti binario. Conviene avanzar por etapas.

---

## 6. Paso 1: modelar el dominio

Primero se definen las entidades.

La pregunta guía es:

> ¿Qué datos necesito representar para que el CRM tenga sentido?

La respuesta mínima es:

- contactos,
- oportunidades,
- actividades.

En esta etapa todavía no hace falta pensar en la UI. Solo importa que el problema esté bien descrito.

### Resultado esperado del paso

- clases del dominio creadas,
- propiedades claras,
- relaciones identificadas (`ContactId` en entidades hijas),
- valores por defecto razonables en algunos campos.

### Qué se gana

- una base semántica clara,
- nombres consistentes para el resto del desarrollo,
- menor probabilidad de rediseñar la base a mitad de camino.

---

## 7. Paso 2: resolver persistencia primero

Una vez que sabemos qué existe en el dominio, resolvemos cómo guardarlo.

En esta etapa se crea:

- el archivo SQLite,
- las tablas,
- los índices,
- las operaciones CRUD,
- los datos semilla.

### ¿Por qué conviene hacerlo temprano?

Porque gran parte de la app depende de que los datos existan y puedan leerse.

Si primero armamos una UI preciosa pero no tenemos operaciones de persistencia claras, después hay que retocar media interfaz para adaptarla a cómo realmente quedan los datos.

### Resultado esperado del paso

- `Initialize()` crea el esquema,
- `SeedIfEmpty()` deja datos de ejemplo,
- `SearchContacts()` devuelve contactos,
- métodos de alta/modificación/baja funcionan.

### Qué se está construyendo en realidad

Aunque parezca que solo estamos “metiendo SQL”, en realidad estamos cerrando el contrato entre aplicación y almacenamiento.

---

## 8. Paso 3: armar la ventana principal

Recién ahora conviene entrar fuerte en la UI.

La ventana principal debe responder a una pregunta de diseño importante:

> ¿Qué información necesito mostrar simultáneamente para que el CRM sea útil?

La respuesta de esta app es:

- lista de contactos,
- detalle del contacto,
- oportunidades,
- actividades.

Eso determina el layout general.

### Resultado esperado del paso

- menú superior,
- campo de búsqueda,
- paneles principales con `FrameView`,
- listas iniciales vacías o con datos,
- barra de estado.

### Decisión clave

No hacer muchas pantallas separadas al principio. Para este problema, una **pantalla maestra con detalle contextual** es más simple que navegar entre subpantallas.

---

## 9. Paso 4: conectar selección y refresco

Acá la app deja de ser una maqueta y empieza a comportarse como un sistema.

La idea es simple:

- si cambia el contacto seleccionado,
- cambia el detalle mostrado,
- cambian las oportunidades mostradas,
- cambian las actividades mostradas.

### Resultado esperado del paso

- `contactsList.ValueChanged` actualiza la selección,
- la selección dispara un refresco del área derecha,
- las listas dependientes quedan sincronizadas.

### Por qué este paso es importante

Porque instala el **flujo maestro-detalle**, que es el corazón de la experiencia.

Sin eso, el usuario ve datos desconectados. Con eso, la UI empieza a contar una historia coherente.

---

## 10. Paso 5: agregar formularios modales

Una app de gestión no sirve si solo muestra datos: tiene que poder modificarlos.

Acá aparecen:

- `ContactDialog`
- `OpportunityDialog`
- `ActivityDialog`

Cada uno encapsula:

- campos de entrada,
- validaciones,
- armado del objeto resultante,
- confirmación o cancelación.

### Resultado esperado del paso

- el usuario puede crear y editar datos,
- los diálogos validan campos obligatorios,
- al cerrar, la ventana principal se refresca.

### Decisión de diseño relevante

Usar `Dialog` como clase base fue una buena decisión porque:

- expresa mejor la intención modal,
- reduce trabajo visual extra,
- sigue el patrón idiomático de `Terminal.Gui`.

---

## 11. Paso 6: agregar acciones de negocio derivadas

No todo en la app es CRUD puro.

Hay acciones con significado más específico, por ejemplo:

- marcar una actividad como hecha.

Eso no es simplemente “editar una fila”, sino una operación de negocio puntual.

### Resultado esperado del paso

- la operación tiene su propio método (`CompleteActivity`),
- la UI refleja el nuevo estado,
- la actividad reaparece en el listado con marca visual.

### Lección importante

No todas las acciones del sistema merecen tratarse como “editar registro genérico”. A veces conviene darles una operación específica porque eso comunica mejor la intención.

---

## 12. Paso 7: incorporar búsqueda

La búsqueda mejora muchísimo la usabilidad. En una lista chica parece un lujo; en una lista real, es imprescindible.

### Qué debería hacer bien

- responder rápido,
- filtrar por varios campos,
- ser tolerante al modo en que escribe el usuario.

En una evolución natural de esta app, la búsqueda debería ser:

- insensible a mayúsculas/minúsculas,
- idealmente también insensible a acentos.

### Ubicación conceptual de esta funcionalidad

La búsqueda nace en la capa de presentación (el usuario escribe), pero se resuelve en la capa de aplicación/persistencia.

Eso es importante: el `TextField` no “busca”; solo captura texto. La lógica real de filtrado vive más abajo.

---

## 13. Un paso del desarrollo explicado en detalle: selección de contacto y refresco contextual

De todos los pasos, este es uno de los más importantes para entender cómo una TUI deja de ser un formulario estático y se vuelve una aplicación interactiva real.

La idea es:

1. el usuario mueve la selección en `contactsList`,
2. cambia `selectedContact`,
3. se actualiza el detalle,
4. se recargan oportunidades y actividades del contacto nuevo.

### ¿Por qué este paso merece atención?

Porque une de manera muy concreta las cuatro capas:

- **presentación**: la lista detecta el cambio,
- **aplicación**: se decide qué contacto queda activo,
- **persistencia**: se consultan oportunidades y actividades asociadas,
- **dominio**: todo gira alrededor de la identidad del contacto.

### Flujo conceptual

```text
ListView cambia selección
    ↓
CrmWindow.SelectContact(index)
    ↓
selectedContact = contacts[index]
    ↓
RefreshSelectedContactArea()
    ↓
RefreshOpportunities() + RefreshActivities()
    ↓
La UI redibuja el estado nuevo
```

### ¿Qué problema resuelve este diseño?

Evita que cada componente tenga que “adivinar” qué mostrar.

- El panel de detalle no sale a buscar por su cuenta.
- La lista de oportunidades no mantiene una selección independiente del contexto principal.
- La lista de actividades no necesita saber quién cambió qué.

Todo eso se coordina desde un único punto de verdad: `selectedContact`.

### Qué pasaría si no lo hiciéramos así

Aparecerían problemas típicos:

- oportunidades que quedan mostrando datos del contacto anterior,
- detalle desincronizado con la fila resaltada,
- lógica duplicada en varios eventos,
- mayor dificultad para depurar.

### Lección pedagógica del paso

Una buena UI no consiste solo en dibujar controles; consiste en diseñar bien el **flujo del estado**.

En esta app, el estado principal es:

- lista actual de contactos,
- índice seleccionado,
- contacto seleccionado.

Una vez que ese estado está bien definido, el resto se puede derivar con bastante limpieza.

## 14. Decisiones de diseño que valen la pena remarcar

### a. Archivo único, capas conceptuales separadas

Aunque la implementación esté en un solo archivo, el diseño sigue siendo separable mentalmente. Esto es importante para ejercicios docentes: no hace falta una solución industrial de veinte carpetas para enseñar buen criterio.

### b. Datos semilla

Sembrar datos al inicio mejora mucho la experiencia de prueba. Permite ejecutar la app y ver comportamiento real de inmediato.

### c. Layout con paneles estables

En vez de cambiar de pantalla para todo, la app mantiene contexto visible. Eso reduce carga cognitiva.

### d. SQL explícito

Es ideal para fines didácticos: se ve con claridad qué tablas existen y qué operaciones se realizan.

### e. Validaciones en los diálogos

La validación cerca del punto de edición hace que los errores sean más comprensibles para el usuario.

---

## 15. Cómo seguir evolucionando esta app

Una vez funcionando, hay varias mejoras razonables:

- búsqueda insensible a acentos y mayúsculas,
- encabezados visuales para listas con columnas alineadas,
- shortcuts de teclado más completos,
- barra de estado contextual con ayuda,
- separación física por archivos y carpetas,
- incorporación de servicios de aplicación intermedios,
- tests sobre persistencia y lógica de formateo.

Si el objetivo es didáctico, conviene introducir estas mejoras **de a una**. El orden importa: primero estructura, después ergonomía, después refinamiento.

---

## 16. Resumen final

Este mini CRM es un muy buen ejercicio porque obliga a pensar la aplicación completa, no solo una parte aislada.

En una sola app aparecen:

- modelado de dominio,
- SQL y persistencia,
- eventos y UI reactiva,
- formularios modales,
- coordinación de estado,
- decisiones de diseño con impacto real en la experiencia.

La enseñanza principal no es “cómo usar tal componente de Terminal.Gui”, sino algo más valioso:

> una aplicación crece mejor cuando se construye **en forma progresiva**, con **capas claramente diferenciadas** y con una idea clara de cuál es el estado central del sistema.

Dicho menos elegante pero muy sinceramente: si primero pensás el problema, después la estructura, y recién después los botoncitos, la app te lo agradece.
