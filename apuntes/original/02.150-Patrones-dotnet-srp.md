# Guía práctica de patrones en .NET para aplicar SRP

## Introducción

Toda aplicación útil termina creciendo: lee archivos, valida datos, persiste en una base, llama APIs externas, muestra información, registra logs y maneja errores. Al principio parece cómodo poner todo junto. Después una misma clase empieza a cambiar por demasiados motivos.

El **Single Responsibility Principle** dice:

> Una clase debería tener una sola razón para cambiar.

SRP no significa "una clase, un método". Significa que la responsabilidad conceptual debe ser clara. Una clase puede tener varios métodos si todos pertenecen al mismo trabajo.

```text
Mal criterio:
    "Tiene muchos métodos, entonces viola SRP."

Mejor criterio:
    "Cambia por motivos distintos, entonces mezcla responsabilidades."
```

Ejemplo mezclado:

```csharp
public sealed class AgendaService {
    private readonly string path = "contactos.json";

    public void Agregar(Contacto contacto) {
        List<Contacto> contactos;

        if (File.Exists(path)) {
            string json = File.ReadAllText(path);
            contactos = JsonSerializer.Deserialize<List<Contacto>>(json) ?? [];
        }
        else {
            contactos = [];
        }

        if (string.IsNullOrWhiteSpace(contacto.Nombre))
            throw new InvalidOperationException("El nombre es obligatorio.");

        contactos.Add(contacto);

        string nuevoJson = JsonSerializer.Serialize(contactos, new JsonSerializerOptions {
            WriteIndented = true
        });

        File.WriteAllText(path, nuevoJson);
    }
}
```

Responsabilidades mezcladas:

```text
AgendaService
    ├─► sabe dónde está el archivo
    ├─► sabe leer archivos
    ├─► sabe parsear JSON
    ├─► sabe validar contactos
    ├─► sabe agregar contactos
    └─► sabe guardar archivos
```

Versión separada:

```csharp
public sealed class AgendaService(
    IContactoRepository repository,
    IContactoValidator validator) {

    public void Agregar(Contacto contacto) {
        validator.Validar(contacto);

        List<Contacto> contactos = repository.ObtenerTodos();
        contactos.Add(contacto);
        repository.GuardarTodos(contactos);
    }
}
```

Ahora el servicio expresa el caso de uso. No sabe si los datos vienen de JSON, SQLite, SQL Server, memoria o una API.

Los patrones ayudan a hacer esta separación con formas conocidas. **SRP detecta el problema; el patrón ayuda a ubicar la responsabilidad que sobra.**

## 1. Mapa rápido de patrones por responsabilidad

| Si cambia...                                      | Patrón útil                         |
|--------------------------------------------------|-------------------------------------|
| La creación o selección de objetos               | Dependency Injection, Factory       |
| La construcción compleja de un objeto            | Builder                             |
| La forma de guardar o buscar datos               | Repository, Unit of Work            |
| Un criterio de búsqueda combinable               | Specification                       |
| Un algoritmo o política de negocio               | Strategy                            |
| Un algoritmo con estructura fija y pasos variables | Template Method                   |
| El comportamiento según el estado interno        | State                               |
| Una acción que debe ejecutarse, encolarse o deshacerse | Command                        |
| Una cadena de validaciones o filtros             | Chain of Responsibility             |
| Una operación sobre una jerarquía de tipos       | Visitor                             |
| Una API externa                                  | Adapter                             |
| Logging, cache, retry, métricas o autorización   | Decorator, Proxy                    |
| La coordinación entre varios objetos             | Mediator, Facade                    |
| Reacciones a eventos                             | Observer / Events                   |
| Configuración externa                            | Options Pattern                     |
| Responsabilidades HTTP transversales             | Middleware, Filters                 |
| Contrato externo vs modelo interno               | DTO + Mapper                        |
| UI, estado y acciones                            | MVVM                                |

Regla práctica: primero se detecta la razón de cambio; después se elige el patrón. No todo refactor necesita un nombre de catálogo.

## 2. Dependency Injection - separar uso de creación

Sin DI, una clase crea sus dependencias:

```csharp
public sealed class ContactoService {
    private readonly JsonContactoRepository repository = new("contactos.json");
}
```

El servicio queda acoplado a JSON. Con DI, recibe una abstracción:

```csharp
public sealed class ContactoService(IContactoRepository repository) {
    public void Agregar(Contacto contacto) {
        List<Contacto> contactos = repository.ObtenerTodos();
        contactos.Add(contacto);
        repository.GuardarTodos(contactos);
    }
}
```

Registro típico en .NET:

```csharp
builder.Services.AddSingleton<IContactoRepository>(
    new JsonContactoRepository("contactos.json"));

builder.Services.AddScoped<ContactoService>();
```

SRP aplicado:

```text
ContactoService        → ejecuta el caso de uso
JsonContactoRepository → persiste en JSON
DI Container           → construye y conecta objetos
```

## 3. Repository y Unit of Work

### 3.1 Repository - separar dominio de persistencia

Repository oculta el mecanismo concreto de almacenamiento.

```csharp
public interface IContactoRepository {
    List<Contacto> ObtenerTodos();
    void GuardarTodos(List<Contacto> contactos);
}
```

Implementación:

```csharp
public sealed class JsonContactoRepository(string path) : IContactoRepository {
    public List<Contacto> ObtenerTodos() {
        if (!File.Exists(path))
            return [];

        string json = File.ReadAllText(path);
        return JsonSerializer.Deserialize<List<Contacto>>(json) ?? [];
    }

    public void GuardarTodos(List<Contacto> contactos) {
        string json = JsonSerializer.Serialize(contactos, new JsonSerializerOptions {
            WriteIndented = true
        });

        File.WriteAllText(path, json);
    }
}
```

Implementación para tests:

```csharp
public sealed class InMemoryContactoRepository : IContactoRepository {
    private readonly List<Contacto> data = [];

    public List<Contacto> ObtenerTodos() => data.ToList();

    public void GuardarTodos(List<Contacto> contactos) {
        data.Clear();
        data.AddRange(contactos);
    }
}
```

SRP aplicado:

```text
Servicio               → reglas del caso de uso
JsonContactoRepository → persistencia en archivo
InMemoryRepository     → persistencia en memoria
```

### 3.2 Unit of Work - separar cambios de confirmación

Unit of Work agrupa cambios y los confirma como una unidad. En EF Core, `DbContext` cumple en gran parte el rol de Repository + Unit of Work:

```csharp
context.Contactos.Add(contacto);
context.Contactos.Remove(contactoViejo);
context.Contactos.Update(contactoEditado);

await context.SaveChangesAsync();
```

Hasta `SaveChangesAsync()`, EF Core solo rastrea cambios.

```text
Servicio   → decide qué operación de negocio hacer
DbContext  → rastrea y confirma cambios
Base       → persiste físicamente
```

## 4. Specification - separar criterios de búsqueda

Cuando el repositorio empieza a llenarse de combinaciones, hay una responsabilidad escondida: el criterio.

```csharp
Task<List<Producto>> ObtenerActivosAsync();
Task<List<Producto>> ObtenerActivosDeCategoriaAsync(int categoriaId);
Task<List<Producto>> ObtenerActivosDeCategoriaConPrecioMayorAsync(int categoriaId, decimal min);
```

Specification convierte cada criterio en una pieza componible:

```csharp
public abstract class Specification<T> {
    public abstract Expression<Func<T, bool>> AsExpression();

    public Specification<T> And(Specification<T> otra) =>
        new AndSpec<T>(this, otra);
}

public sealed class ProductoActivo : Specification<Producto> {
    public override Expression<Func<Producto, bool>> AsExpression() =>
        p => p.Activo;
}

public sealed class ProductoEnCategoria(int id) : Specification<Producto> {
    public override Expression<Func<Producto, bool>> AsExpression() =>
        p => p.CategoriaId == id;
}
```

Uso:

```csharp
var spec = new ProductoActivo()
    .And(new ProductoEnCategoria(3));

var productos = await db.Productos
    .Where(spec.AsExpression())
    .ToListAsync();
```

SRP aplicado: el repositorio busca; cada specification expresa una regla de negocio.

## 5. Strategy, Template Method y State

### 5.1 Strategy - separar algoritmos intercambiables

Strategy sirve cuando hay variantes de un algoritmo.

Código mezclado:

```csharp
public sealed class ExportacionService {
    public void Exportar(IEnumerable<Contacto> contactos, string formato, string path) {
        if (formato == "csv") {
            File.WriteAllLines(path, contactos.Select(c => $"{c.Apellido},{c.Nombre}"));
            return;
        }

        if (formato == "json") {
            File.WriteAllText(path, JsonSerializer.Serialize(contactos));
            return;
        }

        throw new InvalidOperationException("Formato no soportado.");
    }
}
```

Con Strategy:

```csharp
public interface IExportadorContactos {
    string Formato { get; }
    void Exportar(IEnumerable<Contacto> contactos, string path);
}

public sealed class CsvExportadorContactos : IExportadorContactos {
    public string Formato => "csv";

    public void Exportar(IEnumerable<Contacto> contactos, string path) =>
        File.WriteAllLines(path, contactos.Select(c => $"{c.Apellido},{c.Nombre}"));
}

public sealed class JsonExportadorContactos : IExportadorContactos {
    public string Formato => "json";

    public void Exportar(IEnumerable<Contacto> contactos, string path) =>
        File.WriteAllText(path, JsonSerializer.Serialize(contactos));
}
```

Servicio:

```csharp
public sealed class ExportacionService(IEnumerable<IExportadorContactos> exportadores) {
    public void Exportar(string formato, IEnumerable<Contacto> contactos, string path) {
        IExportadorContactos exportador = exportadores.Single(e => e.Formato == formato);
        exportador.Exportar(contactos, path);
    }
}
```

SRP aplicado: cada exportador cambia por su formato; el servicio solo coordina.

### 5.2 Template Method - separar estructura fija de pasos variables

Template Method sirve cuando varios algoritmos comparten estructura pero cambian algunos pasos.

```csharp
public abstract class GeneradorReporte {
    public string Generar() {
        var sb = new StringBuilder();
        sb.AppendLine($"=== {Titulo()} ===");
        sb.AppendLine($"Fecha: {DateTime.Today:d}");

        var datos = ObtenerDatos();
        foreach (var fila in datos)
            sb.AppendLine($"{fila.Id} - ${fila.Total}");

        return sb.ToString();
    }

    protected abstract string Titulo();
    protected abstract IEnumerable<Fila> ObtenerDatos();
}

public sealed class ReporteVentas : GeneradorReporte {
    protected override string Titulo() => "VENTAS";
    protected override IEnumerable<Fila> ObtenerDatos() => ConsultaVentas();
}
```

SRP aplicado: la clase base cambia si cambia la estructura; cada subclase cambia si cambia su reporte.

### 5.3 State - separar comportamiento por estado

State sirve cuando varios métodos preguntan por el mismo campo `estado`.

Código mezclado:

```csharp
public sealed class Pedido {
    public string Estado { get; set; } = "pendiente";

    public void Pagar() {
        if (Estado == "pendiente") Estado = "pagado";
        else throw new InvalidOperationException();
    }
}
```

Con State:

```csharp
public abstract class EstadoPedido {
    public abstract EstadoPedido Pagar();
    public abstract EstadoPedido Cancelar();
}

public sealed class Pendiente : EstadoPedido {
    public override EstadoPedido Pagar() => new Pagado();
    public override EstadoPedido Cancelar() => new Cancelado();
}

public sealed class Pagado : EstadoPedido {
    public override EstadoPedido Pagar() => throw new InvalidOperationException("Ya pagado");
    public override EstadoPedido Cancelar() => new Cancelado();
}

public sealed class Pedido {
    public EstadoPedido Estado { get; private set; } = new Pendiente();

    public void Pagar() => Estado = Estado.Pagar();
    public void Cancelar() => Estado = Estado.Cancelar();
}
```

SRP aplicado: cada estado concentra las reglas permitidas desde ese estado.

## 6. Factory y Builder

### 6.1 Factory - separar creación o selección

Factory sirve cuando el código de negocio no debería decidir qué clase concreta instanciar.

```csharp
public interface IExportadorFactory {
    IExportadorContactos Crear(string formato);
}

public sealed class ExportadorFactory(IEnumerable<IExportadorContactos> exportadores)
    : IExportadorFactory {
    public IExportadorContactos Crear(string formato) =>
        exportadores.Single(e => e.Formato == formato);
}
```

Uso:

```csharp
public sealed class ExportacionService(IExportadorFactory factory) {
    public void Exportar(string formato, IEnumerable<Contacto> contactos, string path) {
        factory.Crear(formato).Exportar(contactos, path);
    }
}
```

Factories comunes en .NET:

```text
IHttpClientFactory
ILoggerFactory
IDbContextFactory<TContext>
```

SRP aplicado: el servicio ejecuta el caso de uso; la factory selecciona o crea el objeto adecuado.

### 6.2 Builder - separar construcción compleja

Builder sirve cuando construir un objeto requiere muchos datos, valores opcionales o validaciones.

```csharp
public sealed class RequestBuilder {
    private string method = "GET";
    private string url = "";
    private readonly Dictionary<string, string> headers = new();
    private string? body;

    public RequestBuilder Post(string value) { method = "POST"; url = value; return this; }
    public RequestBuilder Auth(string token) { headers["Authorization"] = $"Bearer {token}"; return this; }
    public RequestBuilder Json(object payload) {
        headers["Content-Type"] = "application/json";
        body = JsonSerializer.Serialize(payload);
        return this;
    }

    public HttpRequest Build() => new(method, url, headers, body);
}
```

Uso:

```csharp
var request = new RequestBuilder()
    .Post("https://api.empresa.com/clientes")
    .Auth(token)
    .Json(payload)
    .Build();
```

SRP aplicado: el builder sabe construir; el objeto construido representa el resultado; el consumidor decide qué necesita.

## 7. Adapter, Decorator y Proxy

### 7.1 Adapter - aislar APIs externas

Adapter traduce entre el contrato propio de la aplicación y una API externa.

```csharp
public interface INotificador {
    Task EnviarAsync(string destino, string mensaje);
}

public sealed class WhatsAppSdkClient {
    public Task SendTextAsync(string phoneNumber, string body) => Task.CompletedTask;
}

public sealed class WhatsAppNotificador(WhatsAppSdkClient client) : INotificador {
    public Task EnviarAsync(string destino, string mensaje) =>
        client.SendTextAsync(destino, mensaje);
}
```

SRP aplicado: el servicio decide cuándo notificar; el adapter conoce el SDK.

### 7.2 Decorator - agregar comportamiento transversal

Decorator envuelve un objeto con otro que implementa la misma interfaz.

```csharp
public sealed class LoggingContactoRepository(
    IContactoRepository inner,
    ILogger<LoggingContactoRepository> logger) : IContactoRepository {

    public List<Contacto> ObtenerTodos() {
        logger.LogInformation("Leyendo contactos.");
        return inner.ObtenerTodos();
    }

    public void GuardarTodos(List<Contacto> contactos) {
        logger.LogInformation("Guardando {Cantidad} contactos.", contactos.Count);
        inner.GuardarTodos(contactos);
    }
}
```

SRP aplicado: el repositorio persiste; el decorator registra logs.

En .NET, `Stream` usa esta idea constantemente:

```csharp
using var file = new FileStream("datos.gz", FileMode.Open);
using var gzip = new GZipStream(file, CompressionMode.Decompress);
using var reader = new StreamReader(gzip, Encoding.UTF8);
```

Cada capa agrega una responsabilidad sin modificar la anterior.

### 7.3 Proxy - controlar acceso o carga diferida

Proxy se parece estructuralmente a Decorator, pero su intención es controlar el acceso: lazy loading, autorización, remoto, etc.

```csharp
public interface IDocumento {
    string Contenido { get; }
}

public sealed class DocumentoReal(string ruta) : IDocumento {
    public string Contenido { get; } = File.ReadAllText(ruta);
}

public sealed class DocumentoLazy(string ruta) : IDocumento {
    private string? cache;
    public string Contenido => cache ??= File.ReadAllText(ruta);
}
```

SRP aplicado: el documento real representa el contenido; el proxy controla cuándo o cómo acceder.

## 8. Command, Chain of Responsibility y Visitor

### 8.1 Command - encapsular acciones

Command convierte una acción en un objeto. Es común en WPF, MAUI, Avalonia, WinUI y MVVM.

```csharp
public interface ICommand {
    bool CanExecute(object? parameter);
    void Execute(object? parameter);
    event EventHandler? CanExecuteChanged;
}

public sealed class GuardarContactoCommand(ContactoService service) : ICommand {
    public bool CanExecute(object? parameter) =>
        parameter is Contacto c && !string.IsNullOrWhiteSpace(c.Nombre);

    public void Execute(object? parameter) {
        if (parameter is Contacto contacto)
            service.Agregar(contacto);
    }

    public event EventHandler? CanExecuteChanged;
}
```

SRP aplicado: el botón dispara; el command representa la acción; el service ejecuta negocio.

### 8.2 Chain of Responsibility - separar pasos de validación

Chain sirve cuando un proceso tiene pasos independientes que pueden continuar o cortar.

```csharp
public abstract class HandlerRequest {
    private HandlerRequest? siguiente;

    public HandlerRequest Encadenar(HandlerRequest handler) {
        siguiente = handler;
        return handler;
    }

    public Resultado Procesar(Request request) {
        Resultado resultado = ProcesarLocal(request);
        if (!resultado.Ok) return resultado;

        return siguiente?.Procesar(request) ?? Resultado.Ok();
    }

    protected abstract Resultado ProcesarLocal(Request request);
}
```

Ejemplos de handlers: autenticación, autorización, rate limit, validación de schema.

SRP aplicado: cada handler cambia por un único aspecto del procesamiento.

### 8.3 Visitor - separar operaciones sobre jerarquías

Visitor sirve cuando una jerarquía de tipos recibe muchas operaciones que no pertenecen al dominio central.

```csharp
public abstract class Expr {
    public abstract T Aceptar<T>(IVisitorExpr<T> visitor);
}

public sealed class Numero(double valor) : Expr {
    public double Valor { get; } = valor;
    public override T Aceptar<T>(IVisitorExpr<T> visitor) => visitor.Visitar(this);
}

public sealed class Suma(Expr izquierda, Expr derecha) : Expr {
    public Expr Izquierda { get; } = izquierda;
    public Expr Derecha { get; } = derecha;
    public override T Aceptar<T>(IVisitorExpr<T> visitor) => visitor.Visitar(this);
}

public interface IVisitorExpr<T> {
    T Visitar(Numero numero);
    T Visitar(Suma suma);
}
```

SRP aplicado: los nodos representan estructura; cada visitor representa una operación como evaluar, imprimir o exportar.

En C# moderno, para jerarquías chicas, `record sealed` + pattern matching puede ser más simple que Visitor clásico.

## 9. Coordinación: Facade, Mediator y Observer

### 9.1 Facade - simplificar un subsistema

Facade ofrece una API simple sobre varias piezas internas.

```csharp
public sealed class AgendaExporter(
    IContactoRepository repository,
    IExportadorFactory factory) {

    public void ExportarAgenda(string formato, string path) {
        List<Contacto> contactos = repository.ObtenerTodos();
        factory.Crear(formato).Exportar(contactos, path);
    }
}
```

SRP aplicado: la facade coordina un caso común; las piezas internas mantienen su responsabilidad.

Una facade debe simplificar. Si acumula toda la lógica de la aplicación, dejó de tener una responsabilidad clara.

### 9.2 Mediator - reducir acoplamiento entre colaboradores

Mediator sirve cuando muchos objetos se conocen entre sí para coordinar un flujo.

```csharp
public record PedidoConfirmado(int PedidoId);

public sealed class HandlerReservarStock(ServicioInventario inventario)
    : IHandler<PedidoConfirmado> {
    public Task Handle(PedidoConfirmado evento, CancellationToken ct) =>
        inventario.ReservarStock(evento.PedidoId, ct);
}

public sealed class ServicioPedidos(IMediator mediator) {
    public Task ConfirmarPedido(Pedido pedido) =>
        mediator.Publish(new PedidoConfirmado(pedido.Id));
}
```

SRP aplicado: cada handler responde a un paso del flujo; el servicio publica el evento. MediatR y CQRS usan esta idea.

### 9.3 Observer / Events - publicar sin conocer consumidores

Observer permite que un objeto publique eventos sin saber quién escucha.

```csharp
public sealed class ContactoService {
    public event EventHandler<Contacto>? ContactoAgregado;

    public void Agregar(Contacto contacto) {
        // guardar contacto...
        ContactoAgregado?.Invoke(this, contacto);
    }
}

service.ContactoAgregado += (_, contacto) =>
    Console.WriteLine($"Nuevo contacto: {contacto.Nombre}");
```

SRP aplicado: el servicio publica lo que ocurrió; logger, UI o analítica reaccionan por separado.

## 10. Patrones propios del ecosistema .NET

### 10.1 Options Pattern - configuración tipada

Evita que cada clase lea strings mágicos desde `IConfiguration`.

```csharp
public sealed class AgendaOptions {
    public string ArchivoContactos { get; set; } = "contactos.json";
}

builder.Services.Configure<AgendaOptions>(
    builder.Configuration.GetSection("Agenda"));

public sealed class JsonContactoRepository(IOptions<AgendaOptions> options)
    : IContactoRepository {
    private readonly AgendaOptions opts = options.Value;
}
```

SRP aplicado:

```text
appsettings.json       → valores externos
AgendaOptions          → modelo tipado
Options infrastructure → enlace entre configuración y objeto
Repositorio / servicio → usa configuración ya modelada
```

### 10.2 Middleware Pipeline - responsabilidades HTTP encadenadas

ASP.NET Core procesa requests con una cadena de middlewares.

```csharp
app.UseExceptionHandler("/error");
app.UseHttpsRedirection();
app.UseAuthentication();
app.UseAuthorization();
app.UseRateLimiter();

app.MapControllers();
```

Middleware propio:

```csharp
public sealed class RequestLoggingMiddleware(
    RequestDelegate next,
    ILogger<RequestLoggingMiddleware> logger) {

    public async Task InvokeAsync(HttpContext context) {
        logger.LogInformation("Request: {Method} {Path}",
            context.Request.Method,
            context.Request.Path);

        await next(context);
    }
}
```

SRP aplicado: autenticación autentica, autorización autoriza, logging registra, el endpoint resuelve el caso concreto.

### 10.3 Filters - lógica transversal de controladores

Los filtros actúan dentro de MVC o Minimal APIs, después de que ASP.NET Core ya entendió la action o endpoint.

```csharp
public sealed class LogActionFilter(ILogger<LogActionFilter> logger) : IActionFilter {
    public void OnActionExecuting(ActionExecutingContext context) =>
        logger.LogInformation("Ejecutando {Action}", context.ActionDescriptor.DisplayName);

    public void OnActionExecuted(ActionExecutedContext context) =>
        logger.LogInformation("Terminó {Action}", context.ActionDescriptor.DisplayName);
}
```

SRP aplicado:

```text
Controller → traduce HTTP a caso de uso
Filter     → logging, validación, errores o autorización
Service    → reglas del caso de uso
```

### 10.4 DTO + Mapper - contrato externo vs modelo interno

No siempre conviene exponer la entidad interna por una API.

```csharp
public sealed class Contacto {
    public int Id { get; set; }
    public string Nombre { get; set; } = "";
    public string Apellido { get; set; } = "";
    public string Email { get; set; } = "";
    public Guid TenantId { get; set; }
}

public sealed class ContactoDto {
    public int Id { get; set; }
    public string NombreCompleto { get; set; } = "";
    public string Email { get; set; } = "";
}

public static class ContactoMapper {
    public static ContactoDto ToDto(Contacto contacto) => new() {
        Id = contacto.Id,
        NombreCompleto = $"{contacto.Nombre} {contacto.Apellido}",
        Email = contacto.Email
    };
}
```

SRP aplicado: entidad interna, DTO y mapper cambian por motivos distintos.

### 10.5 MVVM - UI, estado y acciones

MVVM separa presentación, estado de pantalla y casos de uso.

```text
View ← binding → ViewModel → Service → Repository
```

```csharp
public sealed class ContactoViewModel(ContactoService service)
    : INotifyPropertyChanged {

    private string nombre = "";

    public string Nombre {
        get => nombre;
        set { nombre = value; OnPropertyChanged(); }
    }

    public ICommand GuardarCommand { get; } =
        new GuardarContactoCommand(service);

    public event PropertyChangedEventHandler? PropertyChanged;

    private void OnPropertyChanged([CallerMemberName] string? property = null) =>
        PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(property));
}
```

SRP aplicado:

```text
View       → dibuja
ViewModel  → estado y comandos
Command    → acción ejecutable
Service    → caso de uso
Repository → persistencia
```

## 11. SOLID en .NET: ejemplos breves

La BCL de .NET muestra estos principios en producción.

### 11.1 SRP: `FileStream`, `StreamReader`, `TextReader`

Leer líneas de texto tiene varias responsabilidades:

```text
FileStream   → bytes desde un archivo
StreamReader → bytes a caracteres con encoding
TextReader   → lectura de texto y líneas
```

```csharp
using var fs = new FileStream("datos.txt", FileMode.Open);
using var sr = new StreamReader(fs, Encoding.UTF8);

string? linea = sr.ReadLine();
```

### 11.2 OCP y Decorator: `Stream`

`Stream` permite agregar capacidades sin modificar clases existentes.

```csharp
using var archivo = new FileStream("datos.gz", FileMode.Open);
using var descomp = new GZipStream(archivo, CompressionMode.Decompress);
using var lector = new StreamReader(descomp, Encoding.UTF8);
```

Cambiar GZip por Brotli no toca `StreamReader` ni `FileStream`.

### 11.3 ISP: colecciones

.NET separa interfaces según lo que el cliente necesita:

```text
IEnumerable<T>           → iterar
IReadOnlyCollection<T>   → iterar y contar
IReadOnlyList<T>         → leer por índice
ICollection<T>           → agregar y quitar
IList<T>                 → indexar y modificar
```

Aceptar `IEnumerable<T>` comunica que solo se va a iterar. Aceptar `ICollection<T>` comunica que se necesita modificar.

### 11.4 DIP: depender de abstracciones

`StreamReader` depende de `Stream`, no de `FileStream`:

```csharp
var desdeArchivo = new StreamReader(new FileStream("a.txt", FileMode.Open));
var desdeMemoria = new StreamReader(new MemoryStream(bytes));
```

El consumidor no sabe de dónde vienen los bytes. Eso facilita composición y tests.

## 12. Caso integrador

Problema: agregar un contacto, validarlo, guardarlo y notificar.

### 12.1 Versión mezclada

```csharp
public sealed class AgendaService {
    public async Task AgregarAsync(Contacto contacto) {
        if (string.IsNullOrWhiteSpace(contacto.Nombre))
            throw new InvalidOperationException("El nombre es obligatorio.");

        string path = "contactos.json";

        List<Contacto> contactos;
        if (File.Exists(path)) {
            string jsonActual = await File.ReadAllTextAsync(path);
            contactos = JsonSerializer.Deserialize<List<Contacto>>(jsonActual) ?? [];
        }
        else {
            contactos = [];
        }

        contactos.Add(contacto);

        string jsonNuevo = JsonSerializer.Serialize(contactos);
        await File.WriteAllTextAsync(path, jsonNuevo);

        using HttpClient client = new();
        await client.PostAsJsonAsync("https://api.notificador.com/messages", new {
            contacto.Telefono,
            Mensaje = $"Contacto creado: {contacto.Nombre}"
        });
    }
}
```

Razones de cambio mezcladas:

```text
validación
persistencia
serialización JSON
creación de HttpClient
API externa
caso de uso
```

### 12.2 Versión separada

```csharp
public interface IContactoValidator {
    void Validar(Contacto contacto);
}

public interface IContactoRepository {
    Task<List<Contacto>> ObtenerTodosAsync();
    Task GuardarTodosAsync(List<Contacto> contactos);
}

public interface INotificador {
    Task EnviarAsync(string destino, string mensaje);
}
```

```csharp
public sealed class AgendaService(
    IContactoValidator validator,
    IContactoRepository repository,
    INotificador notificador) {

    public async Task AgregarAsync(Contacto contacto) {
        validator.Validar(contacto);

        List<Contacto> contactos = await repository.ObtenerTodosAsync();
        contactos.Add(contacto);
        await repository.GuardarTodosAsync(contactos);

        await notificador.EnviarAsync(
            contacto.Telefono,
            $"Contacto creado: {contacto.Nombre}");
    }
}
```

El servicio coordina varias piezas, pero su responsabilidad sigue siendo una: ejecutar el caso de uso "agregar contacto". Los detalles especializados viven afuera.

## 13. Reglas prácticas

### 13.1 Detectar razones de cambio

Preguntas útiles:

```text
¿Cambia si cambia la base de datos?
¿Cambia si cambia la UI?
¿Cambia si cambia el proveedor externo?
¿Cambia si cambia una regla de validación?
¿Cambia si cambia el formato de salida?
¿Cambia si cambia la configuración?
```

Si varias respuestas son "sí", hay responsabilidades mezcladas.

### 13.2 No crear abstracciones antes de tiempo

SRP no significa llenar todo de interfaces sin necesidad concreta.

```text
Mala señal:
    IContactoServiceFactoryProviderManager

Buena señal:
    IContactoRepository
    INotificador
    IExportadorContactos
```

La abstracción debe responder a una variación real o razonablemente probable.

### 13.3 Preferir nombres que revelen responsabilidad

Buenos nombres:

```text
JsonContactoRepository
ContactoValidator
CsvExportadorContactos
WhatsAppNotificador
AgendaOptions
ContactoMapper
```

Malos nombres:

```text
ContactoManager
AgendaHelper
Utils
GeneralService
ProcessData
```

`Manager`, `Helper` y `Utils` suelen esconder responsabilidades poco claras.

### 13.4 Separar orquestación de detalle

Esto puede estar bien:

```csharp
validator.Validar(contacto);
repository.Guardar(contacto);
notificador.Enviar(...);
```

No viola SRP si la responsabilidad del servicio es coordinar el caso de uso.

Lo problemático es que ese mismo servicio implemente internamente validación, persistencia, serialización, HTTP, logging y configuración.

### 13.5 Usar patrones como herramienta, no como decoración

```text
¿Muchos formatos de exportación?            → Strategy
¿API externa incómoda?                      → Adapter
¿Logging/cache/retry mezclados con negocio? → Decorator
¿Creación de objetos ensucia el servicio?   → Factory
¿Negocio mezclado con datos?                → Repository
¿Pantalla con demasiada lógica?             → MVVM / Command
```

## Resumen

```text
SRP
    └─► Una clase, una razón clara para cambiar

Patrones por responsabilidad:
    ├── Dependency Injection  → separar uso de creación
    ├── Factory / Builder     → separar creación y construcción
    ├── Repository / UoW      → separar persistencia y transacción
    ├── Specification         → separar criterios de búsqueda
    ├── Strategy / Template   → separar algoritmos variables
    ├── State                 → separar comportamiento por estado
    ├── Command / Chain       → separar acciones y pasos
    ├── Visitor               → separar operaciones sobre jerarquías
    ├── Adapter               → aislar APIs externas
    ├── Decorator / Proxy     → agregar o controlar acceso
    ├── Facade / Mediator     → separar coordinación
    ├── Observer / Events     → desacoplar eventos y reacciones
    ├── Options Pattern       → separar configuración
    ├── Middleware / Filters  → separar lógica HTTP transversal
    ├── DTO + Mapper          → separar contrato externo
    └── MVVM                  → separar UI, estado y acciones

Regla final:
    └─► Si una clase cambia por varios motivos distintos,
        probablemente necesita que alguna responsabilidad salga de ahí.
        El nombre del patrón viene después.
```

La meta no es tener más clases. La meta es que cada clase tenga un trabajo reconocible, testeable y reemplazable.
