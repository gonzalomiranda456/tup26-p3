## 1. El problema de la clase 3

El componente Blazor de la clase anterior funciona, pero tiene un acoplamiento problemático:

```razor
@inject TareasContext db
```

El frontend **conoce** la base de datos. Accede a ella directamente. Eso significa que:

- El proyecto Blazor tiene que referenciar EF Core, el `DbContext`, y el modelo de datos.
- Si mañana cambiamos la base de datos, hay que modificar el frontend.
- No hay forma de tener el frontend y el backend en servidores separados.
- Otro cliente (una app móvil, otra página web) no puede reutilizar la lógica de acceso a datos.

La arquitectura correcta es la que construimos en las clases 1 y 2: **el frontend no habla con la base de datos, habla con la API**. La API habla con la base de datos.

```
┌──────────────┐         ┌──────────────┐         ┌──────────────┐
│   Blazor     │ ──HTTP──►   Minimal    │ ──EF ───►   SQLite     │
│  (frontend)  │◄──JSON──   API        │◄── ────  │    (disco)   │
│              │         │  (backend)  │          │              │
└──────────────┘         └──────────────┘         └──────────────┘
```

En esta clase hacemos esa separación. Al terminar, vamos a tener dos proyectos independientes que se comunican via HTTP.

## 2. La arquitectura correcta

Dos proyectos, dos procesos, dos puertos:

```
TareasApi/    → el backend de las clases 1 y 2
              → corre en http://localhost:5000
              → tiene la lógica de negocio y acceso a datos

TareasWeb/    → el frontend de la clase 3
              → corre en http://localhost:5001
              → no conoce la base de datos
              → habla con TareasApi via HTTP
```

El componente Blazor va a reemplazar el `@inject TareasContext db` por `@inject HttpClient http`. En lugar de llamar `db.Tareas.ToListAsync()`, va a hacer `GET http://localhost:5000/tareas` y recibir el JSON.

El flujo completo de "listar tareas" pasa a ser:

```
1. Usuario abre http://localhost:5001
2. Blazor monta el componente
3. OnInitializedAsync() se ejecuta
4. El componente hace GET http://localhost:5000/tareas
5. La API consulta SQLite y devuelve JSON
6. El componente deserializa el JSON a List<Tarea>
7. Blazor renderiza la lista en el browser
```

Son más pasos, pero cada pieza tiene una sola responsabilidad.

## 3. HttpClient: hacer requests desde C#

`HttpClient` es la clase de .NET para hacer requests HTTP desde código. Es el equivalente de REST Client, pero desde C#.

### Operaciones básicas

```csharp
var http = new HttpClient();
http.BaseAddress = new Uri("http://localhost:5000");

// GET — leer datos
var tareas = await http.GetFromJsonAsync<List<Tarea>>("/tareas");

// POST — crear
var nueva = new Tarea(0, "Nueva tarea", false);
await http.PostAsJsonAsync("/tareas", nueva);

// PUT — actualizar
await http.PutAsJsonAsync($"/tareas/{id}", actualizada);

// DELETE — eliminar
await http.DeleteAsync($"/tareas/{id}");
```

Los métodos `GetFromJsonAsync`, `PostAsJsonAsync` y `PutAsJsonAsync` son extensiones de `System.Net.Http.Json` que combinan hacer el request con serializar/deserializar JSON automáticamente. Son el equivalente de lo que hace Minimal API en el servidor, pero en el cliente.

### El ciclo de un GET

```csharp
var tareas = await http.GetFromJsonAsync<List<Tarea>>("/tareas");
```

Por adentro, esto hace tres cosas:

1. Hace `GET http://localhost:5000/tareas`
2. Lee el cuerpo de la respuesta (el JSON)
3. Deserializa el JSON a `List<Tarea>`

Si el servidor devuelve `404` u otro error, `GetFromJsonAsync` lanza una excepción. Si querés manejar el error manualmente:

```csharp
var response = await http.GetAsync("/tareas");

if (response.IsSuccessStatusCode) {
    var tareas = await response.Content.ReadFromJsonAsync<List<Tarea>>();
}
```

### El ciclo de un POST

```csharp
var response = await http.PostAsJsonAsync("/tareas", nueva);
var creada   = await response.Content.ReadFromJsonAsync<Tarea>();
```

`PostAsJsonAsync` serializa `nueva` a JSON y lo manda como cuerpo del request con `Content-Type: application/json`. La API lo recibe, lo deserializa, lo guarda y devuelve la tarea creada (con el `Id` asignado por la base de datos). `ReadFromJsonAsync<Tarea>()` deserializa ese JSON de vuelta a un objeto.

## 4. El problema de CORS

Cuando el browser intenta hacer un request desde `http://localhost:5001` (el frontend) hacia `http://localhost:5000` (el backend), se activa una restricción de seguridad del browser llamada **CORS** (*Cross-Origin Resource Sharing*).

### Por qué existe CORS

Imaginá que estás logueado en tu banco en `banco.com`. Abrís otra pestaña y entrás a un sitio malicioso `evil.com`. Sin CORS, el JavaScript de `evil.com` podría hacer requests a `banco.com` usando tus credenciales y robar tu información.

CORS previene esto: el browser solo permite que un sitio haga requests a **otro origen** si el servidor del destino lo autoriza explícitamente.

Un **origen** es la combinación de protocolo + dominio + puerto:

```
http://localhost:5000   ← un origen
http://localhost:5001   ← otro origen (puerto distinto → origen distinto)
https://miapp.com       ← otro origen
```

### Cómo funciona

Antes de hacer el request real, el browser manda un request de "preflight" (`OPTIONS`) preguntando al servidor: *¿aceptás requests desde `http://localhost:5001`?*

Si el servidor responde con los headers correctos autorizando ese origen, el browser procede con el request real. Si no, el browser bloquea el request y el frontend recibe un error.

### La solución: configurar CORS en el backend

El backend tiene que decirle explícitamente qué orígenes puede aceptar. En Minimal API:

```csharp
// En Program.cs del backend

builder.Services.AddCors(options => {
    options.AddPolicy("frontend", policy => {
        policy.WithOrigins("http://localhost:5001")  // el origen del frontend
              .AllowAnyMethod()
              .AllowAnyHeader();
    });
});

// ... después de builder.Build() ...

app.UseCors("frontend");   // aplicar la política al pipeline
```

`WithOrigins` lista los orígenes que tienen permiso. `AllowAnyMethod` acepta GET, POST, PUT, DELETE, etc. `AllowAnyHeader` acepta cualquier cabecera (incluyendo `Content-Type: application/json`).

> CORS es una restricción **del browser**, no del servidor. Si hacés requests desde REST Client o desde otro servidor, CORS no aplica. Solo aplica cuando el request viene de JavaScript (o Blazor WASM) corriendo en un browser.

## 5. Preparar el backend

El backend de las clases anteriores casi no necesita cambios. Solo agregamos CORS.

`Program.cs` del proyecto `TareasApi`:

```csharp
using Microsoft.EntityFrameworkCore;

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddDbContext<TareasContext>(options =>
    options.UseSqlite("Data Source=tareas.db"));

// Agregar CORS
builder.Services.AddCors(options => {
    options.AddPolicy("frontend", policy => {
        policy.WithOrigins("http://localhost:5001")
              .AllowAnyMethod()
              .AllowAnyHeader();
    });
});

var app = builder.Build();

// Aplicar CORS antes de los endpoints
app.UseCors("frontend");

using (var scope = app.Services.CreateScope())
    scope.ServiceProvider.GetRequiredService<TareasContext>().Database.EnsureCreated();

app.MapGet("/tareas", async (TareasContext db) =>
    await db.Tareas.ToListAsync());

app.MapGet("/tareas/{id}", async (int id, TareasContext db) => {
    var tarea = await db.Tareas.FindAsync(id);
    return tarea is null ? Results.NotFound() : Results.Ok(tarea);
});

app.MapPost("/tareas", async (Tarea nueva, TareasContext db) => {
    db.Tareas.Add(nueva);
    await db.SaveChangesAsync();
    return Results.Created($"/tareas/{nueva.Id}", nueva);
});

app.MapPut("/tareas/{id}", async (int id, Tarea actualizada, TareasContext db) => {
    var tarea = await db.Tareas.FindAsync(id);
    if (tarea is null) return Results.NotFound();
    db.Entry(tarea).CurrentValues.SetValues(actualizada with { Id = id });
    await db.SaveChangesAsync();
    return Results.Ok(tarea);
});

app.MapDelete("/tareas/{id}", async (int id, TareasContext db) => {
    var tarea = await db.Tareas.FindAsync(id);
    if (tarea is null) return Results.NotFound();
    db.Tareas.Remove(tarea);
    await db.SaveChangesAsync();
    return Results.NoContent();
});

app.Run();

record Tarea(int Id, string Texto, bool Completada);

class TareasContext : DbContext {
    public TareasContext(DbContextOptions<TareasContext> options) : base(options) { }
    public DbSet<Tarea> Tareas { get; set; }
}
```

El único cambio respecto a la clase 2 son las líneas de CORS. **El resto es idéntico.**

## 6. Preparar el frontend

### Registrar HttpClient en Program.cs

En el proyecto `TareasWeb`, reemplazamos el registro del `DbContext` por un `HttpClient` configurado con la URL base del backend:

```csharp
// Antes (clase 3):
builder.Services.AddDbContext<TareasContext>(...);

// Ahora:
builder.Services.AddScoped(sp =>
    new HttpClient { BaseAddress = new Uri("http://localhost:5000") });
```

`AddScoped` registra el `HttpClient` con scope por request: cada usuario tiene su propio `HttpClient`, pero se reutiliza durante toda la sesión. La `BaseAddress` evita repetir el host en cada llamada.

### El record Tarea en el frontend

El frontend necesita conocer la estructura de `Tarea` para deserializar el JSON. Definimos el mismo record en el proyecto Blazor:

```csharp
record Tarea(int Id, string Texto, bool Completada);
```

No es duplicación problemática — es el **contrato** entre frontend y backend. Ambos lados tienen que conocer la forma de los datos. En proyectos más grandes este record viviría en un proyecto compartido (`TareasShared`), pero por ahora lo definimos en los dos proyectos.

### Eliminar la referencia a EF Core

El proyecto Blazor ya no necesita EF Core ni `TareasContext`. Si los tenés del proyecto anterior, los eliminás:

```bash
dotnet remove package Microsoft.EntityFrameworkCore.Sqlite
```

Y eliminás la clase `TareasContext` del proyecto.

## 7. Conectar los endpoints uno por uno

Ahora reescribimos cada operación del componente. El patrón es siempre el mismo: reemplazamos la llamada a `db` por una llamada a `http`.

### Leer — GET /tareas

```csharp
// Antes:
tareas = await db.Tareas.ToListAsync();

// Ahora:
tareas = await http.GetFromJsonAsync<List<Tarea>>("/tareas") ?? [];
```

El `?? []` maneja el caso en que la respuesta sea `null` (JSON vacío o error de deserialización).

### Crear — POST /tareas

```csharp
// Antes:
db.Tareas.Add(new Tarea(0, textoNueva.Trim(), false));
await db.SaveChangesAsync();

// Ahora:
await http.PostAsJsonAsync("/tareas", new Tarea(0, textoNueva.Trim(), false));
```

### Actualizar — PUT /tareas/{id}

```csharp
// Antes:
var actualizada = tarea with { Completada = !tarea.Completada };
db.Entry(tarea).CurrentValues.SetValues(actualizada);
await db.SaveChangesAsync();

// Ahora:
var actualizada = tarea with { Completada = !tarea.Completada };
await http.PutAsJsonAsync($"/tareas/{tarea.Id}", actualizada);
```

### Eliminar — DELETE /tareas/{id}

```csharp
// Antes:
db.Tareas.Remove(tarea);
await db.SaveChangesAsync();

// Ahora:
await http.DeleteAsync($"/tareas/{tarea.Id}");
```

La simetría es notable. La API que diseñamos en la clase 1 se mapea directamente a métodos de `HttpClient`. `GET` → `GetFromJsonAsync`, `POST` → `PostAsJsonAsync`, `PUT` → `PutAsJsonAsync`, `DELETE` → `DeleteAsync`. El diseño de la API fue coherente desde el principio y ahora se cosecha esa coherencia.

## 8. El componente completo

```razor
@page "/"
@inject HttpClient http

<h1>Mis Tareas</h1>

<div>
    <input @bind="textoNueva" placeholder="Nueva tarea..." />
    <button @onclick="AgregarTarea">Agregar</button>
</div>

@if (cargando) {
    <p>Cargando...</p>
} else if (!tareas.Any()) {
    <p>No hay tareas. ¡Agregá una!</p>
} else {
    <ul>
        @foreach (var tarea in tareas)
        {
            <li>
                <input type="checkbox"
                       checked="@tarea.Completada"
                       @onchange="() => ToggleCompletada(tarea)" />
                <span style="text-decoration: @(tarea.Completada ? "line-through" : "none")">
                    @tarea.Texto
                </span>
                <button @onclick="() => EliminarTarea(tarea)">✕</button>
            </li>
        }
    </ul>
}

@code {
    List<Tarea> tareas     = new();
    string      textoNueva = "";
    bool        cargando   = true;

    protected override async Task OnInitializedAsync() {
        await CargarTareas();
    }

    async Task CargarTareas() {
        tareas   = await http.GetFromJsonAsync<List<Tarea>>("/tareas") ?? [];
        cargando = false;
    }

    async Task AgregarTarea() {
        if (string.IsNullOrWhiteSpace(textoNueva)) return;

        await http.PostAsJsonAsync("/tareas", new Tarea(0, textoNueva.Trim(), false));
        textoNueva = "";
        await CargarTareas();
    }

    async Task ToggleCompletada(Tarea tarea) {
        var actualizada = tarea with { Completada = !tarea.Completada };
        await http.PutAsJsonAsync($"/tareas/{tarea.Id}", actualizada);
        await CargarTareas();
    }

    async Task EliminarTarea(Tarea tarea) {
        await http.DeleteAsync($"/tareas/{tarea.Id}");
        await CargarTareas();
    }
}

@* Al final del archivo o en un archivo separado Models.cs *@
@code {
    record Tarea(int Id, string Texto, bool Completada);
}
```

Comparalo con el componente de la clase 3. **La plantilla HTML es idéntica**. Solo cambiaron las implementaciones de los métodos en `@code`. Eso es exactamente lo que debe pasar cuando separás responsabilidades correctamente: cambiar el origen de los datos no debería cambiar cómo se muestran.

## 9. Correr los dos proyectos juntos

Para que la app funcione, los dos proyectos tienen que estar corriendo al mismo tiempo, en terminales separadas.

**Terminal 1 — el backend:**

```bash
cd TareasApi
dotnet run
# Escuchando en http://localhost:5000
```

**Terminal 2 — el frontend:**

```bash
cd TareasWeb
dotnet run
# Escuchando en http://localhost:5001
```

Abrí `http://localhost:5001` en el browser. El frontend carga, hace `GET http://localhost:5000/tareas`, recibe el JSON y muestra la lista.

### Verificar que el flujo funciona

Podés observar el tráfico HTTP directamente en el browser. Abrí las herramientas de desarrollo (F12), pestaña **Network**, y recargá la página. Vas a ver los requests que hace el frontend:

```
GET /tareas          200 OK    [lista de tareas en JSON]
```

Cuando agregás una tarea:

```
POST /tareas         201 Created   [tarea nueva con id asignado]
GET  /tareas         200 OK        [lista actualizada]
```

El frontend hace exactamente los mismos requests que hacíamos manualmente con REST Client en las clases anteriores. La diferencia es que ahora los hace automáticamente en respuesta a las acciones del usuario.

### Manejo básico de errores

Si el backend no está corriendo cuando el frontend intenta conectarse, `GetFromJsonAsync` lanza una excepción. Podemos capturarla y mostrar un mensaje:

```razor
@if (error is not null) {
    <p style="color: red">@error</p>
}

@code {
    string? error = null;

    async Task CargarTareas() {
        try {
            tareas   = await http.GetFromJsonAsync<List<Tarea>>("/tareas") ?? [];
            cargando = false;
            error    = null;
        }
        catch (Exception ex) {
            error    = $"No se pudo conectar con el servidor: {ex.Message}";
            cargando = false;
        }
    }
}
```

## 10. Resumen

```
La arquitectura correcta:
  Frontend (Blazor)  →  HTTP  →  Backend (Minimal API)  →  EF Core  →  Base de datos
  Cada capa habla solo con la siguiente. Ninguna se salta niveles.

CORS:
  El browser bloquea requests entre orígenes distintos por seguridad.
  Origen = protocolo + dominio + puerto.
  Solución: el backend declara qué orígenes puede atender.

  builder.Services.AddCors(...)   → registrar la política
  app.UseCors("nombre")           → aplicar al pipeline

HttpClient — el equivalente de REST Client desde C#:
  GetFromJsonAsync<T>(url)       → GET + deserializar JSON
  PostAsJsonAsync(url, objeto)   → POST + serializar JSON
  PutAsJsonAsync(url, objeto)    → PUT + serializar JSON
  DeleteAsync(url)               → DELETE

Mapeo API ↔ HttpClient:
  GET    /tareas      →  GetFromJsonAsync<List<Tarea>>("/tareas")
  POST   /tareas      →  PostAsJsonAsync("/tareas", nueva)
  PUT    /tareas/{id} →  PutAsJsonAsync($"/tareas/{id}", actualizada)
  DELETE /tareas/{id} →  DeleteAsync($"/tareas/{id}")

Lo que cambió respecto a la clase 3:
  @inject TareasContext db   →   @inject HttpClient http
  db.Tareas.ToListAsync()    →   http.GetFromJsonAsync<List<Tarea>>("/tareas")
  db.Tareas.Add(t)           →   http.PostAsJsonAsync("/tareas", t)
  db.Tareas.Remove(t)        →   http.DeleteAsync($"/tareas/{t.Id}")
  La plantilla HTML no cambió — solo la fuente de los datos.

Lo que ganamos con la separación:
  El frontend no conoce la base de datos.
  El backend no conoce Blazor.
  Cualquier otro cliente (app móvil, otra web) puede consumir la misma API.
  Podemos desplegar frontend y backend en servidores distintos.
```

**La app full-stack está completa.** Dos proyectos independientes, comunicados por HTTP, con datos que persisten en SQLite. En la clase 5 miramos hacia atrás, nombramos las decisiones de diseño que tomamos y vemos cómo escalar desde acá.
