# Patrón Decorador en C#

El **patrón Decorador** sirve para agregar comportamiento a un objeto **envolviéndolo** con otro objeto que implementa la misma interfaz.

La idea central es esta:

```csharp
objetoOriginal
```

se transforma en:

```csharp
decorador(objetoOriginal)
```

Y después podés seguir envolviendo:

```csharp
decorador3(decorador2(decorador1(objetoOriginal)))
```

Como una milanesa completa: base, jamón, queso, huevo, papas. El churrasco sigue siendo churrasco, pero cada capa agrega algo.

---

## 1. El problema que resuelve

Supongamos que tenés una interfaz:

```csharp
public interface INotificador
{
    void Enviar(string mensaje);
}
```

Y una implementación básica:

```csharp
public class NotificadorEmail : INotificador
{
    public void Enviar(string mensaje)
    {
        Console.WriteLine($"Enviando email: {mensaje}");
    }
}
```

Ahora aparece un requisito:

> “Además de enviar, quiero registrar en log.”

Podrías modificar `NotificadorEmail`:

```csharp
public class NotificadorEmail : INotificador
{
    public void Enviar(string mensaje)
    {
        Console.WriteLine($"Log: {mensaje}");
        Console.WriteLine($"Enviando email: {mensaje}");
    }
}
```

Pero eso mezcla responsabilidades.

Ahora aparece otro requisito:

> “Además quiero medir tiempo.”

Entonces metés más código.

Luego:

> “Además quiero validar.”  
> “Además quiero cachear.”  
> “Además quiero reintentar si falla.”  
> “Además quiero comprimir.”  
> “Además quiero auditar.”

Y la clase empieza a parecer una empanada recalentada: todo adentro, nada distinguible.

---

## 2. Primer principio: una responsabilidad por objeto

Desde primeros principios, un objeto debería tener una razón clara para cambiar.

`NotificadorEmail` debería cambiar si cambia la forma de enviar emails.

Pero no debería cambiar si cambia:

- la política de logging;
- la validación;
- el control de permisos;
- la medición de tiempo;
- el retry;
- la auditoría;
- el cache.

Entonces conviene separar cada comportamiento en una capa independiente.

---

## 3. Solución básica: envolver

Queremos esto:

```csharp
INotificador notificador = new NotificadorEmail();
```

Y después agregarle logging sin tocar la clase original:

```csharp
INotificador notificador = new NotificadorConLogging(
    new NotificadorEmail()
);
```

El decorador implementa la misma interfaz:

```csharp
public class NotificadorConLogging : INotificador
{
    private readonly INotificador _inner;

    public NotificadorConLogging(INotificador inner)
    {
        _inner = inner;
    }

    public void Enviar(string mensaje)
    {
        Console.WriteLine($"[LOG] Enviando mensaje: {mensaje}");

        _inner.Enviar(mensaje);

        Console.WriteLine("[LOG] Mensaje enviado");
    }
}
```

Uso:

```csharp
INotificador notificador =
    new NotificadorConLogging(
        new NotificadorEmail()
    );

notificador.Enviar("Hola mundo");
```

Salida:

```text
[LOG] Enviando mensaje: Hola mundo
Enviando email: Hola mundo
[LOG] Mensaje enviado
```

El objeto consumidor no sabe si está hablando con el notificador real o con un decorador.

Para él, todo es simplemente:

```csharp
INotificador
```

---

## 4. Filosofía de diseño

El patrón Decorador se basa en una idea muy fuerte:

> Agregar comportamiento por composición, no por herencia.

La herencia dice:

```csharp
class NotificadorEmailConLogging : NotificadorEmail
```

La composición dice:

```csharp
class NotificadorConLogging : INotificador
{
    private readonly INotificador _inner;
}
```

La diferencia es enorme.

Con herencia, la clase nueva **es una especialización** de otra.

Con decoración, la clase nueva **envuelve** a otra.

---

## 5. Por qué no resolverlo con herencia

Supongamos que tenés estas variantes:

- email;
- SMS;
- WhatsApp;
- push notification.

Y estas funcionalidades adicionales:

- logging;
- validación;
- retry;
- auditoría;
- medición de tiempo.

Con herencia podrías terminar con:

```text
EmailConLogging
EmailConRetry
EmailConLoggingYRetry
EmailConValidacion
EmailConValidacionYLogging
SmsConLogging
SmsConRetry
SmsConLoggingYRetry
WhatsAppConLoggingYRetryYAuditoria
```

Eso escala horrible.

Es la famosa **explosión combinatoria de clases**.

Con Decorador, en cambio, tenés piezas combinables:

```csharp
new LoggingNotificador(
    new RetryNotificador(
        new ValidacionNotificador(
            new NotificadorEmail()
        )
    )
);
```

Cada capa hace una cosa.

## 6. Estructura general del patrón

El patrón tiene cuatro partes:

```csharp
public interface IServicio
{
    void Ejecutar();
}
```

Implementación real:

```csharp
public class ServicioReal : IServicio
{
    public void Ejecutar()
    {
        Console.WriteLine("Ejecutando servicio real");
    }
}
```

Decorador base opcional:

```csharp
public abstract class ServicioDecorator : IServicio
{
    protected readonly IServicio Inner;

    protected ServicioDecorator(IServicio inner)
    {
        Inner = inner;
    }

    public virtual void Ejecutar()
    {
        Inner.Ejecutar();
    }
}
```

Decorador concreto:

```csharp
public class ServicioConLogging : ServicioDecorator
{
    public ServicioConLogging(IServicio inner) : base(inner)
    {
    }

    public override void Ejecutar()
    {
        Console.WriteLine("Antes de ejecutar");
        base.Ejecutar();
        Console.WriteLine("Después de ejecutar");
    }
}
```

Uso:

```csharp
IServicio servicio =
    new ServicioConLogging(
        new ServicioReal()
    );

servicio.Ejecutar();
```

## 7. Caso de uso 1: logging

### Problema

Querés registrar cuándo se ejecuta una operación, pero sin contaminar la lógica principal.

Interfaz:

```csharp
public interface IProcesadorPedidos
{
    void Procesar(int pedidoId);
}
```

Implementación real:

```csharp
public class ProcesadorPedidos : IProcesadorPedidos
{
    public void Procesar(int pedidoId)
    {
        Console.WriteLine($"Procesando pedido {pedidoId}");
    }
}
```

Decorador:

```csharp
public class ProcesadorPedidosConLogging : IProcesadorPedidos
{
    private readonly IProcesadorPedidos _inner;

    public ProcesadorPedidosConLogging(IProcesadorPedidos inner)
    {
        _inner = inner;
    }

    public void Procesar(int pedidoId)
    {
        Console.WriteLine($"[LOG] Inicio procesamiento pedido {pedidoId}");

        _inner.Procesar(pedidoId);

        Console.WriteLine($"[LOG] Fin procesamiento pedido {pedidoId}");
    }
}
```

Uso:

```csharp
IProcesadorPedidos procesador =
    new ProcesadorPedidosConLogging(
        new ProcesadorPedidos()
    );

procesador.Procesar(42);
```

## 8. Caso de uso 2: validación

### Problema

Querés validar datos antes de ejecutar una operación, pero sin meter validaciones en todos lados.

```csharp
public class ProcesadorPedidosConValidacion : IProcesadorPedidos
{
    private readonly IProcesadorPedidos _inner;

    public ProcesadorPedidosConValidacion(IProcesadorPedidos inner)
    {
        _inner = inner;
    }

    public void Procesar(int pedidoId)
    {
        if (pedidoId <= 0)
        {
            throw new ArgumentException("El ID del pedido debe ser positivo.");
        }

        _inner.Procesar(pedidoId);
    }
}
```

Uso combinado:

```csharp
IProcesadorPedidos procesador =
    new ProcesadorPedidosConLogging(
        new ProcesadorPedidosConValidacion(
            new ProcesadorPedidos()
        )
    );

procesador.Procesar(42);
```

Acá el orden importa.

Primero loguea, después valida, después procesa.

Si querés validar antes de loguear:

```csharp
IProcesadorPedidos procesador =
    new ProcesadorPedidosConValidacion(
        new ProcesadorPedidosConLogging(
            new ProcesadorPedidos()
        )
    );
```

## 9. Caso de uso 3: medición de tiempo

```csharp
using System.Diagnostics;

public class ProcesadorPedidosConMedicionTiempo : IProcesadorPedidos
{
    private readonly IProcesadorPedidos _inner;

    public ProcesadorPedidosConMedicionTiempo(IProcesadorPedidos inner)
    {
        _inner = inner;
    }

    public void Procesar(int pedidoId)
    {
        Stopwatch sw = Stopwatch.StartNew();

        try
        {
            _inner.Procesar(pedidoId);
        }
        finally
        {
            sw.Stop();
            Console.WriteLine($"Tiempo: {sw.ElapsedMilliseconds} ms");
        }
    }
}
```

Uso:

```csharp
IProcesadorPedidos procesador =
    new ProcesadorPedidosConMedicionTiempo(
        new ProcesadorPedidosConLogging(
            new ProcesadorPedidos()
        )
    );

procesador.Procesar(10);
```

Este decorador no sabe nada de pedidos. Solo mide duración.

Esa es la gracia.

## 10. Caso de uso 4: retry

### Problema

Querés reintentar una operación si falla.

```csharp
public class ProcesadorPedidosConRetry : IProcesadorPedidos
{
    private readonly IProcesadorPedidos _inner;
    private readonly int _intentos;

    public ProcesadorPedidosConRetry(IProcesadorPedidos inner, int intentos = 3)
    {
        _inner = inner;
        _intentos = intentos;
    }

    public void Procesar(int pedidoId)
    {
        for (int intento = 1; intento <= _intentos; intento++)
        {
            try
            {
                _inner.Procesar(pedidoId);
                return;
            }
            catch (Exception ex) when (intento < _intentos)
            {
                Console.WriteLine($"Intento {intento} falló: {ex.Message}");
            }
        }
    }
}
```

Uso:

```csharp
IProcesadorPedidos procesador =
    new ProcesadorPedidosConRetry(
        new ProcesadorPedidos(),
        intentos: 3
    );

procesador.Procesar(15);
```

## 11. Caso de uso 5: cache

Este es uno de los casos más típicos.

Interfaz:

```csharp
public interface IBuscadorProductos
{
    string BuscarNombre(int productoId);
}
```

Implementación real:

```csharp
public class BuscadorProductosDesdeBaseDeDatos : IBuscadorProductos
{
    public string BuscarNombre(int productoId)
    {
        Console.WriteLine("Consultando base de datos...");
        return $"Producto {productoId}";
    }
}
```

Decorador con cache:

```csharp
public class BuscadorProductosConCache : IBuscadorProductos
{
    private readonly IBuscadorProductos _inner;
    private readonly Dictionary<int, string> _cache = new();

    public BuscadorProductosConCache(IBuscadorProductos inner)
    {
        _inner = inner;
    }

    public string BuscarNombre(int productoId)
    {
        if (_cache.TryGetValue(productoId, out string? nombre))
        {
            Console.WriteLine("Resultado desde cache");
            return nombre;
        }

        nombre = _inner.BuscarNombre(productoId);

        _cache[productoId] = nombre;

        return nombre;
    }
}
```

Uso:

```csharp
IBuscadorProductos buscador =
    new BuscadorProductosConCache(
        new BuscadorProductosDesdeBaseDeDatos()
    );

Console.WriteLine(buscador.BuscarNombre(1));
Console.WriteLine(buscador.BuscarNombre(1));
```

Salida:

```text
Consultando base de datos...
Producto 1
Resultado desde cache
Producto 1
```

La clase original no sabe que existe cache.

Perfecto.

## 13. Caso de uso 7: streams en .NET

Este es el ejemplo clásico en la propia biblioteca de .NET.

Tenés un `FileStream`, que trabaja con bytes de archivo:

```csharp
using FileStream fs = new FileStream("datos.txt", FileMode.OpenOrCreate);
```

Pero si querés comprimir, podés envolverlo:

```csharp
using var gzip = new System.IO.Compression.GZipStream(
    fs,
    System.IO.Compression.CompressionMode.Compress
);
```

La idea es:

```text
GZipStream envuelve FileStream
```

El `FileStream` sabe escribir bytes en disco.

El `GZipStream` sabe comprimir bytes.

No son la misma responsabilidad.

Otro ejemplo:

```csharp
using FileStream fs = File.OpenRead("datos.txt");
using StreamReader reader = new StreamReader(fs);

string texto = reader.ReadToEnd();
```

Acá:

```text
StreamReader envuelve FileStream
```

Aunque `StreamReader` no es el decorador “puro” clásico porque no implementa `Stream`; implementa una abstracción diferente (`TextReader`). Pero la filosofía es muy parecida:

```text
FileStream: bytes
StreamReader: caracteres/texto
```

Se agregan capacidades mediante capas.

## 14. Caso de uso 8: cifrado o compresión

Supongamos:

```csharp
public interface IAlmacenamiento
{
    void Guardar(string contenido);
    string Leer();
}
```

Implementación base:

```csharp
public class AlmacenamientoArchivo : IAlmacenamiento
{
    private readonly string _path;

    public AlmacenamientoArchivo(string path)
    {
        _path = path;
    }

    public void Guardar(string contenido)
    {
        File.WriteAllText(_path, contenido);
    }

    public string Leer()
    {
        return File.ReadAllText(_path);
    }
}
```

Decorador que convierte a mayúsculas antes de guardar, solo como ejemplo simple:

```csharp
public class AlmacenamientoEnMayusculas : IAlmacenamiento
{
    private readonly IAlmacenamiento _inner;

    public AlmacenamientoEnMayusculas(IAlmacenamiento inner)
    {
        _inner = inner;
    }

    public void Guardar(string contenido)
    {
        _inner.Guardar(contenido.ToUpperInvariant());
    }

    public string Leer()
    {
        return _inner.Leer();
    }
}
```

Uso:

```csharp
IAlmacenamiento almacenamiento =
    new AlmacenamientoEnMayusculas(
        new AlmacenamientoArchivo("datos.txt")
    );

almacenamiento.Guardar("hola mundo");
Console.WriteLine(almacenamiento.Leer());
```

Salida:

```text
HOLA MUNDO
```

Podrías tener decoradores más reales:

```text
AlmacenamientoConCifrado
AlmacenamientoConCompresion
AlmacenamientoConLogging
AlmacenamientoConValidacion
AlmacenamientoConCache
```

Y combinarlos:

```csharp
IAlmacenamiento almacenamiento =
    new AlmacenamientoConLogging(
        new AlmacenamientoConCifrado(
            new AlmacenamientoConCompresion(
                new AlmacenamientoArchivo("datos.bin")
            )
        )
    );
```

## 15. Caso de uso 9: decoradores con métodos asincrónicos

En aplicaciones modernas, es muy común decorar servicios `async`.

Interfaz:

```csharp
public interface IClienteApi
{
    Task<string> ObtenerDatosAsync(int id);
}
```

Implementación real:

```csharp
public class ClienteApi : IClienteApi
{
    public async Task<string> ObtenerDatosAsync(int id)
    {
        await Task.Delay(500);
        return $"Datos del recurso {id}";
    }
}
```

Decorador con logging:

```csharp
public class ClienteApiConLogging : IClienteApi
{
    private readonly IClienteApi _inner;

    public ClienteApiConLogging(IClienteApi inner)
    {
        _inner = inner;
    }

    public async Task<string> ObtenerDatosAsync(int id)
    {
        Console.WriteLine($"[LOG] Consultando recurso {id}");

        string resultado = await _inner.ObtenerDatosAsync(id);

        Console.WriteLine($"[LOG] Recurso {id} obtenido");

        return resultado;
    }
}
```

Uso:

```csharp
IClienteApi cliente =
    new ClienteApiConLogging(
        new ClienteApi()
    );

string datos = await cliente.ObtenerDatosAsync(10);

Console.WriteLine(datos);
```

## 16. Decorador y Dependency Injection

En aplicaciones ASP.NET Core o apps modernas con DI, el decorador aparece muchísimo.

Supongamos:

```csharp
public interface IEmailSender
{
    Task SendAsync(string to, string subject, string body);
}
```

Implementación real:

```csharp
public class SmtpEmailSender : IEmailSender
{
    public Task SendAsync(string to, string subject, string body)
    {
        Console.WriteLine($"SMTP: enviando email a {to}");
        return Task.CompletedTask;
    }
}
```

Decorador:

```csharp
public class LoggingEmailSender : IEmailSender
{
    private readonly IEmailSender _inner;

    public LoggingEmailSender(IEmailSender inner)
    {
        _inner = inner;
    }

    public async Task SendAsync(string to, string subject, string body)
    {
        Console.WriteLine($"[LOG] Enviando email a {to}");

        await _inner.SendAsync(to, subject, body);

        Console.WriteLine("[LOG] Email enviado");
    }
}
```

Manual:

```csharp
IEmailSender sender =
    new LoggingEmailSender(
        new SmtpEmailSender()
    );
```

Con DI nativo, decorar no siempre es tan directo como registrar dos servicios del mismo tipo, porque podés crear ciclos si no tenés cuidado.

Una forma manual simple:

```csharp
builder.Services.AddScoped<SmtpEmailSender>();

builder.Services.AddScoped<IEmailSender>(sp =>
{
    var inner = sp.GetRequiredService<SmtpEmailSender>();
    return new LoggingEmailSender(inner);
});
```

Uso en una clase:

```csharp
public class UsuarioService
{
    private readonly IEmailSender _emailSender;

    public UsuarioService(IEmailSender emailSender)
    {
        _emailSender = emailSender;
    }

    public async Task RegistrarAsync(string email)
    {
        await _emailSender.SendAsync(
            email,
            "Bienvenido",
            "Gracias por registrarte."
        );
    }
}
```

`UsuarioService` no sabe si el email sender tiene logging, cache, retry o palomas mensajeras.

Solo depende de:

```csharp
IEmailSender
```

Eso está muy bien.

## 17. Decoradores múltiples con DI

Podés hacer:

```csharp
builder.Services.AddScoped<SmtpEmailSender>();

builder.Services.AddScoped<IEmailSender>(sp =>
{
    IEmailSender inner = sp.GetRequiredService<SmtpEmailSender>();

    inner = new RetryEmailSender(inner);
    inner = new LoggingEmailSender(inner);

    return inner;
});
```

Donde:

```csharp
public class RetryEmailSender : IEmailSender
{
    private readonly IEmailSender _inner;

    public RetryEmailSender(IEmailSender inner)
    {
        _inner = inner;
    }

    public async Task SendAsync(string to, string subject, string body)
    {
        try
        {
            await _inner.SendAsync(to, subject, body);
        }
        catch
        {
            Console.WriteLine("Reintentando...");
            await _inner.SendAsync(to, subject, body);
        }
    }
}
```

El pipeline queda así:

```text
LoggingEmailSender
    RetryEmailSender
        SmtpEmailSender
```

Cuando llamás:

```csharp
await sender.SendAsync(...);
```

la ejecución pasa por capas.

## 18. Relación con principios SOLID

### SRP — Single Responsibility Principle

Cada decorador tiene una sola responsabilidad.

```text
EmailSender: enviar emails
LoggingEmailSender: loguear
RetryEmailSender: reintentar
ValidatingEmailSender: validar
```

No metés todo en una sola clase.

### OCP — Open/Closed Principle

Podés extender comportamiento sin modificar la clase original.

No tocás:

```csharp
SmtpEmailSender
```

Agregás:

```csharp
LoggingEmailSender
```

La clase está cerrada para modificación, pero abierta para extensión.

### LSP — Liskov Substitution Principle

El decorador debe poder sustituir al objeto original.

Si algo espera:

```csharp
IEmailSender
```

debería poder recibir:

```csharp
SmtpEmailSender
LoggingEmailSender
RetryEmailSender
```

sin enterarse.

### DIP — Dependency Inversion Principle

Los consumidores dependen de abstracciones:

```csharp
IEmailSender
```

no de concreciones:

```csharp
SmtpEmailSender
```

Eso hace posible decorar.

## 19. Decorador vs Herencia

Con herencia:

```csharp
public class EmailSenderConLogging : SmtpEmailSender
{
}
```

Problemas:

- queda acoplado a una implementación concreta;
- no podés combinar fácilmente;
- si necesitás `EmailSenderConLoggingYRetryYCache`, empieza la tragedia;
- obliga a diseñar jerarquías rígidas.

Con decorador:

```csharp
public class LoggingEmailSender : IEmailSender
{
    private readonly IEmailSender _inner;
}
```

Ventajas:

- combinable;
- flexible;
- desacoplado;
- testeable;
- respeta SRP;
- se puede aplicar en runtime.

## 20. Decorador vs Proxy

Son parecidos, pero no iguales.

### Decorador

Agrega comportamiento visible o funcional.

Ejemplos:

```text
logging
cache
compresión
validación
retry
cifrado
```

### Proxy

Controla el acceso al objeto.

Ejemplos:

```text
lazy loading
remote proxy
security proxy
virtual proxy
```

En la práctica, se parecen mucho en código:

```csharp
private readonly IServicio _inner;
```

La diferencia está en la intención de diseño.

Decorador:

> “Quiero agregar capacidades.”

Proxy:

> “Quiero controlar el acceso.”

---

## 21. Decorador vs Adapter

Adapter cambia una interfaz para hacer compatible algo incompatible.

Decorador mantiene la misma interfaz y agrega comportamiento.

### Adapter

```text
Interfaz A → Interfaz B
```

### Decorador

```text
IServicio → IServicio
```

Ejemplo:

```csharp
public class ServicioDecorado : IServicio
{
    private readonly IServicio _inner;
}
```

La clave es que el decorador implementa la misma abstracción que envuelve.

## 22. Decorador vs Middleware

En ASP.NET Core, el pipeline de middleware tiene una filosofía muy parecida.

Cada middleware envuelve al siguiente:

```text
MiddlewareLogging
    MiddlewareAuth
        MiddlewareRouting
            Endpoint
```

Ejemplo conceptual:

```csharp
app.Use(async (context, next) =>
{
    Console.WriteLine("Antes");
    await next();
    Console.WriteLine("Después");
});
```

Esto es casi un decorador aplicado a requests HTTP.

El request pasa por capas.

## 23. Ejemplo completo y limpio

```csharp
public interface IOperacion
{
    int Ejecutar(int x, int y);
}
```

Implementación base:

```csharp
public class Suma : IOperacion
{
    public int Ejecutar(int x, int y)
    {
        return x + y;
    }
}
```

Decorador de logging:

```csharp
public class OperacionConLogging : IOperacion
{
    private readonly IOperacion _inner;

    public OperacionConLogging(IOperacion inner)
    {
        _inner = inner;
    }

    public int Ejecutar(int x, int y)
    {
        Console.WriteLine($"Ejecutando operación con {x} y {y}");

        int resultado = _inner.Ejecutar(x, y);

        Console.WriteLine($"Resultado: {resultado}");

        return resultado;
    }
}
```

Decorador de validación:

```csharp
public class OperacionConValidacion : IOperacion
{
    private readonly IOperacion _inner;

    public OperacionConValidacion(IOperacion inner)
    {
        _inner = inner;
    }

    public int Ejecutar(int x, int y)
    {
        if (x < 0 || y < 0)
        {
            throw new ArgumentException("No se permiten números negativos.");
        }

        return _inner.Ejecutar(x, y);
    }
}
```

Uso:

```csharp
IOperacion operacion =
    new OperacionConLogging(
        new OperacionConValidacion(
            new Suma()
        )
    );

int resultado = operacion.Ejecutar(2, 3);

Console.WriteLine(resultado);
```

Salida:

```text
Ejecutando operación con 2 y 3
Resultado: 5
5
```

## 24. Orden de los decoradores

Esto es crucial.

No es lo mismo:

```csharp
new Logging(
    new Validacion(
        new Servicio()
    )
)
```

que:

```csharp
new Validacion(
    new Logging(
        new Servicio()
    )
)
```

En el primer caso:

```text
logging → validación → servicio
```

En el segundo:

```text
validación → logging → servicio
```

Ejemplo:

```csharp
IOperacion a =
    new OperacionConLogging(
        new OperacionConValidacion(
            new Suma()
        )
    );
```

Si hay error de validación, probablemente se loguea el intento.

En cambio:

```csharp
IOperacion b =
    new OperacionConValidacion(
        new OperacionConLogging(
            new Suma()
        )
    );
```

Si hay error de validación, ni siquiera llega al logging.

Ambas pueden estar bien. Depende de la intención.

---

## 25. Cuándo usar Decorador

Usalo cuando:

- querés agregar comportamiento sin modificar la clase original;
- querés combinar funcionalidades opcionales;
- querés evitar herencia explosiva;
- querés aplicar SRP;
- tenés una interfaz estable;
- necesitás capas como logging, cache, validación, retry, seguridad, métricas;
- querés envolver servicios en DI;
- querés construir pipelines.

---

## 26. Cuándo no usarlo

No lo uses si:

- la clase es trivial y no va a crecer;
- el comportamiento adicional es parte esencial del objeto;
- no hay una buena abstracción común;
- el orden de capas se vuelve confuso;
- el debugging se vuelve una cebolla insoportable;
- estás decorando todo “porque patrones”. Eso no es arquitectura, es cosplay.

---

## 27. Señal de que necesitás un Decorador

Cuando ves esto dentro de una clase:

```csharp
public void Ejecutar()
{
    Validar();
    LogInicio();
    MedirTiempo();
    VerificarPermisos();
    Reintentar();
    EjecutarOperacionReal();
    LogFin();
    Auditar();
}
```

Probablemente tenés varias responsabilidades mezcladas.

Podrías separar:

```text
AutorizacionDecorator
    ValidacionDecorator
        LoggingDecorator
            MedicionTiempoDecorator
                RetryDecorator
                    ServicioReal
```

## 29. Diferencia esencial con una simple llamada auxiliar

Podrías decir:

```csharp
logger.Log();
servicio.Ejecutar();
```

Sí, pero entonces el cliente tiene que saber que hay que loguear.

Con decorador, el cliente solo hace:

```csharp
servicio.Ejecutar();
```

El logging queda encapsulado en la composición.

La responsabilidad de “cómo se arma el objeto” queda en otro lugar: factory, DI container, configuración, etc.

## 30. Versión mental definitiva

El Decorador responde esta pregunta:

> ¿Cómo agrego responsabilidades adicionales a un objeto sin modificar su clase y sin crear una jerarquía absurda de herencia?

Respuesta:

> Lo envuelvo con otro objeto que implementa la misma interfaz y delega en el original.

Código mínimo:

```csharp
public class Decorador : IServicio
{
    private readonly IServicio _inner;

    public Decorador(IServicio inner)
    {
        _inner = inner;
    }

    public void Ejecutar()
    {
        // antes

        _inner.Ejecutar();

        // después
    }
}
```

Esa es la esencia.

Todo lo demás es decoración. Literalmente.
