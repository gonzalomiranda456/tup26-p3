# Clases y objetos en C#

## Introducción

Una **clase** es el mecanismo principal para definir tipos propios en C#. Describe la estructura y el comportamiento de un objeto: qué datos guarda y qué operaciones puede realizar.

Cuando escribimos `string nombre = "Ana"`, usamos un objeto de la clase `String`. Cuando hacemos `new List<int>()`, creamos un objeto de la clase `List<T>`. Aprender a diseñar nuestras propias clases es aprender a modelar el dominio del problema que queremos resolver.

***

## 1. Sintaxis básica

```csharp
// Forma mínima — clase pública en su propio archivo (convenio)
public class Persona {
    // campos privados (estado interno)
    private string nombre;
    private int    edad;

    // constructor
    public Persona(string nombre, int edad) {
        this.nombre = nombre;
        this.edad   = edad;
    }

    // propiedad pública
    public string Nombre => this.nombre;

    // método público
    public string Saludar() => $"Hola, soy {this.nombre} y tengo {this.edad} años.";
}

// Uso
var p = new Persona("Ana", 25);
Console.WriteLine(p.Saludar());
```

Los bloques de una clase suelen ordenarse así por convención:

```text
campos privados
constructores
propiedades
métodos públicos
métodos privados
```

## 2. Constructores

El constructor es el método especial que se invoca al crear un objeto con `new`. Su nombre es igual al de la clase y no tiene tipo de retorno.

### 2.1 Constructor explícito

```csharp
public class Producto {
    private string nombre;
    private decimal precio;

    public Producto(string nombre, decimal precio) {
        this.nombre = nombre;
        this.precio = precio;
    }
}
```

### 2.2 Constructor por defecto

Si no se declara ninguno, el compilador genera uno vacío automáticamente. En cuanto declarás al menos uno, el implícito desaparece y hay que declararlo a mano si se necesita:

```csharp
public class Configuracion {
    public int Timeout  { get; set; }
    public bool Activo  { get; set; }

    // Constructor por defecto explícito — necesario si hay otros constructores
    public Configuracion() { }

    public Configuracion(int timeout) {
        Timeout = timeout;
        Activo  = true;
    }
}
```

### 2.3 Constructor delegado — `this(...)`

Un constructor puede llamar a otro de la misma clase con `: this(...)`. Es la forma correcta de reusar lógica de inicialización:

```csharp
public class Conexion {
    public string Host    { get; }
    public int    Puerto  { get; }
    public bool   Segura  { get; }

    // Constructor completo
    public Conexion(string host, int puerto, bool segura) {
        Host   = host;
        Puerto = puerto;
        Segura = segura;
    }

    // Delegado — llama al completo con valores por defecto
    public Conexion(string host) : this(host, 443, true) { }

    public Conexion() : this("localhost") { }
}

// Uso
var c1 = new Conexion("db.ejemplo.com", 5432, false);
var c2 = new Conexion("api.ejemplo.com");  // puerto 443, segura
var c3 = new Conexion();                   // localhost, 443, segura
```

### 2.4 Primary constructor (C# 12+) — la forma moderna

Los parámetros del `primary constructor` (Constructor principal) no son propiedades ni campos. Son variables disponibles en todo el cuerpo de la clase.\
Es la forma más compacta de declarar una clase con estado:

```csharp
// Forma clásica
public class Punto {
    private readonly double x;
    private readonly double y;

    public Punto(double x, double y) { this.x = x; this.y = y; }
    public double X => this.x;
    public double Y => this.y;
}

// Con primary constructor — equivalente y mucho más breve
public class Punto(double x, double y) {
    public double X => x;
    public double Y => y;

    public double Distancia => Math.Sqrt(x * x + y * y);
}

// Uso idéntico
var p = new Punto(3, 4);
Console.WriteLine(p.Distancia);  // 5
```

> El primary constructor no crea propiedades automáticamente (eso es el `record`). Los parámetros son variables disponibles en todo el cuerpo — podés usarlos en campos, propiedades y métodos.

### 2.5 Inicializador de objeto

Permite asignar propiedades después de la construcción, sin necesidad de un constructor con todos los parámetros:

```csharp
public class Empleado {
    public string Nombre        { get; set; } = "";
    public string Departamento  { get; set; } = "";
    public decimal Salario      { get; set; }
}

// Uso con inicializador — conciso y legible
var e = new Empleado {
    Nombre       = "Carlos",
    Departamento = "IT",
    Salario      = 95_000m
};
```

## 3. Propiedades

Las propiedades son la forma idiomática de exponer el estado de un objeto en C#. Combinan la seguridad de los campos privados con la comodidad de acceso directo.

### 3.1 Propiedad completa — getter y setter explícitos

```csharp
public class Temperatura {
    private double celsius;
    public double Celsius {
        get { return this.celsius; }
        set {
            if (value < -273.15)
                throw new ArgumentOutOfRangeException(nameof(value), "Bajo el cero absoluto.");
            this.celsius = value;
        }
    }

    // Propiedad calculada — sólo getter
    public double Fahrenheit {
        get { return this.celsius * 9 / 5 + 32; }
    }
}
```

### 3.2 Auto-propiedad — forma breve

Cuando no se necesita lógica en getter/setter, el compilador genera el campo privado automáticamente:

```csharp
public class Articulo {
    public Guid    Id     { get; }        // sólo lectura — se asigna en constructor
    public string  Nombre { get; set; }   // lectura y escritura
    public decimal Precio { get; set; }
    public int     Stock  { get; private set; }   // escritura sólo desde dentro
    public string  Codigo { get; init; }  // init-only — asignable en new{} pero no después
}
```

### 3.3 Valor por defecto en propiedades

```csharp
public class Configuracion {
    public string  Nombre    { get; set; } = "Sin nombre";
    public bool    Activo    { get; set; } = true;
    public int     Timeout   { get; set; } = 30;
    public List<string> Tags { get; set; } = [];
}
```

### 3.4 Expresión de propiedad — `=>`

Para propiedades calculadas de una sola expresión:

```csharp
public class Rectangulo(double ancho, double alto) {
    public double Ancho      => ancho;
    public double Alto       => alto;
    public double Area       => ancho * alto;
    public double Perimetro  => 2 * (ancho + alto);
    public bool   EsCuadrado => ancho == alto;
}
```

### 3.5 Propiedades indexadas

Las propiedades indexadas permiten acceder a datos con una sintaxis parecida a la de un array o diccionario: `objeto[indice]`. En C# se implementan con el modificador `this` dentro de la clase.

Son útiles cuando una clase encapsula una colección interna o cuando queremos exponer acceso por clave sin mostrar la estructura de almacenamiento real.

#### 3.5.1 Indexador simple por posición

```csharp
public class Agenda {
    private readonly List<string> contactos = [];

    public string this[int index] {
        get => this.contactos[index];
        set => this.contactos[index] = value;
    }

    public int Count => this.contactos.Count;

    public void Agregar(string nombre) => this.contactos.Add(nombre);
}

var agenda = new Agenda();
agenda.Agregar("Ana");
agenda.Agregar("Carlos");

Console.WriteLine(agenda[0]);   // Ana
agenda[1] = "Carla";
```

#### 3.5.2 Indexador por clave

También puede recibir parámetros más expresivos, por ejemplo una cadena:

```csharp
public class ConfiguracionApp {
    private readonly Dictionary<string, string> valores = new();

    public string? this[string clave] {
        get => this.valores.TryGetValue(clave, out var valor) ? valor : null;
        set {
            if (value is null) this.valores.Remove(clave);
            else this.valores[clave] = value;
        }
    }
}

var config = new ConfiguracionApp();
config["modo"] = "desarrollo";
config["tema"] = "oscuro";

Console.WriteLine(config["modo"]);   // desarrollo
```

#### 3.5.3 Cuándo usarlas

| Caso de uso                                   | Conviene un indexador |
| --------------------------------------------- | --------------------- |
| Acceso posicional a elementos                 | Sí                    |
| Acceso por clave o nombre                     | Sí                    |
| La clase representa una colección o mapa      | Sí                    |
| El acceso necesita validación o lógica simple | Sí                    |
| Exponer muchos datos no relacionados          | No                    |

Regla práctica: si la operación se lee naturalmente como `objeto[algo]`, un indexador suele ser la mejor opción.

### 3.6 `required` — propiedad obligatoria (C# 11+)

Obliga a que la propiedad se asigne en el inicializador. El compilador lo verifica en tiempo de compilación:

```csharp
public class Usuario {
    public required string Nombre { get; init; }
    public required string Email  { get; init; }
    public string?  Telefono      { get; init; }
}

// El compilador exige Nombre y Email
var u = new Usuario { Nombre = "Ana", Email = "ana@ejemplo.com" };

// Error de compilación — falta Email
var u2 = new Usuario { Nombre = "Carlos" };  // ❌
```

### 3.7 Tabla resumen de variantes

| Sintaxis                  | Acceso                           | Cuándo usarla                      |
| ------------------------- | -------------------------------- | ---------------------------------- |
| `{ get; set; }`           | Libre lectura y escritura        | Estado mutable                     |
| `{ get; private set; }`   | Lectura libre, escritura interna | Estado con control                 |
| `{ get; }`                | Solo lectura                     | Asignable solo en constructor      |
| `{ get; init; }`          | Solo lectura post-construcción   | DTOs, objetos con inicializador    |
| `=> expresion`            | Solo lectura calculada           | Propiedades derivadas              |
| `required { get; init; }` | Obligatoria en `new{}`           | Propiedades que no pueden omitirse |

## 4. Métodos

### 4.1 Definición

```csharp
public class Calculadora {
    // Método con cuerpo
    public int Sumar(int a, int b) {
        return a + b;
    }

    // Método con expresión — forma breve para una sola expresión
    public int Restar(int a, int b) => a - b;

    // Método void
    public void Imprimir(string msg) => Console.WriteLine(msg);

    // Parámetros opcionales
    public double Potencia(double baseValor, double exp = 2) => Math.Pow(baseValor, exp);

    // Parámetros nombrados (se usan en la llamada)
    public string Formatear(string texto, bool mayus = false, bool trim = false) {
        if (trim)  texto = texto.Trim();
        if (mayus) texto = texto.ToUpper();
        return texto;
    }
}

// Uso
var calc = new Calculadora();
calc.Sumar(3, 4);
calc.Potencia(2);                                     // exp = 2 por defecto
calc.Potencia(2, 10);
calc.Formatear("  hola  ", trim: true, mayus: true);  // parámetros nombrados
```

### 4.2 Modificadores de acceso — alcance

| Modificador          | Visible desde                                           |
| -------------------- | ------------------------------------------------------- |
| `public`             | En cualquier parte                                      |
| `private`            | Sólo dentro de la clase (default para métodos y campos) |
| `protected`          | Dentro de la clase y sus subclases                      |
| `internal`           | Dentro del mismo ensamblado (proyecto)                  |
| `protected internal` | Subclases o mismo ensamblado                            |
| `private protected`  | Subclases dentro del mismo ensamblado                   |

```csharp
public class Cuenta {
    public    decimal Saldo       { get; private set; }
    private   string  pin;
    protected string  TitularId   { get; set; } = "";
    internal  string  AuditoriaId { get; set; } = "";

    public void Depositar(decimal monto) {
        ValidarMonto(monto);   // llama al método privado
        Saldo += monto;
    }

    private void ValidarMonto(decimal monto) {
        if (monto <= 0) {
            throw new ArgumentException("El monto debe ser positivo.");
        }
    }
}
```

### 4.3 Sobrecarga de métodos

Múltiples métodos con el mismo nombre pero distintos parámetros (distinto tipo, cantidad u orden). El compilador elige cuál usar según los argumentos:

```csharp
public class Logger {
    public void Log(string mensaje)
        => Console.WriteLine($"[INFO] {mensaje}");

    public void Log(string mensaje, Exception ex)
        => Console.WriteLine($"[ERROR] {mensaje}: {ex.Message}");

    public void Log(string mensaje, string nivel)
        => Console.WriteLine($"[{nivel.ToUpper()}] {mensaje}");

    public void Log(Exception ex)
        => Console.WriteLine($"[EXCEPTION] {ex.GetType().Name}: {ex.Message}");
}

var log = new Logger();
log.Log("Inicio del sistema");
log.Log("Operación fallida", new InvalidOperationException("sin datos"));
log.Log("Conexión lenta", "WARNING");
```

> La sobrecarga se resuelve en tiempo de **compilación**. No es lo mismo que la sobreescritura (override), que se resuelve en tiempo de ejecución.

## 5. Clases y métodos estáticos

Un miembro `static` pertenece a la **clase**, no a las instancias. Existe uno solo sin importar cuántos objetos se creen.

### 5.1 Métodos estáticos

```csharp
public class Temperatura {
    public double Celsius { get; }

    public Temperatura(double celsius) => Celsius = celsius;

    // Métodos de instancia
    public double ToFahrenheit() => Celsius * 9 / 5 + 32;
    public double ToKelvin()     => Celsius + 273.15;

    // Métodos estáticos — factoría semántica
    public static Temperatura DesdeFahrenheit(double f) => new((f - 32) * 5 / 9);
    public static Temperatura DesdeKelvin(double k) => new(k - 273.15);
    
    public static bool TryParse(string s, out Temperatura temperatura) {
        if (double.TryParse(s, out double c)) {
            temperatura = new Temperatura(c);
            return true;
        }
        temperatura = null!;
        return false;
    }

    public override string ToString() => $"{Celsius:F1}°C";
}

// Uso
var t1 = new Temperatura(100);
var t2 = Temperatura.DesdeFahrenheit(212);
var t3 = Temperatura.DesdeKelvin(373.15);
if (Temperatura.TryParse("36.5", out var t4)) {
    Console.WriteLine(t4);  // 36.5°C
}
```

### 5.2 Campos y propiedades estáticos

```csharp
public class Contador {
    private static int total = 0;

    public int Id { get; }

    public Contador() {
        total++;
        Id = total;
    }

    public static int Total => total;
    public static void Reset() => total = 0;
}

var a = new Contador();   // Id = 1
var b = new Contador();   // Id = 2
var c = new Contador();   // Id = 3
Console.WriteLine(Contador.Total);  // 3
```

### 5.3 Clase estática

Una clase marcada como `static` no puede instanciarse y sólo puede contener miembros estáticos. Ideal para utilidades y helpers:

```csharp
public static class MathEx {
    public const double TauRadianes = 2 * Math.PI;

    public static double Clamp(double valor, double min, double max)
        => Math.Max(min, Math.Min(max, valor));

    public static bool EsPrimo(int n) {
        if (n < 2) return false;
        for (int i = 2; i * i <= n; i++)
            if (n % i == 0) return false;
        return true;
    }

    public static IEnumerable<int> Primos(int hasta) => Enumerable.Range(2, hasta - 1).Where(EsPrimo);
}

// Uso — nunca se instancia
double x = MathEx.Clamp(1.5, 0, 1);
bool esPrimo = MathEx.EsPrimo(17);
```

### 5.4 Constructor estático

Se ejecuta **una sola vez**, antes del primer uso de la clase. No recibe parámetros:

```csharp
public class AppConfig {
    public static readonly string ConnectionString;
    public static readonly string Ambiente;

    // Se ejecuta una sola vez al cargar la clase
    static AppConfig() {
        Ambiente         = Environment.GetEnvironmentVariable("ASPNETCORE_ENVIRONMENT") ?? "Development";
        ConnectionString = Ambiente == "Production"
            ? "Server=prod-db;..."
            : "Server=localhost;...";
    }
}
```

## 6. Sobrecarga de operadores

C# permite definir el comportamiento de los operadores estándar (`+`, `-`, `==`, `<`, etc.) para nuestras clases:

```csharp
public class Vector2D(double x, double y) {
    public double X => x;
    public double Y => y;

    // Operadores aritméticos
    public static Vector2D operator +(Vector2D a, Vector2D b)
        => new(a.X + b.X, a.Y + b.Y);

    public static Vector2D operator -(Vector2D a, Vector2D b)
        => new(a.X - b.X, a.Y - b.Y);

    public static Vector2D operator *(Vector2D v, double escalar)
        => new(v.X * escalar, v.Y * escalar);

    public static Vector2D operator *(double escalar, Vector2D v)
        => v * escalar;   // reusar el de arriba

    public static Vector2D operator -(Vector2D v)
        => new(-v.X, -v.Y);   // operador unario

    // Igualdad — si sobreescribís ==, debés sobreescribir != también
    public static bool operator ==(Vector2D a, Vector2D b)
        => a.X == b.X && a.Y == b.Y;

    public static bool operator !=(Vector2D a, Vector2D b)
        => !(a == b);

    // Comparación de magnitud
    public static bool operator >(Vector2D a, Vector2D b)
        => a.Magnitud > b.Magnitud;

    public static bool operator <(Vector2D a, Vector2D b)
        => a.Magnitud < b.Magnitud;

    // Propiedad calculada
    public double Magnitud => Math.Sqrt(X * X + Y * Y);

    // Siempre sobreescribir Equals y GetHashCode junto con ==
    public override bool Equals(object? obj)
        => obj is Vector2D v && this == v;

    public override int GetHashCode() => HashCode.Combine(X, Y);

    public override string ToString() => $"({X:F2}, {Y:F2})";
}

// Uso natural
var v1 = new Vector2D(3, 4);
var v2 = new Vector2D(1, 2);

var suma   = v1 + v2;      // (4.00, 6.00)
var doble  = v1 * 2;       // (6.00, 8.00)
var doble2 = 2 * v1;       // ídem
bool mayor = v1 > v2;      // true — magnitud 5 > magnitud 2.24
```

## 7. Conversiones — `implicit` y `explicit`

Se pueden definir conversiones entre tipos propios, o desde/hacia tipos del sistema:

```csharp
public class Porcentaje(double valor) {
    public double Valor => valor;

    // Conversión implícita desde double — no requiere cast
    // Usarla cuando la conversión es siempre segura y natural
    public static implicit operator Porcentaje(double d) => new(d);

    // Conversión implícita hacia double
    public static implicit operator double(Porcentaje p) => p.Valor;

    // Conversión implícita desde int
    public static implicit operator Porcentaje(int i) => new(i);

    public override string ToString() => $"{Valor:F1}%";
}

// Uso
Porcentaje iva       = 21.0;    // implicit desde double
Porcentaje retencion = 10;     // implicit desde int
double     valor     = iva;    // implicit hacia double

double precio  = 1000;
double conIva  = precio * (1 + iva / 100);   // iva se convierte a double implícito
```

Para conversiones que pueden perder información o fallar, usar `explicit`:

```csharp
public class Celsius(double grados) {
    public double Grados => grados;

    // Conversión explícita — requiere cast (puede perder info de Fahrenheit)
    public static explicit operator Celsius(double fahrenheit)
        => new((fahrenheit - 32) * 5 / 9);

    public static implicit operator double(Celsius c) => c.Grados;

    public override string ToString() => $"{Grados:F1}°C";
}

// Uso
Celsius t = (Celsius)212.0;   // requiere cast explícito
double g  = t;                // implícito — siempre seguro
```

## 8. Herencia

La **herencia** permite que una clase *derivada* adquiera los miembros de una clase *base*, extendiendo o especializando su comportamiento. En C# la herencia de clases es siempre **simple**: una sola clase base directa.

La relación que modela es **"es un"**: un `Perro` *es un* `Animal`, un `OperadorBinario` *es un* `Nodo`. Cuando esa frase no tiene sentido natural, la herencia probablemente no es la herramienta correcta.

### 8.1 Sintaxis básica

```csharp
// Clase base
public class Animal {
    public string Nombre { get; }
    public int    Edad   { get; }

    public Animal(string nombre, int edad) {
        this.Nombre = nombre;
        this.Edad   = edad;
    }

    public virtual string HacerSonido() => "...";

    public override string ToString() => $"{this.Nombre} ({this.Edad} años)";
}

// Clase derivada — el operador ':' indica la clase base
public class Perro : Animal {
    public string Raza { get; }

    // El constructor derivado inicializa la clase base con 'base(...)'
    public Perro(string nombre, int edad, string raza)
        : base(nombre, edad) {
        this.Raza = raza;
    }

    public override string HacerSonido() => "Guau!";

    public override string ToString() => $"{base.ToString()} — {this.Raza}";
}

// Uso
var d = new Perro("Rex", 3, "Labrador");
Console.WriteLine(d.Nombre);          // heredado de Animal
Console.WriteLine(d.HacerSonido());   // "Guau!" — versión propia
Console.WriteLine(d);                 // "Rex (3 años) — Labrador"
```

### 8.2 Cadena de constructores — `base(...)`

El constructor derivado **siempre** inicializa la base, explícita o implícitamente. Si la base no tiene constructor sin parámetros, la llamada a `base(...)` es obligatoria. El orden de ejecución va de la clase más alta a la más derivada:

```csharp
public class Vehiculo {
    public string Marca { get; }
    public int    Anio  { get; }

    public Vehiculo(string marca, int año) {
        this.Marca = marca;
        this.Anio  = año;
    }
}

public class Automovil : Vehiculo {
    public int Puertas { get; }

    public Automovil(string marca, int año, int puertas) : base(marca, año) { // 1° se ejecuta Vehiculo(...)
        this.Puertas = puertas;                                               // 2° luego este cuerpo
    }
}

public class AutoElectrico : Automovil {
    public int AutonomiaKm { get; }

    public AutoElectrico(string marca, int año, int autonomia)
        : base(marca, año, 4) {            // 1° Vehiculo, 2° Automovil, 3° aquí
        this.AutonomiaKm = autonomia;
    }
}
```

### 8.3 `protected` — estado accesible en subclases

`protected` hace que un miembro sea invisible fuera de la jerarquía pero accesible en todas las clases derivadas. Es el nivel de acceso correcto para estado que las subclases necesitan manipular directamente:

```csharp
public class Cuenta {
    protected decimal saldo;   // visible en subclases, invisible fuera

    public decimal Saldo => this.saldo;

    public Cuenta(decimal saldoInicial) {
        this.saldo = saldoInicial;
    }

    public virtual void Depositar(decimal monto) {
        this.saldo += monto;
    }
}

public class CuentaAhorro : Cuenta {
    private decimal tasaInteres;

    public CuentaAhorro(decimal saldoInicial, decimal tasa)
        : base(saldoInicial) {
        this.tasaInteres = tasa;
    }

    public void AplicarInteres() {
        this.saldo += this.saldo * this.tasaInteres;  // accede a saldo porque es protected
    }
}
```

### 8.4 `base` — invocar miembros del padre

La palabra clave `base` llama a la implementación de la clase base desde una clase derivada. Es útil cuando el `override` quiere *extender* el comportamiento en lugar de reemplazarlo por completo:

```csharp
public class Empleado {
    public string  Nombre  { get; }
    public decimal Salario { get; protected set; }

    public Empleado(string nombre, decimal salario) {
        this.Nombre  = nombre;
        this.Salario = salario;
    }

    public virtual string Descripcion()
        => $"{this.Nombre} — Salario: {this.Salario:C}";
}

public class Gerente : Empleado {
    public string Area { get; }

    public Gerente(string nombre, decimal salario, string area)
        : base(nombre, salario) {
        this.Area = area;
    }

    public override string Descripcion()
        => $"{base.Descripcion()} | Área: {this.Area}";  // reutiliza + agrega
}

var g = new Gerente("Laura", 150_000m, "Tecnología");
Console.WriteLine(g.Descripcion());
// "Laura — Salario: $150.000,00 | Área: Tecnología"
```

### 8.5 `sealed` — bloquear la herencia

`sealed` en una clase impide que se derive de ella. También puede aplicarse a un `override` individual para bloquear sobreescrituras en niveles más profundos de la jerarquía:

```csharp
// Clase completamente sellada — nadie puede heredar de ella
public sealed class Id {
    public Guid Valor { get; } = Guid.NewGuid();
    public override string ToString() => this.Valor.ToString("N");
}

// Método sellado — Circulo permite herencia pero bloquea la sobreescritura de Area()
public class Figura {
    public virtual double Area() => 0;
}

public class Circulo : Figura {
    private double radio;

    public Circulo(double radio) {
        this.radio = radio;
    }

    public sealed override double Area() => Math.PI * this.radio * this.radio;
}

// Error de compilación — Area() está sellado en Circulo
// public class CirculoColoreado : Circulo {
//     public override double Area() => ...;  // ❌
// }
```

### 8.6 Tabla — cuándo usar herencia

| Situación                                                | ¿Heredar?          |
| -------------------------------------------------------- | ------------------ |
| La clase derivada **es un** tipo de la clase base        | ✅ Sí               |
| Sólo se quiere reutilizar código sin relación conceptual | ❌ Usar composición |
| Hay comportamiento compartido que puede sobreescribirse  | ✅ Sí               |
| La clase base ya tiene `sealed`                          | ❌ Imposible        |
| Se necesita "herencia múltiple"                          | ❌ Usar interfaces  |

> En C# sólo existe herencia **simple de clase**. Si un tipo necesita comportamiento de múltiples fuentes, se usan interfaces.

## 9. Polimorfismo

El **polimorfismo** permite tratar objetos de distintos tipos derivados a través de una referencia del tipo base, ejecutando en cada caso la implementación correcta. Es el mecanismo que hace útil a la herencia en la práctica.

La idea central: el código que llama a un método no necesita saber qué tipo concreto tiene el objeto. Lo decide el runtime en cada ejecución.

### 9.1 `virtual` y `override` — despacho dinámico

`virtual` marca un método como sobreescribible. `override` reemplaza esa implementación en la clase derivada. El método ejecutado se determina en **tiempo de ejecución** según el tipo real del objeto, no el tipo declarado de la variable:

```csharp
public class Notificador {
    public virtual void Enviar(string mensaje)
        => Console.WriteLine($"[BASE] {mensaje}");
}

public class NotificadorEmail : Notificador {
    public override void Enviar(string mensaje)
        => Console.WriteLine($"[EMAIL] → {mensaje}");
}

public class NotificadorSMS : Notificador {
    public override void Enviar(string mensaje)
        => Console.WriteLine($"[SMS] → {mensaje}");
}

// El método no sabe ni le importa el tipo concreto
void NotificarATodos(List<Notificador> lista, string msg) {
    foreach (var n in lista)
        n.Enviar(msg);   // despacho dinámico: cada objeto ejecuta su override
}

NotificarATodos(
    new List<Notificador> { new NotificadorEmail(), new NotificadorSMS() },
    "Nuevo pedido"
);
// [EMAIL] → Nuevo pedido
// [SMS]   → Nuevo pedido
```

### 9.2 `new` — ocultar sin polimorfismo

`new` en un método derivado **oculta** el miembro base sin participar del despacho dinámico. La versión ejecutada depende del tipo *declarado* de la variable, no del tipo real. Es una diferencia crítica respecto de `override`:

```csharp
public class Base {
    public virtual void A() => Console.WriteLine("Base.A (virtual)");
    public        void B() => Console.WriteLine("Base.B (no virtual)");
}

public class Derivada : Base {
    public override void A() => Console.WriteLine("Derivada.A (override)");
    public new      void B() => Console.WriteLine("Derivada.B (new)");
}

Base obj = new Derivada();

obj.A();   // "Derivada.A (override)" — tipo real gana    ← polimorfismo
obj.B();   // "Base.B (no virtual)"   — tipo declarado gana  ← NO es polimorfismo
```

> Preferir siempre `override` sobre `new`. Si el código lleva a necesitar `new`, normalmente es señal de un diseño incorrecto en la jerarquía.

### 9.3 Ejemplo completo — AST de expresiones

El AST de expresiones matemáticas es el ejemplo canónico de polimorfismo: un único `Evaluar()` recorre un árbol sin saber el tipo de cada nodo.

```csharp
// Clase base abstracta — toda expresión sabe evaluarse
public abstract class Nodo {
    public abstract double Evaluar();
    public abstract override string ToString();
}

// Hoja: número literal — no tiene hijos
public class Numero(double valor) : Nodo {
    public override double Evaluar() => valor;

    public override string ToString() => valor.ToString("G");
}

// Nodo intermedio abstracto — define la estructura de un operador binario
public abstract class OperadorBinario(Nodo izquierdo, Nodo derecho) : Nodo {
    protected Nodo Izquierdo { get; } = izquierdo;
    protected Nodo Derecho   { get; } = derecho;

    // Cada operador concreto define su símbolo, pero el método ToString() lo reutiliza sin conocer el tipo específico
    public abstract string Simbolo { get; }

    // ToString reutiliza Simbolo sin conocer el tipo concreto
    // Las subclases heredan este método sin necesidad de redefinirlo
    public override string ToString() => $"({this.Izquierdo} {this.Simbolo} {this.Derecho})";
}

// Operadores concretos — cada uno sólo define su símbolo y su operación
public class Suma(Nodo izquierdo, Nodo derecho) : OperadorBinario(izquierdo, derecho) {
    public override string Simbolo   => "+";
    public override double Evaluar() => this.Izquierdo.Evaluar() + this.Derecho.Evaluar();
}

public class Resta(Nodo izquierdo, Nodo derecho) : OperadorBinario(izquierdo, derecho) {
    public override string Simbolo   => "-";
    public override double Evaluar() => this.Izquierdo.Evaluar() - this.Derecho.Evaluar();
}

public class Producto(Nodo izquierdo, Nodo derecho) : OperadorBinario(izquierdo, derecho) {
    public override string Simbolo   => "*";
    public override double Evaluar() => this.Izquierdo.Evaluar() * this.Derecho.Evaluar();
}

public class Division(Nodo izquierdo, Nodo derecho) : OperadorBinario(izquierdo, derecho) {
    public override string Simbolo => "/";
    public override double Evaluar() {
        double divisor = this.Derecho.Evaluar();
        if (divisor == 0) throw new DivideByZeroException("División por cero.");
        return this.Izquierdo.Evaluar() / divisor;
    }
}
```

El polimorfismo se hace visible cuando se recorre una lista de nodos heterogéneos:

```csharp
// (3 + 4) * (10 - 2)
Nodo expr = new Producto(
    new Suma (new Numero(3), new Numero(4)),
    new Resta(new Numero(10), new Numero(2))
);

Console.WriteLine(expr);            // ((3 + 4) * (10 - 2))
Console.WriteLine(expr.Evaluar());  // 56

// Lista heterogénea — todas las referencias son de tipo Nodo
List<Nodo> expresiones = [
    new Suma     (new Numero(2), new Numero(3)),
    new Resta    (new Numero(9), new Numero(4)),
    new Producto (new Numero(3), new Numero(3)),
    new Division (new Numero(8), new Numero(4)),
];

foreach (Nodo n in expresiones)
    Console.WriteLine($"{n} = {n.Evaluar()}");

// (2 + 3) = 5
// (9 - 4) = 5
// (3 * 3) = 9
// (8 / 4) = 2
```

El `foreach` llama a `Evaluar()` y `ToString()` sin ningún `if` ni `switch`: el runtime determina en cada iteración qué implementación ejecutar. Eso es polimorfismo.

### 9.4 Verificación de tipo — `is`, `as`, pattern matching

A veces se necesita acceder a miembros específicos de un tipo derivado. C# ofrece tres herramientas para esto:

```csharp
Nodo nodo = new Suma(new Numero(1), new Numero(2));

// 'is' con pattern variable — verifica tipo y declara variable en una sola expresión
if (nodo is OperadorBinario op) {
    Console.WriteLine($"Es un operador: {op.Simbolo}");   // "+"
}

// 'as' — intenta convertir; devuelve null si falla, no lanza excepción
OperadorBinario? opONull = nodo as OperadorBinario;
if (opONull is not null) {
    Console.WriteLine(opONull.Evaluar());
}

// switch con patrones — la forma más idiomática en C# moderno
string DescribirNodo(Nodo n) => n switch {
    Numero      num => $"Número: {num.Evaluar()}",
    Suma        s   => $"Suma de dos términos",
    Division    d   => $"División (cuidado con el cero)",
    OperadorBinario op => $"Operador '{op.Simbolo}'",  // caso genérico
    _               => "Nodo desconocido"
};
```

### 9.5 Tabla — modificadores de sobreescritura

| Modificador       | Dónde va       | Efecto                                         | ¿Polimórfico? |
| ----------------- | -------------- | ---------------------------------------------- | ------------- |
| `virtual`         | Clase base     | Permite sobreescritura en derivadas            | Sí            |
| `override`        | Clase derivada | Reemplaza la implementación base               | Sí            |
| `abstract`        | Clase base     | Sin implementación; obliga a sobreescribir     | Sí            |
| `new`             | Clase derivada | Oculta el miembro base sin despacho dinámico   | No            |
| `sealed override` | Clase derivada | Override final; la subclase no puede cambiarlo | Sí (último)   |

## 10. Clases abstractas

Una clase `abstract` no puede instanciarse directamente; se usa como base para otras clases.\
Es útil cuando varias implementaciones comparten una estructura común o parte del comportamiento, pero cada una completa su propia lógica.\
Las clases abstractas permiten definir el contrato sin cerrar del todo la implementación, y las subclases concretas deben completar los métodos abstractos.

```csharp
public abstract class Figura {
    // Propiedad concreta — disponible en todas las subclases
    public string Color { get; set; } = "negro";

    // Métodos abstractos — cada subclase DEBE implementarlos
    public abstract double Area();
    public abstract double Perimetro();

    // Método concreto — comportamiento compartido, las subclases pueden sobreescribirlo
    public virtual string Describir()
        => $"{GetType().Name} de color {Color}: área={Area():F2}, perímetro={Perimetro():F2}";
}

public class Circulo(double radio) : Figura {
    public double Radio => radio;

    public override double Area()      => Math.PI * radio * radio;
    public override double Perimetro() => 2 * Math.PI * radio;
}

public class Rectangulo(double ancho, double alto) : Figura {
    public override double Area()      => ancho * alto;
    public override double Perimetro() => 2 * (ancho + alto);

    // Sobreescribe el método concreto
    public override string Describir()
        => base.Describir() + $" [{ancho}×{alto}]";
}

// Uso polimórfico
List<Figura> figuras = [new Circulo(5), new Rectangulo(4, 6)];
foreach (var f in figuras)
    Console.WriteLine(f.Describir());
```

### Abstracta vs. interfaz — cuándo usar cada una

| Característica              | Clase abstracta | Interfaz             |
| --------------------------- | --------------- | -------------------- |
| Puede tener estado (campos) | Sí              | No                   |
| Puede tener implementación  | Sí              | Sí (default members) |
| Herencia múltiple           | No              | Sí                   |
| Expresa "es un"             | Sí              | No necesariamente    |
| Expresa "puede hacer"       | Indirecto       | Sí                   |

Regla práctica: si los tipos comparten **estado y comportamiento base**, usá clase abstracta. Si sólo comparten **contrato de comportamiento**, usá interfaz.

## 11. Inmutabilidad

Una clase **inmutable** no permite modificar su estado después de la construcción.\
Es más segura en concurrencia, más fácil de razonar y mejor para caching.

### 11.1 Clase inmutable con `readonly` y propiedades `init`

```csharp
public sealed class Dinero {
    public decimal Monto  { get; }
    public string  Moneda { get; }

    public Dinero(decimal monto, string moneda) {
        if (monto < 0) throw new ArgumentException("El monto no puede ser negativo.");
        Monto  = monto;
        Moneda = moneda.ToUpper();
    }

    // Los "mutadores" retornan un nuevo objeto en lugar de modificar el actual
    public Dinero Sumar(Dinero otro) {
        if (Moneda != otro.Moneda) {
            throw new InvalidOperationException("No se pueden sumar monedas distintas.");
        }
        return new Dinero(Monto + otro.Monto, Moneda);
    }
    public Dinero Multiplicar(decimal factor) => new(Monto * factor, Moneda);
    
    // Sobrecarga de operadores para facilitar la sintaxis
    public static Dinero operator +(Dinero a, Dinero b)  => a.Sumar(b);
    public static Dinero operator -(Dinero a, Dinero b)  => a + b * -1.0m;
    public static Dinero operator *(Dinero d, decimal f) => d.Multiplicar(f);

    public override string ToString() => $"{Monto:N2} {Moneda}";
}

// Uso
var precio    = new Dinero(100, "ARS");
var descuento = precio * 0.10m;
var impuesto  = (precio - descuento) * 0.21m;
var total     = precio + impuesto - descuento;
Console.WriteLine(total);                       // 108,90 ARS
// precio sigue siendo 100 ARS — nunca se modificó
```

### 11.2 `sealed` — prohibir la herencia

`sealed` impide que otras clases hereden de la nuestra. Útil para clases que no deben extenderse:

```csharp
public sealed class Id {
    public Guid Valor { get; } = Guid.NewGuid();
    public override string ToString() => Valor.ToString("N");
}
```

## 12. Clases como DTOs

Un **DTO** (Data Transfer Object) es un objeto cuyo único propósito es transportar datos entre capas del sistema. No tiene comportamiento de negocio.

### 12.1 DTO clásico con propiedades

```csharp
// DTO de entrada (request)
public class CrearProductoRequest {
    public required string  Nombre      { get; init; }
    public required decimal Precio      { get; init; }
    public required int     Stock       { get; init; }
    public string?          Descripcion { get; init; }
    public string           Categoria   { get; init; } = "General";
}

// DTO de salida (response)
public class ProductoResponse {
    public required int     Id          { get; init; }
    public required string  Nombre      { get; init; }
    public required decimal Precio      { get; init; }
    public required int     Stock       { get; init; }
    public required bool    Disponible  { get; init; }
}
```

### 12.2 DTO con `record` — la alternativa moderna

Para DTOs simples, `record` es la opción preferida: más breve, con igualdad por valor y `ToString` generado:

```csharp
// Posicional
record CrearProductoRequest(string Nombre, decimal Precio, int Stock);

// Con cuerpo para propiedades opcionales y valores por defecto
record ProductoResponse(int Id, string Nombre, decimal Precio, int Stock) {
    public bool Disponible => Stock > 0;
}
```

Los `record` se comportan como un `class` con varias características adicionales: igualdad por valor, `ToString` automático y un constructor primario que asigna las propiedades.\
Son ideales para DTOs porque reducen el boilerplate y dejan claro que el objeto sólo transporta datos.

Se puede usar `record class` para enfatizar que es una clase, o `record struct` para obtener semántica de valor con la ergonomía de un `record`.\
En general, `record` por sí solo es suficiente para la mayoría de los casos.

Con `with` expressions, los `record` permiten crear nuevas instancias modificando sólo algunas propiedades, lo que es muy útil para DTOs:

```csharp
var producto = new ProductoResponse(1, "Laptop", 1500m, 10);
var actualizado = producto with { Precio = 1400m, Stock = 5 };
Console.WriteLine(producto);     // ProductoResponse { Id = 1, Nombre = Laptop, Precio = 1500, Stock = 10 }
Console.WriteLine(actualizado);  
```

Otra ventaja de los `record` es que su igualdad se basa en el valor de las propiedades, no en la referencia. Esto es especialmente útil para pruebas unitarias y comparaciones de DTOs:

```csharp
var p1 = new ProductoResponse(1, "Laptop", 1500m, 10);
var p2 = new ProductoResponse(1, "Laptop", 1500m, 10);
Console.WriteLine(p1 == p2);  // true — igualdad por valor
```

### 12.3 `struct` — cuándo conviene usarlo

Un `struct` es un **tipo de valor**: se copia por valor y suele ser adecuado para datos pequeños, simples e inmutables.\
No tiene herencia de clases y su diseño apunta a representar cosas que se entienden mejor como un valor que como una identidad.

Como regla práctica, un `struct` funciona bien cuando:

- el dato es pequeño
- la instancia se crea y copia mucho
- no necesitás herencia
- querés evitar asignaciones al heap en escenarios muy concretos

No conviene usarlo cuando el objeto tiene comportamiento complejo, muchas propiedades mutables o una vida larga compartida entre muchas referencias.

Ejemplo clásico:

```csharp
public struct Punto {
    public double X { get; }
    public double Y { get; }

    public Punto(double x, double y) {
        X = x;
        Y = y;
    }

    public override string ToString() => $"({X}, {Y})";
}
```

También existe `record struct`, que combina semántica de valor con igualdad por valor y sintaxis más breve:

```csharp
public record struct Punto(double X, double Y);
```

### 12.4 Cuándo usar `class`, `record` o `struct` como DTO

| Situación                                            | Usar                                     |
| ---------------------------------------------------- | ---------------------------------------- |
| DTO simple, sólo datos, sin validación               | `record`                                 |
| DTO con lógica de validación o cálculos              | `class`                                  |
| DTO pequeño, inmutable y de valor                    | `struct` o `record struct`               |
| DTO que se deserializa desde JSON (System.Text.Json) | `class`, `record` o `record struct`      |
| Necesitás herencia de DTO                            | `record` (herencia de records) o `class` |

## 13. Extensión de clases (C# 14)

Las extensiones permiten agregar miembros a tipos existentes **sin heredar ni modificar** la clase original. Es especialmente útil para extender tipos de bibliotecas externas o tipos propios en capas separadas.

### 13.1 Métodos de extensión — el enfoque clásico

Antes de C# 14, sólo se podían agregar métodos mediante clases estáticas con `this` en el primer parámetro:

```csharp
public static class StringExtensions {
    public static bool EsEmail(this string s)
        => s.Contains('@') && s.Contains('.');

    public static string Capitalizar(this string s)
        => string.IsNullOrEmpty(s) ? s : char.ToUpper(s[0]) + s[1..].ToLower();

    public static string Truncar(this string s, int max, string sufijo = "…")
        => s.Length <= max ? s : s[..(max - sufijo.Length)] + sufijo;
}

// Uso — parecen métodos del propio tipo
"ana@ejemplo.com".EsEmail();        // true
"hola MUNDO".Capitalizar();         // "Hola mundo"
"texto muy largo".Truncar(10);      // "texto mu…"
```

### 13.2 Extension types (C# 14) — la forma nueva

C# 14 introduce `extension` como palabra clave propia, con una sintaxis dedicada que permite agregar no sólo métodos sino también propiedades, propiedades estáticas y operadores:

```csharp
// Extensión explícita del tipo string
extension StringEx for string {
    // Propiedad de instancia
    public bool EsEmail => this.Contains('@') && this.Contains('.');
    public bool EsVacio  => string.IsNullOrWhiteSpace(this);
    public int  Palabras => this.Split(' ', StringSplitOptions.RemoveEmptyEntries).Length;

    // Método de instancia
    public string Capitalizar() => 
        string.IsNullOrEmpty(this) ? this : char.ToUpper(this[0]) + this[1..].ToLower();

    public string Truncar(int max, string sufijo = "…") => 
        this.Length <= max ? this : this[..(max - sufijo.Length)] + sufijo;

    // Propiedad estática — accedida como string.Vacio
    public static string Vacio => string.Empty;
}

// Uso — idéntico a si fueran miembros nativos del tipo
string email = "ana@ejemplo.com";
Console.WriteLine(email.EsEmail);      // true  — propiedad
Console.WriteLine(email.Palabras);     // 1     — propiedad
Console.WriteLine(email.Capitalizar()); // método

string raro = string.Vacio;            // propiedad estática
```

También se puede extender un tipo propio o de librería:

```csharp
// Extensión para List<T>
extension ListaEx<T> for List<T> {
    // Propiedad — ¿la lista no está vacía?
    public bool TieneElementos => this.Count > 0;

    // Método
    public List<T> AleatorioMezclado() {
        var copia = new List<T>(this);
        var rnd   = new Random();
        for (int i = copia.Count - 1; i > 0; i--) {
            int j = rnd.Next(i + 1);
            (copia[i], copia[j]) = (copia[j], copia[i]);
        }
        return copia;
    }
}

// Extensión para un tipo propio — agrega presentación sin tocar el modelo
extension ProductoPresenter for Producto {
    public string EtiquetaPrecio => $"${this.Precio:N2}";
    public string ResumenStock   => this.Stock > 0
        ? $"{this.Stock} en stock"
        : "Sin stock";
}
```

### 13.3 Comparación: método de extensión clásico vs. `extension` C# 14

| Aspecto                       | Método de extensión (clásico) | `extension` (C# 14) |
| ----------------------------- | ----------------------------- | ------------------- |
| Métodos de instancia          | Sí                            | Sí                  |
| Propiedades de instancia      | No                            | Sí                  |
| Propiedades estáticas         | No                            | Sí                  |
| Operadores                    | No                            | Sí                  |
| Legibilidad de la declaración | Moderada                      | Alta                |
| Compatibilidad hacia atrás    | Sí                            | .NET 10+            |

## Resumen

```text
Definir estado interno y comportamiento
    └─► class con campos privados + propiedades + métodos

Construir el objeto de forma flexible
    └─► primary constructor (C# 12+) para la forma más compacta
    └─► constructor delegado (:this) para reusar lógica

Exponer el estado correctamente
    └─► { get; }             → sólo lectura
    └─► { get; private set;} → escritura interna
    └─► { get; init; }       → asignable en new{}
    └─► required { get; init; } → obligatoria en el inicializador
    └─► => expresion         → propiedad calculada
    └─► this[tipo param]     → acceso indexado con []

Compartir comportamiento sin instancia
    └─► static

Especializar un tipo existente (herencia)
    └─► class Derivada : Base { }
    └─► base(...) para encadenar constructores
    └─► protected para estado visible en subclases
    └─► base.Metodo() para extender sin reemplazar
    └─► sealed para bloquear herencia o sobreescritura

Variar el comportamiento según el tipo real (polimorfismo)
    └─► virtual + override → despacho dinámico en tiempo de ejecución
    └─► abstract → obliga a implementar, nunca instanciable directamente
    └─► is / as / switch con patrones → verificar tipo en tiempo de ejecución
    └─► new → oculta (no despacha) — evitar salvo casos muy específicos

Forzar implementación en subclases
    └─► abstract

Definir aritmética y comparación propias
    └─► operator +  -  *  /  ==  !=  <  >

Convertir entre tipos
    └─► implicit operator  /  explicit operator

Hacer el tipo inmutable
    └─► readonly fields + propiedades sin set + sealed

Transportar datos entre capas
    └─► class con required + init  /  record  /  record struct  /  struct

Agregar funcionalidad a tipos existentes
    └─► extension ... for ... (C# 14)
```
