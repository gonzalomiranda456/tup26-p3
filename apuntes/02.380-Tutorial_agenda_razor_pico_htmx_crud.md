## 1. Qué vamos a construir

Una agenda de contactos con las seis operaciones de un CRUD: listar, buscar, ver detalle, crear, editar y borrar. La pantalla es de tipo **maestro-detalle**: a la izquierda la lista de contactos, a la derecha un panel que cambia según lo que estés haciendo (ver, crear, editar o confirmar un borrado).

Lo distintivo de esta arquitectura es que **no escribimos JavaScript**. Cada interacción la maneja HTMX con atributos HTML, y el servidor responde con fragmentos de HTML ya renderizados por Razor. No hay API JSON, no hay framework de frontend, no hay estado duplicado en el navegador.

La estructura de carpetas queda así:

```txt
AgendaRazorHtmx/
├── Program.cs
├── Pages/
│   ├── _ViewStart.cshtml        <- aplica el layout (IMPRESCINDIBLE)
│   ├── Index.cshtml             <- pantalla principal + handlers inline
│   └── Shared/
│       ├── _Layout.cshtml
│       ├── _ContactoLista.cshtml
│       ├── _ContactoDetalle.cshtml
│       ├── _ContactoCrear.cshtml
│       ├── _ContactoEditar.cshtml
│       └── _ContactoBorrar.cshtml
├── Models/
│   └── Contacto.cs
├── Dtos/
│   ├── CrearContactoDto.cs
│   └── EditarContactoDto.cs
├── Repositories/
│   ├── IContactoRepository.cs
│   └── ContactoRepository.cs
└── Services/
    ├── IContactoService.cs
    └── ContactoService.cs
```

> **Por qué separamos en capas (modelo, DTO, repositorio, servicio):** aunque la app es chica, cada pieza tiene una sola responsabilidad. La página solo arma HTML y delega; el servicio resuelve la lógica; el repositorio guarda datos. Esto deja cambiar el almacenamiento (de memoria a SQLite) sin tocar las páginas, y testear el servicio con un repositorio falso. Es la separación mínima para que el proyecto no se vuelva inmanejable cuando crece.

## 2. Cómo fluye una interacción

Antes de escribir código conviene tener el modelo mental claro. Cuando el usuario hace algo (un clic, escribir en el buscador), el recorrido es siempre el mismo:

```txt
Navegador
   │  HTMX dispara un GET o POST
   ▼
Index.cshtml  (un handler: OnGetDetalle, OnPostCrear, etc.)
   │  el handler delega
   ▼
ContactoService   (resuelve la lógica)
   │
   ▼
ContactoRepository   (lee/escribe los datos)
   │
   ▼
Datos en memoria
```

Y la respuesta vuelve convertida en HTML:

```txt
Datos → Servicio → Handler → Partial Razor → HTML
   │
   ▼
HTMX recibe ese HTML y reemplaza solo una parte de la página
```

> **La clave de HTMX:** el servidor no devuelve datos crudos (JSON) que el navegador tenga que interpretar. Devuelve **HTML ya armado**, y HTMX simplemente lo inserta en el lugar indicado. Por eso no hace falta JavaScript: la lógica de "qué se ve" vive en el servidor, en los partials de Razor.

## 4. Por qué Razor Pages y los handlers inline

Razor Pages genera HTML en el servidor. Una página es un archivo `.cshtml` que mezcla HTML con C#, y la directiva `@page` al inicio la convierte en una URL accesible:

```razor
@page

<h1>Hola Razor</h1>
```

Normalmente la lógica de una página Razor vive en un archivo aparte (`Index.cshtml.cs`, el *code-behind*). En este tutorial usamos un enfoque alternativo: **handlers inline** dentro del mismo `.cshtml`, en un bloque `@functions`.

```razor
@functions {
    public void OnGet() {
        // Se ejecuta con: GET /Index
    }

    public PartialViewResult OnGetDetalle(int id) {
        // Se ejecuta con: GET /Index?handler=Detalle&id=1
    }
}
```

La convención de nombres de Razor traduce automáticamente la URL al método:

| Petición HTTP | Método que se ejecuta |
|---|---|
| `GET /Index` | `OnGet()` |
| `GET /Index?handler=Detalle&id=1` | `OnGetDetalle(int id)` |
| `POST /Index?handler=Crear` | `OnPostCrear(...)` |

El verbo (`GET`/`POST`) y el parámetro `handler` arman el nombre del método. Los parámetros restantes de la URL (`id=1`) se asignan a los argumentos del método por nombre: eso es **model binding**.

> **Por qué handlers inline en este tutorial:** mantenerlos en el `.cshtml` deja toda la lógica de la pantalla en un solo archivo, fácil de leer de corrido. En un proyecto grande conviene el code-behind separado, pero para aprender el flujo completo de un vistazo, lo inline es más claro. Es una decisión pedagógica, no una recomendación general.

## 5. Por qué PicoCSS

PicoCSS es un framework CSS minimalista que estiliza el **HTML semántico** directamente, sin obligarte a llenar los elementos de clases.

```html
<main class="container">
    <article>
        <header><strong>Título</strong></header>
        <label>Nombre <input name="nombre" /></label>
        <button>Aceptar</button>
    </article>
</main>
```

Ese HTML, sin una sola clase de utilidad, ya se ve prolijo con PicoCSS cargado. Elementos como `article`, `header`, `label`, `input`, `button` y `dl` reciben estilos cuidados automáticamente.

> **Por qué Pico y no Bootstrap o Tailwind:** para una herramienta administrativa como esta, escribir clases en cada elemento es ruido. Pico nos deja concentrarnos en estructurar bien el HTML y obtener una interfaz limpia gratis. El trade-off es menos control fino sobre el diseño, que acá no necesitamos.

## 6. Por qué HTMX

HTMX permite hacer peticiones al servidor y actualizar parte de la página usando solo atributos HTML. Un ejemplo:

```html
<button
    hx-get="/Index?handler=Detalle&id=2"
    hx-target="#detalle"
    hx-swap="outerHTML">
    Ver detalle
</button>
```

Al hacer clic, HTMX hace tres cosas: pide por `GET` a esa URL, recibe el HTML que devuelve el servidor, y reemplaza el elemento `#detalle` con ese HTML. Sin escribir nada de JavaScript.

Estos son los atributos que usamos en el tutorial:

| Atributo | Para qué sirve |
|---|---|
| `hx-get` | Hacer una petición GET |
| `hx-post` | Hacer una petición POST |
| `hx-target` | A qué elemento de la página aplicar la respuesta |
| `hx-swap` | Cómo insertar el HTML (`innerHTML` reemplaza el contenido, `outerHTML` reemplaza el elemento entero) |
| `hx-trigger` | Qué evento dispara la petición (clic, tecla, etc.) |
| `hx-confirm` | Mostrar un cuadro de confirmación antes de enviar |

> **Por qué HTML parcial y no JSON:** en una SPA tradicional, el servidor manda JSON y el navegador tiene código JavaScript que lo convierte en HTML. Eso duplica la lógica de presentación en dos lados. Con HTMX, el servidor manda el HTML ya armado por Razor y el navegador solo lo coloca. Una sola fuente de verdad para la presentación, cero JavaScript que mantener.

## 7. El modelo

El modelo representa cómo vive un contacto **dentro** de la aplicación. Es la clase que el repositorio guarda y que el servicio manipula.

Creá `Models/Contacto.cs`:

```csharp
namespace AgendaRazorHtmx.Models;

public class Contacto {
    public int Id { get; set; }
    public string Nombre { get; set; } = "";
    public string Apellido { get; set; } = "";
    public string Telefono { get; set; } = "";
    public string Email { get; set; } = "";
    public string Direccion { get; set; } = "";
}
```

> **Por qué `= ""` en cada string:** una propiedad `string` no anulable tiene que arrancar con un valor para que el compilador no se queje de posibles nulos. La cadena vacía es un punto de partida seguro: un contacto recién creado tiene campos vacíos, no nulos.

## 8. Los DTOs como record

Un **DTO** (Data Transfer Object) define qué datos entran desde afuera. Usamos `record` porque son inmutables y se declaran en una línea: ideal para datos que solo transportan información.

Vamos a tener dos, y la razón de que sean dos es importante.

> **Por qué dos DTOs distintos:** crear y editar no manejan los mismos datos. Al **crear**, el `Id` todavía no existe (lo genera el sistema), así que el DTO de creación no lo incluye. Al **editar**, el `Id` es imprescindible para saber a qué contacto aplicar los cambios. Forzar un solo DTO para ambos casos nos obligaría a un `Id` opcional que nunca sabríamos si confiar. Dos tipos distintos hacen explícito el contrato de cada operación.

Creá `Dtos/CrearContactoDto.cs`:

```csharp
namespace AgendaRazorHtmx.Dtos;

public record CrearContactoDto(
    string Nombre,
    string Apellido,
    string Telefono,
    string Email,
    string Direccion
);
```

Y `Dtos/EditarContactoDto.cs`:

```csharp
namespace AgendaRazorHtmx.Dtos;

public record EditarContactoDto(
    int Id,
    string Nombre,
    string Apellido,
    string Telefono,
    string Email,
    string Direccion
);
```

> **Por qué no usar `Contacto` directamente en los formularios:** el modelo es lo que vive adentro; el DTO es la frontera de lo que aceptamos de afuera. Si el formulario enviara un `Contacto` completo, el navegador podría intentar setear cualquier propiedad del modelo interno. El DTO define exactamente qué campos son válidos como entrada, y nada más. En resumen: **modelo = cómo vive el dato adentro; DTO = qué acepto desde afuera.**

## 9. El repositorio

El repositorio se encarga del acceso a datos y nada más. Hoy guardamos en memoria; mañana se podría cambiar por SQLite o SQL Server sin que el resto del sistema se entere, porque todos hablan con la **interface**, no con la implementación.

Primero el contrato. Creá `Repositories/IContactoRepository.cs`:

```csharp
using AgendaRazorHtmx.Models;

namespace AgendaRazorHtmx.Repositories;

public interface IContactoRepository {
    List<Contacto> ObtenerTodos();
    Contacto? ObtenerPorId(int id);
    Contacto Agregar(Contacto contacto);
    void Actualizar(Contacto contacto);
    void Borrar(int id);
}
```

Ahora la implementación en memoria. Creá `Repositories/ContactoRepository.cs`:

```csharp
using AgendaRazorHtmx.Models;

namespace AgendaRazorHtmx.Repositories;

public class ContactoRepository : IContactoRepository {
    private readonly List<Contacto> contactos =
    [
        new Contacto {
            Id = 1,
            Nombre = "Juan",
            Apellido = "Pérez",
            Telefono = "3815551111",
            Email = "juan@mail.com",
            Direccion = "San Miguel de Tucumán"
        },
        new Contacto {
            Id = 2,
            Nombre = "Ana",
            Apellido = "Gómez",
            Telefono = "3815552222",
            Email = "ana@mail.com",
            Direccion = "Yerba Buena"
        },
        new Contacto {
            Id = 3,
            Nombre = "Luis",
            Apellido = "Torres",
            Telefono = "3815553333",
            Email = "luis@mail.com",
            Direccion = "Tafí Viejo"
        }
    ];

    private int siguienteId = 4;

    public List<Contacto> ObtenerTodos() {
        return this.contactos;
    }

    public Contacto? ObtenerPorId(int id) {
        return this.contactos.FirstOrDefault(contacto => contacto.Id == id);
    }

    public Contacto Agregar(Contacto contacto) {
        contacto.Id = this.siguienteId;
        this.siguienteId++;

        this.contactos.Add(contacto);
        return contacto;
    }

    public void Actualizar(Contacto contactoActualizado) {
        Contacto? contacto = this.ObtenerPorId(contactoActualizado.Id);

        if (contacto is null) {
            return;
        }

        contacto.Nombre = contactoActualizado.Nombre;
        contacto.Apellido = contactoActualizado.Apellido;
        contacto.Telefono = contactoActualizado.Telefono;
        contacto.Email = contactoActualizado.Email;
        contacto.Direccion = contactoActualizado.Direccion;
    }

    public void Borrar(int id) {
        Contacto? contacto = this.ObtenerPorId(id);

        if (contacto is null) {
            return;
        }

        this.contactos.Remove(contacto);
    }
}
```

> **El campo `siguienteId`** simula el autoincremento de una base de datos: lleva la cuenta del próximo `Id` libre. Arranca en 4 porque ya precargamos tres contactos.
>
> **Importante: los datos viven solo en memoria.** Cada vez que reinicies la aplicación, la lista vuelve a los tres contactos iniciales y se pierde todo lo que hayas creado o editado. Es a propósito, para no meter una base de datos todavía. En la sección final vemos cómo pasar a almacenamiento persistente.

## 10. El servicio

El servicio contiene la **funcionalidad** de la aplicación. La página no debería saber cómo se generan los IDs, cómo se convierte un DTO en modelo, ni cómo se busca internamente. La página pide; el servicio resuelve.

El contrato, en `Services/IContactoService.cs`:

```csharp
using AgendaRazorHtmx.Dtos;
using AgendaRazorHtmx.Models;

namespace AgendaRazorHtmx.Services;

public interface IContactoService {
    List<Contacto> Buscar(string? texto);
    Contacto? ObtenerDetalle(int id);
    EditarContactoDto? ObtenerParaEditar(int id);
    Contacto Crear(CrearContactoDto dto);
    Contacto? Actualizar(EditarContactoDto dto);
    void Borrar(int id);
}
```

La implementación, en `Services/ContactoService.cs`:

```csharp
using AgendaRazorHtmx.Dtos;
using AgendaRazorHtmx.Models;
using AgendaRazorHtmx.Repositories;

namespace AgendaRazorHtmx.Services;

public class ContactoService : IContactoService {
    private readonly IContactoRepository contactoRepository;

    public ContactoService(IContactoRepository contactoRepository) {
        this.contactoRepository = contactoRepository;
    }

    public List<Contacto> Buscar(string? texto) {
        List<Contacto> contactos = this.contactoRepository.ObtenerTodos();

        if (string.IsNullOrWhiteSpace(texto)) {
            return contactos;
        }

        string textoNormalizado = texto.Trim().ToLower();

        return contactos
            .Where(contacto =>
                contacto.Nombre.ToLower().Contains(textoNormalizado) ||
                contacto.Apellido.ToLower().Contains(textoNormalizado) ||
                contacto.Telefono.ToLower().Contains(textoNormalizado) ||
                contacto.Email.ToLower().Contains(textoNormalizado) ||
                contacto.Direccion.ToLower().Contains(textoNormalizado))
            .ToList();
    }

    public Contacto? ObtenerDetalle(int id) {
        return this.contactoRepository.ObtenerPorId(id);
    }

    public EditarContactoDto? ObtenerParaEditar(int id) {
        Contacto? contacto = this.contactoRepository.ObtenerPorId(id);

        if (contacto is null) {
            return null;
        }

        return new EditarContactoDto(
            contacto.Id,
            contacto.Nombre,
            contacto.Apellido,
            contacto.Telefono,
            contacto.Email,
            contacto.Direccion
        );
    }

    public Contacto Crear(CrearContactoDto dto) {
        Contacto contacto = new() {
            Nombre = dto.Nombre,
            Apellido = dto.Apellido,
            Telefono = dto.Telefono,
            Email = dto.Email,
            Direccion = dto.Direccion
        };

        return this.contactoRepository.Agregar(contacto);
    }

    public Contacto? Actualizar(EditarContactoDto dto) {
        Contacto? contacto = this.contactoRepository.ObtenerPorId(dto.Id);

        if (contacto is null) {
            return null;
        }

        Contacto contactoActualizado = new() {
            Id = dto.Id,
            Nombre = dto.Nombre,
            Apellido = dto.Apellido,
            Telefono = dto.Telefono,
            Email = dto.Email,
            Direccion = dto.Direccion
        };

        this.contactoRepository.Actualizar(contactoActualizado);
        return this.contactoRepository.ObtenerPorId(dto.Id);
    }

    public void Borrar(int id) {
        this.contactoRepository.Borrar(id);
    }
}
```

> **El servicio centraliza las conversiones entre modelo y DTO.** Fijate las tres traducciones que ocurren acá y en ningún otro lado: `CrearContactoDto → Contacto` (al crear), `Contacto → EditarContactoDto` (al preparar la edición) y `EditarContactoDto → Contacto` (al guardar). La página nunca ve esta lógica: pide "creá esto" o "editá aquello" y recibe el resultado.
>
> **Por qué `Buscar(null)` devuelve todo:** unificamos listar y buscar en un solo método. Si el texto es nulo o vacío, devuelve la lista completa; si trae algo, filtra. Así la página tiene un solo camino para poblar la lista, con o sin filtro.

## 11. Registrar dependencias en Program.cs

Acá registramos los servicios en el contenedor de inyección de dependencias y configuramos el pipeline.

Reemplazá **todo** el contenido de `Program.cs` por esto:

```csharp
using AgendaRazorHtmx.Repositories;
using AgendaRazorHtmx.Services;

var builder = WebApplication.CreateBuilder(args);

// Deshabilitamos la validacion antiforgery para simplificar el tutorial.
// (Mas abajo se explica el porque y cuando NO conviene hacer esto.)
builder.Services.AddRazorPages(options =>
    options.Conventions.ConfigureFilter(
        new Microsoft.AspNetCore.Mvc.IgnoreAntiforgeryTokenAttribute()));

builder.Services.AddSingleton<IContactoRepository, ContactoRepository>();
builder.Services.AddScoped<IContactoService, ContactoService>();

var app = builder.Build();

app.UseStaticFiles();
app.MapRazorPages();

app.Run();
```

Tres decisiones a entender acá:

> **Por qué deshabilitamos antiforgery.** ASP.NET, por defecto, exige un token de seguridad en cada `POST` para protegerse de ataques CSRF. Hacer que HTMX envíe ese token correctamente requiere configuración extra en el layout y en cada formulario. Para que este tutorial se concentre en el CRUD y en el flujo de HTMX, lo apagamos con `IgnoreAntiforgeryTokenAttribute`. Así los `POST` de crear, editar y borrar funcionan sin pasos adicionales.
>
> **Cuándo NO hacer esto.** En una aplicación real, expuesta a internet, la protección antiforgery es importante y **no** se desactiva. Si más adelante querés activarla, el camino es configurar HTMX para que mande el token en un header. Lo dejamos como mejora para no complicar la primera versión.
>
> **`AddSingleton` para el repositorio:** una sola instancia para toda la aplicación. Como guardamos los datos en una `List` en memoria, necesitamos que esa lista sea **la misma** en cada petición. Si fuera scoped o transient, cada request tendría su propia lista vacía y los datos "desaparecerían" entre una operación y la siguiente.
>
> **`AddScoped` para el servicio:** una instancia por petición HTTP. El servicio no guarda estado propio, solo coordina, así que una instancia fresca por request es lo correcto y lo habitual.

## 12. El archivo de arranque de Razor (imprescindible)

Este archivo es el que hace que la aplicación use el layout. Sin él la app compila pero aparece **sin estilos**, porque PicoCSS y HTMX viven en el layout y nada le diría a la página que lo use. Por eso lo creamos antes que las vistas.

Creá `Pages/_ViewStart.cshtml`:

```razor
@{
    Layout = "_Layout";
}
```

> **Por qué es imprescindible:** `_ViewStart` se ejecuta antes de cada página y le indica qué layout usar. Sin él, `Index.cshtml` se renderiza **sin** el `_Layout.cshtml`, lo que significa que las etiquetas que cargan PicoCSS y HTMX (que viven en el layout) nunca llegan al navegador. Resultado: la página funciona "a medias" pero sin estilos y sin que HTMX responda a nada.

> **Por qué NO necesitamos `_ViewImports.cshtml` con `@addTagHelper`:** un proyecto Razor típico incluye ese archivo para habilitar *tag helpers* de ASP.NET como `<partial>`. Nosotros no usamos `<partial>` en ningún lado: la lista inicial la carga HTMX con `hx-trigger="load"` (sección 14) y todos los fragmentos posteriores llegan también por HTMX. Como toda la inserción de HTML pasa por un único mecanismo, no hay ninguna etiqueta especial de Razor que registrar, y el `_ViewImports.cshtml` se vuelve innecesario. Es la consecuencia directa de la decisión de "HTMX para todo".
>
> Si en el futuro agregás validación con tag helpers (`asp-validation-for`) o volvés a usar `<partial>`, ahí sí vas a necesitar crear `Pages/_ViewImports.cshtml` con la línea `@addTagHelper *, Microsoft.AspNetCore.Mvc.TagHelpers`.

```razor
    </main>

    <script src="https://unpkg.com/htmx.org@@2.0.4"></script>

    @await RenderSectionAsync("Scripts", required: false)
</body>
</html>
```

> **Por qué los `@@` dobles:** en Razor, el carácter `@` introduce código C#. Cuando necesitamos un `@` **literal** en el HTML o CSS de salida, lo escribimos doble. Por eso `@@picocss` (parte de la URL del CDN), `@@media` (regla CSS) y `@@2.0.4` (versión en la URL de HTMX) llevan doble arroba: queremos que el navegador reciba un solo `@`, no que Razor intente interpretarlo.
>
> **`@RenderBody()`** es donde se inserta el contenido de cada página. **`RenderSectionAsync("Scripts", ...)`** deja un hueco opcional por si alguna página necesita scripts propios; con `required: false` no falla si no se usa.

## 14. La página principal con handlers inline

Esta es la pantalla central: el HTML del maestro-detalle más todos los handlers en un bloque `@functions`. Como deshabilitamos la validación antiforgery en `Program.cs`, los `POST` de HTMX funcionan directo, sin tokens ni configuración extra.

Creá `Pages/Index.cshtml`:

```razor
@page
@using AgendaRazorHtmx.Dtos
@using AgendaRazorHtmx.Models
@using AgendaRazorHtmx.Services
@using Microsoft.AspNetCore.Mvc
@inject IContactoService ContactoService

@{
    ViewData["Title"] = "Agenda";
}

<h1>Agenda de contactos</h1>

<div class="app-grid">
    <article>
        <header>
            <strong>Contactos</strong>
        </header>

        <div class="acciones">
            <input type="search"
                   name="texto"
                   placeholder="Buscar..."
                   hx-get="/Index?handler=Buscar"
                   hx-trigger="keyup changed delay:300ms"
                   hx-target="#contactos"
                   hx-swap="innerHTML" />

            <button hx-get="/Index?handler=Crear"
                    hx-target="#detalle"
                    hx-swap="outerHTML">
                Nuevo
            </button>
        </div>

        <div id="contactos"
             hx-get="/Index?handler=Buscar"
             hx-trigger="load"
             hx-swap="innerHTML">
        </div>
    </article>

    <article id="detalle">
        <header>
            <strong>Detalle</strong>
        </header>

        <p>Seleccioná un contacto o creá uno nuevo.</p>
    </article>
</div>

@functions {
    public PartialViewResult OnGetBuscar(string? texto) {
        List<Contacto> contactos = ContactoService.Buscar(texto);
        return Partial("Shared/_ContactoLista", contactos);
    }

    public PartialViewResult OnGetDetalle(int id) {
        Contacto? contacto = ContactoService.ObtenerDetalle(id);
        return Partial("Shared/_ContactoDetalle", contacto);
    }

    public PartialViewResult OnGetCrear() {
        CrearContactoDto dto = new("", "", "", "", "");
        return Partial("Shared/_ContactoCrear", dto);
    }

    public PartialViewResult OnPostCrear(CrearContactoDto dto) {
        if (!ModelState.IsValid) {
            return Partial("Shared/_ContactoCrear", dto);
        }

        Contacto contactoCreado = ContactoService.Crear(dto);
        return Partial("Shared/_ContactoDetalle", contactoCreado);
    }

    public PartialViewResult OnGetEditar(int id) {
        EditarContactoDto? dto = ContactoService.ObtenerParaEditar(id);
        return Partial("Shared/_ContactoEditar", dto);
    }

    public PartialViewResult OnPostGuardar(EditarContactoDto dto) {
        if (!ModelState.IsValid) {
            return Partial("Shared/_ContactoEditar", dto);
        }

        Contacto? contactoActualizado = ContactoService.Actualizar(dto);
        return Partial("Shared/_ContactoDetalle", contactoActualizado);
    }

    public PartialViewResult OnGetBorrar(int id) {
        Contacto? contacto = ContactoService.ObtenerDetalle(id);
        return Partial("Shared/_ContactoBorrar", contacto);
    }

    public PartialViewResult OnPostBorrar(int id) {
        ContactoService.Borrar(id);

        List<Contacto> contactos = ContactoService.Buscar(null);
        return Partial("Shared/_ContactoLista", contactos);
    }
}
```

Esta página concentra el HTML de la pantalla, la inyección del servicio con `@inject`, y los ocho handlers que responden a HTMX. Lo que **no** contiene es igual de importante: no tiene el modelo, ni los DTOs, ni el repositorio, ni lógica de almacenamiento. Todo eso vive en sus capas.

> **Por qué la lista se carga con HTMX y no con un partial en el servidor:** el `<div id="contactos">` arranca vacío y trae `hx-trigger="load"`, que dispara una petición a `OnGetBuscar` apenas el elemento aparece en la página. Así, **toda la inserción de HTML pasa por un solo mecanismo: HTMX**. No mezclamos un `<partial>` renderizado en el servidor con los fragmentos que vienen por HTMX; todo entra igual. Como efecto secundario, ya no necesitamos el tag helper `<partial>` y podemos prescindir del `_ViewImports.cshtml` (lo vemos en la sección siguiente).
>
> **El costo a tener en cuenta:** con `hx-trigger="load"` el navegador pinta primero la página con la lista vacía y, una fracción de segundo después, HTMX la rellena. Es una segunda petición HTTP y un parpadeo mínimo. En esta app (datos en memoria, red local) es imperceptible, pero conviene saber que se cambió "una carga" por "carga + fetch" a cambio de la coherencia de usar HTMX para todo.
>
> **Por qué ya no hay `OnGet`:** antes, `OnGet` preparaba la lista inicial. Ahora esa lista la trae `OnGetBuscar` (con texto nulo, que devuelve todo), igual que cualquier búsqueda. Un solo handler sirve para el primer render y para los filtros posteriores.
>
> **`hx-swap` según el objetivo:** la lista usa `innerHTML` (reemplazamos el *contenido* de `#contactos`, el `<div>` queda). El panel de detalle usa `outerHTML` (reemplazamos el *artículo entero* `#detalle`, porque el partial trae su propio `<article id="detalle">`). Confundir uno con otro es un error típico: con `innerHTML` en el detalle terminarías con dos `<article>` anidados.

## 15. Los partials

Los partials son fragmentos de HTML reutilizables. Cada handler devuelve uno, y HTMX lo inserta en la página. Como deshabilitamos la validación antiforgery, los formularios no necesitan ningún token oculto.

### Lista (el panel maestro)

Creá `Pages/Shared/_ContactoLista.cshtml`:

```razor
@model List<AgendaRazorHtmx.Models.Contacto>

@if (Model.Count == 0) {
    <p>No hay contactos.</p>
}
else {
    <ul>
        @foreach (var contacto in Model) {
            <li>
                <a href="#"
                   class="contacto-link"
                   hx-get="/Index?handler=Detalle&id=@contacto.Id"
                   hx-target="#detalle"
                   hx-swap="outerHTML">
                    @contacto.Nombre @contacto.Apellido
                </a>
            </li>
        }
    </ul>
}
```

> Cada contacto es un enlace que, al hacer clic, pide `OnGetDetalle(id)` y reemplaza el panel de detalle. El `id` se interpola desde el modelo (`@contacto.Id`).

### Detalle (solo lectura)

Creá `Pages/Shared/_ContactoDetalle.cshtml`:

```razor
@model AgendaRazorHtmx.Models.Contacto?

<article id="detalle">
    <header>
        <strong>Detalle</strong>
    </header>

    @if (Model is null) {
        <p>Contacto no encontrado.</p>
    }
    else {
        <dl>
            <dt>Nombre</dt>
            <dd>@Model.Nombre</dd>

            <dt>Apellido</dt>
            <dd>@Model.Apellido</dd>

            <dt>Teléfono</dt>
            <dd>@Model.Telefono</dd>

            <dt>Email</dt>
            <dd>@Model.Email</dd>

            <dt>Dirección</dt>
            <dd>@Model.Direccion</dd>
        </dl>

        <div class="acciones">
            <button hx-get="/Index?handler=Editar&id=@Model.Id"
                    hx-target="#detalle"
                    hx-swap="outerHTML">
                Editar
            </button>

            <button class="secondary"
                    hx-get="/Index?handler=Borrar&id=@Model.Id"
                    hx-target="#detalle"
                    hx-swap="outerHTML">
                Borrar
            </button>
        </div>
    }
</article>
```

> **Por qué el partial trae su propio `<article id="detalle">`:** como los botones que apuntan al detalle usan `hx-swap="outerHTML"`, reemplazan el elemento completo. El partial tiene que volver a traer ese contenedor con el mismo `id`, o la próxima interacción no encontraría a dónde apuntar.

### Creación

Creá `Pages/Shared/_ContactoCrear.cshtml`:

```razor
@model AgendaRazorHtmx.Dtos.CrearContactoDto?

<article id="detalle">
    <header>
        <strong>Nuevo contacto</strong>
    </header>

    <form hx-post="/Index?handler=Crear"
          hx-target="#detalle"
          hx-swap="outerHTML">

        <label>
            Nombre
            <input name="Nombre" value="@Model?.Nombre" autocomplete="off" />
        </label>

        <label>
            Apellido
            <input name="Apellido" value="@Model?.Apellido" autocomplete="off" />
        </label>

        <label>
            Teléfono
            <input name="Telefono" value="@Model?.Telefono" autocomplete="off" />
        </label>

        <label>
            Email
            <input type="email" name="Email" value="@Model?.Email" autocomplete="off" />
        </label>

        <label>
            Dirección
            <input name="Direccion" value="@Model?.Direccion" autocomplete="off" />
        </label>

        <div class="acciones">
            <button type="submit">Crear</button>

            <button type="button"
                    class="secondary"
                    hx-get="/Index?handler=Buscar"
                    hx-target="#contactos"
                    hx-swap="innerHTML">
                Limpiar lista
            </button>
        </div>
    </form>
</article>
```

> Este formulario **no** envía `Id`: lo genera el repositorio al crear. Los `name` de cada input coinciden exactamente con las propiedades de `CrearContactoDto`, y por eso el model binding arma el DTO solo al hacer `POST`.

### Edición

Creá `Pages/Shared/_ContactoEditar.cshtml`:

```razor
@model AgendaRazorHtmx.Dtos.EditarContactoDto?

<article id="detalle">
    <header>
        <strong>Editar contacto</strong>
    </header>

    @if (Model is null) {
        <p>Contacto no encontrado.</p>
    }
    else {
        <form hx-post="/Index?handler=Guardar"
              hx-target="#detalle"
              hx-swap="outerHTML">

            <input type="hidden" name="Id" value="@Model.Id" />

            <label>
                Nombre
                <input name="Nombre" value="@Model.Nombre" autocomplete="off" />
            </label>

            <label>
                Apellido
                <input name="Apellido" value="@Model.Apellido" autocomplete="off" />
            </label>

            <label>
                Teléfono
                <input name="Telefono" value="@Model.Telefono" autocomplete="off" />
            </label>

            <label>
                Email
                <input type="email" name="Email" value="@Model.Email" autocomplete="off" />
            </label>

            <label>
                Dirección
                <input name="Direccion" value="@Model.Direccion" autocomplete="off" />
            </label>

            <div class="acciones">
                <button type="submit">Guardar</button>

                <button type="button"
                        class="secondary"
                        hx-get="/Index?handler=Detalle&id=@Model.Id"
                        hx-target="#detalle"
                        hx-swap="outerHTML">
                    Cancelar
                </button>
            </div>
        </form>
    }
</article>
```

> A diferencia del de creación, este formulario **sí** envía `Id`, en un campo oculto. Es lo que le permite al servicio saber qué contacto actualizar. El botón Cancelar simplemente vuelve a pedir el detalle del mismo contacto.

### Borrado (confirmación)

Creá `Pages/Shared/_ContactoBorrar.cshtml`:

```razor
@model AgendaRazorHtmx.Models.Contacto?

<article id="detalle">
    <header>
        <strong>Borrar contacto</strong>
    </header>

    @if (Model is null) {
        <p>Contacto no encontrado.</p>
    }
    else {
        <p>
            ¿Seguro que querés borrar a
            <strong>@Model.Nombre @Model.Apellido</strong>?
        </p>

        <div class="acciones">
            <button class="contrast"
                    hx-post="/Index?handler=Borrar&id=@Model.Id"
                    hx-target="#contactos"
                    hx-swap="innerHTML"
                    hx-confirm="¿Confirmás el borrado?">
                Sí, borrar
            </button>

            <button type="button"
                    class="secondary"
                    hx-get="/Index?handler=Detalle&id=@Model.Id"
                    hx-target="#detalle"
                    hx-swap="outerHTML">
                Cancelar
            </button>
        </div>
    }
</article>
```

> **`hx-confirm`** muestra un cuadro de confirmación del navegador antes de enviar el `POST`. Es una segunda red de seguridad sobre el paso de confirmación que ya representa este partial.
>
> **Un detalle de comportamiento que conviene conocer:** al confirmar el borrado, el botón apunta a `#contactos` con `innerHTML`, así que **se actualiza la lista** pero el panel de detalle sigue mostrando la confirmación vieja. Para una primera versión es aceptable. Refrescar lista y detalle a la vez requiere una técnica de HTMX llamada *out-of-band swap* (`hx-swap-oob`), que devuelve dos fragmentos en una sola respuesta. Lo dejamos anotado como mejora para no complicar esta versión.

## 16. Los flujos completos

Vale la pena seguir dos flujos de punta a punta para fijar cómo encajan las piezas.

### Flujo de alta

```txt
Clic en "Nuevo"
   │  hx-get /Index?handler=Crear
   ▼
OnGetCrear()  →  devuelve _ContactoCrear (formulario vacío)
   │  HTMX reemplaza #detalle
   ▼
El usuario completa el formulario y envía
   │  hx-post /Index?handler=Crear  (el binding arma CrearContactoDto)
   ▼
OnPostCrear(dto)  →  ContactoService.Crear(dto)  →  el repositorio asigna Id
   │  devuelve _ContactoDetalle del nuevo contacto
   ▼
HTMX reemplaza #detalle con el detalle del contacto recién creado
```

> En esta versión, al crear un contacto la lista de la izquierda **no** se refresca sola: el panel de detalle muestra el contacto nuevo, pero para verlo en la lista hay que volver a buscar. Igual que con el borrado, refrescar ambos paneles a la vez es trabajo de `hx-swap-oob`.

### Flujo de borrado

```txt
El usuario ve un detalle y hace clic en "Borrar"
   │  hx-get /Index?handler=Borrar&id=2
   ▼
OnGetBorrar(2)  →  devuelve _ContactoBorrar (la confirmación)
   │  HTMX reemplaza #detalle
   ▼
El usuario confirma
   │  hx-post /Index?handler=Borrar&id=2  (hx-confirm pide confirmación extra)
   ▼
OnPostBorrar(2)  →  ContactoService.Borrar(2)
   │  devuelve _ContactoLista actualizada
   ▼
HTMX reemplaza el contenido de #contactos con la lista sin ese contacto
```

## 18. Resumen y próximos pasos

### Qué resuelve cada pieza

| Pieza | Responsabilidad |
|---|---|
| `_ViewStart.cshtml` | Aplica el layout a cada página |
| `_Layout.cshtml` | Plantilla base: PicoCSS, HTMX, estilos |
| `Index.cshtml` | Pantalla principal y los ocho handlers |
| Partials | Fragmentos de HTML que devuelven los handlers |
| `Contacto` | Modelo interno |
| `CrearContactoDto` / `EditarContactoDto` | Contratos de entrada de cada operación |
| `ContactoService` | Funcionalidad de la app y conversiones modelo/DTO |
| `ContactoRepository` | Acceso a datos (hoy, en memoria) |

### Los ocho handlers de un vistazo

```csharp
public PartialViewResult OnGetBuscar(string? texto);  // listar (carga inicial) y filtrar
public PartialViewResult OnGetDetalle(int id);        // ver
public PartialViewResult OnGetCrear();                // formulario de alta
public PartialViewResult OnPostCrear(CrearContactoDto dto);
public PartialViewResult OnGetEditar(int id);         // formulario de edición
public PartialViewResult OnPostGuardar(EditarContactoDto dto);
public PartialViewResult OnGetBorrar(int id);         // confirmación
public PartialViewResult OnPostBorrar(int id);
```

### Para llevarlo más lejos

Esta versión ya tiene un CRUD funcional. Algunas mejoras naturales, más o menos en orden de dificultad:

- **Validaciones con `DataAnnotations`** en los DTOs (por ejemplo `[Required]`, `[EmailAddress]`), aprovechando que ya chequeamos `ModelState.IsValid` en los handlers.
- **Reactivar la protección antiforgery** (que acá apagamos por simplicidad): se vuelve a habilitar la validación y se configura HTMX para que mande el token en un header. Es el paso obligado antes de exponer la app a internet.
- **Refrescar lista y detalle a la vez** con out-of-band swaps (`hx-swap-oob`): una sola respuesta que actualiza los dos paneles tras crear o borrar.
- **Persistencia con SQLite** vía Entity Framework Core: se cambia solo la implementación del repositorio; las interfaces, el servicio y las páginas quedan igual. Esa es justamente la ventaja de haber separado en capas.
- **Mensajes de éxito o error** y confirmación visual después de guardar.
- **Foco automático** en el primer input al abrir un formulario.
- **Paginación y ordenamiento** de la lista cuando los contactos crecen.

El patrón completo —Razor Pages con handlers inline, PicoCSS para la estética, HTMX para la interacción, partials para los fragmentos, DTOs como contratos, servicio para la lógica y repositorio para los datos— permite construir un CRUD administrativo mantenible sin SPA y sin escribir JavaScript a mano.
