# Terminal.Gui: apuntes para construir aplicaciones TUI en C#

> Estos apuntes usan **Terminal.Gui v2**, la librería de .NET para construir interfaces de terminal modernas. Vamos progresivamente desde el ejemplo más chico (dos paneles) hasta una agenda completa con persistencia en **SQLite** vía **Dapper.Contrib** e importación/exportación a **JSON**.

El objetivo es que al final puedas construir una aplicación básica de gestión con:

- ventanas y paneles (`Window`, `FrameView`);
- diálogos modales (`Dialog`, `Dialog<T>`);
- listas con búsqueda activa (`ListView`, `TextField`);
- vista de detalle con formato (`Markdown`);
- menú desplegable superior (`MenuBar`, `MenuItem`);
- manejo de teclado vía atajos en el menú;
- persistencia en SQLite usando Dapper.Contrib como mini-ORM;
- importación y exportación de los datos en JSON.

## 1. La idea: una interfaz gráfica dentro de la terminal

Una aplicación de consola tradicional imprime texto y lee texto:

``` csharp
Console.Write("Nombre: ");
string nombre = Console.ReadLine() ?? "";
```

Eso alcanza para programas chicos, pero se vuelve incómodo cuando hay muchas opciones, listas, formularios, búsqueda, edición y navegación.

Una **TUI** (*Terminal User Interface*) usa la terminal completa como superficie visual. En lugar de imprimir líneas una debajo de otra, dibuja ventanas, paneles, menús, botones, listas y formularios.

La terminal sigue siendo texto, pero la experiencia deja de ser una secuencia lineal de preguntas y respuestas.

Terminal.Gui resuelve varias tareas difíciles:

- dibujar correctamente en la pantalla;
- redibujar cuando cambia el tamaño de la terminal;
- manejar foco;
- capturar teclado y mouse;
- organizar componentes visuales;
- abrir diálogos modales;
- evitar que el programador tenga que usar `Console.SetCursorPosition` manualmente.

La idea central es esta:

``` text
Application
└── Runnable / Window
    ├── MenuBar
    ├── FrameView
    │   └── ListView
    ├── FrameView
    │   └── Label
    └── Dialog, cuando se abre uno
```

La aplicación no "imprime pantallas". Construye un árbol de objetos visuales. Terminal.Gui se encarga de dibujarlo, mover el foco, recibir eventos y redibujar cuando haga falta.

## 2. Filosofía de diseño

Terminal.Gui se parece más a WinForms, WPF o Avalonia que a `Console.WriteLine`.

Sus decisiones de diseño principales son:

### Todo lo visible es un `View`

Un botón, una etiqueta, una lista, una ventana y un diálogo son `View` o derivan de `View`.

Eso significa que todos comparten conceptos:

- posición: `X`, `Y`;
- tamaño: `Width`, `Height`;
- padre visual: `SuperView`;
- hijos visuales: `SubViews`;
- foco;
- eventos;
- dibujo;
- layout.

Ejemplo:

``` csharp
Label label = new() {
    Text = "Buscar:",
    X = 1,
    Y = 2
};

TextField input = new() {
    X = Pos.Right(label) + 1,
    Y = Pos.Top(label),
    Width = Dim.Percent(40)
};
```

### El layout se expresa, no se calcula a mano

En vez de calcular coordenadas con enteros todo el tiempo, Terminal.Gui usa:

- `Pos` para posiciones;
- `Dim` para tamaños.

Eso permite decir:

``` csharp
X = Pos.Right(label) + 1;
Width = Dim.Fill(2);
```

En castellano: "poné este campo una columna a la derecha del label y hacelo ocupar el resto del ancho, dejando dos columnas libres".

### La aplicación es reactiva

No hay un `while` manual preguntando qué quiere hacer el usuario.

Terminal.Gui ejecuta un bucle de eventos:

1.  dibuja la interfaz;
2.  espera una tecla, mouse o evento interno;
3.  manda ese evento al componente con foco;
4.  ejecuta handlers;
5.  redibuja lo que cambió.

Por eso escribimos código como este:

``` csharp
buscar.TextChanged += (_, _) => Actualizar();
btnGuardar.Accepting += (_, _) => Guardar();
```

### El teclado es un ciudadano de primera clase

Una buena TUI se puede usar sin mouse. Terminal.Gui soporta:

- foco con Tab;
- menús con F10 o Alt+letra;
- hotkeys con `_`;
- atajos como `Ctrl+S`;
- eventos de teclado;
- comandos.

### La interfaz no debería contener la lógica de negocio

El `Button` no debería saber cómo se persiste un contacto. El botón llama a un método de la ventana, y ese método delega en una clase que maneja datos.

En el ejemplo final integrado hay cuatro responsabilidades separadas:

- `AgendaWindow` organiza la UI y coordina acciones;
- `Editar : Dialog<Contacto?>` edita un contacto;
- `SqliteAgendaStore` persiste contactos en SQLite con Dapper.Contrib;
- `JsonAgendaIO` lee y escribe JSON para importación y exportación;
- `Contacto` representa los datos.

## 3. Crear el proyecto

Para clase conviene una app C# de archivo único (file based) con .NET 10. Las dependencias se declaran al principio del `.cs`:

``` csharp
#!/usr/bin/env -S dotnet run
#:package Terminal.Gui@2.0.1
#:property PublishAot=false

using Terminal.Gui.App;       // Application, IApplication, Runnable
using Terminal.Gui.Drawing;   // LineStyle, colores y dibujo
```

Luego se ejecuta:

``` bash
dotnet run agenda-terminal-gui.cs
```

O, si el archivo tiene el shebang `#!/usr/bin/env -S dotnet run` y permisos de ejecución, directamente:

``` bash
./agenda-terminal-gui.cs
```

## 4. Ciclo de vida de la aplicación

El ciclo de vida mínimo es:

``` csharp
using( IApplication app = Application.Create().Init()){
    Window mainWindow = new() { Title = "Tutorial Terminal.Gui v2" };
    app.Run(mainWindow);
}
```

Qué pasa ahí:

1.  `Application.Create()` crea una instancia de aplicación.
2.  `Init()` inicializa el driver de terminal, el loop principal y el manejo de input.
3.  `Run(...)` ejecuta una pantalla, ventana o diálogo.
4.  Cuando la pantalla pide detenerse, `Run` termina.
5.  El `using` llama a `Dispose()` y restaura la terminal.

## 5. Configuración

Para una primera aplicación no es obligatorio configurar nada. Conviene concentrarse primero en layout, eventos y arquitectura.

Hay dos niveles de configuración que conviene conocer:

**Nivel 1: ajustes puntuales en código.** Son una o dos líneas antes de crear la aplicación, para cambiar defaults visuales:

``` csharp
Menu.DefaultBorderStyle = LineStyle.Rounded;
```

Eso le da bordes redondeados a todos los submenús sin tocar nada más. Es lo que usa el ejemplo de menús de la sección 13.

**Nivel 2: `ConfigurationManager`** para temas, colores y bindings globales:

``` csharp
using Terminal.Gui.Configuration;

ConfigurationManager.RuntimeConfig = """
{
  "Theme": "Dark",
  "Application.DefaultKeyBindings": { "Quit": { "All": ["Ctrl+Q"] } }
}
""";

ConfigurationManager.Enable(ConfigLocations.Runtime);

using IApplication app = Application.Create().Init();
```

Regla práctica:

- si estás aprendiendo, no configures demasiado;
- si la app ya funciona, agregá tema, colores y bindings globales;
- si algo de configuración no se aplica, revisá que `ConfigurationManager.Enable(...)` ocurra **antes** de `Application.Create().Init()`.

## 6. Vistas como organizador visual

Una `View` puede ser:

- un control concreto: `Label`, `TextField`, `Button`, `CheckBox`;
- un contenedor: `FrameView`, `Window`, `Dialog`;
- una pantalla propia creada por nosotros.

Se agregan vistas con `Add`. Un primer ejemplo concreto, con dos paneles lado a lado:

``` csharp
using( IApplication app = Application.Create().Init()){
    Window mainWindow = new() { Title = "Tutorial Terminal.Gui v2" };

    FrameView maestro = new() {
        Title = "Maestro",
        X = 0, Y = 0,
        Width = Dim.Percent(25), Height = Dim.Fill()
    };
    FrameView detalle = new() {
        Title = "Detalle",
        X = Pos.Right(maestro) + 5,
        Width = Dim.Fill(), Height = Dim.Fill()
    };
    mainWindow.Add(maestro, detalle);

    app.Run( mainWindow);
}
```

Cada panel se vuelve el `SuperView` de lo que pongamos dentro. Lo que agreguemos al panel se posiciona dentro del área interna del panel, no contra toda la pantalla.

`Pos.Right(maestro) + 5` quiere decir "ubicá este panel cinco columnas a la derecha del borde derecho del panel maestro". Si después corremos el maestro o cambiamos su ancho, el detalle se reacomoda solo.

Esto permite pensar por capas:

``` text
MainWindow
├── MenuBar
├── FrameView "Maestro"
│   ├── Label encabezado
│   └── ListView
├── FrameView "Detalle"
│   └── Markdown o Label multilinea
└── Label de estado
```

Esa estructura visual también ayuda a separar responsabilidades en el código.

## 7. Runners y pantallas ejecutables

En Terminal.Gui no se ejecuta cualquier objeto: se ejecuta un objeto que participa del ciclo de vida de la app. Para eso existe `IRunnable`; las clases cómodas para empezar son `Window` y `Runnable`.

### Primer paso: todo en el constructor

Antes de empezar a separar en clases, conviene ver un ejemplo donde todo vive adentro del constructor de una `Window`. Aparece un botón centrado y, al apretarlo, se abre un diálogo de confirmación:

``` csharp
using( IApplication app = Application.Create().Init()){
    app.Run<Agenda>();
}

class Agenda : Window {
    public Agenda() {
        Title  = "Agenda";
        Width  = Dim.Fill();
        Height = Dim.Fill();

        Button boton = new() {
            Text = "¡Confirmar!",
            X = Pos.Center(),
            Y = Pos.Center()
        };

        Label resultado = new() {
            Text = "",
            X = Pos.Center(),
            Y = Pos.Bottom(boton) + 1
        };

        // Definimos un diálogo de confirmación que se mostrará al hacer click en el botón
        Dialog confirmar = new() {
            Title  = "Confirmar acción",
            Width  = 60,
            Height = 10
        };

        confirmar.Add(new Label() {
            Text = "¿Estás seguro de que quieres confirmar esta acción?",
            X = Pos.Center(),
            Y = 2
        });
        confirmar.AddButton(new Button(){Title = "Cancelar"});  // Resultado 0
        confirmar.AddButton(new Button(){Title = "Confirmar"}); // Resultado 1

        boton.Accepting += (_,_) => {
            // Apila el diálogo de confirmación y espera a que el usuario elija una opción
            App!.Run(confirmar);
            if (confirmar.Result == 1) {
                resultado.Text = "Resultado: ¡Confirmado!";
            } else {
                resultado.Text = "Resultado: Cancelado.";
            }
        };

        Add(boton, resultado);
    }
}
```

Tres cosas importantes ya están acá:

- El diálogo se ejecuta con `App!.Run(confirmar)`. Eso apila el diálogo encima de la ventana y espera a que el usuario elija una opción.
- El orden en que se agregan los botones define el `Result`: el primero es `0`, el segundo es `1`. Ese resultado se lee después de que `Run` retorna.
- La ventana de atrás no recibe input mientras el diálogo está abierto.

### Segundo paso: el diálogo como clase aparte

Cuando un diálogo crece o cuando se reutiliza, conviene sacarlo a su propia clase. El comportamiento es el mismo, pero el código de la ventana queda más limpio:

``` csharp
using( IApplication app = Application.Create().Init()){
    app.Run<Agenda>();
}

class Agenda : Window {
    public Agenda() {
        Title = "Agenda";

        Button boton     = new() { Text = "¡Confirmar!", X = Pos.Center(), Y = Pos.Center() };
        Label  resultado = new() { Text = "Resultado: ", X = Pos.Center(), Y = Pos.Bottom(boton) + 1 };

        boton.Accepting += (_,_) => {
            using Confirmar confirmar = new("¿Estás seguro de que quieres confirmar esta acción?");
            App!.Run(confirmar);
            resultado.Text = "Resultado " + (confirmar.Result==1 ? "¡Confirmado!" : "Cancelado.");
        };

        Add(boton);
        Add(resultado);
    }
}

class Confirmar : Dialog {
    public Confirmar(string Mensaje){
        Title  = "Confirmar acción";
        Width  = 60; Height = 10;

        Add(new Label() { Text = Mensaje, X = Pos.Center(), Y = 2 });

        AddButton(new Button(){Title = "Cancelar" });  // Resultado 0
        AddButton(new Button(){Title = "Confirmar"});  // Resultado 1
    }
}
```

Mismo comportamiento, pero ahora `Confirmar` se puede reutilizar desde cualquier otra ventana y aceptar un mensaje distinto cada vez.

### Tres formas equivalentes de correr una ventana

Las tres líneas siguientes producen el mismo resultado. Conviene conocer las tres porque aparecen mezcladas en código existente:

``` csharp
using (IApplication app = Application.Create().Init()) {
    app.Run(new Agenda());
}
```

``` csharp
using (IApplication app = Application.Create().Init()) {
    app.Run<Agenda>();
}
```

``` csharp
using IApplication app = Application.Create().Init();
using Agenda agenda = new();
app.Run(agenda);
```

La diferencia entre `app.Run(new Agenda())` y `app.Run<Agenda>()` es estética: en el segundo, la app construye la instancia por nosotros usando el constructor sin parámetros.

### Cómo funciona la apertura de un diálogo

``` csharp
// Crea la caja de dialogo
Confirmar confirmar = new("¿Estás seguro de que quieres confirmar esta acción?");
// 'Apila' la caja de dialogo y espera a que el usuario elija una opción
App!.Run(confirmar);
// Al cerrar el diálogo, el control vuelve a esta línea, y se puede leer el resultado
resultado.Text = "Resultado " + (confirmar.Result==1 ? "¡Confirmado!" : "Cancelado.");
```

`App!.Run(dialog)` abre un loop modal encima del actual. Mientras el diálogo está abierto, la ventana de atrás no recibe input. Cuando el diálogo se cierra (porque el usuario aprietó un botón sin marcar `e.Handled = true`), el control vuelve a la línea siguiente.

## 8. Ventanas, paneles y diálogos

Hay tres contenedores que conviene distinguir.

### `Window`

Sirve como ventana principal tradicional. Tiene borde y título.

### `Runnable`

Es útil como pantalla ejecutable sin marco. Puede ocupar todo el terminal y contener otros views. Se usa típicamente cuando se quiere combinar `MenuBar` con una `Window` adentro (ver sección sobre menús).

### `FrameView`

Es un panel con borde y título, ideal para organizar áreas dentro de una pantalla.

``` csharp
FrameView detalle = new() {
    Title = "Detalle",
    X = Pos.Right(maestro) + 5,
    Width = Dim.Fill(), Height = Dim.Fill()
};
```

### `Dialog<T>`

Es una ventana modal **tipada**: además de mostrarse en primer plano y bloquear lo de atrás, retorna un valor del tipo `T` cuando se cierra. Es el contenedor ideal para edición, confirmaciones o formularios cortos que necesitan devolver datos.

El ejemplo completo de un diálogo de edición de contactos:

``` csharp
using( IApplication app = Application.Create().Init()){
    Window mainWindow = new() { Title = "Tutorial Terminal.Gui v2" };

    Button editar    = new() { Title = "Editar contacto",    X = Pos.Center(), Y = Pos.Center() };
    Label  resultado = new() { Text  = "Resultado: Ninguno", X = Pos.Center(), Y = Pos.Bottom(editar) };

    editar.Accepting += (_,_) => {
        app.Run<Editar>();
        resultado.Text = $"Resultado: {app.GetResult<Contacto>()}";
    };

    mainWindow.Add(editar);
    mainWindow.Add(resultado);

    app.Run( mainWindow);
}

record Contacto(string Nombre, string Telefono) {
    public override string ToString() => $"{Nombre} ({Telefono})";
};

class Editar : Dialog<Contacto?> {
    public Editar(){
        Title = "Editar Contacto";
        Width = 60; Height = 10;

        // Agrego el campo Nombre
        Label labelNombre       = new() { Text = "  Nombre:", X = 1, Y = 1 };
        TextField inputNombre   = new() { X = Pos.Right(labelNombre) + 1, Y = 1, Width = 30 };
        Add(labelNombre, inputNombre);

        // Agrego el campo Teléfono
        Label labelTelefono     = new() { Text = "Teléfono:", X = 1, Y = 3 };
        TextField inputTelefono = new() { X = Pos.Right(labelTelefono) + 1, Y = 3, Width = 15 };
        Add(labelTelefono, inputTelefono);

        // Si cancelo retorno null
        Button btnCancelar = new() { Title = "Cancelar" };
        btnCancelar.Accepting += (_,_) => Result = null;
        AddButton(btnCancelar);

        // Si confirmo, valido y retorno el contacto
        Button btnConfirmar = new() { Title = "Confirmar" };
        btnConfirmar.Accepting += (_, e) => {
            if(string.IsNullOrWhiteSpace(inputNombre.Text) || inputNombre.Text.Length < 3) {
                e.Handled = true; // Evito que el diálogo se cierre automáticamente
                inputNombre.SetFocus();
                return;
            }

            if(string.IsNullOrWhiteSpace(inputTelefono.Text) || !inputTelefono.Text.All( char.IsDigit )) {
                e.Handled = true; // Evito que el diálogo se cierre automáticamente
                inputTelefono.SetFocus();
                return;
            }

            Result = new Contacto(inputNombre.Text, inputTelefono.Text);
        };
        AddButton(btnConfirmar);
    }
}
```

Hay varias decisiones de diseño que valen la pena leer despacio:

- **El tipo del diálogo es `Dialog<Contacto?>`**. Eso obliga a setear `Result` antes de cerrar y permite que la ventana de afuera obtenga el contacto sin armar canales raros.
- **El botón Cancelar setea `Result = null`** y deja que el diálogo se cierre normalmente. Eso es lo que diferencia un cancel de un confirm sin escribir lógica extra.
- **La validación usa `e.Handled = true` solo cuando falla**. Si todos los campos son válidos, no se toca `e.Handled` y el diálogo se cierra solo. Si algún campo está mal, marcamos `e.Handled = true` para abortar el cierre y devolvemos el foco al campo problemático. No hay que llamar `RequestStop()` manualmente: el cierre lo dispara el botón cuando no se intercepta.
- **Las dos formas de leer el resultado**: una es usar la API tipada `app.GetResult<Contacto>()` después de `app.Run<Editar>()`. La otra es construir el diálogo a mano y leer la propiedad `Result`:

``` csharp
// Alternativa más explícita, equivalente:
editar.Accepting += (_,_) => {
    using Editar dialog = new();
    app.Run(dialog);
    if (dialog.Result is not null) {
        resultado.Text = $"Resultado: {dialog.Result}";
    }
};
```

La diferencia conceptual entre los tres contenedores:

- `FrameView` organiza una pantalla.
- `Dialog` (o `Dialog<T>`) interrumpe la pantalla para pedir una decisión o datos.
- `Runnable`/`Window` es lo que corre en `app.Run(...)` como pantalla principal.

## 9. Layout: ubicación con `Pos` y tamaño con `Dim`

La pantalla de terminal se mide en celdas de caracteres. Una coordenada no es un pixel; es una columna y fila de texto.

Cada `View` tiene:

``` csharp
X      // posición horizontal
Y      // posición vertical
Width  // ancho
Height // alto
```

Pero no conviene pensar que siempre son enteros. En Terminal.Gui son expresiones.

### Posiciones absolutas

``` csharp
X = 1;
Y = 2;
```

Útil para cosas simples.

### Posiciones relativas a otros views

``` csharp
X = Pos.Right(searchLabel) + 1;
Y = Pos.Top(searchLabel);
```

Esto dice: "a la derecha del label, en la misma fila".

### Debajo de otro view

``` csharp
Y = Pos.Bottom(emailLabel) + 1;
```

Muy usado en formularios.

### Anclar al final

``` csharp
Y = Pos.AnchorEnd(1);
```

Sirve para una barra de estado en la última fila.

### Tamaño fijo

``` csharp
Width = 74;
Height = 22;
```

Es razonable para diálogos.

### Porcentaje

``` csharp
Width = Dim.Percent(50);
```

Sirve para partir la pantalla en paneles.

### Rellenar el espacio disponible

``` csharp
Width  = Dim.Fill();
Height = Dim.Fill(2);
```

`Dim.Fill(2)` significa ocupar el espacio disponible dejando 2 celdas libres al final.

Ejemplo completo de formulario:

``` csharp
Label nameLabel = new() { Text = "Nombre:", X = 2, Y = 1 };

TextField nameField = new() {
    X = 14, Y = Pos.Top(nameLabel),
    Width = Dim.Fill(2)
};

Label phoneLabel = new() {
    Text = "Teléfono:",
    X = Pos.Left(nameLabel),    Y = Pos.Bottom(nameLabel) + 1
};

TextField phoneField = new() {
    X = Pos.Left(nameField),    Y = Pos.Top(phoneLabel),
    Width = Dim.Fill(2)
};
```

La ventaja de este estilo es que si movemos `nameLabel`, lo demás sigue acomodándose de manera coherente.

## 10. Componentes básicos de entrada

### `Label`

Muestra texto fijo.

``` csharp
new Label { Text = "Buscar:", X = 1, Y = 2 };
```

### `TextField`

Entrada de una sola línea.

``` csharp
TextField buscar = new() {
    X = 10, Y = 2,
    Width = Dim.Percent(42)
};

buscar.TextChanged += (_, _) => Actualizar();
```

`TextChanged` es lo que permite búsqueda activa: cada vez que cambia el texto, se vuelve a calcular la lista filtrada. (En `ListView`, el evento análogo cuando cambia la selección se llama `ValueChanged`, no confundir.)

### `TextView`

Entrada o visualización de texto multilinea.

``` csharp
TextView notesField = new() {
    X = 14, Y = 11,
    Width = Dim.Fill(2),  Height = 4
};
```

### `Button`

Ejecuta una acción.

``` csharp
Button saveButton = new() {
    Text = "Guardar",
    IsDefault = true
};

saveButton.Accepting += (_, e) => {
    Save();
    e.Handled = true;
};
```

`IsDefault = true` permite que Enter active ese botón cuando el diálogo lo usa como acción principal.

### `CheckBox`

Entrada booleana.

``` csharp
CheckBox favoriteField = new() {
    Text = "_Favorito",
    Value = contact.Favorito ? CheckState.Checked : CheckState.UnChecked
};
```

El estado se lee con `Value`:

``` csharp
bool favorito = favoriteField.Value == CheckState.Checked;
```

## 11. Listas, búsqueda activa y vista de detalle

Llegado este punto, ya sabemos armar paneles y diálogos. Toca combinar esos elementos en una pantalla de tipo "maestro-detalle": una lista a la izquierda, los datos del elemento seleccionado a la derecha, y una búsqueda arriba que filtra la lista en tiempo real.

### Ejemplo completo

``` csharp
using System.Collections.ObjectModel;
using Terminal.Gui.App;       // Application, IApplication, Runnable
using Terminal.Gui.ViewBase;  // View, Pos, Dim
using Terminal.Gui.Views;     // Window, FrameView, Button, TextField, etc.

List<Contacto> agenda = [
    new("Ana",      "Andrade",   "(381)456-7890", 20),
    new("Bruno",    "Bianchi",   "(381)567-8901", 35),
    new("Carla",    "Cortés",    "(381)678-9012", 28),
    new("Diego",    "Díaz",      "(381)789-0123", 42),
    new("Elena",    "Espinosa",  "(381)890-1234", 30),
    new("Federico", "Fernández", "(381)901-2345", 45),
    new("Gabriela", "García",    "(381)012-3456", 25)
];

using( IApplication app = Application.Create().Init()){
    Window mainWindow= new() { Title = "Tutorial Terminal.Gui v2" };

    Label etiqueta   = new() { Text = "Buscar:", X = 1, Y = 1 };
    TextField buscar = new() { X = Pos.Right(etiqueta), Y = 1, Width = 43 };

    ListView maestro = new() { X = 1, Y = 3, Width = 50, Height = Dim.Fill(4) };
    Markdown detalle = new() { Text = "", X = Pos.Right(maestro) + 1, Y = 3, Width = 40, Height = Dim.Fill(4) };

    List<Contacto> filtrados = [];

    void Actualizar() {
        string texto = buscar.Text ?? "";
        filtrados = agenda.Where(c => c.Coincide(texto)).ToList();
        maestro.SetSource(new ObservableCollection<string>(filtrados.Select(c => c.ToString()).ToList()));
    }

    buscar.TextChanged += (_, _) => Actualizar();
    Actualizar();

    maestro.ValueChanged += (_, _) => {
        int indice = maestro.SelectedItem ?? 0;
        if (indice >= 0 && indice < filtrados.Count) {
            Contacto contacto = filtrados[indice];
            detalle.Text = $"""
                # Contacto

                **{contacto.NombreCompleto}**

                *Teléfono:* **{contacto.Telefono}**

                *Edad:* **{contacto.Edad}** años
                """;
        } else {
            detalle.Text = "_Sin contacto seleccionado._";
        }
    };

    mainWindow.Add(etiqueta, buscar, maestro, detalle);

    app.Run( mainWindow);
}

record Contacto(string Nombre, string Apellido, string Telefono, int Edad) {
    public string NombreCompleto => $"{Apellido}, {Nombre}";

    public bool Coincide(string texto) =>
        NombreCompleto.Contains(texto.Trim(), StringComparison.OrdinalIgnoreCase)
        || Telefono.Contains(texto.Trim());

    public override string ToString() => $" {NombreCompleto,-22} | {Telefono,10} | {Edad,2} años";
};
```

### Cómo se piensa este código

**El modelo es un `record`.** El `Contacto` no es solo un contenedor de datos: tiene tres comportamientos que la UI necesita y que conviene que estén en el modelo, no en la ventana:

- `NombreCompleto` es la concatenación `Apellido, Nombre` usada en la lista.
- `Coincide(texto)` decide si un contacto matchea la búsqueda. Concentrar la regla de búsqueda en el modelo evita que la ventana sepa demasiado.
- `ToString()` formatea la fila de la lista con anchos fijos (`,-22`, `,10`, `,2`), que es lo que hace que las columnas queden alineadas en el `ListView`.

**Dos listas en memoria.** `agenda` es la lista completa; `filtrados` es la que está mostrándose en este momento. La búsqueda no toca `agenda`: solo recalcula `filtrados`. Eso permite limpiar la búsqueda y volver al estado original sin perder nada.

**El `ListView` muestra strings, no objetos.** `SetSource` recibe una `ObservableCollection<string>`. La selección se resuelve usando el mismo índice contra `filtrados`:

``` csharp
int indice = maestro.SelectedItem ?? 0;
Contacto contacto = filtrados[indice];
```

Ese detalle es importante: nunca conviene intentar reconstruir el contacto a partir de la fila visible. La fuente de verdad es siempre el objeto en `filtrados`.

**`buscar.TextChanged` es lo que hace que la búsqueda sea activa.** Cada vez que el usuario tipea una letra, se dispara `Actualizar`, que reconstruye `filtrados` y se lo pasa al `ListView`. No hay botón "Buscar" porque no hace falta.

**`maestro.ValueChanged` se dispara cuando cambia el ítem seleccionado** (con flechas, click o búsqueda). El handler regenera el texto del panel derecho.

**El panel derecho es un `Markdown`.** Esa vista interpreta `**negrita**`, `*itálica*`, `# titulo` y demás. Para mostrar datos formateados ahorra mucho código respecto a armar `Label`s manualmente. El string usa raw strings (`$"""..."""`) para que las llaves y los saltos de línea queden legibles.

**Por qué no consultar la base en cada `ValueChanged`.** Aunque después agreguemos persistencia, conviene mantener la lista en memoria: la búsqueda activa filtra cientos de veces por segundo y no queremos golpear el almacenamiento con tantas consultas. La regla termina siendo: cada operación CRUD modifica el almacenamiento **y** la lista en memoria, en ese orden.

## 12. Diálogos de edición: separación de responsabilidades

El ejemplo de `Editar : Dialog<Contacto?>` de la sección 8 ilustra un principio que vale la pena explicitar para cuando los diálogos crezcan.

Una caja de diálogo de edición tiene tres responsabilidades, y nada más:

1.  mostrar campos,
2.  validar lo que el usuario ingresa,
3.  devolver un resultado tipado.

Cosas que **no** tiene que hacer un diálogo:

- no toca la base de datos,
- no modifica la lista global de contactos en memoria,
- no decide si la operación se persiste o no.

La ventana principal decide qué hacer con el resultado:

``` csharp
editar.Accepting += (_, _) => {
    app.Run<Editar>();
    Contacto? nuevo = app.GetResult<Contacto>();
    if (nuevo is not null) {
        // primero: la ventana decide si persistir
        // segundo: la ventana actualiza la lista en memoria
        // tercero: la ventana refresca la UI
    }
};
```

Esa separación tiene una consecuencia práctica importante: el botón **Cancelar** del diálogo funciona "gratis", porque nada se persistió todavía. No hay nada que deshacer.

Cuando el diálogo edita un contacto existente (no uno nuevo), conviene pasarle una **copia** del contacto, no el original:

``` csharp
Contacto copia = original with { };   // record con copia por valor
```

Así, mientras el usuario está editando, el original sigue intacto. Si cancela, no pasó nada. Si confirma, la ventana sabe que tiene que reemplazar el original por la copia editada.

Para clases (no records) hace falta un método `Clone()`:

``` csharp
public Contacto Clone() => new() {
    Id       = this.Id,
    Nombre   = this.Nombre,
    Telefono = this.Telefono
};
```

## 13. Menús desplegables

Un menú superior se arma con tres tipos:

- `MenuBar`: la barra de menú entera, en la parte de arriba de la pantalla.
- `MenuBarItem`: cada menú principal (`Archivo`, `Contacto`, ...).
- `MenuItem`: cada acción concreta dentro de un menú (`Salir`, `Agregar`, ...).

### Ejemplo completo

``` csharp
using Terminal.Gui.App;       // Application, IApplication, Runnable
using Terminal.Gui.Drawing;   // LineStyle
using Terminal.Gui.Input;     // Key
using Terminal.Gui.ViewBase;  // View, Pos, Dim
using Terminal.Gui.Views;     // Window, Label, MenuBar, MenuItem

using IApplication app = Application.Create().Init();

Menu.DefaultBorderStyle = LineStyle.Rounded;

Runnable raiz = new() { };

Label mensaje = new() {
    Text = "Seleccioná una opción desde la barra superior.",
    X = 1, Y = 1,
    Width = Dim.Fill(2), Height = 2,
};

MenuBar menu = new(new MenuBarItem[] {
    new("Archivo", new MenuItem[] {
        new("_Importar", "Importar desde JSON",        () => mensaje.Text = "Archivo > Importar"),
        new("_Exportar", "Exportar a JSON",            () => mensaje.Text = "Archivo > Exportar"),
        null!,
        new("_Salir", "Salir de la aplicación",        () => app.RequestStop(),                   Key.S.WithCtrl)
    }),
    new("Contacto", new MenuItem[] {
        new("_Agregar", "Agrega un nuevo contacto",    () => mensaje.Text = "Contacto > Agregar", Key.A.WithCtrl),
        new("_Editar",  "Edita un contacto existente", () => mensaje.Text = "Contacto > Editar",  Key.E.WithCtrl),
        null!,
        new("_Borrar",  "Borra un contacto existente", () => mensaje.Text = "Contacto > Borrar",  Key.B.WithCtrl)
    })
});

Window ventana = new() {
    Title = "Agenda",
    X = 0, Y = 1,
};

ventana.Add(mensaje);

raiz.Add(menu, ventana);
app.Run(raiz);
```

### Cómo se piensa este código

**Por qué hay un `Runnable` por encima de la `Window`.** El `MenuBar` ocupa la fila `Y = 0`. Si pusiéramos todo dentro de la `Window`, su borde superior taparía el menú. La solución es tener un `Runnable` como raíz (sin borde), agregarle el `MenuBar` directamente en `Y = 0`, y poner la `Window` debajo, en `Y = 1`. El `Runnable` no se ve: solo coordina.

**El constructor de `MenuBar` recibe un array de `MenuBarItem`.** Cada `MenuBarItem` toma un título y un array de `MenuItem`. No es un *initializer* `{ Menus = [...] }`: es un parámetro posicional.

**El constructor de `MenuItem` tiene cuatro variantes útiles.** Las que aparecen acá son las dos más comunes:

``` csharp
new("_Importar", "Importar desde JSON", () => ...)                  // sin atajo
new("_Salir",    "Salir de la aplicación", () => ..., Key.S.WithCtrl) // con atajo
```

El primer argumento es el texto visible. El guion bajo marca la letra rápida (`_Salir` habilita Alt+S dentro del menú abierto). El segundo es la ayuda que aparece en la barra de estado. El tercero es el callback. El cuarto, opcional, es el atajo global (`Ctrl+S`, `Ctrl+A`, etc.).

**Los separadores son `null!`.** Esa línea horizontal que divide grupos de opciones en un menú se obtiene insertando un `null!` (un null forzado para que C# no se queje) entre los `MenuItem`. Es feo pero es lo que la API espera.

**`Menu.DefaultBorderStyle = LineStyle.Rounded` antes de crear el menú.** Eso da bordes redondeados a todos los submenús que se abran. Es opcional, pero se ve mejor que el default. Tiene que ir antes de instanciar la `MenuBar`.

**El menú no implementa la lógica.** En este ejemplo los callbacks son tan chicos que están como lambdas inline, pero en una aplicación real cada `MenuItem` llamaría a un método con nombre:

``` csharp
new("_Importar", "Importar desde JSON", ImportarJson, Key.I.WithCtrl)
```

Y la lógica real vive en `ImportarJson()`. El menú solo conecta la opción visual con el método.

## 14. Manejo de teclado

Terminal.Gui ya maneja muchas teclas sin que tengamos que hacer nada:

- Tab cambia el foco entre controles.
- Las flechas navegan listas.
- Enter acepta botones o abre el ítem seleccionado.
- F9 (o Alt al solo) activa el menú.
- Escape cierra menús o diálogos según el contexto.

### Atajos en el menú

La forma más natural de agregar un atajo es pasarlo como cuarto argumento del `MenuItem`:

``` csharp
new("_Salir", "Salir de la aplicación", () => app.RequestStop(), Key.S.WithCtrl)
```

Esto hace dos cosas a la vez:

- muestra `Ctrl+S` al lado del texto del menú,
- registra el atajo globalmente.

`Ctrl+S` queda disponible en cualquier pantalla mientras el `MenuBar` esté visible, sin que tengamos que tocar ningún evento de teclado.

### La API de `Key`

Las teclas son objetos `Key`. Los modificadores se expresan con propiedades:

``` csharp
Key.S.WithCtrl    // Ctrl+S
Key.A.WithAlt     // Alt+A
Key.F.WithShift   // Shift+F
Key.N.WithCtrl.WithShift  // Ctrl+Shift+N
```

Las teclas especiales tienen nombres propios: `Key.F1`, `Key.Esc`, `Key.Enter`, `Key.Tab`, `Key.Delete`, `Key.PageUp`, etc.

### Cuándo conviene sobrescribir `OnKeyDown`

Para atajos que **no corresponden a una opción de menú** (por ejemplo, "F4 para enfocar el campo de búsqueda"), se puede sobrescribir `OnKeyDown` en la ventana:

``` csharp
protected override bool OnKeyDown(Key key) {
    if (key == Key.F4) {
        this.buscar.SetFocus();
        return true;
    }
    return base.OnKeyDown(key);
}
```

`return true` significa "esta tecla ya fue manejada"; `base.OnKeyDown(key)` deja que Terminal.Gui siga con su comportamiento por defecto.

La regla práctica: si la acción ya está en el menú, pasale el `Key` ahí. Si es una acción que solo tiene sentido como atajo (foco, navegación interna), usá `OnKeyDown`.

## 15. Persistencia con SQLite y Dapper.Contrib

### Por qué este stack

Para una agenda no necesitamos un ORM completo como Entity Framework. Tampoco queremos escribir SQL crudo para cada operación CRUD. Dapper resuelve el medio:

- **ADO.NET** + **`SqliteConnection`** abre la conexión a la base.
- **Dapper** convierte filas en objetos con `Query<T>` y ejecuta SQL con `Execute`.
- **Dapper.Contrib** agrega métodos de extensión `GetAll`, `Insert`, `Update`, `Delete` que generan el SQL básico por convención, basándose en atributos del modelo.

El resultado es un store muy chico: una clase, una conexión, y métodos de una línea para el 90% de los casos.

### El modelo

El modelo se anota con atributos de Dapper.Contrib:

```csharp
#:package Terminal.Gui@2.0.1
#:package Microsoft.Data.Sqlite@9.0.0
#:package Dapper@2.1.35
#:package Dapper.Contrib@2.0.78

using Microsoft.Data.Sqlite;  // SqliteConnection
using Dapper;                 // Execute, Query
using Dapper.Contrib.Extensions; // GetAll, Insert, Update, Delete, [Key], [Table]

using System.Text.Json;       // JsonSerializer
```

```csharp
[Table("Contactos")]
public sealed class Contacto {
    [Key]  public int    Id        { get; set; }
           public string Nombre    { get; set; } = "";
           public string Telefono  { get; set; } = "";
           public string Email     { get; set; } = "";
           public string Notas     { get; set; } = "";
           public bool   Favorito  { get; set; }

    public Contacto Clone() => new() {
        Id        = this.Id,
        Nombre    = this.Nombre,
        Telefono  = this.Telefono,
        Email     = this.Email,
        Notas     = this.Notas,
        Favorito  = this.Favorito
    };
}
```

Qué hace cada atributo:

- `[Table("Contactos")]` indica el nombre real de la tabla. Sin esto, Dapper.Contrib pluraliza el nombre de la clase, lo que no siempre acierta en español. Mejor explicitarlo siempre.
- `[Key]` marca la columna de clave primaria **auto-generada por la base** (en SQLite, `INTEGER PRIMARY KEY AUTOINCREMENT`). En `Insert`, esa propiedad se rellena con el id devuelto por la base.

### El store

``` csharp
public sealed class SqliteAgendaStore {
    private readonly string connectionString;

    public SqliteAgendaStore(string dbPath) {
        this.connectionString = $"Data Source={dbPath}";
        EnsureSchema();
    }

    private SqliteConnection Open() {
        SqliteConnection connection = new(this.connectionString);
        connection.Open();
        return connection;
    }

    private void EnsureSchema() {
        using SqliteConnection db = Open();
        db.Execute("""
            CREATE TABLE IF NOT EXISTS Contactos (
                Id       INTEGER PRIMARY KEY AUTOINCREMENT,
                Nombre   TEXT    NOT NULL,
                Telefono TEXT    NOT NULL DEFAULT '',
                Email    TEXT    NOT NULL DEFAULT '',
                Notas    TEXT    NOT NULL DEFAULT '',
                Favorito INTEGER NOT NULL DEFAULT 0
            );
        """);
    }

    public List<Contacto> GetAll() {
        using SqliteConnection db = Open();
        return db.GetAll<Contacto>().ToList();
    }

    public int Insert(Contacto contact) {
        using SqliteConnection db = Open();
        return (int)db.Insert(contact);
    }

    public bool Update(Contacto contact) {
        using SqliteConnection db = Open();
        return db.Update(contact);
    }

    public bool Delete(int id) {
        using SqliteConnection db = Open();
        return db.Delete(new Contacto { Id = id });
    }

    public void DeleteAll() {
        using SqliteConnection db = Open();
        db.Execute("DELETE FROM Contactos;");
    }
}
```

### Cómo se traduce cada operación

| Método del store        | Lo que hace Dapper.Contrib                                  |
| ----------------------- | ----------------------------------------------------------- |
| `db.GetAll<Contacto>()` | `SELECT * FROM Contactos`                                   |
| `db.Get<Contacto>(id)`  | `SELECT * FROM Contactos WHERE Id = @Id`                    |
| `db.Insert(c)`          | `INSERT INTO Contactos (...) VALUES (...)` y devuelve el id |
| `db.Update(c)`          | `UPDATE Contactos SET ... WHERE Id = @Id`                   |
| `db.Delete(c)`          | `DELETE FROM Contactos WHERE Id = @Id`                      |

El detalle a recordar es que `Delete` recibe **el objeto**, no el id. Cuando solo tenemos el id (como en una eliminación desde la lista), construimos un contacto vacío con ese id:

``` csharp
db.Delete(new Contacto { Id = id });
```

### Mapeo de tipos

SQLite no tiene tipos estrictos, pero Dapper resuelve el mapeo común:

| Tipo C#    | Tipo SQLite | Comentario                        |
| ---------- | ----------- | --------------------------------- |
| `int`      | `INTEGER`   | Directo                           |
| `string`   | `TEXT`      | Directo                           |
| `bool`     | `INTEGER`   | `1` para `true`, `0` para `false` |
| `DateTime` | `TEXT`      | Formato ISO 8601                  |
| `decimal`  | `TEXT`      | Para evitar pérdida de precisión  |

### Por qué abrir y cerrar la conexión cada vez

`Open()` se llama en cada método y se libera con `using`. Para una app de escritorio con un solo usuario y una base SQLite local, esto es trivial en costo y elimina toda una categoría de bugs (conexiones colgadas, transacciones huérfanas). En aplicaciones servidor sí tendría sentido un pool, pero acá no.

### Integración con la ventana

La ventana ya no tiene un flag `hasUnsavedChanges`. Cada operación CRUD persiste de inmediato. Reutilizando el patrón `Dialog<Contacto?>` de la sección 8:

``` csharp
private void NuevoContacto() {
    using Editar dialog = new();   // Editar : Dialog<Contacto?>
    App!.Run(dialog);

    if (dialog.Result is null) {
        return;
    }

    Contacto nuevo = dialog.Result;
    int newId = this.store.Insert(nuevo);
    nuevo.Id  = newId;

    this.contactos.Add(nuevo);
    SetStatus($"Contacto '{nuevo.Nombre}' creado.");
    Actualizar(seleccionado: newId);
}
```

El diálogo devuelve `null` si el usuario canceló y un `Contacto` válido si confirmó. El `store` se encarga de generar el `Id` nuevo en SQLite.

## 16. Importación y exportación JSON

La base de datos es la fuente de verdad. JSON sirve para dos cosas distintas:

- **Exportar**: dar al usuario un archivo legible para versionar, mandar por mail o respaldar.
- **Importar**: cargar contactos provistos en un archivo, por ejemplo al iniciar el sistema o al recibir datos de otro lado.

Estas dos operaciones no son persistencia: son **interoperabilidad**. Por eso viven en una clase separada del store.

### Opciones en el menú

La importación y la exportación tienen que aparecer como acciones del menú `Archivo`, junto con recargar y salir. Usando la API que vimos en la sección 13:

``` csharp
new("Archivo", new MenuItem[] {
    new("_Importar JSON", "Importar desde JSON", ImportarJson, Key.I.WithCtrl),
    new("_Exportar JSON", "Exportar a JSON",     ExportarJson, Key.E.WithCtrl),
    new("_Recargar",      "Recargar desde la base", Recargar,  Key.R.WithCtrl),
    null!,
    new("_Salir",         "Salir de la aplicación", () => app.RequestStop(), Key.Q.WithCtrl)
})
```

El menú no implementa la operación. Solo conecta la opción visual con un método de la ventana principal:

- `ImportarJson` lee un archivo JSON y agrega esos contactos a la base.
- `ExportarJson` escribe los contactos actuales en un archivo JSON.

Los mismos métodos se invocan desde el teclado con `Ctrl+I` y `Ctrl+E`, sin escribir nada más, porque el atajo va como cuarto argumento del `MenuItem`.

### El conversor JSON

``` csharp
public sealed class JsonAgendaIO {
    private static readonly JsonSerializerOptions JsonOptions = new() {
        WriteIndented = true,
        Encoder       = JavaScriptEncoder.UnsafeRelaxedJsonEscaping
    };

    public List<Contacto> Read(string path) {
        if (!File.Exists(path)) {
            throw new FileNotFoundException($"Archivo no encontrado: '{path}'");
        }

        string json = File.ReadAllText(path, Encoding.UTF8);
        return JsonSerializer.Deserialize<List<Contacto>>(json, JsonOptions) ?? new();
    }

    public void Write(string path, IEnumerable<Contacto> contacts) {
        string json = JsonSerializer.Serialize(
            contacts.OrderBy(contact => contact.Id),
            JsonOptions);

        File.WriteAllText(path, json, Encoding.UTF8);
    }
}
```

`UnsafeRelaxedJsonEscaping` permite que las tildes y la `ñ` se escriban como tales en el archivo, en vez de aparecer como `\u00f1`. Para datos en español es lo que casi siempre se quiere.

### Pedirle un path al usuario

Necesitamos un diálogo simple para que el usuario tipee la ruta del archivo. Se puede hacer con `FileDialog`, pero un diálogo propio que extiende `Dialog<string?>` ya alcanza para el ejemplo y mantiene el mismo patrón de tipado que `Editar`:

``` csharp
class PedirRuta : Dialog<string?> {
    public PedirRuta(string titulo, string prompt, string rutaDefault){
        Title = titulo;
        Width = 60; Height = 8;

        Label label    = new() { Text = prompt, X = 1, Y = 1 };
        TextField input = new() {
            Text = rutaDefault,
            X = 1, Y = Pos.Bottom(label) + 1,
            Width = Dim.Fill(2)
        };
        Add(label, input);

        Button btnCancelar = new() { Title = "Cancelar" };
        btnCancelar.Accepting += (_,_) => Result = null;
        AddButton(btnCancelar);

        Button btnAceptar = new() { Title = "Aceptar" };
        btnAceptar.Accepting += (_,_) => Result = input.Text ?? "";
        AddButton(btnAceptar);
    }
}
```

Y se usa así desde la ventana:

``` csharp
private string? PedirRutaArchivo(string titulo, string prompt, string rutaDefault) {
    using PedirRuta dialog = new(titulo, prompt, rutaDefault);
    App!.Run(dialog);
    return string.IsNullOrWhiteSpace(dialog.Result) ? null : dialog.Result;
}
```

### Exportar

Tomamos los contactos en memoria y los escribimos al archivo:

``` csharp
private void ExportarJson() {
    string? ruta = PedirRutaArchivo(
        "Exportar JSON",
        "Ruta de salida:",
        "agenda-export.json");

    if (ruta is null) {
        return;
    }

    try {
        this.json.Write(ruta, this.contactos);
        SetStatus($"Exportados {this.contactos.Count} contactos a '{ruta}'.");
    } catch (Exception ex) {
        MessageBox.ErrorQuery(App!, "Exportar", $"Error: {ex.Message}", "OK");
    }
}
```

### Importar

Importar es más delicado: hay que decidir qué pasa con los contactos ya existentes. La estrategia más simple es **agregar**: cada contacto del JSON entra como nuevo registro en la base, con un id fresco asignado por SQLite.

``` csharp
private void ImportarJson() {
    string? ruta = PedirRutaArchivo(
        "Importar JSON",
        "Ruta del archivo a importar:",
        "agenda-export.json");

    if (ruta is null) {
        return;
    }

    try {
        List<Contacto> importados = this.json.Read(ruta);

        int respuesta = MessageBox.Query(
            App!,
            "Importar",
            $"Se agregarán {importados.Count} contactos del archivo a la base. ¿Continuar?",
            "No", "Sí") ?? 0;

        if (respuesta != 1) {
            return;
        }

        foreach (Contacto contacto in importados) {
            contacto.Id = 0;                          // SQLite asigna id nuevo
            int newId = this.store.Insert(contacto);
            contacto.Id = newId;
        }

        CargarContactos();
        SetStatus($"{importados.Count} contactos importados desde '{ruta}'.");
    } catch (Exception ex) {
        MessageBox.ErrorQuery(App!, "Importar", $"Error: {ex.Message}", "OK");
    }
}
```

Puntos clave:

- **Reseteamos `Id = 0` antes de insertar** para que SQLite genere un id nuevo. Si dejáramos los ids del archivo, podríamos chocar con ids ya existentes en la base.
- **Confirmación explícita**: importar es una operación que agrega datos que después hay que limpiar si fue un error.
- **Recarga completa al final**: después de importar, refrescamos `contactos` desde la base para que la lista en memoria refleje exactamente lo que hay persistido (incluidos los ids nuevos).
- **Manejo de errores con `MessageBox.ErrorQuery`**: archivos inexistentes, JSON malformado o problemas de permisos no deben tirar el programa.

### Dos formatos, un solo modelo

Lo importante es que el `Contacto` se serializa y deserializa **igual** en JSON y en SQLite. No hay dos versiones del modelo. Los atributos `[Table]` y `[Key]` no afectan la serialización JSON, y la propiedad `Favorito` se mapea a `INTEGER` en SQLite y a `true`/`false` en JSON sin que tengamos que hacer nada.

``` csharp
private void NuevoContacto() {

        return;
```

``` csharp
private void CargarContactos() {

private void Recargar() {
    CargarContactos();
```

### Actualizar

``` csharp
private void EditarContacto() {
    Contacto? seleccionado = ContactoSeleccionado();
    if (seleccionado is null) {
        MessageBox.Query(App!, "Editar", "Seleccioná un contacto.", "OK");
        return;
    }

    using Editar dialog = new(seleccionado.Clone());
    App!.Run(dialog);

    if (dialog.Result is null) {
        return;
    }

    Contacto editado = dialog.Result;
    this.store.Update(editado);
    int index = this.contactos.FindIndex(c => c.Id == seleccionado.Id);
    this.contactos[index] = editado;

    SetStatus($"Contacto '{editado.Nombre}' actualizado.");
    Actualizar(seleccionado: editado.Id);
}
```

El `Clone()` evita editar el objeto original mientras el usuario todavía puede cancelar.

### Eliminar

``` csharp
private void EliminarContacto() {
    Contacto? seleccionado = ContactoSeleccionado();
    if (seleccionado is null) {
        return;
    }

    int respuesta = MessageBox.Query(
        App!,
        "Eliminar contacto",
        $"¿Eliminar a {seleccionado.Nombre}?",
        "No",
        "Sí") ?? 0;

    if (respuesta == 1) {
        this.store.Delete(seleccionado.Id);
        this.contactos.RemoveAll(c => c.Id == seleccionado.Id);

        SetStatus("Contacto eliminado.");
        Actualizar();
    }
}
```

## 18. Orden recomendado para construir una TUI

El orden que seguimos en clase, replicable en cualquier TUI:

1.  **Paneles** (`25.2`). `Window` con dos `FrameView` lado a lado. Sin lógica, solo layout.
2.  **Subclase de `Window`** (`25.3`). Mover la pantalla a su propia clase con botón y diálogo de confirmación inline.
3.  **Refactor del diálogo** (`25.4`). Sacar el `Dialog` a su propia clase para que sea reutilizable.
4.  **Diálogo tipado con validación** (`25.5`). Pasar de `Dialog` a `Dialog<T>` y validar campos.
5.  **`ListView` + búsqueda activa + detalle** (`25.6`). Vista maestro-detalle con `TextChanged` y `Markdown`.
6.  **Menú superior** (`25.7`). `MenuBar` con atajos de teclado vía `Key.X.WithCtrl`.
7.  **Modelo de datos** (`Contacto` con `[Table]` y `[Key]`) y store de persistencia (`SqliteAgendaStore`). Conviene verificarlo con un script chico antes de tocar la UI: que `Insert/GetAll/Update/Delete` funcionen.
8.  **Conversor JSON** (`JsonAgendaIO`) y verificación con un round-trip de prueba.
9.  **Acciones CRUD conectadas al store**: el menú dispara métodos de la ventana, que persisten y refrescan la lista en memoria.
10. **Importación y exportación JSON** desde el menú `Archivo`.
11. **Recargar** desde la base y **confirmar salida** si hay cambios pendientes.

Si se intenta hacer todo junto, los errores de layout, eventos y persistencia se mezclan. Especialmente el store: probarlo desde un archivo aparte antes de integrarlo a la UI evita muchas horas de depuración cruzada.

## 19. Errores comunes

### Olvidar `[Table]` o `[Key]` en el modelo

Síntoma: `Insert` falla con un error sobre tabla inexistente, o el id no se rellena después de insertar.

Solución: anotar la clase con `[Table("Contactos")]` y la propiedad clave con `[Key]`. Sin `[Key]`, Dapper.Contrib busca una propiedad llamada `Id` por convención, pero ser explícito evita sorpresas.

### `Delete` recibe el objeto, no el id

`db.Delete(5)` no compila o no hace lo que parece. Hay que pasar un objeto con el id seteado:

``` csharp
db.Delete(new Contacto { Id = id });
```

### Conexión no cerrada

Síntoma: el archivo `.db` queda bloqueado, o las pruebas dejan procesos colgados.

Solución: siempre usar `using SqliteConnection db = Open();`. El `using` garantiza que la conexión se cierra incluso si la operación lanza excepción.

### Importar JSON conservando los ids del archivo

Si el JSON viene con ids `1, 2, 3` y la base ya tiene contactos con esos ids, `Insert` falla por clave duplicada. Conviene resetear `contact.Id = 0` antes de insertar y dejar que SQLite asigne ids frescos.

### Mantener la lista en memoria desactualizada

Después de cada operación CRUD, hay que actualizar **dos** lugares: la base con el `store`, y la lista `contactos` en memoria. Si solo se actualiza uno, la UI queda inconsistente con la base hasta el próximo `Recargar`. La regla: persistir primero, actualizar memoria después, refrescar la lista por último.

### Guardar directamente desde el diálogo

No conviene. El diálogo debe devolver datos (vía `Result`). La ventana decide cuándo persistir. Esto es lo que hace posible que el botón "Cancelar" del diálogo funcione: no hay nada para deshacer porque nada se persistió todavía.

### Perder la selección al filtrar

Si al refrescar la lista siempre seleccionás el primer elemento, la UI se siente errática. Conviene mantener el `Id` seleccionado:

``` csharp
int? idAMantener = idSolicitado ?? ContactoSeleccionado()?.Id;
```

### Intentar parsear el texto de la fila

No uses la fila formateada como fuente de verdad. La fuente de verdad es el objeto `Contacto` en `filtrados`.
