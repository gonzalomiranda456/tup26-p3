## 1. El problema de la clase 1

La API que construimos en la clase anterior funciona. Podés crear tareas, listarlas, obtenerlas por id. Pero tiene un problema fundamental:

```csharp
var tareas = new List<Tarea> {
    new(1, "Estudiar HTTP",        false),
    new(2, "Practicar Minimal API", false),
};
```

Esa lista vive en la **RAM** del proceso. Cuando el servidor se detiene, la RAM se libera. Todo lo que el usuario creó desaparece. La próxima vez que arranque el servidor, la lista empieza de cero.

Esto no es un bug — es una consecuencia directa de cómo funciona la memoria. La RAM es rápida y abundante, pero **volátil**: no sobrevive al apagado del proceso.

Para que los datos persistan necesitamos guardarlos en algo que sobreviva: un **archivo en disco**. Y la forma estructurada y eficiente de guardar datos en disco es una **base de datos**.

---

## 2. Bases de datos relacionales desde primeros principios

Una base de datos relacional organiza los datos en **tablas**. Una tabla es exactamente lo que parece: filas y columnas.

Si la clase anterior hubiéramos guardado las tareas en una base de datos, la tabla `Tareas` se vería así:

```
┌────┬───────────────────────┬────────────┐
│ Id │ Texto                 │ Completada │
├────┼───────────────────────┼────────────┤
│  1 │ Estudiar HTTP         │ false      │
│  2 │ Practicar Minimal API │ false      │
│  3 │ Hacer el parcial      │ true       │
└────┴───────────────────────┴────────────┘
```

Cada **columna** es un campo (equivale a una propiedad del objeto). Cada **fila** es un registro (equivale a una instancia del objeto). La columna `Id` identifica unívocamente cada fila — es la **clave primaria**.

Para interactuar con esa tabla, las bases de datos relacionales usan un lenguaje llamado **SQL**:

```sql
-- Leer todas las tareas
SELECT * FROM Tareas;

-- Leer una tarea por id
SELECT * FROM Tareas WHERE Id = 1;

-- Crear una tarea nueva
INSERT INTO Tareas (Texto, Completada) VALUES ('Nueva tarea', false);

-- Actualizar una tarea
UPDATE Tareas SET Completada = true WHERE Id = 1;

-- Eliminar una tarea
DELETE FROM Tareas WHERE Id = 1;
```

Estas cinco operaciones son exactamente las mismas que los cinco verbos HTTP que vimos en la clase anterior. No es casualidad:

```
GET    /tareas/{id}  ←→  SELECT * FROM Tareas WHERE Id = ?
POST   /tareas       ←→  INSERT INTO Tareas ...
PUT    /tareas/{id}  ←→  UPDATE Tareas SET ... WHERE Id = ?
DELETE /tareas/{id}  ←→  DELETE FROM Tareas WHERE Id = ?
```

La API es la capa que traduce HTTP a operaciones de base de datos.

## 3. El problema de hablar con la base de datos

Podríamos escribir SQL directamente en C#:

```csharp
var comando = new SqliteCommand("SELECT * FROM Tareas WHERE Id = @id", conexion);
comando.Parameters.AddWithValue("@id", 1);
var reader = await comando.ExecuteReaderAsync();

while (await reader.ReadAsync()) {
    var id         = reader.GetInt32(0);
    var texto      = reader.GetString(1);
    var completada = reader.GetBoolean(2);
    // ...
}
```

Funciona. Pero tiene varios problemas:

**Es repetitivo.** Para cada tabla hay que escribir el mismo código de apertura de conexión, creación de comando, lectura columna por columna, mapeo a objeto.

**Es frágil.** Si renombrás una propiedad en C# o una columna en la base de datos, el compilador no te avisa. El error aparece en runtime.

**Es verboso.** Para hacer un simple `SELECT *` necesitás diez líneas.

Hay una forma mejor.

---

## 4. EF Core: un ORM

**Entity Framework Core** es un **ORM** (*Object-Relational Mapper*): una librería que traduce automáticamente entre objetos C# y tablas relacionales.

La idea es simple: vos escribís C#, EF Core genera el SQL.

```csharp
// En lugar de esto:
var comando = new SqliteCommand("SELECT * FROM Tareas WHERE Id = @id", conexion);
// ... diez líneas de código ...

// Escribís esto:
var tarea = await db.Tareas.FindAsync(id);
```

EF Core mira tu clase `Tarea`, sabe que tiene una tabla `Tareas` en la base de datos, genera el SQL correcto y mapea el resultado de vuelta a un objeto C#. Todo automático.

Ventajas concretas:

- **El compilador verifica tu código.** Si renombrás `Tarea.Texto` a `Tarea.Descripcion`, el compilador te avisa en todos los lugares donde se usa. Con SQL en strings, no.
- **Menos código.** Un `FindAsync(id)` reemplaza diez líneas de ADO.NET.
- **Independencia de la base de datos.** El mismo código C# funciona con SQLite, SQL Server, PostgreSQL, MySQL. Solo cambia la cadena de conexión y el paquete de driver.

**¿Por qué SQLite?**

SQLite es una base de datos que vive en **un solo archivo**. No necesita servidor, no necesita configuración, no necesita instalación. El archivo se crea automáticamente. Es perfecta para desarrollo y para aplicaciones pequeñas.

## 5. Los tres conceptos centrales

### `DbContext` — la sesión con la base de datos

El `DbContext` es la clase central de EF Core. Representa **una sesión de trabajo** con la base de datos: abrís el contexto, hacés operaciones, cerrás el contexto.

Pensalo como la "puerta" a la base de datos. Todas las consultas y modificaciones pasan por él.

```csharp
public class TareasContext : DbContext {
    // ...
}
```

### `DbSet<T>` — una tabla

Dentro del `DbContext`, cada propiedad `DbSet<T>` representa una tabla. `DbSet<Tarea>` es la tabla de tareas.

```csharp
public class TareasContext : DbContext {
    public DbSet<Tarea> Tareas { get; set; }
}
```

`DbSet<T>` implementa `IQueryable<T>`, así que podés usar LINQ directamente sobre él:

```csharp
var tareas = await db.Tareas.Where(t => !t.Completada).ToListAsync();
```

EF Core traduce ese LINQ a SQL y ejecuta la consulta en la base de datos.

### `SaveChangesAsync` — confirmar los cambios

EF Core no guarda los cambios inmediatamente. Cuando agregás, modificás o eliminás objetos, los registra en memoria como "cambios pendientes". Recién cuando llamás `SaveChangesAsync()` los escribe en la base de datos en una sola transacción.

```csharp
db.Tareas.Add(nuevaTarea);     // registra el cambio en memoria
await db.SaveChangesAsync();   // escribe en la base de datos
```

## 7. Definir el modelo y el contexto

### El modelo

Vamos a usar el mismo `record Tarea` de la clase anterior, con un pequeño cambio: EF Core necesita que la clave primaria sea una **propiedad mutable**. Los `record` con parámetros en el constructor tienen propiedades de solo lectura (`init`), pero EF Core puede leer claves primarias de esa forma sin problema.

```csharp
record Tarea(int Id, string Texto, bool Completada);
```

> EF Core identifica la clave primaria por convención: una propiedad llamada `Id` o `{NombreClase}Id` se convierte automáticamente en clave primaria. No hace falta ningún atributo.

### El contexto

```csharp
using Microsoft.EntityFrameworkCore;

class TareasContext : DbContext {
    public DbSet<Tarea> Tareas { get; set; }

    protected override void OnConfiguring(DbContextOptionsBuilder options)
        => options.UseSqlite("Data Source=tareas.db");
}
```

`OnConfiguring` le dice a EF Core qué base de datos usar. `Data Source=tareas.db` indica que el archivo se llama `tareas.db` y se crea en el directorio de trabajo.

---

## 8. Crear la base de datos

EF Core no crea la base de datos automáticamente al arrancar. Hay que decirle explícitamente que cree las tablas que corresponden a los `DbSet` del contexto. Hay dos formas.

### Opción A — `EnsureCreated` (simple, para desarrollo)

```csharp
using var db = new TareasContext();
db.Database.EnsureCreated();
```

Crea la base de datos y las tablas **si no existen todavía**. Si ya existen, no hace nada. Es la forma más simple.

Limitación: si después cambiás el modelo (agregás una columna, por ejemplo), `EnsureCreated` no actualiza la base de datos existente. Para eso están las migraciones.

### Opción B — Migraciones (la forma correcta)

Las migraciones son archivos C# que describen los cambios al esquema de la base de datos a lo largo del tiempo. Son la forma recomendada porque permiten **evolucionar** el esquema sin perder datos.

**Crear la primera migración:**

```bash
dotnet ef migrations add CrearTablaTareas
```

Esto genera una carpeta `Migrations/` con un archivo que describe cómo crear la tabla `Tareas`.

**Aplicar las migraciones a la base de datos:**

```bash
dotnet ef database update
```

Crea el archivo `tareas.db` y ejecuta las migraciones pendientes.

> Para esta clase usamos `EnsureCreated` — es más simple y suficiente. Las migraciones son la herramienta correcta en proyectos reales donde el esquema evoluciona.

### Registrar el contexto en el contenedor de DI

En lugar de crear el contexto a mano con `new TareasContext()`, lo registramos en el sistema de inyección de dependencias de ASP.NET Core. Así los handlers lo reciben automáticamente como parámetro:

```csharp
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddDbContext<TareasContext>(options =>
    options.UseSqlite("Data Source=tareas.db"));

var app = builder.Build();

// Crear la base de datos si no existe
using (var scope = app.Services.CreateScope())
    scope.ServiceProvider.GetRequiredService<TareasContext>().Database.EnsureCreated();
```

Con esto registrado, cualquier handler puede pedir un `TareasContext` como parámetro y ASP.NET Core se lo entrega:

```csharp
app.MapGet("/tareas", async (TareasContext db) => {
    return await db.Tareas.ToListAsync();
});
```

## 9. Async/await en operaciones de base de datos

Las operaciones de base de datos son **lentas** comparadas con la RAM — involucran acceso a disco y, en aplicaciones reales, viajes de red. Si el servidor espera bloqueado a que termine cada consulta, no puede atender otros requests mientras tanto.

`async`/`await` resuelve esto: el servidor **suspende** el handler mientras espera la base de datos y atiende otros requests. Cuando la base de datos responde, retoma el handler desde donde lo dejó.

```csharp
// Sin async: el hilo queda bloqueado esperando la BD
app.MapGet("/tareas", (TareasContext db) => {
    return db.Tareas.ToList();   // bloquea hasta que termina
});

// Con async: el hilo queda libre mientras espera la BD
app.MapGet("/tareas", async (TareasContext db) => {
    return await db.Tareas.ToListAsync();   // suspende y libera el hilo
});
```

La regla práctica es simple: **todas las operaciones de EF Core tienen versión `Async`**. Siempre usá la versión async y poné `await` delante:

| Sincrónico | Asincrónico |
|------------|-------------|
| `ToList()` | `ToListAsync()` |
| `Find(id)` | `FindAsync(id)` |
| `FirstOrDefault()` | `FirstOrDefaultAsync()` |
| `SaveChanges()` | `SaveChangesAsync()` |
| `Any()` | `AnyAsync()` |

Cuando un handler usa `await`, su firma necesita `async` y retornar `Task<T>` o simplemente usar la inferencia de tipos:

```csharp
app.MapGet("/tareas", async (TareasContext db) =>
    await db.Tareas.ToListAsync());
```

## 10. Los cuatro endpoints con EF Core

Ahora reemplazamos la lista en memoria por operaciones reales de base de datos.

### GET /tareas — listar todas

```csharp
app.MapGet("/tareas", async (TareasContext db) =>
    await db.Tareas.ToListAsync());
```

`ToListAsync()` ejecuta `SELECT * FROM Tareas` y mapea cada fila a un objeto `Tarea`.

### GET /tareas/{id} — obtener una

```csharp
app.MapGet("/tareas/{id}", async (int id, TareasContext db) => {
    var tarea = await db.Tareas.FindAsync(id);
    return tarea is null ? Results.NotFound() : Results.Ok(tarea);
});
```

`FindAsync(id)` ejecuta `SELECT * FROM Tareas WHERE Id = @id`. Si no encuentra nada, retorna `null`.

### POST /tareas — crear

```csharp
app.MapPost("/tareas", async (Tarea nueva, TareasContext db) => {
    db.Tareas.Add(nueva);
    await db.SaveChangesAsync();
    return Results.Created($"/tareas/{nueva.Id}", nueva);
});
```

Tres pasos: agregar el objeto al `DbSet`, guardar los cambios (ejecuta el `INSERT`), retornar `201` con la tarea creada. 

Nota importante: **no asignamos el `Id` manualmente**. La base de datos lo genera automáticamente (autoincrement). Después de `SaveChangesAsync()`, EF Core actualiza la propiedad `Id` del objeto con el valor asignado por la base de datos.

Por eso el cliente puede mandar `"id": 0` en el JSON — ese valor se ignora y EF Core asigna el id real.

### PUT /tareas/{id} — actualizar

```csharp
app.MapPut("/tareas/{id}", async (int id, Tarea actualizada, TareasContext db) => {
    var tarea = await db.Tareas.FindAsync(id);

    if (tarea is null)
        return Results.NotFound();

    db.Entry(tarea).CurrentValues.SetValues(actualizada with { Id = id });
    await db.SaveChangesAsync();

    return Results.Ok(tarea);
});
```

`db.Entry(tarea).CurrentValues.SetValues(...)` actualiza todas las propiedades del objeto rastreado por EF Core con los valores del objeto nuevo. El `with { Id = id }` asegura que el id en el cuerpo del request no pueda sobreescribir el id de la URL.

### DELETE /tareas/{id} — eliminar

```csharp
app.MapDelete("/tareas/{id}", async (int id, TareasContext db) => {
    var tarea = await db.Tareas.FindAsync(id);

    if (tarea is null)
        return Results.NotFound();

    db.Tareas.Remove(tarea);
    await db.SaveChangesAsync();

    return Results.NoContent();
});
```

`Remove` marca el objeto para eliminar. `SaveChangesAsync` ejecuta el `DELETE`. El código de respuesta correcto es `204 No Content`: la operación fue exitosa pero no hay nada que devolver.

## 11. El programa completo

```csharp
using Microsoft.EntityFrameworkCore;

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddDbContext<TareasContext>(options =>
    options.UseSqlite("Data Source=tareas.db"));

var app = builder.Build();

// Crear la base de datos si no existe
using (var scope = app.Services.CreateScope())
    scope.ServiceProvider.GetRequiredService<TareasContext>().Database.EnsureCreated();

// --- Endpoints ---

app.MapGet("/tareas", async (TareasContext db) =>
    await db.Tareas.ToListAsync());

app.MapGet("/tareas/{id}", async (int id, TareasContext db) => {
    var tarea = await db.Tareas.FindAsync(id);
    return tarea is null ? Results.NotFound() : Results.Ok(tarea);
});

app.MapPost("/tareas", async (Tarea nueva, TareasContext db) => {
    db.Tareas.Add(nueva);
    await db.SaveChangesAsync();
    return Results.Created($"/tareas/{nueva.Id}", nueva);
});

app.MapPut("/tareas/{id}", async (int id, Tarea actualizada, TareasContext db) => {
    var tarea = await db.Tareas.FindAsync(id);
    if (tarea is null) return Results.NotFound();
    db.Entry(tarea).CurrentValues.SetValues(actualizada with { Id = id });
    await db.SaveChangesAsync();
    return Results.Ok(tarea);
});

app.MapDelete("/tareas/{id}", async (int id, TareasContext db) => {
    var tarea = await db.Tareas.FindAsync(id);
    if (tarea is null) return Results.NotFound();
    db.Tareas.Remove(tarea);
    await db.SaveChangesAsync();
    return Results.NoContent();
});

app.Run();

// --- Modelo ---

record Tarea(int Id, string Texto, bool Completada);

// --- Contexto ---

class TareasContext : DbContext {
    public TareasContext(DbContextOptions<TareasContext> options) : base(options) { }
    public DbSet<Tarea> Tareas { get; set; }
}
```

## 12. Resumen

```
El problema:
  La RAM es volátil — los datos desaparecen al reiniciar el proceso.
  Necesitamos guardar en disco.

La solución:
  Base de datos relacional — datos en tablas (filas y columnas).
  EF Core — traduce entre objetos C# y tablas SQL.

Los tres conceptos de EF Core:
  DbContext   → la sesión con la base de datos
  DbSet<T>    → una tabla; soporta LINQ directamente
  SaveChangesAsync() → confirma todos los cambios pendientes en una transacción

El ciclo de una operación de escritura:
  1. Modificar el DbSet (Add, Remove, o editar propiedades)
  2. await db.SaveChangesAsync()   ← recién acá toca la base de datos

El ciclo de una operación de lectura:
  await db.Entidades.ToListAsync()          → SELECT *
  await db.Entidades.FindAsync(id)          → SELECT WHERE Id = ?
  await db.Entidades.Where(...).ToListAsync() → SELECT WHERE condición

Async/await:
  Las operaciones de BD son lentas → siempre usar la versión Async
  El handler declara async → retorna Task automáticamente
  await suspende el handler y libera el hilo para otros requests

Lo que cambia respecto a la clase 1:
  Antes: var tareas = new List<Tarea>()
  Ahora: DbSet<Tarea> en una base de datos SQLite
  Los endpoints son casi idénticos — solo cambia la fuente de datos
```

**Lo que le falta a esta API:**

Los datos persisten, pero el frontend sigue siendo REST Client. En la próxima clase construimos la interfaz con Blazor: una página web que consume esta API.

## Ejercicio para la casa

1. Agregá una propiedad `FechaCreacion` de tipo `DateTime` al record `Tarea`. Asignala automáticamente en el endpoint `POST` con `DateTime.UtcNow`. El cliente no debería tener que mandarla.

2. Creá un endpoint `PATCH /tareas/{id}/completar` que solo marca una tarea como completada, sin necesidad de mandar el objeto completo. Debe devolver `404` si la tarea no existe y `200` con la tarea actualizada si existe.

   > Tip: `PATCH` es otro verbo HTTP que significa "modificar parcialmente". Se mapea con `app.MapPatch(...)`.
