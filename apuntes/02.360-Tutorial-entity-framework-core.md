## ¿Qué vamos a construir?

Una agenda de contactos de línea de comandos donde cada contacto puede tener:

- Un **nombre**.
- Una **categoría** (Familia, Trabajo, Amigos...).
- Varias **direcciones**, donde una dirección puede ser un teléfono o un email.

Nos enfocamos a propósito en consola: sin formularios, sin UI, sin frameworks web. Queremos que toda la atención vaya al **modelado de datos y al acceso a la base**. Una vez que entendés EF Core en consola, usarlo en MAUI, ASP.NET o lo que sea es *exactamente el mismo código* — cambia lo de alrededor.

El objetivo de esta clase es entender **qué es un ORM y cómo trabaja con relaciones**. Para eso elegimos un modelo con tres tablas relacionadas: es lo suficientemente simple para entenderlo de un vistazo, y lo suficientemente rico para ver aparecer las relaciones 1:N y N:1, las foreign keys, la carga de datos relacionados, y el borrado en cascada.

Lo que queremos entender al final:

- Qué es un ORM y cuándo conviene usarlo.
- Cómo se traduce una clase C# en una tabla SQL.
- Cómo se traducen las relaciones entre clases en foreign keys.
- Qué es un `DbContext` y por qué su ciclo de vida importa.
- Cómo funciona el *change tracking*.
- Cómo se cargan datos relacionados con `Include`.
- Dónde está el límite entre "dejá que EF Core decida" y "necesito pensar el SQL".

> **Una aclaración sobre migraciones.** En las partes prácticas del proyecto (partes 1 a 8) NO vamos a usar migraciones — vamos a dejar que EF Core cree la base automáticamente con `EnsureCreated()`. Las migraciones son una pieza fundamental cuando un proyecto evoluciona, pero introducirlas hoy en el contexto del ejercicio agrega ruido que nos aleja del objetivo central. Las cubrimos en profundidad en la **Parte 12** de este mismo documento.

## Parte 1 — Primeros principios

Antes de tocar código, necesitás tener claros algunos conceptos. Si te los saltás, vas a terminar copiando código sin entender qué hace.

### 1.1 — El problema del impedance mismatch

Tu programa piensa en **objetos**: un `Contacto` con propiedades, referencias a otros objetos, listas, herencia. Tu base de datos piensa en **tablas**: filas y columnas, claves primarias, foreign keys, sin polimorfismo.

Los dos modelos no coinciden. Este desajuste tiene un nombre clásico: *object-relational impedance mismatch*. Sin un ORM, el código de acceso a datos termina siendo así:

```csharp
// Sin ORM — SQL a mano, mapeo manual, propenso a errores
static Contacto? BuscarContacto(int id) {
    using var cmd = new SqliteCommand(
        "SELECT Id, Nombre FROM Contactos WHERE Id = @id", conn);
    cmd.Parameters.AddWithValue("@id", id);
    using var reader = cmd.ExecuteReader();
    if (reader.Read()) {
        var contacto = new Contacto {
            Id     = reader.GetInt32(0),
            Nombre = reader.GetString(1),
        };
        // ahora otra query para las direcciones...
        // y otra para la categoría...
        return contacto;
    }
    return null;
}
```

Multiplicá eso por cada tabla, cada operación, cada cambio de schema, cada join. Los cambios en la base exigen rastrear cada query afectado, y no hay forma de verificar en compilación que una consulta es válida.

Un **ORM** (Object-Relational Mapper) automatiza esa traducción. Vos trabajás con objetos; la librería traduce a SQL por vos, y sabe cómo seguir las referencias entre ellos.

### 1.2 — Qué es Entity Framework Core

EF Core es el ORM oficial de Microsoft para .NET. Hace cuatro cosas principales:

1. **Mapea clases a tablas.** Una clase `Contacto` se convierte en la tabla `Contactos`. Sus propiedades se vuelven columnas. EF Core infiere los tipos, las claves, los nombres.
2. **Traduce LINQ a SQL.** Escribís `db.Contactos.Where(c => c.Nombre.StartsWith("A")).ToList()` y EF Core genera el `SELECT ... WHERE Nombre LIKE 'A%'` por vos.
3. **Entiende relaciones.** Si `Contacto` tiene una lista de `Direccion`, EF Core sabe generar la foreign key, sabe hacer el JOIN cuando las pedís juntas, y sabe borrarlas en cascada cuando borrás el contacto.
4. **Rastrea cambios.** Cargás un contacto, le modificás el nombre, llamás `SaveChanges()`, y EF Core genera el `UPDATE` correcto — solo para las columnas que cambiaron.

EF Core 10 salió en noviembre 2025 y es release LTS soportado hasta noviembre 2028. Corre sobre .NET 10.

### 1.3 — Por qué SQLite

SQLite es un motor de base de datos **embebido**: toda la base vive en un único archivo. No hay servidor, no hay conexiones de red, no hay cuentas de usuario.

Ventajas para nuestro caso:

- **Cero configuración.** No hay que instalar nada. El paquete NuGet ya trae el motor.
- **Portable.** El archivo `.db` lo podés copiar, mover, versionar, compartir.
- **Multiplataforma.** Windows, Linux, macOS, sin cambios.
- **Real.** No es una simulación — es un motor SQL completo, usado en producción por Android, iOS, Firefox, y básicamente toda la industria.

Desventajas (que no nos afectan en este ejercicio):

- Mala concurrencia de escritura: si varios procesos escriben a la vez, se traban.
- No soporta stored procedures ni triggers avanzados.
- No escala horizontalmente.

Para una app de escritorio o una agenda personal, estas desventajas no existen. Para una app web con 10000 usuarios escribiendo al mismo tiempo, cambiás a PostgreSQL o SQL Server **sin tocar el código de EF Core** — solo el provider. Ese es parte del valor del ORM.

### 1.4 — Code First

Hay dos formas de empezar un proyecto con EF Core:

- **Database First**: ya existe una base de datos y vos generás clases C# a partir de ella. Típico cuando heredás un sistema legacy.
- **Code First**: escribís las clases en C# y EF Core genera la base. Es el enfoque moderno por defecto.

Nosotros usamos **Code First**. Nuestras clases C# son la fuente de verdad; la base se deriva de ellas. Los cambios al modelo se pueden expresar como **migraciones** que EF Core convierte en `ALTER TABLE`, `CREATE TABLE`, etc. — pero eso lo veremos en la Parte 12.

### 1.5 — Las relaciones, traducidas

Esta es la parte central de la clase. Pensemos cómo se traducen las relaciones entre objetos a tablas SQL.

**Relación N:1 (muchos contactos pertenecen a una categoría):**

En objetos:
```csharp
class Contacto  { Categoria Categoria; }
class Categoria { string Nombre; }
```

En tablas:
```
Contactos(Id, Nombre, CategoriaId → Categorias.Id)
Categorias(Id, Nombre)
```

La clase `Contacto` guarda una *referencia* a `Categoria`. La tabla `Contactos` guarda una *foreign key* a la fila de `Categorias`.

**Relación 1:N (un contacto tiene muchas direcciones):**

En objetos:
```csharp
class Contacto  { List<Direccion> Direcciones; }
class Direccion { Contacto Contacto; }
```

En tablas:
```
Contactos(Id, Nombre)
Direcciones(Id, Valor, ContactoId → Contactos.Id)
```

La lista vive del lado de `Contacto` en el modelo de objetos. En el modelo relacional, la foreign key vive del lado "muchos" — es decir, cada `Direccion` recuerda a qué `Contacto` pertenece. Esa es la forma natural de expresarlo en SQL: en una base relacional no hay "listas", solo relaciones.

**Relación N:N (un libro tiene muchos géneros y un género aplica a muchos libros):**

En objetos:
```csharp
class Libro   { List<Genero> Generos; }
class Genero  { List<Libro> Libros; }
```

En tablas SQL se necesita una **tabla de unión**:
```
Libros(Id, Titulo)
Generos(Id, Nombre)
LibroGeneros(LibroId → Libros.Id, GeneroId → Generos.Id)
```

EF Core puede crear la tabla de unión automáticamente, o podés definirla explícitamente si necesitás guardar datos extras en la relación (como fecha de publicación o rol del autor).

**Relación 1:1 (un libro tiene un único detalle editorial):**

En tablas, la FK vive en la tabla del lado "dependiente":
```
Libros(Id, Titulo)
DetallesEditoriales(Id, ISBN, LibroId → Libros.Id)
```

Cubrimos los tres tipos con ejemplos completos en la **Parte 11**.

### 1.6 — El DbContext: puerta de entrada a todo

Todo lo que hagas con EF Core pasa por un objeto `DbContext`. Es la "unidad de trabajo" (*unit of work*) — representa una sesión con la base. Sus responsabilidades son:

- Abre y cierra la conexión a la base.
- Contiene las `DbSet<T>`, que son las colecciones consultables (una por cada tabla).
- Rastrea los cambios que hacés sobre las entidades cargadas.
- Traduce operaciones LINQ a SQL.
- Gestiona transacciones.
- Ejecuta las operaciones pendientes cuando llamás `SaveChanges()`.

Regla importantísima: **el `DbContext` no es thread-safe y es barato de crear**. Creás uno, lo usás para una unidad de trabajo (una operación del menú), lo desechás. No lo compartas entre hilos. No lo hagas `static`. No lo tengas vivo horas.

Usaremos `using` para que se desechen solos.

## Parte 2 — Preparación del proyecto

### 2.1 — Requisitos

- .NET 10 SDK instalado (`dotnet --version` debería mostrar 10.x).
- Un editor: VS Code, Visual Studio 2026 o Rider.

### 2.2 — Crear el proyecto

```bash
dotnet new console -n AgendaEF -f net10.0
cd AgendaEF
```

### 2.3 — Instalar los paquetes NuGet

Dos paquetes, cada uno con su razón de ser:

```bash
dotnet add package Microsoft.EntityFrameworkCore --version 10.*
dotnet add package Microsoft.EntityFrameworkCore.Sqlite --version 10.*
```

- **`Microsoft.EntityFrameworkCore`**: el core del ORM. Las clases `DbContext`, `DbSet`, el change tracker, el query pipeline.
- **`Microsoft.EntityFrameworkCore.Sqlite`**: el *provider* de SQLite. Sin esto EF Core sabe hablar LINQ pero no sabe a qué base traducir.

### 2.4 — Alternativa: configuración file-based (.NET 10)

> **Nota — programas basados en archivo.** .NET 10 introduce los *file-based programs*: podés escribir y ejecutar un programa C# **sin crear un proyecto ni un `.csproj`**, declarando las dependencias directamente dentro del archivo `.cs` con directivas `#:`.

En lugar del flujo `dotnet new` → `dotnet add package`, el archivo declara sus propias dependencias en el encabezado:

```csharp
#:package Microsoft.EntityFrameworkCore@10.*
#:package Microsoft.EntityFrameworkCore.Sqlite@10.*

// todo el código del programa a continuación...
using Microsoft.EntityFrameworkCore;

var db = new AgendaRapidaContext();
db.Database.EnsureCreated();

// ... resto del programa
```

Y se ejecuta directamente:

```bash
dotnet run agenda.cs
```

El SDK descarga los paquetes, compila y corre — sin carpeta de proyecto, sin `bin/`, sin `obj/`.

**¿Cuándo usar file-based y cuándo el proyecto tradicional?**

| Situación | Enfoque recomendado |
| --------- | ------------------- |
| Script de prueba rápida, exploración de una API, ejemplo para clase | File-based — menos fricción |
| App con múltiples archivos, capas (Models/, Data/, Services/) | Proyecto tradicional con `.csproj` |
| Necesitás migraciones (`dotnet ef migrations add`) | Proyecto tradicional — las tools de EF requieren `.csproj` |
| Querés distribuir o versionar el proyecto | Proyecto tradicional |

Para **este tutorial** usamos el proyecto tradicional porque la arquitectura en capas (Models, Data, Services) y las migraciones de la Parte 12 requieren un `.csproj`. Pero si querés hacer una prueba rápida de una query o experimentar con el `DbContext` antes de estructurar el proyecto, el enfoque file-based es ideal.

## Parte 3 — Arquitectura del proyecto

```
AgendaEF/
├── AgendaEF.csproj
├── Program.cs                ← menú interactivo (UI de consola)
├── Models/
│   ├── Categoria.cs
│   ├── Contacto.cs
│   ├── Direccion.cs
│   └── TipoDireccion.cs      ← enum
├── Data/
│   └── AgendaContext.cs      ← el DbContext
└── Services/
    └── ContactoService.cs    ← la lógica CRUD
```

### Por qué separamos en capas

Podríamos poner todo en `Program.cs`. Sería más corto. Pero quiero que practiquemos separación de responsabilidades desde el día uno:

| Capa         | Responsabilidad                         | Qué NO debe hacer                |
| ------------ | --------------------------------------- | -------------------------------- |
| **Models**   | Representar los datos del dominio       | Saber de SQL, consola, o EF Core |
| **Data**     | Configurar EF Core y el mapeo a la base | Tener lógica de negocio          |
| **Services** | Operaciones CRUD y reglas de negocio    | Leer de `Console` ni imprimir    |
| **Program**  | Menú y entrada/salida por consola       | Hablar directo con la base       |

Esta separación no es un capricho académico. Es lo que te permite, mañana, reutilizar `ContactoService` desde una app MAUI o una Web API sin cambiar una línea. El CRUD no "sabe" que hoy lo llama una consola.

### Decisiones de diseño explícitas

| Decisión                                  | Por qué                                                                 |
| ----------------------------------------- | ----------------------------------------------------------------------- |
| SQLite con archivo local `agenda.db`      | Cero infra para el ejercicio, portable                                  |
| `EnsureCreated()` en lugar de migraciones | Nos enfocamos en el ORM, no en la evolución del schema                  |
| Tabla única `Direcciones` con enum `Tipo` | Evita duplicar estructura "teléfono/email" en dos tablas casi idénticas |
| `DbContext` nuevo por operación           | Respeta el ciclo de vida correcto, evita memory leaks                   |
| `Id` autoincremental en todas las tablas  | Convención de EF Core, cero configuración                               |
| Operaciones síncronas                     | Simplicidad pedagógica; async se verá en la Parte 9                     |
| Service sin `Console.WriteLine`           | La lógica no depende de la UI; es testeable                             |

## Parte 4 — Los modelos (el corazón del diseño)

Acá está la parte más importante del tutorial. **Cómo modelás las clases determina cómo se ven las tablas y cómo fluye todo el resto del código.**

### 4.1 — El enum `TipoDireccion`

Archivo: `Models/TipoDireccion.cs`

```csharp
namespace AgendaEF.Models;

public enum TipoDireccion {
    Telefono = 1,
    Email    = 2,
}
```

**Qué está pasando:**

- Un enum es la forma más segura de representar un conjunto cerrado de valores. El compilador te obliga a usar solo los valores permitidos; no podés tipear `"telefono"` con minúscula y romper todo.
- Los valores numéricos (`= 1`, `= 2`) los pongo explícitos: así, si mañana alguien agrega un valor en el medio de la lista, los que ya existen no cambian su número. Esto importa porque **EF Core guarda el enum en la base como número por defecto**. Si los valores cambian, la base queda inconsistente.
- No uso 0. Es buena costumbre: el 0 es el valor por defecto de cualquier enum, y que represente un estado "sin asignar" ayuda a detectar bugs.

### 4.2 — La entidad `Categoria`

Archivo: `Models/Categoria.cs`

```csharp
using System.ComponentModel.DataAnnotations;

namespace AgendaEF.Models;

public class Categoria {
    public int Id { get; set; }

    [Required]
    [MaxLength(50)]
    public string Nombre { get; set; } = string.Empty;

    // Navegación inversa: una categoría puede tener muchos contactos
    public List<Contacto> Contactos { get; set; } = [];
}
```

**Puntos clave:**

- **`Id`** sin atributos. EF Core usa convención: una propiedad llamada `Id` se asume clave primaria. En SQLite, un `int` PK se auto-incrementa.
- **`[Required]`** en `Nombre` → se traduce a `NOT NULL`.
- **`[MaxLength(50)]`** documenta la intención y se respeta en providers estrictos.
- **`List<Contacto> Contactos`** es una **propiedad de navegación**. No es una columna en la tabla — es la forma que tiene el modelo de objetos de expresar "una categoría conoce a todos los contactos que la usan". EF Core la usa para hacer joins cuando pedís categoría + contactos juntos.
- **`= []`** inicializa la lista vacía. Evita `NullReferenceException` cuando accedés a una categoría recién creada antes de `SaveChanges()`.

### 4.3 — La entidad `Contacto`

Archivo: `Models/Contacto.cs`

```csharp
using System.ComponentModel.DataAnnotations;

namespace AgendaEF.Models;

public class Contacto {
    public int Id { get; set; }

    [Required]
    [MaxLength(100)]
    public string Nombre       { get; set; } = string.Empty;

    public DateTime CreadoEn   { get; set; } = DateTime.UtcNow;

    // Relación N:1 con Categoria
    public int CategoriaId     { get; set; }
    public Categoria Categoria { get; set; } = null!;

    // Relación 1:N con Direccion
    public List<Direccion> Direcciones { get; set; } = [];
}
```

Este es el centro de gravedad del modelo. Miralo despacio.

**La relación con Categoria:**

Tenemos dos propiedades que representan lo mismo desde ángulos distintos:

- `CategoriaId` es la **foreign key** — el número que en la tabla `Contactos` apunta a una fila de `Categorias`. Es un dato crudo.
- `Categoria` es la **propiedad de navegación** — el objeto `Categoria` cargado en memoria. Lo usás cuando querés acceder a datos de la categoría desde el contacto (ej: `contacto.Categoria.Nombre`).

¿Por qué tener las dos? Porque resuelven problemas distintos:

- Si solo querés *asignar* una categoría (ej: "este contacto ahora es de Trabajo, cuyo id es 2"), con `CategoriaId = 2` alcanza. No necesitás traer el objeto `Categoria` de la base.
- Si querés *leer* datos de la categoría, usás la propiedad de navegación.

El patrón "FK + navegación" es una convención fuerte en EF Core. Aprendelo bien. El nombre `CategoriaId` es clave: EF Core por convención busca una propiedad con ese nombre para mapearla como la FK de la navegación `Categoria`.

**El `= null!` en `Categoria`:**

Es una promesa al compilador: "sé que esta propiedad parece que podría ser null, pero en la práctica EF Core la va a poblar cuando cargue el contacto, así que ignorá el warning". Es un truco común con propiedades de navegación no-opcionales.

**La relación con Direcciones:**

Del lado del `Contacto`, la relación 1:N se expresa como una `List<Direccion>`. Del lado de la `Direccion` vive la foreign key `ContactoId`. EF Core ve este par y entiende "ajá, uno a muchos".

### 4.4 — La entidad `Direccion`

Archivo: `Models/Direccion.cs`

```csharp
using System.ComponentModel.DataAnnotations;

namespace AgendaEF.Models;

public class Direccion {
    public int Id             { get; set; }

    [Required]
    public TipoDireccion Tipo { get; set; }

    [Required]
    [MaxLength(150)]
    public string Valor       { get; set; } = string.Empty;

    // Relación N:1 con Contacto
    public int ContactoId     { get; set; }
    public Contacto Contacto  { get; set; } = null!;
}
```

**Qué observar:**

- **`Tipo`** es nuestro enum. EF Core lo guardará como `INTEGER` (los valores `1` o `2`). Simple y eficiente.
- **`Valor`** es el string que contiene el número de teléfono o el email. Un solo campo para ambos casos. El tipo del contenido lo indica `Tipo`.
- Otra vez el par **`ContactoId` + `Contacto`**, pero esta vez del lado "muchos" de la relación. La FK vive acá porque en una base relacional, **la FK siempre vive en el lado "muchos"** de una relación 1:N.

### 4.5 — Diagrama mental

Pongamos todo junto:

```
 ┌──────────────┐
 │ Categoria    │       ┌───────────────┐       ┌──────────────┐
 │──────────────│       │ Contacto      │       │ Direccion    │
 │ Id (PK)      │──────<│ Id (PK)       │──────<│ Id (PK)      │
 │ Nombre       │       │ Nombre        │       │ Tipo         │
 └──────────────┘       │ CreadoEn      │       │ Valor        │
     1 categoría        │ CategoriaId FK│       │ ContactoId FK│
    muchos contactos    └───────────────┘       └──────────────┘
                         1 contacto
                      muchas direcciones
```

Las flechas `──────<` indican "uno a muchos": el lado del `<` es el "muchos". Los tres `Id` son claves primarias autoincrementales. Las dos FK viven del lado "muchos" de cada relación.

## Parte 5 — El DbContext

Archivo: `Data/AgendaContext.cs`

```csharp
using AgendaEF.Models;
using Microsoft.EntityFrameworkCore;

namespace AgendaEF.Data;

public class AgendaContext : DbContext {
    public DbSet<Contacto>  Contactos   => Set<Contacto>();
    public DbSet<Categoria> Categorias  => Set<Categoria>();
    public DbSet<Direccion> Direcciones => Set<Direccion>();

    protected override void OnConfiguring(DbContextOptionsBuilder options) {
        string path = Path.Combine(AppContext.BaseDirectory, "agenda.db");
        options.UseSqlite($"Data Source={path}");
    }

    protected override void OnModelCreating(ModelBuilder modelBuilder) {
        // Índice para búsquedas rápidas por nombre de contacto
        modelBuilder.Entity<Contacto>()
            .HasIndex(c => c.Nombre);

        // Cada categoría tiene un nombre único
        modelBuilder.Entity<Categoria>()
            .HasIndex(c => c.Nombre)
            .IsUnique();
    }

    /// <summary>
    /// Crea la base si no existe, con las tablas según el modelo.
    /// Llamar una vez al arrancar la app.
    /// </summary>
    public static void InicializarBase() {
        using var db = new AgendaContext();
        db.Database.EnsureCreated();

        // Semilla: si no hay categorías, crear las típicas
        if (!db.Categorias.Any()) {
            db.Categorias.AddRange(
                new Categoria { Nombre = "Familia" },
                new Categoria { Nombre = "Trabajo" },
                new Categoria { Nombre = "Amigos"  },
                new Categoria { Nombre = "Otros"   }
            );
            db.SaveChanges();
        }
    }
}
```

**Qué está pasando:**

- **Tres `DbSet`**, uno por tabla. EF Core necesita verlos para saber qué clases son entidades.
- **`OnConfiguring`** configura la conexión. Para SQLite, la connection string es simplemente `Data Source=ruta/al/archivo`.
- **`OnModelCreating`** es donde se configura el modelo con la **Fluent API**. Pusimos dos índices: uno común sobre `Contacto.Nombre` (vamos a buscar por nombre seguido), y uno único sobre `Categoria.Nombre` (no queremos dos categorías llamadas "Trabajo").
- **`EnsureCreated()`** mira el modelo actual y, si la base no existe, la crea con todas las tablas, índices y foreign keys. Si ya existe, no hace nada.
- **La semilla inicial** se ejecuta una sola vez gracias al check `if (!db.Categorias.Any())`.

> **¿Por qué `EnsureCreated()` y no migraciones?**
>
> `EnsureCreated()` es perfecto para el caso que nos ocupa: queremos que la base aparezca automáticamente para poder enfocarnos en el ORM. Pero tiene un límite grave: **no soporta cambios al modelo después de la primera ejecución**. Si mañana agregás una propiedad a `Contacto`, `EnsureCreated()` ve que la base existe y se queda tranquilo — aunque le falte la columna nueva. El programa después va a explotar.
>
> **Regla práctica para este ejercicio**: si cambiás el modelo, borrá el archivo `agenda.db` y dejá que el programa lo cree de nuevo.
>
> El manejo correcto de cambios al schema — con **migraciones** — lo cubrimos en la Parte 12.

### Fluent API vs Data Annotations

Hay dos formas de configurar el modelo:

- **Data Annotations**: atributos en la entidad (`[Required]`, `[MaxLength]`). Simples, viven junto a la propiedad. Mezclan responsabilidades: la clase de dominio sabe cosas de la base de datos.
- **Fluent API**: código en `OnModelCreating` usando `modelBuilder`. Más expresivo, puede hacer cosas que los atributos no pueden (índices únicos, relaciones complejas, configuración específica del provider). Mantiene las clases de dominio limpias.

**Regla**: lo básico va en atributos, lo avanzado en Fluent API. Coexisten sin problema; si hay conflicto, gana Fluent API.

Para proyectos más grandes hay una forma más escalable: delegar la configuración de cada entidad a su propia clase. Lo vemos en la **Parte 13**.

## Parte 6 — El servicio CRUD

Archivo: `Services/ContactoService.cs`

Acá viven las operaciones de negocio. Una por cada operación del menú.

```csharp
using AgendaEF.Data;
using AgendaEF.Models;
using Microsoft.EntityFrameworkCore;

namespace AgendaEF.Services;

public class ContactoService {
    // ─── Categorías ──────────────────────────────────────

    public List<Categoria> ListarCategorias() {
        using var db = new AgendaContext();

        return db.Categorias
            .AsNoTracking()
            .OrderBy(c => c.Nombre)
            .ToList();
    }

    // ─── Contactos: Create ───────────────────────────────

    public Contacto Agregar(string nombre, int categoriaId, List<Direccion> direcciones) {
        using var db = new AgendaContext();

        var contacto = new Contacto {
            Nombre      = nombre.Trim(),
            CategoriaId = categoriaId,
            Direcciones = direcciones,
        };

        db.Contactos.Add(contacto);
        db.SaveChanges();

        return contacto;
    }

    // ─── Contactos: Read ─────────────────────────────────

    public List<Contacto> ListarTodos() {
        using var db = new AgendaContext();

        return db.Contactos
            .AsNoTracking()
            .Include(c => c.Categoria)
            .Include(c => c.Direcciones)
            .OrderBy(c => c.Nombre)
            .ToList();
    }

    public Contacto? BuscarPorId(int id) {
        using var db = new AgendaContext();

        return db.Contactos
            .AsNoTracking()
            .Include(c => c.Categoria)
            .Include(c => c.Direcciones)
            .FirstOrDefault(c => c.Id == id);
    }

    public List<Contacto> BuscarPorNombre(string termino) {
        using var db = new AgendaContext();

        string t = termino.Trim();
        return db.Contactos
            .AsNoTracking()
            .Include(c => c.Categoria)
            .Include(c => c.Direcciones)
            .Where(c => EF.Functions.Like(c.Nombre, $"%{t}%"))
            .OrderBy(c => c.Nombre)
            .ToList();
    }

    public bool Actualizar(int id, string nombre, int categoriaId) {
        using var db = new AgendaContext();

        var contacto = db.Contactos.Find(id);
        if (contacto is null) {
            return false;
        }

        contacto.Nombre      = nombre.Trim();
        contacto.CategoriaId = categoriaId;

        db.SaveChanges();
        return true;
    }

    // ─── Contactos: Delete ───────────────────────────────

    public bool Eliminar(int id) {
        using var db = new AgendaContext();

        var contacto = db.Contactos
            .Include(c => c.Direcciones)
            .FirstOrDefault(c => c.Id == id);

        if (contacto is null) {
            return false;
        }

        db.Contactos.Remove(contacto);
        db.SaveChanges();
        return true;
    }
```

Hay varias decisiones clave acá. Vamos por partes.

### 6.1 — El patrón `using var db = new AgendaContext()`

Cada método abre un `DbContext` nuevo y lo desecha al terminar. Esto es el patrón *short-lived context*. Te garantiza tres cosas:

1. La conexión a la base se cierra limpiamente.
2. El change tracker se libera (no acumula entidades en memoria entre operaciones).
3. Dos operaciones no se contaminan entre sí.

En apps grandes con inyección de dependencias, esto se configura automáticamente. En una consola simple, lo hacemos a mano.

### 6.2 — `AsNoTracking()` en las lecturas

Cuando cargás una entidad con EF Core, por defecto el context la **trackea** — registra su estado para detectar cambios después. Si solo querés leer (no modificar), esto es desperdicio: consume memoria y CPU para nada.

`.AsNoTracking()` le dice a EF Core "esto es solo lectura, no te molestes en trackear". Resultado: consultas más rápidas, menos memoria. **Usalo siempre en lecturas puras**.

En los métodos `Actualizar` y `Eliminar` *no* usamos `AsNoTracking`, porque justamente necesitamos que EF Core trackee los cambios para generar el `UPDATE` / `DELETE` correcto.

### 6.3 — `.Include()`: la clave para cargar relaciones

Esta es la nueva estrella del tutorial:

```csharp
db.Contactos
  .Include(c => c.Categoria)
  .Include(c => c.Direcciones)
```

**Sin `Include`**, si cargás un contacto y accedés a `contacto.Categoria`, te va a venir `null`. EF Core por defecto **no trae las relaciones** — las cargás explícitamente. Esto es *eager loading*: decís por adelantado qué relaciones querés.

El SQL que genera EF Core con estos `Include` es algo como:

```sql
SELECT c.Id, c.Nombre, c.CategoriaId,
       cat.Id, cat.Nombre,
       d.Id, d.Tipo, d.Valor, d.ContactoId
FROM Contactos c
LEFT JOIN Categorias cat ON cat.Id = c.CategoriaId
LEFT JOIN Direcciones d ON d.ContactoId = c.Id
ORDER BY c.Nombre
```

EF Core después arma los objetos a partir de estas filas: agrupa las direcciones por contacto y las mete en la lista correspondiente.

**¿Por qué `Include` y no cargar automáticamente todo?** Porque traer relaciones es caro. EF Core te obliga a pedir explícitamente lo que querés: es un default conservador y correcto.

**El problema N+1**: si tuvieras *lazy loading* activado (que no activamos), cada acceso a `c.Categoria` dispararía una query nueva. 1000 contactos = 1001 queries. `Include` lo evita: una sola query. Es de los problemas de performance más comunes con ORMs.

### 6.4 — `Find` vs `FirstOrDefault`

- **`Find(id)`**: busca por clave primaria. Primero revisa el change tracker (si la entidad ya está cargada en memoria, la devuelve sin ir a la base). Solo funciona con la PK. **No soporta `Include`**.
- **`FirstOrDefault(c => ...)`**: busca con cualquier condición LINQ. Siempre va a la base. Acepta `Include`.

Por eso en `Eliminar` usamos `FirstOrDefault` con `Include`: necesitamos que EF Core cargue las direcciones junto al contacto para poder borrarlas en cascada. En `Actualizar`, como solo vamos a tocar propiedades del contacto, `Find` alcanza y es más rápido.

### 6.5 — Borrado en cascada

Cuando borramos un contacto, sus direcciones quedarían huérfanas. EF Core detecta automáticamente la relación 1:N y configura **cascade delete** por defecto — cuando borrás el contacto, sus direcciones se borran también.

Por eso cargamos el contacto con `Include(c => c.Direcciones)` antes de eliminarlo: para que EF Core las trackee y sepa que tiene que borrarlas. La categoría, en cambio, *no* se borra — está compartida por muchos contactos.

### 6.6 — Agregar direcciones en dos momentos distintos

**Al crear el contacto** (`Agregar`), pasamos la lista ya construida:

```csharp
var contacto = new Contacto {
    Nombre      = nombre,
    CategoriaId = categoriaId,
    Direcciones = direcciones,  // ya tiene la lista adentro
};
db.Contactos.Add(contacto);
db.SaveChanges();
```

EF Core hace un `INSERT` al contacto, y después `INSERT`s a cada dirección usando el `Id` recién generado. Todo en una sola transacción.

**Al agregar una dirección a un contacto existente** (`AgregarDireccion`), pasamos el `ContactoId`:

```csharp
var direccion = new Direccion {
    ContactoId = contactoId,  // apuntamos al contacto existente
    Tipo       = tipo,
    Valor      = valor,
};
db.Direcciones.Add(direccion);
db.SaveChanges();
```

No necesitamos cargar el contacto entero. Con saber su id, creamos una nueva fila en `Direcciones` apuntando a él. Esta es la magia de separar FK y navegación.

## Parte 9 — Consultas LINQ en profundidad

El servicio del ejercicio usa las operaciones más comunes. Acá expandimos el repertorio que vas a necesitar en proyectos reales.

### 9.1 — Ejecución diferida vs inmediata

EF Core **no ejecuta la consulta hasta que se itera**. Hasta entonces es un objeto `IQueryable<T>` que describe *qué* querés, no *cuándo* buscarlo.

```csharp
// IQueryable — consulta construida, NO ejecutada todavía
IQueryable<Contacto> query = db.Contactos.Where(c => c.CategoriaId == 2);

// Podés seguir refinando sin ir a la base
query = query.OrderBy(c => c.Nombre);
query = query.Take(10);

// Acá se genera y ejecuta el SQL
List<Contacto> resultado = query.ToList();
// SELECT TOP 10 * FROM Contactos WHERE CategoriaId = 2 ORDER BY Nombre
```

Métodos que ejecutan inmediatamente: `ToList()`, `ToArray()`, `FirstOrDefault()`, `Count()`, `Any()`, `Sum()`, `foreach`.

Este comportamiento permite construir queries condicionalmente sin duplicar código:

```csharp
IQueryable<Contacto> q = db.Contactos.AsNoTracking();

if (!string.IsNullOrWhiteSpace(filtroNombre))
    q = q.Where(c => c.Nombre.Contains(filtroNombre));

if (filtroCategoria.HasValue)
    q = q.Where(c => c.CategoriaId == filtroCategoria);

return q.OrderBy(c => c.Nombre).ToList();
```

### 9.2 — Proyecciones con `Select`

En lugar de cargar la entidad completa, proyectás solo las columnas que necesitás. Más eficiente, menos datos transferidos.

```csharp
// Sin proyección — trae todas las columnas (incluyendo las que no usás)
List<Contacto> todos = db.Contactos.ToList();

// Con proyección a tipo anónimo — solo las columnas necesarias
var resumen = db.Contactos
    .Select(c => new { c.Id, c.Nombre, c.CategoriaId })
    .ToList();

// Proyección a un DTO (record) — más tipado, más limpio
record ContactoResumen(int Id, string Nombre, string Categoria);

List<ContactoResumen> dtos = db.Contactos
    .AsNoTracking()
    .Include(c => c.Categoria)
    .Select(c => new ContactoResumen(c.Id, c.Nombre, c.Categoria.Nombre))
    .ToList();
```

Las proyecciones siempre van del lado del servidor (SQL), no en memoria. EF Core genera el `SELECT` con solo las columnas proyectadas.

### 9.3 — `ThenInclude`: relaciones anidadas

Cuando una relación incluida tiene a su vez relaciones propias, usás `ThenInclude`:

```csharp
// En un sistema hipotético donde Categoria tiene una lista de Tags
var contactos = db.Contactos
    .Include(c => c.Categoria)
        .ThenInclude(cat => cat.Tags)   // relaciones de la relación
    .Include(c => c.Direcciones)
    .ToList();
```

En nuestro modelo de agenda no hace falta porque las relaciones no tienen más de dos niveles, pero en modelos más ricos (pedidos → ítems → productos → categorías) `ThenInclude` es esencial.

### 9.4 — Consultas comunes

```csharp
// Paginación — fundamental en cualquier listado
int pagina = 2;
int tamaño = 10;

var paginados = db.Contactos
    .AsNoTracking()
    .OrderBy(c => c.Nombre)
    .Skip((pagina - 1) * tamaño)
    .Take(tamaño)
    .ToList();

// Conteos y agregaciones

int soloFamilia      = db.Contactos.Count(c => c.CategoriaId == 1);
bool hayAlguno       = db.Contactos.Any();
bool existeAda       = db.Contactos.Any(c => c.Nombre == "Ada Lovelace");

// Agrupación: cuántos contactos por categoría
var porCategoria = db.Contactos
    .AsNoTracking()
    .Include(c => c.Categoria)
    .GroupBy(c => c.Categoria.Nombre)
    .Select(g => new {
        Categoria = g.Key,
        Cantidad  = g.Count(),
    })
    .OrderByDescending(x => x.Cantidad)
    .ToList();

// Búsqueda insensible a mayúsculas (SQLite es insensible por defecto con ASCII)
var resultados = db.Contactos
    .Where(c => EF.Functions.Like(c.Nombre, "%lovelace%"))
    .ToList();
```

## Parte 10 — Operaciones de escritura avanzadas

### 10.1 — `ExecuteUpdate`: actualización masiva sin cargar entidades

El patrón estándar (cargar entidad → modificar → `SaveChanges`) tiene un costo: trae filas de la base al servidor para modificarlas y enviarlas de vuelta. Para operaciones sobre muchas filas, hay una forma más eficiente:

```csharp
// Forma estándar — carga cada contacto en memoria
var contactos = db.Contactos.Where(c => c.CategoriaId == 4).ToList();
foreach (var c in contactos)
    c.CategoriaId = 3;
db.SaveChanges();

// ExecuteUpdate — genera un solo UPDATE en la base, sin cargar nada
await db.Contactos
    .Where(c => c.CategoriaId == 4)
    .ExecuteUpdateAsync(setter =>
        setter.SetProperty(c => c.CategoriaId, 3));
// SQL: UPDATE Contactos SET CategoriaId = 3 WHERE CategoriaId = 4
```

Podés encadenar varios `SetProperty` para actualizar múltiples columnas a la vez.

### 10.2 — `ExecuteDelete`: borrado masivo

```csharp
// Forma estándar — carga, marca, guarda
var viejas = db.Direcciones.Where(d => d.Tipo == TipoDireccion.Email).ToList();
db.Direcciones.RemoveRange(viejas);
db.SaveChanges();

// ExecuteDelete — un solo DELETE, sin cargar nada
await db.Direcciones
    .Where(d => d.Tipo == TipoDireccion.Email)
    .ExecuteDeleteAsync();
// SQL: DELETE FROM Direcciones WHERE Tipo = 2
```

**Importante**: `ExecuteUpdate` y `ExecuteDelete` **no usan el change tracker** — bypasean la unidad de trabajo. Se ejecutan inmediatamente, como SQL crudo. No esperan `SaveChanges()`. Úsalos cuando la operación es masiva y no necesitás los objetos en memoria.

### 10.3 — Transacciones explícitas

Por defecto, cada `SaveChanges()` es una transacción automática. Para operaciones que involucran múltiples `SaveChanges()` y necesitás que todo se confirme o revierta junto:

```csharp
using var db = new AgendaContext();
using var tx = await db.Database.BeginTransactionAsync();

try {
    // Primera operación
    var categoria = new Categoria { Nombre = "VIP" };
    db.Categorias.Add(categoria);
    await db.SaveChangesAsync();

    // Segunda operación que depende de la primera
    var contacto = new Contacto {
        Nombre      = "Ada Lovelace",
        CategoriaId = categoria.Id,
    };
    db.Contactos.Add(contacto);
    await db.SaveChangesAsync();

    await tx.CommitAsync();  // todo o nada
} catch {
    await tx.RollbackAsync();
    throw;
}
```

## Parte 11 — Relaciones avanzadas

El modelo de la agenda solo usa 1:N y N:1. Los proyectos reales también necesitan N:N y 1:1.

### 11.1 — N:N simple (tabla de unión automática)

Un libro puede tener muchos géneros; un género puede aplicar a muchos libros.

```csharp
public class Libro {
    public int    Id     { get; set; }
    public string Titulo { get; set; } = string.Empty;

    public List<Genero> Generos { get; set; } = [];
}

public class Genero {
    public int    Id     { get; set; }
    public string Nombre { get; set; } = string.Empty;

    public List<Libro> Libros { get; set; } = [];
}
```

Configuración en `OnModelCreating` (EF Core crea la tabla de unión automáticamente):

```csharp
modelBuilder.Entity<Libro>()
    .HasMany(l => l.Generos)
    .WithMany(g => g.Libros)
    .UsingEntity(j => j.ToTable("libro_generos"));
```

Uso:

```csharp
using var db = new LibreriaContext();

var novela    = new Genero { Nombre = "Novela" };
var historica = new Genero { Nombre = "Histórica" };
db.Generos.AddRange(novela, historica);

var libro = new Libro {
    Titulo  = "La guerra y la paz",
    Generos = [novela, historica],
};
db.Libros.Add(libro);
db.SaveChanges();

// Leer con géneros incluidos
var libros = db.Libros
    .Include(l => l.Generos)
    .ToList();
```

### 11.2 — N:N con tabla de unión explícita

Cuando la relación tiene **atributos propios** (por ejemplo, un autor puede ser "principal" o "coautor" de un libro), definís la tabla de unión como entidad:

```csharp
public class LibroAutor {
    public int    LibroId { get; set; }
    public int    AutorId { get; set; }
    public string Rol     { get; set; } = "principal";
    public int    Orden   { get; set; } = 1;

    public Libro Libro { get; set; } = null!;
    public Autor Autor { get; set; } = null!;
}

public class Libro {
    public int    Id          { get; set; }
    public string Titulo      { get; set; } = string.Empty;
    public List<LibroAutor> LibroAutores { get; set; } = [];
}

public class Autor {
    public int    Id     { get; set; }
    public string Nombre { get; set; } = string.Empty;
    public List<LibroAutor> LibroAutores { get; set; } = [];
}
```

Configuración:

```csharp
modelBuilder.Entity<LibroAutor>(entity => {
    entity.HasKey(la => new { la.LibroId, la.AutorId });  // PK compuesta

    entity.HasOne(la => la.Libro)
          .WithMany(l => l.LibroAutores)
          .HasForeignKey(la => la.LibroId);

    entity.HasOne(la => la.Autor)
          .WithMany(a => a.LibroAutores)
          .HasForeignKey(la => la.AutorId);
});
```

Consulta con `ThenInclude` para navegar por la tabla de unión:

```csharp
var libros = db.Libros
    .Include(l => l.LibroAutores)
        .ThenInclude(la => la.Autor)
    .ToList();

foreach (var libro in libros) {
    var autores = libro.LibroAutores
        .OrderBy(la => la.Orden)
        .Select(la => $"{la.Autor.Nombre} ({la.Rol})");
    Console.WriteLine($"{libro.Titulo} — {string.Join(", ", autores)}");
}
```

### 11.3 — Relación 1:1

Un libro tiene un único detalle editorial; ese detalle pertenece a un único libro.

```csharp
public class DetalleEditorial {
    public int    Id        { get; set; }
    public string ISBN      { get; set; } = string.Empty;
    public string Editorial { get; set; } = string.Empty;
    public int    Paginas   { get; set; }

    public int    LibroId   { get; set; }
    public Libro  Libro     { get; set; } = null!;
}

public class Libro {
    public int    Id      { get; set; }
    public string Titulo  { get; set; } = string.Empty;

    // Navegación hacia el detalle (puede ser null si no se cargó)
    public DetalleEditorial? Detalle { get; set; }
}
```

Configuración:

```csharp
modelBuilder.Entity<Libro>()
    .HasOne(l => l.Detalle)
    .WithOne(d => d.Libro)
    .HasForeignKey<DetalleEditorial>(d => d.LibroId);
```

La FK vive del lado del "dependiente" (`DetalleEditorial`), igual que en 1:N.

## Parte 12 — Migraciones

`EnsureCreated()` crea la base una sola vez y no soporta cambios. En proyectos reales necesitás **migraciones**: clases generadas automáticamente que describen la diferencia entre el modelo C# actual y el schema de la base, y que se pueden aplicar de forma incremental y reversible.

### 12.1 — Paquetes adicionales

Para usar las herramientas de migraciones:

```bash
dotnet add package Microsoft.EntityFrameworkCore.Tools  --version 10.*
dotnet add package Microsoft.EntityFrameworkCore.Design --version 10.*
```

### 12.2 — Flujo de trabajo

```bash
# 1. Definir o modificar el modelo C#

# 2. Crear la migración
dotnet ef migrations add CrearAgenda

# 3. Revisar el archivo generado (siempre — antes de aplicar)
# Migrations/20251201120000_CrearAgenda.cs

# 4. Aplicar a la base
dotnet ef database update
```

Cada migración tiene dos métodos: `Up()` (aplica el cambio) y `Down()` (lo revierte). EF Core los genera automáticamente comparando el modelo actual con el snapshot anterior.

Comandos útiles:

```bash
dotnet ef migrations list                     # ver todas las migraciones
dotnet ef database update NombreMigracion     # ir a una migración específica
dotnet ef migrations remove                   # eliminar la última (si no se aplicó)
dotnet ef database update 0                   # revertir todo
```

### 12.3 — Qué contiene una migración

```csharp
// Migrations/20251201120000_CrearAgenda.cs — generado por EF Core
public partial class CrearAgenda : Migration {
    protected override void Up(MigrationBuilder migrationBuilder) {
        migrationBuilder.CreateTable(
            name: "Categorias",
            columns: table => new {
                Id     = table.Column<int>(type: "INTEGER", nullable: false)
                              .Annotation("Sqlite:Autoincrement", true),
                Nombre = table.Column<string>(maxLength: 50, nullable: false),
            },
            constraints: table => {
                table.PrimaryKey("PK_Categorias", x => x.Id);
            });

        // ... más tablas ...
    }

    protected override void Down(MigrationBuilder migrationBuilder) {
        migrationBuilder.DropTable(name: "Contactos");
        migrationBuilder.DropTable(name: "Categorias");
        // ...
    }
}
```

**Señales de advertencia que deben hacer pausar antes de aplicar:**

- `DropTable` o `DropColumn` — se pierden datos permanentemente.
- `AddColumn NOT NULL` sin `defaultValue` sobre una tabla con datos existentes — la base puede rechazar la operación.
- `RenameColumn` donde esperabas `AddColumn` — EF Core a veces interpreta mal los renames.

### 12.4 — Datos semilla via `HasData`

Para datos que siempre deben existir (catálogos, configuraciones iniciales), en lugar de insertar a mano podés declarar semilla en la configuración del modelo:

```csharp
// En la configuración de Categoria:
entity.HasData(
    new Categoria { Id = 1, Nombre = "Familia"  },
    new Categoria { Id = 2, Nombre = "Trabajo"  },
    new Categoria { Id = 3, Nombre = "Amigos"   },
    new Categoria { Id = 4, Nombre = "Otros"    }
);
```

Los IDs en `HasData` son **obligatorios y fijos**. EF Core genera la migración con los `INSERT` correspondientes y los aplica solo una vez.

### 12.5 — Aplicar migraciones al arrancar la app

En lugar de `EnsureCreated()`, una app con migraciones hace:

```csharp
using var db = new AgendaContext();
await db.Database.MigrateAsync();  // aplica todas las migraciones pendientes
```

`MigrateAsync()` es seguro de llamar siempre: si no hay migraciones pendientes, no hace nada. Si la base no existe, la crea. Es el reemplazo correcto de `EnsureCreated()` en proyectos que evolucionan.

### 12.6 — Nombrar bien las migraciones

```bash
# Descriptivo — cada nombre dice qué cambió
dotnet ef migrations add CrearTablaContactos
dotnet ef migrations add AgregarCampoEmailContacto
dotnet ef migrations add AgregarIndiceNombreContacto
dotnet ef migrations add SeedCategoriasIniciales

# Genérico — inútil para entender el historial
dotnet ef migrations add Migration1
dotnet ef migrations add Cambios
```

## Parte 13 — Configuración escalable con `IEntityTypeConfiguration`

Para proyectos con muchas entidades, mezclar toda la configuración en `OnModelCreating` lo vuelve inmanejable. La solución es delegar la configuración de cada entidad a su propia clase:

```csharp
// Data/Configurations/ContactoConfiguration.cs
using Microsoft.EntityFrameworkCore;
using Microsoft.EntityFrameworkCore.Metadata.Builders;

namespace AgendaEF.Data.Configurations;

public class ContactoConfiguration : IEntityTypeConfiguration<Contacto> {
    public void Configure(EntityTypeBuilder<Contacto> entity) {
        entity.HasKey(c => c.Id);

        entity.Property(c => c.Nombre)
              .IsRequired()
              .HasMaxLength(100);

        entity.HasIndex(c => c.Nombre);

        entity.HasOne(c => c.Categoria)
              .WithMany(cat => cat.Contactos)
              .HasForeignKey(c => c.CategoriaId)
              .OnDelete(DeleteBehavior.Restrict);

        entity.HasMany(c => c.Direcciones)
              .WithOne(d => d.Contacto)
              .HasForeignKey(d => d.ContactoId)
              .OnDelete(DeleteBehavior.Cascade);
    }
}
```

```csharp
// Data/Configurations/CategoriaConfiguration.cs
public class CategoriaConfiguration : IEntityTypeConfiguration<Categoria> {
    public void Configure(EntityTypeBuilder<Categoria> entity) {
        entity.HasKey(c => c.Id);

        entity.Property(c => c.Nombre)
              .IsRequired()
              .HasMaxLength(50);

        entity.HasIndex(c => c.Nombre)
              .IsUnique();
    }
}
```

El contexto queda limpio y aplica todas las configuraciones automáticamente:

```csharp
public class AgendaContext : DbContext {
    public DbSet<Contacto>  Contactos   => Set<Contacto>();
    public DbSet<Categoria> Categorias  => Set<Categoria>();
    public DbSet<Direccion> Direcciones => Set<Direccion>();

    protected override void OnConfiguring(DbContextOptionsBuilder options) {
        string path = Path.Combine(AppContext.BaseDirectory, "agenda.db");
        options.UseSqlite($"Data Source={path}");
    }

    protected override void OnModelCreating(ModelBuilder modelBuilder) {
        // Aplica automáticamente todas las IEntityTypeConfiguration del assembly
        modelBuilder.ApplyConfigurationsFromAssembly(typeof(AgendaContext).Assembly);
    }
}
```

Esta estructura escala a proyectos con 20, 50, 100 entidades sin que el contexto crezca.

## Parte 14 — Para apps con inyección de dependencias

En una app con DI (ASP.NET Core, MAUI con Microsoft.Extensions.DependencyInjection), el contexto se configura en el contenedor y se inyecta donde se necesita. No se instancia manualmente.

### 14.1 — Configurar el contexto con DI

```csharp
// Program.cs (ASP.NET Core o app con DI)
builder.Services.AddDbContext<AgendaContext>(options =>
    options.UseSqlite(builder.Configuration.GetConnectionString("Agenda")));
```

```json
// appsettings.json
{
  "ConnectionStrings": {
    "Agenda": "Data Source=agenda.db"
  }
}
```

El contexto recibe las opciones por constructor:

```csharp
public class AgendaContext : DbContext {
    public AgendaContext(DbContextOptions<AgendaContext> options) : base(options) { }

    public DbSet<Contacto>  Contactos   => Set<Contacto>();
    public DbSet<Categoria> Categorias  => Set<Categoria>();
    public DbSet<Direccion> Direcciones => Set<Direccion>();

    protected override void OnModelCreating(ModelBuilder modelBuilder) {
        modelBuilder.ApplyConfigurationsFromAssembly(typeof(AgendaContext).Assembly);
    }
}
```

### 14.2 — El patrón Repositorio

Con DI, el servicio recibe el contexto por constructor en lugar de instanciarlo:

```csharp
public class ContactoService {
    private readonly AgendaContext _db;

    public ContactoService(AgendaContext db) {
        this._db = db;
    }

    public async Task<List<Contacto>> ListarTodosAsync() =>
        await this._db.Contactos
            .AsNoTracking()
            .Include(c => c.Categoria)
            .Include(c => c.Direcciones)
            .OrderBy(c => c.Nombre)
            .ToListAsync();

    public async Task<Contacto?> BuscarPorIdAsync(int id) =>
        await this._db.Contactos
            .AsNoTracking()
            .Include(c => c.Categoria)
            .Include(c => c.Direcciones)
            .FirstOrDefaultAsync(c => c.Id == id);

    public async Task<Contacto> AgregarAsync(string nombre, int categoriaId, List<Direccion> direcciones) {
        bool categoriaExiste = await this._db.Categorias.AnyAsync(c => c.Id == categoriaId);
        if (!categoriaExiste)
            throw new ArgumentException($"La categoría {categoriaId} no existe.");

        var contacto = new Contacto {
            Nombre      = nombre.Trim(),
            CategoriaId = categoriaId,
            Direcciones = direcciones,
        };

        this._db.Contactos.Add(contacto);
        await this._db.SaveChangesAsync();

        return contacto;
    }
}
```

Registro en el contenedor:

```csharp
builder.Services.AddScoped<ContactoService>();
```

`AddScoped` significa que el contenedor crea una instancia por request (web) o por scope (consola). El ciclo de vida del `DbContext` lo maneja el contenedor automáticamente.

## Parte 15 — Principios de diseño para llevar a cualquier proyecto

Cerramos con los principios transferibles. Estos aplicán a cualquier app de acceso a datos que construyas en el futuro, independientemente del ORM.

### 15.1 — El DbContext tiene ciclo de vida corto

Un `DbContext` es como una conversación con la base. Abrís la conversación, hacés lo tuyo, cortás. Razones concretas:

- **Memory leaks**: el change tracker acumula entidades a medida que consultás. Un contexto largo crece indefinidamente.
- **Datos stale**: si otro proceso modifica la base, tu contexto largo sigue viendo la versión vieja.
- **Transacciones confusas**: un contexto largo con muchos saves puede dejar la base en estados inconsistentes si algo falla.

Regla operativa: **un `DbContext` por unidad de trabajo** (una operación del menú, una request web, una pantalla de MAUI).

```csharp
// ✓ Correcto — contexto corto
async Task ActualizarNombre(int id, string nombre) {
    using var db = new AgendaContext();
    var c = await db.Contactos.FindAsync(id);
    if (c is null) return;
    c.Nombre = nombre;
    await db.SaveChangesAsync();
}

// ✗ Incorrecto — contexto de vida larga
public class Servicio {
    private readonly AgendaContext _db = new();  // vive toda la vida del servicio
}
```

### 15.2 — Separá lectura y escritura mentalmente

Las operaciones de lectura y escritura son fundamentalmente distintas.

- **Lecturas**: usá `AsNoTracking`. Son idempotentes. No requieren transacción.
- **Escrituras**: sin `AsNoTracking` (necesitás tracking para `SaveChanges`). No son idempotentes. Pueden requerir transacción.

Esta distinción es el germen del patrón **CQRS** (Command Query Responsibility Segregation). Para una agenda no hace falta llegar a CQRS, pero la mentalidad sí.

### 15.3 — FK + propiedad de navegación: el patrón cardinal

Cuando modelás una relación, casi siempre querés ambas:

```csharp
public int CategoriaId     { get; set; }  // la FK cruda — para asignar
public Categoria Categoria { get; set; }  // la navegación — para leer
```

- **Asignar es barato**: `CategoriaId = 2` y listo. No traés el objeto `Categoria` de la base.
- **Leer es cómodo**: `contacto.Categoria.Nombre` en lugar de queries manuales.
- **EF Core los relaciona automáticamente** por convención de nombre.

### 15.4 — `Include` explícito es la forma correcta

EF Core por defecto **no carga las relaciones automáticamente**. Vos decidís qué traer en cada query. Consecuencia: si accedés a `contacto.Categoria` sin `Include`, obtenés `null`. No es un bug del ORM; es que olvidaste pedirlo.

### 15.5 — Proyectar cuando solo necesitás parte de los datos

```csharp
// ✗ Trae todo para usar solo el nombre
var lista = db.Contactos.ToList().Select(c => c.Nombre);

// ✓ El SQL solo trae la columna necesaria
var nombres = db.Contactos.Select(c => c.Nombre).ToList();
```

### 15.6 — Pensá el modelo relacional, no solo el de objetos

Antes de escribir la clase, dibujá el diagrama de tablas. Si el diagrama es raro, el modelo de objetos también lo va a ser.

Pregunta de chequeo: *¿dónde vive la foreign key?* La respuesta es siempre "en el lado muchos" de una relación 1:N, o "en una tabla intermedia" en N:N.

### 15.7 — Revisá siempre las migraciones antes de aplicar

Las migraciones son SQL que va a producción. `DropTable`, `DropColumn`, `AddColumn NOT NULL` sin default sobre datos existentes — son operaciones que destruyen datos si no las revisás con cuidado.

### 15.8 — Mirá el SQL generado

**Nunca uses un ORM sin saber qué SQL está generando.** Detectás el problema N+1. Te volvés mejor en SQL. Cuando el ORM genera algo subóptimo, podés saltarlo con `FromSqlRaw` o `ExecuteSqlRaw`. El ORM es una comodidad, no un reemplazo del conocimiento de bases de datos.

### 15.9 — Validación en capas

Hay tres lugares donde validar que un campo es requerido:

1. **En la UI** (antes de llamar al service): feedback inmediato.
2. **En el service** (regla de negocio): garantía aunque la UI falle.
3. **En la base** (`NOT NULL`): última línea de defensa.

Parece redundante, pero cada capa protege de un escenario distinto. *Defense in depth.*

Lo que acabamos de cubrir va de un extremo al otro de EF Core: desde el proyecto más simple (agenda en consola con tres tablas) hasta las herramientas que usás en proyectos de producción (migraciones, `IEntityTypeConfiguration`, `ExecuteUpdate`/`ExecuteDelete`, transacciones, repositorios con DI).

El hilo conductor es siempre el mismo: **entender el por qué**. Por qué `AsNoTracking`, por qué `Include`, por qué contextos cortos, por qué tanto FK como navegación, por qué revisar el SQL. El cómo lo vas a recordar cinco minutos; el por qué te sirve para los próximos diez años.

Y lo más importante: los principios no son específicos de EF Core. El impedance mismatch, el unit of work, las relaciones 1:N con FK del lado muchos, el problema N+1, la separación lectura/escritura — todo esto vale con Dapper, con SQLAlchemy, con Hibernate, con cualquier ORM que uses en los próximos 20 años.
