# Patrones de diseño en .NET

## 1. El vínculo entre patrones y SRP

Cada vez que detectás una clase que carga con dos responsabilidades, hay que decidir qué pieza extraer, dónde ponerla, y cómo conectarla. Esas decisiones tienen un puñado de **formas recurrentes**, y a esas formas la comunidad les puso nombres hace 30 años: son los patrones de diseño orientado a objetos.

Vale ser honestos sobre la dirección causal. Los patrones no son la causa de un buen diseño. La causa es la presión del SRP — la necesidad de separar lo que tiene motivos distintos para cambiar. Lo que pasa es que, cuando uno extrae responsabilidades repetidamente, va descubriendo que **la mayoría de las extracciones caen en unas pocas formas**. Esas formas son los patrones. Estudiarlos no reemplaza al diagnóstico; lo acelera. Cuando uno mira una clase y dice "acá hay dos responsabilidades mezcladas: la decisión de qué hacer y la ejecución", reconocer que eso es un Strategy clasifica el refactor inmediatamente.

> **Tesis de esta parte**: SRP es el *diagnóstico*; el patrón es la *forma típica* que toma la extracción. Ningún refactor SRP requiere conocer el nombre del patrón; pero conocer los nombres ayuda a comunicar, a anticipar el código resultante, y a no reinventar variantes.

Otro acuerdo necesario: **no toda violación de SRP necesita un patrón con nombre**. A veces basta con extraer una clase y listo. Si el resultado se parece a uno de los patrones, perfecto; si no, también. Evitar el dogma del catálogo es parte de aplicar patrones bien.

Tomemos un ejemplo típico: una agenda de contactos que guarda datos en JSON.

```csharp
// ── Solución mezclada — demasiadas responsabilidades ─────────────────────
public sealed class AgendaService {
    private readonly string path = "contactos.json";

    public void Agregar(Contacto contacto) {
        List<Contacto> contactos;

        if (File.Exists(path)) {
            string json = File.ReadAllText(path);
            contactos = JsonSerializer.Deserialize<List<Contacto>>(json) ?? [];
        }
        else {
            contactos = [];
        }

        if (string.IsNullOrWhiteSpace(contacto.Nombre))
            throw new InvalidOperationException("El nombre es obligatorio.");

        contactos.Add(contacto);

        string nuevoJson = JsonSerializer.Serialize(contactos,
            new JsonSerializerOptions { WriteIndented = true });

        File.WriteAllText(path, nuevoJson);
    }
}
```

```text
AgendaService
    ├─► sabe dónde está el archivo
    ├─► sabe leer archivos
    ├─► sabe parsear JSON
    ├─► sabe validar contactos
    ├─► sabe agregar contactos
    └─► sabe guardar archivos
```

La pregunta de diseño no es "¿funciona?". La pregunta correcta es: **¿por cuántas razones distintas puede cambiar esta clase?**. Demasiadas. Una versión más limpia separa responsabilidades:

```csharp
public sealed class AgendaService {
    private readonly IContactoRepository repository;
    private readonly IContactoValidator  validator;

    public AgendaService(IContactoRepository repository, IContactoValidator validator) {
        this.repository = repository;
        this.validator  = validator;
    }

    public void Agregar(Contacto contacto) {
        validator.Validar(contacto);

        List<Contacto> contactos = repository.ObtenerTodos();
        contactos.Add(contacto);
        repository.GuardarTodos(contactos);
    }
}
```

Ahora el servicio expresa el caso de uso. No sabe si los datos vienen de JSON, SQLite, SQL Server, memoria o una API. Ese detalle vive en otra clase. Ese es el papel de los patrones: **dar formas conocidas de separar responsabilidades sin inventar una arquitectura nueva cada martes**.

## 2. Taxonomía operativa

Los patrones del catálogo clásico (GoF y posteriores) se pueden organizar según qué *tipo* de responsabilidad extraen. Para SRP, esa taxonomía es más útil que la división tradicional creacional/estructural/comportamiento:

| Familia                  | Qué responsabilidad extrae                            | Patrones                                   |
|--------------------------|-------------------------------------------------------|--------------------------------------------|
| Política / algoritmo     | "Cómo se hace cierta operación" cuando hay variantes  | Strategy, Template Method, State           |
| Operación                | "Qué se hace" como cosa de primera clase              | Command, Chain of Responsibility           |
| Creación                 | "Cómo se construye" un objeto cuando es no trivial    | Factory, Builder                           |
| Coordinación             | "Quién le habla a quién"                              | Mediator, Observer, Facade                 |
| Persistencia / consulta  | "Cómo se guarda y se busca"                           | Repository, Unit of Work, Specification    |
| Adaptación / envoltura   | "Cómo se interopera o se decora"                      | Adapter, Decorator, Proxy                  |
| Idiomáticos de .NET      | "Convenciones de framework"                           | DI, Options, Middleware, Filters, MVVM, DTO |

Cada fila representa un tipo de motivo de cambio. Cuando uno detecta que una clase mezcla, por ejemplo, "lógica de negocio" con "cómo se elige el algoritmo de cálculo", el motivo extra encaja en la primera fila — la solución va a tener forma de Strategy, Template Method o State, según el subcaso. La taxonomía clasifica el problema antes de elegir el patrón.

## 3. Política y algoritmo: Strategy, Template Method, State

Estos patrones extraen la pregunta "¿cómo se hace?" cuando existe más de una respuesta válida.

### 3.1 Strategy — extraer la decisión sobre cómo se calcula

**El motivo de cambio que separa.** Una clase decide qué algoritmo usar y, dentro de ella, implementa los algoritmos. Hay dos motivos de cambio: cuándo se elige cada algoritmo, y qué hace cada algoritmo.

**Síntoma típico.** Un `switch` o cadena de `if` que enumera alternativas y dentro de cada rama hay lógica sustantiva.

```csharp
// Una clase, dos responsabilidades:
//   (1) decidir qué política aplicar
//   (2) implementar cada política
public class CalculadoraEnvio {
    public decimal Calcular(string tipo, decimal peso, decimal distancia) {
        switch (tipo) {
            case "estandar":
                return peso * 0.5m + distancia * 0.1m;
            case "express":
                return peso * 1.0m + distancia * 0.3m + 50m;
            case "internacional":
                var aduanas = 100m;
                var seguro  = peso > 10 ? 80m : 30m;
                return peso * 2.0m + distancia * 0.5m + aduanas + seguro;
        }
        throw new ArgumentException(tipo);
    }
}
```

**Refactor.** Cada algoritmo se vuelve una clase con un único método; la calculadora orquesta sin conocer detalles.

```csharp
public interface IPoliticaEnvio {
    decimal Calcular(decimal peso, decimal distancia);
}

public class EnvioEstandar : IPoliticaEnvio {
    public decimal Calcular(decimal peso, decimal distancia)
        => peso * 0.5m + distancia * 0.1m;
}

public class EnvioExpress : IPoliticaEnvio {
    public decimal Calcular(decimal peso, decimal distancia)
        => peso * 1.0m + distancia * 0.3m + 50m;
}

public class EnvioInternacional : IPoliticaEnvio {
    public decimal Calcular(decimal peso, decimal distancia) {
        var aduanas = 100m;
        var seguro  = peso > 10 ? 80m : 30m;
        return peso * 2.0m + distancia * 0.5m + aduanas + seguro;
    }
}

public class CalculadoraEnvio(IPoliticaEnvio politica) {
    public decimal Calcular(decimal peso, decimal distancia) =>
        politica.Calcular(peso, distancia);
}
```

**Por qué satisface SRP.** La calculadora cambia si cambia cómo se elige la política. Cada implementación cambia si cambia su algoritmo. Tres áreas (envíos estándar, express, internacional) pueden pedir cambios sobre sus respectivas clases sin colisionar.

> **Cuándo aplicar Strategy**: cuando hay variantes de un algoritmo y los cambios sobre cada variante vienen de motivos distintos. Si las variantes cambian siempre juntas — por ejemplo, porque dependen de una misma fórmula tributaria — el `switch` original es preferible.

### 3.2 Template Method — extraer los pasos variables del esqueleto

**El motivo de cambio que separa.** Un algoritmo tiene una estructura fija pero algunos pasos varían. La estructura cambia por una razón, los pasos por otra.

**Síntoma típico.** Varios métodos de una misma clase con la misma estructura repetida y diferencias localizadas.

```csharp
public class GeneradorReporteVentas {
    public string Generar() {
        var sb = new StringBuilder();
        sb.AppendLine("=== REPORTE VENTAS ===");
        sb.AppendLine($"Fecha: {DateTime.Today:d}");
        var datos = ConsultaSql("SELECT * FROM ventas WHERE fecha=@hoy");
        foreach (var fila in datos) sb.AppendLine($"{fila.Id} - ${fila.Total}");
        sb.AppendLine($"Total: ${datos.Sum(f => f.Total)}");
        return sb.ToString();
    }
}

public class GeneradorReporteCompras {
    public string Generar() {
        var sb = new StringBuilder();
        sb.AppendLine("=== REPORTE COMPRAS ===");
        sb.AppendLine($"Fecha: {DateTime.Today:d}");
        var datos = ConsultaSql("SELECT * FROM compras WHERE fecha=@hoy");
        foreach (var fila in datos) sb.AppendLine($"{fila.Id} - ${fila.Total}");
        sb.AppendLine($"Total: ${datos.Sum(f => f.Total)}");
        return sb.ToString();
    }
}
```

Tres responsabilidades mezcladas: la estructura del reporte, qué consulta se hace, qué título tiene. La estructura es idéntica; los pasos varían.

**Refactor.**

```csharp
public abstract class GeneradorReporte {
    public string Generar() {
        var sb = new StringBuilder();
        sb.AppendLine($"=== {Titulo()} ===");
        sb.AppendLine($"Fecha: {DateTime.Today:d}");
        var datos = ObtenerDatos();
        foreach (var fila in datos) sb.AppendLine($"{fila.Id} - ${fila.Total}");
        sb.AppendLine($"Total: ${datos.Sum(f => f.Total)}");
        return sb.ToString();
    }

    protected abstract string Titulo();
    protected abstract IEnumerable<Fila> ObtenerDatos();
}

public class ReporteVentas : GeneradorReporte {
    protected override string Titulo() => "REPORTE VENTAS";
    protected override IEnumerable<Fila> ObtenerDatos() =>
        ConsultaSql("SELECT * FROM ventas WHERE fecha=@hoy");
}

public class ReporteCompras : GeneradorReporte {
    protected override string Titulo() => "REPORTE COMPRAS";
    protected override IEnumerable<Fila> ObtenerDatos() =>
        ConsultaSql("SELECT * FROM compras WHERE fecha=@hoy");
}
```

**Por qué satisface SRP.** La clase base es responsable de la **estructura**; cambia si cambia el formato general. Cada subclase es responsable de **un tipo de reporte**; cambia si cambia su consulta o su título. Dos motivos disjuntos, dos lugares disjuntos.

> **Variante moderna**: Template Method puede implementarse con composición y delegados en lugar de herencia. Un `Func<IEnumerable<Fila>>` y un `string titulo` pasados al constructor logran el mismo SRP sin la rigidez de la herencia. El patrón es la idea ("estructura fija + pasos variables"), no su implementación con herencia.

### 3.3 State — extraer el comportamiento que cambia con el estado interno

**El motivo de cambio que separa.** Un objeto se comporta distinto según su estado, y la clase termina llena de `if` por estado. Cada estado tiene su propio motivo de cambio.

**Síntoma típico.** Métodos con `switch` o `if` sobre un campo `estado` o `tipo`, repetido en varios métodos.

```csharp
public class Pedido {
    public string Estado { get; set; } = "pendiente";

    public void Pagar() {
        if      (Estado == "pendiente")  Estado = "pagado";
        else if (Estado == "cancelado")  throw new InvalidOperationException("Ya cancelado");
        else                              throw new InvalidOperationException($"No se puede pagar desde {Estado}");
    }

    public void Despachar() {
        if (Estado == "pagado") Estado = "despachado";
        else throw new InvalidOperationException($"No se puede despachar desde {Estado}");
    }

    public void Cancelar() {
        if (Estado == "pendiente" || Estado == "pagado") Estado = "cancelado";
        else throw new InvalidOperationException($"No se puede cancelar desde {Estado}");
    }
}
```

**Refactor.** Cada estado es una clase, y el `Pedido` delega su comportamiento al estado actual.

```csharp
public abstract class EstadoPedido {
    public abstract EstadoPedido Pagar();
    public abstract EstadoPedido Despachar();
    public abstract EstadoPedido Cancelar();
}

public class Pendiente : EstadoPedido {
    public override EstadoPedido Pagar()     => new Pagado();
    public override EstadoPedido Despachar() => throw new InvalidOperationException();
    public override EstadoPedido Cancelar()  => new Cancelado();
}

public class Pagado : EstadoPedido {
    public override EstadoPedido Pagar()     => throw new InvalidOperationException("Ya pagado");
    public override EstadoPedido Despachar() => new Despachado();
    public override EstadoPedido Cancelar()  => new Cancelado();
}

public class Despachado : EstadoPedido { /* todas las transiciones inválidas */ }
public class Cancelado  : EstadoPedido { /* todas las transiciones inválidas */ }

public class Pedido {
    public EstadoPedido Estado { get; private set; } = new Pendiente();

    public void Pagar()     => Estado = Estado.Pagar();
    public void Despachar() => Estado = Estado.Despachar();
    public void Cancelar()  => Estado = Estado.Cancelar();
}
```

**Por qué satisface SRP.** Cada clase de estado responde a un único actor: el área que define qué transiciones permite. Si Operaciones decide que un pedido cancelado puede reactivarse, eso es modificar `Cancelado`, sin tocar las demás. Si Producto decide agregar un nuevo estado `EnRevision`, es agregar una clase nueva — no modificar las existentes (acá converge con OCP).

> **Variante moderna**: con records sealed y pattern matching, un `Pedido` puede modelarse con `record Pedido(EstadoPedido Estado, ...)` y un método que use pattern matching exhaustivo. Es la misma separación, otra implementación.

## 4. Operación: Command, Chain of Responsibility

Acá la responsabilidad extraída no es "cómo se hace" sino "qué se hace" — la operación misma se vuelve un objeto.

### 4.1 Command — convertir operaciones en objetos de primera clase

**El motivo de cambio que separa.** El código que **invoca** una operación se mezcla con el código que la **ejecuta**. Si querés agregar logging, undo, queuing, programación diferida, transaccionalidad — todo eso son responsabilidades distintas que no caben adentro del invocador ni adentro de la operación.

**Síntoma típico.** Un método grande que hace varias cosas, sin posibilidad de revertir, registrar, programar o reordenar.

```csharp
// Editor de texto con operaciones inseparables
public class Editor {
    private string contenido = "";

    public void Insertar(int pos, string texto) {
        contenido = contenido.Insert(pos, texto);
    }

    public void Borrar(int pos, int largo) {
        contenido = contenido.Remove(pos, largo);
    }

    // ¿Cómo agregamos undo? ¿Cómo registramos un log de cambios?
    // Tendríamos que tocar Insertar y Borrar.
}
```

**Refactor.** Cada operación es un objeto que sabe ejecutarse y revertirse.

```csharp
public interface IComando {
    void Ejecutar();
    void Deshacer();
}

public class Insertar(Editor editor, int pos, string texto) : IComando {
    public void Ejecutar() => editor.InsertarTexto(pos, texto);
    public void Deshacer() => editor.BorrarTexto(pos, texto.Length);
}

public class Borrar(Editor editor, int pos, int largo) : IComando {
    private string textoBorrado = "";

    public void Ejecutar() {
        textoBorrado = editor.LeerTexto(pos, largo);
        editor.BorrarTexto(pos, largo);
    }

    public void Deshacer() => editor.InsertarTexto(pos, textoBorrado);
}

public class HistorialComandos {
    private readonly Stack<IComando> historial = new();

    public void Ejecutar(IComando c) {
        c.Ejecutar();
        historial.Push(c);
    }

    public void Deshacer() {
        if (historial.Count > 0) historial.Pop().Deshacer();
    }
}
```

**Por qué satisface SRP.** El editor cambia si cambia su contenido. Cada comando cambia si cambia esa operación específica. El historial cambia si cambia la política de undo/redo. Tres motivos disjuntos.

> **Caso real**: el patrón Command es la base de cualquier sistema con undo/redo, queuing, transacciones distribuidas (cada paso reversible es un comando), y de los handlers de MediatR — un `IRequest<T>` es esencialmente un Command serializable. En aplicaciones de escritorio (WPF, MAUI, Avalonia, WinUI) la interfaz `System.Windows.Input.ICommand` cumple el mismo rol, conectando botones de la UI con la acción del ViewModel:
>
> ```csharp
> public interface ICommand {
>     bool CanExecute(object? parameter);
>     void Execute(object? parameter);
>     event EventHandler? CanExecuteChanged;
> }
> ```

### 4.2 Chain of Responsibility — extraer cada paso de un proceso

**El motivo de cambio que separa.** Un proceso de validación, autorización o filtrado tiene varios pasos. Si todos viven en una clase, agregar un paso requiere modificarla.

**Síntoma típico.** Un método con muchos `if` consecutivos donde cada uno chequea una condición y decide si continúa o aborta.

```csharp
public class ProcesadorRequest {
    public Resultado Procesar(Request r) {
        if (r.Token is null)              return Resultado.Error("Sin token");
        if (!ValidarToken(r.Token))       return Resultado.Error("Token inválido");
        if (!TienePermiso(r.Usuario, r.Recurso))  return Resultado.Error("Sin permiso");
        if (ExcedioLimite(r.Usuario))     return Resultado.Error("Límite excedido");
        if (!FormatoValido(r.Payload))    return Resultado.Error("Formato inválido");
        return Ejecutar(r);
    }
}
```

Cada paso es una responsabilidad: autenticación, autorización, rate limit, validación de formato, ejecución. Vivir todos juntos significa que cualquier cambio en uno arriesga regresión en los demás.

**Refactor.** Cada paso es un handler en una cadena.

```csharp
public abstract class Handler {
    private Handler? siguiente;

    public Handler EncadenarCon(Handler h) {
        siguiente = h;
        return h;
    }

    public Resultado Manejar(Request r) {
        var local = HacerLoMio(r);
        if (!local.EsExitoso) return local;
        return siguiente?.Manejar(r) ?? Resultado.Ok();
    }

    protected abstract Resultado HacerLoMio(Request r);
}

public class HandlerAutenticacion : Handler {
    protected override Resultado HacerLoMio(Request r) =>
        r.Token is null || !ValidarToken(r.Token)
            ? Resultado.Error("Autenticación fallida")
            : Resultado.Ok();
}

public class HandlerAutorizacion : Handler { /* ... */ }
public class HandlerRateLimit    : Handler { /* ... */ }

// Composición
var pipeline = new HandlerAutenticacion();
pipeline.EncadenarCon(new HandlerAutorizacion())
        .EncadenarCon(new HandlerRateLimit());
```

**Por qué satisface SRP.** Cada handler tiene un único motivo de cambio. Reordenar pasos, agregar uno nuevo, o reemplazar el algoritmo de uno existente son operaciones aditivas y aisladas.

> **Conexión .NET**: el pipeline de middleware de ASP.NET Core es la materialización idiomática de Chain of Responsibility. Cada middleware decide si pasa o no al siguiente, y se componen en `Program.cs`.

## 5. Creación: Factory, Builder

Cuando el problema no es ejecutar un comportamiento sino *construir* un objeto, el patrón apropiado pertenece a esta familia.

### 5.1 Factory — extraer la decisión de qué tipo crear

**El motivo de cambio que separa.** Una clase elige qué tipo concreto instanciar y, dentro de ella, mezcla esa decisión con la lógica que usa el objeto. Cambiar la regla de selección obliga a tocar el código de uso, y agregar un nuevo tipo concreto fuerza modificación.

**Síntoma típico.** Un `new` distinto según un parámetro, dentro del código de negocio.

```csharp
public class ProcesadorPagos {
    public void Procesar(string medio, decimal monto) {
        IGateway gw;
        if      (medio == "tarjeta")       gw = new GatewayTarjeta();
        else if (medio == "transferencia") gw = new GatewayTransferencia();
        else if (medio == "mercado_pago")  gw = new GatewayMercadoPago();
        else throw new ArgumentException(medio);

        gw.Cobrar(monto);
    }
}
```

**Refactor.**

```csharp
public interface IFabricaGateways {
    IGateway Crear(string medio);
}

public class FabricaGateways : IFabricaGateways {
    public IGateway Crear(string medio) => medio switch {
        "tarjeta"        => new GatewayTarjeta(),
        "transferencia"  => new GatewayTransferencia(),
        "mercado_pago"   => new GatewayMercadoPago(),
        _ => throw new ArgumentException(medio),
    };
}

public class ProcesadorPagos(IFabricaGateways fabrica) {
    public void Procesar(string medio, decimal monto) =>
        fabrica.Crear(medio).Cobrar(monto);
}
```

**Por qué satisface SRP.** El procesador cambia si cambia la lógica de cobro. La fábrica cambia si cambia la lógica de elección o si aparece un nuevo gateway. Agregar un nuevo medio (criptomonedas, por ejemplo) es modificar sólo la fábrica.

> **Nota .NET**: el contenedor de DI (`IServiceProvider`) es esencialmente una *abstract factory* genérica. Cuando registrás `services.AddScoped<IGateway, GatewayTarjeta>()`, estás delegando la decisión de construcción al contenedor. Para casos donde la elección depende de un parámetro de runtime, una factory explícita sigue siendo la herramienta correcta. Factories comunes en el framework: `IHttpClientFactory`, `ILoggerFactory`, `IDbContextFactory<TContext>`. Estos nombres no son casualidad: separan creación/configuración del código que usa el objeto.

### 5.2 Builder — extraer la construcción compleja

**El motivo de cambio que separa.** Un objeto requiere muchos parámetros para construirse, algunos opcionales, con dependencias entre ellos y validaciones. Mezclar esa lógica con el código que usa el objeto produce constructores monstruosos o construcciones repetitivas.

**Síntoma típico.** Constructores con 8 parámetros, varios opcionales, con interdependencias ("si configurás autenticación tenés que dar también credenciales").

```csharp
// Cada lugar que arma una request HTTP repite mucho boilerplate
var req = new HttpRequest(
    method: "POST",
    url: "https://api.empresa.com/v1/clientes",
    headers: new Dictionary<string, string> {
        ["Authorization"] = $"Bearer {token}",
        ["Content-Type"]  = "application/json",
        ["X-Tenant-Id"]   = tenantId,
        ["X-Trace-Id"]    = traceId,
    },
    body: JsonSerializer.Serialize(payload),
    timeout: TimeSpan.FromSeconds(30),
    followRedirects: true,
    retries: 3);
```

**Refactor.**

```csharp
public class RequestBuilder {
    private string method = "GET";
    private string url    = "";
    private readonly Dictionary<string, string> headers = new();
    private string?  body;
    private TimeSpan timeout = TimeSpan.FromSeconds(30);

    public RequestBuilder Post(string u)              { method = "POST"; url = u; return this; }
    public RequestBuilder Get(string u)               { method = "GET";  url = u; return this; }
    public RequestBuilder Header(string k, string v)  { headers[k] = v; return this; }
    public RequestBuilder Auth(string token)          { headers["Authorization"] = $"Bearer {token}"; return this; }
    public RequestBuilder Tenant(string id)           { headers["X-Tenant-Id"] = id; return this; }
    public RequestBuilder Trace(string id)            { headers["X-Trace-Id"]  = id; return this; }
    public RequestBuilder Json(object body) {
        headers["Content-Type"] = "application/json";
        this.body = JsonSerializer.Serialize(body);
        return this;
    }
    public RequestBuilder Timeout(TimeSpan t)         { timeout = t; return this; }

    public HttpRequest Build() => new HttpRequest(method, url, headers, body, timeout);
}

// Uso
var req = new RequestBuilder()
    .Post("https://api.empresa.com/v1/clientes")
    .Auth(token)
    .Tenant(tenantId)
    .Trace(traceId)
    .Json(payload)
    .Build();
```

**Por qué satisface SRP.** El builder es responsable de las reglas de construcción: qué combinaciones son válidas, qué headers se derivan de qué llamadas, cómo se serializa el body. La clase `HttpRequest` es responsable de representar la request armada. El consumidor es responsable de decidir qué request quiere. Tres responsabilidades, tres lugares.

> **En .NET**: el patrón aparece en `WebApplicationBuilder`, `HostBuilder`, `EntityTypeBuilder<T>` (mapeo de EF Core), y muchos métodos `Use*` y `With*` del framework. Es uno de los patrones más usados en la API moderna del framework.

## 6. Coordinación: Mediator, Observer, Facade

Cuando el problema es "quién le habla a quién", la responsabilidad extraída es la coordinación misma.

### 6.1 Mediator — extraer la coordinación entre componentes

**El motivo de cambio que separa.** Varios objetos colaboran, y cada uno conoce a los demás directamente. Cualquier cambio en cómo colaboran afecta a varios.

**Síntoma típico.** Una pieza donde un servicio le habla a otros tres, esos tres se conocen entre sí, y el conjunto forma un grafo de dependencias mutuas.

```csharp
public class ServicioPedidos(
    ServicioInventario      inventario,
    ServicioPagos           pagos,
    ServicioNotificaciones  notif,
    ServicioFacturacion     factura,
    ServicioLogistica       logistica) {

    public async Task ConfirmarPedido(Pedido p) {
        await inventario.ReservarStock(p);
        var pago = await pagos.Cobrar(p);
        await factura.Emitir(p, pago);
        await logistica.AgendarEnvio(p);
        await notif.NotificarCliente(p);
    }
}
```

Si mañana cambia el flujo (validar fraude antes de cobrar), el servicio se modifica. Si aparece un nuevo paso (puntos de fidelidad), también. Si un paso se elimina, también.

**Refactor.** Cada paso es un handler independiente; un mediador los coordina mediante un evento o un pipeline.

```csharp
// Notificación de dominio: "se confirmó un pedido"
public record PedidoConfirmado(int PedidoId);

public class HandlerReservarStock(ServicioInventario inv) : IHandler<PedidoConfirmado> {
    public Task Handle(PedidoConfirmado e, CancellationToken ct) =>
        inv.ReservarStock(e.PedidoId, ct);
}

public class HandlerCobrarPago(ServicioPagos pagos)        : IHandler<PedidoConfirmado> { /* ... */ }
public class HandlerEmitirFactura(ServicioFacturacion fc)  : IHandler<PedidoConfirmado> { /* ... */ }
public class HandlerAgendarEnvio(ServicioLogistica log)    : IHandler<PedidoConfirmado> { /* ... */ }
public class HandlerNotificarCliente(ServicioNotif notif)  : IHandler<PedidoConfirmado> { /* ... */ }

// El mediador despacha el evento a todos los handlers registrados
public class ServicioPedidos(IMediator mediator) {
    public Task ConfirmarPedido(Pedido p) =>
        mediator.Publish(new PedidoConfirmado(p.Id));
}
```

**Por qué satisface SRP.** Cada handler responde a un solo equipo. Cada uno cambia por su propio motivo. Agregar un paso es agregar una clase. Eliminar un paso es borrar un archivo. El servicio orquestador queda en lo suyo: traducir un comando de aplicación en un evento de dominio, y delegar.

> **Caso real en .NET**: MediatR (la librería) y los handlers de CQRS son la implementación canónica de este patrón. La librería es opcional; el patrón no.

### 6.2 Observer — extraer la reacción a eventos

**El motivo de cambio que separa.** Una clase que produce un evento (cambio de estado, acción del usuario) tendría que conocer a todos los interesados. Cada nuevo interesado modifica al productor.

**Síntoma típico.** Después de hacer algo importante, una clase llama a varios servicios para "avisarles".

```csharp
public class CarritoCompras {
    public void AgregarItem(Item i) {
        items.Add(i);

        // Avisos directos — la clase conoce a sus consumidores
        analiticas.Registrar(i);
        recomendador.Aprender(i);
        promociones.EvaluarOfertas(items);
    }
}
```

**Refactor.** El carrito sólo notifica un evento; los interesados se registran independientemente.

```csharp
public class CarritoCompras {
    public event Action<Item>? ItemAgregado;

    public void AgregarItem(Item i) {
        items.Add(i);
        ItemAgregado?.Invoke(i);
    }
}

// En la composición de la app
carrito.ItemAgregado += i => analiticas.Registrar(i);
carrito.ItemAgregado += i => recomendador.Aprender(i);
carrito.ItemAgregado += i => promociones.EvaluarOfertas(carrito.Items);
```

**Por qué satisface SRP.** El carrito maneja su contenido y avisa cambios; cambia si cambia la lógica del carrito. Cada observador es responsable de su reacción específica; cambia por su propio motivo. Si analítica decide que ya no le interesa registrar items, se desuscribe sin que el carrito se entere.

> **En .NET**: el patrón aparece naturalmente con `event`, `EventHandler<T>`, `IObservable<T>` y la familia de operadores reactivos de System.Reactive. En Blazor, los `EventCallback<T>` cumplen un rol similar entre componentes padre e hijo.

### 6.3 Facade — extraer una interfaz simple sobre un subsistema complejo

**El motivo de cambio que separa.** Un cliente del código tiene que orquestar varias clases internas para hacer una tarea común. Esa orquestación se replica en cada cliente, y cada cliente conoce demasiado de la estructura interna.

**Síntoma típico.** El mismo bloque de cinco líneas que instancia varios objetos y los enlaza, repetido en muchos lugares.

```csharp
// Cada cliente orquesta los detalles
var datos      = new SqlConnection(...).Query("SELECT ...");
var procesado  = new Procesador(opciones).Procesar(datos);
var formateado = new Formateador(formato).Formatear(procesado);
var pdf        = new GeneradorPdf().Generar(formateado);
File.WriteAllBytes(ruta, pdf);
```

**Refactor.** Una clase con un método claro encapsula toda la orquestación.

```csharp
public class GeneradorReportes(
    SqlConnection      conn,
    OpcionesProcesador opts,
    FormatoSalida      fmt) {

    public byte[] Generar(string consulta) {
        var datos      = conn.Query(consulta);
        var procesado  = new Procesador(opts).Procesar(datos);
        var formateado = new Formateador(fmt).Formatear(procesado);
        return new GeneradorPdf().Generar(formateado);
    }
}
```

**Por qué satisface SRP.** El facade es responsable de la orquestación; cambia si cambia el flujo. Cada subsistema interno cambia por su propio motivo. Y el cliente recupera el SRP que había perdido: ya no tiene que conocer ni orquestar la maquinaria interna.

> **Cuidado con los facades anémicos**: si un facade sólo redirige a un único subsistema sin agregar valor, no extrae responsabilidad — sólo agrega indirección. Un buen facade orquesta; uno malo es un alias.

## 7. Persistencia: Repository, Unit of Work, Specification

Acceso a datos y reglas de búsqueda son responsabilidades distintas del dominio.

### 7.1 Repository — extraer el acceso a datos del dominio

**El motivo de cambio que separa.** Una clase de dominio que se persiste a sí misma mezcla reglas de negocio (qué es válido, qué se puede hacer) con reglas de infraestructura (qué motor de base de datos, qué esquema, qué consultas).

**Síntoma típico.** Active Record: clases de dominio con métodos `Save`, `Load`, `Delete`, conocidas como "hacen todo solas".

```csharp
public class Cliente {
    public int    Id     { get; set; }
    public string Nombre { get; set; } = "";

    public void Save() {
        using var conn = new SqlConnection(...);
        conn.Open();
        // SQL aquí
    }

    public static Cliente Load(int id) { /* ... */ }
}
```

**Refactor.**

```csharp
public class Cliente {
    public int    Id     { get; private set; }
    public string Nombre { get; private set; } = "";

    public void CambiarNombre(string nuevo) {
        if (string.IsNullOrWhiteSpace(nuevo)) throw new DomainException();
        Nombre = nuevo;
    }
}

public interface IRepositorioClientes {
    Task<Cliente?> ObtenerAsync(int id, CancellationToken ct);
    Task GuardarAsync(Cliente c, CancellationToken ct);
}

public class RepositorioClientesEf(AppDbContext db) : IRepositorioClientes {
    public Task<Cliente?> ObtenerAsync(int id, CancellationToken ct) =>
        db.Clientes.FindAsync(id, ct).AsTask();

    public async Task GuardarAsync(Cliente c, CancellationToken ct) {
        db.Clientes.Update(c);
        await db.SaveChangesAsync(ct);
    }
}

// Versión en memoria — para tests
public class RepositorioClientesEnMemoria : IRepositorioClientes {
    private readonly Dictionary<int, Cliente> almacen = new();

    public Task<Cliente?> ObtenerAsync(int id, CancellationToken ct) =>
        Task.FromResult(almacen.GetValueOrDefault(id));

    public Task GuardarAsync(Cliente c, CancellationToken ct) {
        almacen[c.Id] = c;
        return Task.CompletedTask;
    }
}
```

**Por qué satisface SRP.** `Cliente` cambia si cambian las reglas del negocio. `RepositorioClientesEf` cambia si cambian las decisiones de infraestructura. La interfaz `IRepositorioClientes` aísla al dominio de cualquier conocimiento sobre EF, SQL o el motor de base de datos. Y el servicio que usa el repositorio no necesita modificarse para tests: simplemente recibe la implementación en memoria.

### 7.2 Unit of Work — agrupar cambios y confirmarlos como una unidad

**El motivo de cambio que separa.** Un caso de uso modifica varias entidades y hay que decidir cuándo se confirman al medio persistente. Si cada repositorio confirma por su cuenta, los cambios pueden quedar a medias en caso de error.

**En .NET**, `DbContext` cumple en gran medida los roles de Repository + Unit of Work:

```csharp
context.Contactos.Add(contacto);
context.Contactos.Remove(contactoViejo);
context.Contactos.Update(contactoEditado);

// Hasta acá, sólo se registraron cambios en memoria.
await context.SaveChangesAsync(); // Confirma todo en una sola transacción.
```

```text
DbContext
    ├─► rastrea entidades
    ├─► detecta cambios
    ├─► agrupa operaciones
    └─► confirma con SaveChangesAsync()
```

Servicio típico:

```csharp
public sealed class ContactoService(AgendaDbContext context) {
    public async Task AgregarAsync(Contacto contacto) {
        context.Contactos.Add(contacto);
        await context.SaveChangesAsync();
    }
}
```

**SRP aplicado**:

```text
ContactoService → decide qué operación de negocio hacer
DbContext       → rastrea y confirma cambios
Base de datos   → persiste físicamente
```

El servicio no arma SQL, no abre conexiones y no maneja manualmente el estado de cada entidad. Eso queda encapsulado en EF Core.

### 7.3 Specification — extraer las reglas de criterio del dominio

**El motivo de cambio que separa.** Las reglas de "qué cuenta como X" tienden a vivir en consultas o en condicionales del repositorio o del servicio. Eso mezcla lógica de negocio con detalles de cómo se ejecuta la búsqueda.

**Síntoma típico.** Métodos de repositorio con nombres como `ObtenerProductosActivosDeCategoriaConPrecioMayor`, que crecen sin parar.

```csharp
public interface IRepositorioProductos {
    Task<List<Producto>> ObtenerActivosAsync();
    Task<List<Producto>> ObtenerActivosDeCategoriaAsync(int catId);
    Task<List<Producto>> ObtenerActivosConPrecioMayorAsync(decimal min);
    Task<List<Producto>> ObtenerActivosDeCategoriaConPrecioMayorAsync(int catId, decimal min);
    // ...
}
```

**Refactor.** Cada criterio es una clase; se componen.

```csharp
public abstract class Specification<T> {
    public abstract Expression<Func<T, bool>> AsExpression();
    public Specification<T> And(Specification<T> otra) => new AndSpec<T>(this, otra);
}

public class ProductoActivo : Specification<Producto> {
    public override Expression<Func<Producto, bool>> AsExpression() => p => p.Activo;
}

public class ProductoEnCategoria(int catId) : Specification<Producto> {
    public override Expression<Func<Producto, bool>> AsExpression() => p => p.CategoriaId == catId;
}

public class ProductoConPrecioMayor(decimal min) : Specification<Producto> {
    public override Expression<Func<Producto, bool>> AsExpression() => p => p.Precio > min;
}

// Uso compuesto
var spec = new ProductoActivo()
    .And(new ProductoEnCategoria(3))
    .And(new ProductoConPrecioMayor(1000));

var resultados = await db.Productos.Where(spec.AsExpression()).ToListAsync();
```

**Por qué satisface SRP.** Cada specification responde a una regla de negocio. La regla "qué es un producto activo" cambia si cambia el negocio, y se modifica una sola clase. El repositorio queda con una API genérica (`Buscar(Specification<T>)`) que no necesita crecer con cada combinación nueva.

## 8. Adaptación: Adapter, Decorator, Proxy

A veces la responsabilidad extraída es la adaptación entre dos contratos, o el agregado de comportamiento transversal sin contaminar al original.

### 8.1 Adapter — extraer la traducción entre contratos

**El motivo de cambio que separa.** El código que usa una librería externa se acopla a su API. Si la librería cambia o se reemplaza, hay que modificar todos los consumidores.

**Síntoma típico.** Llamadas directas a un SDK de tercero esparcidas por la lógica de negocio.

```csharp
// Acoplado al SDK de Twilio
public class ServicioNotificaciones(TwilioRestClient cliente) {
    public void EnviarSms(string telefono, string texto) {
        MessageResource.Create(
            body: texto,
            from: new PhoneNumber("+5491100000000"),
            to:   new PhoneNumber(telefono),
            client: cliente);
    }
}
```

**Refactor.**

```csharp
// Contrato propio de la aplicación
public interface IEnviadorSms {
    Task EnviarAsync(string telefono, string texto, CancellationToken ct);
}

// Adapter — traduce nuestra API a la del SDK externo
public class EnviadorSmsTwilio(TwilioRestClient cliente, string remitente) : IEnviadorSms {
    public Task EnviarAsync(string telefono, string texto, CancellationToken ct) {
        return MessageResource.CreateAsync(
            body: texto,
            from: new PhoneNumber(remitente),
            to:   new PhoneNumber(telefono),
            client: cliente);
    }
}

// Otro proveedor — mismo contrato, distinta traducción
public class EnviadorSmsInfobip(InfobipClient cliente) : IEnviadorSms {
    public Task EnviarAsync(string telefono, string texto, CancellationToken ct) =>
        cliente.Sms.SendAsync(new InfobipSmsRequest { To = telefono, Body = texto }, ct);
}
```

**Por qué satisface SRP.**

```text
ContactoService     → decide cuándo notificar
IEnviadorSms        → contrato propio de la aplicación
EnviadorSmsTwilio   → adapta el SDK de Twilio
EnviadorSmsInfobip  → adapta el SDK de Infobip
```

La aplicación no queda contaminada por los nombres, modelos y rarezas del proveedor externo. Cambiar de proveedor es cambiar el registro de DI; el resto del código no se entera. Todo SDK trae su zoológico; mejor dejarlo en una jaula.

### 8.2 Decorator — agregar comportamiento sin modificar la clase original

**El motivo de cambio que separa.** Una clase carga con responsabilidades transversales (logging, caché, retry, métricas, autorización) que no tienen nada que ver con su lógica principal pero que aparecen en muchos lugares.

**Síntoma típico.** Una clase con su lógica + logs + caché + medición de tiempos + manejo de errores, todo enredado.

```csharp
public sealed class RepoProductos(SqlConnection conn, IMemoryCache cache, ILogger<RepoProductos> logger) {
    public async Task<Producto?> ObtenerAsync(int id) {
        if (cache.TryGetValue($"prod:{id}", out Producto? p)) {
            logger.LogInformation("Hit de caché");
            return p;
        }
        var sw = Stopwatch.StartNew();
        p = await conn.QuerySingleOrDefaultAsync<Producto>(
            "SELECT * FROM productos WHERE id=@id", new { id });
        logger.LogInformation("Query en {Ms}ms", sw.ElapsedMilliseconds);
        if (p is not null) cache.Set($"prod:{id}", p);
        return p;
    }
}
```

Tres responsabilidades en una: acceder a datos, cachear, loguear.

**Refactor.**

```csharp
public interface IRepoProductos {
    Task<Producto?> ObtenerAsync(int id, CancellationToken ct);
}

public class RepoProductosSql(SqlConnection conn) : IRepoProductos {
    public Task<Producto?> ObtenerAsync(int id, CancellationToken ct) =>
        conn.QuerySingleOrDefaultAsync<Producto?>(
            "SELECT * FROM productos WHERE id=@id", new { id });
}

public class RepoProductosLogging(IRepoProductos inner, ILogger<RepoProductosLogging> log)
    : IRepoProductos {
    public async Task<Producto?> ObtenerAsync(int id, CancellationToken ct) {
        log.LogInformation("Buscando producto {Id}", id);
        var sw = Stopwatch.StartNew();
        var r  = await inner.ObtenerAsync(id, ct);
        log.LogInformation("{Resultado} en {Ms}ms",
            r is null ? "no encontrado" : "encontrado", sw.ElapsedMilliseconds);
        return r;
    }
}

public class RepoProductosCache(IRepoProductos inner, IMemoryCache cache) : IRepoProductos {
    public Task<Producto?> ObtenerAsync(int id, CancellationToken ct) =>
        cache.GetOrCreateAsync($"prod:{id}", async _ => await inner.ObtenerAsync(id, ct));
}
```

```csharp
// Composición — orden inverso al orden de ejecución
IRepoProductos repo =
    new RepoProductosLogging(
        new RepoProductosCache(
            new RepoProductosSql(conn),
            cache),
        logger);
```

**Por qué satisface SRP.** `RepoProductosSql` cambia si cambia la query o el esquema. `RepoProductosCache` cambia si cambia la política de caché. `RepoProductosLogging` cambia si cambia qué se loguea. Tres motivos distintos, tres clases, tres archivos. Y la composición es flexible: en tests se usa sólo el sql, en producción los tres.

> **En .NET**: la familia `Stream` (`FileStream` → `BufferedStream` → `GZipStream` → `CryptoStream`) es el ejemplo canónico de Decorator en la BCL. Cada capa agrega una responsabilidad ortogonal sin tocar las inferiores.

### 8.3 Proxy — extraer el control de acceso o la carga diferida

**El motivo de cambio que separa.** Un objeto caro de crear o sensible al acceso requiere lógica especial alrededor de su uso (lazy loading, autorización, control remoto). Esa lógica no debería vivir adentro del objeto.

**Síntoma típico.** Lógica de "¿está cargado? ¿puedo acceder?" mezclada con la lógica de uso.

**Refactor.**

```csharp
public interface IDocumento {
    string Contenido { get; }
}

public class DocumentoReal(string ruta) : IDocumento {
    public string Contenido { get; } = File.ReadAllText(ruta);  // costoso, eager
}

public class DocumentoLazy(string ruta) : IDocumento {
    private string? cache;
    public string Contenido => cache ??= File.ReadAllText(ruta);
}

public class DocumentoProtegido(IDocumento inner, IAutorizador auth) : IDocumento {
    public string Contenido =>
        auth.PuedeLeer() ? inner.Contenido : throw new UnauthorizedAccessException();
}
```

**Por qué satisface SRP.** El proxy se ocupa del control (lazy load, autorización, instrumentación remota); el documento real se ocupa de su contenido. La diferencia con Decorator es semántica: el decorator agrega comportamiento ortogonal; el proxy controla el acceso al original.

## 9. Idiomáticos de .NET: DI, Options, Middleware, Filters, DTO + Mapper, MVVM

Esta sección reúne patrones que en otras plataformas requerirían librerías o configuración manual, pero que en .NET vienen de fábrica como parte del framework. Conviene reconocerlos por nombre porque son la lengua común del ecosistema.

Algunos nombres —como DI, middleware, filters, repository o unit of work— ya aparecieron antes desde la lente de SOLID aplicado a la BCL y al stack moderno. Acá reaparecen a propósito, pero con otra pregunta guía: no “qué principio ilustran”, sino “qué forma concreta conviene usar al separar responsabilidades en código propio”.

### 9.1 Dependency Injection — separar construcción de uso

Ya vimos DIP como principio. El patrón **DI** es la implementación: en lugar de que cada clase haga `new` de sus dependencias, las recibe por constructor; un contenedor central las construye y las conecta.

```csharp
// Sin DI
public sealed class ContactoService {
    private readonly JsonContactoRepository repository = new("contactos.json"); // acoplado
}

// Con DI
public sealed class ContactoService(IContactoRepository repository) {
    public void Agregar(Contacto contacto) {
        var contactos = repository.ObtenerTodos();
        contactos.Add(contacto);
        repository.GuardarTodos(contactos);
    }
}
```

Registro en ASP.NET Core / Worker Service / MAUI:

```csharp
builder.Services.AddSingleton<IContactoRepository>(new JsonContactoRepository("contactos.json"));
builder.Services.AddScoped<ContactoService>();
```

**SRP aplicado**:

```text
ContactoService        → ejecuta el caso de uso
JsonContactoRepository → persiste contactos en JSON
DI Container           → construye y conecta objetos
```

La responsabilidad de construir objetos sale del servicio y queda en la composición de la aplicación.

### 9.2 Options Pattern — separar configuración de lógica

**Mal enfoque**: el repositorio parsea configuración por todos lados.

```csharp
public sealed class JsonContactoRepository(IConfiguration configuration) {
    public List<Contacto> ObtenerTodos() {
        string path = configuration["Agenda:ArchivoContactos"]!;
        // ...
        return [];
    }
}
```

**Con Options Pattern**: la configuración se modela como una clase fuertemente tipada.

```csharp
public sealed class AgendaOptions {
    public string ArchivoContactos { get; set; } = "contactos.json";
}

builder.Services.Configure<AgendaOptions>(builder.Configuration.GetSection("Agenda"));

public sealed class JsonContactoRepository(IOptions<AgendaOptions> options) : IContactoRepository {
    private readonly AgendaOptions options = options.Value;

    public List<Contacto> ObtenerTodos() {
        string path = this.options.ArchivoContactos;
        // ...
        return [];
    }
}
```

**SRP aplicado**:

```text
appsettings.json       → valores externos
AgendaOptions          → modelo tipado de configuración
Options infrastructure → enlaza configuración
Repository             → usa configuración ya modelada
```

El servicio usa configuración. No la busca con strings mágicos en cada esquina.

### 9.3 Middleware Pipeline — separar responsabilidades HTTP

ASP.NET Core procesa requests usando un pipeline donde cada middleware es una pieza con una sola responsabilidad:

```text
Request → ExceptionHandling → Routing → Authentication → Authorization → Endpoint
```

```csharp
app.UseExceptionHandler("/error");
app.UseRouting();
app.UseAuthentication();
app.UseAuthorization();
app.MapControllers();
```

Middleware propio:

```csharp
public sealed class RequestLoggingMiddleware(
    RequestDelegate next,
    ILogger<RequestLoggingMiddleware> logger) {

    public async Task InvokeAsync(HttpContext context) {
        logger.LogInformation("Request: {Method} {Path}",
            context.Request.Method, context.Request.Path);
        await next(context);
    }
}

app.UseMiddleware<RequestLoggingMiddleware>();
```

El endpoint no debería ocuparse de autenticar, autorizar, loguear, manejar errores globales y además resolver negocio. Para eso está el pipeline.

### 9.4 Filters — separar lógica transversal de controladores

Los filtros (`IActionFilter`, `IAuthorizationFilter`, `IExceptionFilter`, `IResultFilter`, `IEndpointFilter`) sacan lógica transversal de los controladores:

```csharp
public sealed class LogActionFilter(ILogger<LogActionFilter> logger) : IActionFilter {
    public void OnActionExecuting(ActionExecutingContext context)
        => logger.LogInformation("Ejecutando action.");

    public void OnActionExecuted(ActionExecutedContext context)
        => logger.LogInformation("Action ejecutada.");
}

[ServiceFilter(typeof(LogActionFilter))]
public sealed class ContactosController(ContactoService service) : ControllerBase {
    [HttpPost("contactos")]
    public IActionResult Crear(Contacto contacto) {
        service.Agregar(contacto);
        return Ok();
    }
}
```

**SRP aplicado**:

```text
Controller   → traduce HTTP a caso de uso
Filter       → logging, validación, errores, autorización
Service      → reglas del caso de uso
```

El controlador no termina siendo repositorio, validador, logger, mapper y psicólogo familiar.

### 9.5 DTO + Mapper — separar contrato externo de modelo interno

Una entidad del dominio o de EF Core no siempre debería exponerse directamente por una API.

```csharp
// Modelo interno (entidad)
public sealed class Contacto {
    public int    Id       { get; set; }
    public string Nombre   { get; set; } = "";
    public string Apellido { get; set; } = "";
    public string Email    { get; set; } = "";
}

// Contrato externo (DTO)
public sealed class ContactoDto {
    public int    Id              { get; set; }
    public string NombreCompleto  { get; set; } = "";
    public string Email           { get; set; } = "";
}

// Mapper
public static class ContactoMapper {
    public static ContactoDto ToDto(Contacto c) => new ContactoDto {
        Id              = c.Id,
        NombreCompleto  = $"{c.Nombre} {c.Apellido}",
        Email           = c.Email
    };
}

List<ContactoDto> resultado = contactos.Select(ContactoMapper.ToDto).ToList();
```

**SRP aplicado**:

```text
Contacto       → modelo interno
ContactoDto    → contrato externo
ContactoMapper → conversión entre ambos
```

Si cambia la API pública, no necesariamente cambia el dominio. Si cambia el dominio, no necesariamente cambia el contrato externo.

### 9.6 MVVM — separar UI, estado e interacción

MVVM es el patrón estándar en .NET desktop/mobile (WPF, MAUI, Avalonia, WinUI). Separa la aplicación así:

```text
View
    ↓ binding
ViewModel
    ↓ usa
Service
    ↓ usa
Repository
```

| Pieza      | Responsabilidad                      |
|------------|--------------------------------------|
| View       | Dibujar la interfaz                  |
| ViewModel  | Estado y acciones de la pantalla     |
| Model      | Datos y reglas del dominio           |
| Service    | Casos de uso                         |
| Repository | Persistencia                         |

```csharp
public sealed class ContactoViewModel {
    private readonly ContactoService service;

    public string Nombre   { get; set; } = "";
    public string Apellido { get; set; } = "";

    public ICommand GuardarCommand { get; }

    public ContactoViewModel(ContactoService service) {
        this.service    = service;
        GuardarCommand  = new GuardarContactoCommand(this.service);
    }
}
```

Esto evita meter consultas SQL, parseo JSON y validaciones dentro del code-behind. El code-behind cargado es humedad en los cimientos.

## 10. Caso integrador: refactorizar una god class

Veamos un caso completo que combina varios patrones para extraer responsabilidades de una god class de venta online.

### 10.1 La god class original

```csharp
public class ServicioVentas(SqlConnection conn, ILogger logger) {

    public async Task<int> ProcesarVenta(VentaDto dto) {
        // Validar
        if (dto.Items is null || dto.Items.Count == 0)
            throw new ArgumentException("Sin items");
        if (dto.ClienteId <= 0)
            throw new ArgumentException("Cliente inválido");

        // Calcular total con descuento por tipo de cliente
        var subtotal = dto.Items.Sum(i => i.Precio * i.Cantidad);
        decimal descuento;
        var clienteRow = await conn.QuerySingleAsync(
            "SELECT tipo, email FROM clientes WHERE id=@id", new { id = dto.ClienteId });
        if      (clienteRow.tipo == "vip")     descuento = subtotal * 0.15m;
        else if (clienteRow.tipo == "premium") descuento = subtotal * 0.10m;
        else if (clienteRow.tipo == "regular") descuento = subtotal * 0.05m;
        else                                    descuento = 0;

        var total = subtotal - descuento;

        // Procesar pago
        if (dto.MedioPago == "tarjeta") {
            var resp = await new HttpClient().PostAsJsonAsync(
                "https://api.tarjetas.com/cobro", new { total });
            if (!resp.IsSuccessStatusCode) throw new Exception("Pago rechazado");
        } else if (dto.MedioPago == "transferencia") {
            // ...
        }

        // Persistir
        var ventaId = await conn.ExecuteScalarAsync<int>(
            "INSERT INTO ventas (cliente, total) VALUES (@c, @t); SELECT LAST_INSERT_ROWID();",
            new { c = dto.ClienteId, t = total });
        foreach (var item in dto.Items)
            await conn.ExecuteAsync("INSERT INTO venta_items ...", item);

        // Notificar
        var smtp = new SmtpClient("smtp.empresa.com");
        smtp.Send("ventas@empresa.com", clienteRow.email,
                  "Su compra", $"Total: ${total}");

        // Auditar
        logger.LogInformation("Venta {Id} procesada por {Total}", ventaId, total);
        await File.AppendAllTextAsync("audit.log",
            $"{DateTime.UtcNow:o}|venta|{ventaId}|{total}\n");

        return ventaId;
    }
}
```

Inventario de responsabilidades — y los actores asociados:

1. **Validación**: equipo de Producto / Ventas.
2. **Política de descuento**: equipo de Marketing.
3. **Pago**: equipo de Finanzas / Plataforma de pagos.
4. **Persistencia**: equipo de Datos / Infraestructura.
5. **Notificación al cliente**: equipo de Customer Experience.
6. **Auditoría / logging**: equipo de Compliance / Observabilidad.

Seis actores, seis razones distintas para que esta clase cambie. Cada cambio arriesga regresión en lo que ya funciona.

### 10.2 Descomposición patrón por patrón

**Descuento → Strategy**:

```csharp
public interface IPoliticaDescuento {
    bool    AplicaA(Cliente c);
    decimal Calcular(decimal subtotal);
}

public class DescuentoVip      : IPoliticaDescuento { /* 15% si tipo=="vip" */ }
public class DescuentoPremium  : IPoliticaDescuento { /* 10% si tipo=="premium" */ }
public class DescuentoRegular  : IPoliticaDescuento { /*  5% si tipo=="regular" */ }
public class SinDescuento      : IPoliticaDescuento { /*  0% si no aplica otra */ }
```

**Pago → Strategy + Factory**:

```csharp
public interface IGatewayPago {
    Task<ResultadoPago> CobrarAsync(decimal monto, CancellationToken ct);
}

public class GatewayTarjeta       : IGatewayPago { /* HTTP a api.tarjetas.com */ }
public class GatewayTransferencia : IGatewayPago { /* ... */ }

public interface IFabricaGateways {
    IGatewayPago Crear(string medio);
}
```

**Persistencia → Repository**:

```csharp
public interface IRepositorioVentas {
    Task<int> AgregarAsync(Venta v, CancellationToken ct);
}

public class RepositorioVentasSql : IRepositorioVentas { /* INSERTs vía Dapper */ }
```

**Notificación → Adapter (sobre SMTP)**:

```csharp
public interface INotificadorCliente {
    Task EnviarAsync(string destino, string asunto, string cuerpo, CancellationToken ct);
}

public class NotificadorEmail : INotificadorCliente { /* adapter sobre SmtpClient */ }
```

**Auditoría → Decorator (sobre cualquier servicio)**:

```csharp
public class ServicioVentasConAuditoria(
    IServicioVentas inner,
    ILogger<ServicioVentasConAuditoria> log,
    IAuditor auditor) : IServicioVentas {

    public async Task<int> ProcesarVenta(VentaDto dto, CancellationToken ct) {
        var id = await inner.ProcesarVenta(dto, ct);
        log.LogInformation("Venta {Id} procesada", id);
        await auditor.RegistrarAsync("venta", id, ct);
        return id;
    }
}
```

**Validación → un único validator** (o Chain of Responsibility si crece):

```csharp
public interface IValidador<T> {
    Resultado Validar(T input);
}

public class ValidadorVenta : IValidador<VentaDto> { /* ... */ }
```

### 10.3 La coordinación final

```csharp
public class ServicioVentas(
    IValidador<VentaDto>            validador,
    IRepositorioClientes            repoClientes,
    IEnumerable<IPoliticaDescuento> politicas,
    IFabricaGateways                fabricaGateways,
    IRepositorioVentas              repoVentas,
    INotificadorCliente             notificador) : IServicioVentas {

    public async Task<int> ProcesarVenta(VentaDto dto, CancellationToken ct) {
        validador.Validar(dto).LanzarSiInvalido();

        var cliente   = await repoClientes.ObtenerAsync(dto.ClienteId, ct);
        var subtotal  = dto.Items.Sum(i => i.Precio * i.Cantidad);
        var politica  = politicas.First(p => p.AplicaA(cliente));
        var total     = subtotal - politica.Calcular(subtotal);

        var gateway = fabricaGateways.Crear(dto.MedioPago);
        await gateway.CobrarAsync(total, ct);

        var venta = Venta.From(dto, total);
        var id    = await repoVentas.AgregarAsync(venta, ct);

        await notificador.EnviarAsync(cliente.Email, "Su compra",
                                      $"Total: ${total}", ct);
        return id;
    }
}
```

Y la auditoría se compone como decorator en el registro de DI:

```csharp
builder.Services.AddScoped<IServicioVentas>(sp => {
    var inner = ActivatorUtilities.CreateInstance<ServicioVentas>(sp);
    return new ServicioVentasConAuditoria(inner,
        sp.GetRequiredService<ILogger<ServicioVentasConAuditoria>>(),
        sp.GetRequiredService<IAuditor>());
});
```

### 10.4 El resultado en términos de SRP

| Clase                         | Responsabilidad única                    | Cambia si...                            |
|-------------------------------|------------------------------------------|-----------------------------------------|
| `ServicioVentas`              | Orquestar el caso de uso                 | Cambia el flujo del proceso             |
| `ValidadorVenta`              | Reglas de validación de DTO              | Cambian las reglas de validez           |
| `IPoliticaDescuento`          | Una política de descuento                | Cambia esa política                     |
| `IGatewayPago`                | Cobrar por un medio                      | Cambia ese medio                        |
| `IFabricaGateways`            | Decidir qué gateway crear                | Aparece un nuevo medio                  |
| `RepositorioVentasSql`        | Persistencia de ventas                   | Cambia el esquema o el motor            |
| `NotificadorEmail`            | Adaptar SMTP a nuestra abstracción       | Cambia el SDK / proveedor               |
| `ServicioVentasConAuditoria`  | Logging y auditoría                      | Cambian las políticas de observabilidad |

Cada clase responde a un único actor. Las pruebas son chicas, mockeables sin esfuerzo. Agregar un nuevo medio de pago, una nueva política de descuento, o cambiar el proveedor de email son operaciones aditivas.

## 11. Síntesis

### 11.1 Heurísticas: cuándo aplicar cada cosa

Una guía rápida de diagnóstico → patrón:

| Si veo...                                                                          | Probablemente necesite... |
|------------------------------------------------------------------------------------|---------------------------|
| `switch` o `if-else` enumerando alternativas con lógica sustancial en cada rama    | Strategy                  |
| Misma estructura de método repetida con variaciones puntuales                      | Template Method           |
| `if` sobre un campo `estado` o `tipo` repetido en varios métodos                   | State                     |
| Operaciones que se invocan y quiero loguear, deshacer, encolar o programar         | Command                   |
| Cadena de validaciones / filtros con `if`s consecutivos                            | Chain of Responsibility   |
| `new` distinto según un parámetro, dentro del código de negocio                    | Factory                   |
| Constructores con muchos parámetros, varios opcionales                             | Builder                   |
| Lógica de orquestación entre varios servicios que se conocen mutuamente            | Mediator                  |
| Una clase que avisa a varias después de hacer algo                                 | Observer                  |
| El mismo bloque que enlaza varios objetos, replicado en muchos lugares             | Facade                    |
| Clases de dominio con métodos `Save`/`Load`                                        | Repository                |
| Métodos de búsqueda combinatorios que crecen sin parar                             | Specification             |
| SDK externo invocado directamente desde la lógica de negocio                       | Adapter                   |
| Logging / caché / retry / métricas mezclados con el código de negocio              | Decorator                 |
| Lógica de "está cargado / puedo acceder" antes de cada uso                         | Proxy                     |
| Cadena de responsabilidades HTTP sin estructura clara                              | Middleware Pipeline       |
| Lógica transversal repetida en muchos controladores                                | Filters                   |
| Configuración leída con strings mágicos en muchas clases                           | Options Pattern           |
| Entidades de dominio expuestas tal cual por una API                                | DTO + Mapper              |
| Code-behind de UI con lógica de negocio adentro                                    | MVVM                      |

---

### 11.2 Cómo se entrelazan los principios y los patrones

Los patrones de extracción para SRP suelen colaborar con otros principios — y eso no es accidente:

- **OCP** se obtiene casi gratis cuando se aplica Strategy, Decorator o Chain of Responsibility: agregar una nueva variante es agregar una clase, no modificar las existentes.
- **DIP** aparece cuando el patrón usa interfaces (Strategy, Repository, Adapter) e inyecta las implementaciones: el código de alto nivel queda dependiendo de abstracciones.
- **ISP** guía a hacer las interfaces de los patrones lo más chicas posibles: un `IComando` con `Ejecutar` y `Deshacer`, un `IPoliticaDescuento` con dos métodos. Las interfaces de los patrones tienden a ser pequeñas por naturaleza.
- **LSP** entra cuando uno escribe variantes de un patrón: las implementaciones de `IPoliticaDescuento` tienen que cumplir el mismo contrato — todas devuelven un descuento sobre el subtotal, ninguna lanza excepción inesperada.

Los patrones bien aplicados no satisfacen sólo SRP: tiran del resto de SOLID en la dirección correcta como efecto colateral. Esa es una de las razones por las que el catálogo de patrones es persistente: no son trucos aislados, son las **formas naturales que toma un diseño respetuoso de los principios**.

---

### 11.3 Reglas para no caer en sobrediseño

Los patrones tienen costo. Aplicarlos preventivamente o por gusto académico produce código pesado y poco legible.

1. **El patrón aparece después del diagnóstico**, nunca antes. Primero detectás SRP violado, después extraés, después reconocés que el resultado se llama Strategy o Repository.
2. **Si el refactor no se justifica por una violación real de SRP, no lo hagas**. Cuatro descuentos hardcodeados sin perspectiva de cambio están bien como un `switch`.
3. **Los nombres son secundarios**. Si extraés una clase y no es ningún patrón conocido, está bien igual.
4. **No mezcles patrones por moda**. Una clase no necesita ser Strategy + Decorator + Factory + Observer simultáneamente para ser "elegante". A menudo eso es síntoma de over-engineering.
5. **Releé después del refactor**. Si el código quedó más difícil de seguir que antes, deshacé el cambio. La meta es facilidad de cambio futuro, no purismo arquitectónico.
6. **Preferí nombres que revelen responsabilidad**. `JsonContactoRepository`, `WhatsAppNotificador`, `CsvExportador` revelan qué hace cada clase. `ContactoManager`, `AgendaHelper`, `Utils` muchas veces significan "no sé qué responsabilidad tiene esto, pero compila".
7. **No crees abstracciones antes de tiempo**. Una interfaz por cada clase concreta del sistema es ruido. La abstracción debe responder a una variación real o razonablemente probable.

### 11.4 Resumen final

```text
Los cinco principios SOLID:
    ├── SRP  → una clase, una razón clara para cambiar
    ├── OCP  → extender sin modificar
    ├── LSP  → subclases que reemplacen sin sorpresas
    ├── ISP  → muchas interfaces chicas, no una grande
    └── DIP  → depender de abstracciones, no de implementaciones

Cómo lucen en la BCL y en el stack moderno de .NET:
    ├── SRP  → FileStream/StreamReader/TextReader, List<T> vs Enumerable,
    │         DbContext, handlers de Minimal API, middleware
    ├── OCP  → jerarquía Stream, LINQ sobre IEnumerable<T>, IQueryable<T>,
    │         filtros globales, componentes Blazor
    ├── LSP  → IEnumerable<T>, IActionResult, RenderFragment,
    │         (con la concesión pragmática de Stream y CanRead/CanSeek)
    ├── ISP  → jerarquía de IEnumerable/IReadOnlyList/ICollection/IList,
    │         IDisposable vs IAsyncDisposable, IServiceCollection vs
    │         IApplicationBuilder vs IEndpointRouteBuilder
    └── DIP  → StreamReader(Stream), ILogger<T>, IHttpClientFactory,
              IServiceCollection, repositorio sobre EF Core

Patrones para extraer responsabilidades cuando SRP se viola:
    ├── Política / algoritmo  → Strategy, Template Method, State
    ├── Operación             → Command, Chain of Responsibility
    ├── Creación              → Factory, Builder
    ├── Coordinación          → Mediator, Observer, Facade
    ├── Persistencia          → Repository, Unit of Work, Specification
    ├── Adaptación            → Adapter, Decorator, Proxy
    └── Idiomáticos de .NET   → DI, Options, Middleware, Filters,
                                 DTO + Mapper, MVVM

Regla final:
    └─► El patrón es la geometría de una extracción que ya estaba motivada
        por SRP. No se aprenden patrones para aplicarlos: se aprenden
        para reconocerlos cuando aparecen, comunicarlos rápido, y no
        inventar variantes inferiores.
```

El buen diseño no hace ruido: simplemente deja de romperse por cualquier pavada. Cuando una clase tiene un trabajo reconocible, testeable y reemplazable, los cambios futuros son aditivos en vez de invasivos. Esa es la promesa de SOLID, y los patrones son las formas conocidas de cumplirla.
