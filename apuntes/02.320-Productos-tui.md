# Terminal.Gui: cómo se piensa una interfaz de usuario en la terminal

> Estos apuntes son sobre **diseñar interfaces** con **Terminal.Gui**, la librería de .NET para construir TUIs modernas. El ejemplo que recorremos es un editor de una pequeña base de productos (código, descripción, precio, existencias), pero el foco no está en el programa: está en la interfaz.
>
> Vamos a esbozar las ideas, los componentes y las decisiones de diseño. **No vamos a escribir un programa terminado**. La persistencia, las validaciones de negocio y la importación/exportación las mencionamos solo cuando sirven para explicar una decisión de UI.

El objetivo es que, después de leer estos apuntes, sepas pensar la pantalla antes de tipear código y reconozcas los bloques con los que se arma una TUI:

- ventanas, paneles y diálogos;
- menús desplegables y atajos de teclado;
- formularios con campos de texto, números y checks;
- listas tipadas y búsqueda activa;
- separación entre la UI y la lógica del dominio;
- mensajes al usuario, validación y foco.

---

## 1. La idea: una interfaz gráfica dentro de la terminal

Una aplicación de consola tradicional imprime texto y lee texto:

``` csharp
Console.Write("Código: ");
string codigo = Console.ReadLine() ?? "";
```

Para un programa chico alcanza. Pero cuando hay muchas opciones, listas, formularios, búsqueda, edición y navegación, esa forma se vuelve incómoda muy rápido.

Una **TUI** (*Terminal User Interface*) usa la terminal entera como superficie visual. En lugar de imprimir líneas una debajo de otra, dibuja ventanas, paneles, menús, botones, listas y formularios. La terminal sigue siendo texto, pero la experiencia deja de ser una secuencia lineal de preguntas y respuestas.

Terminal.Gui resuelve por nosotros las tareas difíciles:

- dibujar correctamente en la pantalla;
- redibujar cuando cambia el tamaño de la terminal;
- manejar el foco entre componentes;
- capturar teclado y mouse;
- organizar componentes visuales;
- abrir diálogos modales;
- evitar que tengamos que usar `Console.SetCursorPosition` a mano.

La idea central es que **una aplicación no "imprime pantallas"**. Construye un árbol de objetos visuales y Terminal.Gui se encarga de dibujarlo, mover el foco, recibir eventos y redibujar cuando haga falta:

``` text
Application
└── ProductsWindow
    ├── MenuBar
    ├── controles de búsqueda
    ├── FrameView "Productos"
    │   └── ListView<Producto>
    ├── FrameView "Detalle"
    │   └── Label
    └── Label de estado
```

Esto es lo primero que conviene internalizar: dejamos de pensar en "imprimo, leo, imprimo, leo" y empezamos a pensar en **un árbol de vistas**.

## 2. Filosofía de diseño

Terminal.Gui se parece más a WinForms, WPF o Avalonia que a `Console.WriteLine`.

### Todo lo visible es un `View`

Un botón, una etiqueta, una lista, una ventana y un diálogo son `View` o derivan de `View`. Eso significa que todos comparten los mismos conceptos:

- posición: `X`, `Y`;
- tamaño: `Width`, `Height`;
- padre visual: `SuperView`;
- hijos visuales: `SubViews`;
- foco;
- eventos;
- dibujo;
- layout.

### El layout se expresa, no se calcula a mano

En vez de calcular coordenadas con enteros, Terminal.Gui usa:

- `Pos` para posiciones;
- `Dim` para tamaños.

Eso permite decir cosas como "pone este campo una columna a la derecha del label" o "que ocupe el resto del ancho dejando dos columnas libres", sin tener que recalcular nada cuando la terminal cambie de tamaño.

### La aplicación es reactiva

No hay un `while` manual preguntando al usuario qué quiere hacer. Terminal.Gui ejecuta un bucle de eventos:

1. dibuja la interfaz;
2. espera una tecla, un click o un evento interno;
3. manda ese evento al componente con foco;
4. ejecuta los handlers;
5. redibuja lo que cambió.

Nuestro código casi no llama a nada activamente; **reacciona**. Se suscribe a eventos como `ValueChanged`, `Accepting`, `OpenSelectedItem`, y deja que el framework dispare la lógica cuando corresponda.

### El teclado es un ciudadano de primera clase

Una buena TUI se puede usar sin mouse. Terminal.Gui ofrece foco con Tab, menús con F10 o Alt+letra, hotkeys con `_`, atajos como `Ctrl+N`, eventos de teclado y comandos. Antes de agregar mouse, conviene que todo se pueda hacer con el teclado.

### La interfaz no debería contener la lógica de negocio

El botón "Guardar" no tiene que saber cómo se persiste un producto. El botón llama a un método de la ventana, y ese método delega en una clase que se ocupa de los datos.

En nuestro ejemplo separamos cuatro responsabilidades:

- `ProductsWindow` organiza la UI y coordina las acciones;
- `ProductDialog` edita un producto;
- un **store** (cualquiera: SQLite, archivos, memoria) persiste productos;
- `Producto` representa los datos del dominio.

Esa separación es lo que nos permite cambiar el almacenamiento sin tocar la pantalla, y al revés.

## 3. Crear el proyecto

Para clase conviene una app C# de archivo único con .NET 10. Las dependencias se declaran al principio del `.cs`. Para una TUI básica alcanza con:

``` csharp
#:package Terminal.Gui@2.0.1
#:property PublishAot=false
```

Los `using` más usados son:

``` csharp
using Terminal.Gui.App;       // Application, IApplication, Runnable
using Terminal.Gui.Drawing;   // LineStyle, colores y dibujo
using Terminal.Gui.Input;     // Key, Command
using Terminal.Gui.ViewBase;  // View, Pos, Dim
using Terminal.Gui.Views;     // Window, FrameView, Button, TextField, etc.

using System.Collections.ObjectModel;   // ObservableCollection
using System.Globalization;             // CultureInfo (formato de moneda/fecha)
```

Si más adelante el ejemplo necesita persistir en SQLite, JSON o lo que sea, eso se agrega aparte. No es parte de la TUI.

## 4. Ciclo de vida de la aplicación

El ciclo mínimo es:

``` csharp
using IApplication app = Application.Create().Init();
app.Run(new ProductsWindow(store));
```

Qué pasa ahí:

1. `Application.Create()` crea una instancia de aplicación.
2. `Init()` inicializa el driver de terminal, el loop principal y el manejo de input.
3. `Run(...)` ejecuta una pantalla, ventana o diálogo.
4. Cuando la pantalla pide detenerse, `Run` termina.
5. El `using` llama a `Dispose()` y restaura la terminal.

Para cerrar una ventana o diálogo se usa:

``` csharp
App!.RequestStop();
```

`RequestStop` no mata el proceso. Le pide al loop activo que termine de forma ordenada.

## 5. Pensar la pantalla antes de codear

Antes de tipear una línea, conviene dibujar la pantalla en papel o en un comentario ASCII. Para nuestro editor de productos imaginamos algo así:

``` text
┌────────────────────────────────────────────────────────────────────────────┐
│ Archivo  Productos  Ver                                                    │
├────────────────────────────────────────────────────────────────────────────┤
│ Buscar: [_______________]  [x] Solo con stock bajo                         │
├─────────────────────────────┬──────────────────────────────────────────────┤
│ Productos                   │ Detalle                                      │
│   P-0001  Lápiz HB negro    │  Código:        P-0003                       │
│   P-0002  Cuaderno A4 80h   │  Descripción:   Birome azul                  │
│ > P-0003  Birome azul       │  Precio:        $ 150.00                     │
│   P-0004  Goma de borrar    │  Existencias:   0 unidades                   │
│   P-0005  Tijera escolar    │  Estado:        Sin stock                    │
│   P-0006  Regla 30 cm       │  Activo:        Sí                           │
│   ...                       │                                              │
│                             │  Última modif:  14/05/2026 09:32             │
├─────────────────────────────┴──────────────────────────────────────────────┤
│ F2 Nuevo  F3 Editar  Del Eliminar  Ctrl+R Recargar  Ctrl+Q Salir           │
└────────────────────────────────────────────────────────────────────────────┘
```

Decisiones tomadas en este sketch:

- el **menú** vive arriba y agrupa acciones por categoría;
- la **búsqueda** está a la vista, no escondida en un submenú, porque es lo que más se usa;
- la **lista (maestro)** solo muestra código y descripción: lo justo para identificar y elegir;
- el **detalle** a la derecha muestra **todos** los datos del producto seleccionado, con formato amable (precio con símbolo de moneda, fecha legible, estado calculado);
- abajo, una **barra de ayuda** recuerda los atajos.

La razón pedagógica de ese reparto es importante: **la lista no es un reporte**. La lista sirve para *encontrar* un producto. El detalle sirve para *verlo entero*. Si metemos precio y stock en la lista, el usuario empieza a leer columnas en vez de buscar; y si encima los productos tienen 10 campos, la lista se vuelve ilegible. **Maestro escueto, detalle completo.**

Este sketch vale más que media hora de prueba y error con `Pos` y `Dim`. Una vez que sabemos qué queremos dibujar, el resto es traducción.

## 6. Vistas como organizador visual

Una `View` puede ser:

- un control concreto: `Label`, `TextField`, `Button`, `CheckBox`, `ListView`;
- un contenedor con borde y título: `FrameView`, `Window`, `Dialog`;
- una pantalla propia creada por nosotros (heredando de `Runnable`).

Las vistas se agregan a su contenedor con `Add`. Por ejemplo, el panel "Productos" tendrá adentro un `ListView`, no contra toda la pantalla sino contra el área interna del panel. Eso permite mover el panel y que la lista lo siga.

La estructura visual de nuestra pantalla refleja exactamente el sketch:

``` text
ProductsWindow
├── MenuBar
├── Label "Buscar:"
├── TextField searchField
├── CheckBox lowStockOnly
├── FrameView "Productos"
│   └── ListView<Producto> productsList
├── FrameView "Detalle"
│   └── Label detailsLabel
└── Label statusBar
```

Cuando esta estructura está clara, el código casi se escribe solo.

## 7. Runners y pantallas ejecutables

En Terminal.Gui no se ejecuta cualquier objeto: se ejecuta uno que participa del ciclo de vida de la app. Para eso existe `IRunnable`; la clase cómoda para empezar es `Runnable`.

Esquema típico:

``` csharp
public sealed class ProductsWindow : Runnable {
    public ProductsWindow(IProductStore store) {
        Title  = "Productos - Terminal.Gui";
        Width  = Dim.Fill();
        Height = Dim.Fill();

        BuildLayout();
        LoadProducts();
    }
}
```

Cuando una pantalla necesita abrir un diálogo:

``` csharp
ProductDialog dialog = new("Editar producto", selected.Clone());
App!.Run(dialog);

if (dialog.Saved && dialog.Product is not null) {
    // aplicar cambios
}
```

`App!.Run(dialog)` abre un loop modal encima del actual. Mientras el diálogo está abierto, la ventana de atrás no recibe input. Cuando el diálogo llama a `App!.RequestStop()`, el control vuelve a la línea siguiente.

## 8. Ventanas, paneles y diálogos

Conviene distinguir cuatro contenedores:

| Contenedor   | Cuándo usarlo                                                |
|--------------|--------------------------------------------------------------|
| `Runnable`   | Pantalla ejecutable que pasa a `app.Run(...)`. Nuestra raíz. |
| `Window`     | Ventana principal tradicional con borde y título.            |
| `FrameView`  | Panel con borde y título, dentro de una pantalla.            |
| `Dialog`     | Ventana modal para edición, confirmación o formularios.      |

En la pantalla de productos:

- `ProductsWindow` es un `Runnable` (raíz);
- los paneles "Productos" y "Detalle" son `FrameView`;
- el diálogo de edición es un `Dialog`;
- las confirmaciones de borrado las hace `MessageBox`, que internamente es otro `Dialog`.

La diferencia conceptual:

- `FrameView` **organiza** una pantalla;
- `Dialog` **interrumpe** la pantalla para pedir una decisión o datos;
- `Runnable`/`Window` es lo que corre en `app.Run(...)`.

## 9. Layout: ubicación con `Pos` y tamaño con `Dim`

La pantalla de terminal se mide en celdas de caracteres. Una coordenada no es un pixel; es una columna y una fila de texto.

Cada `View` tiene `X`, `Y`, `Width` y `Height`. La clave es entender que **no son enteros, son expresiones**.

### Posiciones absolutas

``` csharp
X = 1;
Y = 2;
```

Útil para cosas simples cerca del borde superior izquierdo.

### Posiciones relativas a otros views

``` csharp
X = Pos.Right(searchLabel) + 1;
Y = Pos.Top(searchLabel);
```

"A la derecha del label, en la misma fila". Esto es lo que se usa la mayor parte del tiempo en formularios.

### Debajo de otro view

``` csharp
Y = Pos.Bottom(priceLabel) + 1;
```

Muy útil para apilar campos.

### Anclar al final

``` csharp
Y = Pos.AnchorEnd(1);
```

La barra de estado va siempre en la última fila, sin importar el alto de la terminal.

### Tamaños

| Expresión                 | Significado                                         |
|---------------------------|-----------------------------------------------------|
| `Width = 30`              | Ancho fijo de 30 columnas.                          |
| `Width = Dim.Percent(60)` | El 60% del ancho del padre.                         |
| `Width = Dim.Fill()`      | Lo que sobre.                                       |
| `Width = Dim.Fill(2)`     | Lo que sobre, dejando 2 columnas libres al final.   |

La ventaja de este estilo es que si movemos un control, los demás se acomodan solos. Y si el usuario agranda la terminal, la pantalla se redibuja sin que tengamos que recalcular nada.

## 10. Componentes básicos de entrada

Para un editor de productos vamos a necesitar cinco tipos de componentes de entrada.

### `Label`

Texto fijo. Se usa para títulos de campos y mensajes cortos.

``` csharp
new Label { Text = "Código:", X = 2, Y = 1 };
```

### `TextField`

Entrada de una sola línea. Es la base de los campos de código y descripción.

``` csharp
TextField codeField = new() { X = 14, Y = 1, Width = 14 };
codeField.ValueChanged += (_, _) => /* validar formato */ ;
```

`ValueChanged` se dispara con cada cambio de texto. Para el campo `searchField` esto habilita la **búsqueda activa**: cada tecla refina la lista.

### Campos numéricos

Terminal.Gui 2.x ofrece un `NumericTextField` (o se puede usar un `TextField` validando manualmente). Para el dominio de productos hay dos campos numéricos clave:

- **Precio**: `decimal`, con dos decimales, no negativo.
- **Existencias**: `int`, no negativo.

Pedagógicamente vale la pena distinguirlos: el precio tiene parte fraccionaria, las existencias no. La UI debería:

- alinear los números **a la derecha**, no a la izquierda como un texto cualquiera;
- mostrar el precio con formato (`$ 120,00` o `120.00`, según corresponda);
- impedir tipear letras o signos no esperados.

### `Button`

Ejecuta una acción.

``` csharp
Button saveButton = new() { Text = "Guardar", IsDefault = true };
saveButton.Accepting += (_, e) => { Save(); e.Handled = true; };
```

`IsDefault = true` hace que Enter active ese botón cuando el diálogo está enfocado. Es la acción principal.

### `CheckBox`

Entrada booleana. En nuestro caso, para marcar "Solo con stock bajo" arriba, y "Activo" en el diálogo de edición.

``` csharp
CheckBox active = new() { Text = "_Activo", Value = CheckState.Checked };
```

El guion bajo marca la letra de atajo (Alt+A).

## 11. El patrón maestro-detalle

Llegamos al corazón de la pantalla. Dos paneles que se hablan:

- el **maestro** (panel izquierdo): una lista corta y escaneable. **Solo código y descripción.**
- el **detalle** (panel derecho): toda la información del producto seleccionado, con formato amable.

El usuario navega el maestro con las flechas; el detalle se actualiza solo.

### El modelo del producto

Vamos a tener algo así:

``` csharp
public sealed class Producto {
    public int      Id              { get; set; }
    public string   Codigo          { get; set; } = "";
    public string   Descripcion     { get; set; } = "";
    public decimal  Precio          { get; set; }
    public int      Existencias     { get; set; }
    public bool     Activo          { get; set; } = true;
    public DateTime UltimaModif     { get; set; } = DateTime.Now;

    public Producto Clone() => new() {
        Id          = this.Id,
        Codigo      = this.Codigo,
        Descripcion = this.Descripcion,
        Precio      = this.Precio,
        Existencias = this.Existencias,
        Activo      = this.Activo,
        UltimaModif = this.UltimaModif
    };

    // Solo código y descripción: este es el secreto del maestro escueto
    public override string ToString() {
        string cod  = (this.Codigo ?? "").PadRight(8);
        string desc = this.Descripcion ?? "";
        if (desc.Length > 40) {
            desc = desc.Substring(0, 39) + "…";
        }
        return $"{cod} {desc}";
    }
}
```

La decisión clave está en `ToString()`. **El maestro no es un reporte**. Si querés ver precio o stock, lo ves en el detalle. La lista queda limpia, los códigos se alinean visualmente, y el ojo escanea rápido.

### Los dos paneles

En el método `BuildLayout()` de la ventana, los paneles del maestro y el detalle se arman como dos `FrameView` lado a lado:

``` csharp
// Panel izquierdo: el maestro
FrameView listPanel = new() {
    Title  = "Productos",
    X      = 0,
    Y      = 4,                      // debajo del menú y la búsqueda
    Width  = Dim.Percent(42),        // ocupa menos: solo código + descripción
    Height = Dim.Fill(2)             // hasta arriba de la barra de estado
};

this.productsList = new ListView<Producto>() {
    X      = 0, Y = 0,
    Width  = Dim.Fill(),
    Height = Dim.Fill()
};

listPanel.Add(this.productsList);

// Panel derecho: el detalle
FrameView detailsPanel = new() {
    Title  = "Detalle",
    X      = Pos.Right(listPanel),   // pegado al maestro
    Y      = Pos.Top(listPanel),
    Width  = Dim.Fill(),
    Height = Dim.Fill(2)
};

this.detailsLabel = new Label() {
    X      = 1, Y = 0,
    Width  = Dim.Fill(1),
    Height = Dim.Fill()
};

detailsPanel.Add(this.detailsLabel);

Add(listPanel, detailsPanel);
```

Dos cosas que merecen atención:

- `Pos.Right(listPanel)` posiciona el panel derecho **a la derecha del izquierdo, sea cual sea el ancho del izquierdo**. Si mañana cambiamos `Dim.Percent(42)` por `Dim.Percent(35)`, el detalle se reacomoda solo.
- el `Label` del detalle puede contener varias líneas separadas por `\n`. No hace falta un control especial para mostrar varios campos.

### La sincronización: el evento que las une

Para que el detalle se actualice cuando el usuario navega la lista, nos suscribimos al evento de cambio de selección del `ListView<Producto>`:

``` csharp
this.productsList.SelectedItemChanged += (_, _) => RefreshDetails();
```

Ese `RefreshDetails()` es el que escribe el contenido del panel derecho. Lo escribimos completo porque es el ejemplo concreto que pediste:

``` csharp
private void RefreshDetails() {
    Producto? selected = this.productsList.Value;

    if (selected is null) {
        this.detailsLabel.Text = "  (sin selección)";
        return;
    }

    string estado = (selected.Existencias, selected.Activo) switch {
        (_, false)          => "Producto inactivo",
        (0, _)              => "Sin stock",
        ( < 5, _)           => "Stock bajo",
        _                   => "OK"
    };

    string activo = selected.Activo ? "Sí" : "No";

    // Cultura argentina para el formato de moneda y fecha
    CultureInfo ar = CultureInfo.GetCultureInfo("es-AR");

    this.detailsLabel.Text =
        $"""
          Código:        {selected.Codigo}
          Descripción:   {selected.Descripcion}

          Precio:        {selected.Precio.ToString("C2", ar)}
          Existencias:   {selected.Existencias}} unidade
          Estado:        {estado}

          Activo:        {activo}
          Última modif:  {selected.UltimaModif.ToString("dd/MM/yyyy HH:mm", ar)}
        """;
}
```

Los puntos pedagógicos:

- **`switch` por patrones** para calcular el estado. Es un caso clásico donde C# moderno gana legibilidad: el orden de los casos importa (primero `inactivo`, después `sin stock`, después `stock bajo`, después `OK`) y leer las reglas como una tabla es más claro que cinco `if`.
- **Formato de moneda con cultura**: `Precio.ToString("C2", ar)` da `$ 150,00`. Si dejamos la cultura por defecto en una terminal angloparlante, sale `$150.00` o algo más raro. Para apps argentinas, vale la pena fijarla.
- **Espacios en blanco al inicio** (`"  Código: ..."`): el `Label` no tiene padding propio. Dos espacios manuales evitan que el texto se pegue al borde del panel.
- **Líneas en blanco** entre grupos lógicos: identificación, datos comerciales, metadatos. Eso hace que el detalle se lea como una ficha, no como un párrafo.

### El flujo completo cuando el usuario navega

Vale la pena seguirlo mentalmente:

1. el usuario aprieta la flecha abajo;
2. el `ListView<Producto>` mueve el cursor a la siguiente fila;
3. internamente, dispara `SelectedItemChanged`;
4. nuestro handler llama a `RefreshDetails()`;
5. `RefreshDetails()` lee `productsList.Value` y vuelve a armar el texto del label de detalle;
6. Terminal.Gui redibuja el panel derecho.

Y todo eso sin un solo `while` ni un `Console.ReadKey`. Esa es la diferencia entre programar para consola y programar para una TUI.

### Una nota sobre `SetSource`

Cuando recargamos productos, hay que pasarle la colección al `ListView`. La forma idiomática en Terminal.Gui 2.x es con `SetSource` sobre una colección observable:

``` csharp
this.productsList.SetSource(
    new ObservableCollection<Producto>(this.filteredProducts));
```

Eso garantiza que cuando la fuente cambia, el control se redibuja. Después de cualquier filtro, recarga o edición, llamamos a este `SetSource` con la lista filtrada actual y el `ListView` se actualiza.

## 12. Búsqueda en caliente

"Búsqueda en caliente" o "búsqueda activa" significa que **no hay botón Buscar**: el usuario tipea en un campo y la lista se filtra con cada tecla. Es la forma estándar en TUIs modernas y, una vez que se prueba, se vuelve adictiva.

Para implementarla bien hacen falta tres ingredientes: dos listas en memoria, un handler de cambio de texto y una rutina de refresco que respete la selección.

### Dos listas en memoria

El primer instinto sería filtrar la lista original, pero eso es un error: si filtramos y después borramos el texto del campo de búsqueda, ya no tenemos los datos originales. La solución es mantener dos colecciones:

``` csharp
private List<Producto> products         = new();   // todos los productos
private List<Producto> filteredProducts = new();   // los visibles ahora mismo
```

`products` se modifica al insertar, editar o eliminar (y al recargar desde el store). `filteredProducts` se recalcula cada vez que cambia el filtro. **El `ListView` siempre se alimenta de `filteredProducts`**, nunca de `products` directamente.

### Cargar productos desde el store

La carga inicial llena `products` y dispara el primer refresco:

``` csharp
private void LoadProducts() {
    this.products = this.store.GetAll();
    RefreshProducts();
}
```

### Suscribirse al campo de búsqueda

Suponiendo que en `BuildLayout()` creamos así el campo de búsqueda:

``` csharp
Label searchLabel = new() {
    Text = "Buscar:",
    X    = 1, Y = 2
};

this.searchField = new TextField() {
    X     = Pos.Right(searchLabel) + 1,
    Y     = Pos.Top(searchLabel),
    Width = Dim.Percent(40)
};

this.lowStockOnly = new CheckBox() {
    Text  = "Solo con _stock bajo",
    X     = Pos.Right(this.searchField) + 2,
    Y     = Pos.Top(this.searchField),
    Value = CheckState.UnChecked
};

Add(searchLabel, this.searchField, this.lowStockOnly);
```

… el cableado de la búsqueda en caliente es de dos líneas:

``` csharp
this.searchField.TextChanged          += (_, _) => RefreshProducts();
this.lowStockOnly.CheckedStateChanged += (_, _) => RefreshProducts();
```

`TextChanged` se dispara con cada cambio de texto en el campo. No esperamos a que el usuario apriete Enter. No hay timer. No hay debounce. Cada tecla refresca.

> **Nota.** En Terminal.Gui 2.x el evento se llama `TextChanged`. Versiones anteriores usaron `ValueChanged`. Si encontrás un nombre y no funciona, probá el otro.

### El predicado de filtrado

Antes del refresco, definimos qué significa "coincide con la búsqueda". Lo aislamos en un método chico para que después se pueda extender:

``` csharp
private static bool Matches(Producto product, string search, bool onlyLowStock) {
    if (onlyLowStock && product.Existencias > 5) {
        return false;
    }

    if (string.IsNullOrWhiteSpace(search)) {
        return true;
    }

    search = search.Trim().ToLowerInvariant();

    return product.Codigo.ToLowerInvariant().Contains(search)
        || product.Descripcion.ToLowerInvariant().Contains(search);
}
```

Tres detalles deliberados:

- **`ToLowerInvariant`** en ambos lados: la búsqueda es **case-insensitive**, como espera cualquier usuario.
- **`Contains`, no `StartsWith`**: el usuario puede tipear `"birome"` o `"azul"` y encontrar "Birome azul". Buscar por prefijo se siente mezquino.
- **El filtro de stock se evalúa primero**: si está activo y el producto no califica, ni hace falta mirar el texto.

### El refresco que respeta la selección

Acá está el método que junta todo. Es probablemente el método más cargado de la ventana, pero cada línea cumple un propósito:

``` csharp
private void RefreshProducts(int? selectedId = null) {
    // 1. Recordar qué producto estaba seleccionado, si no nos lo pasaron
    selectedId ??= this.productsList.Value?.Id;

    // 2. Leer el estado actual del filtro
    string search       = this.searchField.Text?.ToString()?.Trim() ?? "";
    bool   onlyLowStock = this.lowStockOnly.Value == CheckState.Checked;

    // 3. Recalcular la lista filtrada
    this.filteredProducts = this.products
        .Where(p => Matches(p, search, onlyLowStock))
        .OrderBy(p => p.Codigo)
        .ToList();

    // 4. Pasarle al ListView la nueva fuente
    this.productsList.SetSource(
        new ObservableCollection<Producto>(this.filteredProducts));

    // 5. Reposicionar el cursor sobre el producto que estaba seleccionado
    int index;
    if (this.filteredProducts.Count == 0) {
        index = -1;
    } else if (selectedId is null) {
        index = 0;
    } else {
        int found = this.filteredProducts.FindIndex(p => p.Id == selectedId.Value);
        index = found >= 0 ? found : 0;
    }
    this.productsList.Index = index;

    // 6. Forzar el refresco del detalle (por si cambió el seleccionado)
    RefreshDetails();

    // 7. Mensaje para el usuario
    SetStatus($"{this.filteredProducts.Count} de {this.products.Count} productos.");
}
```

El paso 5 es **lo que evita la sensación errática**. Sin él, cada vez que el usuario tipea una letra el cursor salta al primer producto. Con él, el cursor se queda sobre el producto que el usuario estaba mirando, **mientras siga apareciendo en los resultados**. Si deja de aparecer, el cursor cae elegantemente al primero.

### Lo que el usuario ve

Imaginemos que en el maestro hay 50 productos y el usuario tipea, letra por letra, `b-i-r`:

| Tecla | Filtro | Productos visibles                          |
|-------|--------|---------------------------------------------|
|(vacío)| `""`   | 50 productos                                |
| `b`   | `"b"`  | 12 (los que tienen una "b" en algún lado)   |
| `bi`  | `"bi"` | 4                                           |
| `bir` | `"bir"`| 2 (Birome azul, Birome negra)               |

Y todo esto sucede **mientras tipea**, sin pulsar Enter, sin abrir un diálogo, sin clickear nada. Esa es la sensación que vale la pena entregarle al usuario.

### Variantes que conviene conocer

- **Búsqueda con debounce**: en aplicaciones donde cada filtro cuesta caro (porque pega a una base remota), conviene esperar 200-300 ms desde la última tecla antes de filtrar. En nuestro ejemplo, con los datos en memoria, es innecesario.
- **Resaltar coincidencias**: pintar en otro color las letras que matchean dentro de la fila. Vistoso, pero requiere dibujar la fila a mano, en vez de delegar en `ToString()`.
- **Búsqueda por más campos**: si el día de mañana se quiere buscar también por proveedor, basta con extender `Matches` con un `||` más. La rutina de refresco no cambia.

## 13. Cajas de diálogo para edición

Una caja de diálogo de edición tiene tres responsabilidades, ni una más:

1. **mostrar campos** con los valores actuales;
2. **validar** lo que el usuario tipeó;
3. **devolver un resultado** (guardó o canceló).

Lo que **no** hace:

- no toca la base de datos;
- no modifica la lista global;
- no decide qué hacer con el resultado.

La ventana principal es la que decide qué pasa con el producto editado. Esta separación es lo que hace posible que el botón "Cancelar" funcione: no hay nada para deshacer porque nada se persistió todavía.

### El sketch del diálogo

Antes de codear, dibujamos:

``` text
┌─ Editar producto ───────────────────────────────────┐
│                                                     │
│  Código:        [P-0003___________]                 │
│  Descripción:   [Birome azul______________]         │
│                                                     │
│  Precio:        [        150.00]                    │
│  Existencias:   [             0]                    │
│                                                     │
│  [x] Activo                                         │
│                                                     │
│                            [ Guardar ] [ Cancelar ] │
└─────────────────────────────────────────────────────┘
```

Decisiones:

- los campos se apilan de arriba abajo, con líneas en blanco para agrupar (identificación, datos comerciales, metadatos);
- los **labels van alineados a la izquierda** y los campos en una columna alineada, para que el ojo siga una sola línea vertical;
- los **botones van abajo a la derecha**, "Guardar" como acción principal y "Cancelar" al lado;
- Enter activa "Guardar" (es el botón por defecto), Escape activa "Cancelar".

### El esqueleto de la clase

``` csharp
public sealed class ProductDialog : Dialog {
    private readonly TextField codeField;
    private readonly TextField descriptionField;
    private readonly TextField priceField;
    private readonly TextField stockField;
    private readonly CheckBox  activeField;

    // Resultado que la ventana principal va a leer
    public bool      Saved   { get; private set; }
    public Producto? Product { get; private set; }

    public ProductDialog(string title, Producto product) {
        this.Product = product;

        this.Title   = title;
        this.Width   = 60;
        this.Height  = 14;

        // ... crear campos, botones, suscribir eventos ...
    }
}
```

Notar dos cosas:

- el diálogo **recibe el producto desde afuera**, no lo crea él. Esto es lo que permite reutilizarlo para Nuevo (producto vacío) y para Editar (producto clonado).
- `Saved` y `Product` son las dos propiedades que la ventana principal va a consultar al cerrarse el diálogo. Es la única forma de comunicación de adentro hacia afuera.

### Armar el layout dentro del constructor

Acá viene la parte más larga, pero es directa: una pareja `Label` + `TextField` por cada campo, posicionado en relación al anterior con `Pos.Bottom`. Todos los campos arrancan en la misma columna `X` para que se vean alineados.

``` csharp
// Campo: Código
Label codeLabel = new() {
    Text = "Código:",
    X    = 2, Y = 1
};

this.codeField = new TextField() {
    Text  = product.Codigo,
    X     = 18,                           // columna fija para alinear
    Y     = Pos.Top(codeLabel),
    Width = 12                            // los códigos son cortos
};

// Campo: Descripción
Label descLabel = new() {
    Text = "Descripción:",
    X    = Pos.Left(codeLabel),
    Y    = Pos.Bottom(codeLabel) + 1
};

this.descriptionField = new TextField() {
    Text  = product.Descripcion,
    X     = Pos.Left(this.codeField),
    Y     = Pos.Top(descLabel),
    Width = Dim.Fill(2)                   // hasta el borde del diálogo
};

// Campo: Precio
Label priceLabel = new() {
    Text = "Precio:",
    X    = Pos.Left(codeLabel),
    Y    = Pos.Bottom(descLabel) + 2      // un renglón en blanco para separar
};

CultureInfo ar = CultureInfo.GetCultureInfo("es-AR");

this.priceField = new TextField() {
    Text  = product.Precio.ToString("F2", ar),    // formato 0.00
    X     = Pos.Left(this.codeField),
    Y     = Pos.Top(priceLabel),
    Width = 14
};

// Campo: Existencias
Label stockLabel = new() {
    Text = "Existencias:",
    X    = Pos.Left(codeLabel),
    Y    = Pos.Bottom(priceLabel) + 1
};

this.stockField = new TextField() {
    Text  = product.Existencias.ToString(),
    X     = Pos.Left(this.codeField),
    Y     = Pos.Top(stockLabel),
    Width = 14
};

// CheckBox: Activo
this.activeField = new CheckBox() {
    Text  = "_Activo",
    X     = Pos.Left(codeLabel),
    Y     = Pos.Bottom(stockLabel) + 2,
    Value = product.Activo ? CheckState.Checked : CheckState.UnChecked
};

Add(codeLabel, this.codeField,
    descLabel, this.descriptionField,
    priceLabel, this.priceField,
    stockLabel, this.stockField,
    this.activeField);
```

Hay varios patrones interesantes acá:

- **`Pos.Left(codeLabel)`** en todos los labels hace que la columna izquierda se alinee sola. Si cambiamos `X = 2` por `X = 3`, los seis labels se mueven juntos.
- **`Pos.Left(this.codeField)`** en todos los campos hace lo mismo con la columna de la derecha. Una sola decisión de diseño, replicada.
- **`Pos.Bottom(...) + 1`** apila los campos uno bajo el otro. `+2` cuando queremos una línea en blanco visible (separa grupos).
- **`Width = Dim.Fill(2)`** en la descripción la hace flexible hasta el borde derecho, mientras que `Width = 14` deja los campos numéricos cortos. **Eso es comunicación visual**: el ancho del campo le sugiere al usuario qué se espera adentro.
- **El precio se inicializa formateado** con `ToString("F2", ar)`: si el producto vale 150, el campo arranca con `"150,00"`, no con `"150"`. Pequeño, pero da sensación de pulcritud.

### Los botones

Un `Dialog` tiene un método `AddButton` que ubica los botones en la fila inferior automáticamente. No nos preocupamos por las coordenadas:

``` csharp
Button saveButton = new() {
    Text      = "_Guardar",
    IsDefault = true                      // Enter dispara este botón
};

saveButton.Accepting += (_, e) => {
    Save();                               // método que valida y cierra
    e.Handled = true;
};

Button cancelButton = new() {
    Text = "_Cancelar"
};

cancelButton.Accepting += (_, e) => {
    this.Saved = false;
    App!.RequestStop();                   // cerrar sin guardar nada
    e.Handled = true;
};

AddButton(saveButton);
AddButton(cancelButton);
```

Tres cosas para subrayar:

- **`IsDefault = true`** en Guardar hace que Enter, desde cualquier campo del diálogo, dispare el guardado. Es lo que el usuario espera de un formulario.
- **El guion bajo** (`"_Guardar"`, `"_Cancelar"`) marca la letra del atajo: Alt+G y Alt+C.
- **`e.Handled = true`** corta la propagación del evento. Sin eso, ciertas situaciones disparan el handler dos veces.

### El método `Save` con validación

Acá vive toda la lógica del diálogo. Es el único método largo de la clase, pero su estructura es la misma para cualquier formulario: validar campo por campo, mostrar error si falla, llenar el modelo si todo está bien.

``` csharp
private void Save() {
    // 1. Limpiar y validar el código
    string code = (this.codeField.Text?.ToString() ?? "").Trim();
    if (string.IsNullOrWhiteSpace(code)) {
        MessageBox.ErrorQuery(App!, "Validación",
            "El código es obligatorio.", "OK");
        this.codeField.SetFocus();
        return;
    }

    // 2. Limpiar y validar la descripción
    string desc = (this.descriptionField.Text?.ToString() ?? "").Trim();
    if (string.IsNullOrWhiteSpace(desc)) {
        MessageBox.ErrorQuery(App!, "Validación",
            "La descripción es obligatoria.", "OK");
        this.descriptionField.SetFocus();
        return;
    }

    // 3. Parsear y validar el precio (decimal, no negativo)
    CultureInfo ar = CultureInfo.GetCultureInfo("es-AR");
    string priceText = (this.priceField.Text?.ToString() ?? "").Trim();
    if (!decimal.TryParse(priceText, NumberStyles.Number, ar, out decimal price)
        || price < 0) {
        MessageBox.ErrorQuery(App!, "Validación",
            "El precio debe ser un número mayor o igual a cero.", "OK");
        this.priceField.SetFocus();
        return;
    }

    // 4. Parsear y validar las existencias (entero, no negativo)
    string stockText = (this.stockField.Text?.ToString() ?? "").Trim();
    if (!int.TryParse(stockText, out int stock) || stock < 0) {
        MessageBox.ErrorQuery(App!, "Validación",
            "Las existencias deben ser un entero mayor o igual a cero.", "OK");
        this.stockField.SetFocus();
        return;
    }

    // 5. Todo OK: volcar al modelo
    this.Product!.Codigo      = code;
    this.Product.Descripcion  = desc;
    this.Product.Precio       = price;
    this.Product.Existencias  = stock;
    this.Product.Activo       = this.activeField.Value == CheckState.Checked;
    this.Product.UltimaModif  = DateTime.Now;

    this.Saved = true;
    App!.RequestStop();
}
```

Los puntos pedagógicos:

- **El patrón "validar - mostrar - enfocar - return"** se repite para cada campo. Es deliberadamente verboso: cada bloque es independiente, fácil de leer y fácil de modificar. No conviene meterlo en un loop sobre un array de validadores hasta que haya 15 campos.
- **`decimal.TryParse` con `CultureInfo` argentina**: acepta `"150,00"` como precio. Si usáramos la cultura por defecto, en una máquina con locale en inglés `"150,00"` se rechazaría. Es una decisión de UX: queremos que el usuario tipee como tipea normalmente.
- **`int.TryParse` para las existencias**: rechaza `"0.5"` o `"abc"` sin necesidad de un try/catch. `TryParse` siempre es mejor que `Parse` en código de UI.
- **Mantener el foco**: después de mostrar el error, `SetFocus()` lleva el cursor al campo culpable. Esto es lo que separa un formulario amable de uno que castiga al usuario.
- **`UltimaModif = DateTime.Now`** se setea acá, en el lugar único donde se acepta un cambio. Así nunca queda desincronizado con la edición real.

### Edición sobre una copia

Cuando abrimos el diálogo para editar un producto existente, le pasamos una **copia** del original, no el original:

``` csharp
ProductDialog dialog = new("Editar producto", selected.Clone());
App!.Run(dialog);

if (dialog.Saved && dialog.Product is not null) {
    // recién acá decidimos qué hacer con el producto editado
    this.store.Update(dialog.Product);
    int index = this.products.FindIndex(p => p.Id == dialog.Product.Id);
    this.products[index] = dialog.Product;
    SetStatus($"Producto '{dialog.Product.Codigo}' actualizado.");
    RefreshProducts(dialog.Product.Id);
}
```

Si el usuario edita y cancela, el original queda intacto sin necesidad de "deshacer" nada: la copia se descarta y listo. Si el usuario guarda, recién entonces la ventana principal:

1. persiste con el store (primero la base);
2. reemplaza el objeto en la lista en memoria (después la memoria);
3. da feedback en la barra de estado;
4. refresca la lista pidiendo que mantenga seleccionado el producto recién editado.

El **orden importa**: si la base falla (por ejemplo, código duplicado), no queremos haber actualizado la memoria. Y si todo sale bien, el refresco final asegura que la fila del producto editado se vea con sus datos nuevos.

### El mismo diálogo, dos usos

El mismo `ProductDialog` sirve para **crear** y para **editar**. La única diferencia está en lo que la ventana hace después:

``` csharp
private void NewProduct() {
    Producto product = new();                       // sin Id, sin datos

    ProductDialog dialog = new("Nuevo producto", product);
    App!.Run(dialog);

    if (!dialog.Saved || dialog.Product is null) {
        return;
    }

    int newId = this.store.Insert(dialog.Product);  // primero la base
    dialog.Product.Id = newId;
    this.products.Add(dialog.Product);              // después la memoria
    SetStatus($"Producto '{dialog.Product.Codigo}' creado.");
    RefreshProducts(newId);
}
```

Es el mismo diálogo, la misma validación, la misma experiencia visual. Cambia solo el título y lo que se hace al cerrarse: `Insert` para Nuevo, `Update` para Editar. Esa simetría es lo que hace que la UI se sienta consistente: el usuario aprende a editar y ya sabe crear.

Un menú superior se arma con tres clases:

- `MenuBar`: la barra completa arriba;
- `MenuBarItem`: cada menú principal (Archivo, Productos, Ver);
- `MenuItem`: cada acción dentro del menú.

Para nuestro editor de productos:

``` csharp
MenuBar menu = new() {
```

## 15. Manejo de teclado

Terminal.Gui ya maneja muchas teclas por nosotros:

- Tab cambia el foco;
- las flechas navegan la lista;
- Enter acepta botones o abre el ítem seleccionado;
- F10 activa el menú;
- Escape cierra menús o diálogos.

Para los atajos propios sobreescribimos `OnKeyDown` en la ventana:

``` csharp
protected override bool OnKeyDown(Key key) {
    bool Handle(Action action) {
        action();
        return true;
    }

    return key switch {
        var k when k == Key.F2          => Handle(NewProduct),
        var k when k == Key.F3          => Handle(EditSelectedProduct),
        var k when k == Key.Delete      => Handle(DeleteSelectedProduct),
        var k when k == Key.R.WithCtrl  => Handle(ReloadProducts),
        _ => base.OnKeyDown(key)
    };
}
```

- `return true` significa "esta tecla ya fue manejada, no la sigas propagando";
- `return base.OnKeyDown(key)` permite que Terminal.Gui haga lo suyo (mover foco, scrollear, etc.).

Atajos sugeridos para el editor de productos:

``` text
F2              Nuevo producto
F3              Editar producto
Enter           Editar, si la lista tiene foco
Del             Eliminar producto
Ctrl+R          Recargar
Ctrl+Q          Salir
F4              Llevar foco al campo de búsqueda
```

Una buena TUI **se puede usar entera sin tocar el mouse**. Si al usar la app vos misma sentís que tenés que estirar la mano al mouse, falta un atajo.

## 16. Mensajes al usuario

Tres canales para hablarle al usuario, cada uno con su propósito:

### Barra de estado

Un `Label` en `Y = Pos.AnchorEnd(1)` que muestra un mensaje corto: "Producto 'Lápiz HB' creado.", "5 productos cargados.", "Sin resultados.".

Es la forma menos invasiva de dar feedback. No interrumpe, no requiere clickear OK. Se usa para resultados de operaciones que salieron bien.

### `MessageBox.Query`

Para confirmaciones y decisiones binarias:

``` csharp
int answer = MessageBox.Query( App!, "Eliminar producto", $"¿Eliminar el producto '{selected.Descripcion}'?", "No", "Sí") ?? 0;

if (answer == 1) {
    // borrar
}
```

Devuelve el índice del botón apretado. Para evitar bugs por culpa de un click distraído, **el botón por defecto** ("No") debería ser el seguro.

### `MessageBox.ErrorQuery`

Para errores reales: archivo no encontrado, conexión rota, validación fallida.

``` csharp
MessageBox.ErrorQuery(App!, "Validación", "El precio no puede ser negativo.", "OK");
```

Regla práctica: **un error nunca debería tirar el programa**. Si una operación puede fallar, va en un `try`/`catch` y el error se muestra acá.

## 17. Validación: dónde y cuándo

En el dominio de productos hay reglas naturales:

- el **código** no puede estar vacío y, según el negocio, podría tener formato fijo (`P-9999`, por ejemplo);
- la **descripción** no puede estar vacía;
- el **precio** no puede ser negativo;
- las **existencias** no pueden ser negativas;
- el **código** debería ser único.

Acá entra una decisión de diseño: **dónde valida cada cosa**.

| Regla                                | Dónde valida                                            |
|--------------------------------------|---------------------------------------------------------|
| Tipo (que el precio sea un número)   | En el campo, mientras se tipea (impide letras).         |
| Formato (que el código sigue patrón) | En el diálogo, al apretar Guardar.                      |
| No vacío, no negativo                | En el diálogo, al apretar Guardar.                      |
| Código único                         | En el store o en la ventana, antes de insertar.         |

La validación **inmediata** (al tipear) sirve para errores obvios y de bajo costo. La validación **al guardar** sirve para reglas que dependen de varios campos a la vez o de información que no está en el formulario (¿existe ya un producto con ese código?).

No querés validar la unicidad del código en cada pulsación de tecla: sería tirar una consulta a la base por cada letra. Pero sí querés impedir que el usuario tipee letras en el campo de precio.

## 18. Separación entre UI y datos

La ventana no conoce SQLite. No conoce JSON. No conoce archivos. La ventana habla con un **store** a través de una interfaz mínima:

``` csharp
public interface IProductStore {
    List<Producto> GetAll();
    int            Insert(Producto product);
    bool           Update(Producto product);
    bool           Delete(int id);
}
```

Detrás de esa interfaz puede haber:

- una implementación en memoria (`InMemoryProductStore`) para probar la UI sin tener una base;
- una implementación con SQLite (`SqliteProductStore`);
- una implementación que lea/escriba un JSON (`JsonProductStore`);
- una implementación que llame a una API HTTP.

La ventana no se entera de cuál es. Esta es la **inyección de dependencias** más sencilla que existe: el constructor recibe un `IProductStore`, y arriba de todo decidimos qué pasar:

``` csharp
IProductStore store = new InMemoryProductStore();   // o el que sea
app.Run(new ProductsWindow(store));
```

Esto es lo que hace que la UI sea testeable, evolucionable y reutilizable. Si después decidimos persistir en otra cosa, no tocamos ni una línea de la pantalla.

## 19. Visualización condicional: stock bajo

El dominio de productos da una oportunidad pedagógica linda: **resaltar visualmente** los productos con poco stock. Hay tres formas principales y conviene conocer las tres.

### 1. Indicador en el texto del maestro

Lo más simple. En el `ToString()` del producto, agregamos un caracter al principio:

``` csharp
public override string ToString() {
    string flag = (this.Existencias <= 5) ? "!" : " ";
    string cod  = (this.Codigo ?? "").PadRight(8);
    string desc = this.Descripcion ?? "";
    if (desc.Length > 38) { desc = desc.Substring(0, 37) + "…"; }
    return $"{flag} {cod} {desc}";
}
```

Resultado en la lista:

``` text
  P-0001  Lápiz HB negro
! P-0002  Cuaderno A4 80h
  P-0003  Birome azul
! P-0004  Goma de borrar
```

Funciona en cualquier terminal, no requiere colores, y respeta la regla del maestro escueto: ni precio ni cantidad invaden la lista. Es el equivalente al asterisco de los favoritos en una agenda.

### 2. Color en la fila del maestro

Terminal.Gui permite dibujar con `ColorScheme` propios. Una fila con stock bajo puede pintarse en rojo. Es vistoso, pero hay que cuidarlo: necesita ser legible en terminales con tema claro y oscuro, y conviene no abusar para que no parezca un árbol de Navidad.

### 3. Panel de detalle con campo "Estado"

Es lo que ya hicimos en `RefreshDetails`: el campo `Estado` se calcula con un `switch` sobre existencias y `Activo`, y se muestra como una línea más en la ficha. Es el modo más conservador y suele ser el más útil en aplicaciones de negocio: el usuario ve el estado **cuando elige el producto**, no necesita escanear toda la lista.

Las tres se pueden combinar: indicador discreto en la fila + texto explícito en el detalle. Cada una refuerza la otra sin saturar.

## 20. Orden recomendado para construir una TUI

Para no perderse, conviene construir en este orden:

1. **Sketch en papel o ASCII** de la pantalla principal.
2. **Modelo de datos** (`Producto`) con los campos básicos.
3. **Store en memoria** que cumpla `IProductStore`, con dos o tres productos de prueba hardcodeados.
4. **Ventana principal vacía** con título y borde.
5. **Menú superior**, aunque las acciones no hagan nada todavía.
6. **Panel izquierdo con `ListView`** mostrando los productos de prueba.
7. **Panel derecho con detalle** del producto seleccionado.
8. **Búsqueda activa** que filtre la lista.
9. **Diálogo de edición** que se abra y vuelva, sin todavía persistir.
10. **Acciones CRUD** conectadas al store en memoria.
11. **Atajos de teclado**.
12. **Recargar y salida limpia**.
13. **Cambiar el store a uno persistente** (SQLite, archivos, lo que toque).

Si se intenta hacer todo junto, los errores de layout, eventos y datos se mezclan y no se sabe dónde está la falla. Empezar con un store en memoria es lo que permite enfocarse en la UI primero.

## 21. Errores comunes

### Querer parsear el texto de la fila

La fila se dibuja con `ToString()`, pero la fuente de verdad sigue siendo el objeto `Producto`. Si necesitamos saber qué seleccionó el usuario, leemos `productsList.Value`, no el texto.

### Validar dos veces lo mismo (y diferente)

Si el diálogo valida que el precio no sea negativo pero el store no lo hace, una llamada directa al store puede meter datos malos. Si el store valida pero el diálogo no, el usuario apreta Guardar y recién ahí se entera del error. La regla: **el diálogo valida para que el usuario no se frustre, el store valida para garantizar la integridad de los datos**. Ambos, no uno solo.

### Editar el objeto original antes de Guardar

Si el diálogo modifica directamente el `Producto` que vino de la lista, no hay forma de cancelar. Hay que trabajar sobre una copia (`Clone()`) y solo aplicar los cambios cuando el usuario confirma.

### Mantener la lista en memoria desactualizada

Después de cada operación CRUD hay que actualizar **dos** lugares: el store, y la lista en memoria que alimenta al `ListView`. Si solo actualizamos uno, la UI queda inconsistente. La regla: persistir primero, actualizar memoria después, refrescar la lista por último.

### Confiar en el orden visual

Si el usuario ordena la lista por precio descendente y nosotros guardamos "el producto en el índice 3", a la próxima ordenada eso ya no significa nada. Siempre que necesitemos referenciar un producto, usamos su `Id`, no su posición.

### Botones por defecto peligrosos

Un diálogo de confirmación de borrado cuyo botón por defecto es "Sí" es una trampa. El default tiene que ser el seguro, en general "No" o "Cancelar".

### Olvidarse del teclado

Toda funcionalidad de menú o botón debería tener un atajo de teclado. Probar la app **sin tocar el mouse** es el mejor test de usabilidad de una TUI.

### Sobrecargar el sketch inicial

Empezar con 12 botones y 8 paneles. La pantalla cabe, pero el usuario se pierde. Es mejor empezar con poco y agregar lo que falte cuando un caso de uso lo pida.
