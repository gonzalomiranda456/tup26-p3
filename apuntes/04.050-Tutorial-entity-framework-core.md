# Entity Framework Core

## Introducción — el problema que resuelve

Las aplicaciones trabajan con objetos. Las bases de datos relacionales trabajan con tablas, filas y columnas. Esta diferencia fundamental se llama **impedancia objeto-relacional** (_object-relational impedance mismatch_), y fue durante décadas el mayor punto de fricción del desarrollo backend.

Sin un ORM, el código de acceso a datos se ve así:

```csharp
// Código sin ORM — escribir SQL a mano, mapear manualmente
static List<Producto> ObtenerProductosPorCategoria(int categoriaId) {
    var productos = new List<Producto>();

    using var conn = new SqlConnection(connectionString);
    conn.Open();

    using var cmd = new SqlCommand(
        "SELECT Id, Nombre, Precio, Stock FROM Productos WHERE CategoriaId = @catId",
        conn);
    cmd.Parameters.AddWithValue("@catId", categoriaId);

    using var reader = cmd.ExecuteReader();
    while (reader.Read()) {
        productos.Add(new Producto {
            Id     = reader.GetInt32(0),
            Nombre = reader.GetString(1),
            Precio = reader.GetDecimal(2),
            Stock  = reader.GetInt32(3),
        });
    }

    return productos;
}
```

Esto tiene varios problemas: el SQL está duplicado por toda la aplicación, el mapeo es manual y propenso a errores, los cambios en la base de datos exigen rastrear cada query afectado, y no hay forma de verificar en compilación que la consulta es válida.

**Entity Framework Core (EF Core)** resuelve esto siendo un **Object-Relational Mapper**: traduce automáticamente entre el mundo de objetos C# y el mundo relacional de SQL. Con EF Core el mismo código se ve así:

```csharp
// Con EF Core — limpio, verificable en compilación, sin SQL manual
static List<Producto> ObtenerProductosPorCategoria(int categoriaId) {
    using var db = new TiendaContext();
    return db.Productos
             .Where(p => p.CategoriaId == categoriaId)
             .ToList();
}
```

EF Core genera el SQL correcto para cada base de datos, rastrea cambios automáticamente, y hace que los errores de tipeo en nombres de columnas o tablas se detecten en tiempo de compilación, no en producción.

## 1. Filosofía de diseño

### 1.1 ORM — mapear objetos a tablas

La idea central es que cada **clase** de C# representa una **tabla** de la base de datos, y cada **instancia** de esa clase representa una **fila**.

```
Clase Producto          →   Tabla Productos
────────────────────────────────────────────
int    Id               →   columna Id  (PK)
string Nombre           →   columna Nombre
decimal Precio          →   columna Precio
int    Stock            →   columna Stock
int    CategoriaId      →   columna CategoriaId (FK)
```

Las relaciones entre tablas (FK) se expresan como propiedades de navegación entre clases:

```
Categoria.Productos  →  relación 1:N con Producto
Producto.Categoria   →  propiedad de navegación hacia Categoria
```

### 1.2 Code-First — las clases definen la base de datos

EF Core soporta dos flujos de trabajo:

**Code-First (enfoque recomendado):** se definen primero las clases C# y EF Core genera el esquema de base de datos a partir de ellas. Los cambios al modelo se expresan como **migraciones** que EF Core convierte en `ALTER TABLE`, `CREATE TABLE`, etc.

**Database-First:** se parte de una base de datos existente y EF Core genera las clases. Útil al integrar bases de datos legacy.

En un proyecto nuevo, Code-First es siempre la elección correcta. Permite versionar el esquema junto al código fuente.

### 1.3 Convención sobre configuración

EF Core funciona con **convenciones sensatas** por defecto. Una clase llamada `Producto` con una propiedad `Id` de tipo `int` genera automáticamente una tabla `Productos` con clave primaria autoincremental — sin configuración explícita.

Cuando las convenciones no alcanzan, se puede sobrescribir con **Data Annotations** (atributos) o con la **Fluent API** (configuración en código). La Fluent API es la forma recomendada para configuraciones no triviales porque mantiene las clases de dominio limpias.

## 2. Conceptos principales

### 2.1 Entidad

Una **entidad** es una clase C# que EF Core mapea a una tabla. Las entidades son clases simples (POCO — _Plain Old CLR Objects_) sin dependencias de EF Core:

```csharp
// Entidad simple — no hereda de nada, no tiene atributos de EF
public class Producto {
    public int     Id       { get; set; }
    public string  Nombre   { get; set; } = string.Empty;
    public decimal Precio   { get; set; }
    public int     Stock    { get; set; }
}
```

### 2.2 DbContext

El `DbContext` es el corazón de EF Core. Representa una **sesión con la base de datos** y es la puerta de entrada para todas las operaciones. Sus responsabilidades son:

- Mantener las configuraciones de mapeo
- Rastrear cambios a las entidades cargadas (_change tracking_)
- Traducir operaciones LINQ a SQL
- Gestionar transacciones
- Ejecutar migraciones

```csharp
public class TiendaContext : DbContext {
    // Cada DbSet<T> representa una tabla
    public DbSet<Producto>  Productos  { get; set; }
    public DbSet<Categoria> Categorias { get; set; }
    public DbSet<Pedido>    Pedidos    { get; set; }

    protected override void OnConfiguring(DbContextOptionsBuilder options) {
        options.UseSqlite("Data Source=tienda.db");
    }
}
```

### 2.3 DbSet\<T>

Un `DbSet<T>` representa una **tabla completa** en la base de datos. Es el punto de entrada para consultar, agregar, modificar y eliminar registros de esa tabla. Implementa `IQueryable<T>`, por lo que acepta expresiones LINQ que se traducen a SQL.

### 2.4 Change Tracking — rastreo de cambios

Cuando EF Core carga entidades de la base de datos, las **rastrea** en memoria. Al llamar a `SaveChanges()`, compara el estado actual con el estado original y genera automáticamente los `INSERT`, `UPDATE` o `DELETE` necesarios.

```
Entidad cargada → modificada → SaveChanges()
                                     ↓
                          EF Core detecta cambios
                                     ↓
                          Genera UPDATE automático
```

Los estados posibles de una entidad son:

| Estado       | Descripción                  | SaveChanges genera |
|--------------|------------------------------|--------------------|
| `Detached`   | No rastreada por el contexto | Nada               |
| `Unchanged`  | Cargada, sin modificaciones  | Nada               |
| `Added`      | Nueva, pendiente de insertar | `INSERT`           |
| `Modified`   | Cargada y modificada         | `UPDATE`           |
| `Deleted`    | Marcada para eliminar        | `DELETE`           |

### 2.5 Migraciones

Las migraciones son **clases generadas automáticamente** que describen la diferencia entre el modelo C# actual y el esquema de base de datos. Cada migración tiene dos métodos: `Up()` (aplica el cambio) y `Down()` (lo revierte).

```
Modelo C# → dotnet ef migrations add NombreMigración → archivo de migración
Migración → dotnet ef database update                → SQL ejecutado en DB
```

## 3. Ejemplo progresivo — sistema de gestión de una librería

Vamos a construir paso a paso un sistema de gestión para una librería. Empezamos simple y vamos incorporando conceptos a medida que aparece la necesidad.

### 3.1 Configuración inicial

```bash
# Crear proyecto
dotnet new console -n Libreria
cd Libreria

# Instalar paquetes
dotnet add package Microsoft.EntityFrameworkCore.Sqlite
dotnet add package Microsoft.EntityFrameworkCore.Tools
dotnet add package Microsoft.EntityFrameworkCore.Design
```

El archivo `.csproj` resultante incluye:

```xml
<ItemGroup>
  <PackageReference Include="Microsoft.EntityFrameworkCore.Sqlite" Version="10.*" />
  <PackageReference Include="Microsoft.EntityFrameworkCore.Tools"  Version="10.*" />
  <PackageReference Include="Microsoft.EntityFrameworkCore.Design" Version="10.*" />
</ItemGroup>
```

### 3.2 Primera entidad y contexto

Empezamos con la entidad más simple posible:

```csharp
// Models/Libro.cs
public class Libro {
    public int    Id      { get; set; }
    public string Titulo  { get; set; } = string.Empty;
    public string Autor   { get; set; } = string.Empty;
    public int    Año     { get; set; }
    public decimal Precio { get; set; }
}
```

```csharp
// Data/LibreriaContext.cs
using Microsoft.EntityFrameworkCore;

public class LibreriaContext : DbContext {
    public DbSet<Libro> Libros { get; set; }

    protected override void OnConfiguring(DbContextOptionsBuilder options) {
        options.UseSqlite("Data Source=libreria.db");
    }
}
```

Creamos la primera migración y la aplicamos:

```bash
dotnet ef migrations add CrearTablaLibros
dotnet ef database update
```

Esto genera la carpeta `Migrations/` con el archivo de migración y crea `libreria.db` con la tabla `Libros`.

### 3.3 Operaciones CRUD básicas

```csharp
// Program.cs
using var db = new LibreriaContext();

// ── CREATE ────────────────────────────────────────────────────────────────
var libro = new Libro {
    Titulo = "Clean Code",
    Autor  = "Robert C. Martin",
    Año    = 2008,
    Precio = 4500m,
};

db.Libros.Add(libro);
db.SaveChanges();
Console.WriteLine($"Libro guardado con Id: {libro.Id}");

// ── READ ──────────────────────────────────────────────────────────────────
// Buscar por clave primaria — más eficiente
Libro? encontrado = db.Libros.Find(1);

// Buscar con condición
List<Libro> librosCaros = db.Libros
    .Where(l => l.Precio > 3000)
    .OrderBy(l => l.Titulo)
    .ToList();

// Obtener uno solo
Libro? primerLibro = db.Libros
    .FirstOrDefault(l => l.Autor == "Robert C. Martin");

// ── UPDATE ────────────────────────────────────────────────────────────────
if (encontrado is not null) {
    encontrado.Precio = 5000m;   // solo modificar la propiedad
    db.SaveChanges();            // EF Core detecta el cambio y genera UPDATE
}

// ── DELETE ────────────────────────────────────────────────────────────────
if (primerLibro is not null) {
    db.Libros.Remove(primerLibro);
    db.SaveChanges();
}
```

## 4. Modelado de entidades

### 4.1 Convenciones automáticas

EF Core aplica estas convenciones sin configuración explícita:

| Convención                                 | Resultado                                         |
|--------------------------------------------|---------------------------------------------------|
| Clase `Libro`                              | Tabla `Libros`                                    |
| Propiedad `Id` o `LibroId`                 | Clave primaria autoincremental                    |
| Propiedad `string`                         | Columna `NVARCHAR(MAX)` / `TEXT`, nullable en DB  |
| Propiedad `string` (C# non-null)           | EF la trata como requerida desde el modelo        |
| Propiedad `int`, `decimal`, etc.           | Columna NOT NULL                                  |
| Propiedad `int?`, `string?`                | Columna NULL                                      |
| Propiedad `CategoriaId` + nav. `Categoria` | FK hacia tabla `Categorias`                       |

### 4.2 Data Annotations — configuración con atributos

Los Data Annotations permiten personalizar el mapeo directamente en la clase:

```csharp
using System.ComponentModel.DataAnnotations;
using System.ComponentModel.DataAnnotations.Schema;

[Table("libros")]                          // nombre de tabla personalizado
public class Libro {
    [Key]                                  // clave primaria (redundante si se llama Id)
    [DatabaseGenerated(DatabaseGeneratedOption.Identity)]
    public int Id { get; set; }

    [Required]                             // NOT NULL
    [MaxLength(200)]                       // VARCHAR(200)
    public string Titulo { get; set; } = string.Empty;

    [Required]
    [MaxLength(100)]
    public string Autor { get; set; } = string.Empty;

    [Range(1450, 2100)]                    // validación (no afecta el esquema)
    public int Año { get; set; }

    [Column(TypeName = "decimal(10,2)")]   // tipo SQL exacto
    public decimal Precio { get; set; }

    [NotMapped]                            // esta propiedad NO se guarda en DB
    public string Descripcion => $"{Titulo} ({Año})";
}
```

Las Data Annotations mezclan responsabilidades: la clase de dominio sabe cosas de la base de datos. Para proyectos medianos/grandes, la Fluent API es la alternativa más limpia.

### 4.3 Fluent API — configuración en el contexto

La Fluent API mueve toda la configuración de mapeo al contexto, dejando las entidades como clases puras:

```csharp
// Models/Libro.cs — entidad limpia, sin atributos de EF
public class Libro {
    public int     Id     { get; set; }
    public string  Titulo { get; set; } = string.Empty;
    public string  Autor  { get; set; } = string.Empty;
    public int     Año    { get; set; }
    public decimal Precio { get; set; }
}
```

```csharp
// Data/LibreriaContext.cs
public class LibreriaContext : DbContext {
    public DbSet<Libro> Libros { get; set; }

    protected override void OnModelCreating(ModelBuilder modelBuilder) {
        modelBuilder.Entity<Libro>(entity => {
            entity.ToTable("libros");

            entity.HasKey(l => l.Id);

            entity.Property(l => l.Titulo)
                  .IsRequired()
                  .HasMaxLength(200);

            entity.Property(l => l.Autor)
                  .IsRequired()
                  .HasMaxLength(100);

            entity.Property(l => l.Precio)
                  .HasColumnType("decimal(10,2)");

            // Índice para búsquedas frecuentes por autor
            entity.HasIndex(l => l.Autor);
        });
    }
}
```

### 4.4 Clases de configuración separadas — la forma escalable

Para proyectos con muchas entidades, cada entidad tiene su propia clase de configuración:

```csharp
// Data/Configurations/LibroConfiguration.cs
using Microsoft.EntityFrameworkCore;
using Microsoft.EntityFrameworkCore.Metadata.Builders;

public class LibroConfiguration : IEntityTypeConfiguration<Libro> {
    public void Configure(EntityTypeBuilder<Libro> entity) {
        entity.ToTable("libros");
        entity.HasKey(l => l.Id);

        entity.Property(l => l.Titulo)
              .IsRequired()
              .HasMaxLength(200);

        entity.Property(l => l.Autor)
              .IsRequired()
              .HasMaxLength(100);

        entity.Property(l => l.Precio)
              .HasColumnType("decimal(10,2)");

        entity.HasIndex(l => l.Autor);
    }
}
```

```csharp
// Data/LibreriaContext.cs — limpio, delega a las configuraciones
public class LibreriaContext : DbContext {
    public DbSet<Libro>     Libros     { get; set; }
    public DbSet<Categoria> Categorias { get; set; }

    protected override void OnModelCreating(ModelBuilder modelBuilder) {
        // Aplica automáticamente todas las IEntityTypeConfiguration del assembly
        modelBuilder.ApplyConfigurationsFromAssembly(typeof(LibreriaContext).Assembly);
    }
}
```

## 5. Relaciones entre entidades

### 5.1 Uno a muchos (1:N) — la más común

Un libro pertenece a una categoría; una categoría tiene muchos libros.

```csharp
// Models/Categoria.cs
public class Categoria {
    public int    Id     { get; set; }
    public string Nombre { get; set; } = string.Empty;

    // Propiedad de navegación — colección del lado "uno"
    public List<Libro> Libros { get; set; } = [];
}
```

```csharp
// Models/Libro.cs — agregar FK y propiedad de navegación
public class Libro {
    public int     Id          { get; set; }
    public string  Titulo      { get; set; } = string.Empty;
    public string  Autor       { get; set; } = string.Empty;
    public int     Año         { get; set; }
    public decimal Precio      { get; set; }

    // FK hacia Categoria
    public int     CategoriaId { get; set; }

    // Propiedad de navegación — referencia al lado "uno"
    public Categoria Categoria { get; set; } = null!;
}
```

```csharp
// Data/Configurations/LibroConfiguration.cs — configurar la relación
public class LibroConfiguration : IEntityTypeConfiguration<Libro> {
    public void Configure(EntityTypeBuilder<Libro> entity) {
        // ... propiedades anteriores ...

        entity.HasOne(l => l.Categoria)          // un libro tiene una categoría
              .WithMany(c => c.Libros)            // una categoría tiene muchos libros
              .HasForeignKey(l => l.CategoriaId)  // la FK está en Libro
              .OnDelete(DeleteBehavior.Restrict);  // no eliminar categorías con libros
    }
}
```

Uso:

```csharp
using var db = new LibreriaContext();

// Crear categoría y libros relacionados
var programacion = new Categoria { Nombre = "Programación" };
db.Categorias.Add(programacion);

db.Libros.Add(new Libro {
    Titulo     = "Clean Code",
    Autor      = "Robert C. Martin",
    Año        = 2008,
    Precio     = 4500m,
    Categoria  = programacion   // asignar objeto, EF maneja la FK
});

db.SaveChanges();

// Leer con relación incluida — Include() carga la propiedad de navegación
var librosConCategoria = db.Libros
    .Include(l => l.Categoria)
    .Where(l => l.Categoria.Nombre == "Programación")
    .ToList();

foreach (var l in librosConCategoria) {
    Console.WriteLine($"{l.Titulo} [{l.Categoria.Nombre}]");
}
```

### 5.2 Muchos a muchos (N:N)

Un libro puede tener muchos géneros; un género puede aplicar a muchos libros.

```csharp
// Models/Genero.cs
public class Genero {
    public int    Id     { get; set; }
    public string Nombre { get; set; } = string.Empty;

    public List<Libro> Libros { get; set; } = [];
}
```

```csharp
// Models/Libro.cs — agregar navegación hacia géneros
public class Libro {
    // ... propiedades anteriores ...
    public List<Genero> Generos { get; set; } = [];
}
```

```csharp
// Configuración — EF Core crea la tabla de unión automáticamente
entity.HasMany(l => l.Generos)
      .WithMany(g => g.Libros)
      .UsingEntity(j => j.ToTable("libro_generos"));  // nombre de tabla de unión
```

**Tabla de unión con datos extra** — cuando la relación tiene atributos propios:

```csharp
// Models/LibroAutor.cs — tabla de unión explícita con datos extra
public class LibroAutor {
    public int    LibroId  { get; set; }
    public int    AutorId  { get; set; }
    public string Rol      { get; set; } = string.Empty;  // "principal", "coautor"
    public int    Orden    { get; set; }

    public Libro  Libro  { get; set; } = null!;
    public Autor  Autor  { get; set; } = null!;
}
```

```csharp
// Configuración de la tabla de unión explícita
public class LibroAutorConfiguration : IEntityTypeConfiguration<LibroAutor> {
    public void Configure(EntityTypeBuilder<LibroAutor> entity) {
        entity.HasKey(la => new { la.LibroId, la.AutorId });  // PK compuesta

        entity.HasOne(la => la.Libro)
              .WithMany(l => l.LibroAutores)
              .HasForeignKey(la => la.LibroId);

        entity.HasOne(la => la.Autor)
              .WithMany(a => a.LibroAutores)
              .HasForeignKey(la => la.AutorId);
    }
}
```

## 6. Consultas con LINQ

EF Core traduce expresiones LINQ a SQL. El principio clave: **la consulta no se ejecuta hasta que se itera**.

### 6.1 Ejecución diferida vs inmediata

```csharp
// IQueryable<Libro> — consulta construida pero NO ejecutada todavía
IQueryable<Libro> query = db.Libros.Where(l => l.Precio > 1000);

// Agregar más condiciones sin ir a la base de datos
query = query.OrderBy(l => l.Titulo);
query = query.Take(10);

// Ejecución inmediata — AQUÍ se genera y ejecuta el SQL
List<Libro> resultado = query.ToList();
// SQL generado: SELECT TOP 10 * FROM libros WHERE Precio > 1000 ORDER BY Titulo
```

Métodos que ejecutan la consulta inmediatamente: `ToList()`, `ToArray()`, `FirstOrDefault()`, `Count()`, `Any()`, `Sum()`, `Single()`, `foreach`.

### 6.2 Proyecciones — seleccionar solo lo necesario

```csharp
// Sin proyección — trae todas las columnas
List<Libro> libros = db.Libros.ToList();

// Con proyección — trae solo lo necesario (más eficiente)
var resumen = db.Libros
    .Where(l => l.Precio > 1000)
    .Select(l => new { l.Id, l.Titulo, l.Precio })   // solo 3 columnas
    .ToList();

// Proyectar a un DTO
record LibroDto(int Id, string Titulo, decimal Precio);

List<LibroDto> dtos = db.Libros
    .Select(l => new LibroDto(l.Id, l.Titulo, l.Precio))
    .ToList();
```

### 6.3 Include — carga de relaciones

```csharp
// Carga eager — incluir relaciones en la misma consulta
var libros = db.Libros
    .Include(l => l.Categoria)                    // incluir Categoria
    .Include(l => l.Generos)                      // incluir Generos (colección)
    .Include(l => l.LibroAutores)                 // incluir tabla de unión
        .ThenInclude(la => la.Autor)              // desde la tabla de unión, incluir Autor
    .ToList();

// Acceso seguro a relaciones incluidas
foreach (var libro in libros) {
    Console.WriteLine($"{libro.Titulo} — {libro.Categoria.Nombre}");
    foreach (var la in libro.LibroAutores.OrderBy(la => la.Orden)) {
        Console.WriteLine($"  {la.Autor.Nombre} ({la.Rol})");
    }
}
```

### 6.4 Consultas comunes

```csharp
// Filtrar
var librosBaratos = db.Libros
    .Where(l => l.Precio < 2000 && l.Año >= 2020)
    .ToList();

// Ordenar
var porPrecio = db.Libros
    .OrderByDescending(l => l.Precio)
    .ThenBy(l => l.Titulo)
    .ToList();

// Paginación
int pagina   = 2;
int Tamaño   = 10;

var paginados = db.Libros
    .OrderBy(l => l.Id)
    .Skip((pagina - 1) * Tamaño)
    .Take(Tamaño)
    .ToList();

// Conteos y agregaciones
int total         = db.Libros.Count();
int libros2024    = db.Libros.Count(l => l.Año == 2024);
decimal precioMax = db.Libros.Max(l => l.Precio);
decimal promedio  = db.Libros.Average(l => l.Precio);
bool hayStock     = db.Libros.Any(l => l.Precio < 1000);

// Agrupar
var porCategoria = db.Libros
    .GroupBy(l => l.Categoria.Nombre)
    .Select(g => new {
        Categoria = g.Key,
        Cantidad  = g.Count(),
        Promedio  = g.Average(l => l.Precio),
    })
    .ToList();

// Búsqueda por texto
var resultados = db.Libros
    .Where(l => l.Titulo.Contains("Clean") || l.Autor.Contains("Martin"))
    .ToList();

// Verificar existencia
bool existe = db.Libros.Any(l => l.Titulo == "Clean Code");
```

## 7. Operaciones de escritura

### 7.1 Insertar

```csharp
// Insertar uno
var libro = new Libro { Titulo = "Domain-Driven Design", Autor = "Eric Evans", Precio = 6000m };
db.Libros.Add(libro);
await db.SaveChangesAsync();
Console.WriteLine(libro.Id);   // Id asignado por la DB

// Insertar varios
var libros = new List<Libro> {
    new() { Titulo = "Refactoring", Autor = "Fowler", Precio = 5500m },
    new() { Titulo = "SICP", Autor = "Abelson", Precio = 4000m },
};
db.Libros.AddRange(libros);
await db.SaveChangesAsync();
```

### 7.2 Actualizar

```csharp
// Actualizar entidad rastreada
var libro = await db.Libros.FindAsync(id);
if (libro is not null) {
    libro.Precio = 7000m;
    libro.Stock  = 50;
    await db.SaveChangesAsync();  // genera UPDATE solo para las columnas cambiadas
}

// Actualizar sin cargar la entidad (más eficiente para updates masivos)
await db.Libros
    .Where(l => l.Año < 2000)
    .ExecuteUpdateAsync(setter => setter
        .SetProperty(l => l.Precio, l => l.Precio * 0.9m));  // 10% de descuento
```

### 7.3 Eliminar

```csharp
// Eliminar entidad rastreada
var libro = await db.Libros.FindAsync(id);
if (libro is not null) {
    db.Libros.Remove(libro);
    await db.SaveChangesAsync();
}

// Eliminar sin cargar (más eficiente)
await db.Libros
    .Where(l => l.Stock == 0)
    .ExecuteDeleteAsync();
```

### 7.4 Transacciones

`SaveChanges()` envuelve todos los cambios en una transacción automáticamente. Para operaciones que abarcan múltiples `SaveChanges()`:

```csharp
using var transaction = await db.Database.BeginTransactionAsync();
try {
    var categoria = new Categoria { Nombre = "Clásicos" };
    db.Categorias.Add(categoria);
    await db.SaveChangesAsync();

    var libro = new Libro {
        Titulo     = "Don Quijote",
        Autor      = "Cervantes",
        CategoriaId = categoria.Id,
        Precio     = 3500m,
    };
    db.Libros.Add(libro);
    await db.SaveChangesAsync();

    await transaction.CommitAsync();
}
catch {
    await transaction.RollbackAsync();
    throw;
}
```

## 8. Migraciones

### 8.1 Flujo de trabajo

```bash
# 1. Modificar el modelo (agregar propiedad, relación, etc.)

# 2. Crear la migración
dotnet ef migrations add AgregarStockALibro

# 3. Revisar el archivo generado (siempre)
# Migrations/20240315123456_AgregarStockALibro.cs

# 4. Aplicar a la base de datos
dotnet ef database update

# Comandos útiles adicionales:
dotnet ef migrations list                    # ver todas las migraciones
dotnet ef database update NombreMigración   # ir a una migración específica
dotnet ef migrations remove                  # quitar la última migración (no aplicada)
dotnet ef database update 0                  # revertir todas las migraciones
```

### 8.2 Qué contiene una migración

```csharp
// Migrations/20240315123456_AgregarStockALibro.cs
// Este archivo lo genera EF Core — revisar antes de aplicar
public partial class AgregarStockALibro : Migration {
    protected override void Up(MigrationBuilder migrationBuilder) {
        migrationBuilder.AddColumn<int>(
            name: "Stock",
            table: "libros",
            type: "INTEGER",
            nullable: false,
            defaultValue: 0);
    }

    protected override void Down(MigrationBuilder migrationBuilder) {
        migrationBuilder.DropColumn(
            name: "Stock",
            table: "libros");
    }
}
```

### 8.3 Datos semilla — `HasData`

Cargar datos iniciales que siempre deben existir (catálogos, configuraciones):

```csharp
// En la configuración de la entidad
public class CategoriaConfiguration : IEntityTypeConfiguration<Categoria> {
    public void Configure(EntityTypeBuilder<Categoria> entity) {
        entity.HasKey(c => c.Id);
        entity.Property(c => c.Nombre).IsRequired().HasMaxLength(100);

        // Datos semilla — IDs fijos, obligatorios
        entity.HasData(
            new Categoria { Id = 1, Nombre = "Programación" },
            new Categoria { Id = 2, Nombre = "Arquitectura de Software" },
            new Categoria { Id = 3, Nombre = "Clásicos de la Computación" }
        );
    }
}
```

Luego crear y aplicar la migración normalmente — EF Core detecta los datos semilla y genera los `INSERT` correspondientes.

## 10. Mejores prácticas

### 10.1 El contexto es una unidad de trabajo

El `DbContext` representa una **unidad de trabajo** (_unit of work_): agrupa operaciones relacionadas y las confirma todas juntas. No debe vivir más de lo necesario.

```csharp
// ✓ Correcto — un contexto por operación lógica
async Task TransferirStock(int origenId, int destinoId, int cantidad) {
    using var db = new LibreriaContext();
    using var tx = await db.Database.BeginTransactionAsync();

    var origen  = await db.Libros.FindAsync(origenId);
    var destino = await db.Libros.FindAsync(destinoId);

    if (origen is null || destino is null || origen.Stock < cantidad)
        throw new InvalidOperationException("Transferencia inválida");

    origen.Stock  -= cantidad;
    destino.Stock += cantidad;

    await db.SaveChangesAsync();
    await tx.CommitAsync();
}

// ✗ Incorrecto — contexto de vida larga acumula entidades en memoria
public class Servicio {
    private readonly LibreriaContext _db = new();   // ← vive toda la vida del servicio

    // Potenciales problemas: stale data, memory leaks, concurrencia
}
```

### 10.2 Proyectar en las consultas de solo lectura

```csharp
// ✓ Solo cargar las columnas necesarias
var titulos = await db.Libros
    .Where(l => l.CategoriaId == id)
    .Select(l => new { l.Id, l.Titulo })
    .ToListAsync();

// ✗ Cargar la entidad completa cuando solo se necesita parte
var libros = await db.Libros
    .Where(l => l.CategoriaId == id)
    .ToListAsync();   // trae todas las columnas innecesariamente
```

### 10.3 `AsNoTracking` para consultas de solo lectura

```csharp
// ✓ AsNoTracking — más rápido porque no registra el estado de los objetos
var libros = await db.Libros
    .AsNoTracking()
    .Include(l => l.Categoria)
    .ToListAsync();

// Solo usar entidades rastreadas cuando se van a modificar
var libro = await db.Libros.FindAsync(id);   // rastreado — puede modificarse
libro!.Precio = nuevoPrecio;
await db.SaveChangesAsync();
```

### 10.4 Evitar N+1 queries

El problema N+1 ocurre cuando se hace una consulta para obtener N entidades y luego N consultas adicionales para acceder a sus relaciones:

```csharp
// ✗ N+1 — una query por cada libro para cargar su categoría
var libros = await db.Libros.ToListAsync();
foreach (var libro in libros) {
    Console.WriteLine(libro.Categoria.Nombre);   // query extra por cada libro
}

// ✓ Include — una sola query con JOIN
var libros = await db.Libros
    .Include(l => l.Categoria)
    .ToListAsync();
foreach (var libro in libros) {
    Console.WriteLine(libro.Categoria.Nombre);   // sin queries extra
}
```

### 10.5 Ejecutar lógica de validación fuera del contexto

```csharp
// ✓ Validar antes de persistir
async Task AgregarLibroAsync(Libro libro) {
    if (string.IsNullOrWhiteSpace(libro.Titulo))
        throw new ArgumentException("El título es requerido.");

    if (libro.Precio <= 0)
        throw new ArgumentException("El precio debe ser positivo.");

    bool categoriaExiste = await db.Categorias.AnyAsync(c => c.Id == libro.CategoriaId);
    if (!categoriaExiste)
        throw new ArgumentException($"La categoría {libro.CategoriaId} no existe.");

    db.Libros.Add(libro);
    await db.SaveChangesAsync();
}
```

### 10.6 Revisar siempre las migraciones generadas

```csharp

// Siempre revisar el archivo .cs generado antes de aplicar

// Señales de advertencia en una migración:

// - DropColumn (se pierden datos)

```

### 10.7 Nominar bien las migraciones


```csharp
// Data/Configurations/LibroAutorConfiguration.cs
public class LibroAutorConfiguration : IEntityTypeConfiguration<LibroAutor> {
    public void Configure(EntityTypeBuilder<LibroAutor> entity) {
        entity.HasKey(la => new { la.LibroId, la.AutorId });

        entity.Property(la => la.Rol).IsRequired().HasMaxLength(50);
        entity.Property(la => la.Orden).HasDefaultValue(1);

        entity.HasOne(la => la.Libro)
              .WithMany(l => l.LibroAutores)
              .HasForeignKey(la => la.LibroId);

        entity.HasOne(la => la.Autor)
              .WithMany(a => a.LibroAutores)
              .HasForeignKey(la => la.AutorId);
```

```csharp
// Repositories/LibroRepository.cs
public class LibroRepository {
    private readonly LibreriaContext _db;

    public LibroRepository(LibreriaContext db) {
        _db = db;
    }

    public async Task<List<Libro>> BuscarAsync(string? titulo, int? categoriaId, int pagina = 1, int Tamaño = 20) {
        var query = _db.Libros
            .AsNoTracking()
            .Include(l => l.Categoria)
            .Include(l => l.LibroAutores)
                .ThenInclude(la => la.Autor)
            .AsQueryable();

        if (!string.IsNullOrWhiteSpace(titulo))
            query = query.Where(l => l.Titulo.Contains(titulo));

        if (categoriaId.HasValue)
            query = query.Where(l => l.CategoriaId == categoriaId);

        return await query
            .OrderBy(l => l.Titulo)
            .Skip((pagina - 1) * Tamaño)
            .Take(Tamaño)
            .ToListAsync();
    }

    public async Task<Libro?> ObtenerPorIdAsync(int id) =>
        await _db.Libros
            .Include(l => l.Categoria)
            .Include(l => l.LibroAutores)
                .ThenInclude(la => la.Autor)
            .FirstOrDefaultAsync(l => l.Id == id);

    public async Task<Libro> CrearAsync(Libro libro) {
        ArgumentNullException.ThrowIfNull(libro);

        bool categoriaExiste = await _db.Categorias.AnyAsync(c => c.Id == libro.CategoriaId);
        if (!categoriaExiste)
            throw new ArgumentException($"Categoría {libro.CategoriaId} no encontrada.");

        _db.Libros.Add(libro);
        await _db.SaveChangesAsync();
        return libro;
    }

    public async Task ActualizarPrecioAsync(int id, decimal nuevoPrecio) {
        if (nuevoPrecio <= 0)
            throw new ArgumentException("El precio debe ser positivo.");

        int filas = await _db.Libros
            .Where(l => l.Id == id)
            .ExecuteUpdateAsync(s => s.SetProperty(l => l.Precio, nuevoPrecio));

        if (filas == 0)
            throw new KeyNotFoundException($"Libro {id} no encontrado.");
    }

    public async Task<Dictionary<string, int>> ContarPorCategoriaAsync() =>
        await _db.Libros
            .AsNoTracking()
            .GroupBy(l => l.Categoria.Nombre)
            .Select(g => new { Categoria = g.Key, Cantidad = g.Count() })
            .ToDictionaryAsync(x => x.Categoria, x => x.Cantidad);
}
```

```csharp
    }
}
```

## Resumen

```
¿Qué es EF Core?
    └─► ORM que traduce entre objetos C# y tablas relacionales

Flujo de trabajo Code-First:
    └─► Definir clases (entidades)
    └─► Configurar el DbContext
    └─► Crear migración: dotnet ef migrations add Nombre
    └─► Aplicar: dotnet ef database update

Operaciones fundamentales:
    ├── Leer    → db.Entidades.Where(...).Include(...).ToListAsync()
    ├── Crear   → db.Entidades.Add(obj) + SaveChangesAsync()
    ├── Editar  → cargar entidad, modificar propiedades + SaveChangesAsync()
    └── Borrar  → db.Entidades.Remove(obj) + SaveChangesAsync()

Relaciones:
    ├── 1:N  → HasOne / WithMany + FK en el lado N
    ├── N:N  → HasMany / WithMany  (EF crea tabla de unión)
    └── 1:1  → HasOne / WithOne + FK explícita

Reglas clave:
    ├── AsNoTracking() en consultas de solo lectura
    ├── Include() para evitar el problema N+1
    ├── Select() para proyectar solo las columnas necesarias
    ├── Un contexto por unidad de trabajo (no de vida larga)
    ├── Revisar siempre las migraciones antes de aplicar
    └── IEntityTypeConfiguration por entidad para mantener el contexto limpio
```
