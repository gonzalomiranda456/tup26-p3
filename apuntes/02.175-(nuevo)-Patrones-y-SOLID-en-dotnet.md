# Patrones de diseño y SOLID en .NET — una guía desde SRP

## Introducción

Toda aplicación útil termina creciendo: lee archivos, valida datos, guarda en una base de datos, llama APIs externas, muestra información en pantalla, registra logs, envía notificaciones y maneja errores. Al principio parece cómodo poner todo junto en una sola clase. Después esa clase empieza a cambiar por cualquier motivo: cambió la base de datos, cambió el formato JSON, cambió la UI, cambió una regla de negocio, cambió el proveedor de email.

Ese es el problema que los principios de diseño y los patrones intentan atacar. La idea central de este apunte es simple:

> **SRP es el diagnóstico. Los patrones son la geometría del resultado.**

El documento se organiza en cinco partes:

1. **El problema y el principio**: qué es SRP y por qué nace.
2. **Catálogo de patrones**: las formas recurrentes que toma la extracción de responsabilidades, organizadas según *qué tipo de responsabilidad extraen*.
3. **Patrones específicos del ecosistema .NET**: DI, Options, Middleware, Filters, DTO + Mapper, MVVM.
4. **SOLID en la BCL**: cómo los cinco principios SOLID viven en las librerías de .NET (`Stream`, `IEnumerable<T>`, colecciones, EF Core, ASP.NET Core).
5. **Cierre**: caso integrador y heurísticas para no caer en sobrediseño.

# Parte 1 — El problema y el principio

## 1. El problema: clases que hacen demasiado

Tomemos un ejemplo típico: una agenda de contactos que guarda datos en JSON.

```csharp
public sealed class AgendaService {
    private readonly string path = "contactos.json";

    public void Agregar(Contacto contacto) {
        List<Contacto> contactos;

        if (File.Exists(path)) {
            string json = File.ReadAllText(path);
            contactos = JsonSerializer.Deserialize<List<Contacto>>(json) ?? [];
        }
        else {
            contactos = [];
        }

        if (string.IsNullOrWhiteSpace(contacto.Nombre))
            throw new InvalidOperationException("El nombre es obligatorio.");

        contactos.Add(contacto);

        string nuevoJson = JsonSerializer.Serialize(contactos, new JsonSerializerOptions {
            WriteIndented = true
        });

        File.WriteAllText(path, nuevoJson);
    }
}
```

Funciona. Pero la clase está haciendo de todo:

```text
AgendaService
    ├─► sabe dónde está el archivo
    ├─► sabe leer archivos
    ├─► sabe parsear JSON
    ├─► sabe validar contactos
    ├─► sabe agregar contactos
    └─► sabe guardar archivos
```

La pregunta de diseño no es "¿funciona?". La pregunta correcta es:

> ¿Por cuántas razones distintas puede cambiar esta clase?

La respuesta: demasiadas. Si cambia el formato a XML, hay que tocar la clase. Si cambia la regla de validación, también. Si en vez de archivo hay que usar SQLite, también. Cada cambio arriesga romper algo que ya funcionaba.

## 2. SRP — el diagnóstico

El **Single Responsibility Principle** lo enuncia así:

> Una clase debería tener una sola razón para cambiar.

El principio no exige clases con un solo método ni microscópicas. Exige que la responsabilidad conceptual sea clara. Cuando una clase cambia por motivos distintos, probablemente tiene responsabilidades mezcladas.

```text
Mal criterio:
    "Esta clase tiene muchos métodos, entonces viola SRP."

Buen criterio:
    "Esta clase puede cambiar por motivos distintos, entonces probablemente viola SRP."
```

Una clase puede tener varios métodos si todos pertenecen a la misma responsabilidad. Un repositorio con `ObtenerTodos`, `ObtenerPorId`, `Guardar` y `Eliminar` sigue siendo una sola responsabilidad: persistir contactos. En cambio, si la misma clase suma `ValidarEmail`, `RenderizarHtml`, `EnviarWhatsapp` y `CalcularDeuda`, ya no es una clase: es un placard antes de que llegue visita.

Una versión más limpia del ejemplo separa responsabilidades:

```csharp
public sealed class AgendaService(
    IContactoRepository repository,
    IContactoValidator validator) {

    public void Agregar(Contacto contacto) {
        validator.Validar(contacto);

        List<Contacto> contactos = repository.ObtenerTodos();
        contactos.Add(contacto);
        repository.GuardarTodos(contactos);
    }
}
```

El servicio expresa el caso de uso. No sabe si los datos vienen de JSON, SQLite, SQL Server, memoria o una API. Ese detalle vive en otra clase.

## 3. Los patrones como formas recurrentes de extracción

Cuando uno detecta que una clase carga con dos responsabilidades, hay que decidir qué pieza extraer, dónde ponerla y cómo conectarla con el resto. Esas decisiones tienen un puñado de **formas recurrentes**, y a esas formas la comunidad les puso nombre hace décadas: son los patrones de diseño.

Vale la pena ser honestos sobre la dirección causal. Los patrones no son la causa de un buen diseño. La causa es la presión del SRP: la necesidad de separar lo que tiene motivos distintos para cambiar. Lo que pasa es que, cuando uno extrae responsabilidades repetidamente, va descubriendo que la mayoría de las extracciones caen en unas pocas formas. Esas formas son los patrones. Estudiarlos no reemplaza al diagnóstico; lo acelera. Cuando uno mira una clase y dice "acá se mezclan dos cosas: la decisión de qué algoritmo usar y la implementación de cada algoritmo", reconocer que eso es un Strategy clasifica el refactor inmediatamente.

> **No toda violación de SRP necesita un patrón con nombre.** A veces basta con extraer una clase y listo. Si el resultado se parece a uno de los patrones, perfecto; si no, también. Evitar el dogma del catálogo es parte de aplicar patrones bien.

## 4. Taxonomía operativa: qué responsabilidad extrae cada familia

Los patrones del catálogo clásico (GoF y posteriores) pueden organizarse según qué *tipo* de responsabilidad extraen. Para SRP, esa taxonomía es más útil que la división tradicional creacional/estructural/comportamiento:

| Familia                 | Qué responsabilidad extrae                            | Patrones                                  |
|-------------------------|-------------------------------------------------------|-------------------------------------------|
| Política / algoritmo    | "Cómo se hace cierta operación" cuando hay variantes  | Strategy, Template Method, State          |
| Operación               | "Qué se hace" como cosa de primera clase              | Command, Chain of Responsibility, Visitor |
| Creación                | "Cómo se construye" un objeto cuando es no trivial    | Factory, Builder                          |
| Coordinación            | "Quién le habla a quién"                              | Mediator, Observer, Facade                |
| Persistencia / consulta | "Cómo se guarda y se busca"                           | Repository, Unit of Work, Specification   |
| Adaptación / envoltura  | "Cómo se interopera o se decora"                      | Adapter, Decorator, Proxy                 |

Cada fila representa un *tipo* de motivo de cambio. Cuando una clase mezcla, por ejemplo, "lógica de negocio" con "cómo se elige el algoritmo de cálculo", el motivo extra encaja en la primera fila — la solución va a tener forma de Strategy, Template Method o State, según el subcaso. La taxonomía clasifica el problema antes de elegir el patrón.

Más allá del catálogo clásico, el ecosistema .NET aporta sus propios patrones recurrentes. Los vemos en la parte 3.

# Parte 2 — Catálogo de patrones por responsabilidad extraída

## 5. Algoritmo y política

Estos patrones extraen la pregunta "¿cómo se hace?" cuando hay más de una respuesta válida.

### 5.1 Strategy — extraer la decisión sobre cómo se calcula

**Motivo de cambio que separa.** Una clase decide qué algoritmo usar y, dentro de ella, implementa los algoritmos. Hay dos motivos de cambio: cuándo se elige cada algoritmo y qué hace cada algoritmo.

**Síntoma.** Un `switch` o cadena de `if` que enumera alternativas y dentro de cada rama hay lógica sustantiva.

```csharp
public class CalculadoraEnvio {
    public decimal Calcular(string tipo, decimal peso, decimal distancia) {
        switch (tipo) {
            case "estandar":
                return peso * 0.5m + distancia * 0.1m;
            case "express":
                return peso * 1.0m + distancia * 0.3m + 50m;
            case "internacional":
                var aduanas = 100m;
                var seguro  = peso > 10 ? 80m : 30m;
                return peso * 2.0m + distancia * 0.5m + aduanas + seguro;
        }
        throw new ArgumentException(tipo);
    }
}
```

**Refactor: Strategy.** Cada algoritmo se vuelve una clase con un único método; la calculadora orquesta sin conocer los detalles.

```csharp
public interface IPoliticaEnvio {
    decimal Calcular(decimal peso, decimal distancia);
}

public class EnvioEstandar      : IPoliticaEnvio { /* ... */ }
public class EnvioExpress       : IPoliticaEnvio { /* ... */ }
public class EnvioInternacional : IPoliticaEnvio { /* ... */ }

public class CalculadoraEnvio(IPoliticaEnvio politica) {
    public decimal Calcular(decimal peso, decimal distancia) =>
        politica.Calcular(peso, distancia);
}
```

**Por qué satisface SRP.** La calculadora cambia si cambia cómo se elige la política. Cada implementación cambia si cambia su algoritmo. Tres áreas de negocio (envíos estándar, express, internacional) pueden pedir cambios sobre sus respectivas clases sin colisionar.

> **Cuándo no aplicar Strategy.** Si las variantes cambian siempre juntas — por ejemplo, porque dependen de una misma fórmula tributaria — el `switch` original es preferible. La extracción se justifica cuando los motivos de cambio son realmente disjuntos.

### 5.2 Template Method — extraer los pasos variables del esqueleto

**Motivo de cambio que separa.** Un algoritmo tiene una estructura fija pero algunos pasos varían. La estructura cambia por una razón (cómo se orquestan los pasos); los pasos cambian por otra (qué hace cada uno).

**Síntoma.** Varios métodos de distintas clases con la misma estructura repetida y diferencias localizadas.

```csharp
public class GeneradorReporteVentas {
    public string Generar() {
        var sb = new StringBuilder();
        sb.AppendLine("=== REPORTE VENTAS ===");
        sb.AppendLine($"Fecha: {DateTime.Today:d}");
        var datos = ConsultaSql("SELECT * FROM ventas WHERE fecha=@hoy");
        foreach (var fila in datos) sb.AppendLine($"{fila.Id} - ${fila.Total}");
        sb.AppendLine($"Total: ${datos.Sum(f => f.Total)}");
        return sb.ToString();
    }
}

public class GeneradorReporteCompras {
    public string Generar() {
        var sb = new StringBuilder();
        sb.AppendLine("=== REPORTE COMPRAS ===");
        sb.AppendLine($"Fecha: {DateTime.Today:d}");
        var datos = ConsultaSql("SELECT * FROM compras WHERE fecha=@hoy");
        foreach (var fila in datos) sb.AppendLine($"{fila.Id} - ${fila.Total}");
        sb.AppendLine($"Total: ${datos.Sum(f => f.Total)}");
        return sb.ToString();
    }
}
```

La estructura del reporte es idéntica; los pasos varían (título, consulta).

**Refactor: Template Method.**

```csharp
public abstract class GeneradorReporte {
    public string Generar() {
        var sb = new StringBuilder();
        sb.AppendLine($"=== {Titulo()} ===");
        sb.AppendLine($"Fecha: {DateTime.Today:d}");
        var datos = ObtenerDatos();
        foreach (var fila in datos) sb.AppendLine($"{fila.Id} - ${fila.Total}");
        sb.AppendLine($"Total: ${datos.Sum(f => f.Total)}");
        return sb.ToString();
    }

    protected abstract string Titulo();
    protected abstract IEnumerable<Fila> ObtenerDatos();
}

public class ReporteVentas : GeneradorReporte {
    protected override string Titulo() => "REPORTE VENTAS";
    protected override IEnumerable<Fila> ObtenerDatos() =>
        ConsultaSql("SELECT * FROM ventas WHERE fecha=@hoy");
}
```

**Por qué satisface SRP.** La clase base es responsable de la **estructura** del reporte; cambia si cambia el formato general. Cada subclase es responsable de **un tipo** de reporte; cambia si cambia su consulta o su título.

> **Variante moderna.** Template Method puede implementarse con composición y delegados en vez de herencia. Un `Func<IEnumerable<Fila>>` y un `string titulo` pasados al constructor logran el mismo SRP sin la rigidez de la herencia. El patrón es la idea ("estructura fija + pasos variables"), no su implementación específica.

### 5.3 State — extraer el comportamiento que cambia con el estado interno

**Motivo de cambio que separa.** Un objeto se comporta distinto según su estado, y la clase termina llena de `if` por estado. Cada estado tiene su propio motivo de cambio: cambian las reglas de "estar pendiente" sin que cambien las reglas de "estar pagado".

**Síntoma.** Métodos con `switch` o `if` sobre un campo `estado` o `tipo`, repetido en varios métodos.

```csharp
public class Pedido {
    public string Estado { get; set; } = "pendiente";

    public void Pagar() {
        if (Estado == "pendiente") Estado = "pagado";
        else if (Estado == "cancelado") throw new InvalidOperationException("Ya cancelado");
        else throw new InvalidOperationException($"No se puede pagar desde {Estado}");
    }

    public void Despachar() {
        if (Estado == "pagado") Estado = "despachado";
        else throw new InvalidOperationException($"No se puede despachar desde {Estado}");
    }

    public void Cancelar() {
        if (Estado == "pendiente" || Estado == "pagado") Estado = "cancelado";
        else throw new InvalidOperationException($"No se puede cancelar desde {Estado}");
    }
}
```

**Refactor: State.** Cada estado es una clase, y el `Pedido` delega su comportamiento al estado actual.

```csharp
public abstract class EstadoPedido {
    public abstract EstadoPedido Pagar();
    public abstract EstadoPedido Despachar();
    public abstract EstadoPedido Cancelar();
}

public class Pendiente : EstadoPedido {
    public override EstadoPedido Pagar()     => new Pagado();
    public override EstadoPedido Despachar() => throw new InvalidOperationException();
    public override EstadoPedido Cancelar()  => new Cancelado();
}

public class Pagado : EstadoPedido {
    public override EstadoPedido Pagar()     => throw new InvalidOperationException("Ya pagado");
    public override EstadoPedido Despachar() => new Despachado();
    public override EstadoPedido Cancelar()  => new Cancelado();
}

public class Despachado : EstadoPedido { /* todas las transiciones inválidas */ }
public class Cancelado  : EstadoPedido { /* todas las transiciones inválidas */ }

public class Pedido {
    public EstadoPedido Estado { get; private set; } = new Pendiente();

    public void Pagar()     => Estado = Estado.Pagar();
    public void Despachar() => Estado = Estado.Despachar();
    public void Cancelar()  => Estado = Estado.Cancelar();
}
```

**Por qué satisface SRP.** Cada clase de estado responde a un único actor: el área que define qué transiciones permite desde ahí. Si Operaciones decide que un pedido cancelado puede reactivarse, eso es modificar `Cancelado`, sin tocar las demás. Si Producto agrega un nuevo estado `EnRevision`, es agregar una clase nueva. Acá converge con OCP: agregar es preferible a modificar.

## 6. Operación como objeto de primera clase

La responsabilidad extraída acá no es "cómo se hace" sino "qué se hace" — la operación misma se vuelve un objeto.

### 6.1 Command — convertir operaciones en objetos

**Motivo de cambio que separa.** El código que **invoca** una operación se mezcla con el código que la **ejecuta**. Si se quiere agregar logging, undo, queuing, programación diferida o transaccionalidad, todo eso son responsabilidades distintas que no caben dentro del invocador ni dentro de la operación.

**Síntoma.** Un método grande que hace varias cosas, sin posibilidad de revertir, registrar, programar o reordenar lo que hizo.

```csharp
public class Editor {
    private string contenido = "";

    public void Insertar(int pos, string texto) {
        contenido = contenido.Insert(pos, texto);
    }

    public void Borrar(int pos, int largo) {
        contenido = contenido.Remove(pos, largo);
    }

    // ¿Cómo agregamos undo? ¿Cómo registramos un log de cambios?
    // Tendríamos que tocar Insertar y Borrar.
}
```

**Refactor: Command.** Cada operación es un objeto que sabe ejecutarse y revertirse.

```csharp
public interface IComando {
    void Ejecutar();
    void Deshacer();
}

public class InsertarTexto(Editor editor, int pos, string texto) : IComando {
    public void Ejecutar() => editor.InsertarTexto(pos, texto);
    public void Deshacer() => editor.BorrarTexto(pos, texto.Length);
}

public class BorrarTexto(Editor editor, int pos, int largo) : IComando {
    private string textoBorrado = "";
    public void Ejecutar() {
        textoBorrado = editor.LeerTexto(pos, largo);
        editor.BorrarTexto(pos, largo);
    }
    public void Deshacer() => editor.InsertarTexto(pos, textoBorrado);
}

public class HistorialComandos {
    private readonly Stack<IComando> historial = new();

    public void Ejecutar(IComando c) {
        c.Ejecutar();
        historial.Push(c);
    }

    public void Deshacer() {
        if (historial.Count > 0) historial.Pop().Deshacer();
    }
}
```

**Por qué satisface SRP.** El editor cambia si cambia su contenido. Cada comando cambia si cambia esa operación específica. El historial cambia si cambia la política de undo/redo (límite de pasos, persistencia). Tres motivos disjuntos.

> **Caso real.** Command es la base de cualquier sistema con undo/redo, queuing, transacciones distribuidas (cada paso reversible es un comando), y de los handlers de MediatR — un `IRequest<T>` es esencialmente un Command serializable. En WPF, MAUI, Avalonia y WinUI, la interfaz `ICommand` permite que los botones se enlacen directamente a comandos del ViewModel.

### 6.2 Chain of Responsibility — extraer cada paso de un proceso

**Motivo de cambio que separa.** Un proceso de validación, autorización o filtrado tiene varios pasos. Si todos viven en una clase, agregar un paso requiere modificarla.

**Síntoma.** Un método con muchos `if` consecutivos donde cada uno chequea una condición y decide si continúa o aborta.

```csharp
public class ProcesadorRequest {
    public Resultado Procesar(Request r) {
        if (r.Token is null) return Resultado.Error("Sin token");
        if (!ValidarToken(r.Token)) return Resultado.Error("Token inválido");

        if (!TienePermiso(r.Usuario, r.Recurso)) return Resultado.Error("Sin permiso");

        if (ExcedioLimite(r.Usuario)) return Resultado.Error("Límite excedido");

        if (r.Body is null) return Resultado.Error("Body vacío");
        if (!ValidarSchema(r.Body)) return Resultado.Error("Schema inválido");

        return EjecutarAccion(r);
    }
}
```

**Refactor: Chain of Responsibility.**

```csharp
public abstract class HandlerRequest {
    private HandlerRequest? siguiente;

    public HandlerRequest Encadenar(HandlerRequest h) {
        siguiente = h;
        return h;
    }

    public Resultado Procesar(Request r) {
        var resultado = ProcesarLocal(r);
        if (!resultado.Ok) return resultado;
        return siguiente?.Procesar(r) ?? Resultado.Ok();
    }

    protected abstract Resultado ProcesarLocal(Request r);
}

public class HandlerAuth : HandlerRequest {
    protected override Resultado ProcesarLocal(Request r) =>
        r.Token is null      ? Resultado.Error("Sin token")
      : ValidarToken(r.Token) ? Resultado.Ok()
                              : Resultado.Error("Token inválido");
}

public class HandlerRateLimit : HandlerRequest { /* ... */ }
public class HandlerSchema    : HandlerRequest { /* ... */ }
```

**Por qué satisface SRP.** Cada handler responde a un solo aspecto del procesamiento, y a un solo equipo (auth → seguridad, rate limit → plataforma, schema → API). Agregar un nuevo paso es agregar un handler y enlazarlo, sin modificar los existentes.

### 6.3 Visitor — extraer operaciones de una jerarquía de tipos

**Motivo de cambio que separa.** Cuando una jerarquía de clases tiene múltiples operaciones que se aplican a sus tipos, esas operaciones tienden a vivir como métodos virtuales en cada clase. Pero las operaciones no son responsabilidad de la clase: son responsabilidad de quien las consume.

**Síntoma.** Una clase de dominio (típicamente nodos de un AST, formas geométricas, instrumentos financieros) que tiene muchos métodos no esenciales: `Imprimir`, `Serializar`, `Validar`, `CalcularImpuestos`, `GenerarHtml`. Cada uno responde a un actor distinto.

```csharp
// Cada nodo carga con TODAS las operaciones — y cambia con cada nueva
public abstract class Expr {
    public abstract double Evaluar();           // matemáticos
    public abstract string AImprimir();         // UI
    public abstract string ALatex();            // exportación científica
    public abstract Expr Derivar(string var);   // cálculo simbólico
    public abstract Expr Simplificar();         // optimización
}

public class Suma(Expr izq, Expr der) : Expr {
    public override double Evaluar() => izq.Evaluar() + der.Evaluar();
    public override string AImprimir() => $"({izq.AImprimir()} + {der.AImprimir()})";
    // ... cinco más
}
```

Cada nuevo tipo de operación obliga a modificar **todos los nodos**.

**Refactor: Visitor.** Cada operación se vuelve un visitor; los nodos sólo aceptan visitors.

```csharp
public abstract class Expr {
    public abstract T Aceptar<T>(IVisitorExpr<T> v);
}

public class Suma(Expr izq, Expr der) : Expr {
    public Expr Izq { get; } = izq;
    public Expr Der { get; } = der;
    public override T Aceptar<T>(IVisitorExpr<T> v) => v.Visitar(this);
}

public class Numero(double valor) : Expr {
    public double Valor { get; } = valor;
    public override T Aceptar<T>(IVisitorExpr<T> v) => v.Visitar(this);
}

public interface IVisitorExpr<T> {
    T Visitar(Suma s);
    T Visitar(Numero n);
}

public class Evaluador : IVisitorExpr<double> {
    public double Visitar(Suma s)   => s.Izq.Aceptar(this) + s.Der.Aceptar(this);
    public double Visitar(Numero n) => n.Valor;
}

public class Impresor : IVisitorExpr<string> {
    public string Visitar(Suma s)   => $"({s.Izq.Aceptar(this)} + {s.Der.Aceptar(this)})";
    public string Visitar(Numero n) => n.Valor.ToString();
}
```

**Por qué satisface SRP.** Los nodos son responsables de la estructura del AST; cambian si aparece un nuevo tipo de nodo. Cada visitor es responsable de una operación; cambia si cambia esa operación. Cinco operaciones, cinco visitors, ningún acoplamiento accidental.

> **Conexión con C# moderno.** Si los nodos se modelan como `record sealed`, un `switch` exhaustivo en una función pura suele ser más limpio que el Visitor clásico — especialmente con menos de 10 nodos. El compilador avisa si te olvidás de un caso. Visitor sigue siendo mejor cuando las operaciones superan en cantidad a los tipos y querés que cada operación viva en su propio archivo.

## 7. Creación

La construcción de un objeto puede ser una responsabilidad propia, distinta de su uso.

### 7.1 Factory — extraer la decisión de qué se construye

**Motivo de cambio que separa.** Una clase consume objetos de varios tipos posibles, y dentro de ella decide cuál instanciar. Si la regla de elección cambia, hay que modificar la clase consumidora aunque sus operaciones no hayan cambiado.

**Síntoma.** Un `new` distinto según un parámetro, dentro del código de negocio.

```csharp
public class ProcesadorPagos {
    public void Procesar(string medio, decimal monto) {
        IGateway gw;
        if      (medio == "tarjeta")       gw = new GatewayTarjeta();
        else if (medio == "transferencia") gw = new GatewayTransferencia();
        else if (medio == "mercado_pago")  gw = new GatewayMercadoPago();
        else throw new ArgumentException(medio);

        gw.Cobrar(monto);
    }
}
```

**Refactor: Factory.**

```csharp
public interface IFabricaGateways {
    IGateway Crear(string medio);
}

public class FabricaGateways : IFabricaGateways {
    public IGateway Crear(string medio) => medio switch {
        "tarjeta"        => new GatewayTarjeta(),
        "transferencia"  => new GatewayTransferencia(),
        "mercado_pago"   => new GatewayMercadoPago(),
        _ => throw new ArgumentException(medio),
    };
}

public class ProcesadorPagos(IFabricaGateways fabrica) {
    public void Procesar(string medio, decimal monto) =>
        fabrica.Crear(medio).Cobrar(monto);
}
```

**Por qué satisface SRP.** El procesador cambia si cambia la lógica de cobro. La fábrica cambia si cambia la lógica de elección o aparece un nuevo gateway. Agregar criptomonedas ahora es modificar sólo la fábrica.

> **Factories en .NET.** Nombres como `IHttpClientFactory`, `ILoggerFactory` e `IDbContextFactory<TContext>` no son casualidad: separan creación/configuración del código que usa los objetos. Notá también que el contenedor de DI (`IServiceProvider`) es esencialmente una abstract factory genérica: cuando registrás `services.AddScoped<IGateway, GatewayTarjeta>()`, estás delegando la decisión de construcción al contenedor. Para casos donde la elección depende de un parámetro de runtime, una factory explícita sigue siendo la herramienta correcta.

### 7.2 Builder — extraer la construcción compleja

**Motivo de cambio que separa.** Un objeto requiere muchos parámetros para construirse, algunos opcionales, con dependencias entre ellos y validaciones. Mezclar esa lógica con el código que usa el objeto produce constructores monstruosos o construcciones repetitivas.

**Síntoma.** Constructores con muchos parámetros, varios opcionales, con interdependencias.

```csharp
var req = new HttpRequest(
    method: "POST",
    url: "https://api.empresa.com/v1/clientes",
    headers: new Dictionary<string, string> {
        ["Authorization"] = $"Bearer {token}",
        ["Content-Type"]  = "application/json",
        ["X-Tenant-Id"]   = tenantId,
        ["X-Trace-Id"]    = traceId,
    },
    body: JsonSerializer.Serialize(payload),
    timeout: TimeSpan.FromSeconds(30),
    followRedirects: true,
    retries: 3);
```

**Refactor: Builder.**

```csharp
public class RequestBuilder {
    private string method = "GET";
    private string url    = "";
    private readonly Dictionary<string, string> headers = new();
    private string? body;
    private TimeSpan timeout = TimeSpan.FromSeconds(30);

    public RequestBuilder Post(string u)       { method = "POST"; url = u; return this; }
    public RequestBuilder Auth(string token)   { headers["Authorization"] = $"Bearer {token}"; return this; }
    public RequestBuilder Tenant(string id)    { headers["X-Tenant-Id"] = id; return this; }
    public RequestBuilder Trace(string id)     { headers["X-Trace-Id"] = id; return this; }
    public RequestBuilder Json(object payload) {
        headers["Content-Type"] = "application/json";
        body = JsonSerializer.Serialize(payload);
        return this;
    }
    public RequestBuilder Timeout(TimeSpan t)  { timeout = t; return this; }

    public HttpRequest Build() => new(method, url, headers, body, timeout);
}

// Uso
var req = new RequestBuilder()
    .Post("https://api.empresa.com/v1/clientes")
    .Auth(token)
    .Tenant(tenantId)
    .Trace(traceId)
    .Json(payload)
    .Build();
```

**Por qué satisface SRP.** El builder es responsable de las reglas de construcción: qué combinaciones son válidas, qué headers se derivan de qué llamadas, cómo se serializa el body. La clase `HttpRequest` es responsable de representar la request ya armada. El consumidor es responsable de decidir qué request quiere. Tres responsabilidades, tres lugares.

## 8. Coordinación

Cuando el problema es "quién le habla a quién", la responsabilidad extraída es la coordinación misma.

### 8.1 Mediator — extraer la coordinación entre componentes

**Motivo de cambio que separa.** Varios objetos colaboran y cada uno conoce a los demás directamente. Cualquier cambio en cómo colaboran afecta a varios.

**Síntoma.** Una pieza de código donde un servicio le habla a otros tres, esos tres se conocen entre sí, y el conjunto forma un grafo de dependencias mutuas.

```csharp
public class ServicioPedidos(
    ServicioInventario inventario,
    ServicioPagos pagos,
    ServicioNotificaciones notif,
    ServicioFacturacion factura,
    ServicioLogistica logistica) {

    public async Task ConfirmarPedido(Pedido p) {
        await inventario.ReservarStock(p);
        var pago = await pagos.Cobrar(p);
        await factura.Emitir(p, pago);
        await logistica.AgendarEnvio(p);
        await notif.NotificarCliente(p);
    }
}
```

Si mañana cambia el flujo (validación de fraude antes de cobrar, puntos de fidelidad, eliminación de un paso), el servicio se modifica. Cada paso del workflow es un motivo distinto de cambio.

**Refactor: Mediator.** Cada paso es un handler independiente; un mediador los coordina mediante eventos o un pipeline.

```csharp
// Notificación de dominio: "se confirmó un pedido"
public record PedidoConfirmado(int PedidoId);

// Cada handler es responsable de UN paso
public class HandlerReservarStock(ServicioInventario inv) : IHandler<PedidoConfirmado> {
    public Task Handle(PedidoConfirmado e, CancellationToken ct) =>
        inv.ReservarStock(e.PedidoId, ct);
}

public class HandlerCobrarPago     (ServicioPagos pagos)       : IHandler<PedidoConfirmado> { /* ... */ }
public class HandlerEmitirFactura  (ServicioFacturacion fc)    : IHandler<PedidoConfirmado> { /* ... */ }
public class HandlerAgendarEnvio   (ServicioLogistica log)     : IHandler<PedidoConfirmado> { /* ... */ }
public class HandlerNotificarCliente(ServicioNotif notif)      : IHandler<PedidoConfirmado> { /* ... */ }

// El mediador despacha el evento a todos los handlers registrados
public class ServicioPedidos(IMediator mediator) {
    public Task ConfirmarPedido(Pedido p) =>
        mediator.Publish(new PedidoConfirmado(p.Id));
}
```

**Por qué satisface SRP.** Cada handler responde a un solo equipo y cambia por su propio motivo. Agregar un paso es agregar una clase; eliminar un paso es borrar un archivo. El servicio orquestador queda en lo suyo: traducir un comando de aplicación en un evento de dominio, y delegar.

> **Caso real en .NET.** MediatR (la librería) y los handlers de comandos/eventos del CQRS son la implementación canónica de este patrón en el ecosistema. La librería es opcional; el patrón no.

### 8.2 Observer / Events — extraer la reacción a eventos

**Motivo de cambio que separa.** Una clase que produce un evento (un cambio de estado, una acción del usuario) tendría que conocer a todos los interesados en ese evento. Cada nuevo interesado modifica al productor.

**Síntoma.** Después de hacer algo importante, una clase llama a varios servicios para "avisarles".

```csharp
public class CarritoCompras {
    public void AgregarItem(Item i) {
        items.Add(i);

        // Avisos directos — la clase conoce a sus consumidores
        analiticas.Registrar(i);
        recomendador.Aprender(i);
        promociones.EvaluarOfertas(items);
    }
}
```

**Refactor: Observer.** El carrito sólo notifica un evento; los interesados se registran independientemente.

```csharp
public class CarritoCompras {
    public event Action<Item>? ItemAgregado;

    public void AgregarItem(Item i) {
        items.Add(i);
        ItemAgregado?.Invoke(i);
    }
}

// En la composición de la app
carrito.ItemAgregado += i => analiticas.Registrar(i);
carrito.ItemAgregado += i => recomendador.Aprender(i);
carrito.ItemAgregado += i => promociones.EvaluarOfertas(carrito.Items);
```

**Por qué satisface SRP.** El carrito maneja su contenido y avisa cambios; cambia si cambia la lógica del carrito. Cada observador maneja su reacción específica; cambia por su propio motivo. Si analítica decide que ya no le interesa registrar items, se desuscribe sin que el carrito se entere.

> **En .NET** este patrón aparece naturalmente con `event`, `EventHandler<T>` e `IObservable<T>` (Reactive Extensions). En la BCL es la base de eventos de UI (`Click`, `TextChanged`), de cambios de propiedad (`INotifyPropertyChanged`) y del modelo de eventos del dominio.

### 8.3 Facade — extraer una interfaz simple sobre un subsistema complejo

**Motivo de cambio que separa.** Un cliente del código tiene que orquestar varias clases internas para hacer una tarea común. Esa orquestación se replica en cada cliente y cada cliente conoce demasiado de la estructura interna.

**Síntoma.** El mismo bloque de cinco líneas que instancia varios objetos y los enlaza, repetido en muchos lugares.

```csharp
// Cada cliente orquesta los detalles
var datos     = new SqlConnection(...).Query("SELECT ...");
var procesado = new Procesador(opciones).Procesar(datos);
var formateado = new Formateador(formato).Formatear(procesado);
var pdf       = new GeneradorPdf().Generar(formateado);
File.WriteAllBytes(ruta, pdf);
```

**Refactor: Facade.** Una clase con un método claro encapsula toda la orquestación.

```csharp
public class GeneradorReportes(
    SqlConnection conn,
    OpcionesProcesador opts,
    FormatoSalida fmt) {

    public byte[] Generar(string consulta) {
        var datos      = conn.Query(consulta);
        var procesado  = new Procesador(opts).Procesar(datos);
        var formateado = new Formateador(fmt).Formatear(procesado);
        return new GeneradorPdf().Generar(formateado);
    }
}
```

**Por qué satisface SRP.** El facade es responsable de la orquestación; cambia si cambia el flujo. Cada subsistema interno cambia por su propio motivo. El cliente recupera el SRP que había perdido: ya no tiene que conocer ni orquestar la maquinaria interna.

> **Cuidado con los facades anémicos.** Si un facade sólo redirige a un único subsistema sin agregar valor, no extrae responsabilidad: sólo agrega indirección. Un buen facade orquesta; uno malo es un alias. Y si la facade empieza a acumular toda la lógica de la aplicación, dejó de ser fachada y pasó a ser baldío municipal.

## 9. Persistencia y consulta

Acceso a datos y reglas de búsqueda son responsabilidades distintas del dominio.

### 9.1 Repository — extraer el acceso a datos del dominio

**Motivo de cambio que separa.** Una clase de dominio que se persiste a sí misma mezcla reglas de negocio (qué es válido, qué se puede hacer) con reglas de infraestructura (qué motor de base de datos, qué esquema, qué consultas). Dos actores muy distintos.

**Síntoma.** Active Record: clases de dominio con métodos `Save`, `Load`, `Delete`.

```csharp
public class Cliente {
    public int Id { get; set; }
    public string Nombre { get; set; } = "";

    public void Save() {
        using var conn = new SqlConnection(...);
        conn.Open();
        // SQL aquí
    }

    public static Cliente Load(int id) { /* ... */ }
}
```

**Refactor: Repository.**

```csharp
public class Cliente {
    public int Id { get; private set; }
    public string Nombre { get; private set; } = "";

    public void CambiarNombre(string nuevo) {
        if (string.IsNullOrWhiteSpace(nuevo)) throw new DomainException();
        Nombre = nuevo;
    }
}

public interface IRepositorioClientes {
    Task<Cliente?> ObtenerAsync(int id, CancellationToken ct);
    Task GuardarAsync(Cliente c, CancellationToken ct);
}

public class RepositorioClientesEf(AppDbContext db) : IRepositorioClientes {
    public Task<Cliente?> ObtenerAsync(int id, CancellationToken ct) =>
        db.Clientes.FindAsync(id, ct).AsTask();

    public async Task GuardarAsync(Cliente c, CancellationToken ct) {
        db.Clientes.Update(c);
        await db.SaveChangesAsync(ct);
    }
}
```

**Por qué satisface SRP.** `Cliente` cambia si cambian las reglas del negocio. `RepositorioClientesEf` cambia si cambian las decisiones de infraestructura. La interfaz `IRepositorioClientes` aísla al dominio de cualquier conocimiento sobre EF, SQL o el motor.

> **Un beneficio inmediato es testabilidad.** Una implementación en memoria del mismo contrato permite probar la lógica de negocio sin tocar la base:
> ```csharp
> public class InMemoryRepoClientes : IRepositorioClientes {
>     private readonly Dictionary<int, Cliente> data = new();
>     public Task<Cliente?> ObtenerAsync(int id, CancellationToken ct) =>
>         Task.FromResult(data.GetValueOrDefault(id));
>     public Task GuardarAsync(Cliente c, CancellationToken ct) {
>         data[c.Id] = c;
>         return Task.CompletedTask;
>     }
> }
> ```

### 9.2 Unit of Work — agrupar cambios y confirmarlos en una transacción

**Motivo de cambio que separa.** "Hacer cambios" y "confirmar cambios" son responsabilidades distintas. Un servicio puede querer ejecutar varias operaciones de dominio y recién al final confirmar todo (o cancelar todo) como una unidad.

En EF Core, `DbContext` cumple en gran medida los roles de Repository **y** Unit of Work:

```csharp
context.Contactos.Add(contacto);
context.Contactos.Remove(contactoViejo);
context.Contactos.Update(contactoEditado);

await context.SaveChangesAsync();  // ← acá se confirma todo en una transacción
```

Hasta `SaveChangesAsync()`, EF Core sólo está registrando cambios en su change tracker.

```text
DbContext
    ├─► rastrea entidades
    ├─► detecta cambios
    ├─► agrupa operaciones
    └─► confirma con SaveChangesAsync()
```

**Por qué satisface SRP.** El servicio de dominio decide qué operación de negocio hacer; el `DbContext` rastrea y confirma cambios; la base de datos los persiste físicamente. El servicio no arma SQL, no abre conexiones y no maneja manualmente el estado de cada entidad.

### 9.3 Specification — extraer las reglas de criterio del dominio

**Motivo de cambio que separa.** Las reglas de "qué cuenta como X" tienden a vivir en consultas o en condicionales del repositorio. Eso mezcla lógica de negocio con detalles de cómo se ejecuta la búsqueda.

**Síntoma.** Métodos de repositorio con nombres como `ObtenerProductosActivosDeCategoriaConPrecioMayor`, que crecen sin parar.

```csharp
public interface IRepositorioProductos {
    Task<List<Producto>> ObtenerActivosAsync();
    Task<List<Producto>> ObtenerActivosDeCategoriaAsync(int catId);
    Task<List<Producto>> ObtenerActivosConPrecioMayorAsync(decimal min);
    Task<List<Producto>> ObtenerActivosDeCategoriaConPrecioMayorAsync(int catId, decimal min);
    // sigue creciendo...
}
```

**Refactor: Specification.** Cada criterio es una clase y se componen.

```csharp
public abstract class Specification<T> {
    public abstract Expression<Func<T, bool>> AsExpression();

    public Specification<T> And(Specification<T> otra) =>
        new AndSpec<T>(this, otra);
}

public class ProductoActivo : Specification<Producto> {
    public override Expression<Func<Producto, bool>> AsExpression() =>
        p => p.Activo;
}

public class ProductoEnCategoria(int catId) : Specification<Producto> {
    public override Expression<Func<Producto, bool>> AsExpression() =>
        p => p.CategoriaId == catId;
}

public class ProductoConPrecioMayor(decimal min) : Specification<Producto> {
    public override Expression<Func<Producto, bool>> AsExpression() =>
        p => p.Precio > min;
}

// Uso compuesto
var spec = new ProductoActivo()
    .And(new ProductoEnCategoria(3))
    .And(new ProductoConPrecioMayor(1000));

var resultados = await db.Productos.Where(spec.AsExpression()).ToListAsync();
```

**Por qué satisface SRP.** Cada specification responde a una regla de negocio; cambia si cambia la regla. El repositorio queda con una API genérica (`Buscar(Specification<T>)`) que no necesita crecer con cada combinación nueva.

## 10. Adaptación y envoltura

A veces la responsabilidad extraída es la adaptación entre dos contratos, o el agregado de comportamiento transversal sin contaminar al original.

### 10.1 Adapter — extraer la traducción entre contratos

**Motivo de cambio que separa.** El código que usa una librería externa se acopla a su API. Si la librería cambia o se reemplaza, hay que modificar todos los consumidores.

**Síntoma.** Llamadas directas a un SDK de tercero esparcidas por la lógica de negocio.

```csharp
// Acoplado al SDK de Twilio
public class ServicioNotificaciones(TwilioRestClient cliente) {
    public void EnviarSms(string telefono, string texto) {
        MessageResource.Create(
            body: texto,
            from: new PhoneNumber("+5491100000000"),
            to:   new PhoneNumber(telefono),
            client: cliente);
    }
}
```

**Refactor: Adapter.**

```csharp
public interface IEnviadorSms {
    Task EnviarAsync(string telefono, string texto, CancellationToken ct);
}

public class EnviadorSmsTwilio(TwilioRestClient cliente, string remitente) : IEnviadorSms {
    public Task EnviarAsync(string telefono, string texto, CancellationToken ct) =>
        MessageResource.CreateAsync(
            body: texto,
            from: new PhoneNumber(remitente),
            to:   new PhoneNumber(telefono),
            client: cliente);
}

public class ServicioNotificaciones(IEnviadorSms enviador) {
    public Task EnviarSms(string telefono, string texto, CancellationToken ct) =>
        enviador.EnviarAsync(telefono, texto, ct);
}
```

**Por qué satisface SRP.** `EnviadorSmsTwilio` es responsable de traducir entre nuestra abstracción y el SDK de Twilio; cambia si cambia el SDK. `ServicioNotificaciones` es responsable de la lógica de negocio de notificaciones; cambia si cambian las reglas. Si mañana migramos a otro proveedor, escribimos `EnviadorSmsAws` y cambiamos la línea de registro en DI: nada más.

> Todo SDK trae su zoológico; mejor dejarlo en una jaula.

### 10.2 Decorator — extraer comportamiento transversal

**Motivo de cambio que separa.** Logging, caché, retry, métricas, autorización, transaccionalidad: son preocupaciones que querés aplicar a operaciones existentes pero que **no son responsabilidad** de esas operaciones. Si las metés adentro, contaminás cada clase con conocimientos que no le competen.

**Síntoma.** Métodos que empiezan con `_logger.LogInformation`, `_cache.Get`, `_metrics.Increment`, antes de hacer lo suyo.

```csharp
public class RepoProductos(SqlConnection conn, IMemoryCache cache, ILogger logger) {
    public async Task<Producto?> Obtener(int id) {
        logger.LogInformation("Buscando producto {Id}", id);
        if (cache.TryGetValue($"prod:{id}", out Producto? p)) {
            logger.LogInformation("Hit de caché");
            return p;
        }
        var sw = Stopwatch.StartNew();
        p = await conn.QuerySingleOrDefaultAsync<Producto>(
            "SELECT * FROM productos WHERE id=@id", new { id });
        logger.LogInformation("Query en {Ms}ms", sw.ElapsedMilliseconds);
        if (p is not null) cache.Set($"prod:{id}", p);
        return p;
    }
}
```

Tres responsabilidades en una: acceder a datos, cachear, loguear.

**Refactor: Decorator.** Cada capa implementa el mismo contrato y delega al siguiente.

```csharp
public interface IRepoProductos {
    Task<Producto?> ObtenerAsync(int id, CancellationToken ct);
}

public class RepoProductosSql(SqlConnection conn) : IRepoProductos {
    public Task<Producto?> ObtenerAsync(int id, CancellationToken ct) =>
        conn.QuerySingleOrDefaultAsync<Producto?>(
            "SELECT * FROM productos WHERE id=@id", new { id });
}

public class RepoProductosLogging(IRepoProductos inner, ILogger<RepoProductosLogging> log)
    : IRepoProductos {
    public async Task<Producto?> ObtenerAsync(int id, CancellationToken ct) {
        log.LogInformation("Buscando producto {Id}", id);
        var sw = Stopwatch.StartNew();
        var r = await inner.ObtenerAsync(id, ct);
        log.LogInformation("{Resultado} en {Ms}ms",
            r is null ? "no encontrado" : "encontrado", sw.ElapsedMilliseconds);
        return r;
    }
}

public class RepoProductosCache(IRepoProductos inner, IMemoryCache cache) : IRepoProductos {
    public Task<Producto?> ObtenerAsync(int id, CancellationToken ct) =>
        cache.GetOrCreateAsync($"prod:{id}", _ => inner.ObtenerAsync(id, ct));
}

// Composición
IRepoProductos repo =
    new RepoProductosLogging(
        new RepoProductosCache(
            new RepoProductosSql(conn),
            cache),
        logger);
```

**Por qué satisface SRP.** `RepoProductosSql` cambia si cambia la query o el esquema. `RepoProductosCache` cambia si cambia la política de caché. `RepoProductosLogging` cambia si cambia qué se loguea. Tres motivos, tres clases, tres archivos. Y la composición es flexible: en tests podés usar sólo el SQL; en producción los tres.

### 10.3 Proxy — extraer el control de acceso o la carga diferida

**Motivo de cambio que separa.** Un objeto caro de crear o sensible al acceso requiere lógica especial alrededor de su uso (lazy loading, autorización, control remoto). Esa lógica no debería vivir adentro del objeto.

**Síntoma.** Lógica de "¿está cargado? ¿puedo acceder?" mezclada con la lógica de uso.

```csharp
public interface IDocumento {
    string Contenido { get; }
}

public class DocumentoReal(string ruta) : IDocumento {
    public string Contenido { get; } = File.ReadAllText(ruta);  // costoso, eager
}

public class DocumentoLazy(string ruta) : IDocumento {
    private string? cache;
    public string Contenido => cache ??= File.ReadAllText(ruta);
}

public class DocumentoProtegido(IDocumento inner, IAutorizador auth) : IDocumento {
    public string Contenido =>
        auth.PuedeLeer() ? inner.Contenido : throw new UnauthorizedAccessException();
}
```

**Por qué satisface SRP.** El proxy se ocupa del control (lazy load, autorización, instrumentación remota); el documento real se ocupa de su contenido.

> **Decorator vs Proxy.** La diferencia es semántica: el decorator agrega comportamiento ortogonal al original (logging, caché, retry); el proxy controla el acceso al original (lazy, autorización, remoto). Estructuralmente son casi idénticos.

# Parte 3 — Patrones del ecosistema .NET

Más allá del catálogo clásico, el ecosistema .NET tiene un conjunto de patrones recurrentes que conviene conocer porque aparecen sistemáticamente en plantillas de proyectos modernas (ASP.NET Core, Worker Services, MAUI).

## 11. Dependency Injection — separar uso de creación

**Motivo de cambio que separa.** Una clase que crea sus propias dependencias queda acoplada a la implementación concreta. Si mañana cambia el almacenamiento de JSON a SQLite, hay que modificar el servicio aunque su lógica de negocio no haya cambiado.

```csharp
public sealed class ContactoService {
    private readonly JsonContactoRepository repository = new("contactos.json");

    public void Agregar(Contacto contacto) {
        var contactos = repository.ObtenerTodos();
        contactos.Add(contacto);
        repository.GuardarTodos(contactos);
    }
}
```

**Refactor: DI.** La clase recibe sus dependencias y no sabe qué implementación concreta usa.

```csharp
public sealed class ContactoService(IContactoRepository repository) {
    public void Agregar(Contacto contacto) {
        var contactos = repository.ObtenerTodos();
        contactos.Add(contacto);
        repository.GuardarTodos(contactos);
    }
}
```

En ASP.NET Core, Worker Services y MAUI, el contenedor de DI ya viene integrado:

```csharp
builder.Services.AddSingleton<IContactoRepository>(
    new JsonContactoRepository("contactos.json"));

builder.Services.AddScoped<ContactoService>();

// Uso (Minimal API)
app.MapPost("/contactos", (Contacto contacto, ContactoService service) => {
    service.Agregar(contacto);
    return Results.Ok();
});
```

**Por qué satisface SRP.** El servicio ejecuta el caso de uso. El repositorio persiste. El contenedor de DI construye y conecta objetos. La responsabilidad de armar el grafo de objetos sale del servicio y queda en la composición de la aplicación.

> **Tres tiempos de vida** que hay que conocer: `Singleton` (una instancia para toda la app), `Scoped` (una por request HTTP), `Transient` (una por inyección). Se eligen según si la dependencia tiene estado y si ese estado debe compartirse.

## 12. Options Pattern — separar configuración de lógica

**Motivo de cambio que separa.** En .NET la configuración suele venir de `appsettings.json`, variables de entorno, secretos o parámetros externos. Si cada clase parsea configuración por su lado con strings mágicos, queda contaminada con detalles de presentación de la configuración.

**Mal enfoque:**

```csharp
public sealed class JsonContactoRepository(IConfiguration configuration) : IContactoRepository {
    public List<Contacto> ObtenerTodos() {
        string path = configuration["Agenda:ArchivoContactos"]!;
        // leer archivo...
        return [];
    }
}
```

**Refactor con Options Pattern:**

```csharp
public sealed class AgendaOptions {
    public string ArchivoContactos { get; set; } = "contactos.json";
}

// Registro
builder.Services.Configure<AgendaOptions>(
    builder.Configuration.GetSection("Agenda"));

// Uso
public sealed class JsonContactoRepository(IOptions<AgendaOptions> options) : IContactoRepository {
    private readonly AgendaOptions opts = options.Value;

    public List<Contacto> ObtenerTodos() {
        string path = opts.ArchivoContactos;
        // leer archivo...
        return [];
    }
}
```

**Por qué satisface SRP.** El archivo `appsettings.json` provee valores. `AgendaOptions` modela tipadamente la configuración. La infraestructura de Options enlaza ambos. El repositorio usa configuración ya modelada, no la busca con strings mágicos en cada esquina.

## 13. Middleware Pipeline — separar responsabilidades HTTP

ASP.NET Core procesa cada request usando una cadena de middlewares, donde **cada uno tiene una sola responsabilidad**.

```text
Request
    ↓
Exception handling
    ↓
HTTPS redirection
    ↓
Authentication
    ↓
Authorization
    ↓
Rate limiting
    ↓
Endpoint
```

```csharp
app.UseExceptionHandler("/error");
app.UseHttpsRedirection();
app.UseAuthentication();
app.UseAuthorization();
app.UseRateLimiter();

app.MapControllers();
```

Un middleware propio:

```csharp
public sealed class RequestLoggingMiddleware(
    RequestDelegate next,
    ILogger<RequestLoggingMiddleware> logger) {

    public async Task InvokeAsync(HttpContext context) {
        logger.LogInformation("Request: {Method} {Path}",
            context.Request.Method, context.Request.Path);
        await next(context);
    }
}

app.UseMiddleware<RequestLoggingMiddleware>();
```

**Por qué satisface SRP.** El middleware de autenticación no sabe nada de autorización; el de rate limiting no sabe nada de logging. El endpoint no debería ocuparse de autenticar, autorizar, loguear, manejar errores globales y además resolver negocio. Para eso está el pipeline.

## 14. Filters — separar lógica transversal de controladores

Los filtros (`IActionFilter`, `IExceptionFilter`, `IAuthorizationFilter`) sacan lógica transversal de los controladores **sin modificarlos**.

```csharp
public sealed class LogActionFilter(ILogger<LogActionFilter> logger) : IActionFilter {
    public void OnActionExecuting(ActionExecutingContext context) =>
        logger.LogInformation("Ejecutando {Action}", context.ActionDescriptor.DisplayName);

    public void OnActionExecuted(ActionExecutedContext context) =>
        logger.LogInformation("Terminó {Action}", context.ActionDescriptor.DisplayName);
}

// Aplicado globalmente a todos los controladores
builder.Services.AddControllers(opts => opts.Filters.Add<LogActionFilter>());

// O selectivamente
[ServiceFilter(typeof(LogActionFilter))]
public sealed class ContactosController(ContactoService service) : ControllerBase {
    [HttpPost("contactos")]
    public IActionResult Crear(Contacto contacto) {
        service.Agregar(contacto);
        return Ok();
    }
}
```

**Por qué satisface SRP.** El controlador traduce HTTP a caso de uso. Los filtros se ocupan de logging, validación, errores y autorización. El servicio resuelve el caso de uso. Sin filtros, el controlador termina siendo repositorio, validador, logger, mapper y psicólogo familiar.

> Los filtros y los middlewares se parecen, pero no son lo mismo. Los middlewares operan sobre el `HttpContext` crudo y se ejecutan para todas las requests. Los filtros operan en el contexto de MVC/Controllers (después de model binding) y pueden aplicarse selectivamente por controller o por action.

## 15. DTO + Mapper — separar contrato externo de modelo interno

**Motivo de cambio que separa.** Una entidad de dominio o de EF Core puede tener detalles internos que no querés exponer en una API pública. Y al revés: si la API tiene un formato pensado para el consumidor, ese formato no tiene por qué dictar cómo se modela el dominio.

```csharp
// Entidad interna
public sealed class Contacto {
    public int Id { get; set; }
    public string Nombre { get; set; } = "";
    public string Apellido { get; set; } = "";
    public string Email { get; set; } = "";
    public DateTime CreadoEn { get; set; }
    public Guid TenantId { get; set; }
}

// Contrato externo
public sealed class ContactoDto {
    public int Id { get; set; }
    public string NombreCompleto { get; set; } = "";
    public string Email { get; set; } = "";
}

// Mapper
public static class ContactoMapper {
    public static ContactoDto ToDto(Contacto c) => new() {
        Id             = c.Id,
        NombreCompleto = $"{c.Nombre} {c.Apellido}",
        Email          = c.Email
    };
}

// Uso
List<ContactoDto> resultado = contactos.Select(ContactoMapper.ToDto).ToList();
```

**Por qué satisface SRP.** `Contacto` representa el modelo interno. `ContactoDto` representa el contrato externo. `ContactoMapper` traduce entre ambos. Si cambia la API pública (agregar un campo, cambiar el formato del nombre), no cambia el dominio. Si cambia el dominio (agregar `TenantId`), no cambia el contrato externo.

## 16. MVVM — separar UI, estado e interacción

MVVM se usa intensamente en .NET desktop/mobile (WPF, MAUI, Avalonia, WinUI). Separa la aplicación así:

```text
View ←──binding──→ ViewModel ───usa──→ Service ───usa──→ Repository
```

| Pieza      | Responsabilidad                  |
|------------|----------------------------------|
| View       | Dibujar la interfaz              |
| ViewModel  | Estado y acciones de la pantalla |
| Model      | Datos y reglas del dominio       |
| Service    | Casos de uso                     |
| Repository | Persistencia                     |

```csharp
public sealed class ContactoViewModel(ContactoService service) : INotifyPropertyChanged {
    private string nombre = "";
    public string Nombre {
        get => nombre;
        set { nombre = value; OnPropertyChanged(); }
    }

    public string Apellido { get; set; } = "";

    public ICommand GuardarCommand { get; } = new GuardarContactoCommand(service);

    public event PropertyChangedEventHandler? PropertyChanged;
    private void OnPropertyChanged([CallerMemberName] string? p = null) =>
        PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(p));
}
```

**Por qué satisface SRP.** La View dibuja. El ViewModel maneja estado de pantalla y comandos. El Command representa una acción ejecutable. El Service resuelve el caso de uso. El Repository persiste. Esto evita meter consultas SQL, parseo JSON y validaciones dentro del code-behind. Code-behind cargado es humedad en los cimientos.

# Parte 4 — SOLID en la BCL: los patrones puestos a producción

La **BCL** (Base Class Library) de .NET — `List<T>`, `Dictionary<TKey, TValue>`, `Stream`, `FileStream`, `StreamReader`, `IEnumerable<T>`, LINQ — fue diseñada por ingenieros que vienen aplicando estos principios hace décadas, y está pulida por millones de desarrolladores que la usan todos los días.

Por eso es un **laboratorio fantástico** para ver SOLID en la práctica. Hay un detalle importante: **no vas a ver "SOLID puro de libro"**, porque la BCL también carga con compatibilidad hacia atrás, rendimiento y ergonomía. Vas a ver aplicaciones claras y alguna que otra concesión pragmática. Eso es parte del aprendizaje: **los principios guían, no mandan**.

## 17. SRP en .NET: cada pieza con su trabajo

### 17.1 El trío `FileStream` + `StreamReader` + `TextReader`

Pensemos en el problema de **leer líneas de un archivo de texto**. Suena simple, pero descomponiéndolo desde primeros principios hay tres responsabilidades distintas:

1. **Acceder a los bytes** que viven en un disco físico (sectores, handles del SO, buffering, permisos).
2. **Convertir bytes en caracteres** usando una codificación (UTF-8, UTF-16, Latin-1).
3. **Agrupar caracteres en líneas** (detectar `\r\n`, `\n`, `\r`).

La BCL tiene **una clase por responsabilidad**:

```csharp
// Responsabilidad 1: bytes desde/hacia un dispositivo
FileStream fs   = new FileStream("datos.txt", FileMode.Open);

// Responsabilidad 2: bytes ↔ caracteres (con encoding)
StreamReader sr = new StreamReader(fs, Encoding.UTF8);

// Responsabilidad 3: caracteres → líneas (heredada de TextReader)
string? linea   = sr.ReadLine();
```

Si todo estuviera en una única clase `ArchivoDeTexto`, tendría tres razones distintas para cambiar (sistema de archivos, encoding, detección de líneas). Por eso en .NET son tres clases.

### 17.2 `List<T>` vs `Enumerable` (LINQ)

`List<T>` tiene la responsabilidad de **almacenar** elementos en memoria (agregar, quitar, indexar, redimensionar). Nada más.

Pero queremos hacer cosas con las colecciones: filtrar, mapear, ordenar, agrupar. ¿Esas operaciones son responsabilidad de `List<T>`? **No.** Si lo fueran, cada colección (`List<T>`, `Queue<T>`, `Stack<T>`, `HashSet<T>`) tendría que reimplementar `Where`, `Select`, `OrderBy`, y cambiar la lógica de filtrado obligaría a tocar **todas** las colecciones.

Los diseñadores de LINQ pusieron esas operaciones en una clase **estática separada**: `System.Linq.Enumerable`. Todos los métodos son **métodos de extensión** sobre `IEnumerable<T>`:

```csharp
public static class Enumerable {
    public static IEnumerable<TSource> Where<TSource>(
        this IEnumerable<TSource> source,
        Func<TSource, bool> predicate) { /* ... */ }

    public static IEnumerable<TResult> Select<TSource, TResult>(
        this IEnumerable<TSource> source,
        Func<TSource, TResult> selector) { /* ... */ }
}
```

Resultado:

- `List<T>` sólo almacena. Cambia si cambia cómo se almacena.
- `Enumerable` sólo opera. Cambia si cambia cómo se opera.
- Una colección custom que implemente `IEnumerable<T>` hereda **gratis** todo LINQ.

Es SRP llevado al extremo más elegante.

### 17.3 Otros ejemplos breves

- **`DbContext` en EF Core** representa una sesión con la base de datos y expone los conjuntos como `DbSet<T>`. **No sabe** mapear columnas (lo delega a `IEntityTypeConfiguration<T>`), **no sabe** ejecutar SQL (lo delega al driver), **no sabe** lógica de negocio.
- **Handlers en Minimal API**: cada endpoint es un método con una única responsabilidad — manejar un tipo específico de request. Si cambia la búsqueda, sólo cambia ese método.
- **Middlewares en ASP.NET Core**: ya vistos en sección 13. Cada uno una responsabilidad.

## 18. OCP en .NET: streams y LINQ

El Open/Closed Principle dice que las clases deben estar **abiertas a la extensión, cerradas a la modificación**. Este principio es el que más brilla en .NET.

### 18.1 `Stream` como jerarquía extensible

La clase base `System.IO.Stream` es una abstracción mínima: "un lugar de donde puedo leer bytes y a donde puedo escribir bytes":

```csharp
public abstract class Stream : IDisposable {
    public abstract bool CanRead  { get; }
    public abstract bool CanWrite { get; }
    public abstract bool CanSeek  { get; }
    public abstract int  Read(byte[] buffer, int offset, int count);
    public abstract void Write(byte[] buffer, int offset, int count);
    // ...
}
```

A partir de esa base, la BCL (y cualquier librería de terceros) agrega implementaciones **sin modificar `Stream`**:

- `FileStream` — bytes desde/hacia un archivo.
- `MemoryStream` — bytes en un `byte[]`.
- `NetworkStream` — bytes desde/hacia un socket.
- `GZipStream`, `DeflateStream`, `BrotliStream` — compresión envolviendo otro stream.
- `CryptoStream` — cifrado envolviendo otro stream.
- `BufferedStream` — buffering para cualquier otro stream.

El detalle clave: `GZipStream` **no es un archivo comprimido**. Es un `Stream` que, cuando le pedís bytes, los saca de **otro stream** y los descomprime al vuelo. Esto permite componer:

```csharp
using var fs      = new FileStream("datos.gz", FileMode.Open);
using var descomp = new GZipStream(fs, CompressionMode.Decompress);
using var lector  = new StreamReader(descomp, Encoding.UTF8);

string? linea;
while ((linea = lector.ReadLine()) != null)
    Console.WriteLine(linea);
```

Cada línea agrega una capa **sin tocar las clases de abajo**. Si necesitás leer un archivo comprimido y además cifrado, metés un `CryptoStream` en el medio. Ninguna clase sabe de la existencia de las otras, pero todas colaboran porque dependen de la misma abstracción.

> Esto es **el patrón Decorator (sección 10.2) en su forma más limpia**, y es OCP aplicado: se pueden agregar capacidades nuevas (cifrado, compresión, buffering, throttling) sin modificar una sola línea de `Stream`, `FileStream` ni `StreamReader`.

### 18.2 LINQ sobre `IEnumerable<T>`

El mismo patrón en otro dominio. LINQ tiene cerca de 50 operadores. Todos trabajan sobre `IEnumerable<T>` y devuelven `IEnumerable<T>`. Eso permite **agregar operadores nuevos sin tocar la BCL**:

```csharp
public static class MisExtensiones {
    public static IEnumerable<T> SaltearCada<T>(this IEnumerable<T> fuente, int n) {
        int i = 0;
        foreach (var x in fuente)
            if (i++ % n != 0) yield return x;
    }
}

// Uso compuesto con operadores estándar
var pares = numeros.Where(x => x % 2 == 0).SaltearCada(3).Select(x => x * 10);
```

`SaltearCada` se compone naturalmente con `Where` y `Select` porque todos hablan el mismo idioma: `IEnumerable<T>`. Microsoft no previó tu operador, y sin embargo funciona.

### 18.3 `IQueryable<T>` y especificaciones en EF Core

`IQueryable<T>` es el equivalente de `IEnumerable<T>` para consultas que se ejecutan en la base de datos. Permite agregar operadores en el mismo estilo:

```csharp
public static class ProductoQueryExtensions {
    public static IQueryable<Producto> Activos(this IQueryable<Producto> q)
        => q.Where(p => p.Activo);

    public static IQueryable<Producto> DeLaCategoria(this IQueryable<Producto> q, int catId)
        => q.Where(p => p.CategoriaId == catId);

    public static IQueryable<Producto> OrdenadosPorPrecio(this IQueryable<Producto> q)
        => q.OrderBy(p => p.Precio);
}

// Se componen igual que LINQ
var resultado = await db.Productos
    .Activos()
    .DeLaCategoria(3)
    .OrdenadosPorPrecio()
    .ToListAsync();
```

EF Core no fue modificado. Vos no modificaste `DbSet<T>`. Sólo extendiste el sistema. La SQL generada incluirá todos los filtros correctamente.

### 18.4 Componentes en Blazor

Blazor está diseñado para extenderse sin modificar lo existente:

```razor
@* BtnBase.razor — componente base, no se modifica *@
<button class="btn @CssExtra" @onclick="OnClick">@Texto</button>

@code {
    [Parameter] public string         Texto    { get; set; } = "";
    [Parameter] public string         CssExtra { get; set; } = "";
    [Parameter] public EventCallback  OnClick  { get; set; }
}
```

```razor
@* BtnPrimario.razor — extiende sin modificar BtnBase *@
<BtnBase Texto="@Texto" CssExtra="btn-primary" OnClick="@OnClick" />
```

`BtnBase` nunca se toca. Agregar `BtnSecundario`, `BtnLink` o `BtnIcono` sólo requiere crear un componente nuevo que componga.

## 19. LSP en .NET: la concesión pragmática de Stream

El Liskov Substitution Principle dice que **un subtipo debe poder usarse en lugar de su tipo base sin sorpresas**. En .NET hay ejemplos limpios y uno famoso de violación parcial que vale la pena entender.

### 19.1 `IEnumerable<T>` y `foreach` (LSP en su forma más pura)

`IEnumerable<T>` es la abstracción más usada de .NET. Su contrato es simple: "podés pedirme un enumerador, y el enumerador te devuelve elementos de a uno". Un `foreach` no sabe si está iterando una `List<T>`, un `HashSet<T>`, los valores de un `Dictionary<K,V>`, un archivo línea por línea con `File.ReadLines()`, o el resultado de un `Where().Select()`. Y no le importa:

```csharp
void Imprimir<T>(IEnumerable<T> fuente) {
    foreach (var item in fuente)
        Console.WriteLine(item);
}

Imprimir(new List<int> { 1, 2, 3 });           // funciona
Imprimir(new HashSet<int> { 1, 2, 3 });        // funciona
Imprimir(File.ReadLines("datos.txt"));         // funciona (¡streaming!)
Imprimir(Enumerable.Range(1, 1_000_000));      // funciona (lazy)
```

**Todas las implementaciones de `IEnumerable<T>` son intercambiables** desde el punto de vista del cliente. Esto es LSP en su forma más pura.

### 19.2 `Stream` — el caso interesante (violación parcial)

`Stream` define métodos como `Seek`, `Position` y `Length`. Pero no todos los streams pueden posicionarse: un `NetworkStream` o un `GZipStream` son secuenciales por naturaleza.

¿Qué se hizo en la BCL? Dos cosas. Primero, **propiedades de capacidad** — cada stream declara explícitamente qué puede hacer:

```csharp
public abstract bool CanRead  { get; }
public abstract bool CanWrite { get; }
public abstract bool CanSeek  { get; }
```

Segundo, **excepciones documentadas**: si llamás a `Seek` en un stream que no soporta seek, tira `NotSupportedException`.

```csharp
using var red = new NetworkStream(socket);
red.Position = 0; // 💥 NotSupportedException
```

Estrictamente hablando, **esto viola LSP**: un método que recibe un `Stream` no puede confiar en que `Seek` funcione, tiene que chequear `CanSeek` primero. La sustituibilidad del contrato base no es completa.

¿Por qué se hizo así? Porque la alternativa era partir `Stream` en `ISeekableStream` vs `IForwardOnlyStream`, y eso habría complicado enormemente el uso común. Los ingenieros de .NET priorizaron **ergonomía** sobre pureza del principio. Es una concesión consciente.

> **La lección a llevarse.** Cuando veas este patrón (capacidades declaradas + excepciones para lo no soportado), sabé que es un compromiso. A veces es razonable, a veces es señal de que la jerarquía está mal pensada. Si tu subtipo tira `NotSupportedException` en la mitad de los métodos, probablemente conviene dividir la interfaz.

### 19.3 `DbSet<T>` como `IQueryable<T>` en EF Core

`DbSet<T>` implementa `IQueryable<T>` e `IEnumerable<T>`. Cualquier método que acepte `IQueryable<Producto>` puede recibir un `DbSet<Producto>` real **o una lista en memoria** en los tests:

```csharp
static async Task<List<Producto>> BuscarActivos(IQueryable<Producto> fuente)
    => await fuente.Where(p => p.Activo).ToListAsync();

// Producción
var produccion = await BuscarActivos(db.Productos);

// Test
var enMemoria = new List<Producto> {
    new() { Nombre = "A", Activo = true  },
    new() { Nombre = "B", Activo = false }
}.AsQueryable();
var test = await BuscarActivos(enMemoria);
```

`BuscarActivos` no sabe si habla con SQL Server o con una lista. Ambos subtipos son intercambiables.

## 20. ISP en .NET: la jerarquía de colecciones

El Interface Segregation Principle dice que **los clientes no deberían depender de métodos que no usan**. .NET segregó sus colecciones en interfaces chicas y específicas.

### 20.1 La jerarquía de `System.Collections.Generic`

```csharp
public interface IEnumerable<out T> {                        // sólo iterar
    IEnumerator<T> GetEnumerator();
}

public interface IReadOnlyCollection<out T> : IEnumerable<T> {   // + contar
    int Count { get; }
}

public interface IReadOnlyList<out T> : IReadOnlyCollection<T> { // + acceder por índice (readonly)
    T this[int index] { get; }
}

public interface ICollection<T> : IEnumerable<T> {           // + modificar
    int Count { get; }
    void Add(T item);
    bool Remove(T item);
    void Clear();
    bool Contains(T item);
    bool IsReadOnly { get; }
}

public interface IList<T> : ICollection<T> {                 // + indexar y modificar
    T this[int index] { get; set; }
    int IndexOf(T item);
    void Insert(int index, T item);
    void RemoveAt(int index);
}
```

Esto permite expresar **exactamente** lo que necesita cada método:

```csharp
void Imprimir<T>(IEnumerable<T> items) { /* sólo iterar */ }

string Describir<T>(IReadOnlyCollection<T> items)             // sé cuántos hay, no modifico
    => $"Tengo {items.Count} elementos";

T ObtenerTercero<T>(IReadOnlyList<T> items) => items[2];      // leo por índice, no modifico

void AgregarVarios<T>(ICollection<T> items, IEnumerable<T> nuevos) { // necesito poder agregar
    foreach (var n in nuevos) items.Add(n);
}
```

Con interfaces segregadas, el tipo del parámetro **documenta qué vas a hacer con él**. En una API pública, aceptar `IReadOnlyList<Pedido>` en lugar de `List<Pedido>` es contrato y documentación al mismo tiempo: garantiza que el método no va a modificar la lista del llamador.

### 20.2 `IDisposable` vs `IAsyncDisposable`

ISP en chiquito, pero con mucha claridad. El contrato "liberar recursos" empezó siendo:

```csharp
public interface IDisposable {
    void Dispose();
}
```

Con async/await, algunos recursos necesitan liberarse asincrónicamente. ¿Qué se hizo? **No se modificó `IDisposable`**. Se creó una interfaz nueva:

```csharp
public interface IAsyncDisposable {
    ValueTask DisposeAsync();
}
```

Una clase puede implementar una, la otra, o las dos. Y `using` en C# soporta las dos:

```csharp
using       var fs  = new FileStream("a.txt", FileMode.Open);  // Dispose()
await using var fs2 = new FileStream("a.txt", FileMode.Open);  // DisposeAsync()
```

Si `IDisposable` hubiera tenido siempre los dos métodos, **toda clase que la implementa** habría tenido que incluir un `DisposeAsync` que nadie pidió.

### 20.3 Las tres interfaces del bootstrapping de ASP.NET Core

```csharp
var builder = WebApplication.CreateBuilder(args);

// IServiceCollection — sólo registrar dependencias
builder.Services.AddDbContext<TiendaContext>(opts => opts.UseSqlite("..."));
builder.Services.AddScoped<IRepositorioProductos, RepositorioEF>();

var app = builder.Build();

// IApplicationBuilder — sólo configurar el pipeline
app.UseHttpsRedirection();
app.UseAuthentication();

// IEndpointRouteBuilder — sólo mapear rutas
app.MapControllers();
app.MapGet("/salud", () => "OK");

app.Run();
```

`IServiceCollection` no sabe nada de middleware. `IApplicationBuilder` no sabe nada de rutas. Cada interfaz tiene exactamente las operaciones que su responsabilidad requiere.

## 21. DIP en .NET: inyección de dependencias por todos lados

El Dependency Inversion Principle dice que **el código de alto nivel no debe depender de detalles concretos, sino de abstracciones**. En .NET esto está integrado al punto de ser invisible.

### 21.1 `StreamReader` depende de `Stream`, no de `FileStream`

Mirá la firma del constructor:

```csharp
public StreamReader(Stream stream, Encoding encoding) { /* ... */ }
```

El parámetro es `Stream` — la abstracción. No `FileStream`, no `NetworkStream`. Esto es DIP puro y es la razón por la que se puede hacer todo esto:

```csharp
// Desde un archivo
var r1 = new StreamReader(new FileStream("a.txt", FileMode.Open));

// Desde un archivo comprimido
var r2 = new StreamReader(
    new GZipStream(new FileStream("a.gz", FileMode.Open), CompressionMode.Decompress));

// Desde una respuesta HTTP
using var http = new HttpClient();
var r3 = new StreamReader(await http.GetStreamAsync("https://..."));

// Desde memoria (típico en tests)
var bytes = Encoding.UTF8.GetBytes("hola\nmundo");
var r4    = new StreamReader(new MemoryStream(bytes));
```

`StreamReader` no sabe ni le importa de dónde vienen los bytes. Y eso da un regalo enorme para los tests: cualquier lógica que lea texto puede probarse pasándole un `MemoryStream` con datos fabricados, sin tocar el disco.

### 21.2 El contenedor de DI de ASP.NET Core

Ya lo vimos en la sección 11. Acá lo importante es ver que DIP **no es una opción** sino la forma esperada de trabajar:

```csharp
public class TarjetasController(
    ILogger<TarjetasController> logger,
    IRepositorioTarjetas        repo,
    INotificador                notificador) : ControllerBase {

    [HttpPost("debitar")]
    public async Task<IActionResult> Debitar(string numero, decimal monto) {
        var tarjeta = await repo.ObtenerAsync(numero);
        tarjeta.Debitar(monto);
        await repo.GuardarAsync(tarjeta);
        await notificador.NotificarAsync(tarjeta.Email, "Débito", $"${monto}");
        logger.LogInformation("Débito de {Monto} en {Numero}", monto, numero);
        return Ok();
    }
}
```

El controller **no hace `new` de nada**. Sólo depende de tres interfaces. Cuál implementación se usa se decide en `Program.cs`:

```csharp
builder.Services.AddScoped<IRepositorioTarjetas, RepositorioSqlServer>();
builder.Services.AddScoped<INotificador, NotificadorEmailSmtp>();
builder.Logging.AddConsole();
```

Si mañana cambia SQL Server por PostgreSQL, o email por SMS, **se cambia una línea**.

### 21.3 `IHttpClientFactory`

`HttpClient` tiene un problema conocido: instanciarlo a mano puede agotar sockets del SO. ASP.NET Core resuelve esto con `IHttpClientFactory`:

```csharp
builder.Services.AddHttpClient("Clima", client => {
    client.BaseAddress = new Uri("https://api.open-meteo.com/");
});

public class ServicioClima(IHttpClientFactory factory) {
    public async Task<string> ObtenerTemperaturaAsync() {
        var client = factory.CreateClient("Clima");
        return await client.GetStringAsync(
            "v1/forecast?latitude=-26.8&longitude=-65.2&current=temperature_2m");
    }
}
```

`ServicioClima` depende de `IHttpClientFactory`, no de un `HttpClient` concreto. En tests se reemplaza la factory sin hacer requests HTTP reales.

### 21.4 Corolario: `ILogger<T>` como ejemplo de los cinco principios juntos

`ILogger<T>` es un caso precioso de SOLID concentrado:

- Es una **interfaz chica** (ISP).
- Permite **implementaciones intercambiables** (LSP, DIP).
- Se pueden **agregar loggers nuevos** (archivo, Console, Seq, Datadog) sin modificar nada existente (OCP).
- Cada implementación tiene **una única responsabilidad**: registrar logs (SRP).

## 22. Mini caso: leer un archivo comprimido con LINQ

Antes del caso grande, un caso chiquito que sintetiza todo lo visto en la parte 4. Imaginemos un pipeline que procesa un archivo de **JSON Lines comprimido con GZip**, filtra registros y los agrega. Sin pensar explícitamente en SOLID, estás usando los cinco principios porque la BCL te los ofrece de fábrica:

```csharp
using System.IO;
using System.IO.Compression;
using System.Text;
using System.Text.Json;

public record Movimiento(string Tarjeta, string Tipo, decimal Monto);

public static class Procesador {
    public static decimal TotalDebitosGold(string rutaArchivoGz) {
        // [DIP] StreamReader depende de Stream, no de archivos
        // [OCP] Apilamos GZipStream sobre FileStream sin tocar nada
        // [SRP] Cada capa tiene una única responsabilidad
        using var archivo = new FileStream(rutaArchivoGz, FileMode.Open);
        using var descomp = new GZipStream(archivo, CompressionMode.Decompress);
        using var lector  = new StreamReader(descomp, Encoding.UTF8);

        // [LSP] El enumerable es intercambiable con cualquier IEnumerable<string>
        // [ISP] Sólo necesitamos iterar (IEnumerable), no modificar
        return LeerLineas(lector)
            .Select(linea => JsonSerializer.Deserialize<Movimiento>(linea)!)
            .Where(m => m.Tipo == "debito" && m.Tarjeta.StartsWith("GOLD"))
            .Sum(m => m.Monto);
    }

    private static IEnumerable<string> LeerLineas(TextReader lector) {
        string? linea;
        while ((linea = lector.ReadLine()) != null)
            yield return linea;
    }
}
```

Ocho líneas de lógica real. Cada decisión de diseño de la BCL te está ayudando:

- Si mañana el archivo viene de una API en vez de disco, cambiás `FileStream` por el stream de `HttpClient`. **El resto no se toca.**
- Si el archivo usa Brotli en vez de GZip, cambiás `GZipStream` por `BrotliStream`. **El resto no se toca.**
- Si en los tests querés probar con un string en memoria, usás `new StringReader(texto)`. **El resto no se toca.**
- Si aparece una regla nueva (agrupar por fecha), agregás un `.GroupBy()` al pipeline. **Nada existente se toca.**

Esa sensación de "todo compone como Lego" no es accidente: es el resultado de aplicar SOLID consistentemente durante 20 años.

## 23. Caso integrador: refactorizar una god class

Veamos un caso completo que combina varios patrones para extraer responsabilidades de una god class de venta online.

### 23.1 La god class original

```csharp
public class ServicioVentas(SqlConnection conn, ILogger logger) {

    public async Task<int> ProcesarVenta(VentaDto dto) {
        // 1. Validar
        if (dto.Items is null || dto.Items.Count == 0)
            throw new ArgumentException("Sin items");
        if (dto.ClienteId <= 0)
            throw new ArgumentException("Cliente inválido");

        // 2. Calcular total con descuento por tipo de cliente
        var subtotal = dto.Items.Sum(i => i.Precio * i.Cantidad);
        decimal descuento;
        var clienteRow = await conn.QuerySingleAsync(
            "SELECT tipo, email FROM clientes WHERE id=@id", new { id = dto.ClienteId });
        if      (clienteRow.tipo == "vip")     descuento = subtotal * 0.15m;
        else if (clienteRow.tipo == "premium") descuento = subtotal * 0.10m;
        else if (clienteRow.tipo == "regular") descuento = subtotal * 0.05m;
        else                                    descuento = 0;

        var total = subtotal - descuento;

        // 3. Procesar pago
        if (dto.MedioPago == "tarjeta") {
            var resp = await new HttpClient().PostAsJsonAsync(
                "https://api.tarjetas.com/cobro", new { total });
            if (!resp.IsSuccessStatusCode) throw new Exception("Pago rechazado");
        } else if (dto.MedioPago == "transferencia") {
            // ...
        }

        // 4. Persistir
        var ventaId = await conn.ExecuteScalarAsync<int>(
            "INSERT INTO ventas (cliente, total) VALUES (@c, @t); SELECT LAST_INSERT_ROWID();",
            new { c = dto.ClienteId, t = total });
        foreach (var item in dto.Items)
            await conn.ExecuteAsync("INSERT INTO venta_items ...", item);

        // 5. Notificar
        var smtp = new SmtpClient("smtp.empresa.com");
        smtp.Send("ventas@empresa.com", clienteRow.email,
                  "Su compra", $"Total: ${total}");

        // 6. Auditar
        logger.LogInformation("Venta {Id} procesada por {Total}", ventaId, total);
        await File.AppendAllTextAsync("audit.log",
            $"{DateTime.UtcNow:o}|venta|{ventaId}|{total}\n");

        return ventaId;
    }
}
```

Inventario de responsabilidades — y los actores asociados:

| # | Responsabilidad         | Actor                                |
|---|-------------------------|--------------------------------------|
| 1 | Validación              | Producto / Ventas                    |
| 2 | Política de descuento   | Marketing                            |
| 3 | Pago                    | Finanzas / Plataforma de pagos       |
| 4 | Persistencia            | Datos / Infraestructura              |
| 5 | Notificación al cliente | Customer Experience                  |
| 6 | Auditoría / logging     | Compliance / Observabilidad          |

Seis actores, seis razones distintas para que esta clase cambie. Cada cambio arriesga regresión en lo que ya funciona.

public class ServicioVentasConAuditoria(
    IServicioVentas inner,
    ILogger<ServicioVentasConAuditoria> log,
    IAuditor auditor) : IServicioVentas {

    public async Task<int> ProcesarVenta(VentaDto dto, CancellationToken ct) {
        var id = await inner.ProcesarVenta(dto, ct);

**Coordinación final → Facade limpio**:

```csharp
public class ServicioVentas(
    IValidador<VentaDto> validador,
    IRepositorioClientes repoClientes,
    IEnumerable<IPoliticaDescuento> politicas,
    IFabricaGateways fabricaGateways,
    IRepositorioVentas repoVentas,
    INotificadorCliente notificador) : IServicioVentas {

    public async Task<int> ProcesarVenta(VentaDto dto, CancellationToken ct) {
        validador.Validar(dto).LanzarSiInvalido();

        var cliente   = await repoClientes.ObtenerAsync(dto.ClienteId, ct);
        var subtotal  = dto.Items.Sum(i => i.Precio * i.Cantidad);
        var politica  = politicas.First(p => p.AplicaA(cliente));
        var total     = subtotal - politica.Calcular(subtotal);

        var gateway = fabricaGateways.Crear(dto.MedioPago);
        await gateway.CobrarAsync(total, ct);

        var venta = Venta.From(dto, total);
        var id    = await repoVentas.AgregarAsync(venta, ct);

        await notificador.EnviarAsync(cliente.Email, "Su compra",
                                      $"Total: ${total}", ct);
        return id;
    }
}
```

### 23.3 El resultado en términos de SRP

| Clase                          | Responsabilidad única                  | Cambia si...                            |
|--------------------------------|----------------------------------------|-----------------------------------------|
| `ServicioVentas`               | Orquestar el caso de uso               | Cambia el flujo del proceso             |
| `ValidadorVenta`               | Reglas de validación de DTO            | Cambian las reglas de validez           |
| `IPoliticaDescuento`           | Una política de descuento              | Cambia esa política                     |
| `IGatewayPago`                 | Cobrar por un medio                    | Cambia ese medio                        |
| `IFabricaGateways`             | Decidir qué gateway crear              | Aparece un nuevo medio                  |
| `RepositorioVentasSql`         | Persistencia de ventas                 | Cambia el esquema o el motor            |
| `NotificadorEmail`             | Adaptar SMTP a nuestra abstracción     | Cambia el SDK / proveedor               |
| `ServicioVentasConAuditoria`   | Logging y auditoría                    | Cambian las políticas de observabilidad |

Cada clase responde a un único actor. Las pruebas son chicas, mockeables sin esfuerzo. Agregar un nuevo medio de pago, una nueva política de descuento, o cambiar el proveedor de email son operaciones aditivas.

## 24. Heurísticas: cuándo y cuándo no

### 24.1 Tabla diagnóstico → patrón

Una guía rápida para reconocer cuándo aplicar cada patrón:

| Si veo...                                                                          | Probablemente necesite...   |
|------------------------------------------------------------------------------------|-----------------------------|
| `switch` o `if-else` enumerando alternativas con lógica sustancial en cada rama    | Strategy                    |
| Misma estructura de método repetida con variaciones puntuales                      | Template Method             |
| `if` sobre un campo `estado` o `tipo` repetido en varios métodos                   | State                       |
| Múltiples operaciones aplicadas a una jerarquía de tipos                           | Visitor                     |
| Operaciones que se invocan y quiero loguear, deshacer, encolar o programar         | Command                     |
| Cadena de validaciones / filtros con `if`s consecutivos                            | Chain of Responsibility     |
| `new` distinto según un parámetro, dentro del código de negocio                    | Factory                     |
| Constructores con muchos parámetros, varios opcionales                             | Builder                     |
| Lógica de orquestación entre varios servicios que se conocen mutuamente            | Mediator                    |
| Una clase que avisa a varias después de hacer algo                                 | Observer                    |
| El mismo bloque de código que enlaza varios objetos, replicado en muchos lugares   | Facade                      |
| Clases de dominio con métodos `Save`/`Load`                                        | Repository                  |
| Métodos de búsqueda combinatorios que crecen sin parar                             | Specification               |
| SDK externo invocado directamente desde la lógica de negocio                       | Adapter                     |
| Logging / caché / retry / métricas mezclados con el código de negocio              | Decorator                   |
| Lógica de "está cargado / puedo acceder" antes de cada uso                         | Proxy                       |

### 24.2 Reglas para no caer en sobrediseño

Los patrones tienen costo. Aplicarlos preventivamente o por gusto académico produce código pesado y poco legible.

1. **El patrón aparece después del diagnóstico**, nunca antes. Primero detectás SRP violado, después extraés, después reconocés que el resultado se llama Strategy o Repository.
2. **Si el refactor no se justifica por una violación real de SRP, no lo hagas.** Cuatro descuentos hardcodeados sin perspectiva de cambio están bien como un `switch`.
3. **Los nombres son secundarios.** Si extraés una clase y no es ningún patrón conocido, está bien igual. La lectura del código no se mejora porque el patrón tenga nombre.
4. **No mezcles patrones por moda.** Una clase no necesita ser Strategy + Decorator + Factory + Observer simultáneamente para ser "elegante". A menudo eso es síntoma de over-engineering.
5. **Releé después del refactor.** Si el código quedó más difícil de seguir que antes, deshacé el cambio. La meta es facilidad de cambio futuro, no purismo arquitectónico.
6. **No crear abstracciones antes de tiempo.** SRP no significa llenar todo de interfaces por deporte:
   ```text
   Mala señal:   IContactoServiceFactoryProviderManager
   Buena señal:  IContactoRepository, INotificador, IExportadorContactos
   ```
   La abstracción debe responder a una variación real o razonablemente probable.
7. **Nombres que revelen responsabilidad.** `JsonContactoRepository`, `WhatsAppNotificador`, `ContactoMapper` son buenos. `ContactoManager`, `AgendaHelper`, `Utils`, `GeneralService` muchas veces significan "no sé qué responsabilidad tiene esto, pero compila".

### 24.3 Conexión con los otros principios SOLID

Los patrones de extracción para SRP suelen colaborar con el resto:

- **OCP** se obtiene casi gratis cuando se aplica Strategy, Visitor, Decorator o Chain of Responsibility: agregar una nueva variante es agregar una clase, no modificar las existentes.
- **DIP** aparece cuando el patrón usa interfaces (Strategy, Repository, Adapter) e inyecta las implementaciones: el código de alto nivel depende de abstracciones.
- **ISP** guía a hacer las interfaces de los patrones lo más chicas posibles: un `IComando` con `Ejecutar` y `Deshacer`, un `IPoliticaDescuento` con dos métodos. Las interfaces de los patrones tienden a ser pequeñas por naturaleza.
- **LSP** entra cuando se escriben variantes: las implementaciones de `IPoliticaDescuento` tienen que cumplir el mismo contrato — todas devuelven un descuento sobre el subtotal, ninguna lanza excepción inesperada.

Los patrones bien aplicados no satisfacen sólo SRP: tiran del resto de SOLID en la dirección correcta como efecto colateral. Esa es una de las razones por las que el catálogo de patrones es persistente: no son trucos aislados, son las formas naturales que toma un diseño respetuoso de los principios.

### 24.4 Cierre

> Los patrones de POO son las **formas recurrentes** que toma la extracción de responsabilidades. SRP es el diagnóstico que motiva la extracción; el patrón es la geometría del resultado. No se aprenden patrones para aplicarlos: se aprenden para reconocerlos cuando aparecen, comunicarlos rápido, y no inventar variantes inferiores.
>
> El estudiante que entiende esta inversión causal — primero el problema, después el patrón — es el que sabe diseñar. El que recita el catálogo y lo aplica a todo, todavía no.

## Resumen final

```text
SRP
    └─► Una clase, una razón clara para cambiar

Patrones organizados por la responsabilidad que extraen:

    Política / algoritmo
        ├── Strategy                 → algoritmos intercambiables
        ├── Template Method          → estructura fija + pasos variables
        └── State                    → comportamiento por estado

    Operación
        ├── Command                  → operaciones como objetos
        ├── Chain of Responsibility  → cadena de pasos independientes
        └── Visitor                  → operaciones sobre jerarquía de tipos

    Creación
        ├── Factory                  → decidir qué construir
        └── Builder                  → construir objetos complejos

    Coordinación
        ├── Mediator                 → centralizar coordinación
        ├── Observer / Events        → publicar sin conocer consumidores
        └── Facade                   → simplificar subsistema

    Persistencia y consulta
        ├── Repository               → abstraer acceso a datos
        ├── Unit of Work             → agrupar y confirmar cambios
        └── Specification            → reglas de criterio componibles

    Adaptación y envoltura
        ├── Adapter                  → traducir entre contratos
        ├── Decorator                → comportamiento transversal
        └── Proxy                    → control de acceso o lazy

Patrones específicos del ecosistema .NET:
    ├── Dependency Injection         → separar uso de creación
    ├── Options Pattern              → configuración tipada
    ├── Middleware Pipeline          → responsabilidades HTTP encadenadas
    ├── Filters                      → lógica transversal de controllers
    ├── DTO + Mapper                 → contrato externo vs modelo interno
    └── MVVM                         → UI, estado y acciones separados

Regla final:
    └─► Si una clase cambia por varios motivos distintos,
        probablemente necesita que alguna responsabilidad salga de ahí.
        El nombre del patrón viene después.
```

La idea no es diseñar con más clases porque sí. La idea es que cada clase tenga un trabajo reconocible, testeable y reemplazable. El buen diseño no hace ruido: simplemente deja de romperse por cualquier pavada.
