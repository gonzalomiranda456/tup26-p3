## 1. Qué problema resuelve un backend

Una aplicación de consola corre en una computadora. Sus datos viven ahí. Si otra aplicación necesita leer o modificar esos datos, hace falta una forma clara de comunicarse.

Un **backend web** es un programa que:

1. escucha pedidos HTTP;
2. interpreta qué quiere el cliente;
3. ejecuta lógica de aplicación;
4. consulta o modifica datos;
5. devuelve una respuesta HTTP.

Esquema general:

```txt
Cliente HTTP
   ↓ request
Backend ASP.NET Core
   ↓ lógica
Base de datos SQLite
   ↑ datos
Backend ASP.NET Core
   ↑ response
Cliente HTTP
```

El cliente puede ser:

- un navegador;
- una app mobile;
- otro backend;
- un programa de consola;
- un test automatizado;
- una página HTML con JavaScript.

El contrato entre cliente y servidor es **HTTP**.

## 2. HTTP como protocolo de mensajes

HTTP es un protocolo basado en mensajes. En cada interacción hay dos mensajes:

```txt
Request  → pedido que envía el cliente
Response → respuesta que devuelve el servidor
```

### Request

Una request tiene esta estructura conceptual:

```txt
Método + ruta + versión
Headers
Body opcional
```

Ejemplo:

```http
POST /productos HTTP/1.1
Host: localhost:5000
Content-Type: application/json
Accept: application/json

{
  "nombre": "Monitor",
  "precio": 250000
}
```

Se lee así:

```txt
Quiero crear algo dentro de /productos.
Te envío datos en JSON.
Espero una respuesta JSON.
```

### Response

Una response tiene esta estructura conceptual:

```txt
Versión + status code
Headers
Body opcional
```

Ejemplo:

```http
HTTP/1.1 201 Created
Content-Type: application/json
Location: /productos/7

{
  "id": 7,
  "nombre": "Monitor",
  "precio": 250000
}
```

Se lee así:

```txt
El recurso fue creado correctamente.
La representación está en JSON.
El nuevo recurso está en /productos/7.
```

### Métodos HTTP

El método indica la intención del pedido.

| Método   | Intención principal                                    |
| -------- | ------------------------------------------------------ |
| `GET`    | Leer datos.                                            |
| `POST`   | Crear un recurso o ejecutar una operación de creación. |
| `PUT`    | Reemplazar un recurso completo.                        |
| `PATCH`  | Modificar parcialmente un recurso.                     |
| `DELETE` | Eliminar un recurso.                                   |

### URL, ruta y query string

Ejemplo:

```txt
http://localhost:5000/productos/5?incluirInactivos=true
│      │              │          │
│      │              │          └── query string
│      │              └──────────── ruta
│      └─────────────────────────── host y puerto
└────────────────────────────────── esquema
```

La **ruta** identifica el recurso dentro del servidor:

```txt
/productos
/productos/5
/clientes/10/facturas
```

La **query string** manda parámetros opcionales:

```txt
/productos?texto=mouse
/productos?pagina=2&tamanio=20
/productos?orden=precio
```

Regla práctica:

```txt
Ruta         → identifica el recurso.
Query string → filtra, ordena o pagina una colección.
```

### Headers

Los headers son metadatos del mensaje.

| Header           | Uso                                      |
| ---------------- | ---------------------------------------- |
| `Content-Type`   | Formato del body que se envía.           |
| `Accept`         | Formato que el cliente prefiere recibir. |
| `Authorization`  | Credenciales o tokens.                   |
| `Content-Length` | Tamaño del body.                         |

### Body

El body contiene datos. En APIs REST modernas, casi siempre usamos JSON.

```json
{
  "nombre": "Teclado",
  "precio": 15000
}
```

`GET` normalmente no lleva body.  
`POST`, `PUT` y `PATCH` normalmente sí.

### Status codes principales

El status code comunica el resultado de la operación.

| Código | Nombre                | Uso típico                                                                 |
| -----: | --------------------- | -------------------------------------------------------------------------- |
| 200    | OK                    | La operación salió bien y devuelve contenido.                              |
| 201    | Created               | Se creó un recurso.                                                        |
| 204    | No Content            | La operación salió bien pero no devuelve body.                             |
| 400    | Bad Request           | El cliente mandó datos inválidos.                                          |
| 401    | Unauthorized          | Falta autenticación.                                                       |
| 403    | Forbidden             | Está autenticado, pero no tiene permiso.                                   |
| 404    | Not Found             | El recurso no existe.                                                      |
| 409    | Conflict              | Hay conflicto con el estado actual.                                        |
| 422    | Unprocessable Entity  | El JSON está bien formado, los datos no son válidos para regla de negocio. |
| 500    | Internal Server Error | Falló el servidor.                                                         |

Una API no solo devuelve datos. Devuelve datos más significado.

```txt
404 no significa "no hay nada".
404 significa "el recurso solicitado no existe".
```

## 3. JSON

JSON es texto plano para representar datos.

Ejemplo:

```json
{
  "id": 5,
  "nombre": "Mouse",
  "precio": 10000,
  "activo": true
}
```

Tipos comunes:

| Tipo     | Ejemplo                |
| -------- | ---------------------- |
| Número   | `42`, `3.14`           |
| Texto    | `"Hola"`               |
| Booleano | `true`, `false`        |
| Nulo     | `null`                 |
| Objeto   | `{ "clave": "valor" }` |
| Array    | `[1, 2, 3]`            |

ASP.NET Core convierte objetos C# a JSON automáticamente cuando una API devuelve objetos.

## 4. REST

REST es una forma de diseñar APIs alrededor de **recursos**.

Un recurso es algo con identidad dentro del sistema:

```txt
Producto
Cliente
Factura
Turno
Alumno
Inscripción
```

En REST no diseñamos URLs como acciones:

```txt
/crearProducto
/borrarProducto
/listarProductos
```

Diseñamos URLs como recursos:

```txt
/productos
/productos/5
```

Y usamos el método HTTP para expresar la operación:

| Request               | Significado                     |
| --------------------- | ------------------------------- |
| `GET /productos`      | Leer la colección de productos. |
| `GET /productos/5`    | Leer el producto con id 5.      |
| `POST /productos`     | Crear un producto nuevo.        |
| `PUT /productos/5`    | Reemplazar el producto 5.       |
| `PATCH /productos/5`  | Modificar parte del producto 5. |
| `DELETE /productos/5` | Eliminar el producto 5.         |

### Colección y recurso individual

```txt
/productos
```

representa la colección completa.

```txt
/productos/5
```

representa un producto concreto.

### Representación

El recurso real no viaja por la red. Lo que viaja es una representación.

El producto vive en la base de datos. La API devuelve una representación JSON:

```json
{
  "id": 5,
  "nombre": "Mouse",
  "precio": 10000
}
```

Conviene no confundir:

```txt
Tabla de base de datos ≠ entidad C# ≠ DTO ≠ JSON público
```

A veces se parecen. No son lo mismo.

### Métodos seguros e idempotentes

`GET` es seguro: no debería modificar el estado del servidor.

`PUT` y `DELETE` son idempotentes: repetir la misma operación debería dejar el sistema en el mismo estado final.

Ejemplo:

```txt
DELETE /productos/5
DELETE /productos/5
DELETE /productos/5
```

Después de la primera llamada, el producto ya no existe. Repetir la request no debería eliminar productos distintos.

`POST` no es necesariamente idempotente:

```txt
POST /productos
POST /productos
POST /productos
```

puede crear tres productos distintos.

### PUT vs PATCH

`PUT` reemplaza el recurso completo.

```http
PUT /productos/5
Content-Type: application/json

{
  "nombre": "Mouse inalámbrico",
  "precio": 18000
}
```

`PATCH` modifica parcialmente.

```http
PATCH /productos/5
Content-Type: application/json

{
  "precio": 18000
}
```

Para un primer CRUD alcanza con `PUT`. `PATCH` se puede dejar como mejora.

## 5. Primera Minimal API

Crear un archivo `app.cs`:

```csharp
#:sdk Microsoft.NET.Sdk.Web

var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

app.MapGet("/", () => "API funcionando");

app.MapGet("/saludo", () => "Hola desde el servidor");

app.Run();
```

Ejecutar:

```bash
dotnet run app.cs
```

La línea importante:

```csharp
app.MapGet("/saludo", () => "Hola desde el servidor");
```

Se lee así:

```txt
Cuando llegue un GET a /saludo,
ejecutá esta función
y devolvé el resultado como response HTTP.
```

Un **endpoint** es una función disponible por HTTP.

## 6. Rutas y parámetros

Ahora agregamos un parámetro en la ruta:

```csharp
#:sdk Microsoft.NET.Sdk.Web

var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

app.MapGet("/productos/{id:int}", (int id) => {
    return $"Producto solicitado: {id}";
});

app.Run();
```

Request:

```http
GET /productos/10
```

Respuesta:

```txt
Producto solicitado: 10
```

La ruta:

```csharp
"/productos/{id:int}"
```

indica que `id` viene desde la URL y debe poder convertirse a `int`.

Si llega:

```txt
/productos/abc
```

no coincide con esa ruta porque `abc` no es entero.

También podemos recibir datos desde la **query string**.

En Minimal APIs, si un parámetro simple no está en la ruta, ASP.NET Core lo busca en la query string.

Ejemplo:

```csharp
app.MapGet("/productos/buscar", (string texto) => {
    return $"Buscando productos que contengan: {texto}";
});
```

Request:

```http
GET /productos/buscar?texto=mouse
```

Respuesta:

```txt
Buscando productos que contengan: mouse
```

En este caso:

```txt
/productos/buscar?texto=mouse
                  └── texto llega al parámetro string texto
```

La ruta identifica la operación:

```txt
/productos/buscar
```

La query string manda el criterio de búsqueda:

```txt
?texto=mouse
```

También se pueden recibir varios valores:

```csharp
app.MapGet("/productos", (int pagina, int tamanio) => {
    return $"Página {pagina}, tamaño {tamanio}";
});
```

Request:

```http
GET /productos?pagina=2&tamanio=20
```

Si un parámetro de query string puede faltar, conviene declararlo como nullable o darle un valor por defecto:

```csharp
app.MapGet("/productos", (string? texto, int pagina = 1) => {
    return $"Texto: {texto}, página: {pagina}";
});
```

Request válida:

```http
GET /productos?texto=teclado
```

Como `pagina` no llegó en la query string, toma el valor por defecto `1`.

También podemos recibir valores desde las **cabeceras** o **headers** de la request.

Para eso usamos el atributo `[FromHeader]`.

Primero hay que agregar este `using`:

```csharp
using Microsoft.AspNetCore.Mvc;
```

Ejemplo:

```csharp
app.MapGet("/perfil", ([FromHeader(Name = "Authorization")] string authorization) => {
    return $"Header Authorization recibido: {authorization}";
});
```

Request:

```http
GET /perfil HTTP/1.1
Host: localhost:5000
Authorization: Bearer abc123
```

Respuesta:

```txt
Header Authorization recibido: Bearer abc123
```

En este caso:

```txt
Authorization: Bearer abc123
               └── llega al parámetro string authorization
```

Si el nombre del parámetro coincide con el nombre del header, se puede escribir más corto:

```csharp
app.MapGet("/cliente", ([FromHeader] string origen) => {
    return $"Origen: {origen}";
});
```

Request:

```http
GET /cliente HTTP/1.1
Host: localhost:5000
Origen: web
```

Pero algunos headers tienen guiones, como `User-Agent`. En C# no podemos tener un parámetro llamado `user-agent`, por eso hay que indicar el nombre real del header:

```csharp
app.MapGet("/cliente", ([FromHeader(Name = "User-Agent")] string userAgent) => {
    return $"User-Agent: {userAgent}";
});
```

Si un header puede faltar, conviene declararlo como nullable:

```csharp
app.MapGet("/perfil", ([FromHeader(Name = "Authorization")] string? authorization) => {
    if (authorization is null) {
        return Results.Unauthorized();
    }

    return Results.Ok($"Header recibido: {authorization}");
});
```

Otra forma es recibir el **contexto HTTP** completo.

El contexto representa toda la request y toda la response. Desde ahí podemos leer la ruta, la query string, los headers, el body, el usuario autenticado, la IP del cliente y otros datos del pedido.

Ejemplo leyendo un header desde `HttpContext`:

```csharp
app.MapGet("/perfil", (HttpContext context) => {
    string authorization = context.Request.Headers["Authorization"].ToString();

    return $"Header Authorization recibido: {authorization}";
});
```

Request:

```http
GET /perfil HTTP/1.1
Host: localhost:5000
Authorization: Bearer abc123
```

El valor se lee desde:

```csharp
context.Request.Headers["Authorization"]
```

Si el header puede faltar, es mejor verificarlo:

```csharp
app.MapGet("/perfil", (HttpContext context) => {
    if (!context.Request.Headers.TryGetValue("Authorization", out var authorization)) {
        return Results.Unauthorized();
    }

    return Results.Ok($"Header recibido: {authorization}");
});
```

También se pueden leer valores de la query string desde el contexto:

```csharp
app.MapGet("/productos/buscar", (HttpContext context) => {
    string texto = context.Request.Query["texto"].ToString();

    return $"Buscando productos que contengan: {texto}";
});
```

Request:

```http
GET /productos/buscar?texto=mouse
```

El valor se lee desde:

```csharp
context.Request.Query["texto"]
```

En general, conviene usar parámetros directos cuando el endpoint es simple:

```csharp
app.MapGet("/productos/buscar", (string texto) => {
    return $"Buscando productos que contengan: {texto}";
});
```

Y conviene usar `HttpContext` cuando necesitamos acceder a información más general de la request:

```csharp
app.MapGet("/diagnostico", (HttpContext context) => {
    string metodo = context.Request.Method;
    string ruta = context.Request.Path;
    string userAgent = context.Request.Headers["User-Agent"].ToString();

    return Results.Ok(new {
        Metodo = metodo,
        Ruta = ruta,
        UserAgent = userAgent
    });
});
```

Regla práctica:

```txt
Ruta         → identifica el recurso.
Query string → manda filtros, orden, paginación u opciones simples.
Headers      → mandan metadatos del pedido, como autenticación o formato esperado.
Body         → manda datos principales para crear o modificar recursos.
HttpContext  → permite acceder manualmente a toda la request y la response.
```

### Parámetros que vienen de servicios

No todos los parámetros de un endpoint salen de la request.

ASP.NET Core también puede completar parámetros usando objetos registrados como servicios.

Ejemplo:

```csharp
var builder = WebApplication.CreateBuilder(args);

DatabaseOptions databaseOptions = new DatabaseOptions {
    ConnectionString = "Data Source=productos.db"
};

builder.Services.AddSingleton(databaseOptions);
builder.Services.AddScoped<ProductoRepositorio>();

var app = builder.Build();

app.MapGet("/productos", (ProductoRepositorio repositorio) => {
    IEnumerable<Producto> productos = repositorio.ObtenerTodos();

    return Results.Ok(productos);
});
```

En este endpoint:

```csharp
(ProductoRepositorio repositorio) =>
```

`repositorio` no viene de la ruta, ni de la query string, ni del body.

Viene del contenedor de servicios porque antes fue registrado con:

```csharp
builder.Services.AddScoped<ProductoRepositorio>();
```

La configuración también se puede registrar como servicio.

```csharp
DatabaseOptions databaseOptions = new DatabaseOptions {
    ConnectionString = "Data Source=productos.db"
};

builder.Services.AddSingleton(databaseOptions);
```

Y otro servicio puede recibir esa configuración por constructor:

```csharp
public class ProductoRepositorio {
    private readonly string connectionString;

    public ProductoRepositorio(DatabaseOptions options) {
        connectionString = options.ConnectionString;
    }
}

public class DatabaseOptions {
    public string ConnectionString { get; set; } = "";
}
```

También se pueden mezclar parámetros que vienen de lugares distintos:

```csharp
app.MapGet("/productos/{id:int}", (int id, ProductoRepositorio repositorio) => {
    Producto? producto = repositorio.ObtenerPorId(id);

    if (producto is null) {
        return Results.NotFound();
    }

    return Results.Ok(producto);
});
```

En ese caso:

```txt
int id                          → viene de la ruta /productos/{id:int}
ProductoRepositorio repositorio → viene de los servicios registrados
```

Regla práctica:

```txt
Parámetros simples    → ruta, query string o headers.
Objetos del body      → JSON enviado por el cliente.
Servicios registrados → contenedor de servicios de ASP.NET Core.
```

## 7. CRUD en memoria

Antes de usar base de datos, armamos una API en memoria. Esto permite entender HTTP, REST, DTOs y status codes sin mezclar todavía SQL.

```csharp
#:sdk Microsoft.NET.Sdk.Web

var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

List<Producto> productos = [
    new Producto(1, "Teclado", 15000),
    new Producto(2, "Mouse", 10000)
];

app.MapGet("/productos", () => {
    return Results.Ok(productos);
});

app.MapGet("/productos/{id:int}", (int id) => {
    Producto? producto = productos.FirstOrDefault(p => p.Id == id);

    if (producto is null) { return Results.NotFound(); }

    return Results.Ok(producto);
});

app.MapPost("/productos", (CrearProductoDto dto) => {
    if (string.IsNullOrWhiteSpace(dto.Nombre)) { return Results.BadRequest("El nombre es obligatorio."); }
    if (dto.Precio <= 0) { return Results.BadRequest("El precio debe ser mayor que cero."); }

    int nuevoId = productos.Count == 0 ? 1 : productos.Max(p => p.Id) + 1;

    Producto producto = new Producto(nuevoId, dto.Nombre, dto.Precio);
    productos.Add(producto);

    return Results.Created($"/productos/{producto.Id}", producto);
});

app.MapPut("/productos/{id:int}", (int id, ActualizarProductoDto dto) => {
    if (string.IsNullOrWhiteSpace(dto.Nombre)) { return Results.BadRequest("El nombre es obligatorio."); }
    if (dto.Precio <= 0) { return Results.BadRequest("El precio debe ser mayor que cero."); }

    int indice = productos.FindIndex(p => p.Id == id);

    if (indice == -1) {
        return Results.NotFound();
    }

    Producto productoActualizado = new Producto(id, dto.Nombre, dto.Precio);
    productos[indice] = productoActualizado;

    return Results.Ok(productoActualizado);
});

app.MapDelete("/productos/{id:int}", (int id) => {
    Producto? producto = productos.FirstOrDefault(p => p.Id == id);

    if (producto is null) {
        return Results.NotFound();
    }

    productos.Remove(producto);

    return Results.NoContent();
});

app.Run();

public record Producto(int Id, string Nombre, decimal Precio);
public record CrearProductoDto(string Nombre, decimal Precio);
public record ActualizarProductoDto(string Nombre, decimal Precio);
```

Este ejemplo ya tiene la estructura de una API real:

```txt
GET colección
GET por id
POST creación
PUT actualización
DELETE eliminación
DTOs de entrada
status codes correctos
JSON automático
```

## 8. API completa con SQLite y repositorio

Crear `app.cs`:

```csharp
#:sdk Microsoft.NET.Sdk.Web
#:package Microsoft.Data.Sqlite@*
#:package Dapper@*
#:package Dapper.Contrib@*

using Dapper;
using Dapper.Contrib.Extensions;
using Microsoft.Data.Sqlite;

var builder = WebApplication.CreateBuilder(args);

DatabaseOptions databaseOptions = new DatabaseOptions {
    ConnectionString = "Data Source=productos.db"
};

builder.Services.AddSingleton(databaseOptions);
builder.Services.AddScoped<ProductoRepositorio>();

var app = builder.Build();

ProductoRepositorio.InicializarBaseDeDatos(databaseOptions);

app.MapGet("/", () => "API de productos");

app.MapGet("/productos", (ProductoRepositorio repositorio) => {
    IEnumerable<Producto> productos = repositorio.ObtenerTodos();

    return Results.Ok(productos);
});

app.MapGet("/productos/{id:int}", (int id, ProductoRepositorio repositorio) => {
    Producto? producto = repositorio.ObtenerPorId(id);

    if (producto is null) {
        return Results.NotFound();
    }

    return Results.Ok(producto);
});

app.MapGet("/productos/buscar", (string texto, ProductoRepositorio repositorio) => {
    IEnumerable<Producto> productos = repositorio.Buscar(texto);

    return Results.Ok(productos);
});

app.MapPost("/productos", (CrearProductoDto dto, ProductoRepositorio repositorio) => {
    List<string> errores = ValidarProducto(dto.Nombre, dto.Precio);

    if (errores.Count > 0) {
        return Results.BadRequest(errores);
    }

    Producto producto = repositorio.Crear(dto.Nombre, dto.Precio);

    return Results.Created($"/productos/{producto.Id}", producto);
});

app.MapPut("/productos/{id:int}", (int id, ActualizarProductoDto dto, ProductoRepositorio repositorio) => {
    List<string> errores = ValidarProducto(dto.Nombre, dto.Precio);

    if (errores.Count > 0) {
        return Results.BadRequest(errores);
    }

    Producto? producto = repositorio.Actualizar(id, dto.Nombre, dto.Precio);

    if (producto is null) {
        return Results.NotFound();
    }

    return Results.Ok(producto);
});

app.MapDelete("/productos/{id:int}", (int id, ProductoRepositorio repositorio) => {
    bool eliminado = repositorio.Eliminar(id);

    if (!eliminado) {
        return Results.NotFound();
    }

    return Results.NoContent();
});

app.Run();

static List<string> ValidarProducto(string nombre, decimal precio) {
    List<string> errores = [];

    if (string.IsNullOrWhiteSpace(nombre)) {
        errores.Add("El nombre es obligatorio.");
    }

    if (precio <= 0) {
        errores.Add("El precio debe ser mayor que cero.");
    }

    return errores;
}

public class ProductoRepositorio {
    private readonly string connectionString;

    public ProductoRepositorio(DatabaseOptions options) {
        connectionString = options.ConnectionString;
    }

    public static void InicializarBaseDeDatos(DatabaseOptions options) {
        using SqliteConnection connection = new SqliteConnection(options.ConnectionString);

        string sql = """
            CREATE TABLE IF NOT EXISTS Productos (
                Id INTEGER PRIMARY KEY AUTOINCREMENT,
                Nombre TEXT NOT NULL,
                Precio REAL NOT NULL
            );
            """;

        connection.Execute(sql);
    }

    public IEnumerable<Producto> ObtenerTodos() {
        using SqliteConnection connection = new SqliteConnection(connectionString);

        return connection.GetAll<Producto>();
    }

    public Producto? ObtenerPorId(int id) {
        using SqliteConnection connection = new SqliteConnection(connectionString);

        return connection.Get<Producto>(id);
    }

    public IEnumerable<Producto> Buscar(string texto) {
        using SqliteConnection connection = new SqliteConnection(connectionString);

        string sql = """
            SELECT Id, Nombre, Precio
            FROM Productos
            WHERE Nombre LIKE @Patron
            ORDER BY Nombre;
            """;

        return connection.Query<Producto>(
            sql,
            new { Patron = $"%{texto}%" }
        );
    }

    public Producto Crear(string nombre, decimal precio) {
        using SqliteConnection connection = new SqliteConnection(connectionString);

        Producto producto = new Producto {
            Nombre = nombre,
            Precio = precio
        };

        long id = connection.Insert(producto);
        producto.Id = (int)id;

        return producto;
    }

    public Producto? Actualizar(int id, string nombre, decimal precio) {
        using SqliteConnection connection = new SqliteConnection(connectionString);

        Producto? producto = connection.Get<Producto>(id);

        if (producto is null) {
            return null;
        }

        producto.Nombre = nombre;
        producto.Precio = precio;

        connection.Update(producto);

        return producto;
    }

    public bool Eliminar(int id) {
        using SqliteConnection connection = new SqliteConnection(connectionString);

        Producto? producto = connection.Get<Producto>(id);

        if (producto is null) {
            return false;
        }

        connection.Delete(producto);

        return true;
    }
}

[Table("Productos")]
public class Producto {
    [Key]
    public int Id { get; set; }

    public string Nombre { get; set; } = "";

    public decimal Precio { get; set; }
}

public record CrearProductoDto(string Nombre, decimal Precio);

public record ActualizarProductoDto(string Nombre, decimal Precio);

public class DatabaseOptions {
    public string ConnectionString { get; set; } = "";
}
```

| Herramienta    | Para qué sirve                                                    |
| -------------- | ----------------------------------------------------------------- |
| SQLite         | Base de datos liviana en un archivo local.                        |
| ADO.NET        | Infraestructura base de conexión y comandos SQL.                  |
| Dapper         | Ejecutar SQL y mapear resultados a objetos.                       |
| Dapper.Contrib | CRUD simple con métodos como `Get`, `Insert`, `Update`, `Delete`. |
| Repositorio    | Concentrar el acceso a datos fuera de los endpoints.              |
| DatabaseOptions | Guardar la configuración de conexión registrada como servicio.    |

Los endpoints reciben HTTP, validan datos y devuelven status codes.

El repositorio habla con SQLite.

```txt
Endpoint      → recibe HTTP, valida datos, devuelve status codes.
Repositorio   → habla con SQLite.
Configuración → guarda el connection string.
```

No conviene compartir una única `SqliteConnection` para toda la aplicación. Una API puede recibir varias requests al mismo tiempo. Si todas comparten la misma conexión, aparecen problemas de estado, concurrencia y errores difíciles de seguir.

Por eso cada operación del repositorio crea su propia conexión:

```csharp

```

La conexión se usa durante esa operación y después se cierra automáticamente al salir del `using`.


## 15. Middleware

Una request no entra directamente al endpoint. Pasa por una tubería.

```txt
Request
   ↓
Middleware
   ↓
Middleware
   ↓
Endpoint
   ↓
Response
```

Cada middleware puede:

- leer la request;
- modificar la request;
- decidir si la ejecución sigue;
- ejecutar código antes del endpoint;
- ejecutar código después del endpoint;
- modificar la response.

Ejemplo simple para ver el flujo:

```csharp
app.Use(async (context, next) => {
    Console.WriteLine($"Entró {context.Request.Method} {context.Request.Path}");

    await next();

    Console.WriteLine($"Salió {context.Response.StatusCode}");
});
```

Si llega:

```http
GET /productos
```

la consola muestra algo parecido a:

```txt
Entró GET /productos
Salió 200
```

El orden del pipeline importa. En general:

```csharp
var app = builder.Build();

app.UseCors("Frontend");
app.UseDefaultFiles();
app.UseStaticFiles();

app.Use(async (context, next) => {
    Console.WriteLine($"{context.Request.Method} {context.Request.Path}");
    await next();
});

app.MapGet("/productos", () => {
    // ...
});
```

### Configurar y usar middleware

Un middleware se configura agregándolo al pipeline con `app.Use(...)`.

La forma general es:

```csharp
app.Use(async (context, next) => {
    // Código antes del endpoint.

    await next();

    // Código después del endpoint.
});
```

`context` es el `HttpContext`. Permite acceder a la request y a la response.

`next` representa el próximo paso de la tubería. Si llamamos a `await next()`, la request sigue avanzando. Si no lo llamamos, el middleware corta la ejecución y el endpoint no se ejecuta.

```txt
Con await next() → la request sigue.
Sin await next() → la request se corta ahí.
```

Esto sirve para validaciones generales que se repiten en muchos endpoints.

### Ejemplo: validar usuario con un header

Supongamos que la API espera recibir un usuario en el header `X-Usuario`.

Request válida:

```http
GET /productos HTTP/1.1
Host: localhost:5000
X-Usuario: ana
```

Request inválida:

```http
GET /productos HTTP/1.1
Host: localhost:5000
```

Podemos validar ese header con un middleware:

```csharp
app.Use(async (context, next) => {
    if (!context.Request.Headers.TryGetValue("X-Usuario", out var usuario)) {
        context.Response.StatusCode = 401;
        await context.Response.WriteAsync("Falta el header X-Usuario.");
        return;
    }

    if (string.IsNullOrWhiteSpace(usuario)) {
        context.Response.StatusCode = 401;
        await context.Response.WriteAsync("El usuario no puede estar vacío.");
        return;
    }

    await next();
});
```

Se lee así:

```txt
Si no viene X-Usuario, responder 401 y cortar.
Si X-Usuario viene vacío, responder 401 y cortar.
Si X-Usuario es válido, continuar con el endpoint.
```

Código completo:

```csharp
#:sdk Microsoft.NET.Sdk.Web

var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

app.Use(async (context, next) => {
    if (!context.Request.Headers.TryGetValue("X-Usuario", out var usuario)) {
        context.Response.StatusCode = 401;
        await context.Response.WriteAsync("Falta el header X-Usuario.");
        return;
    }

    if (string.IsNullOrWhiteSpace(usuario)) {
        context.Response.StatusCode = 401;
        await context.Response.WriteAsync("El usuario no puede estar vacío.");
        return;
    }

    await next();
});

app.MapGet("/productos", () => {
    return Results.Ok(new[] {
        "Teclado",
        "Mouse",
        "Monitor"
    });
});

app.Run();
```

Si probamos sin header:

```http
GET /productos
```

Respuesta:

```txt
401 Unauthorized
Falta el header X-Usuario.
```

Si probamos con header:

```http
GET /productos
X-Usuario: ana
```

Respuesta:

```json
[
  "Teclado",
  "Mouse",
  "Monitor"
]
```

### Usar el usuario dentro del endpoint

Además de validar, el middleware puede guardar datos en `context.Items`.

`context.Items` es una bolsa de datos que vive durante una sola request.

```csharp
app.Use(async (context, next) => {
    if (!context.Request.Headers.TryGetValue("X-Usuario", out var usuario)) {
        context.Response.StatusCode = 401;
        await context.Response.WriteAsync("Falta el header X-Usuario.");
        return;
    }

    context.Items["Usuario"] = usuario.ToString();

    await next();
});
```

Luego el endpoint puede leer ese dato desde el contexto:

```csharp
app.MapGet("/perfil", (HttpContext context) => {
    string usuario = context.Items["Usuario"]?.ToString() ?? "";

    return Results.Ok(new { Usuario = usuario });
});
```

Flujo:

```txt
Request
   ↓
Middleware valida X-Usuario
   ↓
Middleware guarda Usuario en context.Items
   ↓
Endpoint lee context.Items["Usuario"]
   ↓
Response
```

### Cuidar qué rutas protege el middleware

Un middleware se aplica a los endpoints que están después en el pipeline.

Ejemplo:

```csharp
app.MapGet("/", () => "API funcionando");

app.Use(async (context, next) => {
    if (!context.Request.Headers.TryGetValue("X-Usuario", out var usuario)) {
        context.Response.StatusCode = 401;
        await context.Response.WriteAsync("Falta el header X-Usuario.");
        return;
    }

    context.Items["Usuario"] = usuario.ToString();

    await next();
});

app.MapGet("/productos", () => {
    return Results.Ok("Listado de productos");
});
```

En este ejemplo:

```txt
/          → no queda protegido por este middleware.
/productos → sí queda protegido por este middleware.
```

En aplicaciones reales, la autenticación suele hacerse con middleware ya provisto por ASP.NET Core, como `UseAuthentication()` y `UseAuthorization()`. El ejemplo anterior sirve para entender la idea: un middleware puede inspeccionar la request, decidir si sigue y dejar datos preparados para los endpoints.

## 16. Logging con ILogger

ASP.NET Core trae infraestructura de logging.

Se puede pedir un logger en un endpoint:

```csharp
app.MapPost("/productos", (CrearProductoDto dto, ILogger<Program> logger) => {
    logger.LogInformation("Intentando crear producto {Nombre} con precio {Precio}", dto.Nombre, dto.Precio);

    if (string.IsNullOrWhiteSpace(dto.Nombre)) {
        logger.LogWarning("Intento de crear producto con nombre vacío");
        return Results.BadRequest("El nombre es obligatorio.");
    }

    if (dto.Precio <= 0) {
        logger.LogWarning("Intento de crear producto con precio inválido: {Precio}", dto.Precio);
        return Results.BadRequest("El precio debe ser mayor que cero.");
    }

    // insertar producto...

    logger.LogInformation("Producto creado correctamente");

    return Results.Ok();
});
```

Niveles comunes:

| Nivel         | Uso                        |
| ------------- | -------------------------- |
| `Trace`       | Detalle extremo.           |
| `Debug`       | Información de depuración. |
| `Information` | Operación normal.          |
| `Warning`     | Algo raro pero controlado. |
| `Error`       | Falló una operación.       |
| `Critical`    | Fallo grave.               |

Esta forma:

```csharp
logger.LogInformation("Producto creado con id {Id}", producto.Id);
```

es preferible a:

```csharp
logger.LogInformation($"Producto creado con id {producto.Id}");
```

porque conserva el mensaje estructurado.

```csharp
        return Results.BadRequest(errores);

    Producto? producto = repositorio.Actualizar(id, dto.Nombre, dto.Precio);

    if (producto is null) {
        logger.LogWarning("No se pudo actualizar. Producto {Id} no encontrado", id);
        return Results.NotFound();
    }

    logger.LogInformation("Producto actualizado: {Id}", producto.Id);

    return Results.Ok(producto);
});

app.MapDelete("/productos/{id:int}", (int id, ProductoRepositorio repositorio, ILogger<Program> logger) => {
    bool eliminado = repositorio.Eliminar(id);

    if (!eliminado) {
        logger.LogWarning("No se pudo eliminar. Producto {Id} no encontrado", id);
        return Results.NotFound();
    }

    logger.LogInformation("Producto eliminado: {Id}", id);

    return Results.NoContent();
});

static List<string> ValidarProducto(string nombre, decimal precio) {
    List<string> errores = [];

    if (string.IsNullOrWhiteSpace(nombre)) {
        errores.Add("El nombre es obligatorio.");
    }

    if (precio <= 0) {
        errores.Add("El precio debe ser mayor que cero.");
    }

    return errores;
}

public class ProductoRepositorio {
    private readonly string connectionString;

    public ProductoRepositorio(DatabaseOptions options) {
        connectionString = options.ConnectionString;
    }

    public static void InicializarBaseDeDatos(DatabaseOptions options) {
        using SqliteConnection connection = new SqliteConnection(options.ConnectionString);

        string sql = """
            CREATE TABLE IF NOT EXISTS Productos (
                Id INTEGER PRIMARY KEY AUTOINCREMENT,
                Nombre TEXT NOT NULL,
                Precio REAL NOT NULL
            );
            """;

        connection.Execute(sql);
    }

    public IEnumerable<Producto> ObtenerTodos() {
        using SqliteConnection connection = new SqliteConnection(connectionString);

        return connection.GetAll<Producto>();
    }

    public Producto? ObtenerPorId(int id) {
        using SqliteConnection connection = new SqliteConnection(connectionString);

        return connection.Get<Producto>(id);
    }

    public IEnumerable<Producto> Buscar(string texto) {
        using SqliteConnection connection = new SqliteConnection(connectionString);

        string sql = """
            SELECT Id, Nombre, Precio
            FROM Productos
            WHERE Nombre LIKE @Patron
            ORDER BY Nombre;
            """;

        return connection.Query<Producto>(
            sql,
            new { Patron = $"%{texto}%" }
        );
    }

    public Producto Crear(string nombre, decimal precio) {
        using SqliteConnection connection = new SqliteConnection(connectionString);

        Producto producto = new Producto {
            Nombre = nombre,
            Precio = precio
        };

        long id = connection.Insert(producto);
        producto.Id = (int)id;

        return producto;
    }

    public Producto? Actualizar(int id, string nombre, decimal precio) {
        using SqliteConnection connection = new SqliteConnection(connectionString);

        Producto? producto = connection.Get<Producto>(id);

        if (producto is null) {
            return null;
        }

        producto.Nombre = nombre;
        producto.Precio = precio;

        connection.Update(producto);

        return producto;
    }

    public bool Eliminar(int id) {
        using SqliteConnection connection = new SqliteConnection(connectionString);

        Producto? producto = connection.Get<Producto>(id);

        if (producto is null) {
            return false;
        }

        connection.Delete(producto);

        return true;
    }
}
```

## 20. Resumen

| Necesidad                                    | Herramienta                                                                                    |
| :------------------------------------------- | :--------------------------------------------------------------------------------------------- |
| Exponer endpoints HTTP simples               | Minimal APIs con `app.MapGet`, `app.MapPost`, `app.MapPut`, `app.MapDelete`                    |

| Guardar datos en SQLite                      | `Microsoft.Data.Sqlite`                                                                        |
| Ejecutar SQL simple                          | ADO.NET con `SqliteConnection` y `SqliteCommand`                                               |
| Simplificar consultas SQL                    | Dapper con `Query`, `Execute` y parámetros                                                     |

| Evitar inyección SQL                         | parámetros como `@Patron`, no concatenación de strings                                         |

| Registrar eventos de la API                  | `ILogger<T>`                                                                                   |
| Comprimir responses                          | `builder.Services.AddResponseCompression` y `app.UseResponseCompression`                       |
| Consumir una API desde C#                    | `HttpClient` y `System.Net.Http.Json`                                                          |
| Consumir una API desde HTML                  | `fetch`                                                                                        |
