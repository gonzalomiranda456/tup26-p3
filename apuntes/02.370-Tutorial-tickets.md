## 1. Qué vamos a construir

Un sistema donde **clientes y usuarios internos comparten una misma tabla**, pueden iniciar sesión, y cada ticket registra:

- **Quién lo origina** y **quién es el responsable**.
- Un **estado** (`Abierto`, `EnProceso`, `Cerrado`).
- Una **bitácora de acciones**, donde cada acción tiene una fecha: si la fecha ya pasó, es algo que se hizo; si es futura, es un **compromiso pendiente**.

La arquitectura sigue un patrón en capas clásico:

```
HTTP (Minimal API + middleware)
        │
        ▼
   Servicios  ← lógica del sistema
        │
        ▼
  Repositorios ← acceso a datos (interface + implementación)
        │
        ▼
   DbContext (EF Core) ← SQLite
```

> **Por qué capas:** cada capa solo conoce a la de abajo a través de una interface. Eso nos deja cambiar SQLite por otra base, o testear los servicios con un repositorio falso, sin tocar el resto. No es sobre-ingeniería: es la separación mínima para que el código no se vuelva una bola de barro.

## 2. Crear el proyecto

Abrí una terminal y ejecutá:

```bash
dotnet new web -n TicketSystem
cd TicketSystem
```

Agregamos el paquete de EF Core con proveedor SQLite:

```bash
dotnet add package Microsoft.EntityFrameworkCore.Sqlite
```

> **`dotnet new web`** crea el template más liviano de ASP.NET Core, ideal para Minimal API: no trae controladores ni vistas, solo un `Program.cs` con lo justo. Como vamos a trabajar **code-first sin migraciones** (la base se crea sola a partir de las entidades), nos alcanza con el proveedor SQLite: no hace falta `EntityFrameworkCore.Design` ni la herramienta `dotnet-ef`.

Vamos a organizar el código en carpetas. Creálas ahora:

```bash
mkdir Domain Data Repositories Security Services Api
```

Cada carpeta es un namespace y una responsabilidad:

| Carpeta        | Contiene                                         |
| -------------- | ------------------------------------------------ |
| `Domain`       | Las entidades (Usuario, Ticket, Accion) y enums  |
| `Data`         | El DbContext                                     |
| `Repositories` | Interfaces e implementaciones de acceso a datos  |
| `Security`     | El hasher de contraseñas                         |
| `Services`     | La lógica del sistema                            |
| `Api`          | DTOs, filtros y helpers de la capa HTTP          |

## 3. Las entidades del dominio

Las entidades son las clases que EF Core mapea a tablas. Empezamos por los **enums** y la entidad `Usuario`.

Creá el archivo `Domain/Usuario.cs`:

```csharp
namespace TicketSystem.Domain;

public enum TipoUsuario {
    Cliente,
    Interno
}

public class Usuario {
    public int Id { get; set; }
    public string Nombre { get; set; } = string.Empty;
    public string Email { get; set; } = string.Empty;
    public TipoUsuario Tipo { get; set; }

    // Autenticación
    public string PasswordHash { get; set; } = string.Empty;
    public string PasswordSalt { get; set; } = string.Empty;
    public string? Token { get; set; }
    public DateTime? TokenExpira { get; set; }

    // Navegación
    public ICollection<Ticket> TicketsOriginados { get; set; } = new List<Ticket>();
    public ICollection<Ticket> TicketsAsignados { get; set; } = new List<Ticket>();
}
```

> **`= string.Empty`** evita advertencias del compilador por valores nulos: una propiedad `string` no anulable tiene que arrancar con algo. El `Token` y `TokenExpira` sí son anulables (`string?`, `DateTime?`) porque un usuario recién creado todavía no inició sesión.

Ahora `Domain/Ticket.cs`:

```csharp
namespace TicketSystem.Domain;

public enum EstadoTicket {
    Abierto,
    EnProceso,
    Cerrado
}

public class Ticket {
    public int Id { get; set; }
    public string Titulo { get; set; } = string.Empty;
    public string? Descripcion { get; set; }
    public int OriginadoPorId { get; set; }
    public int? ResponsableId { get; set; }
    public EstadoTicket Estado { get; set; } = EstadoTicket.Abierto;
    public DateTime FechaCreacion { get; set; } = DateTime.Now;

    // Navegación
    public Usuario OriginadoPor { get; set; } = null!;
    public Usuario? Responsable { get; set; }
    public ICollection<Accion> Acciones { get; set; } = new List<Accion>();
}
```

> **Dos FK al mismo tipo:** `OriginadoPorId` y `ResponsableId` apuntan ambas a `Usuario`. El responsable es anulable (`int?`) porque un ticket puede entrar sin asignar. El `= null!` en `OriginadoPor` le dice al compilador "confiá, EF lo va a llenar", evitando la advertencia de nulo sin hacer la propiedad anulable.

Y por último `Domain/Accion.cs`:

```csharp
namespace TicketSystem.Domain;

public class Accion {
    public int Id { get; set; }
    public int TicketId { get; set; }
    public string Descripcion { get; set; } = string.Empty;
    public int RegistradaPorId { get; set; }
    public DateTime Fecha { get; set; }
    public bool Realizada { get; set; }

    // Navegación
    public Ticket Ticket { get; set; } = null!;
    public Usuario RegistradaPor { get; set; } = null!;
}
```

> **El corazón del requerimiento "acciones pasadas o futuras":** el campo `Fecha` resuelve todo sin tablas extra. Fecha pasada → ya se hizo. Fecha futura → compromiso. El flag `Realizada` permite marcar cuándo un compromiso futuro efectivamente se cumplió.

## 4. El DbContext

El `DbContext` es el puente entre las entidades y la base. Cada `DbSet<T>` es una tabla.

Creá `Data/TicketContext.cs`:

```csharp
using Microsoft.EntityFrameworkCore;
using TicketSystem.Domain;

namespace TicketSystem.Data;

public class TicketContext : DbContext {
    public DbSet<Usuario> Usuarios => Set<Usuario>();
    public DbSet<Ticket> Tickets => Set<Ticket>();
    public DbSet<Accion> Acciones => Set<Accion>();

    public TicketContext(DbContextOptions<TicketContext> options)
        : base(options) {
    }

    protected override void OnModelCreating(ModelBuilder modelBuilder) {
        modelBuilder.Entity<Ticket>(entity => {
            entity.HasOne(t => t.OriginadoPor)
                  .WithMany(u => u.TicketsOriginados)
                  .HasForeignKey(t => t.OriginadoPorId)
                  .OnDelete(DeleteBehavior.Restrict);

            entity.HasOne(t => t.Responsable)
                  .WithMany(u => u.TicketsAsignados)
                  .HasForeignKey(t => t.ResponsableId)
                  .OnDelete(DeleteBehavior.Restrict);
        });

        modelBuilder.Entity<Accion>(entity => {
            entity.HasOne(a => a.Ticket)
                  .WithMany(t => t.Acciones)
                  .HasForeignKey(a => a.TicketId)
                  .OnDelete(DeleteBehavior.Cascade);

            entity.HasOne(a => a.RegistradaPor)
                  .WithMany()
                  .HasForeignKey(a => a.RegistradaPorId)
                  .OnDelete(DeleteBehavior.Restrict);
        });
    }
}
```

> **Por qué configuramos las relaciones a mano:** cuando una entidad tiene **dos** FK hacia el mismo tipo (Ticket → Usuario, por origen y por responsable), EF Core no puede adivinar qué propiedad de navegación corresponde a cada una. Hay que decírselo explícito con `HasOne / WithMany / HasForeignKey`.
>
> **`Restrict` vs `Cascade`:** con `Restrict`, EF no te deja borrar un usuario si tiene tickets (te protege de borrados en cadena accidentales). Con `Cascade` en las acciones, sí: borrar un ticket borra sus acciones, porque una acción no tiene sentido sin su ticket.

## 5. Los repositorios

Un repositorio aísla el acceso a datos detrás de una interface. El resto del sistema pide datos sin saber que abajo hay EF Core.

### Repositorio de Ticket

Creá `Repositories/ITicketRepository.cs`:

```csharp
using TicketSystem.Domain;

namespace TicketSystem.Repositories;

public interface ITicketRepository {
    Task<Ticket?> ObtenerPorIdAsync(int id);
    Task<IEnumerable<Ticket>> ObtenerTodosAsync();
    Task<IEnumerable<Ticket>> ObtenerPorEstadoAsync(EstadoTicket estado);
    Task AgregarAsync(Ticket ticket);
    Task AgregarAccionAsync(Accion accion);
    Task GuardarCambiosAsync();
}
```

Y su implementación, `Repositories/TicketRepository.cs`:

```csharp
using Microsoft.EntityFrameworkCore;
using TicketSystem.Data;
using TicketSystem.Domain;

namespace TicketSystem.Repositories;

public class TicketRepository : ITicketRepository {
    private readonly TicketContext context;

    public TicketRepository(TicketContext context) {
        this.context = context;
    }

    public async Task<Ticket?> ObtenerPorIdAsync(int id) {
        return await this.context.Tickets
            .Include(t => t.OriginadoPor)
            .Include(t => t.Responsable)
            .Include(t => t.Acciones)
                .ThenInclude(a => a.RegistradaPor)
            .FirstOrDefaultAsync(t => t.Id == id);
    }

    public async Task<IEnumerable<Ticket>> ObtenerTodosAsync() {
        return await this.context.Tickets
            .Include(t => t.OriginadoPor)
            .Include(t => t.Responsable)
            .ToListAsync();
    }

    public async Task<IEnumerable<Ticket>> ObtenerPorEstadoAsync(EstadoTicket estado) {
        return await this.context.Tickets
            .Include(t => t.OriginadoPor)
            .Include(t => t.Responsable)
            .Where(t => t.Estado == estado)
            .ToListAsync();
    }

    public async Task AgregarAsync(Ticket ticket) {
        await this.context.Tickets.AddAsync(ticket);
    }

    public async Task AgregarAccionAsync(Accion accion) {
        await this.context.Acciones.AddAsync(accion);
    }

    public async Task GuardarCambiosAsync() {
        await this.context.SaveChangesAsync();
    }
}
```

> **`Include` / `ThenInclude`:** EF Core por defecto NO trae las propiedades de navegación (las relaciones). Si querés que `ticket.OriginadoPor` venga lleno, hay que pedirlo con `Include`. El `ThenInclude` baja un nivel más: de la acción a quién la registró.
>
> **Por qué `GuardarCambiosAsync` está separado:** el repo no persiste en cada método. Acumula cambios y los guarda cuando el servicio lo decide. Así varias operaciones pueden ir en una sola transacción.

### Repositorio de Usuario

Creá `Repositories/IUsuarioRepository.cs`:

```csharp
using TicketSystem.Domain;

namespace TicketSystem.Repositories;

public interface IUsuarioRepository {
    Task<Usuario?> ObtenerPorIdAsync(int id);
    Task<Usuario?> ObtenerPorEmailAsync(string email);
    Task<Usuario?> ObtenerPorTokenAsync(string token);
    Task<IEnumerable<Usuario>> ObtenerTodosAsync();
    Task AgregarAsync(Usuario usuario);
    Task GuardarCambiosAsync();
}
```

Y `Repositories/UsuarioRepository.cs`:

```csharp
using Microsoft.EntityFrameworkCore;
using TicketSystem.Data;
using TicketSystem.Domain;

namespace TicketSystem.Repositories;

public class UsuarioRepository : IUsuarioRepository {
    private readonly TicketContext context;

    public UsuarioRepository(TicketContext context) {
        this.context = context;
    }

    public async Task<Usuario?> ObtenerPorIdAsync(int id) {
        return await this.context.Usuarios.FindAsync(id);
    }

    public async Task<Usuario?> ObtenerPorEmailAsync(string email) {
        return await this.context.Usuarios
            .FirstOrDefaultAsync(u => u.Email == email);
    }

    public async Task<Usuario?> ObtenerPorTokenAsync(string token) {
        return await this.context.Usuarios
            .FirstOrDefaultAsync(u => u.Token == token);
    }

    public async Task<IEnumerable<Usuario>> ObtenerTodosAsync() {
        return await this.context.Usuarios.ToListAsync();
    }

    public async Task AgregarAsync(Usuario usuario) {
        await this.context.Usuarios.AddAsync(usuario);
    }

    public async Task GuardarCambiosAsync() {
        await this.context.SaveChangesAsync();
    }
}
```

> **`FindAsync` vs `FirstOrDefaultAsync`:** `FindAsync` busca por clave primaria y, si la entidad ya está en memoria, ni va a la base. `FirstOrDefaultAsync` lo usamos cuando buscamos por otra columna (email, token) con una condición.

## 6. El helper de contraseñas

**Nunca** se guarda una contraseña en texto plano. Guardamos un *hash* (resultado irreversible) junto con un *salt* (valor aleatorio que hace único cada hash).

Creá `Security/PasswordHasher.cs`:

```csharp
using System.Security.Cryptography;

namespace TicketSystem.Security;

public static class PasswordHasher {
    private const int SaltSize = 16;      // 128 bits
    private const int HashSize = 32;      // 256 bits
    private const int Iterations = 100_000;

    public static (string Hash, string Salt) Hashear(string password) {
        byte[] salt = RandomNumberGenerator.GetBytes(SaltSize);
        byte[] hash = Pbkdf2(password, salt);

        return (Convert.ToBase64String(hash), Convert.ToBase64String(salt));
    }

    public static bool Verificar(string password, string hashGuardado, string saltGuardado) {
        byte[] salt = Convert.FromBase64String(saltGuardado);
        byte[] hash = Pbkdf2(password, salt);
        byte[] esperado = Convert.FromBase64String(hashGuardado);

        return CryptographicOperations.FixedTimeEquals(hash, esperado);
    }

    private static byte[] Pbkdf2(string password, byte[] salt) {
        return Rfc2898DeriveBytes.Pbkdf2(
            password,
            salt,
            Iterations,
            HashAlgorithmName.SHA256,
            HashSize);
    }
}
```

> **PBKDF2** aplica el hash 100.000 veces a propósito: hace que probar contraseñas por fuerza bruta sea lentísimo para un atacante. Viene en .NET, sin paquetes externos.
>
> **El salt** es por qué dos usuarios con la misma contraseña terminan con hashes distintos: se mezcla un valor aleatorio antes de hashear.
>
> **`FixedTimeEquals`** compara siempre en el mismo tiempo, aunque los bytes difieran desde el primero. Un `==` común "corta" apenas encuentra una diferencia, y midiendo cuánto tarda un atacante podría deducir el hash byte a byte (ataque de *timing*).

## 7. Los servicios

Los servicios contienen la **lógica del sistema**. Hablan con los repositorios (vía interface) y exponen operaciones de negocio.

### Servicio de autenticación

Creá `Services/AuthService.cs`:

```csharp
using System.Security.Cryptography;
using TicketSystem.Domain;
using TicketSystem.Repositories;
using TicketSystem.Security;

namespace TicketSystem.Services;

public class AuthService {
    private readonly IUsuarioRepository repositorio;
    private static readonly TimeSpan DuracionSesion = TimeSpan.FromHours(8);

    public AuthService(IUsuarioRepository repositorio) {
        this.repositorio = repositorio;
    }

    public async Task<Usuario> RegistrarAsync(
        string nombre,
        string email,
        string password,
        TipoUsuario tipo) {
        var existente = await this.repositorio.ObtenerPorEmailAsync(email);
        if (existente is not null) {
            throw new InvalidOperationException("Ya existe un usuario con ese email.");
        }

        var (hash, salt) = PasswordHasher.Hashear(password);

        var usuario = new Usuario {
            Nombre = nombre,
            Email = email,
            Tipo = tipo,
            PasswordHash = hash,
            PasswordSalt = salt
        };

        await this.repositorio.AgregarAsync(usuario);
        await this.repositorio.GuardarCambiosAsync();
        return usuario;
    }

    public async Task<string?> LoginAsync(string email, string password) {
        var usuario = await this.repositorio.ObtenerPorEmailAsync(email);
        if (usuario is null) {
            return null;
        }

        bool valido = PasswordHasher.Verificar(
            password, usuario.PasswordHash, usuario.PasswordSalt);

        if (!valido) {
            return null;
        }

        usuario.Token = GenerarToken();
        usuario.TokenExpira = DateTime.Now.Add(DuracionSesion);
        await this.repositorio.GuardarCambiosAsync();

        return usuario.Token;
    }

    public async Task<Usuario?> ValidarTokenAsync(string token) {
        var usuario = await this.repositorio.ObtenerPorTokenAsync(token);
        if (usuario is null) {
            return null;
        }

        if (usuario.TokenExpira is null || usuario.TokenExpira < DateTime.Now) {
            return null;
        }

        return usuario;
    }

    public async Task LogoutAsync(string token) {
        var usuario = await this.repositorio.ObtenerPorTokenAsync(token);
        if (usuario is not null) {
            usuario.Token = null;
            usuario.TokenExpira = null;
            await this.repositorio.GuardarCambiosAsync();
        }
    }

    private static string GenerarToken() {
        byte[] bytes = RandomNumberGenerator.GetBytes(32);
        return Convert.ToBase64String(bytes);
    }
}
```

> **El flujo de sesión:** `LoginAsync` verifica la contraseña, y si está bien genera un token aleatorio con su vencimiento (8 horas). `ValidarTokenAsync` chequea las dos cosas: que el token exista y que no haya vencido. `LogoutAsync` simplemente borra el token.
>
> **`RandomNumberGenerator`, no `Random`:** el token tiene que ser impredecible. `Random` es predecible (sirve para juegos, no para seguridad); `RandomNumberGenerator` es criptográficamente seguro.

### Servicio de tickets

Creá `Services/TicketService.cs`:

```csharp
using TicketSystem.Domain;
using TicketSystem.Repositories;

namespace TicketSystem.Services;

public class TicketService {
    private readonly ITicketRepository repositorio;

    public TicketService(ITicketRepository repositorio) {
        this.repositorio = repositorio;
    }

    public async Task<Ticket> CrearTicketAsync(
        string titulo,
        string? descripcion,
        int originadoPorId,
        int? responsableId = null) {
        var ticket = new Ticket {
            Titulo = titulo,
            Descripcion = descripcion,
            OriginadoPorId = originadoPorId,
            ResponsableId = responsableId,
            Estado = EstadoTicket.Abierto,
            FechaCreacion = DateTime.Now
        };

        await this.repositorio.AgregarAsync(ticket);
        await this.repositorio.GuardarCambiosAsync();
        return ticket;
    }

    public async Task<Ticket?> ObtenerTicketAsync(int id) {
        return await this.repositorio.ObtenerPorIdAsync(id);
    }

    public async Task<IEnumerable<Ticket>> ListarTicketsAsync() {
        return await this.repositorio.ObtenerTodosAsync();
    }

    public async Task<IEnumerable<Ticket>> ListarPorEstadoAsync(EstadoTicket estado) {
        return await this.repositorio.ObtenerPorEstadoAsync(estado);
    }

    public async Task CambiarEstadoAsync(int ticketId, EstadoTicket nuevoEstado) {
        var ticket = await this.repositorio.ObtenerPorIdAsync(ticketId)
            ?? throw new InvalidOperationException($"No existe el ticket {ticketId}.");

        ticket.Estado = nuevoEstado;
        await this.repositorio.GuardarCambiosAsync();
    }

    public async Task AsignarResponsableAsync(int ticketId, int responsableId) {
        var ticket = await this.repositorio.ObtenerPorIdAsync(ticketId)
            ?? throw new InvalidOperationException($"No existe el ticket {ticketId}.");

        ticket.ResponsableId = responsableId;
        await this.repositorio.GuardarCambiosAsync();
    }

    public async Task<Accion> RegistrarAccionAsync(
        int ticketId,
        string descripcion,
        int registradaPorId,
        DateTime fecha) {
        var ticket = await this.repositorio.ObtenerPorIdAsync(ticketId)
            ?? throw new InvalidOperationException($"No existe el ticket {ticketId}.");

        var accion = new Accion {
            TicketId = ticketId,
            Descripcion = descripcion,
            RegistradaPorId = registradaPorId,
            Fecha = fecha,
            Realizada = fecha <= DateTime.Now
        };

        await this.repositorio.AgregarAccionAsync(accion);
        await this.repositorio.GuardarCambiosAsync();
        return accion;
    }

    public async Task MarcarAccionRealizadaAsync(int ticketId, int accionId) {
        var ticket = await this.repositorio.ObtenerPorIdAsync(ticketId)
            ?? throw new InvalidOperationException($"No existe el ticket {ticketId}.");

        var accion = ticket.Acciones.FirstOrDefault(a => a.Id == accionId)
            ?? throw new InvalidOperationException($"No existe la acción {accionId} en el ticket.");

        accion.Realizada = true;
        await this.repositorio.GuardarCambiosAsync();
    }
}
```

> **El `?? throw`:** si `ObtenerPorIdAsync` devuelve `null`, lanzamos la excepción en la misma línea. Es el operador de coalescencia nula combinado con `throw` como expresión: corto y claro.
>
> **`Realizada = fecha <= DateTime.Now`:** acá se materializa la regla. Si registrás una acción con fecha pasada, queda como ya hecha. Si la fecha es futura, queda pendiente (compromiso), y después se cierra con `MarcarAccionRealizadaAsync`.

## 8. Los DTOs

Los **DTOs** (Data Transfer Objects) son los objetos que viajan por HTTP. Usamos `record` porque son inmutables y concisos: ideal para datos que solo transportan.

Creá `Api/Dtos.cs`:

```csharp
using TicketSystem.Domain;

namespace TicketSystem.Api;

public record RegistroDto(string Nombre, string Email, string Password, TipoUsuario Tipo);
public record LoginDto(string Email, string Password);
public record CrearTicketDto(string Titulo, string? Descripcion, int? ResponsableId);
public record CambiarEstadoDto(EstadoTicket Estado);
public record AsignarResponsableDto(int ResponsableId);
public record RegistrarAccionDto(string Descripcion, DateTime Fecha);
```

> **Por qué DTOs y no las entidades directo:** nunca exponemos la entidad `Usuario` en la API, porque tiene el `PasswordHash`. Los DTOs definen exactamente qué entra y qué sale, sin filtrar datos sensibles ni provocar ciclos infinitos al serializar las navegaciones.

## 9. El helper y la verificación de autenticación

Necesitamos dos piezas para la capa HTTP: un **helper** para leer el usuario logueado y una **extensión reutilizable** para proteger rutas.

Creá `Api/AuthHelper.cs`:

```csharp
using TicketSystem.Domain;

namespace TicketSystem.Api;

public static class AuthHelper {
    public static Usuario? UsuarioActual(this HttpContext context) {
        return context.Items["Usuario"] as Usuario;
    }
}
```

> **`HttpContext.Items`** es un diccionario que vive solo durante el request actual. El middleware Bearer va a dejar ahí al usuario, y cualquier endpoint lo lee con `ctx.UsuarioActual()`. Es un método de extensión (`this HttpContext`) para que se lea natural.

Creá `Api/AuthExtensions.cs`:

```csharp
using TicketSystem.Domain;

namespace TicketSystem.Api;

public static class AuthExtensions {
    public static TBuilder RequiereAuth<TBuilder>(this TBuilder builder)
        where TBuilder : IEndpointConventionBuilder {
        return builder.AddEndpointFilter(async (context, next) => {
            var usuario = context.HttpContext.Items["Usuario"] as Usuario;
            if (usuario is null) {
                return Results.Unauthorized();
            }

            return await next(context);
        });
    }
}
```

> **Qué hace la extensión:** envuelve un filtro inline y le da un nombre reutilizable. Si no hay usuario en `HttpContext.Items` (porque no vino token o era inválido), corta con `401 Unauthorized` y el handler ni se ejecuta. Al exponerlo como `.RequiereAuth()`, la misma verificación sirve tanto para un grupo entero como para un endpoint aislado, sin copiar y pegar la lambda.

## 10. El seeder de datos (solo desarrollo)

Para no registrar todo a mano y, sobre todo, para tener material variado con que probar los filtros por estado, precargamos varios usuarios y tickets en distintos estados, cada uno con acciones pasadas y futuras mezcladas. El seeder **es solo para desarrollo**: en producción NUNCA se siembran usuarios con contraseñas conocidas ni datos de prueba.

Creá `Data/SeedData.cs`:

```csharp
using TicketSystem.Domain;
using TicketSystem.Security;

namespace TicketSystem.Data;

// ATENCION: este seeder es EXCLUSIVAMENTE para desarrollo y pruebas.
// Crea usuarios con contrasenas conocidas. NUNCA debe ejecutarse en produccion.
public static class SeedData {
    public static async Task SembrarAsync(TicketContext context) {
        // Si ya hay usuarios, asumimos que la base ya fue sembrada y salimos.
        // Esto hace que el seeder sea idempotente: correrlo dos veces no duplica.
        if (context.Usuarios.Any()) {
            return;
        }

        // --- Usuarios ---
        // Dos internos (los que resuelven) y tres clientes (los que abren tickets).
        var ana = CrearUsuario("Soporte Ana", "ana@empresa.com", "clave1234", TipoUsuario.Interno);
        var beto = CrearUsuario("Soporte Beto", "beto@empresa.com", "clave1234", TipoUsuario.Interno);
        var clara = CrearUsuario("Clara Cliente", "clara@cliente.com", "clave5678", TipoUsuario.Cliente);
        var diego = CrearUsuario("Diego Cliente", "diego@cliente.com", "clave5678", TipoUsuario.Cliente);
        var elsa = CrearUsuario("Elsa Cliente", "elsa@cliente.com", "clave5678", TipoUsuario.Cliente);

        await context.Usuarios.AddRangeAsync(ana, beto, clara, diego, elsa);
        await context.SaveChangesAsync();

        // A partir de aca los Id ya estan asignados por la base.
        var hoy = DateTime.Now;

        // --- Tickets ---
        // Usamos una fecha base relativa a "hoy" para que las acciones pasadas
        // y futuras queden siempre coherentes, corras esto cuando lo corras.

        // 1) CERRADO: ya resuelto, todas sus acciones en el pasado.
        var t1 = CrearTicket(
            "No llegan los mails de aviso",
            "Deje de recibir notificaciones por correo",
            originadoPor: clara, responsable: ana,
            estado: EstadoTicket.Cerrado, creado: hoy.AddDays(-10));

        // 2) EN PROCESO: arrancado, con trabajo hecho y un compromiso pendiente.
        var t2 = CrearTicket(
            "La factura sale con el total mal",
            "El IVA se calcula sobre el monto equivocado",
            originadoPor: diego, responsable: beto,
            estado: EstadoTicket.EnProceso, creado: hoy.AddDays(-3));

        // 3) EN PROCESO: otro mas, para que el filtro devuelva varios.
        var t3 = CrearTicket(
            "La app se cierra al exportar PDF",
            "Al tocar 'Exportar' la pantalla se va a blanco",
            originadoPor: elsa, responsable: ana,
            estado: EstadoTicket.EnProceso, creado: hoy.AddDays(-1));

        // 4) ABIERTO: recien entrado, sin responsable todavia.
        var t4 = CrearTicket(
            "Quiero cambiar mi correo de contacto",
            "Necesito actualizar el email de mi cuenta",
            originadoPor: clara, responsable: null,
            estado: EstadoTicket.Abierto, creado: hoy.AddHours(-2));

        await context.Tickets.AddRangeAsync(t1, t2, t3, t4);
        await context.SaveChangesAsync();

        // --- Acciones ---
        // Mezcla deliberada de pasadas (Realizada se infiere true) y futuras
        // (compromisos, Realizada false). Replicamos la regla del servicio aca.

        var acciones = new List<Accion> {
            // Ticket 1 (cerrado): historia completa, todo en el pasado.
            CrearAccion(t1, "Reproduje el problema en el servidor de prueba", ana, hoy.AddDays(-9)),
            CrearAccion(t1, "Detecte el filtro de spam mal configurado", ana, hoy.AddDays(-8)),
            CrearAccion(t1, "Aplique el fix y confirme con la clienta", ana, hoy.AddDays(-7)),

            // Ticket 2 (en proceso): algo hecho + un compromiso a futuro.
            CrearAccion(t2, "Revise la formula de calculo del IVA", beto, hoy.AddDays(-2)),
            CrearAccion(t2, "Reunion con contabilidad para validar la alicuota", beto, hoy.AddDays(2)),

            // Ticket 3 (en proceso): solo un compromiso pendiente.
            CrearAccion(t3, "Analizar los logs de la exportacion", ana, hoy.AddDays(1)),

            // Ticket 4 (abierto): un compromiso de primer contacto.
            CrearAccion(t4, "Llamar a la clienta para verificar identidad", ana, hoy.AddDays(1)),
        };

        await context.Acciones.AddRangeAsync(acciones);
        await context.SaveChangesAsync();
    }

    // --- Helpers privados: arman las entidades sin repetir codigo ---

    private static Usuario CrearUsuario(
        string nombre, string email, string password, TipoUsuario tipo) {
        var (hash, salt) = PasswordHasher.Hashear(password);
        return new Usuario {
            Nombre = nombre,
            Email = email,
            Tipo = tipo,
            PasswordHash = hash,
            PasswordSalt = salt
        };
    }

    private static Ticket CrearTicket(
        string titulo, string descripcion,
        Usuario originadoPor, Usuario? responsable,
        EstadoTicket estado, DateTime creado) {
        return new Ticket {
            Titulo = titulo,
            Descripcion = descripcion,
            OriginadoPorId = originadoPor.Id,
            ResponsableId = responsable?.Id,
            Estado = estado,
            FechaCreacion = creado
        };
    }

    private static Accion CrearAccion(
        Ticket ticket, string descripcion, Usuario registradaPor, DateTime fecha) {
        return new Accion {
            TicketId = ticket.Id,
            Descripcion = descripcion,
            RegistradaPorId = registradaPor.Id,
            Fecha = fecha,
            // Misma regla que el servicio: pasado -> hecho, futuro -> pendiente.
            Realizada = fecha <= DateTime.Now
        };
    }
}
```

Esto deja la base con un panorama realista: **5 usuarios** (2 internos, 3 clientes) y **4 tickets** repartidos en `Cerrado` (1), `EnProceso` (2) y `Abierto` (1), con un total de 7 acciones donde conviven historial pasado y compromisos a futuro.

> **Por qué fechas relativas a `hoy` (`hoy.AddDays(-9)`, `hoy.AddDays(2)`):** si pusiéramos fechas fijas tipo `"2026-05-30"`, con el correr de las semanas todas quedarían en el pasado y perderíamos los casos de "compromiso futuro". Anclando todo a `DateTime.Now`, la mezcla pasado/futuro se mantiene siempre, sin importar cuándo corras el seeder.
>
> **Por qué los helpers privados (`CrearUsuario`, `CrearTicket`, `CrearAccion`):** sin ellos, armar 5 usuarios + 4 tickets + 7 acciones sería un muro de inicializadores repetidos. Cada helper concentra el patrón de creación en un solo lugar; el cuerpo del seeder queda legible, casi como leer los datos en una tabla.
>
> **Por qué pasamos los objetos (`originadoPor: clara`) y no los Id:** es más legible y a prueba de errores. Como sembramos usuarios y guardamos primero, cuando los helpers leen `.Id` ya están asignados. El `responsable?.Id` con el `?` permite el caso del ticket abierto sin responsable (`null`).
>
> **Por qué dos bloques de `SaveChangesAsync` encadenados:** mismo motivo que antes, escalado. Guardamos usuarios → obtenemos sus Id → creamos tickets que los referencian → guardamos tickets → obtenemos sus Id → creamos acciones que los referencian. Cada nivel necesita los Id del anterior.
>
> **El comentario de advertencia en el código** no es decorativo: deja explícito que esto jamás va a producción. Un seeder con contraseñas conocidas en un sistema real es un agujero de seguridad.

Acá juntamos todo: registro de servicios, middleware como funciones inline, y los endpoints agrupados.

Reemplazá **todo** el contenido de `Program.cs` por esto:

```csharp
using System.Diagnostics;
using System.Text.Json.Serialization;
using Microsoft.EntityFrameworkCore;
using TicketSystem.Api;
using TicketSystem.Data;
using TicketSystem.Domain;
using TicketSystem.Repositories;
using TicketSystem.Services;

var builder = WebApplication.CreateBuilder(args);

    options.SerializerOptions.Converters.Add(new JsonStringEnumConverter());
});

    options.UseSqlite("Data Source=tickets.db"));

builder.Services.AddScoped<IUsuarioRepository, UsuarioRepository>();
builder.Services.AddScoped<TicketService>();
builder.Services.AddScoped<AuthService>();

var app = builder.Build();

// --- Base de datos: code-first sin migraciones ---
// EnsureCreated() crea el archivo y las tablas a partir de las entidades
// si todavia no existen. Ademas, SOLO en desarrollo, sembramos datos de prueba.
using (var scope = app.Services.CreateScope()) {
    var context = scope.ServiceProvider.GetRequiredService<TicketContext>();
    await context.Database.EnsureCreatedAsync();

    if (app.Environment.IsDevelopment()) {
        await SeedData.SembrarAsync(context);
    }
}

// ============ MIDDLEWARE COMO FUNCIONES ============

// 1° Logging: mide tiempo y registra método, ruta y status.
app.Use(async (context, next) => {
    var logger = context.RequestServices
        .GetRequiredService<ILogger<Program>>();
    var cronometro = Stopwatch.StartNew();

    logger.LogInformation("-> {Metodo} {Ruta}",
        context.Request.Method, context.Request.Path);

    await next(context);

    cronometro.Stop();
    logger.LogInformation("<- {Metodo} {Ruta} respondio {Status} en {Ms}ms",
        context.Request.Method, context.Request.Path,

// 2° Bearer: si viene un token valido, deja el usuario en HttpContext.Items.
app.Use(async (context, next) => {

    var header = context.Request.Headers.Authorization.ToString();

    if (header.StartsWith("Bearer ", StringComparison.OrdinalIgnoreCase)) {
        var token = header["Bearer ".Length..].Trim();
        var usuario = await auth.ValidarTokenAsync(token);

        if (usuario is not null) {
            context.Items["Usuario"] = usuario;
        }
    }

    await next(context);

authGroup.MapPost("/login", async (LoginDto dto, AuthService auth) => {

        ? Results.Unauthorized()

});

authGroup.MapPost("/logout", async (HttpContext ctx, AuthService auth) => {

// ============ GRUPO PROTEGIDO: /tickets ============

    .RequiereAuth();

    var tickets = await svc.ListarTicketsAsync();
    return Results.Ok(tickets.Select(MapearTicket));
});

ticketsGroup.MapGet("/{id:int}", async (int id, TicketService svc) => {

});

ticketsGroup.MapGet("/estado/{estado}", async (EstadoTicket estado, TicketService svc) => {

});

            id, dto.Descripcion, usuario.Id, dto.Fecha);
        return Results.Created($"/tickets/{id}/acciones/{accion.Id}",

    t.Id,
    t.Titulo,
    t.Descripcion,
```

Tres decisiones de diseño que conviene entender:

**El orden del pipeline importa.** El logging va primero para registrar absolutamente todo, incluso requests que después fallan en autenticación. El Bearer va segundo, así cuando un endpoint corre, el usuario ya está identificado.

**Servicios scoped dentro de un middleware.** Los delegates de `app.Use` no son scoped, así que para usar `AuthService` (que sí lo es, porque arrastra el `DbContext`) lo pedimos con `context.RequestServices.GetRequiredService<>()` dentro del delegate. Eso respeta el ciclo de vida por request.

**`.RequiereAuth()` evita duplicación.** La autorización sigue siendo un filtro de endpoint, pero envuelta en un método de extensión reutilizable. Eso deja la protección con una sintaxis corta y consistente tanto en `POST /auth/logout` como en todo el grupo `/tickets`.

**Enums como texto en JSON.** `JsonStringEnumConverter` hace que el cliente pueda mandar y recibir `"EnProceso"`, `"Abierto"` o `"Cerrado"` en vez de números como `1` o `2`. Sin esa línea, el `PUT /tickets/{id}/estado` con un body como `{ "estado": "EnProceso" }` termina en `400 Bad Request` porque el binder intenta leer el enum solo como entero.

**`MapearTicket` aplana la entidad.** Si devolvieras un `Ticket` directo, el serializador entraría en loop: Ticket → Usuario → TicketsOriginados → Ticket... El mapeo a un objeto plano corta el ciclo y, de paso, oculta el `PasswordHash`.

## 12. Crear la base con code-first

No usamos migraciones. La base se crea sola: el bloque `EnsureCreatedAsync()` que pusimos en `Program.cs` inspecciona las entidades al arrancar y genera el archivo `tickets.db` con todas las tablas si todavía no existe. No hay comandos extra que correr.

Simplemente levantá la API:

```bash
dotnet run
```

En el primer arranque pasan tres cosas, en orden: se crea `tickets.db` con sus tablas, y como estamos en entorno de desarrollo, el seeder precarga los 5 usuarios, los 4 tickets y sus acciones. La consola te va a mostrar la URL (algo como `http://localhost:5000`). Dejala corriendo.

> **Code-first con `EnsureCreated` vs migraciones:** `EnsureCreated` es perfecto para aprender y prototipar porque es cero configuración: crea el esquema y listo. Su límite es que **no sabe evolucionar** una base ya creada. Si después cambiás una entidad, `EnsureCreated` no toca la tabla existente; tendrías que borrar `tickets.db` para que la regenere (perdiendo los datos). Para un sistema real que evoluciona, ahí sí se usan migraciones. Para esta práctica, code-first directo es lo más simple.
>
> **¿Querés empezar de cero?** Borrá el archivo `tickets.db` y volvé a correr `dotnet run`: se recrea y se vuelve a sembrar.

## Resumen: cómo encajan las piezas

| Capa             | Archivo(s)                   | Responsabilidad                    |
| ---------------- | ---------------------------- | ---------------------------------- |
| **Dominio**      | `Domain/*.cs`                | Entidades y enums                  |
| **Datos**        | `Data/TicketContext.cs`      | Mapeo a SQLite, relaciones         |
| **Seed (dev)**   | `Data/SeedData.cs`           | Datos de prueba, **solo desarrollo** |
| **Repositorios** | `Repositories/*.cs`          | Acceso a datos tras interface      |
| **Seguridad**    | `Security/PasswordHasher.cs` | Hash + salt de contraseñas         |
| **Servicios**    | `Services/*.cs`              | Lógica del sistema                 |
| **API**          | `Api/*.cs` + `Program.cs`    | HTTP, middleware, rutas            |

**Árbol de decisión rápido** para cuando extiendas el sistema:

- ¿Agregar un dato a una entidad? → tocás `Domain/`. Como usamos `EnsureCreated`, borrá `tickets.db` para que se regenere con el campo nuevo (en un sistema real, ahí pasarías a migraciones).
- ¿Una nueva consulta a la base? → método nuevo en la interface del repo + su implementación.
- ¿Una regla de negocio? → va en el `Service`, nunca en el endpoint.
- ¿Un nuevo endpoint? → al grupo que corresponda (`/auth` público, `/tickets` protegido).

Eso es todo: el sistema completo, de la entidad al endpoint, listo para correr y para que lo amplíes.
