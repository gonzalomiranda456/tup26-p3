## Introducción: ¿por qué mirar la BCL?

La **BCL** (Base Class Library) de .NET es el código que más usás en tu vida como programador C#: `List<T>`, `Dictionary<TKey, TValue>`, `Stream`, `FileStream`, `StreamReader`, `IEnumerable<T>`, LINQ, etc. Todo ese código lo diseñaron ingenieros de Microsoft que vienen aplicando principios de diseño hace décadas, y está pulido por millones de desarrolladores que lo usan todos los días.

Por eso, la BCL es un **laboratorio fantástico** para ver SOLID en la práctica. Y hay un detalle importante: **no vas a ver "SOLID puro de libro"**, porque la BCL también tiene que cargar con decisiones de compatibilidad hacia atrás, rendimiento y ergonomía. Vas a ver aplicaciones claras, pero también alguna que otra concesión pragmática. Eso es parte del aprendizaje: **los principios guían, no mandan**.

El objetivo de este tutorial es que cuando termines de leerlo, cada vez que uses `new StreamReader(new GZipStream(new FileStream(...)))` o `foreach` sobre una `List<T>`, veas el diseño detrás y entiendas por qué funciona tan bien.

## Las dos librerías que hacen el trabajo pesado

Los ejemplos están tomados de librerías de distinto nivel: las clásicas de la BCL y el stack web y de acceso a datos moderno de .NET.

**BCL:**
- **`System.IO`**: streams, archivos, lectura/escritura.
- **`System.Collections.Generic`**: listas, diccionarios, enumerables, LINQ.

**Stack moderno:**
- **Entity Framework Core**: ORM para acceso a bases de datos relacionales.
- **ASP.NET Core / Minimal API**: framework web, middleware, routing.
- **Blazor**: framework de componentes UI para la web.

## S — Single Responsibility Principle

> **Una clase debe tener una sola razón para cambiar.**

La BCL es un caso de estudio magistral de SRP porque, en lugar de diseñar clases gigantes que "hagan todo con archivos", los ingenieros de .NET partieron el problema en **piezas muy finas**, cada una con una única responsabilidad.

### Ejemplo 1: el trío `FileStream` + `StreamReader` + `TextReader`

Pensemos en el problema de **leer líneas de un archivo de texto**. Suena simple, ¿no? Pero si lo descomponemos desde primeros principios, hay al menos tres responsabilidades distintas:

1. **Acceder a los bytes** que viven en un disco físico (sectores, handles del sistema operativo, buffering, permisos).
2. **Convertir bytes en caracteres** usando una codificación (UTF-8, UTF-16, Latin-1, etc.).
3. **Agrupar caracteres en líneas** (detectar `\r\n`, `\n`, `\r`).

La BCL tiene **una clase por responsabilidad**:

```csharp
// Responsabilidad 1: bytes desde/hacia un dispositivo
FileStream fs   = new FileStream("datos.txt", FileMode.Open);

// Responsabilidad 2: bytes ↔ caracteres (con encoding)
StreamReader sr = new StreamReader(fs, Encoding.UTF8);

// Responsabilidad 3: caracteres → líneas (heredada de TextReader)
string? linea   = sr.ReadLine();
```

Pensá qué pasaría si todo esto estuviera en una única clase `ArchivoDeTexto`:

- Si cambia el sistema de archivos (agregás S3, FTP, memoria), tenés que modificar esa clase.
- Si aparece un encoding nuevo, tenés que modificar esa misma clase.
- Si cambia la lógica de detección de líneas, la misma clase otra vez.

Tres razones distintas para cambiar → tres responsabilidades → **debe ser tres clases**. Y por eso en .NET **lo son**. Podés verlo claramente en la jerarquía: `FileStream` hereda de `Stream` (bytes), `StreamReader` hereda de `TextReader` (texto), y las dos se *componen*, no se heredan.

### Ejemplo 2: `List<T>` vs `Enumerable` (LINQ)

Mirá esta diferencia sutil pero poderosa. `List<T>` tiene la responsabilidad de **almacenar** elementos en memoria (agregar, quitar, indexar, capacidad, redimensionamiento interno). Nada más.

Pero uno quiere hacer cosas sobre las colecciones: filtrar, mapear, ordenar, agrupar. ¿Esas operaciones son responsabilidad de `List<T>`? **No.** Si lo fueran, cada colección (`List<T>`, `Queue<T>`, `Stack<T>`, `HashSet<T>`, etc.) tendría que reimplementar Where, Select, OrderBy... y cambiar la lógica de filtrado obligaría a tocar **todas** las colecciones.

Los diseñadores de LINQ pusieron esas operaciones en una clase **estática separada**: `System.Linq.Enumerable`. Todos los métodos son **métodos de extensión** sobre `IEnumerable<T>`:

```csharp
// Enumerable.cs, simplificado
public static class Enumerable {
    public static IEnumerable<TSource> Where<TSource>(
        this IEnumerable<TSource> source,
        Func<TSource, bool> predicate) { /* ... */ }

    public static IEnumerable<TResult> Select<TSource, TResult>(
        this IEnumerable<TSource> source,
        Func<TSource, TResult> selector) { /* ... */ }
}
```

Resultado:

- `List<T>` solo almacena. Cambia si cambia cómo se almacena.
- `Enumerable` solo opera. Cambia si cambia cómo se opera.
- Un dev que escriba su propia colección y haga `: IEnumerable<T>` hereda **gratis** todo LINQ.

Es SRP llevado al extremo más elegante: **una clase no implementa operaciones que no le son propias**.

### Ejemplo 3: `DbContext` en Entity Framework Core

EF Core separa las responsabilidades del acceso a datos de forma similar. Un `DbContext` tiene una sola responsabilidad: **representar una sesión con la base de datos** y exponer los conjuntos de datos (`DbSet<T>`) como propiedades.

```csharp
public class TiendaContext : DbContext {
    public DbSet<Producto>  Productos  { get; set; }
    public DbSet<Categoria> Categorias { get; set; }
    public DbSet<Pedido>    Pedidos    { get; set; }

    protected override void OnConfiguring(DbContextOptionsBuilder options)
        => options.UseSqlite("Data Source=tienda.db");
}
```

El `DbContext` **no sabe cómo mapear columnas a propiedades** (eso lo delega a las configuraciones `IEntityTypeConfiguration<T>`), **no sabe cómo ejecutar la SQL** (eso lo delega al driver, `Microsoft.EntityFrameworkCore.Sqlite` en este caso), y **no sabe nada de la lógica de negocio**. Cada pieza tiene su lugar.

### Ejemplo 4: handlers en Minimal API

En Minimal API cada endpoint es un método con **una única responsabilidad**: manejar un tipo de solicitud HTTP específico.

```csharp
// Cada lambda tiene una sola razón para cambiar
app.MapGet("/productos",         ObtenerTodos);
app.MapGet("/productos/{id}",    ObtenerPorId);
app.MapPost("/productos",        Crear);
app.MapPut("/productos/{id}",    Actualizar);
app.MapDelete("/productos/{id}", Eliminar);

// Handler — responsabilidad única: leer y devolver la lista
static async Task<Ok<List<Producto>>> ObtenerTodos(TiendaContext db)
    => TypedResults.Ok(await db.Productos.ToListAsync());

// Handler — responsabilidad única: buscar un producto por id
static async Task<Results<Ok<Producto>, NotFound>> ObtenerPorId(int id, TiendaContext db)
    => await db.Productos.FindAsync(id)
        is Producto p ? TypedResults.Ok(p) : TypedResults.NotFound();
```

Si cambia la lógica de búsqueda, solo cambia `ObtenerPorId`. Si cambia el formato de la lista, solo cambia `ObtenerTodos`. Cada handler tiene una sola razón para cambiar.

### Ejemplo 5: middleware en ASP.NET Core

El pipeline de ASP.NET Core es una cadena de middlewares donde **cada uno tiene una sola responsabilidad**.

```csharp
app.UseHttpsRedirection();  // redirigir HTTP → HTTPS
app.UseAuthentication();    // verificar quién sos
app.UseAuthorization();     // verificar qué podés hacer
app.UseRateLimiting();      // limitar la tasa de solicitudes
app.MapControllers();       // rutear a controladores
```

Un middleware de autenticación no sabe nada de autorización, y viceversa. Si mañana cambia la lógica de rate limiting, **solo cambia ese middleware**. El resto no se entera. Cada pieza tiene una única razón para cambiar.

## O — Open/Closed Principle

> **Abierto a la extensión, cerrado a la modificación.**

Este principio es el que más brilla en .NET, porque **toda la arquitectura de streams** está construida sobre él.

### Ejemplo 1: `Stream` como jerarquía extensible

La clase base `System.IO.Stream` es una abstracción mínima: "un lugar de donde puedo leer bytes y a donde puedo escribir bytes". Su contrato es básicamente:

```csharp
public abstract class Stream : IDisposable {
    public abstract bool CanRead { get; }
    public abstract bool CanWrite { get; }
    public abstract bool CanSeek { get; }
    public abstract int Read(byte[] buffer, int offset, int count);
    public abstract void Write(byte[] buffer, int offset, int count);
    public abstract long Position { get; set; }
    public abstract long Length { get; }
    // ...
}
```

A partir de esa base, la BCL (y cualquier librería de terceros) agrega implementaciones **sin modificar `Stream`**:

- `FileStream` — bytes desde/hacia un archivo.
- `MemoryStream` — bytes en un `byte[]` en memoria.
- `NetworkStream` — bytes desde/hacia un socket.
- `GZipStream`, `DeflateStream`, `BrotliStream` — compresión/descompresión *envolviendo* otro stream.
- `CryptoStream` — cifrado/descifrado envolviendo otro stream.
- `BufferedStream` — agrega buffering a cualquier otro stream.

El detalle clave: `GZipStream` **no es un archivo comprimido**. Es un `Stream` que, cuando le pedís bytes, los saca de **otro stream** (el que vos le pasás) y los descomprime al vuelo. Esto permite componer:

```csharp
using var fs      = new FileStream("datos.gz", FileMode.Open);
using var descomp = new GZipStream(fs, CompressionMode.Decompress);
using var lector  = new StreamReader(descomp, Encoding.UTF8);

string? linea;
while ((linea = lector.ReadLine()) != null)
    Console.WriteLine(linea);
```

Cada línea agrega una capa de comportamiento **sin tocar las clases de abajo**. Si mañana necesitás leer desde un archivo comprimido con GZip y además cifrado con AES, el patrón es idéntico: metés un `CryptoStream` en el medio. Ninguna de las clases existentes sabe de la existencia de las otras, pero todas **colaboran** porque dependen de la misma abstracción: `Stream`.

Esto es el **patrón Decorator** en su forma más limpia, y es OCP aplicado: se pueden agregar capacidades nuevas (cifrado, compresión, buffering, throttling) **sin modificar una sola línea** de `Stream`, `FileStream` ni `StreamReader`.

### Ejemplo 2: LINQ sobre `IEnumerable<T>`

El mismo patrón en otro dominio. LINQ tiene cerca de 50 operadores (`Where`, `Select`, `OrderBy`, `GroupBy`, `Join`, ...). Todos trabajan sobre `IEnumerable<T>` y devuelven `IEnumerable<T>`.

¿Qué te permite esto? **Agregar operadores nuevos sin tocar la BCL**. Podés escribir tu propio método de extensión:

```csharp
public static class MisExtensiones {
    public static IEnumerable<T> SaltearCada<T>(this IEnumerable<T> fuente, int n) {
        int i = 0;
        foreach (var x in fuente)
            if (i++ % n != 0) yield return x;
    }
}

// Uso:
var pares = numeros.Where(x => x % 2 == 0).SaltearCada(3).Select(x => x * 10);
```

`SaltearCada` se compone naturalmente con `Where` y `Select` porque todos hablan el mismo idioma: `IEnumerable<T>`. Microsoft no previó tu operador custom, y sin embargo funciona. Eso es OCP: **el sistema se extiende sin modificarse**.

### Ejemplo 3: `IQueryable<T>` y especificaciones en EF Core

`IQueryable<T>` es el equivalente de `IEnumerable<T>` para consultas que se ejecutan en la base de datos. Permite agregar operadores en el mismo estilo que LINQ, y EF Core los traduce a SQL al final.

```csharp
// Extensiones propias sobre IQueryable — EF Core no las conoce
public static class ProductoQueryExtensions {
    public static IQueryable<Producto> Activos(this IQueryable<Producto> q)
        => q.Where(p => p.Activo);

    public static IQueryable<Producto> DeLaCategoria(this IQueryable<Producto> q, int categoriaId)
        => q.Where(p => p.CategoriaId == categoriaId);

    public static IQueryable<Producto> OrdenadosPorPrecio(this IQueryable<Producto> q)
        => q.OrderBy(p => p.Precio);
}

// Uso: se componen igual que LINQ
var resultado = await db.Productos
    .Activos()
    .DeLaCategoria(3)
    .OrdenadosPorPrecio()
    .ToListAsync();
```

EF Core no fue modificado. Vos no modificaste `DbSet<T>`. Solo **extendiste** el sistema agregando métodos que hablan el idioma de `IQueryable<T>`. La SQL generada incluirá todos los filtros correctamente.

### Ejemplo 4: componentes en Blazor

Blazor está diseñado para extenderse sin modificar los componentes existentes. Podés crear un componente de botón base y especializarlo:

```razor
@* BtnBase.razor — componente base, no se modifica *@
<button class="btn @CssExtra" @onclick="OnClick">@Texto</button>

@code {
    [Parameter] public string Texto    { get; set; } = "";
    [Parameter] public string CssExtra { get; set; } = "";
    [Parameter] public EventCallback  OnClick { get; set; }
}
```

```razor
@* BtnPrimario.razor — extiende sin modificar BtnBase *@
<BtnBase Texto="@Texto" CssExtra="btn-primary" OnClick="@OnClick" />

@code {
    [Parameter] public string Texto   { get; set; } = "";
    [Parameter] public EventCallback OnClick { get; set; }
}
```

```razor
@* BtnPeligro.razor — otra extensión, misma base *@
<BtnBase Texto="@Texto" CssExtra="btn-danger" OnClick="@OnClick" />
```

`BtnBase` nunca se toca. Agregar un `BtnSecundario`, `BtnLink`, o `BtnIcono` solo requiere crear un componente nuevo que lo componga.

### Ejemplo 5: filtros globales en ASP.NET Core

Los filtros (`IActionFilter`, `IExceptionFilter`) permiten agregar comportamiento transversal a los controladores **sin modificarlos**:

```csharp
// Filtro de logging — se agrega sin tocar ningún controlador
public class LogFiltro : IActionFilter {
    private readonly ILogger<LogFiltro> logger;
    public LogFiltro(ILogger<LogFiltro> logger) => this.logger = logger;

    public void OnActionExecuting(ActionExecutingContext ctx)
        => logger.LogInformation("Ejecutando {Action}", ctx.ActionDescriptor.DisplayName);

    public void OnActionExecuted(ActionExecutedContext ctx)
        => logger.LogInformation("Terminó {Action}", ctx.ActionDescriptor.DisplayName);
}

// Registro global — todos los controladores lo reciben sin saber que existe
builder.Services.AddControllers(opts => opts.Filters.Add<LogFiltro>());
```

Si mañana necesitás un filtro de métricas, de auditoría, de compresión — los agregás. Ningún controlador existente se modifica.

## L — Liskov Substitution Principle

> **Una subclase debe poder reemplazar a su clase base sin romper el código que usa la clase base.**

LSP es más sutil y, justamente, acá es donde vemos las **concesiones pragmáticas** de la BCL. Hay ejemplos muy limpios, y hay uno muy famoso de violación que vale la pena entender.

### Ejemplo 1: `IEnumerable<T>` y `foreach` (cumple LSP)

`IEnumerable<T>` es la abstracción más usada de .NET. Su contrato es simple: "podés pedirme un enumerador, y el enumerador te devuelve elementos de a uno".

```csharp
public interface IEnumerable<out T> : IEnumerable {
    IEnumerator<T> GetEnumerator();
}
```

Un `foreach` no sabe si está iterando un `List<T>`, un `HashSet<T>`, un `Dictionary<K,V>.Values`, un archivo leído línea por línea con `File.ReadLines()`, o el resultado de un `Where().Select()`. **Y no le importa.** Cualquier tipo que implemente `IEnumerable<T>` cumple el contrato:

```csharp
void Imprimir<T>(IEnumerable<T> fuente) {
    foreach (var item in fuente)
        Console.WriteLine(item);
}

Imprimir(new List<int> { 1, 2, 3 });           // funciona
Imprimir(new HashSet<int> { 1, 2, 3 });        // funciona
Imprimir(File.ReadLines("datos.txt"));         // funciona (¡streaming!)
Imprimir(Enumerable.Range(1, 1_000_000));      // funciona (lazy)
```

**Todas las implementaciones de `IEnumerable<T>` son intercambiables** desde el punto de vista del cliente. Esto es LSP en su forma más pura y es la razón por la que LINQ, `foreach`, `async foreach` y toda la maquinaria funcional de C# simplemente *funcionan* con cualquier fuente de datos.

### Ejemplo 2: `Stream` — el caso interesante (violación parcial)

Acá viene la parte honesta. `Stream` define métodos como `Seek`, `Position` y `Length`. Pero no todos los streams pueden posicionarse: un `NetworkStream` o un `GZipStream` son **secuenciales por naturaleza**, no podés "volver atrás" en una conexión TCP.

¿Qué hicieron en la BCL? Dos cosas:

**Primero, propiedades de capacidad**. Cada stream declara explícitamente qué puede hacer:

```csharp
public abstract bool CanRead  { get; }
public abstract bool CanWrite { get; }
public abstract bool CanSeek  { get; }
```

**Segundo, excepciones documentadas**. Si llamás a `Seek` en un stream que no soporta seek, tira `NotSupportedException`.

```csharp
using var red = new NetworkStream(socket);
red.Position = 0; // 💥 NotSupportedException
```

Estrictamente hablando, **esto viola LSP**. Un método que recibe un `Stream` no puede confiar en que `Seek` funcione; tiene que chequear `CanSeek` primero. La "sustituibilidad" del contrato base no es completa.

¿Por qué lo hicieron así? Porque la alternativa era partir `Stream` en `ISeekableStream` vs `IForwardOnlyStream`, y eso habría complicado enormemente el uso común. Los ingenieros de .NET priorizaron la **ergonomía** sobre la pureza del principio. Es una concesión consciente.

**La lección que te tenés que llevar**: cuando veas este patrón (capacidades declaradas + excepciones para lo no soportado) en código ajeno o propio, sabé que es un compromiso. A veces es razonable, a veces es una señal de que tu jerarquía está mal pensada. Si tu subtipo tira `NotSupportedException` en la mitad de los métodos, probablemente tendrías que dividir la interfaz. Lo cual nos lleva al siguiente principio.

### Ejemplo 3: `DbSet<T>` como `IQueryable<T>` en EF Core

`DbSet<T>` implementa `IQueryable<T>` e `IEnumerable<T>`. Cualquier método que acepte `IQueryable<Producto>` puede recibir un `DbSet<Producto>` real o una lista en memoria en los tests:

```csharp
// Método que trabaja con la abstracción
static async Task<List<Producto>> BuscarActivos(IQueryable<Producto> fuente)
    => await fuente.Where(p => p.Activo).ToListAsync();

// Producción: fuente es DbSet<Producto> (ejecuta SQL)
var produccion = await BuscarActivos(db.Productos);

// Test: fuente es una lista en memoria (no toca la base de datos)
var enMemoria = new List<Producto> {
    new() { Nombre = "A", Activo = true },
    new() { Nombre = "B", Activo = false }
}.AsQueryable();

var test = await BuscarActivos(enMemoria);
```

El método `BuscarActivos` no sabe si está hablando con SQL Server o con una lista. Ambos subtipos son **completamente intercambiables** desde su punto de vista. Esto es lo que hace que los tests de EF Core sin base de datos sean posibles.

### Ejemplo 4: `IActionResult` y sus implementaciones en ASP.NET Core

Un controlador declara que retorna `IActionResult`. Cualquier implementación concreta puede usarse:

```csharp
IActionResult Responder(bool encontrado, object? dato) {
    if (!encontrado) return new NotFoundResult();
    if (dato is null)  return new NoContentResult();
    return new OkObjectResult(dato);
}
```

`NotFoundResult`, `OkObjectResult`, `CreatedAtActionResult`, `BadRequestObjectResult` — todos implementan `IActionResult`. El framework los ejecuta con la misma llamada a `ExecuteResultAsync`. **Ningún cliente de `IActionResult` se rompe al recibir cualquiera de ellos**, porque todos cumplen el contrato: escribir una respuesta HTTP.

### Ejemplo 5: `RenderFragment` y composición de componentes Blazor

En Blazor, un componente padre que recibe `RenderFragment` acepta **cualquier fragmento de UI** como hijo. No le importa qué componentes concretos contiene:

```razor
@* Layout.razor — acepta cualquier contenido *@
<div class="contenedor">
    <nav>Menú</nav>
    <main>@Body</main>
</div>

@code {
    [Parameter] public RenderFragment Body { get; set; } = default!;
}
```

```razor
@* Uso — cualquier combinación de componentes cumple el contrato *@
<Layout>
    <Body>
        <h1>Bienvenido</h1>
        <ListadoProductos />
    </Body>
</Layout>
```

`Layout` no sabe qué hay dentro de `Body`. Puede ser un texto, una tabla, un formulario, o componentes anidados arbitrariamente. Todos son **intercambiables** porque todos son `RenderFragment`.

## I — Interface Segregation Principle

> **Los clientes no deben depender de interfaces que no usan.**

.NET segregó sus colecciones en interfaces chicas y específicas. Vale la pena mirar cómo.

### Ejemplo 1: la jerarquía de `System.Collections.Generic`

Mirá cómo están separadas las capacidades de las colecciones:

```csharp
public interface IEnumerable<out T> { // solo iterar
    IEnumerator<T> GetEnumerator();
}

public interface IReadOnlyCollection<out T> : IEnumerable<T> { // + contar
    int Count { get; }
}

public interface IReadOnlyList<out T> : IReadOnlyCollection<T> { // + acceder por índice (readonly)
    T this[int index] { get; }
}

public interface ICollection<T> : IEnumerable<T> { // + modificar
    int Count { get; }
    void Add(T item);
    bool Remove(T item);
    void Clear();
    bool Contains(T item);
    bool IsReadOnly { get; }
}

public interface IList<T> : ICollection<T> { // + indexar y modificar
    T this[int index] { get; set; }
    int IndexOf(T item);
    void Insert(int index, T item);
    void RemoveAt(int index);
}
```

Esto te permite expresar **exactamente** lo que necesita cada método:

```csharp
// Solo necesito iterar
void Imprimir<T>(IEnumerable<T> items) { /* ... */ }

// Necesito saber cuántos hay, pero no modificar
string Describir<T>(IReadOnlyCollection<T> items) => $"Tengo {items.Count} elementos";

// Necesito leer por índice sin modificar (ideal para API pública)
T ObtenerTercero<T>(IReadOnlyList<T> items) => items[2];

// Necesito poder agregar
void AgregarVarios<T>(ICollection<T> items, IEnumerable<T> nuevos) {
    foreach (var n in nuevos) 
        items.Add(n);
}
```

Pensá qué pasaría si .NET solo tuviera una `ICollection<T>` gigante con todo adentro. Cada método que solo quiere iterar estaría aceptando un parámetro con más poder del que necesita: podría modificarlo, vaciarlo, etc. El contrato del método sería **mentiroso**. Con interfaces segregadas, el tipo del parámetro **documenta qué vas a hacer con él**.

Caso concreto: si tenés una API pública, aceptar `IReadOnlyList<Pedido>` en lugar de `List<Pedido>` es una forma de **comunicar y garantizar** que el método no va a modificar la lista del llamador. Es contrato y documentación al mismo tiempo.

### Ejemplo 2: `IDisposable` vs `IAsyncDisposable`

Acá ISP se ve en chiquito pero con mucha claridad. El contrato "liberar recursos" empezó siendo:

```csharp
public interface IDisposable {
    void Dispose();
}
```

Pero con la llegada de async/await, algunos recursos necesitan liberarse con operaciones asincrónicas (por ejemplo, un `FileStream` que tiene que hacer flush al disco, o una conexión de red que tiene que mandar un paquete de cierre). ¿Qué hicieron? **No modificaron `IDisposable`**. Crearon una interfaz nueva:

```csharp
public interface IAsyncDisposable {
    ValueTask DisposeAsync();
}
```

Una clase puede implementar una, la otra, o las dos:

```csharp
public class FileStream : Stream, IDisposable, IAsyncDisposable {
    public void Dispose() { /* sincrónico */ }
    public ValueTask DisposeAsync() { /* asincrónico */ }
}
```

Y `using` en C# ahora soporta las dos:

```csharp
using var fs = new FileStream("a.txt", FileMode.Open);          // usa Dispose()
await using var fs2 = new FileStream("a.txt", FileMode.Open);   // usa DisposeAsync()
```

Si `IDisposable` hubiera tenido siempre los dos métodos, **toda clase que implementa `IDisposable`** habría tenido que incluir un `DisposeAsync` que nadie pidió. Con ISP, cada clase declara **solo lo que ofrece**.

### Ejemplo 3: `IApplicationBuilder`, `IServiceCollection` y `IHostBuilder` en ASP.NET Core

El bootstrapping de ASP.NET Core usa tres interfaces, cada una con un contrato diferente y bien delimitado:

```csharp
var builder = WebApplication.CreateBuilder(args);

// IServiceCollection — solo registrar dependencias
builder.Services.AddDbContext<TiendaContext>(opts => opts.UseSqlite("..."));
builder.Services.AddScoped<IRepositorioProductos, RepositorioEF>();
builder.Services.AddControllers();

var app = builder.Build();

// IApplicationBuilder — solo configurar el pipeline de middleware
app.UseHttpsRedirection();
app.UseAuthentication();
app.UseAuthorization();

// IEndpointRouteBuilder — solo mapear rutas
app.MapControllers();
app.MapGet("/salud", () => "OK");

app.Run();
```

`IServiceCollection` no sabe nada de middleware. `IApplicationBuilder` no sabe nada de rutas. Cada interfaz tiene exactamente las operaciones que su responsabilidad requiere. Si una API solo necesita registrar servicios, depende de `IServiceCollection`. No necesita cargar con métodos de routing que no usa.

### Ejemplo 4: `IEntityTypeConfiguration<T>` en EF Core

EF Core permite definir el mapeo de cada entidad en una clase separada que implementa exactamente una interfaz pequeña:

```csharp
public interface IEntityTypeConfiguration<TEntity> {
    void Configure(EntityTypeBuilder<TEntity> builder);
}

// Configuración de Producto — solo configura Producto
public class ProductoConfig : IEntityTypeConfiguration<Producto> {
    public void Configure(EntityTypeBuilder<Producto> builder) {
        builder.HasKey(p => p.Id);
        builder.Property(p => p.Nombre).IsRequired().HasMaxLength(100);
        builder.HasOne(p => p.Categoria).WithMany(c => c.Productos);
    }
}

// DbContext aplica las configuraciones sin saber sus detalles
protected override void OnModelCreating(ModelBuilder modelBuilder) {
    modelBuilder.ApplyConfiguration(new ProductoConfig());
    modelBuilder.ApplyConfiguration(new CategoriaConfig());
}
```

Cada clase de configuración tiene **solo los métodos que necesita**: `Configure`. No hereda de una clase base con veinte métodos de los cuales usa dos.

### Ejemplo 5: `IComponent` vs `ComponentBase` en Blazor

Blazor define una interfaz mínima para los componentes:

```csharp
public interface IComponent {
    void Attach(RenderHandle renderHandle);
    Task SetParametersAsync(ParameterView parameters);
}
```

Eso es todo lo que el framework necesita de un componente. Si querés el ciclo de vida completo (`OnInitialized`, `OnParametersSet`, `ShouldRender`, etc.), usás `ComponentBase` que lo implementa. Pero si escribís un componente de muy bajo nivel que necesita control total del rendering, podés implementar `IComponent` directamente sin arrastrar todo `ComponentBase`:

```csharp
// Componente ultra liviano — solo lo que el framework necesita
public class MiComponenteRaw : IComponent {
    private RenderHandle handle;

    public void Attach(RenderHandle renderHandle) => handle = renderHandle;

    public Task SetParametersAsync(ParameterView parameters) {
        handle.Render(builder => {
            builder.OpenElement(0, "span");
            builder.AddContent(1, "Hola desde IComponent puro");
            builder.CloseElement();
        });
        return Task.CompletedTask;
    }
}
```

Los componentes comunes no dependen de más contrato del que necesitan. ISP en acción.

## D — Dependency Inversion Principle

> **Depender de abstracciones, no de implementaciones concretas.**

### Ejemplo 1: `StreamReader` depende de `Stream`, no de `FileStream`

Mirá la firma del constructor:

```csharp
public StreamReader(Stream stream, Encoding encoding) { /* ... */ }
```

El parámetro es `Stream` — la abstracción. No `FileStream`, no `NetworkStream`. Esto es DIP puro y es **la razón por la que podés hacer todo esto**:

```csharp
// Leer desde un archivo
var reader1 = new StreamReader(new FileStream("a.txt", FileMode.Open));

// Leer desde un archivo comprimido
var reader2 = new StreamReader(
    new GZipStream(new FileStream("a.gz", FileMode.Open),
                   CompressionMode.Decompress));

// Leer desde una respuesta HTTP
using var http = new HttpClient();
var reader3 = new StreamReader(await http.GetStreamAsync("https://..."));

// Leer desde memoria (típico en tests)
var bytes = Encoding.UTF8.GetBytes("hola\nmundo");
var reader4 = new StreamReader(new MemoryStream(bytes));

// Leer desde un string en memoria, sin Stream (StringReader hereda TextReader)
var reader5 = new StringReader("hola\nmundo");
```

`StreamReader` **no sabe ni le importa** de dónde vienen los bytes. Depende de la abstracción `Stream`, y eso le da una flexibilidad enorme. Y fijate el regalo que trae para los tests: podés probar cualquier lógica que lea texto pasándole un `MemoryStream` con datos fabricados, sin tocar el disco. Eso no es casualidad — es consecuencia directa de haber invertido la dependencia.

### Ejemplo 2: `IServiceCollection` y el contenedor de DI de .NET

En ASP.NET Core, DIP no es una opción sino **la forma esperada** de trabajar. Mirá un controller típico:

```csharp
public class TarjetasController : ControllerBase {
    private readonly ILogger<TarjetasController> logger;
    private readonly IRepositorioTarjetas repo;
    private readonly INotificador notificador;

    public TarjetasController(
        ILogger<TarjetasController> logger,
        IRepositorioTarjetas repo,
        INotificador notificador) {
        this.logger = logger;
        this.repo = repo;
        this.notificador = notificador;
    }

    [HttpPost("debitar")]
    public async Task<IActionResult> Debitar(string numero, decimal monto) {
        var tarjeta = await this.repo.ObtenerAsync(numero);
        tarjeta.Debitar(monto);
        await this.repo.GuardarAsync(tarjeta);
        await this.notificador.NotificarAsync(tarjeta.Email, "Débito", $"${monto}");
        this.logger.LogInformation("Débito de {Monto} en {Numero}", monto, numero);
        return Ok();
    }
}
```

El controller **no hace `new` de nada**. Solo depende de tres interfaces: `ILogger<T>`, `IRepositorioTarjetas`, `INotificador`. Cuál implementación concreta se usa se decide en un único lugar, `Program.cs`:

```csharp
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddScoped<IRepositorioTarjetas, RepositorioSqlServer>();
builder.Services.AddScoped<INotificador, NotificadorEmailSmtp>();
builder.Logging.AddConsole();
```

Si mañana cambiás SQL Server por PostgreSQL, o email por SMS, o logs por un servicio remoto, **cambiás una línea**. El controller no se entera.

Y como corolario que ata todo el tutorial: `ILogger<T>` es en sí mismo un ejemplo precioso de todos los principios juntos. Es una interfaz chica (**ISP**) que permite implementaciones intercambiables (**LSP**, **DIP**), donde podés agregar loggers nuevos (archivos, Console, Seq, Datadog, lo que quieras) sin modificar nada existente (**OCP**), y cada implementación tiene una única responsabilidad: registrar logs (**SRP**).

### Ejemplo 3: repositorio con EF Core

DIP sugiere que la capa de negocio no dependa de EF Core directamente. El patrón repositorio crea la interfaz (abstracción) en la capa de dominio, y la implementación con EF Core vive en infraestructura:

```csharp
// Abstracción — definida en la capa de dominio, sin saber nada de EF
public interface IRepositorioProductos {
    Task<Producto?> ObtenerAsync(int id);
    Task<List<Producto>> ListarAsync();
    Task GuardarAsync(Producto producto);
}

// Implementación concreta — sabe de EF Core, vive en la capa de infraestructura
public class RepositorioEF : IRepositorioProductos {
    private readonly TiendaContext db;
    public RepositorioEF(TiendaContext db) => this.db = db;

    public Task<Producto?> ObtenerAsync(int id)      => db.Productos.FindAsync(id).AsTask();
    public Task<List<Producto>> ListarAsync()         => db.Productos.ToListAsync();
    public async Task GuardarAsync(Producto producto) {
        db.Productos.Update(producto);
        await db.SaveChangesAsync();
    }
}

// Registro — en Program.cs
builder.Services.AddScoped<IRepositorioProductos, RepositorioEF>();
```

La lógica de negocio usa `IRepositorioProductos`. No importa si detrás hay EF Core, un JSON en disco, o una lista en memoria para tests.

### Ejemplo 4: `IHttpClientFactory` en ASP.NET Core

`HttpClient` tiene un problema conocido: instanciarlo a mano puede agotar sockets del sistema operativo. ASP.NET Core resuelve esto con `IHttpClientFactory`, una abstracción que gestiona el ciclo de vida de los clientes internamente:

```csharp
// Registro
builder.Services.AddHttpClient("Clima", client => {
    client.BaseAddress = new Uri("https://api.open-meteo.com/");
});

// Consumo — el servicio depende de IHttpClientFactory, no de HttpClient
public class ServicioClima {
    private readonly IHttpClientFactory factory;
    public ServicioClima(IHttpClientFactory factory) => this.factory = factory;

    public async Task<string> ObtenerTemperaturaAsync() {
        var client = factory.CreateClient("Clima");
        return await client.GetStringAsync("v1/forecast?latitude=-26.8&longitude=-65.2&current=temperature_2m");
    }
}
```

`ServicioClima` depende de `IHttpClientFactory`, no de un `HttpClient` concreto. Podés reemplazar la implementación en tests sin hacer requests HTTP reales.

### Ejemplo 5: inyección directa en handlers de Minimal API

En Minimal API, el framework resuelve automáticamente las dependencias como parámetros del handler. El handler depende de abstracciones, no de implementaciones:

```csharp
// El handler pide IRepositorioProductos — no sabe nada de EF ni de ninguna implementación concreta
app.MapPost("/productos", async (
    Producto producto,
    IRepositorioProductos repo,
    ILogger<Program> logger) =>
{
    await repo.GuardarAsync(producto);
    logger.LogInformation("Producto creado: {Nombre}", producto.Nombre);
    return TypedResults.Created($"/productos/{producto.Id}", producto);
});
```

El framework inyecta lo que encuentre registrado para `IRepositorioProductos`. Si en tests usás `builder.Services.AddScoped<IRepositorioProductos, RepositorioFake>()`, el handler recibe el fake sin ningún cambio.

## Mini caso integrador: leer un archivo comprimido con LINQ

Para cerrar, armemos un pipeline que procesa un archivo de JSON Lines comprimido con GZip, filtra registros y los agrega. Fijate cómo, **sin pensar explícitamente en SOLID**, estás usando los cinco principios porque la BCL te los ofrece de fábrica:

```csharp
using System.IO;
using System.IO.Compression;
using System.Text;
using System.Text.Json;

public record Movimiento(string Tarjeta, string Tipo, decimal Monto);

public static class Procesador {
    public static decimal TotalDebitosGold(string rutaArchivoGz) {
        // [DIP] StreamReader depende de Stream, no de archivos
        // [OCP] Apilamos GZipStream sobre FileStream sin tocar nada
        // [SRP] Cada capa tiene una única responsabilidad
        using var archivo = new FileStream(rutaArchivoGz, FileMode.Open);
        using var descomp = new GZipStream(archivo, CompressionMode.Decompress);
        using var lector  = new StreamReader(descomp, Encoding.UTF8);

        // [LSP] El enumerable de abajo es intercambiable con cualquier IEnumerable<string>
        // [ISP] Solo necesitamos iterar (IEnumerable), no modificar
        return LeerLineas(lector)
            .Select(linea => JsonSerializer.Deserialize<Movimiento>(linea)!)
            .Where(m => m.Tipo == "debito" && m.Tarjeta.StartsWith("GOLD"))
            .Sum(m => m.Monto);
    }

    private static IEnumerable<string> LeerLineas(TextReader lector) {
        string? linea;
        while ((linea = lector.ReadLine()) != null)
            yield return linea;
    }
}
```

Ocho líneas de lógica real. Cada decisión de diseño de la BCL te está ayudando:

- Si mañana el archivo viene de una API en vez de disco, cambiás `FileStream` por el stream de `HttpClient`. **El resto no se toca.**
- Si el archivo usa Brotli en vez de GZip, cambiás `GZipStream` por `BrotliStream`. **El resto no se toca.**
- Si en los tests querés probar con un string en memoria, usás `new StringReader(texto)` directamente. **El resto no se toca.**
- Si aparece una regla nueva (agrupar por fecha, por ejemplo), agregás un `.GroupBy()` al pipeline. **Nada existente se toca.**

Esa sensación de "todo compone como Lego" no es un accidente. Es el resultado de haber aplicado SOLID consistentemente durante 20 años de evolución de la plataforma.

## Resumen

```text
¿Querés que una clase tenga una sola razón para cambiar?
    └─► SRP: una responsabilidad, una clase
        BCL: FileStream / StreamReader / TextReader, List<T> vs Enumerable
        Moderno: DbContext, handlers de Minimal API, middleware de ASP.NET Core

¿Querés agregar comportamiento sin modificar lo existente?
    └─► OCP: diseñá abstracciones y componé con decoradores o extensiones
        BCL: Stream como jerarquía extensible, LINQ sobre IEnumerable<T>
        Moderno: IQueryable<T> + especificaciones EF, componentes Blazor, filtros globales

¿Querés que un subtipo pueda reemplazar al tipo base sin romper clientes?
    └─► LSP: el contrato debe seguir funcionando para todos los subtipos
        BCL: IEnumerable<T> y foreach, Stream y sus concesiones (CanRead/CanSeek)
        Moderno: DbSet<T> como IQueryable<T>, IActionResult, RenderFragment Blazor

¿Querés interfaces chicas y específicas?
    └─► ISP: IEnumerable, IReadOnlyList, IDisposable, IAsyncDisposable
        BCL: jerarquía de colecciones, IDisposable vs IAsyncDisposable
        Moderno: IServiceCollection / IApplicationBuilder / IEndpointRouteBuilder,
                 IEntityTypeConfiguration<T>, IComponent vs ComponentBase

¿Querés depender de contratos y no de clases concretas?
    └─► DIP: StreamReader(Stream), ILogger<T>, DI en ASP.NET Core
        BCL: StreamReader depende de Stream, IServiceCollection con DI
        Moderno: repositorio con EF Core, IHttpClientFactory, inyección en Minimal API
```

La BCL es un buen lugar para aprender diseño porque muchas decisiones están tomadas con cuidado: composición antes que jerarquías enormes, interfaces chicas antes que contratos gigantes y abstracciones pensadas para poder extenderse sin reescribir todo.

Si mirás `Stream`, `IEnumerable<T>`, `IDisposable` o `ILogger<T>` con esa lupa, vas a encontrar SOLID en acción casi en cada esquina.
