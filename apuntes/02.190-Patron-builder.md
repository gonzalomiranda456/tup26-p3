# Patrón Builder en C#

Builder construye objetos complejos **paso a paso**, evitando constructores con demasiados parámetros:

```csharp
// Sin Builder — ilegible
var pizza = new Pizza("fina", "tomate", "mozzarella", true, false, true);

// Con Builder — intención explícita
Pizza pizza = new PizzaBuilder()
    .ConMasa("fina").ConSalsa("tomate").ConJamon().ConMorron()
    .Build();
```

## Implementación básica

```csharp
public class PizzaBuilder {
    private string _masa = "media masa", _salsa = "tomate", _queso = "mozzarella";
    private bool _jamon, _aceitunas, _morron;

    public PizzaBuilder ConMasa(string v)  { _masa = v;      return this; }
    public PizzaBuilder ConSalsa(string v) { _salsa = v;     return this; }
    public PizzaBuilder ConJamon()         { _jamon = true;  return this; }
    public PizzaBuilder ConAceitunas()     { _aceitunas = true; return this; }
    public PizzaBuilder ConMorron()        { _morron = true; return this; }

    public Pizza Build() => new Pizza(_masa, _salsa, _queso, _jamon, _aceitunas, _morron);
}
```

Cada método devuelve `this` → permite encadenar (Fluent Interface).

## Cuándo usarlo

El Builder evita el **constructor telescópico** (muchas sobrecargas para cada combinación de parámetros opcionales) y los **booleanos ambiguos** (`true, false, true` — ¿qué es qué?). Conviene cuando:

- hay muchos parámetros opcionales o booleanos
- el objeto requiere validación antes de crearse
- hay colecciones internas o pasos de construcción
- se quiere un objeto final inmutable
- hay recetas o variantes predefinidas

```csharp
// Con validación en Build()
public Pedido Build() {
    if (string.IsNullOrWhiteSpace(_cliente))
        throw new InvalidOperationException("El cliente es obligatorio.");
    if (_productos.Count == 0)
        throw new InvalidOperationException("Debe agregar al menos un producto.");
    return new Pedido(_cliente, new List<string>(_productos), _metodoPago, _direccion, _esExpress);
}
```

## Ejemplos prácticos

**SQL dinámico:**
```csharp
string sql = new SqlQueryBuilder()
    .Select("Nombre").Select("Email")
    .From("Clientes")
    .Where("Activo = 1").Where("Edad >= 18")
    .OrderBy("Nombre")
    .Build();
// SELECT Nombre, Email FROM Clientes WHERE Activo = 1 AND Edad >= 18 ORDER BY Nombre
```

**Configuración con lógica:**
```csharp
ServidorConfig config = new ServidorConfigBuilder()
    .ConHost("api.midominio.com")
    .UsarHttps()      // también cambia el puerto a 443
    .ConTimeout(60)
    .Build();
```

**Test Data Builder** (muy útil en tests):
```csharp
// Solo cambio lo que importa para cada test
Usuario admin    = new UsuarioTestBuilder().ComoAdmin().Build();
Usuario inactivo = new UsuarioTestBuilder().Inactivo().Build();
Usuario sinEmail = new UsuarioTestBuilder().ConEmail("").Build();
```

## Alternativas más simples

Antes de usar Builder, evaluar si no alcanza con:

```csharp
// Record + with (C#)
var admin = usuarioBase with { Rol = "Admin" };

// Parámetros nombrados
var pizza = new Pizza(masa: "fina", salsa: "tomate", tieneJamon: true, tieneAceitunas: false, tieneMorron: true);

// Object initializer (requiere setters)
var usuario = new Usuario { Nombre = "Ana", Email = "ana@x.com", Rol = "Admin", Activo = true };
```

Builder gana cuando hay validaciones, colecciones, valores por defecto complejos o construcción en múltiples pasos.

## Builder en .NET

`WebApplication.CreateBuilder(args)` + `.Build()` es exactamente este patrón:

```csharp
var builder = WebApplication.CreateBuilder(args);
builder.Services.AddControllers();
builder.Services.AddSwaggerGen();
var app = builder.Build();
```

## Relación con SOLID

SRP: el objeto final representa datos/comportamiento; el Builder se ocupa de construirlo — responsabilidades separadas.  
OCP: se pueden agregar variantes (`PedidoExpressBuilder`) sin tocar el objeto final.
