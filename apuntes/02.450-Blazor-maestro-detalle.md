# Tutorial: Una agenda de contactos con Blazor

Vamos a desarmar pieza por pieza esta aplicación para entender cómo se construye una app Blazor de verdad: desde el modelo de datos hasta los componentes visuales. La app es una agenda de contactos con un layout *maestro-detalle*: a la izquierda la lista de contactos con un buscador, a la derecha el detalle o el formulario de edición.

La idea no es solo "que funcione", sino ver **las técnicas** que Blazor nos da para organizar el código: inyección de dependencias, componentes, parámetros, eventos, binding, validación y parámetros en cascada. Cada sección usa un archivo del proyecto como ejemplo, así que tené el código a mano.

---

## 1. El modelo de datos y el servicio

### El modelo: `Models/Contacto.cs`

Todo arranca con una clase común y corriente (un POCO) que representa un contacto:

```csharp
public class Contacto {
    public int Id { get; set; }

    [Required(ErrorMessage = "El nombre es obligatorio")]
    public string Nombre { get; set; } = "";

    [Required(ErrorMessage = "El teléfono es obligatorio")]
    [Phone(ErrorMessage = "El teléfono no es válido")]
    public string Telefono { get; set; } = "";

    [Required(ErrorMessage = "El email es obligatorio")]
    [EmailAddress(ErrorMessage = "El email no es válido")]
    public string Email { get; set; } = "";
    // ... Empresa, Cargo, Direccion, FechaNacimiento, Notas
}
```

Fijate en las **anotaciones de validación** (`[Required]`, `[Phone]`, `[EmailAddress]`). Vienen de `System.ComponentModel.DataAnnotations` y son *declarativas*: en lugar de escribir código que valide, **describimos las reglas sobre la propiedad** y dejamos que el framework las aplique. Más adelante vamos a ver que el formulario las usa automáticamente — no hay ni un `if` de validación escrito a mano en toda la app.

El modelo también tiene un truco que vamos a necesitar:

```csharp
public Contacto Clonar() => (Contacto)MemberwiseClone();
```

¿Para qué clonar un contacto? Paciencia, lo vemos en la sección 2.

### El servicio: `Services/ContactoService.cs`

Acá está la regla de oro: **los componentes no acceden a los datos directamente**. Toda la lógica de datos vive en una clase de servicio:

```csharp
public class ContactoService {
    private readonly IDbContextFactory<AgendaContext> dbFactory;

    public ContactoService(IDbContextFactory<AgendaContext> dbFactory) =>
        this.dbFactory = dbFactory;

    public async Task<IEnumerable<Contacto>> ListarAsync(string? busqueda = null) { ... }
    public async Task GuardarAsync(Contacto contacto) { ... }
    public async Task EliminarAsync(Contacto contacto) { ... }
}
```

Tres operaciones: listar (con búsqueda opcional), guardar y eliminar. El componente que lo use no tiene idea de que abajo hay Entity Framework y SQLite — y eso es exactamente lo que queremos. Mañana cambiamos SQLite por una API REST y los componentes ni se enteran.

Detalle lindo: `GuardarAsync` resuelve alta y modificación en un solo método mirando el `Id`:

```csharp
if (contacto.Id == 0)
    db.Contactos.Add(contacto);   // Id 0 → es nuevo
else
    db.Contactos.Update(contacto); // tiene Id → ya existe
```

### Inyección de dependencias

¿Y cómo llega el servicio al componente? **Nunca hacemos `new ContactoService()`**. Lo registramos en `Program.cs`:

```csharp
builder.Services
    .AddDbContextFactory<AgendaContext>(
        options => options.UseSqlite("Data Source=contactos.db"));

builder.Services
    .AddScoped<ContactoService>();
```

Y lo pedimos donde lo necesitamos, en `Home.razor`:

```razor
@inject ContactoService Servicio
```

Esto es **inyección de dependencias (DI)**: el contenedor de ASP.NET Core crea las instancias y las "inyecta" a quien las pida. Miralo en cadena: `ContactoService` pide en su constructor un `IDbContextFactory<AgendaContext>`... que también está registrado. El contenedor arma todo el árbol de dependencias solo. Nosotros declaramos *qué* necesitamos, el framework resuelve *cómo* conseguirlo.

¿Por qué `AddScoped`? En Blazor Server, *scoped* significa "una instancia por circuito de usuario": cada visitante tiene su propio servicio mientras dura su conexión.

## 2. La lógica de la aplicación: `Pages/Home.razor`

`Home.razor` es el cerebro de la app. Todo su estado se reduce a cuatro variables:

```csharp
private IEnumerable<Contacto> contactos = [];  // la lista visible
private Contacto? seleccionado;                // el que se muestra en el detalle
private Contacto? editando;                    // el que se está editando (si hay)
private string busqueda = "";                  // el texto del buscador
```

Y acá viene una idea clave de Blazor: **la UI es una función del estado**. No escribimos código que "muestre el formulario" o que "oculte el detalle"; escribimos qué se ve para cada estado y Blazor redibuja solo:

```razor
@if (editando is not null) {
    <FormularioContacto Contacto="@editando" OnGuardar="@Guardar" OnCancelar="@Cancelar" />
} else if (seleccionado is not null) {
    <DetalleContacto Contacto="@seleccionado" OnEditar="@Editar" OnEliminar="@Eliminar" />
} else {
    <p>Seleccioná un contacto para ver su detalle.</p>
}
```

Leelo como una tabla de decisión: ¿hay algo en `editando`? → formulario. ¿No, pero hay `seleccionado`? → detalle. ¿Ninguno? → estado vacío. **Cambiar de "modo" es simplemente asignar variables:**

```csharp
private void Seleccionar(Contacto c) {
    seleccionado = c;
    editando = null;        // si estaba editando, salgo del modo edición
}

private void Agregar() {
    editando = new Contacto();  // contacto nuevo, Id = 0
    seleccionado = null;
}

private void Editar() => editando = seleccionado?.Clonar();

private void Cancelar() => editando = null;
```

### ¿Por qué `Clonar()`?

Acá se resuelve el misterio de la sección 1. Si hiciéramos `editando = seleccionado`, ambas variables apuntarían **al mismo objeto**: cada letra que el usuario tipea en el formulario modificaría el contacto de la lista en vivo. Y si después cancela... el daño ya está hecho.

Editando una **copia**, "Cancelar" es gratis (tiro la copia y listo) y "Guardar" es explícito:

```csharp
private async Task Guardar() {
    await Servicio.GuardarAsync(editando!);   // persistir la copia
    await Cargar();                            // recargar la lista desde la base
    seleccionado = contactos.FirstOrDefault(c => c.Id == editando!.Id);
    editando = null;                           // volver al modo detalle
}
```

Notá que después de guardar **recargamos la lista** y volvemos a buscar el contacto guardado para dejarlo seleccionado. La base de datos es la fuente de verdad; la lista en memoria es solo una vista de ella.

### La carga inicial

```csharp
protected override async Task OnInitializedAsync() {
    tema.Cambio += StateHasChanged;   // esto lo vemos en la sección 7
    await Cargar();
}

private async Task Cargar() => contactos = await Servicio.ListarAsync(busqueda);
```

`OnInitializedAsync` es parte del **ciclo de vida** del componente: Blazor lo llama una vez, cuando el componente arranca. Es el lugar natural para cargar datos.

## 3. RenderFragment: `Controles/Pantalla.razor`

Mirá `Home.razor` de nuevo. ¿Notaste que casi no tiene HTML de layout? Nada de `<div class="row">`, columnas, scroll... Todo eso vive en `Pantalla.razor`, un componente que define la estructura visual y deja **tres huecos** para rellenar:

```razor
<div class="container-fluid d-flex flex-column vh-100 ...">
    <div class="card rounded-4 h-100 ...">

        <div class="card-body ... border-bottom">
            @Cabecera                          @* hueco 1 *@
        </div>

        <div class="row g-0 flex-grow-1 ...">
            <div class="col-4 h-100 border-end">
                @Maestro                       @* hueco 2 *@
            </div>
            <div class="col-8 h-100">
                @Detalle                       @* hueco 3 *@
            </div>
        </div>
    </div>
</div>

@code {
    [Parameter] public RenderFragment? Cabecera { get; set; }
    [Parameter] public RenderFragment? Maestro  { get; set; }
    [Parameter] public RenderFragment? Detalle  { get; set; }
}
```

Un **`RenderFragment` es un parámetro cuyo valor es... contenido Razor**. Así como un parámetro `string` recibe texto, un `RenderFragment` recibe un pedazo de UI. El que usa el componente lo llena con etiquetas que coinciden con el nombre del parámetro:

```razor
<Pantalla>
    <Cabecera>
        <h1>Agenda</h1>
        <BotonTema />
    </Cabecera>
    <Maestro>
        <Buscador ... />
        @* la lista de contactos *@
    </Maestro>
    <Detalle>
        @* formulario o detalle según el modo *@
    </Detalle>
</Pantalla>
```

El resultado es una separación de responsabilidades buenísima: `Pantalla` sabe **dónde** va cada cosa (y pelea ella sola con el CSS), `Home` sabe **qué** va en cada lugar. Si mañana queremos el maestro a la derecha, tocamos un solo archivo.

> 💡 Si el componente tiene un solo hueco, conviene llamarlo `ChildContent`: es el nombre mágico que Blazor llena automáticamente con todo lo que pongas entre las etiquetas, sin tag extra.

## 4. Parámetros y eventos: `Contactos/ItemContacto.razor`

Cada fila de la lista de contactos es un `ItemContacto`. Es el ejemplo perfecto de la anatomía de un componente Blazor: **datos que entran, eventos que salen**.

```razor
<button class="list-group-item ... @(Activo ? "active" : "")"
        @onclick="@OnSeleccionar">
    <span class="badge ...">@Contacto.Iniciales</span>
    <span>
        <span class="d-block fw-semibold">@Contacto.Nombre @Contacto.Apellido</span>
        <small>@Contacto.Empresa</small>
    </span>
</button>

@code {
    [Parameter] public Contacto Contacto           { get; set; } = default!;
    [Parameter] public bool Activo                 { get; set; }
    [Parameter] public EventCallback OnSeleccionar { get; set; }
}
```

Tres parámetros, tres roles:

- **`Contacto`** — los datos a mostrar. Fluyen *del padre al hijo*.
- **`Activo`** — un flag visual: ¿soy el contacto seleccionado? Notá que el componente **no decide** si está activo; se lo dicen. Él solo se pinta.
- **`OnSeleccionar`** — un `EventCallback`: la forma de avisarle al padre que pasó algo. Fluye *del hijo al padre*.

Y así lo usa `Home`:

```razor
@foreach (var c in contactos) {
    <ItemContacto Contacto="@c"
                  Activo="@(c.Id == seleccionado?.Id)"
                  OnSeleccionar="() => Seleccionar(c)" />
}
```

Este es el patrón fundamental de Blazor (y de casi todo framework de componentes): **los datos bajan, los eventos suben**. `ItemContacto` no sabe que existe una lista, ni quién está seleccionado, ni qué pasa al hacer click. Solo dice "me clickearon" y el padre decide. Eso lo hace tonto... y por eso, reutilizable y fácil de razonar.

Bonus de `EventCallback`: cuando el padre maneja el evento, Blazor re-renderiza automáticamente. Por eso al hacer click la fila se marca como activa sin que nadie llame a nada.

## 5. Binding bidireccional: `Controles/Buscador.razor`

En `Home` el buscador se usa así:

```razor
<Buscador @bind-Value="@busqueda" OnBuscar="@Cargar" />
```

Ese `@bind-Value` es **binding bidireccional**: la variable `busqueda` del padre y el texto del input quedan sincronizados en los dos sentidos. ¿Magia? No: es azúcar sintáctico. `@bind-Value` se expande a *dos* parámetros:

```razor
<Buscador Value="@busqueda" ValueChanged="@(v => busqueda = v)" ... />
```

Para que eso funcione, el componente tiene que cumplir una **convención**: exponer un parámetro `X` y su compañero `XChanged`:

```razor
<input class="form-control ps-5" placeholder="Buscar contacto..."
       value="@Value" @oninput="@OnInput" />

@code {
    [Parameter] public string Value { get; set; } = "";
    [Parameter] public EventCallback<string> ValueChanged { get; set; }
    [Parameter] public EventCallback OnBuscar { get; set; }

    private async Task OnInput(ChangeEventArgs e) {
        Value = e.Value?.ToString() ?? "";
        await ValueChanged.InvokeAsync(Value);  // aviso al padre el nuevo valor
        await OnBuscar.InvokeAsync();           // y le aviso que busque
    }
}
```

Seguí el ciclo completo de una tecla:

1. El usuario tipea → se dispara `@oninput` del `<input>`.
2. `OnInput` toma el texto y hace `ValueChanged.InvokeAsync(Value)` → la variable `busqueda` del padre se actualiza.
3. `OnBuscar.InvokeAsync()` → el padre ejecuta `Cargar()`, que llama al servicio con el filtro.
4. La lista se re-renderiza con los resultados.

Todo eso, **en cada tecla** (por eso usamos `@oninput` y no `@onchange`, que esperaría a que el input pierda el foco). Búsqueda en vivo con diez líneas de código.

Fijate además que `Buscador` separa dos cosas que parecen una: *sincronizar el valor* (`ValueChanged`, parte del binding) y *pedir la acción* (`OnBuscar`, un evento de negocio). El binding no obliga al padre a hacer nada; el evento sí.

## 6. Formularios y validación: `Contactos/FormularioContacto.razor`

Llegó el momento en que las anotaciones de la sección 1 pagan el esfuerzo. El formulario completo:

```razor
<EditForm Model="Contacto" OnValidSubmit="@OnGuardar">
    <DataAnnotationsValidator />

    <Boton Texto="Cancelar" Estilo="outline-secondary" OnClick="@OnCancelar" />
    <Boton Icono="save" Texto="Guardar cambios" />   @* sin OnClick → submit *@

    <div class="row g-3">
        <Entrada Etiqueta="Nombre *"   @bind-Valor="Contacto.Nombre" />
        <Entrada Etiqueta="Apellido *" @bind-Valor="Contacto.Apellido" />
        <Entrada Etiqueta="Teléfono *" @bind-Valor="Contacto.Telefono" />
        <Entrada Etiqueta="Email *"    @bind-Valor="Contacto.Email" />
        @* ... *@
        <EntradaFecha Etiqueta="Fecha de nacimiento" @bind-Valor="Contacto.FechaNacimiento" />
    </div>

    <ValidationSummary class="text-danger mt-3" />
</EditForm>
```

Tres piezas trabajan en equipo:

- **`<EditForm Model="Contacto">`** — el contenedor. Crea un *contexto de edición* alrededor del objeto `Contacto` que rastrea qué campos se tocaron y si son válidos.
- **`<DataAnnotationsValidator />`** — el conector. Le dice al `EditForm`: "para validar, leé las anotaciones del modelo". Acá se enchufan los `[Required]`, `[Phone]` y `[EmailAddress]` que escribimos en `Contacto.cs`. Una línea, cero lógica duplicada.
- **`<ValidationSummary />`** — muestra la lista de errores (con los `ErrorMessage` que definimos en el modelo).

Y la jugada maestra: **`OnValidSubmit`**. No es "cuando aprieten el botón" sino "cuando aprieten el botón **y el modelo sea válido**". Si falta el nombre o el email está mal formado, `OnGuardar` directamente no se ejecuta. La validación quedó declarada una sola vez, en el modelo, y el formulario la respeta sin que escribamos un solo `if`.

### Los Inputs

Adentro de un `EditForm` no se usa `<input>` pelado sino los componentes de Blazor: `InputText`, `InputDate`, `InputTextArea`, etc. ¿Por qué? Porque ellos se registran en el contexto de edición: marcan el campo como "modificado", participan de la validación y se pintan con clases CSS según su estado.

En este proyecto los envolvimos en un control propio, `Entrada.razor`, para no repetir el `<label>` y la columna de grilla en cada campo:

```razor
<div class="@Columna">
    <label class="form-label fw-semibold">@Etiqueta</label>
    <InputText class="form-control" Value="@Valor"
               ValueChanged="@ValorChanged" ValueExpression="@ValorExpression" />
</div>

@code {
    [Parameter] public string Etiqueta { get; set; } = "";
    [Parameter] public string? Valor   { get; set; }
    [Parameter] public EventCallback<string?> ValorChanged { get; set; }
    [Parameter] public Expression<Func<string?>>? ValorExpression { get; set; }
}
```

¿Reconocés el patrón `Valor` / `ValorChanged` de la sección 5? Es la misma convención de binding, y por eso en el formulario podemos escribir `@bind-Valor="Contacto.Nombre"`. La novedad es **`ValorExpression`**: el sistema de validación no necesita solo el *valor* del campo, necesita saber *qué propiedad del modelo* es, para encontrar sus anotaciones. `@bind-Valor` completa los tres parámetros automáticamente.

Detalle del botón Guardar: `Boton.razor` decide su `type` solo:

```razor
<button type="@(OnClick.HasDelegate ? "button" : "submit")" ...>
```

Si le pasaron `OnClick`, es un botón común. Si no (como "Guardar cambios"), es `type="submit"` y dispara el submit del `EditForm` que lo contiene. Por eso Cancelar cancela y Guardar guarda, usando el mismo componente.

## 7. CascadingParameter: el tema claro/oscuro

Último desafío: el botón de tema (🌙/☀️) está en la cabecera, pero el atributo `data-bs-theme` que cambia los colores está en `Pantalla`, y el estado lo maneja `Home`. ¿Pasamos el tema por parámetro a cada componente del camino? Se puede, pero es tedioso (el famoso *prop drilling*). Blazor tiene algo mejor: **parámetros en cascada**.

### El estado: `Controles/Tema.cs`

```csharp
public class Tema {
    public bool Oscuro { get; private set; } = true;
    public string Modo => Oscuro ? "dark" : "light";
    public event Action? Cambio;
    public void Alternar() { Oscuro = !Oscuro; Cambio?.Invoke(); }
}
```

Una clase simple: el estado (`Oscuro`), una acción (`Alternar`) y un evento (`Cambio`) para avisar cuando algo cambió.

### El proveedor: `Home.razor`

```razor
<CascadingValue Value="@tema">
    <Pantalla> ... </Pantalla>
</CascadingValue>
```

`CascadingValue` hace que el objeto `tema` esté **disponible para todos los componentes que estén adentro**, a cualquier profundidad. Como un Wi-Fi: no hace falta cablear cada habitación.

### Los consumidores

Cualquier descendiente lo recibe declarando un `[CascadingParameter]` — Blazor lo encuentra por tipo, sin que nadie se lo pase explícitamente:

```razor
@* BotonTema.razor — lee el estado y lo cambia *@
<Boton Icono="@(Tema!.Oscuro ? "sun" : "moon-stars")" OnClick="@Tema.Alternar" />

@code {
    [CascadingParameter] public Tema? Tema { get; set; }
}
```

```razor
@* Pantalla.razor — usa el estado para pintar *@
<div class="..." data-bs-theme="@(Tema?.Modo ?? "dark")">

@code {
    [CascadingParameter] public Tema? Tema { get; set; }
}
```

Miralo bien: `BotonTema` y `Pantalla` **se comunican sin conocerse**. Uno está en la cabecera, el otro es el contenedor de toda la app; no hay ningún parámetro entre ellos. Comparten el mismo objeto `Tema` que cae en cascada desde `Home`.

### La pieza final: el evento `Cambio`

Queda un problema sutil. Cuando clickeás `BotonTema`, se ejecuta `Tema.Alternar()`... pero Blazor re-renderiza al componente donde ocurrió el evento (`BotonTema`), no a `Pantalla`, que es quien tiene que cambiar el color. ¿Cómo se entera `Home` de que tiene que redibujar todo?

Con el evento que pusimos en `Tema`:

```csharp
// Home.razor
protected override async Task OnInitializedAsync() {
    tema.Cambio += StateHasChanged;   // "cuando el tema cambie, redibujame"
    await Cargar();
}

public void Dispose() => tema.Cambio -= StateHasChanged;
```

`Home` se suscribe: cada vez que el tema cambia, llama a `StateHasChanged()` (el "che Blazor, redibujá" manual) y como `Home` contiene a todo lo demás, la app entera se actualiza. Y como buen ciudadano, se **desuscribe en `Dispose`** (por eso el `@implements IDisposable` arriba del archivo) para no dejar referencias colgadas.

Este trío — *objeto de estado + CascadingValue + evento de cambio* — es la versión mínima de un patrón que vas a ver en todas las apps Blazor grandes para manejar estado compartido.

## Recapitulando

| Técnica | Dónde verla | La idea en una frase |
|---|---|---|
| Inyección de dependencias | `Program.cs`, `Home.razor` | Declarás qué necesitás; el framework lo provee. |
| DataAnnotations | `Contacto.cs` | Las reglas de validación viven en el modelo, una sola vez. |
| UI como función del estado | `Home.razor` | No mostrás/ocultás cosas: cambiás variables y Blazor redibuja. |
| RenderFragment | `Pantalla.razor` | Parámetros que reciben UI: layouts con huecos. |
| Parámetros y EventCallback | `ItemContacto.razor` | Los datos bajan, los eventos suben. |
| Binding bidireccional | `Buscador.razor` | Convención `Value` + `ValueChanged` = `@bind-Value`. |
| EditForm y validación | `FormularioContacto.razor` | `OnValidSubmit` solo se dispara si el modelo es válido. |
| CascadingParameter | `Tema.cs`, `BotonTema.razor`, `Pantalla.razor` | Estado compartido sin pasar parámetros mano a mano. |

Ahora sí: abrí los fuentes, rompé algo a propósito (sacá el `Clonar()`, cambiá `@oninput` por `@onchange`, borrá el `DataAnnotationsValidator`) y mirá qué pasa. No hay mejor forma de entender por qué cada pieza está donde está. 🚀
