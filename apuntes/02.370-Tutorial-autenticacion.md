## 1. El problema de fondo

Una API en internet recibe pedidos de cualquiera. Antes de dejar que alguien lea o modifique tickets, hay que responder dos preguntas distintas:

- **¿Quién sos?** → eso es **autenticación**.
- **¿Podés hacer esto?** → eso es **autorización**.

Este tutorial es solo sobre la primera. Y resolverla bien implica encadenar cuatro problemas más chicos:

```
1. Guardar la contraseña sin guardarla    → hash + salt
2. Comprobar la contraseña al entrar      → login
3. No volver a pedirla en cada request    → token de sesión
4. Reconocer al usuario en cada request   → middleware + filtro
```

Cada pieza existe porque la anterior dejó un cabo suelto. Vamos en ese orden.

---

## 2. Por qué no se guarda la contraseña

El instinto es crear una columna `Password` y guardar ahí lo que el usuario escribió. **Es el error clásico de seguridad.** El día que alguien se filtra la base —y pasa todo el tiempo— se lleva las contraseñas de todos en texto plano. Y como la gente reusa contraseñas, ese robo se contagia a sus correos, bancos y todo lo demás.

La idea central es contraintuitiva pero simple:

> **No necesitamos guardar la contraseña. Necesitamos poder reconocerla cuando vuelva.**

Para eso usamos una **función de hash**: agarra cualquier texto y devuelve una huella de tamaño fijo. Tiene dos propiedades que la hacen útil acá:

- Es **determinista**: la misma entrada da siempre la misma huella.
- Es **irreversible**: de la huella no se puede volver al texto original.

Entonces guardamos la huella, no la contraseña. Cuando el usuario vuelve a escribir su clave, le calculamos la huella otra vez y comparamos huella con huella. Si la base se filtra, el atacante se lleva huellas inservibles.

## 3. El hash con salt, paso a paso

Un hash pelado todavía tiene dos agujeros. El salt y el PBKDF2 los tapan. Mirá la clase y después desarmamos cada decisión:

```csharp
using System.Security.Cryptography;

namespace TicketSystem.Security;

public static class PasswordHasher {
    private const int SaltSize = 16;      // 128 bits
    private const int HashSize = 32;      // 256 bits
    private const int Iterations = 100_000;

    public static (string Hash, string Salt) Hashear(string password) {
        byte[] salt = RandomNumberGenerator.GetBytes(SaltSize);
        byte[] hash = Pbkdf2(password, salt);

        return (Convert.ToBase64String(hash), Convert.ToBase64String(salt));
    }

    public static bool Verificar(string password, string hashGuardado, string saltGuardado) {
        byte[] salt = Convert.FromBase64String(saltGuardado);
        byte[] hash = Pbkdf2(password, salt);
        byte[] esperado = Convert.FromBase64String(hashGuardado);

        return CryptographicOperations.FixedTimeEquals(hash, esperado);
    }

    private static byte[] Pbkdf2(string password, byte[] salt) {
        return Rfc2898DeriveBytes.Pbkdf2(
            password,
            salt,
            Iterations,
            HashAlgorithmName.SHA256,
            HashSize);
    }
}
```

### Agujero 1: dos personas con la misma clave dan el mismo hash

Si Ana y Beto usan ambos `clave1234`, un hash simple les da **la misma huella**. Un atacante que ve dos huellas iguales ya sabe que comparten contraseña. Peor: puede armar un diccionario gigante (huella → palabra), las famosas *rainbow tables*, y resolver millones de hashes de un saque.

La solución es el **salt**: un valor aleatorio distinto para cada usuario, que se mezcla con la contraseña antes de hashear.

```
hash = Hash("clave1234" + salt_de_Ana)   → huella A
hash = Hash("clave1234" + salt_de_Beto)  → huella B   (¡distinta!)
```

El salt **no es secreto**: se guarda al lado del hash, en su propia columna. Su trabajo no es esconderse, sino hacer que cada hash sea único. Eso inutiliza las rainbow tables: el atacante tendría que armar una tabla nueva por cada salt. En el código, `RandomNumberGenerator.GetBytes(SaltSize)` genera esos 16 bytes aleatorios.

### Agujero 2: hashear una vez es demasiado rápido

Las funciones de hash comunes están diseñadas para ser **veloces**. Para verificar una contraseña eso suena bien, pero le sirve igual al atacante: con la base filtrada, puede probar miles de millones de combinaciones por segundo contra cada hash.

La solución es **PBKDF2**: aplicar el hash muchas veces seguidas, encadenado. Acá, `Iterations = 100_000` significa cien mil vueltas.

> **Por qué eso ayuda:** para el servidor, hashear una vez al hacer login es imperceptible (milisegundos). Para el atacante que prueba fuerza bruta, multiplicar cada intento por cien mil convierte un ataque de horas en uno de años. La lentitud es el rasgo deseado, no un defecto.

`Rfc2898DeriveBytes.Pbkdf2` viene en .NET, sin paquetes externos. Le pasamos la contraseña, el salt, las iteraciones, el algoritmo base (`SHA256`) y cuántos bytes de salida queremos (32 → 256 bits).

### El detalle final: comparar en tiempo fijo

En `Verificar`, lo natural sería comparar las dos huellas con `==`. Pero un `==` sobre bytes **corta apenas encuentra la primera diferencia**. Eso abre un ataque sutil: midiendo cuánto tarda la comparación en fallar, un atacante puede deducir el hash byte por byte (un *timing attack*).

`CryptographicOperations.FixedTimeEquals` compara **siempre en el mismo tiempo**, recorra o no toda la secuencia. Le saca al atacante esa información de tiempo.

> **Resumen de la pieza:** guardamos `(hash, salt)`. El hash es irreversible, el salt lo hace único, PBKDF2 lo hace lento de atacar, y la comparación en tiempo fijo no filtra pistas. La contraseña original nunca toca el disco.

## 4. El login: verificar sin conocer

Ahora que sabemos guardar la huella, el login es directo. Esta es la parte de `AuthService`:

```csharp
public async Task<string?> LoginAsync(string email, string password) {
    var usuario = await this.repositorio.ObtenerPorEmailAsync(email);
    if (usuario is null) {
        return null;
    }

    bool valido = PasswordHasher.Verificar(
        password, usuario.PasswordHash, usuario.PasswordSalt);

    if (!valido) {
        return null;
    }

    usuario.Token = GenerarToken();
    usuario.TokenExpira = DateTime.Now.Add(DuracionSesion);
    await this.repositorio.GuardarCambiosAsync();

    return usuario.Token;
}
```

El flujo es: buscar al usuario por email, recalcular la huella de la contraseña que mandó y compararla con la guardada. Fijate que el servidor **nunca conoce la contraseña real**: solo confirma que la huella coincide.

> **Por qué `return null` y no un mensaje distinto:** el método devuelve `null` tanto si el email no existe como si la contraseña está mal. Es a propósito. Si dijéramos "ese email no existe", le estaríamos confirmando a un atacante qué cuentas son válidas para después atacarlas. Una sola respuesta vaga para los dos casos no regala esa información.

Si todo da bien, no devolvemos al usuario: devolvemos un **token**. Eso nos lleva a la pieza siguiente.

## 5. El token: no preguntar la clave en cada request

HTTP no tiene memoria. Cada request llega solo, sin saber nada de los anteriores. Si protegemos `/tickets`, el servidor necesita reconfirmar quién sos **en cada pedido**.

Pedir la contraseña en cada request sería absurdo y peligroso (más veces viaja la clave, más chances de que se filtre). La solución es el **token de sesión**: una credencial temporal que reemplaza a la contraseña una vez que ya probaste quién sos.

```csharp
private static string GenerarToken() {
    byte[] bytes = RandomNumberGenerator.GetBytes(32);
    return Convert.ToBase64String(bytes);
}
```

El token es simplemente 32 bytes aleatorios convertidos a texto. Al hacer login se genera, se guarda en el usuario junto con su vencimiento (`TokenExpira`) y se le entrega al cliente. De ahí en más, el cliente lo manda en cada request y el servidor lo reconoce sin volver a tocar la contraseña.

> **`RandomNumberGenerator`, nunca `Random`:** el token tiene que ser **impredecible**. `Random` produce secuencias que un atacante puede anticipar si adivina la semilla; sirve para barajar cartas, no para seguridad. `RandomNumberGenerator` es criptográficamente seguro: aunque veas mil tokens, no podés predecir el próximo.

Validar y cerrar la sesión son el reverso:

```csharp
public async Task<Usuario?> ValidarTokenAsync(string token) {
    var usuario = await this.repositorio.ObtenerPorTokenAsync(token);
    if (usuario is null) {
        return null;
    }

    if (usuario.TokenExpira is null || usuario.TokenExpira < DateTime.Now) {
        return null;
    }

    return usuario;
}

public async Task LogoutAsync(string token) {
    var usuario = await this.repositorio.ObtenerPorTokenAsync(token);
    if (usuario is not null) {
        usuario.Token = null;
        usuario.TokenExpira = null;
        await this.repositorio.GuardarCambiosAsync();
    }
}
```

`ValidarTokenAsync` chequea **dos** cosas: que el token exista y que no haya vencido. Un token sin fecha o vencido vale lo mismo que ninguno.

> **Por qué los tokens vencen:** si un token durara para siempre, robarlo una vez daría acceso eterno. El vencimiento a 8 horas limita el daño de un token filtrado. `LogoutAsync` hace lo mismo de forma manual: borra el token, así deja de servir aunque alguien lo tenga copiado.

## 6. El middleware: identificar al que toca la puerta

Tenemos tokens, pero falta el mecanismo que los lea **en cada request, automáticamente**, antes de que el endpoint corra. Ese es el trabajo del **middleware**.

Un middleware es una función que se mete en el camino de **todos** los requests. En `Program.cs`:

```csharp
// 2° Bearer: si viene un token valido, deja el usuario en HttpContext.Items.
app.Use(async (context, next) => {
    var auth = context.RequestServices.GetRequiredService<AuthService>();
    var header = context.Request.Headers.Authorization.ToString();

    if (header.StartsWith("Bearer ", StringComparison.OrdinalIgnoreCase)) {
        var token = header["Bearer ".Length..].Trim();
        var usuario = await auth.ValidarTokenAsync(token);

        if (usuario is not null) {
            context.Items["Usuario"] = usuario;
        }
    }

    await next(context);
});
```

Paso a paso: lee el header `Authorization`, que por convención llega como `Bearer <token>`. Si está, le recorta el prefijo `"Bearer "` y valida lo que queda. Si el token es válido, **deja al usuario en `HttpContext.Items`** y sigue con `await next(context)`.

> **Qué es `Bearer`:** es el esquema estándar de HTTP para "el que porta este token tiene acceso" (*bearer* = portador). No es magia de .NET, es una convención de la web; por eso el cliente manda `Authorization: Bearer abc123...`.

> **`HttpContext.Items` es un casillero por request:** un diccionario que nace y muere con cada pedido. El middleware deja ahí al usuario y cualquier endpoint posterior lo encuentra, sin volver a validar el token. El helper `ctx.UsuarioActual()` no es más que leer ese casillero.

> **Clave: el middleware NO rechaza a nadie.** Si no hay token o es inválido, simplemente **no** deja usuario en `Items` y deja pasar el request igual. Suena raro, pero es a propósito: identificar y bloquear son trabajos distintos. El middleware solo identifica. Bloquear es del filtro.

> **`context.RequestServices.GetRequiredService<AuthService>()`:** los delegates de `app.Use` no son *scoped*, pero `AuthService` sí lo es (arrastra el `DbContext`). Pedirlo así, dentro del delegate, respeta el ciclo de vida por request en lugar de capturar una instancia equivocada.

## 7. El filtro: cerrar la puerta

El middleware dejó (o no) un usuario en el casillero. Falta quien **decida** si el endpoint corre o se rechaza con `401`. Ese es el trabajo de un **filtro de endpoint**.

### Qué es un filtro

Un filtro de endpoint es una función que se mete **entre el routing y el handler**: corre justo antes de que se ejecute el código del endpoint, y puede dejarlo correr o cortar antes. Su forma es siempre la misma: recibe el `context` del request y un `next`, y decide si llama a `next` (seguir) o devuelve algo sin llamarlo (cortar).

Para nuestra autorización, la lógica es de tres líneas: miramos el casillero `Items["Usuario"]` que dejó el middleware Bearer; si está vacío, cortamos con `401`; si hay usuario, dejamos pasar.

> **Filtro vs middleware, la diferencia clave:** el middleware corre en **todos** los requests del sistema. El filtro corre solo en los endpoints donde uno lo engancha. Por eso el middleware Bearer (que identifica a todos) es middleware, y la autorización (que protege solo `/tickets`) es un filtro: queremos elegir a qué rutas aplicarlo.

### Un filtro es, en el fondo, una función

La forma más directa de escribir un filtro es una **lambda** pasada a `AddEndpointFilter`. Sin clases ni ceremonia: la función `(context, next) => ...` *es* el filtro.

```csharp
app.MapGroup("/tickets")
    .AddEndpointFilter(async (context, next) => {
        var usuario = context.HttpContext.Items["Usuario"] as Usuario;
        if (usuario is null) {
            return Results.Unauthorized();
        }

        return await next(context);
    });
```

Eso ya funciona y protege el grupo entero. Pero tiene un problema: el grupo `/tickets` **no es el único** que necesita protección. El endpoint `POST /auth/logout` también la pide (hay que estar logueado para cerrar sesión). Si dejáramos la lambda inline, tendríamos que **copiar y pegar** esas mismas tres líneas en cada lugar protegido. Y código duplicado es código que algún día vas a corregir en un lado y olvidar en el otro.

### Un nombre para la función: el método de extensión

La solución es darle a esa función un **nombre reutilizable**. Un método de extensión envuelve el `AddEndpointFilter` con la lambda adentro, y lo expone como un método que se lee natural sobre cualquier ruta o grupo.

Creá `Api/AuthExtensions.cs`:

```csharp
using TicketSystem.Domain;

namespace TicketSystem.Api;

public static class AuthExtensions {
    public static TBuilder RequiereAuth<TBuilder>(this TBuilder builder)
        where TBuilder : IEndpointConventionBuilder {
        return builder.AddEndpointFilter(async (context, next) => {
            var usuario = context.HttpContext.Items["Usuario"] as Usuario;
            if (usuario is null) {
                return Results.Unauthorized();
            }

            return await next(context);
        });
    }
}
```

Ahora proteger una ruta es escribir `.RequiereAuth()`, y la lógica vive en **un solo lugar**:

```csharp
var ticketsGroup = app.MapGroup("/tickets")
    .RequiereAuth();
```

> **Por qué `IEndpointConventionBuilder` genérico:** `MapGroup` devuelve un `RouteGroupBuilder`, pero `MapPost`/`MapGet` devuelven un `RouteHandlerBuilder`. Son tipos distintos. Si tipáramos el método para uno solo, no andaría en el otro. Como **ambos** implementan `IEndpointConventionBuilder`, el genérico con ese `where` cubre los dos. Y devolver `TBuilder` (en vez de la interfaz) conserva el tipo original, así podés seguir encadenando métodos después de `.RequiereAuth()`.

> **Por qué lambda y no una clase:** un filtro también puede escribirse como una clase que implementa `IEndpointFilter`. Eso conviene cuando el filtro necesita **dependencias inyectadas por constructor** (un servicio, un logger). El nuestro no necesita ninguna: solo lee `Items["Usuario"]`, que el middleware ya dejó servido. Sin dependencias que inyectar, la lambda dentro de la extensión es más liviana y igual de reutilizable. Como dato: esto que escribimos es, en miniatura, lo mismo que hace el método `RequireAuthorization()` que trae ASP.NET Core —también es un método de extensión sobre `IEndpointConventionBuilder`—, solo que el del framework arrastra todo un sistema de esquemas y *claims* que para aprender el concepto es de más.

> **Por qué separar middleware y filtro:** dividir "identificar" de "autorizar" deja al grupo `/auth` (login, registro) **público** —no lleva `.RequiereAuth()`, cualquiera puede loguearse— y a `/tickets` **protegido**. Si el middleware bloqueara por su cuenta, no habría forma de tener rutas públicas: nadie podría siquiera pedir un token. Separar las dos responsabilidades es lo que hace que el login pueda ser abierto y los tickets cerrados, con el mismo mecanismo.

## 8. El recorrido completo de un request

Juntemos las cuatro piezas siguiendo un `GET /tickets` con token válido:

```
Cliente manda:  GET /tickets
                Authorization: Bearer abc123...
        │
        ▼
┌─────────────────────────────────────────────┐
│ MIDDLEWARE Bearer                             │
│  • lee el header Authorization                │
│  • ValidarTokenAsync("abc123...")             │
│    → existe y no venció → usuario OK          │
│  • HttpContext.Items["Usuario"] = usuario     │
│  • next()  (siempre deja pasar)               │
└─────────────────────────────────────────────┘
        │
        ▼
┌─────────────────────────────────────────────┐
│ FILTRO .RequiereAuth()                        │
│  • ¿hay usuario en Items?  SÍ                 │
│  • next()  → el handler corre                 │
└─────────────────────────────────────────────┘
        │
        ▼
┌─────────────────────────────────────────────┐
│ HANDLER del endpoint                          │
│  • ctx.UsuarioActual() lee el casillero       │
│  • devuelve los tickets                        │
└─────────────────────────────────────────────┘
        │
        ▼
Cliente recibe: 200 OK + lista de tickets
```

Ahora el mismo `GET /tickets` **sin** token, o con uno vencido:

```
MIDDLEWARE: no hay token válido → NO deja usuario en Items → next()
        │
        ▼
FILTRO: ¿hay usuario en Items?  NO → Results.Unauthorized()
        │
        ▼
Cliente recibe: 401 Unauthorized   (el handler nunca corrió)
```

Y así cierra el circuito que arrancó en la sección 1:

| Problema                              | Pieza que lo resuelve            |
| ------------------------------------- | -------------------------------- |
| Guardar la clave sin guardarla        | `PasswordHasher` (hash + salt)   |
| Comprobar la clave al entrar          | `AuthService.LoginAsync`         |
| No pedir la clave en cada request     | Token de sesión + vencimiento    |
| Identificar al usuario en cada request| Middleware Bearer + `Items`      |
| Dejar entrar solo a los identificados | Filtro `.RequiereAuth()`         |

Cada pieza tapa el cabo que dejó la anterior. Esa es toda la autenticación del sistema, de la contraseña al `401`.
