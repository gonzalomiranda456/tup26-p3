# Fundamentos del ecosistema .NET

## Por qué existe .NET

Antes de definir qué es `.NET`, conviene entender qué problema resuelve.

Cuando escribimos un programa en C#, hacen falta varias cosas para que ese programa efectivamente funcione:

- alguien que **traduzca** el código que escribimos a algo que la computadora pueda ejecutar,
- un **motor de ejecución** que corra ese código de forma segura y eficiente,
- una **biblioteca estándar** con clases ya hechas para tareas comunes: leer archivos, manejar fechas, trabajar con strings, hacer HTTP, etc.,
- **herramientas** para compilar, ejecutar, empaquetar, publicar y depurar,
- una forma ordenada de **incorporar dependencias** de terceros.

Si cada uno tuviera que armar todo eso por su cuenta, empezar a programar costaría enormemente. `.NET` es la plataforma que provee todas esas piezas juntas, integradas y funcionando en Windows, Linux y macOS.

## Qué es .NET

`.NET` es una **plataforma de desarrollo**: un conjunto coordinado de partes que permiten escribir, compilar y ejecutar software.

Las piezas principales son:

- un **runtime** que ejecuta el código,
- un **SDK** con todas las herramientas de desarrollo,
- la **Base Class Library** (BCL): miles de clases listas para usar,
- un sistema de **paquetes** (`NuGet`) para sumar dependencias externas,
- un conjunto de **frameworks** especializados para distintos tipos de aplicación.

Con `.NET` se pueden construir aplicaciones de consola, servicios, APIs REST, sitios web, aplicaciones de escritorio, aplicaciones móviles y bibliotecas reutilizables.

### Una analogía

Pensar `.NET` como una plataforma ayuda a no confundirlo con el lenguaje:

- **C#** es el idioma con el que escribimos,
- **.NET** es toda la infraestructura que permite que ese idioma se convierta en programas que efectivamente corren,
- los **frameworks** son ambientes especializados que viven sobre `.NET` para resolver tipos concretos de problema (web, datos, móvil, etc.).

## C# y .NET: cómo se relacionan

C# es **un** lenguaje que corre sobre `.NET`. El runtime también ejecuta otros lenguajes:

- **F#**, lenguaje funcional,
- **VB.NET**, con presencia histórica fuerte en proyectos antiguos.

Todos comparten el mismo runtime, la misma BCL y el mismo ecosistema de paquetes. En esta materia trabajamos con C#, pero conviene saber que la plataforma no depende de un único lenguaje.

## Las piezas en detalle

### Runtime

El runtime es el **motor de ejecución**. Cuando compilamos un programa C# no se genera código de máquina directamente, sino un formato intermedio llamado `IL` (Intermediate Language). El runtime se encarga de:

- cargar ese código intermedio,
- traducirlo a instrucciones de la máquina en tiempo de ejecución (`JIT`, *Just In Time*),
- administrar la memoria (recolección de basura),
- gestionar tipos, hilos y excepciones.

Una computadora con solo el runtime instalado puede **ejecutar** aplicaciones `.NET`, pero no puede compilarlas.

### SDK

El SDK (`Software Development Kit`) contiene todo lo necesario para **desarrollar**:

- el compilador de C# (`Roslyn`),
- la herramienta de línea de comandos `dotnet`,
- la BCL,
- el runtime (incluido dentro del SDK),
- plantillas de proyectos,
- herramientas para restaurar paquetes, correr tests, publicar binarios, etc.

Para programar alcanza con instalar el SDK. El runtime queda incluido.

Dicho de otra forma:

- **runtime** = ejecutar,
- **SDK** = desarrollar (que incluye ejecutar).

### Base Class Library (BCL)

La BCL es el conjunto enorme de clases listas para usar que vienen con la plataforma. Algunos espacios de nombres muy frecuentes:

- `System.IO` — archivos y streams,
- `System.Text.Json` — serialización JSON,
- `System.Linq` — consultas sobre colecciones,
- `System.Net.Http` — clientes HTTP,
- `System.Collections.Generic` — colecciones genéricas,
- `System.Threading.Tasks` — programación asíncrona.

Antes de buscar una biblioteca externa para hacer algo, vale la pena revisar si la BCL ya lo resuelve.

### NuGet

`NuGet` es el gestor oficial de paquetes. Permite incorporar bibliotecas de terceros (o publicar las propias) de forma versionada y reproducible. Cuando un proyecto declara que depende de un paquete, las herramientas descargan, instalan y mantienen actualizada esa dependencia automáticamente.

Sin un gestor de paquetes, cada equipo tendría que copiar manualmente archivos `.dll` y administrar versiones a mano: algo muy frágil y propenso a errores.

## Frameworks vs bibliotecas

Estos dos términos a veces se mezclan, pero conviene distinguirlos.

Una **biblioteca** ofrece clases y funciones que llamamos cuando las necesitamos. *Nuestro código manda*: usa la biblioteca como una herramienta.

Un **framework** define una estructura general en la que nuestro código encaja. *El framework manda*: llama a nuestro código en los puntos que él define.

### Frameworks importantes del ecosistema

- **ASP.NET Core** — sitios web, APIs REST y servicios backend.
- **Entity Framework Core** — ORM para acceso a datos: mapea clases a tablas relacionales.
- **.NET MAUI** — aplicaciones multiplataforma con interfaz gráfica (Windows, macOS, Android, iOS).

Estos frameworks no son parte obligatoria de `.NET`: se suman como paquetes cuando un proyecto los necesita.

## Setup mínimo del ambiente de desarrollo

Para empezar a programar en C# alcanza con dos cosas:

1. el **SDK de .NET** instalado,
2. un editor: **Visual Studio Code**.

No hace falta Visual Studio (el IDE pesado). `VS Code` más el SDK es suficiente para toda la materia.

### 1. Instalar el SDK de .NET

Bajar la versión `.NET 10` desde:

```
https://dotnet.microsoft.com/download
```

Hay instaladores para Windows, Linux y macOS. En Linux suele estar disponible también vía gestor de paquetes (`apt`, `dnf`, etc.).

Para verificar que quedó instalado, abrir una terminal y correr:

``` bash
dotnet --version
```

Tiene que mostrar un número del estilo `10.0.x`.

Para ver más detalles:

``` bash
dotnet --info
```

### 2. Instalar Visual Studio Code

Bajarlo desde:

```
https://code.visualstudio.com/
```

Una vez abierto, instalar la extensión **C# Dev Kit** desde el panel de extensiones (`Ctrl+Shift+X`). Esa extensión agrega:

- soporte de lenguaje (autocompletado, navegación, refactors),
- depurador integrado,
- explorador de soluciones,
- integración con el SDK.

Con esto el ambiente queda listo para escribir, compilar y depurar código C#.

### 3. Probar la CLI

La CLI (`dotnet`) es la puerta de entrada principal a todo. Algunos comandos básicos que vamos a usar mucho:

``` bash
dotnet --version              # ver versión instalada
dotnet new console            # crear un proyecto consola tradicional
dotnet run                    # compilar y ejecutar
dotnet build                  # solo compilar
dotnet add package <nombre>   # agregar un paquete NuGet
```

## Primer programa: archivos C# sueltos (`file-based apps`)

Históricamente, hasta el menor de los programas en C# necesitaba un **proyecto**: una carpeta con archivo `.csproj`, carpetas `bin/` y `obj/`, etc. Eso tiene sentido para proyectos serios, pero es ruido innecesario para algo tan simple como "probar una idea" o "escribir un script de diez líneas".

A partir de `.NET 10` y `C# 14`, se puede ejecutar un único archivo `.cs` **sin proyecto**. La feature se llama `file-based apps`.

### Hola mundo en un solo archivo

Crear un archivo `hola.cs` con este contenido:

``` csharp
Console.WriteLine("¡Hola desde C# 14 y .NET 10!");
```

Ejecutarlo:

``` bash
dotnet run hola.cs
```

No hay `.csproj`. No hay `bin/`. No hay `obj/`. Solo un archivo y un comando.

### Comparación con la forma tradicional

La forma tradicional sigue existiendo y es la que vamos a usar para proyectos más grandes:

``` bash
mkdir saludo
cd saludo
dotnet new console
dotnet run
```

Eso genera una carpeta completa con `Program.cs`, `saludo.csproj`, `bin/` y `obj/`. Es lo correcto cuando el programa crece, tiene varios archivos, dependencias o tests.

Para arrancar, sin embargo, los archivos sueltos son mucho más cómodos.

### Directivas dentro del archivo

Cuando el script necesita paquetes o configuraciones extra, se declaran al inicio del archivo con directivas `#:`:

``` csharp
#:package Humanizer@2.14.1

using Humanizer;

Console.WriteLine("dotnet".Humanize(LetterCasing.Title));
```

Esto reemplaza al `dotnet add package` y al `.csproj` para casos simples.

## Ejemplo completo: lista de estudiantes con notas

Veamos un programa más sustancial, también en un único archivo. Calcula el promedio individual de cada estudiante y el promedio general del curso.

Crear `estudiantes.cs`:

``` csharp
using System;
using System.Collections.Generic;
using System.Linq;

List<Estudiante> estudiantes = new() {
    new Estudiante("Ana",   [8, 9, 7]),
    new Estudiante("Bruno", [6, 5, 7]),
    new Estudiante("Carla", [10, 9, 10])
};

Console.WriteLine("Promedio por estudiante:");
foreach (Estudiante e in estudiantes) {
    Console.WriteLine($"  {e.Nombre,-8} {e.Promedio():F2}");
}

double promedioCurso = estudiantes.Average(e => e.Promedio());
Console.WriteLine();
Console.WriteLine($"Promedio del curso: {promedioCurso:F2}");

public class Estudiante {
    public string Nombre { get; }
    public List<int> Notas { get; }

    public Estudiante(string nombre, List<int> notas) {
        this.Nombre = nombre;
        this.Notas = notas;
    }

    public double Promedio() {
        return this.Notas.Average();
    }
}
```

Ejecutarlo:

``` bash
dotnet run estudiantes.cs
```

Salida esperada:

```
Promedio por estudiante:
  Ana      8.00
  Bruno    6.00
  Carla    9.67

Promedio del curso: 7.89
```

### Qué hay en ese programa

Este ejemplo, en pocas líneas, muestra varias cosas que vamos a estudiar con detalle a lo largo del curso:

- **Top-level statements**: el código que arranca con `List<Estudiante> ...` no está dentro de un `Main` ni de una clase. Es el punto de entrada del programa.
- **Definición de clase** con propiedades de solo lectura y constructor (`Estudiante`).
- **Convención `this.campo`** para distinguir parámetros del constructor de los miembros del objeto.
- **Colecciones genéricas**: `List<Estudiante>`, `List<int>`.
- **Expresión de colección `[...]`**: `[8, 9, 7]` para crear una lista.
- **LINQ**: el método `Average` sobre la colección.
- **Interpolación de strings con formato**: `$"{...:F2}"` para mostrar dos decimales.
- **Todo en un único archivo**, sin proyecto.

A medida que el programa crezca y necesitemos varios archivos, tests, dependencias o publicación, vamos a migrar a la forma tradicional con proyecto `.csproj`. Pero el día a día del aula, sobre todo al principio, se resuelve perfectamente con archivos sueltos.

## Resumen

| Concepto                | Idea clave                                                                  |
|-------------------------|--------|
| `.NET`                  | Plataforma de desarrollo: runtime + SDK + BCL + paquetes + frameworks       |
| Runtime                 | Ejecuta el código intermedio (`IL`) y administra la memoria                 |
| SDK                     | Compilador, CLI, plantillas y herramientas para desarrollar                 |
| BCL                     | Biblioteca estándar de la plataforma: `System.*`                            |
| `NuGet`                 | Gestor oficial de paquetes                                                  |
| Biblioteca              | Conjunto de clases que llamamos cuando las necesitamos                      |
| Framework               | Estructura que llama a nuestro código en los puntos que él define           |
| `ASP.NET Core`          | Framework para web, APIs y backend                                          |
| `Entity Framework Core` | ORM para acceso a datos                                                     |
| `.NET MAUI`             | Aplicaciones multiplataforma con interfaz gráfica                           |
| CLI `dotnet`            | Puerta de entrada a todo: crear, compilar, ejecutar, publicar               |
| `file-based apps`       | Archivos `.cs` sueltos ejecutables sin proyecto, desde `.NET 10` / `C# 14`  |
