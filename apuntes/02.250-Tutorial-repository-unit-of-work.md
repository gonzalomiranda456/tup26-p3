# Persistencia con Repository y Unit of Work

> Tutorial para Programación III — UTN Tucumán
> C# 14 / .NET 10

---

## 1. El problema fundamental

Cuando ejecutamos un programa, todas las variables viven en memoria. Cuando el programa termina, esa memoria se libera y los datos desaparecen.

Sin embargo, muchas aplicaciones necesitan que la información **sobreviva** entre ejecuciones. Una agenda que olvida los contactos cada vez que la cerramos no es una agenda; es un cuaderno mágico que se borra solo.

La pregunta que da origen a todo este tema es:

> **¿Cómo hacemos para que los datos persistan más allá de la vida del proceso?**

La respuesta evidente es: guardándolos en algún lado. Un archivo de texto, un archivo binario, un archivo JSON, una base de datos SQLite, una base de datos remota. Todos esos son **medios de persistencia**.

Pero en cuanto empezamos a programar la solución, aparecen dos preguntas que parecen una sola y son distintas:

1. **¿Dónde** guardamos los datos? (archivo, base, formato)
2. **¿Cuándo** guardamos los datos? (en cada cambio, al cerrar, manualmente)

Casi todos los problemas de diseño en este tema vienen de no haber separado estas dos preguntas. Este tutorial arma el diseño respondiéndolas una por una.

---

## 2. Una primera solución, y por qué no alcanza

Pensemos la versión más directa posible. Tenemos contactos, queremos guardarlos en JSON, los manipulamos. La primera idea natural es:

```csharp
public class Agenda {
    private readonly string path;
    private readonly List<Contacto> contactos;

    public Agenda(string path) {
        this.path = path;
        this.contactos = this.LoadFromFile();
    }

    public void Add(Contacto c) {
        this.contactos.Add(c);
        this.SaveToFile();   // ← persiste en cada operación
    }

    public void Remove(int id) {
        this.contactos.RemoveAll(c => c.Id == id);
        this.SaveToFile();   // ← persiste en cada operación
    }

    private void SaveToFile() {
        var json = JsonSerializer.Serialize(this.contactos);
        File.WriteAllText(this.path, json);
    }

    private List<Contacto> LoadFromFile() { /* ... */ }
}
```

Esto **funciona**. Cumple el requerimiento. Pero al mirarlo con atención aparecen tres problemas distintos.

### Problema 1 — Acoplamiento

`Agenda` sabe que los datos viven en un archivo JSON. Si mañana queremos cambiar a SQLite, hay que reescribir esta clase. La lógica de negocio (manejar contactos) está mezclada con la lógica de persistencia (leer y escribir archivos).

### Problema 2 — Eficiencia

Cada `Add` reescribe el archivo entero. Si insertamos 100 contactos, escribimos 100 veces, y cada escritura es más larga que la anterior. Para JSON puede no doler con pocos datos, pero la forma del código no escala.

### Problema 3 — Atomicidad

Imaginemos una operación que requiere borrar un contacto y agregar otro como reemplazo. Si el programa falla entre el `Remove` y el `Add`, el archivo queda en un estado intermedio inválido. No tenemos forma de decir "estas dos cosas pasan juntas o no pasan".

Estos tres problemas no se resuelven con la misma herramienta. Cada uno necesita su propia respuesta.

## 3. Dos preguntas que estaban mezcladas

Si volvemos a las dos preguntas iniciales:

- **¿Dónde** se guardan los datos → resuelve el problema de acoplamiento
- **¿Cuándo** se guardan los datos → resuelve los problemas de eficiencia y atomicidad

Cada pregunta tiene un patrón asociado:

| Pregunta | Patrón |
|----------|--------|
| ¿Dónde? | **Repository** |
| ¿Cuándo? | **Unit of Work** |

El resto del tutorial es construir estas dos abstracciones, ver cómo encajan, y verificar que el diseño resultante es genuinamente útil.

---

## 4. Repository: separar el "dónde"

El primer principio es simple: **la lógica de negocio no debería saber cómo se guardan los datos**.

Si nuestra app solo necesita "agregar un contacto", "eliminar un contacto", "listar contactos", entonces no le interesa si por debajo eso es un archivo, una base, o una API remota. Lo único que le interesa es que alguien sepa hacer esas operaciones.

Ese "alguien" es el **repositorio**. Un repositorio es un objeto que ofrece operaciones de colección sobre entidades, ocultando dónde y cómo se almacenan.

Lo expresamos como una interfaz, usando los nombres clásicos de CRUD (Create, Read, Update, Delete):

```csharp
public interface IRepository<T, TId> {
    IReadOnlyList<T> ReadAll();
    T? ReadOne(TId id);
    void Create(T item);
    void Update(T item);
    void Remove(TId id);
}
```

Notar que las operaciones son **granulares**: agregar uno, eliminar uno, actualizar uno. No hay un "guardar todo" porque queremos que cada implementación elija cómo materializa el cambio.

Este es un punto importante que vuelve más adelante: el contrato de la interfaz da forma a las implementaciones que pueden vivir debajo. Si la interfaz dice "guardá todo de nuevo", todas las implementaciones tienen que ofrecer esa semántica, aunque sea ineficiente para algunas. Si la interfaz dice "agregá uno", cada implementación puede traducir eso a la operación más natural de su medio.

---

## 5. Unit of Work: separar el "cuándo"

El repositorio nos dice **qué** operaciones existen. Pero no dice **cuándo** esas operaciones impactan al medio físico.

Acá viene el segundo principio: **acumular los cambios y persistirlos cuando lo decidamos explícitamente**.

La idea conceptual: en lugar de que cada `Create` toque el disco, los cambios se anotan en algún lugar y se efectivizan recién cuando llamamos a `SaveChanges()`.

Esto es lo que en patrones de diseño se llama **Unit of Work**: un objeto que representa un conjunto de cambios que se aplican (o se descartan) juntos.

```csharp
public interface IUnitOfWork {
    IRepository<Contacto, int> Contactos { get; }
    void SaveChanges();
}
```

El Unit of Work es el **dueño** de los repositorios. Esto no es accidental: los cambios que un repositorio acumula tienen que coordinarse con los demás cuando llega el momento de persistir. Si tuviéramos `Contactos`, `Categorias` y `Etiquetas`, todos compartirían el mismo Unit of Work y un solo `SaveChanges()` los persistiría a todos en una sola transacción.

Notar lo que ganamos al separar el "cuándo":

- **Eficiencia.** N operaciones, una sola escritura.
- **Atomicidad.** Las N operaciones ocurren juntas o no ocurren.
- **Reversibilidad.** Si no llamamos a `SaveChanges()`, los cambios se descartan al cerrar el programa.

## 6. Representando los cambios pendientes: change tracking

Hasta acá tenemos las dos abstracciones, pero falta una decisión concreta: cuando alguien llama a `Create(contacto)`, ¿qué pasa exactamente? La operación no se ejecuta, pero algo tiene que quedar registrado para ejecutarla después. ¿Qué guardamos?

La técnica que adoptamos es **change tracking**: cada entidad se rastrea junto con un marcador de estado que dice qué le pasó desde la última vez que se sincronizó con el medio de persistencia.

Los estados posibles son cuatro:

```csharp
public enum EntityState {
    Unchanged,
    Added,
    Modified,
    Deleted
}
```

Y el contenedor que une una entidad con su estado:

```csharp
public class TrackedEntry<T> {
    public T Entity { get; set; }
    public EntityState State { get; set; }

    public TrackedEntry(T entity, EntityState state) {
        this.Entity = entity;
        this.State = state;
    }
}
```

Es una clase mutable a propósito: el `State` cambia a lo largo de la vida del Unit of Work, y `Entity` se reemplaza cuando alguien llama a `Update` con una nueva versión.

Con esto, el rastreador de cambios es simplemente un `Dictionary<TId, TrackedEntry<T>>`. La clave es el id de la entidad, el valor es la entidad junto con su estado actual.

La idea central: cuando llega el momento de persistir, recorremos el rastreador y emitimos la operación correspondiente según el estado de cada entrada. Las que están `Unchanged` se ignoran. Las `Added` se insertan. Las `Modified` se actualizan. Las `Deleted` se borran. Después de persistir, el rastreador se "limpia": las eliminadas salen, las demás vuelven a `Unchanged`. Esta operación de aceptar los cambios como persistidos se conoce convencionalmente como `AcceptChanges`.

Esta representación tiene una propiedad muy útil: **consolida cambios automáticamente**. Si alguien crea una entidad y después la elimina antes de guardar, el rastreador la quita y no emite nada. Si alguien modifica una entidad dos veces, solo importa el último valor. El estado se sobreescribe; lo que se persiste es el resultado neto.

### Por qué `Unchanged` no es redundante

Una pregunta natural al ver la lista de estados: si una entidad no tiene cambios pendientes, ¿para qué la guardamos en el rastreador?

La respuesta es que el rastreador cumple **dos roles a la vez**:

1. Es la **fuente de verdad en memoria** de qué entidades existen.
2. Es el **registro de cambios pendientes** que hay que persistir.

Una entidad `Unchanged` cumple el primer rol (existe, hay que poder leerla, sabemos que está sincronizada con el medio) aunque no aporte nada al segundo. Sin ese estado, no podríamos distinguir entre "entidad nueva que nunca fue persistida" y "entidad que ya existe en disco y todavía no la tocamos".

Las cuatro situaciones que representa la entidad respecto del medio físico son:

| Estado | Significado |
|---|---|
| `Unchanged` | Existe en el medio, y la versión en memoria coincide con la persistida |
| `Added` | Existe en memoria, todavía no en el medio |
| `Modified` | Existe en ambos, pero la versión en memoria es más nueva |
| `Deleted` | Existe en el medio, pero hay que borrarla en el próximo guardado |

`Unchanged` es la línea de base contra la cual se miden las transiciones. Cualquier modificación que aparezca después se interpreta partiendo de ese punto.

## 7. El diseño completo

Con todas las piezas a la vista:

```
┌─────────────────┐
│   App / UI      │   ← lógica de presentación
└────────┬────────┘
         │ depende de
         ▼
┌─────────────────┐
│  IUnitOfWork    │   ← cuándo se persiste
│   (interfaz)    │
└────────┬────────┘
         │ contiene
         ▼
┌─────────────────┐
│  IRepository    │   ← qué operaciones existen
│   (interfaz)    │
└─────────────────┘
         ▲
         │ implementan (compartiendo el rastreador)
    ┌────┴────┐
    │         │
┌───┴───┐ ┌───┴────┐
│ JSON  │ │ SQLite │   ← dónde viven los datos
└───────┘ └────────┘
```

Las flechas siempre van de lo concreto hacia lo abstracto. Nadie depende de las clases concretas excepto el punto donde se arma el programa (el *composition root*). Esto es el principio de **inversión de dependencias** de SOLID.

## 8. Implementación con JSON

Para JSON la estrategia es: cargar todos los contactos en el rastreador al arrancar (todos en estado `Unchanged`), modificar el rastreador conforme se llaman las operaciones, y al guardar serializar las entidades vivas (las que no están marcadas como `Deleted`).

```csharp
public class JsonUnitOfWork : IUnitOfWork {
    private readonly string path;
    private readonly Dictionary<int, TrackedEntry<Contacto>> tracked;

    public JsonUnitOfWork(string path) {
        this.path = path;
        this.tracked = this.LoadFromFile();
        this.Contactos = new TrackedContactoRepository(this.tracked);
    }

    public IRepository<Contacto, int> Contactos { get; }

    public void SaveChanges() {
        var hasPending = this.tracked.Values
            .Any(e => e.State != EntityState.Unchanged);
        if (!hasPending) return;

        var aliveEntities = this.tracked.Values
            .Where(e => e.State != EntityState.Deleted)
            .Select(e => e.Entity)
            .ToList();

        var options = new JsonSerializerOptions { WriteIndented = true };
        var json = JsonSerializer.Serialize(aliveEntities, options);
        File.WriteAllText(this.path, json);

        this.AcceptChanges();
    }

    private void AcceptChanges() {
        var ids = this.tracked.Keys.ToList();
        foreach (var id in ids) {
            if (this.tracked[id].State == EntityState.Deleted) {
                this.tracked.Remove(id);
            } else {
                this.tracked[id].State = EntityState.Unchanged;
            }
        }
    }

    private Dictionary<int, TrackedEntry<Contacto>> LoadFromFile() {
        var result = new Dictionary<int, TrackedEntry<Contacto>>();
        if (!File.Exists(this.path)) return result;

        var json = File.ReadAllText(this.path);
        var list = JsonSerializer.Deserialize<List<Contacto>>(json) ?? new();
        foreach (var c in list) {
            result[c.Id] = new TrackedEntry<Contacto>(c, EntityState.Unchanged);
        }
        return result;
    }
}
```

Notar el método `AcceptChanges`: una vez que el archivo se escribió, los cambios pendientes ya no son pendientes. Las entidades modificadas y nuevas pasan a `Unchanged`; las eliminadas se quitan del rastreador. Esto deja al UoW listo para empezar a acumular el siguiente lote de cambios.

## 9. El repositorio que rastrea cambios

El repositorio es el que traduce las operaciones de la interfaz (`Create`, `Update`, `Remove`) a transiciones de estado en el rastreador.

```csharp
public class TrackedContactoRepository : IRepository<Contacto, int> {
    private readonly Dictionary<int, TrackedEntry<Contacto>> tracked;

    public TrackedContactoRepository(Dictionary<int, TrackedEntry<Contacto>> tracked) {
        this.tracked = tracked;
    }

    public IReadOnlyList<Contacto> ReadAll() {
        return this.tracked.Values
            .Where(e => e.State != EntityState.Deleted)
            .Select(e => e.Entity)
            .ToList();
    }

    public Contacto? ReadOne(int id) {
        if (this.tracked.TryGetValue(id, out var entry)
            && entry.State != EntityState.Deleted) {
            return entry.Entity;
        }
        return null;
    }

    public void Create(Contacto c) {
        this.tracked[c.Id] = new TrackedEntry<Contacto>(c, EntityState.Added);
    }

    public void Update(Contacto c) {
        if (this.tracked.TryGetValue(c.Id, out var entry)) {
            entry.Entity = c;
            // Si era Added, sigue siendo Added (con los nuevos valores).
            // Si era Unchanged o Modified, ahora es Modified.
            if (entry.State != EntityState.Added) {
                entry.State = EntityState.Modified;
            }
        } else {
            this.tracked[c.Id] = new TrackedEntry<Contacto>(c, EntityState.Modified);
        }
    }

    public void Remove(int id) {
        if (this.tracked.TryGetValue(id, out var entry)) {
            // Si era Added y se elimina antes de guardar, simplemente sale del rastreador.
            if (entry.State == EntityState.Added) {
                this.tracked.Remove(id);
            } else {
                entry.State = EntityState.Deleted;
            }
        }
    }
}
```

Observar las reglas de transición de estado, que son la lógica medular del rastreador:

| Estado actual | Operación | Estado resultante |
|---|---|---|
| (no rastreada) | Create | Added |
| (no rastreada) | Update | Modified |
| Added | Update | Added (con nuevos valores) |
| Added | Remove | (sale del rastreador) |
| Unchanged | Update | Modified |
| Unchanged | Remove | Deleted |
| Modified | Update | Modified (con nuevos valores) |
| Modified | Remove | Deleted |

Lo elegante es que estas reglas no son arbitrarias: surgen naturalmente de pensar "qué tendría que pasar al final si la base se sincroniza con este estado". Si una entidad nueva nunca se guardó y ya se eliminó, no tiene sentido emitir nada. Si una entidad se modificó dos veces, solo importa el último valor.

## 10. Implementación con SQLite

Para SQLite el rastreador es el mismo, pero el `SaveChanges` ahora emite SQL distinto según el estado de cada entrada, dentro de una transacción.

```csharp
public class SqliteUnitOfWork : IUnitOfWork {
    private readonly string connectionString;
    private readonly Dictionary<int, TrackedEntry<Contacto>> tracked;

    public SqliteUnitOfWork(string path) {
        this.connectionString = $"Data Source={path}";
        this.tracked = new();
        this.InitializeSchema();
        this.Contactos = new SqliteContactoRepository(this.connectionString, this.tracked);
    }

    public IRepository<Contacto, int> Contactos { get; }

    public void SaveChanges() {
        var pending = this.tracked.Values
            .Where(e => e.State != EntityState.Unchanged)
            .ToList();
        if (pending.Count == 0) return;

        using var connection = new SqliteConnection(this.connectionString);
        connection.Open();
        using var transaction = connection.BeginTransaction();

        foreach (var entry in pending) {
            switch (entry.State) {
                case EntityState.Added:
                    this.ExecuteInsert(connection, transaction, entry.Entity);
                    break;
                case EntityState.Modified:
                    this.ExecuteUpdate(connection, transaction, entry.Entity);
                    break;
                case EntityState.Deleted:
                    this.ExecuteDelete(connection, transaction, entry.Entity.Id);
                    break;
            }
        }

        transaction.Commit();
        this.AcceptChanges();
    }

    private void ExecuteInsert(SqliteConnection connection, SqliteTransaction tx, Contacto c) {
        var command = connection.CreateCommand();
        command.Transaction = tx;
        command.CommandText = """
            INSERT INTO Contactos (Id, Nombre, Telefono, Email)
            VALUES ($id, $nombre, $telefono, $email);
        """;
        command.Parameters.AddWithValue("$id", c.Id);
        command.Parameters.AddWithValue("$nombre", c.Nombre);
        command.Parameters.AddWithValue("$telefono", c.Telefono);
        command.Parameters.AddWithValue("$email", c.Email);
        command.ExecuteNonQuery();
    }

    private void ExecuteUpdate(SqliteConnection connection, SqliteTransaction tx, Contacto c) {
        var command = connection.CreateCommand();
        command.Transaction = tx;
        command.CommandText = """
            UPDATE Contactos
            SET Nombre = $nombre, Telefono = $telefono, Email = $email
            WHERE Id = $id;
        """;
        command.Parameters.AddWithValue("$id", c.Id);
        command.Parameters.AddWithValue("$nombre", c.Nombre);
        command.Parameters.AddWithValue("$telefono", c.Telefono);
        command.Parameters.AddWithValue("$email", c.Email);
        command.ExecuteNonQuery();
    }

    private void ExecuteDelete(SqliteConnection connection, SqliteTransaction tx, int id) {
        var command = connection.CreateCommand();
        command.Transaction = tx;
        command.CommandText = "DELETE FROM Contactos WHERE Id = $id;";
        command.Parameters.AddWithValue("$id", id);
        command.ExecuteNonQuery();
    }

    private void AcceptChanges() {
        var ids = this.tracked.Keys.ToList();
        foreach (var id in ids) {
            if (this.tracked[id].State == EntityState.Deleted) {
                this.tracked.Remove(id);
            } else {
                this.tracked[id].State = EntityState.Unchanged;
            }
        }
    }

    private void InitializeSchema() {
        using var connection = new SqliteConnection(this.connectionString);
        connection.Open();
        var command = connection.CreateCommand();
        command.CommandText = """
            CREATE TABLE IF NOT EXISTS Contactos (
                Id       INTEGER PRIMARY KEY,
                Nombre   TEXT NOT NULL,
                Telefono TEXT NOT NULL,
                Email    TEXT NOT NULL
            );
        """;
        command.ExecuteNonQuery();
    }
}
```

El repositorio de SQLite es casi idéntico al de JSON, con dos diferencias:

```csharp
public class SqliteContactoRepository : IRepository<Contacto, int> {
    private readonly string connectionString;
    private readonly Dictionary<int, TrackedEntry<Contacto>> tracked;

    public SqliteContactoRepository(
        string connectionString,
        Dictionary<int, TrackedEntry<Contacto>> tracked
    ) {
        this.connectionString = connectionString;
        this.tracked = tracked;
    }

    // 1. ReadAll consulta la base, no el rastreador.
    public IReadOnlyList<Contacto> ReadAll() {
        using var connection = new SqliteConnection(this.connectionString);
        connection.Open();
        var command = connection.CreateCommand();
        command.CommandText = "SELECT Id, Nombre, Telefono, Email FROM Contactos;";

        var result = new List<Contacto>();
        using var reader = command.ExecuteReader();
        while (reader.Read()) {
            result.Add(new Contacto(
                reader.GetInt32(0),
                reader.GetString(1),
                reader.GetString(2),
                reader.GetString(3)
            ));
        }
        return result;
    }

    public Contacto? ReadOne(int id) {
        using var connection = new SqliteConnection(this.connectionString);
        connection.Open();
        var command = connection.CreateCommand();
        command.CommandText = "SELECT Id, Nombre, Telefono, Email FROM Contactos WHERE Id = $id;";
        command.Parameters.AddWithValue("$id", id);

        using var reader = command.ExecuteReader();
        if (!reader.Read()) return null;
        return new Contacto(
            reader.GetInt32(0),
            reader.GetString(1),
            reader.GetString(2),
            reader.GetString(3)
        );
    }

    public void Create(Contacto c) {
        this.tracked[c.Id] = new TrackedEntry<Contacto>(c, EntityState.Added);
    }

    public void Update(Contacto c) {
        if (this.tracked.TryGetValue(c.Id, out var entry)) {
            entry.Entity = c;
            if (entry.State != EntityState.Added) {
                entry.State = EntityState.Modified;
            }
        } else {
            this.tracked[c.Id] = new TrackedEntry<Contacto>(c, EntityState.Modified);
        }
    }

    // 2. Remove puede recibir un id que nunca fue cargado en el rastreador.
    //    En ese caso creamos un stub para emitir DELETE WHERE Id = id.
    public void Remove(int id) {
        if (this.tracked.TryGetValue(id, out var entry)) {
            if (entry.State == EntityState.Added) {
                this.tracked.Remove(id);
            } else {
                entry.State = EntityState.Deleted;
            }
        } else {
            var stub = new Contacto(id, "", "", "");
            this.tracked[id] = new TrackedEntry<Contacto>(stub, EntityState.Deleted);
        }
    }
}
```

Las dos diferencias respecto del repositorio de JSON son:

1. **`ReadAll` y `ReadOne` consultan la base**, no el rastreador. La razón: con SQLite no cargamos todo en memoria al arrancar, así que las lecturas siempre van al disco. Esto tiene una consecuencia importante: los cambios pendientes (todavía no guardados) **no se ven** en las consultas. Es una limitación deliberada del diseño didáctico para que el código quede simple.

2. **`Remove` puede recibir un id que no esté rastreado**. En ese caso creamos una entidad "stub" (con el Id correcto y el resto vacío) marcada como `Deleted`. Lo único que importa para el `DELETE` es el Id; los demás campos del stub no se usan.

Hay una oportunidad de refactor evidente: las dos clases de repositorio comparten casi toda la lógica de `Create`, `Update` y `Remove`. Podríamos extraer una clase `ChangeTracker<T, TId>` que las dos usen por composición. Lo dejamos como ejercicio: el tutorial gana en claridad mostrando las dos versiones expandidas.

## 11. El composition root

El "composition root" es el único lugar del programa donde se decide qué implementación concreta se usa. Para una app de consola, es `Program.Main`.

```csharp
class Program {
    static void Main() {
        IUnitOfWork uow = new JsonUnitOfWork("agenda.json");
        // o: IUnitOfWork uow = new SqliteUnitOfWork("agenda.db");

        uow.Contactos.Create(new Contacto(1, "Ana", "381-123", "ana@mail.com"));
        uow.Contactos.Create(new Contacto(2, "Beto", "381-456", "beto@mail.com"));
        uow.Contactos.Remove(1);

        uow.SaveChanges();
    }
}
```

Cambiar de medio de persistencia es **una línea**. Toda la lógica que use `IUnitOfWork` queda intacta. Esto no es un truco: es la consecuencia de haber respetado las dos separaciones desde el principio.

## 12. Verificando los principios SOLID

Ahora que está armado, vale revisar que el diseño cumple los cinco principios. No como ritual, sino como verificación de que las decisiones tomadas tienen el efecto que prometían.

**Single Responsibility.** Cada clase tiene una razón de cambio única. `Contacto` cambia si cambia el dominio. `TrackedEntry` cambia si cambia el modelo de rastreo. `JsonUnitOfWork` cambia si cambia el formato del archivo. `SqliteUnitOfWork` cambia si cambia el esquema SQL. Los repositorios cambian si cambia la lógica de transición de estados.

**Open/Closed.** Para soportar XML, una API REST o una base PostgreSQL, no hace falta modificar nada existente: se agrega una nueva implementación de `IUnitOfWork` y se conecta en el composition root.

**Liskov Substitution.** Cualquier `IUnitOfWork` es intercambiable. La lógica de negocio no nota la diferencia, porque el contrato está bien definido: las operaciones son granulares y la persistencia es explícita. Esto no es trivial — fue el problema que aparecía cuando la interfaz tenía un único `Save(IEnumerable<T>)`, donde JSON y SQLite tenían que fingir trabajar igual.

**Interface Segregation.** `IRepository<T, TId>` ofrece exactamente las operaciones que un consumidor típico necesita. Si más adelante aparece un consumidor que solo lee (un reporte, por ejemplo), conviene partirla en `IReader<T, TId>` y `IWriter<T, TId>`. Por ahora, con un único caso de uso, mantener una interfaz cohesiva es razonable.

**Dependency Inversion.** La lógica de negocio depende de `IUnitOfWork`, no de `JsonUnitOfWork`. El cableado entre abstracción e implementación ocurre solo en `Program`. Las flechas de dependencia van desde lo concreto hacia lo abstracto.

---

## 13. Cuándo aplicarlo y cuándo no

Una nota honesta para cerrar.

Para una agenda de cátedra con 20 contactos, este diseño es claramente desproporcionado. La versión simple del punto 2 funcionaría perfectamente y sería más fácil de leer.

El valor de aprender este patrón no está en aplicarlo siempre. Está en:

- **Reconocer cuándo el problema lo pide.** Cuando hay múltiples medios de persistencia posibles, o cuando las operaciones tienen que ser atómicas, o cuando el rendimiento de "guardar todo en cada cambio" empieza a doler, este diseño deja de ser overkill y se vuelve la solución natural.
- **Entender los patrones que vienen después.** Repository pattern en frameworks empresariales, ORMs, arquitectura hexagonal — todos descansan sobre estas dos separaciones.
- **Construir el reflejo de pensar en abstracciones útiles.** No abstraer porque sí, sino preguntarse: ¿qué dos preguntas estoy mezclando? ¿Puedo darle a cada una su propia respuesta?

La regla práctica: empezar simple, identificar la fricción, y solo entonces introducir las abstracciones que la resuelven. Cuando se hace al revés (abstraer primero, ver si sirve después), el código se llena de interfaces que no aportan nada.
