# Principios SOLID en C#

Cinco reglas de diseño orientado a objetos que hacen el código más fácil de mantener, extender y testear:

| Letra | Principio | Idea |
|---|---|---|
| **S** | Single Responsibility | Una clase, una razón para cambiar |
| **O** | Open/Closed | Abierto a extensión, cerrado a modificación |
| **L** | Liskov Substitution | Las subclases reemplazan a la base sin romper nada |
| **I** | Interface Segregation | Muchas interfaces pequeñas > una grande |
| **D** | Dependency Inversion | Depender de abstracciones, no de implementaciones |

## S — Single Responsibility

Una clase que hace demasiado (gestionar saldo + persistir + notificar + calcular puntos) tiene múltiples razones para cambiar. La solución: separar responsabilidades en clases propias.

```csharp
// Solo modela la tarjeta
public class Tarjeta {
    public string Numero { get; set; }
    public decimal Saldo { get; set; }
    public int Puntos { get; set; }
    public string Tipo { get; set; }

    public void Acreditar(decimal monto) {
        if (monto <= 0) throw new MontoInvalidoException();
        Saldo += monto;
    }
    public void Debitar(decimal monto) {
        if (monto <= 0)    throw new MontoInvalidoException();
        if (Saldo < monto) throw new SaldoInsuficienteException();
        Saldo -= monto;
    }
}

public class RegistroMovimientos {
    public void Registrar(string numero, string op, decimal monto) =>
        File.AppendAllText("movimientos.txt", $"{DateTime.Now}: {op} {monto} en {numero}\n");
}

public class NotificadorEmail {
    public void Notificar(string dest, string asunto, string cuerpo) { /* SmtpClient */ }
}

public class CalculadorPuntos {
    public int Calcular(string tipo, decimal monto) => tipo switch {
        "clasica"  => (int)(monto * 0.01m),
        "gold"     => (int)(monto * 0.02m),
        "platinum" => (int)(monto * 0.05m),
        _          => 0
    };
}
```

## O — Open/Closed

Si agregar un tipo nuevo de tarjeta obliga a modificar `CalculadorPuntos`, la clase no está cerrada. Solución: polimorfismo.

```csharp
public abstract class Tarjeta {
    public decimal Saldo { get; protected set; }
    public int Puntos    { get; protected set; }

    public void Debitar(decimal monto) {
        if (Saldo < monto) throw new SaldoInsuficienteException();
        Saldo  -= monto;
        Puntos += CalcularPuntos(monto);
    }
    protected abstract int CalcularPuntos(decimal monto);
}

public class TarjetaClasica  : Tarjeta { protected override int CalcularPuntos(decimal m) => (int)(m * 0.01m); }
public class TarjetaGold     : Tarjeta { protected override int CalcularPuntos(decimal m) => (int)(m * 0.02m); }
public class TarjetaPlatinum : Tarjeta { protected override int CalcularPuntos(decimal m) => (int)(m * 0.05m); }

// Agregar TarjetaBlack no toca nada existente:
public class TarjetaBlack : Tarjeta { protected override int CalcularPuntos(decimal m) => (int)(m * 0.10m); }
```

## L — Liskov Substitution

Si una subclase lanza excepciones inesperadas en métodos de la clase base, viola LSP:

```csharp
// ✗ Mal — rompe el contrato de Tarjeta
public class TarjetaPrepaga : Tarjeta {
    public override void Acreditar(decimal monto) =>
        throw new OperacionNoSoportadaException();
}
```

Señal de violación: `if (tarjeta is TarjetaPrepaga)` diseminado en el código. Solución: rediseñar la jerarquía para que `Acreditar` solo exista donde tiene sentido.

## I — Interface Segregation

Una interfaz enorme obliga a implementar métodos sin sentido. Mejor segregar:

```csharp
public interface IAcreditable  { void Acreditar(decimal monto); }
public interface IDebitable    { void Debitar(decimal monto); }
public interface IAcumulaPuntos { int Puntos { get; } }
public interface ICanjeablePuntos { void CanjearPuntos(int puntos); }

public class TarjetaDebito  : IAcreditable, IDebitable { /* solo lo que le corresponde */ }
public class TarjetaCredito : IDebitable, IAcumulaPuntos, ICanjeablePuntos { /* ... */ }

// La terminal POS solo necesita debitar:
public class TerminalPOS {
    public void Cobrar(IDebitable tarjeta, decimal monto) => tarjeta.Debitar(monto);
}
```

## D — Dependency Inversion

`ServicioTarjeta` no debería instanciar directamente `NotificadorEmail` ni `RegistroMovimientos` — depende de detalles técnicos, no de abstracciones.

```csharp
public interface IRegistroMovimientos {
    void Registrar(string numero, string op, decimal monto);
}
public interface INotificador {
    void Notificar(string dest, string asunto, string cuerpo);
}

// Módulo de alto nivel: depende de abstracciones
public class ServicioTarjeta(IRegistroMovimientos registro, INotificador notificador) {
    public void Debitar(Tarjeta tarjeta, decimal monto) {
        tarjeta.Debitar(monto);
        registro.Registrar(tarjeta.Numero, "Debito", monto);
        notificador.Notificar(tarjeta.EmailTitular, "Débito", $"Debitaste ${monto}");
    }
}

// Composición en Program.cs — cambiar correo por SMS es una línea:
var servicio = new ServicioTarjeta(new RegistroEnArchivo(), new NotificadorEmail());
```

**Ganancia**: `ServicioTarjeta` es testeable con mocks sin SMTP ni archivos reales.

## Resumen

| Principio | Señal de violación | Solución |
|---|---|---|
| SRP | Clase con múltiples razones para cambiar | Separar en clases con una responsabilidad |
| OCP | Agregar variante obliga a modificar código existente | Polimorfismo / herencia / estrategia |
| LSP | `if (x is SubtipoConcreto)` por doquier | Rediseñar jerarquía |
| ISP | `OperacionNoSoportadaException` en métodos heredados | Interfaces pequeñas y específicas |
| DIP | `new` de infraestructura dentro de servicios de negocio | Inyección de dependencias |
