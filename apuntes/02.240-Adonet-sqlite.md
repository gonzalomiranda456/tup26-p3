## 1. ¿Por qué una base de datos?

Cuando empezamos a guardar datos, la primera idea siempre es un archivo. Un `.txt` con líneas, un `.csv` con columnas separadas por comas, un `.json` con objetos serializados. Todas son soluciones legítimas para casos chicos, y de hecho ya las hemos usado.

Pero a medida que la aplicación crece, los archivos planos empiezan a doler. Algunos de los puntos donde duelen:

- **Búsquedas.** Si quiero "todos los contactos cuyo teléfono empieza con 381", tengo que leer todo el archivo y filtrar a mano. Para 50 contactos no importa; para 50.000 sí.
- **Modificaciones parciales.** Para cambiar un solo campo de un solo registro tengo que reescribir el archivo entero.
- **Acceso concurrente.** Si dos partes del programa (o dos programas) quieren modificar el mismo archivo al mismo tiempo, el resultado es impredecible. Uno pisa al otro.
- **Atomicidad.** Si el programa muere a mitad de una escritura, el archivo queda corrupto.
- **Estructura.** Cualquier validación ("este campo es un número", "este campo no puede ser null") la tengo que escribir yo. El archivo no me ayuda.

Una **base de datos** es una pieza de software diseñada específicamente para resolver estos problemas. A cambio de aprender un poco de vocabulario nuevo, recibimos:

- Un lenguaje de consultas optimizado para buscar, filtrar y agregar (SQL).
- Modificaciones puntuales eficientes (cambiar un registro no toca el resto).
- Control de concurrencia.
- Transacciones (un grupo de operaciones se confirma todo junto o no se confirma nada).
- Validación estructural automática (tipos, restricciones, claves).

La pregunta entonces no es "¿base de datos sí o no?" sino "¿cuál?". Y para responderla, primero conviene entender cómo .NET se conecta con bases de datos en general.

## 2. ADO.NET: el patrón común para hablar con cualquier base

En el ecosistema .NET, la forma estándar de hablar con una base de datos relacional se llama **ADO.NET**. La idea fundamental es simple: hay conceptos que aparecen en toda interacción con una base —*conexión*, *comando*, *parámetro*, *transacción*, *cursor de resultados*— y .NET define para cada uno una **clase abstracta** en el namespace `System.Data.Common`.

| Clase abstracta  | Qué representa                           |
|------------------|------------------------------------------|
| `DbConnection`   | Una conexión a la base                   |
| `DbCommand`      | Una sentencia SQL lista para ejecutar    |
| `DbParameter`    | Un parámetro de un comando               |
| `DbDataReader`   | Un cursor sobre los resultados           |
| `DbTransaction`  | Una transacción                          |

Cada motor de base de datos provee su propia implementación concreta de estas clases, empaquetada en una librería que se llama **proveedor**. Cuando hablamos con un motor específico, lo que hacemos es instalar el proveedor correspondiente y usar las clases concretas que provee.

```
DbConnection (abstracta)
├── SqliteConnection       (paquete Microsoft.Data.Sqlite)
├── SqlConnection          (paquete Microsoft.Data.SqlClient, para SQL Server)
├── NpgsqlConnection       (paquete Npgsql, para PostgreSQL)
└── MySqlConnection        (paquete MySqlConnector, para MySQL/MariaDB)
```

Esta arquitectura tiene una consecuencia muy útil: si escribimos nuestro código contra las clases **abstractas** (`DbConnection`, `DbCommand`, etc.), el mismo código funciona con cualquier motor. Lo único que cambia es la línea donde se crea la conexión concreta.

En este tutorial vamos a escribir todo así: tipado contra ADO.NET genérico, usando SQLite como el motor concreto del ejemplo. Al final, vamos a ver concretamente qué hay que cambiar para usar otro motor.

## 3. ¿Qué es SQLite?

Las bases de datos como PostgreSQL, MySQL o SQL Server funcionan como **servidores**: un proceso aparte, corriendo en alguna máquina, escuchando conexiones por la red, gestionando usuarios y permisos. Para usarlas en serio hay que instalarlas, configurarlas y mantenerlas.

**SQLite es diferente: es una base de datos que vive como un archivo.**

Concretamente, SQLite es una librería que se enlaza a tu programa y lee/escribe un único archivo `.db` que contiene toda la base. No hay servidor, no hay puerto, no hay usuario y contraseña. Si querés copiar la base, copiás el archivo. Si querés borrarla, borrás el archivo.

Esto la hace ideal para:

- Aplicaciones de escritorio (cada usuario tiene su propia base local).
- Apps móviles.
- Casos donde un solo proceso accede a los datos.
- Aprender SQL sin tener que instalar un servidor.
- Prototipos y proyectos académicos.

Por eso es nuestra elección para este tutorial: el código que escribimos sigue el patrón ADO.NET universal, pero el motor concreto que corre debajo es trivialmente fácil de poner en marcha.

## 4. SQL: el lenguaje para hablar con la base

Para pedirle cosas a la base de datos usamos un lenguaje llamado **SQL** (Structured Query Language). No es un lenguaje imperativo como C# — no le decimos *cómo* hacer las cosas. Es **declarativo**: le decimos *qué* queremos, y la base se encarga del cómo.

Un ejemplo de la diferencia: si quisiéramos buscar el contacto con id 5 en C# escribiríamos un bucle, una comparación, un return. En SQL escribimos:

```sql
SELECT * FROM Contactos WHERE Id = 5;
```

Le pedimos lo que queremos. La base elige el algoritmo, usa los índices que existan, y devuelve el resultado.

SQL tiene varios "subconjuntos" que conviene distinguir:

- **DDL** (Data Definition Language): define la estructura. `CREATE TABLE`, `ALTER TABLE`, `DROP TABLE`.
- **DML** (Data Manipulation Language): manipula los datos. `INSERT`, `UPDATE`, `DELETE`.
- **DQL** (Data Query Language): consulta. `SELECT`.
- **TCL** (Transaction Control Language): controla transacciones. `BEGIN`, `COMMIT`, `ROLLBACK`.

En este tutorial vamos a tocar todos estos, en ese orden.

Una convención de estilo: **las palabras clave de SQL se escriben en MAYÚSCULAS**. No es obligatorio (el motor las acepta en minúsculas), pero ayuda a distinguirlas de los nombres de tablas y columnas. La vamos a respetar.

Una nota sobre portabilidad: el SQL básico (las cláusulas `SELECT`, `WHERE`, `INSERT`, `UPDATE`, `DELETE`, los operadores, los joins) es estándar y funciona casi igual en todos los motores. Pero hay diferencias en **tipos de datos**, **autoincremento de claves**, **funciones de fecha** y otras cosas particulares de cada dialecto. Cuando aparezca una decisión específica de SQLite, lo vamos a marcar.

## 5. Configuración del proyecto

Para que un programa C# pueda hablar con una base de datos vía ADO.NET, necesitamos dos cosas: el namespace genérico de ADO.NET (que viene con .NET) y un **proveedor** concreto para el motor que queremos usar.

Creamos un proyecto de consola y agregamos el proveedor de SQLite:

```bash
dotnet new console -n AgendaDb
cd AgendaDb
dotnet add package Microsoft.Data.Sqlite
```

En el código, importamos los dos namespaces que vamos a usar:

```csharp
using System.Data;             // tipos comunes (ConnectionState, IDbCommand, etc.)
using System.Data.Common;      // clases abstractas de ADO.NET (DbConnection, DbCommand, ...)
using Microsoft.Data.Sqlite;   // proveedor concreto de SQLite
```

La última línea es la **única** que menciona explícitamente a SQLite. Todo lo demás del programa va a estar tipado contra las clases abstractas. Cuando lleguemos a la sección de "cambiar de motor" vamos a ver que reemplazar esa línea (y la cadena de conexión) es básicamente todo lo que hace falta.

## 6. Conexión: `DbConnection`

Lo primero que necesitamos es una **conexión** a la base. Una conexión representa el "hilo abierto" entre nuestro programa y el motor. La declaramos contra la clase abstracta y la instanciamos contra la concreta:

```csharp
DbConnection connection = new SqliteConnection("Data Source=agenda.db");
connection.Open();
```

Tres cosas para señalar:

**El connection string.** El parámetro `"Data Source=agenda.db"` es lo que se llama una *cadena de conexión*. Para SQLite es trivialmente simple: solo dice qué archivo abrir. Para servidores reales incluye host, puerto, usuario y contraseña. Si el archivo `agenda.db` no existe, SQLite lo crea automáticamente al abrir la conexión.

**`connection.Open()`.** Crear el objeto no abre la conexión; hay que llamar a `Open()` explícitamente. Hasta entonces, no hay diálogo con la base.

**El `using`.** Una conexión es un recurso: ocupa memoria y, en muchos motores, mantiene un canal de red abierto. Hay que cerrarla cuando terminamos. La forma idiomática es la *using declaration*:

```csharp
using DbConnection connection = new SqliteConnection("Data Source=agenda.db");
connection.Open();
// ... usar la conexión ...
// al salir del bloque, se cierra automáticamente
```

`DbConnection` implementa `IDisposable`, así que el `using` funciona sin importar el motor concreto.

Toda interacción con la base va a seguir este patrón: abrir conexión, hacer cosas, cerrar (automáticamente al salir del bloque).

## 7. Crear el esquema: `CREATE TABLE`

Antes de poder guardar datos, hay que decirle a la base qué forma van a tener. Eso se hace con `CREATE TABLE`.

```sql
CREATE TABLE Contactos (
    Id       INTEGER PRIMARY KEY,
    Nombre   TEXT NOT NULL,
    Telefono TEXT NOT NULL,
    Email    TEXT NOT NULL
);
```

Esta sentencia define una **tabla** llamada `Contactos` con cuatro **columnas**. Cada columna tiene un nombre, un tipo, y opcionalmente restricciones.

**Tipos en SQLite.** SQLite tiene una clasificación de tipos más laxa que otras bases. Los principales son:

- `INTEGER` — números enteros.
- `REAL` — números de punto flotante.
- `TEXT` — cadenas de texto.
- `BLOB` — datos binarios crudos.
- `NULL` — ausencia de valor.

Acá aparece la primera diferencia entre dialectos: en SQL Server, los mismos tipos se llaman `INT`, `FLOAT`, `NVARCHAR(100)`; en PostgreSQL, `INTEGER`, `DOUBLE PRECISION`, `VARCHAR`. La estructura de la sentencia es la misma, los tipos cambian.

**Restricciones (constraints).** En la declaración aparecen dos:

- `PRIMARY KEY` — esta columna identifica de forma única a cada fila. No puede haber dos contactos con el mismo `Id`.
- `NOT NULL` — esta columna no puede estar vacía. Si intentamos insertar un contacto sin nombre, la base rechaza la operación.

Hay otras restricciones útiles: `UNIQUE`, `CHECK`, `DEFAULT`, `FOREIGN KEY`. Las vamos a ver más adelante en el cuatrimestre.

**Cómo ejecutarla desde C#:**

```csharp
using DbConnection connection = new SqliteConnection("Data Source=agenda.db");
connection.Open();

DbCommand command = connection.CreateCommand();
command.CommandText = """
    CREATE TABLE IF NOT EXISTS Contactos (
        Id       INTEGER PRIMARY KEY,
        Nombre   TEXT NOT NULL,
        Telefono TEXT NOT NULL,
        Email    TEXT NOT NULL
    );
""";
command.ExecuteNonQuery();
```

Tres ideas nuevas acá:

**`DbCommand`** representa una sentencia SQL lista para ejecutar. Se obtiene siempre desde una conexión con `connection.CreateCommand()`. Como `connection` está tipada como `DbConnection`, el tipo de retorno también es la clase abstracta (`DbCommand`). En el SQL le asignamos la sentencia mediante la propiedad `CommandText`.

**`IF NOT EXISTS`** es una variante muy práctica de `CREATE TABLE` que evita errores cuando la tabla ya existe (por ejemplo, en la segunda ejecución del programa). Sin esa cláusula, el motor tiraría una excepción. SQLite, PostgreSQL y MySQL soportan esta sintaxis; SQL Server no, y hay que usar otra construcción equivalente.

**`ExecuteNonQuery()`** ejecuta el comando cuando *no esperamos un resultado tabular*. `CREATE TABLE`, `INSERT`, `UPDATE`, `DELETE` son todos casos de "ejecutar pero no devolver filas". El método retorna un `int` con el número de filas afectadas (cero para DDL).

Con esto, después de la primera ejecución del programa, queda creado el archivo `agenda.db` con la tabla `Contactos` vacía dentro.

## 8. Insertar datos: `INSERT`

Para agregar una fila a la tabla:

```sql
INSERT INTO Contactos (Id, Nombre, Telefono, Email)
VALUES (1, 'Ana', '381-123', 'ana@mail.com');
```

Notar las comillas simples para los strings. SQL usa comillas simples para literales de texto; las dobles tienen otro significado (identificadores).

Desde C#:

```csharp
DbCommand command = connection.CreateCommand();
command.CommandText = """
    INSERT INTO Contactos (Id, Nombre, Telefono, Email)
    VALUES (1, 'Ana', '381-123', 'ana@mail.com');
""";
command.ExecuteNonQuery();
```

Ya tenemos un contacto guardado. Pero acá viene el punto más importante de todo el tutorial.

## 9. Parámetros: nunca concatenar SQL

La pregunta natural es: ¿cómo hago para insertar un contacto cuyos datos vienen de una variable?

La respuesta **incorrecta**, la que aparece sola en la cabeza:

```csharp
// ⚠️ NUNCA hacer esto
command.CommandText = $"""
    INSERT INTO Contactos (Id, Nombre, Telefono, Email)
    VALUES ({id}, '{nombre}', '{telefono}', '{email}');
""";
```

Esto funciona... hasta que un usuario ingresa como nombre `O'Brien`. Ahí la comilla simple del nombre cierra prematuramente el string SQL y el motor falla con un error de sintaxis.

Y lo peor no es que falle. Lo peor es lo que pasa cuando el "usuario" es malicioso e ingresa como nombre:

```
'); DROP TABLE Contactos; --
```

El SQL resultante es:

```sql
INSERT INTO Contactos (Id, Nombre, Telefono, Email)
VALUES (1, ''); DROP TABLE Contactos; --', ...);
```

Y la base ejecuta dos sentencias: el INSERT trunco, y un `DROP TABLE` que borra toda la información de contactos. Esto se llama **inyección SQL** y es una de las vulnerabilidades más antiguas, más conocidas, y que sin embargo siguen apareciendo en software real todos los meses.

**La solución correcta son los parámetros.** En lugar de armar el SQL concatenando, dejamos huecos identificados y los llenamos por separado.

En ADO.NET genérico, agregar un parámetro a un comando se hace en tres pasos:

```csharp
DbParameter p = command.CreateParameter();
p.ParameterName = "@id";
p.Value = id;
command.Parameters.Add(p);
```

`DbCommand.CreateParameter()` devuelve un `DbParameter` "vacío" del tipo apropiado para el motor. Le asignamos nombre y valor, y lo agregamos a la colección de parámetros del comando. Repetimos para cada parámetro.

Esto es repetitivo, así que vale la pena armar un helper:

```csharp
static void AddParameter(DbCommand command, string name, object? value) {
    DbParameter p = command.CreateParameter();
    p.ParameterName = name;
    p.Value = value ?? DBNull.Value;
    command.Parameters.Add(p);
}
```

El detalle del `?? DBNull.Value`: si el valor es `null` en C#, hay que pasarlo como `DBNull.Value` explícitamente, porque ADO.NET no acepta `null` como valor de parámetro y lo interpretaría como "parámetro sin valor" en lugar de "valor NULL".

Con el helper, el INSERT parametrizado queda:

```csharp
DbCommand command = connection.CreateCommand();
command.CommandText = """
    INSERT INTO Contactos (Id, Nombre, Telefono, Email)
    VALUES (@id, @nombre, @telefono, @email);
""";
AddParameter(command, "@id", id);
AddParameter(command, "@nombre", nombre);
AddParameter(command, "@telefono", telefono);
AddParameter(command, "@email", email);
command.ExecuteNonQuery();
```

Sobre el prefijo de los parámetros: cada motor tiene su convención (`@` en SQL Server, MySQL y PostgreSQL; `:` en Oracle; `$`, `@` o `:` en SQLite). Vamos a usar `@` consistentemente porque es el más portable: SQLite lo acepta, y si mañana cambiamos a otro motor el código sigue funcionando sin tocar el SQL.

Cuando el comando se ejecuta, el motor recibe la sentencia y los valores **por separado** y los combina internamente, escapando cualquier carácter problemático y descartando cualquier intento de inyectar más SQL.

**Regla absoluta: si un valor viene de una variable, va por parámetro. Sin excepciones.**

No es solo una cuestión de seguridad. También es de rendimiento (la base puede cachear el plan de ejecución de la consulta parametrizada y reutilizarlo) y de corrección (el motor maneja los tipos correctamente, sin depender de la conversión a string).

## 10. Leer datos: `SELECT` y `DbDataReader`

Para leer datos usamos `SELECT`. La forma más simple:

```sql
SELECT Id, Nombre, Telefono, Email FROM Contactos;
```

Esto devuelve todas las filas de la tabla, con esas cuatro columnas, en algún orden.

Para filtrar agregamos `WHERE`:

```sql
SELECT Id, Nombre, Telefono, Email
FROM Contactos
WHERE Id = @id;
```

Para ordenar, `ORDER BY`:

```sql
SELECT Id, Nombre, Telefono, Email
FROM Contactos
ORDER BY Nombre ASC;
```

Para limitar la cantidad, `LIMIT`:

```sql
SELECT Id, Nombre FROM Contactos LIMIT 10;
```

Estas cláusulas se combinan en un orden fijo: `SELECT ... FROM ... WHERE ... ORDER BY ... LIMIT ...`. Memorizarlo ahora ahorra muchos errores de sintaxis después.

Una nota de portabilidad: `LIMIT` lo soportan SQLite, PostgreSQL y MySQL; SQL Server usa `SELECT TOP n ...` o la cláusula `OFFSET ... FETCH`. Es una de esas diferencias de dialecto que aparecen al cambiar de motor.

**Cómo recibir los resultados desde C#.**

A diferencia de `INSERT`/`UPDATE`/`DELETE`, un `SELECT` devuelve **filas**. Para recorrerlas usamos `ExecuteReader()`, que devuelve un `DbDataReader`:

```csharp
DbCommand command = connection.CreateCommand();
command.CommandText = "SELECT Id, Nombre, Telefono, Email FROM Contactos;";

using DbDataReader reader = command.ExecuteReader();
while (reader.Read()) {
    int id = reader.GetInt32(0);
    string nombre = reader.GetString(1);
    string telefono = reader.GetString(2);
    string email = reader.GetString(3);

    Console.WriteLine($"{id}: {nombre} ({telefono})");
}
```

Cómo funciona el reader:

- `reader.Read()` avanza a la siguiente fila. Devuelve `true` si hay fila, `false` cuando se acabaron. Por eso el patrón es un `while`.
- Antes del primer `Read()`, el reader no apunta a ninguna fila. **Hay que llamar a `Read()` antes de leer cualquier columna.**
- `reader.GetInt32(0)`, `reader.GetString(1)`, etc., leen la columna correspondiente de la fila actual. El número es el **índice de la columna en el SELECT**, empezando desde 0.
- Hay un `GetXxx` por cada tipo: `GetInt32`, `GetInt64`, `GetString`, `GetDouble`, `GetBoolean`, `GetDateTime`, etc. Es responsabilidad nuestra usar el correcto según el tipo de la columna.
- El reader es un recurso que también necesita `using`. Mantener un reader abierto bloquea la conexión para otras operaciones.

**Una consulta que devuelve un valor único.** Cuando la consulta devuelve una sola fila con una sola columna (un conteo, un máximo, un valor calculado), hay un atajo: `ExecuteScalar()`.

```csharp
DbCommand command = connection.CreateCommand();
command.CommandText = "SELECT COUNT(*) FROM Contactos;";

object? resultado = command.ExecuteScalar();
long cantidad = (long)resultado!;
Console.WriteLine($"Hay {cantidad} contactos.");
```

`ExecuteScalar` devuelve `object?` con el valor de la primera columna de la primera fila. Hay que castearlo al tipo correspondiente. Es más cómodo que armar un reader cuando solo queremos un número.

## 11. Actualizar y eliminar

Las dos operaciones que faltan completan el CRUD.

**`UPDATE` modifica filas existentes:**

```csharp
DbCommand command = connection.CreateCommand();
command.CommandText = """
    UPDATE Contactos
    SET Nombre = @nombre, Telefono = @telefono, Email = @email
    WHERE Id = @id;
""";
AddParameter(command, "@id", id);
AddParameter(command, "@nombre", nombre);
AddParameter(command, "@telefono", telefono);
AddParameter(command, "@email", email);
command.ExecuteNonQuery();
```

La cláusula `WHERE` es **crítica**. Si la olvidamos:

```sql
UPDATE Contactos SET Nombre = 'Pepito';
```

…**todas** las filas pasan a tener el nombre 'Pepito'. La base no nos protege de esto. Cada vez que escribís un `UPDATE`, leelo dos veces buscando el `WHERE`.

**`DELETE` elimina filas:**

```csharp
DbCommand command = connection.CreateCommand();
command.CommandText = "DELETE FROM Contactos WHERE Id = @id;";
AddParameter(command, "@id", id);
command.ExecuteNonQuery();
```

Mismo cuidado con el `WHERE`. Un `DELETE FROM Contactos;` sin condición vacía la tabla entera.

`ExecuteNonQuery()` devuelve la cantidad de filas afectadas, lo cual es útil para verificar que la operación tuvo efecto:

```csharp
int filasAfectadas = command.ExecuteNonQuery();
if (filasAfectadas == 0) {
    Console.WriteLine("No se encontró el contacto a eliminar.");
}
```

## 12. Transacciones: cuando varias operaciones tienen que ser una sola

Pensemos un caso concreto: queremos transferir un teléfono entre dos contactos. Eso implica dos operaciones:

1. Borrar el teléfono del contacto A.
2. Asignárselo al contacto B.

Si ejecutamos las dos como sentencias sueltas y el programa muere entre la primera y la segunda, queda un teléfono perdido en el aire: ya no está en A y nunca llegó a B.

La solución es agruparlas en una **transacción**: una unidad atómica que se confirma toda junta o no se confirma nada.

```csharp
using DbConnection connection = new SqliteConnection("Data Source=agenda.db");
connection.Open();

using DbTransaction transaction = connection.BeginTransaction();
try {
    DbCommand quitar = connection.CreateCommand();
    quitar.Transaction = transaction;
    quitar.CommandText = "UPDATE Contactos SET Telefono = '' WHERE Id = @id;";
    AddParameter(quitar, "@id", idA);
    quitar.ExecuteNonQuery();

    DbCommand asignar = connection.CreateCommand();
    asignar.Transaction = transaction;
    asignar.CommandText = "UPDATE Contactos SET Telefono = @tel WHERE Id = @id;";
    AddParameter(asignar, "@tel", telefono);
    AddParameter(asignar, "@id", idB);
    asignar.ExecuteNonQuery();

    transaction.Commit();   // ← ahora sí, los cambios se persisten juntos
}
catch {
    transaction.Rollback();  // ← si algo falló, descartamos todo
    throw;
}
```

Las cuatro ideas a entender son:

**`BeginTransaction()`** abre una transacción. A partir de ahí, los cambios que hagamos quedan "pendientes" — la base los conoce, pero no los hace visibles a otros lectores ni los considera definitivos.

**`command.Transaction = transaction;`** asocia cada comando a la transacción abierta. Si no lo hacemos, el comando ejecuta fuera de la transacción y los cambios no son atómicos con el resto.

**`Commit()`** confirma la transacción. En ese instante, todos los cambios pendientes se vuelven definitivos, todos juntos.

**`Rollback()`** descarta la transacción. Todos los cambios pendientes se anulan. La base queda como si la transacción nunca hubiera ocurrido.

El patrón `try/commit/catch/rollback` es el idiomático: si llegamos al `Commit` sin excepciones, todo bien; si algo falló a mitad de camino, deshacemos todo y propagamos el error. Esto es lo que hace que la operación sea verdaderamente atómica.

Las transacciones también tienen un beneficio de rendimiento: si vamos a ejecutar muchas inserciones (digamos 1000), envolver todo en una transacción es órdenes de magnitud más rápido que ejecutar 1000 inserts sueltos. La razón es que la base hace fsync a disco al final de cada transacción, no al final de cada sentencia.

## 13. El patrón completo en una clase

Juntando todo, una clase que provee acceso CRUD a la tabla de contactos podría verse así. Notar que la clase no menciona SQLite en ningún lado: recibe una `DbConnection` por inyección, y trabaja contra ella sin saber qué motor concreto es.

```csharp
public class Agenda {
    private readonly DbConnection connection;

    public Agenda(DbConnection connection) {
        this.connection = connection;
        if (this.connection.State != ConnectionState.Open) {
            this.connection.Open();
        }
        this.InitializeSchema();
    }

    private void InitializeSchema() {
        DbCommand command = this.connection.CreateCommand();
        command.CommandText = """
            CREATE TABLE IF NOT EXISTS Contactos (
                Id       INTEGER PRIMARY KEY,
                Nombre   TEXT NOT NULL,
                Telefono TEXT NOT NULL,
                Email    TEXT NOT NULL
            );
        """;
        command.ExecuteNonQuery();
    }

    public void Agregar(Contacto c) {
        DbCommand command = this.connection.CreateCommand();
        command.CommandText = """
            INSERT INTO Contactos (Id, Nombre, Telefono, Email)
            VALUES (@id, @nombre, @telefono, @email);
        """;
        AddParameter(command, "@id", c.Id);
        AddParameter(command, "@nombre", c.Nombre);
        AddParameter(command, "@telefono", c.Telefono);
        AddParameter(command, "@email", c.Email);
        command.ExecuteNonQuery();
    }

    public List<Contacto> ListarTodos() {
        DbCommand command = this.connection.CreateCommand();
        command.CommandText = "SELECT Id, Nombre, Telefono, Email FROM Contactos ORDER BY Nombre;";

        var resultado = new List<Contacto>();
        using DbDataReader reader = command.ExecuteReader();
        while (reader.Read()) {
            resultado.Add(new Contacto(
                reader.GetInt32(0),
                reader.GetString(1),
                reader.GetString(2),
                reader.GetString(3)
            ));
        }
        return resultado;
    }

    public Contacto? Buscar(int id) {
        DbCommand command = this.connection.CreateCommand();
        command.CommandText = "SELECT Id, Nombre, Telefono, Email FROM Contactos WHERE Id = @id;";
        AddParameter(command, "@id", id);

        using DbDataReader reader = command.ExecuteReader();
        if (!reader.Read()) return null;
        return new Contacto(
            reader.GetInt32(0),
            reader.GetString(1),
            reader.GetString(2),
            reader.GetString(3)
        );
    }

    public void Modificar(Contacto c) {
        DbCommand command = this.connection.CreateCommand();
        command.CommandText = """
            UPDATE Contactos
            SET Nombre = @nombre, Telefono = @telefono, Email = @email
            WHERE Id = @id;
        """;
        AddParameter(command, "@id", c.Id);
        AddParameter(command, "@nombre", c.Nombre);
        AddParameter(command, "@telefono", c.Telefono);
        AddParameter(command, "@email", c.Email);
        command.ExecuteNonQuery();
    }

    public void Eliminar(int id) {
        DbCommand command = this.connection.CreateCommand();
        command.CommandText = "DELETE FROM Contactos WHERE Id = @id;";
        AddParameter(command, "@id", id);
        command.ExecuteNonQuery();
    }

    private static void AddParameter(DbCommand command, string name, object? value) {
        DbParameter p = command.CreateParameter();
        p.ParameterName = name;
        p.Value = value ?? DBNull.Value;
        command.Parameters.Add(p);
    }
}
```

Y el modelo de dominio, una vez:

## 14. Cambiar de motor: la promesa cumplida

Llegó el momento de cobrar lo que la abstracción prometía. ¿Qué hace falta para que este mismo programa use SQL Server en lugar de SQLite?

**Paso 1.** Cambiar el paquete:

```bash
dotnet remove package Microsoft.Data.Sqlite
dotnet add package Microsoft.Data.SqlClient
```

**Paso 2.** Cambiar el `using` en `Program.cs`:

```csharp
// Antes:
using Microsoft.Data.Sqlite;

// Después:
using Microsoft.Data.SqlClient;
```

**Paso 3.** Cambiar la línea del composition root:

```csharp
// Antes:
using DbConnection conn = new SqliteConnection("Data Source=agenda.db");

// Después:
using DbConnection conn = new SqlConnection(
    "Server=localhost;Database=Agenda;Trusted_Connection=True;TrustServerCertificate=True");
```

**Y eso es todo en cuanto al código C#.** La clase `Agenda` no se toca.

Lo que sí puede requerir ajustes es el **SQL** dentro de la clase, dependiendo de qué tan portable haya sido lo que escribimos. En este caso particular hay un punto a revisar: la sintaxis `CREATE TABLE IF NOT EXISTS` no es válida en SQL Server. Y los tipos `INTEGER` y `TEXT` se prefieren escribir como `INT` y `NVARCHAR(...)`. Una versión adaptada para SQL Server sería:

```sql
IF OBJECT_ID('Contactos', 'U') IS NULL
CREATE TABLE Contactos (
    Id       INT PRIMARY KEY,
    Nombre   NVARCHAR(100) NOT NULL,
    Telefono NVARCHAR(50) NOT NULL,
    Email    NVARCHAR(200) NOT NULL
);
```

El resto de las consultas (`SELECT`, `INSERT`, `UPDATE`, `DELETE` parametrizados) funcionan igual.

Esto es importante señalarlo: ADO.NET hace **portable el código C#**, pero **no hace portable el SQL**. Los dialectos siguen siendo distintos. Para una aplicación que de verdad quiera funcionar contra varios motores, hay dos caminos: o se restringe a un subconjunto común de SQL (perdiendo features), o se usa un ORM como Entity Framework Core que conoce los dialectos y traduce automáticamente. Para un proyecto académico o un caso donde el motor está fijado, ADO.NET directo es la opción más simple y didáctica.

A modo de referencia rápida, los proveedores principales:

| Motor           | Paquete NuGet                | Clase de conexión    |
|-----------------|------------------------------|----------------------|
| SQLite          | `Microsoft.Data.Sqlite`      | `SqliteConnection`   |
| SQL Server      | `Microsoft.Data.SqlClient`   | `SqlConnection`      |
| PostgreSQL      | `Npgsql`                     | `NpgsqlConnection`   |
| MySQL / MariaDB | `MySqlConnector`             | `MySqlConnection`    |
| Oracle          | `Oracle.ManagedDataAccess.Core` | `OracleConnection` |

## 15. Inspeccionar la base sin escribir C#

Una herramienta muy útil para aprender SQL es poder inspeccionar el archivo `.db` sin pasar por el programa. Algunas opciones para SQLite:

- **DB Browser for SQLite** (`sqlitebrowser.org`): aplicación gráfica gratuita y multiplataforma. Permite abrir el archivo, ver las tablas, ejecutar SQL directamente y modificar datos a mano.
- **Extensión de VS Code "SQLite Viewer"**: cómoda para echar un vistazo rápido sin salir del editor.
- **El comando `sqlite3`**: la CLI oficial. Si la tenés instalada, `sqlite3 agenda.db` abre una sesión interactiva donde podés tipear SQL y ver resultados.

Tener una de estas a mano cuando uno está aprendiendo es invaluable: cuando el código C# no devuelve lo que esperabas, podés abrir el archivo y comprobar si el problema está en la base o en el código que la consulta.

Para los demás motores existen herramientas equivalentes: **SQL Server Management Studio** o **Azure Data Studio** para SQL Server, **pgAdmin** o **DBeaver** para PostgreSQL, **MySQL Workbench** para MySQL. DBeaver, en particular, es gratuito y soporta todos los motores principales — vale la pena probarlo si vas a trabajar con varios.

## 16. Cierre y siguientes pasos

Lo que cubrimos en este tutorial:

- Por qué una base de datos resuelve problemas que los archivos planos no resuelven.
- Qué es ADO.NET y cómo separa el contrato genérico (clases abstractas) de los proveedores concretos.
- SQL como lenguaje declarativo, dividido en DDL, DML, DQL y TCL.
- Las clases `DbConnection`, `DbCommand`, `DbParameter`, `DbDataReader`, `DbTransaction`.
- Las operaciones CRUD con SQL parametrizado.
- El patrón de transacciones para garantizar atomicidad.
- La importancia absoluta de los parámetros para prevenir inyección SQL y manejar tipos correctamente.
- Cómo el código tipado contra ADO.NET genérico es portable a cualquier motor con un cambio mínimo.

Lo que queda pendiente y van a ver más adelante:

- **Relaciones entre tablas:** claves foráneas, joins. El verdadero poder de SQL aparece cuando hay más de una tabla relacionada.
- **Índices:** estructuras de datos que la base usa para acelerar las búsquedas.
- **Diseño de esquemas:** cómo decidir qué tablas crear, qué columnas, qué relaciones. Es la parte más difícil y la más importante.
- **ORMs (Entity Framework Core):** una capa de abstracción que convierte tablas en objetos, traduce LINQ a SQL automáticamente, y maneja los dialectos de cada motor. Una vez que se entiende ADO.NET, EF Core deja de parecer magia: sigue usando estas mismas clases por debajo, solo que automatiza todo lo que nosotros escribimos a mano.

La regla práctica: aprender SQL crudo y ADO.NET primero, usarlos en proyectos chicos hasta que se vuelvan familiares, y solo entonces adoptar herramientas que los abstraen. Si se hace al revés, el ORM se siente como una caja negra y los problemas se vuelven imposibles de diagnosticar.
