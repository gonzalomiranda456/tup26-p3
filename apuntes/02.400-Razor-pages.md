## 1. El punto de partida: qué hace un servidor web

Antes de Razor, antes de C#, está el hecho crudo: un servidor web recibe **peticiones HTTP** y devuelve **respuestas HTTP**. Una petición es básicamente tres cosas: un **verbo** (GET, POST, etc.), una **URL** (`/contactos?id=5`) y opcionalmente un **cuerpo** (los datos de un formulario). La respuesta es un texto (casi siempre HTML) más algunos metadatos llamados *headers*.

Todo lo que viene después es maquinaria para responder esa pregunta: *"llegó esta petición, ¿qué texto le devuelvo?"*. Razor Pages es una forma de organizar esa maquinaria donde la unidad de trabajo es **la página**.

## 2. Qué es una página Razor

Una página Razor es un archivo `.cshtml` que mezcla HTML con C#. El `.cshtml` se compone de dos lenguajes entrelazados por el carácter `@`: todo lo que no está marcado con `@` es HTML literal que sale tal cual; lo que empieza con `@` es código C# cuyo *resultado* se inserta en el HTML.

```razor
@page

@{
    string nombre = "Ana";
    int hora = DateTime.Now.Hour;
}

<h1>Hola, @nombre</h1>
@if (hora < 12) {
    <p>Buenos días</p>
} else {
    <p>Buenas tardes</p>
}
```

Cuando alguien pide esta página, el servidor *ejecuta* ese código y produce HTML plano:

```html
<h1>Hola, Ana</h1>
<p>Buenas tardes</p>
```

El navegador nunca ve C# ni `@if`: ve el resultado. Esto es lo que se llama **renderizar del lado del servidor**. El primer principio fundamental es ese: **el HTML se arma en el servidor ejecutando código, y al navegador llega solo el texto final.**

La diferencia entre un `.cshtml` cualquiera y una página Razor "de verdad" es la primera línea: `@page`. Esa directiva es la que le dice al framework "este archivo es una página accesible por URL", no un simple fragmento. Sin `@page`, el archivo existe pero no responde a ninguna petición por sí mismo.

## 3. El ruteo: de una URL a un archivo

El **ruteo** (routing) es el mecanismo que decide, dada una URL, qué página la atiende. En Razor Pages la regla base es deliciosamente simple: **la ruta es la ubicación del archivo dentro de la carpeta `Pages/`**.

```
Pages/Index.cshtml          →  /
Pages/Contactos.cshtml      →  /Contactos
Pages/Clientes/Lista.cshtml →  /Clientes/Lista
```

`Index` es un nombre especial: corresponde a la raíz de su carpeta (igual que `index.html` en un sitio estático). Por eso `Pages/Index.cshtml` responde en `/` y no en `/Index`.

No hay un archivo de configuración que diga "la URL `/Contactos` va a tal página". La estructura de carpetas *es* la tabla de rutas. Esto es lo que se llama **routing por convención**: el sistema deduce la regla de la ubicación, en vez de que vos la declares.

> Hay una pieza en `Program.cs` que enciende todo esto: `app.MapRazorPages()`. Esa línea le dice a la app "examiná la carpeta `Pages/` y armá las rutas". Sin ella, ningún `.cshtml` responde.

### Handlers: varias acciones dentro de una misma página

Una URL puede pedirse con distintos verbos, y una misma página puede querer responder a varias acciones. Acá entra la segunda regla del ruteo: dentro de una página, los métodos de C# que responden a peticiones se llaman **handlers**, y su nombre se arma con una fórmula fija:

```
On + Verbo + NombreDelHandler
```

| Petición HTTP                       | Método que se ejecuta   |
| ----------------------------------- | ----------------------- |
| `GET /Contactos`                    | `OnGet()`               |
| `GET /Contactos?handler=Ver&id=5`   | `OnGetVer(int id)`      |
| `POST /Contactos?handler=Guardar`   | `OnPostGuardar(...)`    |

Tres observaciones que aclaran de dónde sale cada parte:

- El **verbo** (`Get`/`Post`) viene de cómo se hizo la petición: un enlace o teclear la URL es GET; el envío de un formulario es POST.
- El **nombre del handler** viene del parámetro especial `handler` en la query string. Sin `?handler=...`, se usa el handler "por defecto": `OnGet` para GET, `OnPost` para POST.
- Si no escribís ningún handler, igual existe un `OnGet` implícito que simplemente renderiza la página. Por eso una página con solo `@page` y HTML ya funciona: tiene un `OnGet` invisible que no hace nada más que mostrar el HTML.

Entonces el ruteo opera en dos niveles: **qué archivo** (por la URL/carpeta) y **qué método dentro de ese archivo** (por el verbo + `handler`).

## 4. Los parámetros y el binding: de texto a objetos C#

Acá viene uno de los conceptos más importantes y más fáciles de subestimar. Una petición HTTP es **puro texto**. Cuando llega `GET /Contactos?id=5&buscar=ana`, el `id` y el `buscar` son cadenas de caracteres dentro de la URL. Pero tu handler de C# quiere trabajar con un `int id` y un `string buscar`, no con texto crudo.

El **model binding** es el proceso que tiende ese puente: mira el nombre de cada parámetro de tu método y busca un valor que se llame igual en la petición, lo convierte al tipo que pediste, y te lo entrega listo.

```csharp
public IActionResult OnGetVer(int id, string buscar) {
    // Razor vio "id" en la URL (id=5), lo convirtió a int → id == 5
    // Razor vio "buscar" en la URL (buscar=ana) → buscar == "ana"
}
```

El binding une por **nombre**, no por posición ni por orden. Si tu parámetro se llama `id`, busca `id`; si se llama `contactoId`, busca `contactoId`. De dónde lo busca:

- **Query string**: `?id=5` (lo que va después del `?` en la URL).
- **Datos de ruta**: si la ruta tiene segmentos con nombre.
- **Cuerpo del formulario**: los campos de un `<form>` que se envió por POST.

El binding también funciona con **objetos**, no solo con valores sueltos. Supongamos un formulario con varios campos:

```html
<input name="Nombre" value="Ana" />
<input name="Apellido" value="Gómez" />
```

Y un handler que recibe un objeto completo:

```csharp
public IActionResult OnPostGuardar(ContactoInput input) { ... }
```

¿Cómo pasa de varios `<input name="...">` a un solo objeto? Cuando el parámetro es una clase o `record`, Razor toma cada propiedad (`Nombre`, `Apellido`, ...) y busca un campo del formulario con ese mismo nombre para llenarla. Por eso los `name` de los `<input>` tienen que coincidir con los nombres de las propiedades del objeto. El binding arma el objeto pieza por pieza.

> El principio acá: **el binding es el traductor entre el mundo de texto del HTTP y el mundo tipado de C#.** Y la regla que lo gobierna es siempre "unir por nombre". Si algo "no llega" a tu handler, el 90% de las veces es porque el nombre del campo HTML no coincide con el nombre del parámetro o de la propiedad.

## 5. Los modelos: las formas que toman tus datos

"Modelo" es una palabra sobrecargada, así que vale la pena separarla en los roles concretos que cumple. En el fondo, un modelo es solo una clase que representa datos; lo que cambia es *qué datos* y *para qué*. Aparecen típicamente tres tipos distintos, y la distinción no es burocracia: cada uno protege una frontera.

**El modelo de dominio / entidad** representa cómo vive una cosa *adentro* de tu aplicación. Suele ser la clase que un ORM (como EF Core) mapea a una tabla de la base. Tiene todo lo que la cosa "es", incluido el `Id` que asigna la base.

```csharp
public class Contacto {
    public int    Id       { get; set; }
    public string Nombre   { get; set; } = "";
    public string Apellido { get; set; } = "";
    // ...
}
```

**El DTO de entrada** (*Data Transfer Object*) representa qué datos *aceptás de afuera*, desde un formulario. Es la frontera entre el mundo externo (que no controlás) y tu dominio (que sí). Se lo separa de la entidad por seguridad y claridad: si bindearas el formulario directo contra la entidad, alguien podría mandar campos que no querés que toque. El DTO declara explícitamente "estos son los únicos campos válidos como entrada".

```csharp
public record ContactoInput(int Id, string Nombre, string Apellido, string Telefono);
```

**El view model** representa qué necesita *una vista* para dibujarse, que a veces es más que una sola entidad. Por ejemplo, una lista puede necesitar los elementos *y* cuál está seleccionado para resaltarlo. Eso no es un dato del dominio (un contacto no "sabe" si está seleccionado); es un dato de la pantalla. El view model los junta.

```csharp
public record ListaViewModel(List<Contacto> Contactos, int? SeleccionadoId);
```

> El principio: **cada modelo protege una frontera distinta.** El dominio es lo que vive adentro; el DTO controla qué entra; el view model arma lo que la pantalla necesita ver. En apps chicas a veces se colapsan en uno solo, pero entender por qué existen por separado te evita acoplar cosas que después cuesta desacoplar.

Un detalle de Razor sobre la palabra "modelo": una página o un partial pueden declarar `@model TipoX` arriba. Eso le dice a Razor "el objeto que me pasan para renderizar es de tipo `TipoX`", y dentro del archivo lo accedés con `Model`. Es el modelo *de esa vista*, no necesariamente una entidad de base de datos: puede ser una entidad, un DTO o un view model, según qué necesite la vista.

## 6. Los formularios: el ciclo completo de captura de datos

Un formulario es la forma en que el usuario *manda* datos al servidor. Ya vimos las dos mitades sueltas —el binding convierte los campos en un objeto (sección 4), y el DTO define qué campos aceptamos (sección 5)—. Acá las juntamos en el ciclo completo, que tiene una estructura muy reconocible una vez que la ves entera: **mostrar el formulario → recibir y validar → guardar o volver a mostrar con errores → redirigir.**

### El formulario es un viaje de ida y vuelta

Lo primero que conviene entender es que un formulario involucra **dos peticiones distintas**, no una:

1. Un **GET** que muestra el formulario vacío (o precargado, si es una edición). El usuario lo ve y lo completa.
2. Un **POST** que envía los datos cuando el usuario hace clic en "Guardar". Acá el servidor los procesa.

Esas dos peticiones caen en dos handlers distintos de la misma página: `OnGet` arma el formulario, `OnPost` lo recibe. Tenelo presente, porque toda la mecánica que sigue gira alrededor de esos dos momentos.

### `[BindProperty]`: el modelo que es de la página

En la sección 4 el handler recibía el objeto como *argumento* (`OnPostGuardar(ContactoInput input)`). Funciona, pero Razor Pages ofrece una forma más cómoda y más usada: declarar una **propiedad** de la página marcada con `[BindProperty]`.

```csharp
public class CrearModel : PageModel {
    [BindProperty]
    public ContactoInput Entrada { get; set; } = new();

    public void OnGet() {
        // El formulario se muestra vacío (Entrada recién creado).
    }

    public IActionResult OnPost() {
        // Al llegar el POST, Razor YA llenó Entrada con los campos del formulario.
        // ...
    }
}
```

La diferencia con recibir el objeto por argumento es de *alcance*: una propiedad `[BindProperty]` vive en la página y está disponible tanto al renderizar (el GET la usa para dibujar) como al recibir (el POST la encuentra llena). Eso es justo lo que necesitás cuando un formulario falla la validación y querés volver a mostrarlo con lo que el usuario ya había escrito: los datos siguen ahí, en la propiedad.

> Por defecto, `[BindProperty]` se llena **solo en POST** (no en GET), que es lo que casi siempre querés. Si necesitás que también se complete en GET —por ejemplo, un filtro que viaja en la URL—, se usa `[BindProperty(SupportsGet = true)]`.

### Los tag helpers: HTML que conoce tu modelo

En la sección 4 escribíamos los `<input>` a mano, cuidando que el `name` coincidiera con la propiedad. Razor ofrece **tag helpers**: atributos especiales que generan ese HTML a partir del modelo, sin que tengas que repetir nombres ni valores.

```razor
@page
@model CrearModel

<form method="post">
    <label asp-for="Entrada.Nombre">Nombre</label>
    <input asp-for="Entrada.Nombre" />
    <span asp-validation-for="Entrada.Nombre"></span>

    <label asp-for="Entrada.Email">Email</label>
    <input asp-for="Entrada.Email" />
    <span asp-validation-for="Entrada.Email"></span>

    <button type="submit">Guardar</button>
</form>
```

El tag helper `asp-for="Entrada.Nombre"` hace tres cosas solo: pone el `name` correcto (`Entrada.Nombre`, para que el binding lo reconozca al volver), pone el `value` actual (para que en una edición el campo aparezca precargado), y deduce atributos de tipo a partir de la propiedad (un `Email` genera `type="email"`, un campo requerido agrega `required`). El `asp-validation-for` es el lugar donde aparecerá el mensaje de error de ese campo, si lo hay.

> El principio: **el tag helper ata el HTML al modelo en un solo lugar.** En vez de coordinar a mano el `name` del input con la propiedad de C# (y arriesgarte a que dejen de coincidir), declarás la propiedad una vez y el helper genera el campo consistente con ella. El binding de vuelta funciona porque los nombres los puso el mismo sistema que los va a leer.

### La validación: `ModelState` y las anotaciones

Antes de guardar, casi siempre hay que validar: que el nombre no esté vacío, que el email tenga forma de email. Razor integra esto con **anotaciones de datos**: atributos que ponés sobre las propiedades del DTO declarando las reglas.

```csharp
public class ContactoInput {
    public int Id { get; set; }

    [Required(ErrorMessage = "El nombre es obligatorio.")]
    public string Nombre { get; set; } = "";

    [Required]
    public string Apellido { get; set; } = "";

    [EmailAddress(ErrorMessage = "El email no tiene un formato válido.")]
    public string Email { get; set; } = "";
}
```

Cuando llega el POST, Razor revisa esas reglas **automáticamente** y deja el resultado en una propiedad llamada `ModelState`. Tu handler solo consulta si todo dio bien:

```csharp
public IActionResult OnPost() {
    if (!ModelState.IsValid) {
        // Algo no validó: volvemos a mostrar el MISMO formulario.
        // Entrada conserva lo que el usuario escribió, y los
        // asp-validation-for muestran los mensajes de error.
        return Page();
    }

    repositorio.Guardar(Entrada);          // todo bien: persistimos
    return RedirectToPage("Index");        // y redirigimos (ver más abajo)
}
```

El patrón es siempre el mismo y vale la pena memorizarlo: **si `ModelState` no es válido, devolvé `Page()` para re-mostrar el formulario con errores; si es válido, procesá y redirigí.** `Page()` vuelve a renderizar la página actual —con la propiedad `[BindProperty]` todavía llena, así el usuario no pierde lo que escribió, y con los mensajes de error visibles en los `asp-validation-for`.

> El principio: **la validación vive en el modelo, no en el handler.** Las reglas se declaran como anotaciones sobre el DTO (un solo lugar, junto a los datos que describen), y Razor las aplica solo. El handler no chequea campo por campo: solo pregunta `ModelState.IsValid` y decide entre re-mostrar o seguir.

### POST-Redirect-GET: por qué un POST exitoso siempre redirige

Mirá que en el ejemplo, tras guardar con éxito, el handler hace `RedirectToPage(...)` en vez de devolver HTML directamente. Esto es deliberado y tiene nombre: el patrón **POST-Redirect-GET** (PRG).

El motivo es un problema concreto del navegador. Si un POST respondiera directamente con la página de "guardado con éxito", esa página quedaría asociada a un POST. Si el usuario después aprieta F5 (refrescar), el navegador **reenvía el POST** —y guardás el contacto otra vez—. Quizás viste alguna vez el cartel "¿Reenviar formulario?": es exactamente esto.

La solución del PRG: el POST no devuelve HTML, sino una **redirección** a una URL. El navegador, al recibir el redirect, hace un **GET** a esa URL. Ahora la página que el usuario ve está asociada a un GET inofensivo: si refresca, repite el GET (que no cambia nada), no el POST.

```
GET  /crear              → muestra el formulario vacío
   │  el usuario completa y envía
POST /crear              → OnPost valida y guarda
   │  return RedirectToPage("Index")   ← NO devuelve HTML, redirige
GET  /                   → el navegador pide la página de destino
   │  el usuario ve el resultado; si refresca, repite este GET inofensivo
```

> El principio: **un POST que modifica datos siempre termina en redirección, no en HTML.** Así el resultado queda bajo una URL "de solo lectura" que se puede refrescar sin repetir el cambio. La regla mental: *GET muestra, POST modifica y redirige a un GET.*

### El ciclo completo de un formulario

Juntando todo, un formulario de Razor Pages recorre siempre estos pasos:

```
1. GET  → OnGet arma el formulario ([BindProperty] vacío o precargado)
          los tag helpers (asp-for) generan los campos atados al modelo
2. El usuario completa y envía
3. POST → Razor llena la propiedad [BindProperty] con los campos (binding)
          y aplica las anotaciones de validación → ModelState
4. ¿ModelState.IsValid?
     NO → return Page()          (re-muestra el form con datos y errores)
     SÍ → procesar + RedirectToPage(...)   (PRG: redirige a un GET)
```

Cada pieza que ya conocías encuentra su lugar acá: el binding (paso 3), el DTO y sus anotaciones (paso 3-4), los modelos (la propiedad que viaja), y los handlers `OnGet`/`OnPost` (los dos extremos del viaje). El formulario no es un tema aparte: es la coreografía de todo lo anterior alrededor de capturar datos del usuario.

## 7. El layout: el marco compartido

Cada página produce un fragmento de HTML (su `<h1>`, sus formularios). Pero una página HTML completa necesita siempre lo mismo alrededor: `<!DOCTYPE html>`, el `<head>` con los estilos, las etiquetas `<body>`. Repetir eso en cada página sería absurdo.

El **layout** es una plantilla que contiene todo ese envoltorio común una sola vez, con un hueco donde se inserta el contenido de cada página:

```razor
<!DOCTYPE html>
<html lang="es">
<head>
    <title>@ViewData["Title"]</title>
    <link rel="stylesheet" href="/css/site.css" />
</head>
<body>
    <main>
        @RenderBody()
    </main>
</body>
</html>
```

`@RenderBody()` es el hueco: ahí Razor mete el HTML que generó la página individual. Cuando una página dice `Layout = "_Layout";`, le está pidiendo a Razor "envolveme con esa plantilla". El flujo mental es: la página produce su fragmento, el layout lo recibe en `@RenderBody()`, y juntos forman el documento HTML completo que se envía.

`@ViewData["Title"]` es un pequeño canal de comunicación de la página hacia el layout: la página escribe `ViewData["Title"] = "Mi título"` y el layout lo lee para poner el título. Es un diccionario compartido para esos datos sueltos que la página quiere pasarle al marco.

> El principio: **el layout invierte la relación.** Uno tiende a pensar "la página contiene el HTML"; en realidad el layout contiene el documento y la página le aporta el relleno del medio. Por eso los estilos y scripts comunes viven en el layout: están una vez y aplican a todas las páginas.

## 8. Los partials: fragmentos reutilizables

Si el layout es el marco grande compartido entre páginas, los **partials** son piezas chicas reutilizables *dentro* de las páginas. Un partial es un `.cshtml` (por convención con nombre que empieza en `_`, como `_Detalle.cshtml`) que renderiza un trozo de HTML y normalmente recibe un modelo.

```razor
@* _Detalle.cshtml *@
@model Contacto

<h2>@Model.Nombre @Model.Apellido</h2>
<p>@Model.Telefono</p>
```

Lo usás de dos formas, y la distinción es central:

**Desde otra vista**, con la etiqueta `<partial>`:

```razor
<partial name="_Detalle" model="contacto" />
```

Razor renderiza ese partial pasándole `contacto` como su `Model`, y pega el resultado donde está la etiqueta. Así componés una pantalla a partir de fragmentos.

**Desde un handler**, con `Partial(...)`:

```csharp
public IActionResult OnGetDetalle(int id) {
    Contacto? contacto = repositorio.ObtenerPorId(id);
    return Partial("_Detalle", contacto);
}
```

La diferencia clave: un handler normal renderiza una página entera (con su layout). Pero `Partial(...)` devuelve **solo el fragmento**, sin layout, sin `<html>` ni `<head>`. Es exactamente un pedacito de HTML aislado.

> El principio: **el mismo partial sirve para componer una página completa (vía `<partial>`) y para responder un fragmento suelto (vía `Partial(...)`).** Esa dualidad permite que la presentación viva en un solo lugar y se reutilice en distintos contextos sin duplicar HTML.

## 9. La inyección de dependencias: cómo las piezas se encuentran

Queda una pregunta natural: si un handler usa, por ejemplo, un repositorio de datos, pero nunca escribís `new Repositorio(...)`, ¿de dónde sale esa instancia?

Acá entra la **inyección de dependencias** (DI). El primer principio es separar dos roles: hay objetos que *necesitan* colaboradores para funcionar (un handler necesita un repositorio; el repositorio necesita una conexión a la base), y hay un mecanismo central —el **contenedor** de DI— que se encarga de *crearlos y entregárselos*. En vez de que cada objeto fabrique sus propias dependencias, las *pide* y el contenedor se las da. De ahí "inyección": las dependencias se inyectan desde afuera.

El proceso tiene dos mitades. Primero, en `Program.cs`, **registrás** qué servicios existen y cómo crearlos:

```csharp
builder.Services.AddRazorPages();
builder.Services.AddScoped<Repositorio>();
```

Cada `Add...` le dice al contenedor "cuando alguien necesite un `Repositorio`, sabé cómo dárselo". Después, **consumís** ese servicio pidiéndolo donde lo necesitás. En una página Razor se pide con `@inject`:

```razor
@inject Repositorio Repositorio
```

Esa línea le dice a Razor "necesito un `Repositorio`, conseguímelo del contenedor". Razor lo busca, lo crea (con sus propias dependencias resueltas en cadena: si el repositorio necesita una conexión, el contenedor la crea primero), y te lo deja disponible como una variable que tus handlers pueden usar.

El `Add...` también define el **ciclo de vida** —cuánto vive cada instancia—, y esto importa:

- `AddScoped`: una instancia por petición HTTP. Apropiado para objetos que no guardan estado entre peticiones; uno fresco por request es lo natural.
- `AddSingleton`: una sola instancia para toda la vida de la aplicación, compartida por todas las peticiones. Apropiado para servicios sin estado mutable o con estado seguro de compartir.
- `AddTransient`: una instancia nueva cada vez que se pide. Apropiado para objetos livianos y sin estado.

La elección depende de si el objeto tiene estado y de si es seguro compartirlo. Elegir mal acá es una fuente clásica de errores sutiles (por ejemplo, compartir como singleton un objeto que asume ser de una sola petición).

> El principio: **DI invierte quién manda en la creación de objetos.** Sin DI, cada objeto fabrica y conoce a sus colaboradores, lo que los amarra entre sí. Con DI, los objetos solo declaran qué necesitan, y un mecanismo externo decide cómo construirlo. Eso hace que puedas cambiar la implementación (por ejemplo, una versión falsa para tests) sin tocar el código que la usa.

## 10. Cómo encaja todo en una sola petición

Cerremos con el recorrido completo de una acción, que ata los conceptos. El usuario hace clic en "ver contacto 5":

1. **Ruteo (URL → archivo).** Llega `GET /?handler=Detalle&id=5`. La URL `/` mapea a `Pages/Index.cshtml`.
2. **Ruteo (verbo + handler → método).** Verbo GET + `handler=Detalle` → se ejecuta `OnGetDetalle`.
3. **Binding.** `id=5` (texto en la URL) se convierte al `int id` del método.
4. **Inyección.** El handler usa el repositorio que la página recibió por `@inject`, ya construido por el contenedor.
5. **Modelo.** `repositorio.ObtenerPorId(5)` devuelve un `Contacto` (entidad de dominio).
6. **Partial.** El handler hace `return Partial("_Detalle", contacto)`: renderiza el fragmento con ese modelo.
7. **Respuesta.** Ese HTML vuelve al navegador, que lo muestra.

Cada concepto es una regla chica y predecible. La sensación de "magia" de Razor desaparece cuando ves que es siempre la misma cadena: **una URL elige un método por su nombre, el binding le llena los argumentos, la inyección le presta los colaboradores, el método produce un modelo, y un partial (o la página) lo convierte en HTML.**

## Resumen de los principios

| Concepto    | Regla en una frase                                                              |
| ----------- | ------------------------------------------------------------------------------- |
| Página      | El HTML se arma en el servidor ejecutando código; al navegador llega el texto final. |
| Ruteo       | La URL elige el archivo (por carpeta) y el verbo + `handler` eligen el método.  |
| Binding     | Los datos de texto del HTTP se asignan a los argumentos C# uniendo por nombre.  |
| Modelos     | Cada modelo protege una frontera: dominio (adentro), DTO (entrada), view model (pantalla). |
| Formularios | GET muestra el form (tag helpers atan el HTML al modelo); POST valida con `ModelState` y redirige (PRG). |
| Layout      | Una plantilla común contiene el documento; la página aporta el relleno en `@RenderBody()`. |
| Partials    | Fragmentos reutilizables; sirven para componer páginas y para devolver HTML suelto. |
| Inyección   | Los objetos declaran qué necesitan; un contenedor los crea y se los entrega.    |
