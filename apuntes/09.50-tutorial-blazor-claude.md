# Blazor en .NET 10 / C# 14 — Tutorial completo en español argentino

## TL;DR
- **Blazor es el framework de Microsoft para construir interfaces web interactivas usando C# en lugar de JavaScript**, y en .NET 10 unificó sus dos modelos históricos (Server y WebAssembly) bajo el template **Blazor Web App**, que permite elegir el modo de renderizado por componente con la directiva `@rendermode`.
- Para un curso de segundo año conviene arrancar con **Interactive Server** (`dotnet new blazor -int Server`): tiene tiempo de carga mínimo, debugging idéntico al de cualquier app .NET y oculta el código C# al navegador.
- En este tutorial vas a aprender la teoría (componentes Razor, ciclo de vida, binding, EventCallback, DI, ruteo, formularios) y vas a construir una **agenda de contactos CRUD con lista en memoria** paso a paso.

## Parte 1 — Conceptual

### 1.1 ¿Qué es Blazor y para qué sirve?

Blazor es un framework de UI web que forma parte de ASP.NET Core. La documentación oficial lo define así:

> *"Blazor is a .NET frontend web framework that supports both server-side rendering and client interactivity in a single programming model."* — Microsoft Learn

En criollo: Blazor te permite escribir **toda la app (UI + lógica)** en C# y Razor, y compartir clases, DTOs y validaciones entre cliente y servidor. La página final que ve el navegador sigue siendo HTML + CSS estándar; lo que cambia es **quién** la genera y **dónde** corre el código C# que reacciona a los clics.

¿Por qué te debería interesar como estudiante?

1. **Reutilizás conocimiento de C#/.NET** (POO, LINQ, async/await, colecciones) en el frontend; no hace falta aprender JavaScript de cero.
2. **Tipado fuerte** end-to-end: si renombrás una propiedad del modelo, el compilador te avisa en la vista.
3. **Modelo de componentes** muy parecido al de React/Vue/Angular, pero con C# y sin Node.

> ⚠️ Blazor *no reemplaza* a JavaScript en el navegador: convive con él vía interop (`IJSRuntime`). Para tareas chicas tipo "scroll a un elemento" o "leer del localStorage" todavía necesitás JS.

### 1.2 Modelos de hosting: Server vs WebAssembly

Históricamente Blazor tuvo dos formas de correr el código C# que maneja la UI:

#### Blazor Server (Interactive Server)
El C# corre **en el servidor**. El navegador mantiene una conexión SignalR (WebSocket) abierta; cuando el usuario hace clic, el evento se envía al servidor, éste corre el handler, calcula el **diff** del DOM y manda solo los cambios.

- ✅ **Descarga inicial muy chica**: en .NET 10 el único script de framework, `blazor.web.js`, pesa 43 KB (Telerik / Jon Hilton, ".NET 10 Has Arrived—Here's What's Changed for Blazor", 13-nov-2025: *"blazor.web.js drops from 183 KB to 43 KB (76% size reduction)"*) y **arranque casi instantáneo**.
- ✅ Acceso directo a recursos de servidor (DB, sistema de archivos, secrets) sin exponer nada al cliente.
- ✅ Debugging idéntico a cualquier app .NET (breakpoints en Visual Studio, logs en consola).
- ❌ **Requiere conexión permanente**. Si se cae el WebSocket, la UI se queda colgada (en .NET 10 se mitigó con persistencia de circuito, pero el problema base sigue).
- ❌ **Latencia**: cada interacción es un round-trip de red.
- ❌ **Escalabilidad**: cada usuario activo consume memoria del servidor para mantener su "circuit".

#### Blazor WebAssembly (Interactive WebAssembly)
El **runtime de .NET completo se compila a WebAssembly** y se descarga al navegador, junto con tus DLLs. Una vez cargado, todo el C# corre en la máquina del usuario.

- ✅ **App offline-capable** (después de la carga inicial puede funcionar sin internet).
- ✅ Escalabilidad enorme: el servidor solo sirve archivos estáticos o una Web API.
- ✅ Rendimiento de UI superior una vez cargada (sin latencia de red por cada clic).
- ❌ **Descarga inicial pesada** (varios MB con el runtime; en .NET 10 mejoró con preloading y compresión).
- ❌ **Sin acceso directo a DB**: necesitás una Web API intermedia.
- ❌ El código C# **se descarga al cliente** — no es buen lugar para secretos ni lógica de negocio sensible.

#### .NET 8+: Render modes y el template "Blazor Web App"
Desde .NET 8 Microsoft unificó los dos mundos. Ahora hay un único template (`dotnet new blazor`) que puede elegir el modo de renderizado **por componente** mediante la directiva `@rendermode`:

| Render mode | Dónde corre | Cuándo usarlo |
|---|---|---|
| **Static SSR** | Servidor, una sola vez | Páginas marketing/landing, contenido SEO, sin interactividad |
| **InteractiveServer** | Servidor + SignalR | Apps internas, dashboards, lo que antes era "Blazor Server" |
| **InteractiveWebAssembly** | Navegador (WASM) | SPAs offline, alta interactividad cliente |
| **InteractiveAuto** | Empieza Server, después WASM | Lo mejor de los dos: arranque rápido + cliente luego |

#### Decisión para este curso
Vamos a usar **InteractiveServer**. Razones pedagógicas:
- Setup más simple (un solo proyecto, sin `.Client`).
- Debugging directo desde Visual Studio / Rider / VS Code.
- No hay que armar Web API aparte para acceder a la "base" (la lista en memoria).
- Los conceptos de componentes/binding/lifecycle son **idénticos** a WASM, así que lo aprendido transfiere.

### 1.3 Blazor en el ecosistema .NET 10 / C# 14

.NET 10 (lanzado el 11 de noviembre de 2025) es una versión **LTS** con soporte hasta el **10 de noviembre de 2028**, según el blog oficial de Microsoft (.NET Blog, "Announcing .NET 10", 11-nov-2025: *".NET 10 is a Long Term Support (LTS) release and will be supported for three years, until November 10, 2028"*). Es el target recomendado para producción ahora.

Lo más relevante para Blazor en .NET 10:

- **`[PersistentState]`** — atributo nuevo (clase `PersistentStateAttribute`) que persiste estado entre el prerender y el render interactivo, eliminando el "doble fetch" de datos. La documentación oficial lo describe así: *"To preserve prerendered state, use the `[PersistentState]` attribute to persist state in properties. Properties with this attribute are automatically persisted using the `PersistentComponentState` service during prerendering."*
- **Reducción del 76% en `blazor.web.js`** — de 183 KB a 43 KB gracias a compresión y fingerprinting automáticos (Telerik / Jon Hilton, ".NET 10 Has Arrived—Here's What's Changed for Blazor", 13-nov-2025: *"the compression takes the blazor.web.js script down by about 76%"*).
- **`<ReconnectModal />`** — componente built-in en el template que reemplaza el viejo UI de reconexión y respeta políticas de CSP estrictas.
- **`NotFound.razor`** + parámetro `Router.NotFoundPage` para manejar 404 de forma uniforme.
- **`NavigationManager.NotFound()`** método nuevo para disparar 404 desde código.
- **Validación con `[ValidatableType]`** (experimental) y `AddValidation()` para formularios anidados.
- **Hot Reload habilitado por defecto para builds Debug de Blazor WebAssembly** (Telerik / Jon Hilton, 13-nov-2025: *"Hot Reload is now enabled by default for Blazor WebAssembly Debug builds. Edit a .razor file, save, and see changes instantly without rebuilding or refreshing the browser."*).

Del lado de **C# 14**, lo que más vamos a usar en este tutorial:

- **`field` keyword** — accede al backing field auto-generado dentro de un property accessor, sin declarar el campo a mano:
  ```csharp
  public string Nombre {
      get;
      set => field = value?.Trim() ?? throw new ArgumentNullException(nameof(value));
  }
  ```
- **Null-conditional assignment** — `cliente?.Pedido = ...` ahora compila.
- **`nameof` con generics no construidos** — `nameof(List<>)` devuelve `"List"`.

> 📌 **Estilo de código del tutorial**: usamos llaves K&R (la `{` va en la misma línea que la declaración), y para acceder a campos de instancia escribimos `this.campo` siempre. Es una convención para que el código quede prolijo y para distinguir visualmente lo que es estado del objeto.

### 1.4 Componentes Razor

Un **componente Razor** es la unidad de composición de Blazor. Vive en un archivo `.razor` y es, debajo, una clase de C# que hereda de `ComponentBase`. La documentación oficial lo expresa así: *"Components are .NET C# classes built into .NET assemblies that: Define flexible UI rendering logic. Handle user events. Can be nested and reused."*

#### Estructura básica de un `.razor`

```razor
@* Saludo.razor *@
@page "/saludo"
@using MiApp.Modelos
@inject ILogger<Saludo> Logger

<h3>Hola, @this.nombre</h3>
<button @onclick="Cambiar">Cambiar</button>

@code {
    private string nombre = "Mundo";

    private void Cambiar() {
        this.nombre = "Blazor";
        this.Logger.LogInformation("Nombre cambiado");
    }
}
```

Partes:

1. **Directivas** (`@page`, `@using`, `@inject`, etc.) — siempre arriba.
2. **Markup HTML + Razor** — el cuerpo, con expresiones `@variable` y atributos `@onclick`, `@bind`...
3. **Bloque `@code { ... }`** — el "code-behind" inline: campos, propiedades, métodos.

#### Directivas más usadas

| Directiva | Para qué sirve |
|---|---|
| `@page "/ruta"` | Convierte al componente en una página accesible vía URL |
| `@using Namespace` | Igual que `using` en C# |
| `@inject Tipo nombre` | Inyecta una dependencia |
| `@code { ... }` | Bloque de código C# del componente |
| `@bind`, `@bind-Value` | Data binding bidireccional |
| `@onclick`, `@onchange` | Handlers de eventos del DOM |
| `@ref` | Capturar referencia al elemento o componente |
| `@rendermode` | Especificar render mode (Interactive Server / WebAssembly / Auto) |
| `@layout` | Override del layout para esta página |
| `@attribute [Authorize]` | Aplicar un atributo a la clase generada |

### 1.5 Ciclo de vida de un componente

**El "por qué" antes de la sintaxis**: cuando Blazor renderiza un componente no lo hace todo de una vez. Hay momentos diferenciados (creación, recepción de parámetros, render del DOM real) y necesitás engancharte al correcto. Si cargás datos en el momento equivocado, vas a tener bugs raros tipo "se duplica la llamada al servicio" o "el `@ref` es null".

El flujo, simplificado, es:

```
SetParametersAsync(params)
        │
        ▼
OnInitialized / OnInitializedAsync     ◄── solo la PRIMERA vez
        │
        ▼
OnParametersSet / OnParametersSetAsync  ◄── primera vez y cada vez que cambian parámetros
        │
        ▼
[RENDER al DOM]
        │
        ▼
OnAfterRender(firstRender) / OnAfterRenderAsync
        │
        ▼ (esperando eventos del usuario)
```

Cuándo overridear cada uno:

- **`OnInitialized` / `OnInitializedAsync`** → inicialización única (cargar dropdowns estáticos, subscribirse a un evento global). **Se llama UNA sola vez por instancia del componente**.
- **`OnParametersSet` / `OnParametersSetAsync`** → reaccionar a parámetros que cambian (ej. el padre te pasa un `Id` diferente y tenés que recargar el detalle).
- **`OnAfterRender(bool firstRender)` / `OnAfterRenderAsync`** → cosas que necesitan el DOM ya construido: `@ref`, interop con JavaScript, foco a un input. La sobrecarga toma un `bool firstRender` que es `true` solo la primera vez.
- **`Dispose()` (si implementás `IDisposable`)** → liberar timers, unsubscribirse de eventos.

> ⚠️ **Trampa clásica con prerendering**: si tu app tiene prerender activado (es el default), `OnInitializedAsync` se llama **dos veces**: una en el servidor durante el prerender, otra cuando arranca el modo interactivo. Para evitar duplicar fetches, usá `[PersistentState]` en .NET 10.

```csharp
@code {
    [PersistentState]
    public List<Contacto>? Contactos { get; set; }

    protected override async Task OnInitializedAsync() {
        // Esto solo se ejecuta si Contactos viene null
        this.Contactos ??= await this.ContactoService.ListarAsync();
    }
}
```

### 1.6 Data binding

Blazor distingue tres patrones:

#### Unidireccional (one-way): C# → UI
```razor
<p>Total: @this.total</p>
```
Cualquier cambio de `this.total` que dispare un re-render se refleja. La UI no puede modificarlo desde acá.

#### Bidireccional (two-way) con `@bind`
```razor
<input @bind="this.nombre" />
<p>Hola @this.nombre</p>
```
`@bind="campo"` es azúcar para `value="@campo" @onchange="e => campo = e.Value"`. Por defecto se actualiza con `onchange` (cuando el input pierde foco). Si querés que se actualice en cada tecla:
```razor
<input @bind="this.nombre" @bind:event="oninput" />
```

#### `@bind:after` (.NET 7+) para correr lógica después
```razor
<input @bind="this.filtro" @bind:after="Buscar" />
```

#### Binding a componentes hijos
Si tu componente expone una propiedad `[Parameter] public string Texto { get; set; }`, para que sea bindeable bidireccionalmente tiene que también exponer `[Parameter] public EventCallback<string> TextoChanged { get; set; }` (la convención de nombre `<Prop>Changed` es obligatoria). Después el padre puede hacer:
```razor
<MiInput @bind-Texto="this.busqueda" />
```

#### Eventos DOM
```razor
<button @onclick="Guardar">Guardar</button>
<input @onkeydown="OnTecla" />
```
Los handlers pueden ser `void`, `async Task`, recibir o no un `EventArgs` (`MouseEventArgs`, `KeyboardEventArgs`, etc.).

### 1.7 Parámetros entre componentes

#### Padre → Hijo: `[Parameter]`
```razor
@* TarjetaContacto.razor *@
<div class="card">
    <h5>@this.Contacto.Nombre</h5>
    <p>@this.Contacto.Email</p>
</div>

@code {
    [Parameter, EditorRequired]
    public Contacto Contacto { get; set; } = default!;
}
```
Uso desde el padre:
```razor
<TarjetaContacto Contacto="@this.contactoActual" />
```

#### Hijo → Padre: `EventCallback<T>`
El hijo expone un callback como parámetro, y lo invoca cuando algo pasa. Blazor llama automáticamente a `StateHasChanged()` en el padre cuando el callback termina, así que la UI se actualiza sola.

```razor
@* BotonEliminar.razor *@
<button class="btn btn-danger" @onclick="OnClick">🗑</button>

@code {
    [Parameter] public int Id { get; set; }
    [Parameter] public EventCallback<int> OnEliminar { get; set; }

    private Task OnClick() => this.OnEliminar.InvokeAsync(this.Id);
}
```
Padre:
```razor
<BotonEliminar Id="@c.Id" OnEliminar="Eliminar" />

@code {
    private void Eliminar(int id) { /* ... */ }
}
```

> 💡 **Por qué EventCallback y no `Action`/`Func`**: a diferencia de un delegate normal, `EventCallback` está atado al renderer del componente que lo recibió, así que dispara el re-render automático del padre **sin** tener que llamar `StateHasChanged()` a mano.

### 1.8 Inyección de dependencias

Blazor usa el mismo contenedor de DI que ASP.NET Core. Registrás servicios en `Program.cs`:

```csharp
builder.Services.AddSingleton<IContactoService, ContactoService>();
builder.Services.AddScoped<UsuarioActual>();
builder.Services.AddTransient<IGeneradorDeId, GeneradorDeId>();
```

Los tres lifetimes:

| Lifetime | Blazor Server | Blazor WASM |
|---|---|---|
| **Singleton** | Una sola instancia para toda la app, compartida entre usuarios | Una instancia por sesión de navegador |
| **Scoped** | Una instancia **por circuito** (≈ por usuario conectado) | Equivalente a Singleton |
| **Transient** | Una nueva instancia por cada `@inject` | Igual |

> ⚠️ **Trampa importante**: en Blazor Server, `Scoped` **no es por petición HTTP** como en MVC; es **por circuito SignalR**, o sea por sesión de usuario. Si guardás estado en un servicio Scoped, ese estado dura mientras el usuario tenga la pestaña abierta y conectada.

Para nuestro CRUD en memoria conviene **Singleton** (queremos que todos los usuarios vean la misma agenda).

#### Cómo se inyecta en un componente

Dos formas equivalentes:

```razor
@inject IContactoService Contactos
```
o en `@code`:
```csharp
[Inject]
private IContactoService Contactos { get; set; } = default!;
```

> 📝 Atajo argentino: en componentes solemos usar la primera porque queda más compacta. La segunda es útil cuando tenés code-behind separado.

### 1.9 Routing

#### Definir rutas

Cada página se hace ruteable con `@page`:

```razor
@page "/contactos"
@page "/agenda"      @* podés tener varias rutas para la misma página *@
```

#### Parámetros de ruta con tipo

```razor
@page "/contactos/{Id:int}"

@code {
    [Parameter] public int Id { get; set; }
}
```

Los **constraints** soportados según la documentación oficial de routing en .NET 10 son: `bool`, `datetime`, `decimal`, `double`, `float`, `guid`, `int`, `long`, `nonfile`. La doc aclara: *"Route constraints that verify the URL and are converted to a CLR type (such as `int` or `System.DateTime`) always use the invariant culture."*

#### Parámetros opcionales y query strings

```razor
@page "/buscar"

@code {
    [SupplyParameterFromQuery]
    public string? Termino { get; set; }

    [SupplyParameterFromQuery(Name = "pagina")]
    public int Pagina { get; set; } = 1;
}
```

#### Navegación programática

Se inyecta `NavigationManager`:

```csharp
@inject NavigationManager Nav

private void IrAEditar(int id) {
    this.Nav.NavigateTo($"/contactos/editar/{id}");
}
```

#### `NavLink` vs `<a>`

`<NavLink>` agrega automáticamente la clase CSS `active` cuando la URL coincide. Es preferible para menús de navegación:

```razor
<NavLink href="/contactos" Match="NavLinkMatch.All">Contactos</NavLink>
```

#### Página 404 en .NET 10

La doc oficial muestra el patrón:
```razor
@page "/not-found"
@layout MainLayout

<h3>Not Found</h3>
<p>Sorry, the content you are looking for does not exist.</p>
```
Y en `Routes.razor`:
```razor
<Router AppAssembly="@typeof(Program).Assembly"
        NotFoundPage="typeof(Pages.NotFound)">
    <Found Context="routeData">
        <RouteView RouteData="@routeData" />
        <FocusOnNavigate RouteData="@routeData" Selector="h1" />
    </Found>
</Router>
```

### 1.10 Formularios y validación

Blazor incluye un componente `EditForm` que envuelve un `<form>` HTML y le agrega un `EditContext` que trackea cambios y validaciones.

#### Componentes input built-in
La doc oficial lista estos wrappers (todos heredan de `InputBase<T>`):
- `InputText` → `<input>`
- `InputTextArea` → `<textarea>`
- `InputNumber<T>` → `<input type="number">`
- `InputCheckbox` → `<input type="checkbox">`
- `InputDate<T>` → `<input type="date">`
- `InputSelect<T>` → `<select>`
- `InputRadio<T>` / `InputRadioGroup<T>` → radio buttons
- `InputFile` → upload de archivos

#### Validación con DataAnnotations

```csharp
public class Contacto {
    public int Id { get; set; }

    [Required(ErrorMessage = "El nombre es obligatorio")]
    [StringLength(80, MinimumLength = 2)]
    public string Nombre { get; set; } = "";

    [Required, EmailAddress]
    public string Email { get; set; } = "";

    [Phone]
    public string? Telefono { get; set; }
}
```

Y la vista:

```razor
<EditForm Model="this.modelo" OnValidSubmit="Guardar" FormName="ContactoForm">
    <DataAnnotationsValidator />
    <ValidationSummary />

    <div class="mb-3">
        <label>Nombre</label>
        <InputText @bind-Value="this.modelo.Nombre" class="form-control" />
        <ValidationMessage For="@(() => this.modelo.Nombre)" />
    </div>

    <button type="submit" class="btn btn-primary">Guardar</button>
</EditForm>
```

Los tres callbacks de `EditForm`:
- `OnValidSubmit` → solo si la validación pasa.
- `OnInvalidSubmit` → solo si hay errores.
- `OnSubmit` → siempre, vos manejás la validación a mano.

## Parte 2 — Proyecto práctico: Agenda de contactos CRUD

### 2.1 Lo que vamos a construir

Una pequeña aplicación de gestión de contactos con:

- Lista paginada de contactos.
- Buscador por nombre.
- Alta de contacto (formulario validado).
- Edición de contacto existente.
- Eliminación con confirmación.
- Datos persistidos en una `List<Contacto>` en memoria (singleton) — sin EF, sin DB.

### 2.2 Crear el proyecto

Abrí una terminal en la carpeta donde querés trabajar y ejecutá:

```bash
dotnet new blazor -o Agenda -int Server
cd Agenda
dotnet run
```

#### ¿Qué hicieron esos comandos?

Según la documentación oficial de `dotnet new`, el template `blazor` acepta estas opciones:

- **`-int|--interactivity <None|Server|WebAssembly|Auto>`** — *"Specifies which interactive render mode to use for interactive components"*. Le dimos `Server`.
- **`-ai|--all-interactive`** — *"Makes every page interactive by applying an interactive render mode at the top level. If false, pages use static server-side rendering by default and can be marked interactive on a per-page or per-component basis."* No lo usamos: vamos a marcar la interactividad página por página.
- **`--empty`** — *"Omits sample pages and styling that demonstrate basic usage patterns"*. No lo usamos para tener el layout y CSS de arranque.
- **`-au|--auth <None|Individual>`** — autenticación (None por defecto).
- **`--use-program-main`** — *"If specified, an explicit Program class and Main method is generated instead of top-level statements"*. No lo usamos: top-level statements queda más limpio.

```
Agenda/
├── Components/
│   ├── App.razor              ← html root (<html>, <head>, <body>)
│   ├── Routes.razor           ← <Router> con la tabla de rutas
```

### 2.4 Revisión de `Program.cs`

El archivo generado por `dotnet new blazor -int Server` en .NET 10 contiene las siguientes llamadas clave (la doc oficial las describe así: *"Services for Razor components are added by calling `AddRazorComponents`. `AddInteractiveServerComponents` adds services to support rendering Interactive Server components."*):

```csharp
var builder = WebApplication.CreateBuilder(args);

// 1. Registrar Blazor + servicios de interactividad
builder.Services.AddRazorComponents()
    .AddInteractiveServerComponents();

// 2. ⬇⬇⬇ ACÁ vamos a registrar nuestro servicio
// builder.Services.AddSingleton<IContactoService, ContactoServiceEnMemoria>();

var app = builder.Build();

if (!app.Environment.IsDevelopment()) {
    app.UseExceptionHandler("/Error", createScopeForErrors: true);
    app.UseHsts();
}

app.UseHttpsRedirection();
app.UseAntiforgery();     // protección CSRF para formularios
app.MapStaticAssets();    // sirve wwwroot/ con fingerprinting

app.MapRazorComponents<App>()
    .AddInteractiveServerRenderMode();

app.Run();
```

### 2.5 Modelo de dominio

Creá una carpeta `Modelos/` y dentro `Contacto.cs`:

```csharp
using System.ComponentModel.DataAnnotations;

namespace Agenda.Modelos;

public class Contacto {
    public int Id { get; set; }

    [Required(ErrorMessage = "El nombre es obligatorio.")]
    [StringLength(80, MinimumLength = 2,
        ErrorMessage = "El nombre debe tener entre 2 y 80 caracteres.")]
    public string Nombre {
        get;
        // C# 14: usamos 'field' para acceder al backing field generado
        // y normalizar el valor (trim) antes de guardarlo.
        set => field = (value ?? "").Trim();
    } = "";

    [Required, EmailAddress(ErrorMessage = "Email inválido.")]
    public string Email { get; set; } = "";

    [Phone(ErrorMessage = "Teléfono inválido.")]
    public string? Telefono { get; set; }

    [DataType(DataType.Date)]
    public DateOnly? Cumpleanios { get; set; }

    public Categoria Categoria { get; set; } = Categoria.Personal;
}

public enum Categoria { Personal, Trabajo, Familia, Otro }
```

> Notá el uso del **keyword `field` de C# 14**: nos permite ejecutar lógica en el setter (un `Trim()`) sin tener que declarar a mano un `private string _nombre;`. La sintaxis previa era más verbosa.

### 2.6 Servicio en memoria con inyección de dependencias

Creá `Servicios/IContactoService.cs`:


Implementación `Servicios/ContactoServiceEnMemoria.cs`:

```csharp
using Agenda.Modelos;

namespace Agenda.Servicios;

public class ContactoServiceEnMemoria : IContactoService {
    // Usamos this. para acceder a campos de instancia, como pide el estilo
    // del curso.
    private readonly List<Contacto> contactos;
    private int proximoId;
    private readonly Lock candado = new(); // sincronización: varios usuarios

    public ContactoServiceEnMemoria() {
        // Sembramos algunos contactos de ejemplo.
        this.contactos = [
            new Contacto { Id = 1, Nombre = "Diego Maradona",
                Email = "d10@argentina.ar", Categoria = Categoria.Familia },
            new Contacto { Id = 2, Nombre = "Lionel Messi",
                Email = "messi@psg.fr", Categoria = Categoria.Trabajo },
            new Contacto { Id = 3, Nombre = "Charly García",
                Email = "charly@sui.generis", Categoria = Categoria.Personal },
        ];
        this.proximoId = this.contactos.Count + 1;
    }

    public IReadOnlyList<Contacto> Listar(string? filtro = null) {
        lock (this.candado) {
            if (string.IsNullOrWhiteSpace(filtro)) {
                return this.contactos
                    .OrderBy(c => c.Nombre)
                    .ToList();
            }
            // LINQ: ya lo conocen del año pasado
            return this.contactos
                .Where(c => c.Nombre.Contains(filtro,
                    StringComparison.OrdinalIgnoreCase))
                .OrderBy(c => c.Nombre)
                .ToList();
        }
    }

    public Contacto? ObtenerPorId(int id) {
        lock (this.candado) {
            return this.contactos.FirstOrDefault(c => c.Id == id);
        }
    }

    public Contacto Crear(Contacto nuevo) {
        lock (this.candado) {
            nuevo.Id = this.proximoId++;
            this.contactos.Add(nuevo);
            return nuevo;
        }
    }

    public bool Actualizar(Contacto modificado) {
        lock (this.candado) {
            var existente = this.contactos.FirstOrDefault(
                c => c.Id == modificado.Id);
            if (existente is null) {
                return false;
            }
            existente.Nombre = modificado.Nombre;
            existente.Email = modificado.Email;
            existente.Telefono = modificado.Telefono;
            existente.Cumpleanios = modificado.Cumpleanios;
            existente.Categoria = modificado.Categoria;
            return true;
        }
    }

    public bool Eliminar(int id) {
        lock (this.candado) {
            var existente = this.contactos.FirstOrDefault(c => c.Id == id);
            if (existente is null) {
                return false;
            }
            return this.contactos.Remove(existente);
        }
    }
}
```

> **¿Por qué `lock`?** Porque en Blazor Server hay un solo proceso atendiendo a múltiples usuarios. Sin el lock, dos usuarios creando contactos al mismo tiempo podrían pisarse el `proximoId`. Es buena costumbre incluso si para el curso no importa mucho. `Lock` es el tipo nuevo (.NET 9) para `Monitor`; si no lo tenés, usá `private readonly object candado = new();`.

Registrá el servicio en `Program.cs`:

```csharp
builder.Services.AddSingleton<IContactoService, ContactoServiceEnMemoria>();
```

> Lo registramos como **Singleton** porque queremos que la "base" sea única y se comparta entre todos los usuarios. Si fuera Scoped, cada usuario tendría su propia agenda (no es lo que queremos).

```razor
@page "/contactos"
@rendermode InteractiveServer
@inject IContactoService Contactos
@inject NavigationManager Nav
@using Agenda.Modelos
@using Agenda.Servicios

<PageTitle>Contactos</PageTitle>

<div class="d-flex gap-2 mb-3">

</div>

@if (this.lista is null) {
    <p><em>Cargando...</em></p>
}
else if (this.lista.Count == 0) {
    <div class="alert alert-info">No hay contactos que coincidan.</div>
}
else {
    <table class="table table-striped">

            <tr>

                    <a class="btn btn-sm btn-outline-primary"

                    <button class="btn btn-sm btn-outline-danger"
                            @onclick="() => Confirmar(c)">Eliminar</button>
                </td>
            </tr>
        }
        </tbody>
    </table>
}

@if (this.aEliminar is not null) {
    <div class="modal d-block" tabindex="-1" style="background:rgba(0,0,0,.4)">
        <div class="modal-dialog">
            <div class="modal-content">
                <div class="modal-header">
                    <h5 class="modal-title">¿Borrar contacto?</h5>
                </div>
                <div class="modal-body">
                    <p>Estás por borrar a <strong>@this.aEliminar.Nombre</strong>.
                    Esta acción no se puede deshacer.</p>
                </div>
                <div class="modal-footer">
                    <button class="btn btn-secondary"
                            @onclick="() => this.aEliminar = null">Cancelar</button>
                    <button class="btn btn-danger" @onclick="EliminarConfirmado">
                        Sí, borrar
                    </button>
                </div>
            </div>
        </div>
    </div>
}

@code {
    private string filtro = "";
    private IReadOnlyList<Contacto>? lista;
    private Contacto? aEliminar;

    protected override void OnInitialized() {
        // Carga inicial: ejecutar una sola vez.
        this.Refrescar();
    }

    private void Refrescar() {
        this.lista = this.Contactos.Listar(this.filtro);
    }

    private void Confirmar(Contacto c) => this.aEliminar = c;

    private void EliminarConfirmado() {
        if (this.aEliminar is not null) {
            this.Contactos.Eliminar(this.aEliminar.Id);
            this.aEliminar = null;
            this.Refrescar();
        }
    }
}
```

#### Conceptos que aparecen acá

1. **`@rendermode InteractiveServer`** — porque la página tiene botones que ejecutan código C# en respuesta a clics. Sin esto, los botones serían HTML estático y los `@onclick` no harían nada.
2. **`@bind="filtro" @bind:event="oninput" @bind:after="Refrescar"`** — binding bidireccional que se dispara en cada tecla y, después de actualizar la variable, llama `Refrescar()`. Esto da búsqueda en vivo.
3. **`OnInitialized` vs hacer el `Listar()` en línea**: lo hacemos en el lifecycle para que se ejecute una sola vez cuando el componente se monta.
4. **Lambda inline en el `@onclick="() => Confirmar(c)"`** — necesaria porque queremos pasar `c` como argumento.

### 2.8 Página 2: Crear / editar contacto

`Components/Pages/ContactoFormulario.razor`:

```razor
@page "/contactos/nuevo"
@page "/contactos/editar/{Id:int}"
@rendermode InteractiveServer
@inject IContactoService Contactos
@inject NavigationManager Nav
@using Agenda.Modelos
@using Agenda.Servicios

<PageTitle>@(this.EsNuevo ? "Nuevo contacto" : "Editar contacto")</PageTitle>

<h1>@(this.EsNuevo ? "➕ Nuevo contacto" : $"✏️ Editar: {this.modelo.Nombre}")</h1>

@if (this.cargando) {
    <p><em>Cargando...</em></p>
}
else {
    <EditForm Model="this.modelo" OnValidSubmit="Guardar"
              FormName="ContactoFormulario">
        <DataAnnotationsValidator />
        <ValidationSummary class="alert alert-danger" />

        <div class="mb-3">
            <label class="form-label">Nombre</label>
            <InputText @bind-Value="this.modelo.Nombre" class="form-control" />
            <ValidationMessage For="@(() => this.modelo.Nombre)"
                               class="text-danger" />
        </div>

        <div class="mb-3">
            <label class="form-label">Email</label>
            <InputText @bind-Value="this.modelo.Email" class="form-control"
                       type="email" />
            <ValidationMessage For="@(() => this.modelo.Email)"
                               class="text-danger" />
        </div>

        <div class="mb-3">
            <label class="form-label">Teléfono</label>
            <InputText @bind-Value="this.modelo.Telefono"
                       class="form-control" />
            <ValidationMessage For="@(() => this.modelo.Telefono)"
                               class="text-danger" />
        </div>

        <div class="mb-3">
            <label class="form-label">Cumpleaños</label>
            <InputDate @bind-Value="this.modelo.Cumpleanios"
                       class="form-control" />
        </div>

        <div class="mb-3">
            <label class="form-label">Categoría</label>
            <InputSelect @bind-Value="this.modelo.Categoria"
                         class="form-select">
                @foreach (var cat in Enum.GetValues<Categoria>()) {
                    <option value="@cat">@cat</option>
                }
            </InputSelect>
        </div>

        <div class="d-flex gap-2">
            <button type="submit" class="btn btn-primary">💾 Guardar</button>
            <a class="btn btn-secondary" href="/contactos">Cancelar</a>
        </div>
    </EditForm>
}

@code {
    [Parameter] public int? Id { get; set; }

    [SupplyParameterFromForm]
    public Contacto modelo { get; set; } = new();

    private bool cargando = true;
    private bool EsNuevo => this.Id is null;

    protected override void OnParametersSet() {
        // Si el Id cambia (porque navegaron a otro /contactos/editar/X),
        // volvemos a cargar.
        if (this.Id is int id) {
            var existente = this.Contactos.ObtenerPorId(id);
            if (existente is null) {
                this.Nav.NavigateTo("/contactos");
                return;
            }
            // Clonamos para no mutar el original hasta que se guarde.
            this.modelo = new Contacto {
                Id = existente.Id,
                Nombre = existente.Nombre,
                Email = existente.Email,
                Telefono = existente.Telefono,
                Cumpleanios = existente.Cumpleanios,
                Categoria = existente.Categoria,
            };
        }
        this.cargando = false;
    }

    private void Guardar() {
        if (this.EsNuevo) {
            this.Contactos.Crear(this.modelo);
        }
        else {
            this.Contactos.Actualizar(this.modelo);
        }
        this.Nav.NavigateTo("/contactos");
    }
}
```

#### Por qué dos `@page` en el mismo componente

Es una técnica muy útil: el **mismo componente** sirve tanto para crear (`/contactos/nuevo`) como para editar (`/contactos/editar/123`). La diferencia es que en el caso de edición llega un `Id` como parámetro de ruta; en el caso de creación, llega `null`.

#### Por qué `OnParametersSet` y no `OnInitialized`

Si el usuario navega de `/contactos/editar/1` a `/contactos/editar/2`, Blazor **reutiliza el mismo componente** (no lo destruye). `OnInitialized` no se vuelve a llamar, pero **`OnParametersSet` sí** porque cambió el parámetro `Id`. Por eso cargamos los datos ahí.

**Etapa 2 — Persistencia real (2-3 semanas)**:
4. Reemplazá `ContactoServiceEnMemoria` por una versión con **Entity Framework Core 10 + SQLite** (interfaz `IContactoService` permanece igual: ése es el valor de la abstracción).
5. Agregá manejo de errores con `ILogger<T>` y `try/catch` en el servicio.

**Etapa 3 — Características avanzadas (cuando estés cómodo)**:
6. Probá cambiar el render mode de la página `Contactos` a `InteractiveAuto` y observá cómo arranca en server y migra a WASM.
7. Agregá autenticación con `dotnet new blazor -au Individual` en un proyecto nuevo.
8. Investigá la directiva `[StreamRendering]` para listas que cargan en streaming.

**Cuándo cambiar de InteractiveServer a InteractiveWebAssembly**: cuando necesites trabajar offline, cuando el costo de mantener circuits para muchos usuarios concurrentes se vuelva un problema (la guía oficial "Host and deploy ASP.NET Core server-side Blazor apps" de Microsoft Learn estima *"approximately 250 KB of memory for a minimal Hello World-style app"* por circuito, o sea ~1,3 GB de RAM para soportar 5.000 usuarios concurrentes), o cuando quieras hosting estático puro (GitHub Pages, Azure Static Web Apps).

## Caveats

- **Render modes son sutiles**: `@rendermode InteractiveServer` debe aplicarse a la página *o* a un ancestro. Si te olvidás, los `@onclick` no harán nada y no vas a tener error explícito — solo botones inertes.
- **Prerender + servicios stateful**: si tu app prerenderiza (default), `OnInitializedAsync` corre dos veces (servidor + cliente). En .NET 10 usá `[PersistentState]` para no duplicar trabajo.
- **`Scoped` ≠ "por request"**: en Blazor Server, una instancia Scoped dura toda la sesión del usuario, no una petición HTTP. Confundir esto causa fugas de memoria y datos cruzados.
- **Lista en memoria**: el contenido se pierde al reiniciar el servidor. Sirve para enseñar; **no** lo uses para nada real. Para un cuatrimestre completo, agregá EF Core con SQLite (un solo archivo `.db`, cero configuración).
- **Versiones**: este tutorial asume .NET 10 (LTS, lanzado el 11 de noviembre de 2025) y C# 14. Algunos detalles (el atributo `[PersistentState]`, el `<ReconnectModal>`, la unificación `Router.NotFoundPage`) **no existen** en .NET 8/9. Si te toca una versión más vieja, ajustá. Hasta el final del soporte LTS (10 de noviembre de 2028) este código debería seguir compilando sin cambios.
- **C# 14 keyword `field`**: si tu IDE marca error, asegurate de que el `<LangVersion>preview</LangVersion>` o `<LangVersion>14.0</LangVersion>` esté en el `.csproj`, o de estar usando el SDK 10.0 que lo tiene como default.
- **El template `dotnet new blazor` cambió bastante entre .NET 7, 8 y 10**. Si seguís un tutorial viejo, es muy probable que la estructura de carpetas y `Program.cs` no coincida; mirá siempre la fecha del material.
