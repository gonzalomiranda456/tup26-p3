# SOLID en las librerías de .NET

## 1. La BCL como caso de estudio

La **BCL** (Base Class Library) de .NET es el código que más se usa en la vida diaria como programador C#: `List<T>`, `Dictionary<TKey, TValue>`, `Stream`, `FileStream`, `StreamReader`, `IEnumerable<T>`, LINQ. Todo eso fue diseñado por ingenieros de Microsoft que vienen aplicando principios de diseño hace décadas, y está pulido por millones de desarrolladores que lo usan todos los días.

Por eso, la BCL es un **laboratorio fantástico** para ver SOLID en la práctica. Hay un detalle importante: **no vas a ver "SOLID puro de libro"**, porque la BCL también carga con compatibilidad hacia atrás, rendimiento y ergonomía. Vas a ver aplicaciones claras y alguna que otra concesión pragmática. Eso es parte del aprendizaje: **los principios guían, no mandan**.

Cuando termines esta parte, cada vez que escribas `new StreamReader(new GZipStream(new FileStream(...)))` o un `foreach` sobre una `List<T>`, vas a ver el diseño detrás y entender por qué funciona tan bien.

Los ejemplos vienen de librerías de distinto nivel:

- **BCL**: `System.IO` (streams, archivos), `System.Collections.Generic` (listas, diccionarios, LINQ).
- **Stack moderno**: Entity Framework Core (ORM), ASP.NET Core (web, middleware, routing), Blazor (componentes UI).

## 2. SRP en .NET: cada pieza con su trabajo

La BCL es un caso magistral de SRP porque, en lugar de diseñar clases gigantes que "hagan todo con archivos", los ingenieros de .NET partieron el problema en **piezas finas**, cada una con una sola responsabilidad.

### 2.1 El trío `FileStream` + `StreamReader` + `TextReader`

Pensemos en el problema de **leer líneas de un archivo de texto**. Suena simple, pero descomponiéndolo desde primeros principios hay tres responsabilidades distintas:

1. **Acceder a los bytes** que viven en un disco físico (sectores, handles del SO, buffering, permisos).
2. **Convertir bytes en caracteres** usando una codificación (UTF-8, UTF-16, Latin-1).
3. **Agrupar caracteres en líneas** (detectar `\r\n`, `\n`, `\r`).

La BCL tiene **una clase por responsabilidad**:

```csharp
// Responsabilidad 1: bytes desde/hacia un dispositivo
FileStream fs   = new FileStream("datos.txt", FileMode.Open);

// Responsabilidad 2: bytes ↔ caracteres (con encoding)
StreamReader sr = new StreamReader(fs, Encoding.UTF8);

// Responsabilidad 3: caracteres → líneas (heredada de TextReader)
string? linea   = sr.ReadLine();
```

Pensá qué pasaría si todo estuviera en una única clase `ArchivoDeTexto`: tres razones distintas para cambiar (sistema de archivos, encoding, detección de líneas) → tres responsabilidades → tres clases. Y por eso en .NET **lo son**.

### 2.2 `List<T>` vs `Enumerable` (LINQ)

`List<T>` tiene la responsabilidad de **almacenar** elementos en memoria (agregar, quitar, indexar, redimensionar). Nada más.

Pero queremos hacer cosas con las colecciones: filtrar, mapear, ordenar, agrupar. ¿Esas operaciones son responsabilidad de `List<T>`? **No.** Si lo fueran, cada colección (`List<T>`, `Queue<T>`, `Stack<T>`, `HashSet<T>`) tendría que reimplementar `Where`, `Select`, `OrderBy`... y cambiar la lógica de filtrado obligaría a tocar **todas** las colecciones.

Los diseñadores de LINQ pusieron esas operaciones en una clase **estática separada**: `System.Linq.Enumerable`. Todos los métodos son **métodos de extensión** sobre `IEnumerable<T>`:

```csharp
public static class Enumerable {
    public static IEnumerable<TSource> Where<TSource>(
        this IEnumerable<TSource> source,
        Func<TSource, bool> predicate) { /* ... */ }

    public static IEnumerable<TResult> Select<TSource, TResult>(
        this IEnumerable<TSource> source,
        Func<TSource, TResult> selector) { /* ... */ }
}
```

Resultado:

- `List<T>` sólo almacena. Cambia si cambia cómo se almacena.
- `Enumerable` sólo opera. Cambia si cambia cómo se opera.
- Una colección custom que implemente `IEnumerable<T>` hereda **gratis** todo LINQ.

Es SRP llevado al extremo más elegante.

### 2.3 `DbContext` en EF Core

Un `DbContext` tiene una sola responsabilidad: **representar una sesión con la base de datos** y exponer los conjuntos de datos como propiedades.

```csharp
public class TiendaContext : DbContext {
    public DbSet<Producto>  Productos  { get; set; }
    public DbSet<Categoria> Categorias { get; set; }
    public DbSet<Pedido>    Pedidos    { get; set; }

    protected override void OnConfiguring(DbContextOptionsBuilder options)
        => options.UseSqlite("Data Source=tienda.db");
}
```

El `DbContext` **no sabe cómo mapear columnas a propiedades** (lo delega a `IEntityTypeConfiguration<T>`), **no sabe cómo ejecutar SQL** (lo delega al driver), y **no sabe nada de lógica de negocio**. Cada pieza tiene su lugar.

### 2.4 Handlers en Minimal API

Cada endpoint es un método con una única responsabilidad: manejar un tipo específico de solicitud HTTP.

```csharp
app.MapGet("/productos",         ObtenerTodos);
app.MapGet("/productos/{id}",    ObtenerPorId);
app.MapPost("/productos",        Crear);
app.MapPut("/productos/{id}",    Actualizar);
app.MapDelete("/productos/{id}", Eliminar);

static async Task<Ok<List<Producto>>> ObtenerTodos(TiendaContext db)
    => TypedResults.Ok(await db.Productos.ToListAsync());

static async Task<Results<Ok<Producto>, NotFound>> ObtenerPorId(int id, TiendaContext db)
    => await db.Productos.FindAsync(id) is Producto p
        ? TypedResults.Ok(p)
        : TypedResults.NotFound();
```

Si cambia la búsqueda, sólo cambia `ObtenerPorId`. Si cambia el formato de la lista, sólo cambia `ObtenerTodos`.

### 2.5 Middleware en ASP.NET Core

El pipeline de ASP.NET Core es una cadena de middlewares donde **cada uno tiene una sola responsabilidad**:

```csharp
app.UseHttpsRedirection();  // redirigir HTTP → HTTPS
app.UseAuthentication();    // verificar quién sos
app.UseAuthorization();     // verificar qué podés hacer
app.UseRateLimiting();      // limitar la tasa
app.MapControllers();       // rutear a controladores
```

Un middleware de autenticación no sabe nada de autorización, y viceversa. Si cambia la lógica de rate limiting, sólo cambia ese middleware.

## 3. OCP en .NET: streams y LINQ

Este principio es el que más brilla en .NET, porque toda la arquitectura de streams está construida sobre él.

### 3.1 `Stream` como jerarquía extensible

La clase base `System.IO.Stream` es una abstracción mínima: "un lugar de donde puedo leer bytes y a donde puedo escribir bytes". Su contrato es:

```csharp
public abstract class Stream : IDisposable {
    public abstract bool CanRead  { get; }
    public abstract bool CanWrite { get; }
    public abstract bool CanSeek  { get; }
    public abstract int  Read(byte[] buffer, int offset, int count);
    public abstract void Write(byte[] buffer, int offset, int count);
    public abstract long Position { get; set; }
    public abstract long Length   { get; }
    // ...
}
```

A partir de esa base, la BCL (y cualquier librería de terceros) agrega implementaciones **sin modificar `Stream`**:

- `FileStream` — bytes desde/hacia un archivo.
- `MemoryStream` — bytes en un `byte[]`.
- `NetworkStream` — bytes desde/hacia un socket.
- `GZipStream`, `DeflateStream`, `BrotliStream` — compresión envolviendo otro stream.
- `CryptoStream` — cifrado envolviendo otro stream.
- `BufferedStream` — buffering para cualquier otro stream.

El detalle clave: `GZipStream` **no es un archivo comprimido**. Es un `Stream` que, cuando le pedís bytes, los saca de **otro stream** y los descomprime al vuelo. Esto permite componer:

```csharp
using var fs      = new FileStream("datos.gz", FileMode.Open);
using var descomp = new GZipStream(fs, CompressionMode.Decompress);
using var lector  = new StreamReader(descomp, Encoding.UTF8);

string? linea;
while ((linea = lector.ReadLine()) != null)
    Console.WriteLine(linea);
```

Cada línea agrega una capa **sin tocar las clases de abajo**. Si necesitás leer un archivo comprimido y además cifrado, metés un `CryptoStream` en el medio. Ninguna clase sabe de la existencia de las otras, pero todas colaboran porque dependen de la misma abstracción.

Esto es el **patrón Decorator** en su forma más limpia, y es OCP aplicado: se pueden agregar capacidades nuevas (cifrado, compresión, buffering, throttling) sin modificar una sola línea de `Stream`, `FileStream` ni `StreamReader`.

### 3.2 LINQ sobre `IEnumerable<T>`

El mismo patrón en otro dominio. LINQ tiene cerca de 50 operadores. Todos trabajan sobre `IEnumerable<T>` y devuelven `IEnumerable<T>`. Eso permite **agregar operadores nuevos sin tocar la BCL**:

```csharp
public static class MisExtensiones {
    public static IEnumerable<T> SaltearCada<T>(this IEnumerable<T> fuente, int n) {
        int i = 0;
        foreach (var x in fuente)
            if (i++ % n != 0) yield return x;
    }
}

// Uso compuesto con operadores estándar
var pares = numeros.Where(x => x % 2 == 0).SaltearCada(3).Select(x => x * 10);
```

`SaltearCada` se compone naturalmente con `Where` y `Select` porque todos hablan el mismo idioma: `IEnumerable<T>`. Microsoft no previó tu operador, y sin embargo funciona.

### 3.3 `IQueryable<T>` y especificaciones en EF Core

`IQueryable<T>` es el equivalente de `IEnumerable<T>` para consultas que se ejecutan en la base de datos. Permite agregar operadores en el mismo estilo:

```csharp
public static class ProductoQueryExtensions {
    public static IQueryable<Producto> Activos(this IQueryable<Producto> q)
        => q.Where(p => p.Activo);

    public static IQueryable<Producto> DeLaCategoria(this IQueryable<Producto> q, int categoriaId)
        => q.Where(p => p.CategoriaId == categoriaId);

    public static IQueryable<Producto> OrdenadosPorPrecio(this IQueryable<Producto> q)
        => q.OrderBy(p => p.Precio);
}

// Uso: se componen igual que LINQ
var resultado = await db.Productos
    .Activos()
    .DeLaCategoria(3)
    .OrdenadosPorPrecio()
    .ToListAsync();
```

EF Core no fue modificado. Vos no modificaste `DbSet<T>`. Sólo extendiste el sistema. La SQL generada incluirá todos los filtros correctamente.

### 3.4 Filtros globales y middleware en ASP.NET Core

Los filtros (`IActionFilter`, `IExceptionFilter`) permiten agregar comportamiento transversal a los controladores **sin modificarlos**:

```csharp
public class LogFiltro : IActionFilter {
    private readonly ILogger<LogFiltro> logger;
    public LogFiltro(ILogger<LogFiltro> logger) => this.logger = logger;

    public void OnActionExecuting(ActionExecutingContext ctx)
        => logger.LogInformation("Ejecutando {Action}", ctx.ActionDescriptor.DisplayName);

    public void OnActionExecuted(ActionExecutedContext ctx)
        => logger.LogInformation("Terminó {Action}", ctx.ActionDescriptor.DisplayName);
}

// Registro global — todos los controladores lo reciben sin saber que existe
builder.Services.AddControllers(opts => opts.Filters.Add<LogFiltro>());
```

Si mañana hace falta un filtro de métricas, de auditoría, de compresión, se agregan sin tocar ningún controlador.

### 3.5 Componentes en Blazor

Blazor está diseñado para extenderse sin modificar lo existente:

```razor
@* BtnBase.razor — componente base, no se modifica *@
<button class="btn @CssExtra" @onclick="OnClick">@Texto</button>

@code {
    [Parameter] public string         Texto    { get; set; } = "";
    [Parameter] public string         CssExtra { get; set; } = "";
    [Parameter] public EventCallback  OnClick  { get; set; }
}
```

```razor
@* BtnPrimario.razor — extiende sin modificar BtnBase *@
<BtnBase Texto="@Texto" CssExtra="btn-primary" OnClick="@OnClick" />
```

`BtnBase` nunca se toca. Agregar un `BtnSecundario`, `BtnLink` o `BtnIcono` sólo requiere crear un componente nuevo que componga.

## 4. LSP en .NET: la concesión pragmática de Stream

LSP es más sutil, y acá vemos las concesiones pragmáticas de la BCL: hay ejemplos limpios, y hay uno famoso de violación que vale la pena entender.

### 4.1 `IEnumerable<T>` y `foreach` (LSP en su forma más pura)

`IEnumerable<T>` es la abstracción más usada de .NET. Su contrato es simple: "podés pedirme un enumerador, y el enumerador te devuelve elementos de a uno". Un `foreach` no sabe si está iterando una `List<T>`, un `HashSet<T>`, los valores de un `Dictionary<K,V>`, un archivo línea por línea con `File.ReadLines()`, o el resultado de un `Where().Select()`. Y no le importa:

```csharp
void Imprimir<T>(IEnumerable<T> fuente) {
    foreach (var item in fuente)
        Console.WriteLine(item);
}

Imprimir(new List<int> { 1, 2, 3 });           // funciona
Imprimir(new HashSet<int> { 1, 2, 3 });        // funciona
Imprimir(File.ReadLines("datos.txt"));         // funciona (¡streaming!)
Imprimir(Enumerable.Range(1, 1_000_000));      // funciona (lazy)
```

**Todas las implementaciones de `IEnumerable<T>` son intercambiables** desde el punto de vista del cliente. Esto es LSP en su forma más pura.

### 4.2 `Stream` — el caso interesante (violación parcial)

`Stream` define métodos como `Seek`, `Position` y `Length`. Pero no todos los streams pueden posicionarse: un `NetworkStream` o un `GZipStream` son secuenciales por naturaleza.

¿Qué se hizo en la BCL? Dos cosas. Primero, **propiedades de capacidad** — cada stream declara explícitamente qué puede hacer:

```csharp
public abstract bool CanRead  { get; }
public abstract bool CanWrite { get; }
public abstract bool CanSeek  { get; }
```

Segundo, **excepciones documentadas**: si llamás a `Seek` en un stream que no soporta seek, tira `NotSupportedException`.

```csharp
using var red = new NetworkStream(socket);
red.Position = 0; // 💥 NotSupportedException
```

Estrictamente hablando, **esto viola LSP**: un método que recibe un `Stream` no puede confiar en que `Seek` funcione, tiene que chequear `CanSeek` primero. La sustituibilidad del contrato base no es completa.

¿Por qué se hizo así? Porque la alternativa era partir `Stream` en `ISeekableStream` vs `IForwardOnlyStream`, y eso habría complicado enormemente el uso común. Los ingenieros de .NET priorizaron **ergonomía** sobre pureza del principio. Es una concesión consciente.

**La lección a llevarse**: cuando veas este patrón (capacidades declaradas + excepciones para lo no soportado), sabé que es un compromiso. A veces es razonable, a veces es señal de que la jerarquía está mal pensada. Si tu subtipo tira `NotSupportedException` en la mitad de los métodos, probablemente conviene dividir la interfaz.

### 4.3 `DbSet<T>` como `IQueryable<T>` en EF Core

`DbSet<T>` implementa `IQueryable<T>` e `IEnumerable<T>`. Cualquier método que acepte `IQueryable<Producto>` puede recibir un `DbSet<Producto>` real **o una lista en memoria** en los tests:

```csharp
static async Task<List<Producto>> BuscarActivos(IQueryable<Producto> fuente)
    => await fuente.Where(p => p.Activo).ToListAsync();

// Producción
var produccion = await BuscarActivos(db.Productos);

// Test
var enMemoria = new List<Producto> {
    new() { Nombre = "A", Activo = true  },
    new() { Nombre = "B", Activo = false }
}.AsQueryable();
var test = await BuscarActivos(enMemoria);
```

`BuscarActivos` no sabe si habla con SQL Server o con una lista. Ambos subtipos son completamente intercambiables.

### 4.4 `IActionResult` y `RenderFragment`

Un controlador declara que retorna `IActionResult`. Cualquier implementación concreta puede usarse:

```csharp
IActionResult Responder(bool encontrado, object? dato) {
    if (!encontrado) return new NotFoundResult();
    if (dato is null) return new NoContentResult();
    return new OkObjectResult(dato);
}
```

`NotFoundResult`, `OkObjectResult`, `CreatedAtActionResult`, `BadRequestObjectResult` — todos cumplen el contrato: escribir una respuesta HTTP. El framework los ejecuta con la misma llamada `ExecuteResultAsync`.

En Blazor, un componente padre que recibe `RenderFragment` acepta cualquier fragmento de UI como hijo. No le importa qué componentes concretos contiene: todos son intercambiables porque todos son `RenderFragment`.

## 5. ISP en .NET: la jerarquía de colecciones

.NET segregó sus colecciones en interfaces chicas y específicas. Vale la pena mirar cómo.

### 5.1 La jerarquía de `System.Collections.Generic`

```csharp
public interface IEnumerable<out T> {                       // sólo iterar
    IEnumerator<T> GetEnumerator();
}

public interface IReadOnlyCollection<out T> : IEnumerable<T> {  // + contar
    int Count { get; }
}

public interface IReadOnlyList<out T> : IReadOnlyCollection<T> { // + acceder por índice (readonly)
    T this[int index] { get; }
}

public interface ICollection<T> : IEnumerable<T> {          // + modificar
    int Count { get; }
    void Add(T item);
    bool Remove(T item);
    void Clear();
    bool Contains(T item);
    bool IsReadOnly { get; }
}

public interface IList<T> : ICollection<T> {                // + indexar y modificar
    T this[int index] { get; set; }
    int IndexOf(T item);
    void Insert(int index, T item);
    void RemoveAt(int index);
}
```

Esto permite expresar **exactamente** lo que necesita cada método:

```csharp
void Imprimir<T>(IEnumerable<T> items) { /* sólo iterar */ }

string Describir<T>(IReadOnlyCollection<T> items)             // sé cuántos hay, no modifico
    => $"Tengo {items.Count} elementos";

T ObtenerTercero<T>(IReadOnlyList<T> items) => items[2];      // leo por índice, no modifico

void AgregarVarios<T>(ICollection<T> items, IEnumerable<T> nuevos) {  // necesito poder agregar
    foreach (var n in nuevos) items.Add(n);
}
```

Si .NET tuviera sólo una `ICollection<T>` gigante, cada método que sólo quiere iterar estaría aceptando un parámetro con más poder del que necesita: podría modificarlo, vaciarlo, etc. El contrato del método sería **mentiroso**. Con interfaces segregadas, el tipo del parámetro **documenta qué vas a hacer con él**.

Caso concreto: en una API pública, aceptar `IReadOnlyList<Pedido>` en lugar de `List<Pedido>` es contrato y documentación al mismo tiempo: garantiza que el método no va a modificar la lista del llamador.

### 5.2 `IDisposable` vs `IAsyncDisposable`

ISP en chiquito pero con mucha claridad. El contrato "liberar recursos" empezó siendo:

```csharp
public interface IDisposable {
    void Dispose();
}
```

Pero con async/await, algunos recursos necesitan liberarse asincrónicamente. ¿Qué se hizo? **No se modificó `IDisposable`**. Se creó una interfaz nueva:

```csharp
public interface IAsyncDisposable {
    ValueTask DisposeAsync();
}
```

Una clase puede implementar una, la otra, o las dos. Y `using` en C# soporta las dos:

```csharp
using       var fs  = new FileStream("a.txt", FileMode.Open);  // Dispose()
await using var fs2 = new FileStream("a.txt", FileMode.Open);  // DisposeAsync()
```

Si `IDisposable` hubiera tenido siempre los dos métodos, **toda clase que la implementa** habría tenido que incluir un `DisposeAsync` que nadie pidió.

### 5.3 `IServiceCollection`, `IApplicationBuilder`, `IEndpointRouteBuilder`

El bootstrapping de ASP.NET Core usa tres interfaces, cada una con un contrato bien delimitado:

```csharp
var builder = WebApplication.CreateBuilder(args);

// IServiceCollection — sólo registrar dependencias
builder.Services.AddDbContext<TiendaContext>(opts => opts.UseSqlite("..."));
builder.Services.AddScoped<IRepositorioProductos, RepositorioEF>();

var app = builder.Build();

// IApplicationBuilder — sólo configurar el pipeline
app.UseHttpsRedirection();
app.UseAuthentication();

// IEndpointRouteBuilder — sólo mapear rutas
app.MapControllers();
app.MapGet("/salud", () => "OK");

app.Run();
```

`IServiceCollection` no sabe nada de middleware. `IApplicationBuilder` no sabe nada de rutas. Cada interfaz tiene exactamente las operaciones que su responsabilidad requiere.

### 5.4 `IComponent` vs `ComponentBase` en Blazor

Blazor define una interfaz mínima:

```csharp
public interface IComponent {
    void Attach(RenderHandle renderHandle);
    Task SetParametersAsync(ParameterView parameters);
}
```

Eso es todo lo que el framework necesita. Para usar el ciclo de vida completo (`OnInitialized`, `OnParametersSet`, `ShouldRender`), está `ComponentBase`. Pero un componente de muy bajo nivel puede implementar `IComponent` directamente sin arrastrar `ComponentBase`. Los componentes comunes no dependen de más contrato del que necesitan.

## 6. DIP en .NET: inyección de dependencias por todos lados

### 6.1 `StreamReader` depende de `Stream`, no de `FileStream`

Mirá la firma del constructor:

```csharp
public StreamReader(Stream stream, Encoding encoding) { /* ... */ }
```

El parámetro es `Stream` — la abstracción. No `FileStream`, no `NetworkStream`. Esto es DIP puro y es la razón por la que se puede hacer todo esto:

```csharp
// Desde un archivo
var r1 = new StreamReader(new FileStream("a.txt", FileMode.Open));

// Desde un archivo comprimido
var r2 = new StreamReader(
    new GZipStream(new FileStream("a.gz", FileMode.Open), CompressionMode.Decompress));

// Desde una respuesta HTTP
using var http = new HttpClient();
var r3 = new StreamReader(await http.GetStreamAsync("https://..."));

// Desde memoria (típico en tests)
var bytes = Encoding.UTF8.GetBytes("hola\nmundo");
var r4    = new StreamReader(new MemoryStream(bytes));
```

`StreamReader` no sabe ni le importa de dónde vienen los bytes. Y eso da un regalo enorme para los tests: se puede probar cualquier lógica que lea texto pasándole un `MemoryStream` con datos fabricados, sin tocar el disco.

### 6.2 El contenedor de DI de .NET

En ASP.NET Core, DIP no es una opción sino la forma esperada de trabajar:

```csharp
public class TarjetasController : ControllerBase {
    private readonly ILogger<TarjetasController> logger;
    private readonly IRepositorioTarjetas        repo;
    private readonly INotificador                notificador;

    public TarjetasController(
        ILogger<TarjetasController> logger,
        IRepositorioTarjetas        repo,
        INotificador                notificador) {
        this.logger      = logger;
        this.repo        = repo;
        this.notificador = notificador;
    }

    [HttpPost("debitar")]
    public async Task<IActionResult> Debitar(string numero, decimal monto) {
        var tarjeta = await this.repo.ObtenerAsync(numero);
        tarjeta.Debitar(monto);
        await this.repo.GuardarAsync(tarjeta);
        await this.notificador.NotificarAsync(tarjeta.Email, "Débito", $"${monto}");
        this.logger.LogInformation("Débito de {Monto} en {Numero}", monto, numero);
        return Ok();
    }
}
```

El controller **no hace `new` de nada**. Sólo depende de tres interfaces. Cuál implementación se usa se decide en `Program.cs`:

```csharp
builder.Services.AddScoped<IRepositorioTarjetas, RepositorioSqlServer>();
builder.Services.AddScoped<INotificador, NotificadorEmailSmtp>();
builder.Logging.AddConsole();
```

Si mañana cambia SQL Server por PostgreSQL, o email por SMS, **se cambia una línea**.

### 6.3 Repositorio con EF Core

DIP sugiere que la capa de negocio no dependa de EF Core directamente. La interfaz vive en el dominio; la implementación con EF Core vive en infraestructura:

```csharp
// Abstracción — capa de dominio, sin saber nada de EF
public interface IRepositorioProductos {
    Task<Producto?> ObtenerAsync(int id);
    Task<List<Producto>> ListarAsync();
    Task GuardarAsync(Producto producto);
}

// Implementación concreta — capa de infraestructura
public class RepositorioEF : IRepositorioProductos {
    private readonly TiendaContext db;
    public RepositorioEF(TiendaContext db) => this.db = db;

    public Task<Producto?>     ObtenerAsync(int id) => db.Productos.FindAsync(id).AsTask();
    public Task<List<Producto>> ListarAsync()       => db.Productos.ToListAsync();
    public async Task GuardarAsync(Producto producto) {
        db.Productos.Update(producto);
        await db.SaveChangesAsync();
    }
}

// Registro
builder.Services.AddScoped<IRepositorioProductos, RepositorioEF>();
```

La lógica de negocio usa `IRepositorioProductos`. No importa si detrás hay EF Core, un JSON en disco, o una lista en memoria para tests.

### 6.4 `IHttpClientFactory`

`HttpClient` tiene un problema conocido: instanciarlo a mano puede agotar sockets del SO. ASP.NET Core resuelve esto con `IHttpClientFactory`:

```csharp
builder.Services.AddHttpClient("Clima", client => {
    client.BaseAddress = new Uri("https://api.open-meteo.com/");
});

public class ServicioClima {
    private readonly IHttpClientFactory factory;
    public ServicioClima(IHttpClientFactory factory) => this.factory = factory;

    public async Task<string> ObtenerTemperaturaAsync() {
        var client = factory.CreateClient("Clima");
        return await client.GetStringAsync(
            "v1/forecast?latitude=-26.8&longitude=-65.2&current=temperature_2m");
    }
}
```

`ServicioClima` depende de `IHttpClientFactory`, no de un `HttpClient` concreto. En tests se reemplaza la factory sin hacer requests HTTP reales.

### 6.5 Inyección directa en handlers de Minimal API

```csharp
app.MapPost("/productos", async (
    Producto producto,
    IRepositorioProductos repo,
    ILogger<Program> logger) =>
{
    await repo.GuardarAsync(producto);
    logger.LogInformation("Producto creado: {Nombre}", producto.Nombre);
    return TypedResults.Created($"/productos/{producto.Id}", producto);
});
```

El framework inyecta lo que encuentre registrado para `IRepositorioProductos`. En tests, se registra `RepositorioFake` y el handler lo recibe sin cambios.

### 6.6 Corolario: `ILogger<T>`

`ILogger<T>` es un ejemplo precioso de todos los principios juntos. Es una interfaz chica (**ISP**) que permite implementaciones intercambiables (**LSP**, **DIP**), donde se pueden agregar loggers nuevos (archivos, Console, Seq, Datadog) sin modificar nada existente (**OCP**), y cada implementación tiene una única responsabilidad: registrar logs (**SRP**).

## 7. Mini caso: leer un archivo comprimido con LINQ

Para cerrar esta parte, armemos un pipeline que procesa un archivo de JSON Lines comprimido con GZip, filtra registros y los agrega. Sin pensar explícitamente en SOLID, estás usando los cinco principios porque la BCL te los ofrece de fábrica:

```csharp
using System.IO;
using System.IO.Compression;
using System.Text;
using System.Text.Json;

public record Movimiento(string Tarjeta, string Tipo, decimal Monto);

public static class Procesador {
    public static decimal TotalDebitosGold(string rutaArchivoGz) {
        // [DIP] StreamReader depende de Stream, no de archivos
        // [OCP] Apilamos GZipStream sobre FileStream sin tocar nada
        // [SRP] Cada capa tiene una única responsabilidad
        using var archivo = new FileStream(rutaArchivoGz, FileMode.Open);
        using var descomp = new GZipStream(archivo, CompressionMode.Decompress);
        using var lector  = new StreamReader(descomp, Encoding.UTF8);

        // [LSP] El enumerable es intercambiable con cualquier IEnumerable<string>
        // [ISP] Sólo necesitamos iterar (IEnumerable), no modificar
        return LeerLineas(lector)
            .Select(linea => JsonSerializer.Deserialize<Movimiento>(linea)!)
            .Where(m => m.Tipo == "debito" && m.Tarjeta.StartsWith("GOLD"))
            .Sum(m => m.Monto);
    }

    private static IEnumerable<string> LeerLineas(TextReader lector) {
        string? linea;
        while ((linea = lector.ReadLine()) != null)
            yield return linea;
    }
}
```

Ocho líneas de lógica real. Cada decisión de diseño de la BCL te está ayudando:

- Si mañana el archivo viene de una API en vez de disco, cambiás `FileStream` por el stream de `HttpClient`. **El resto no se toca.**
- Si el archivo usa Brotli en vez de GZip, cambiás `GZipStream` por `BrotliStream`. **El resto no se toca.**
- Si en los tests querés probar con un string en memoria, usás `new StringReader(texto)`. **El resto no se toca.**
- Si aparece una regla nueva (agrupar por fecha), agregás un `.GroupBy()` al pipeline. **Nada existente se toca.**

Esa sensación de "todo compone como Lego" no es accidente: es el resultado de aplicar SOLID consistentemente durante 20 años.
