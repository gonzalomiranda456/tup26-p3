# Patrón Strategy y Patrón Visitor en C#

```
Strategy = variar el algoritmo.
Visitor  = agregar operaciones sobre una jerarquía de objetos.
```

---

# Parte 1: Strategy

## El problema

Un `if/else if` que crece con cada variante nueva viola OCP y mezcla lógica:

```csharp
// ✗ — cada tipo nuevo obliga a modificar esta clase
public decimal Calcular(string tipo, decimal total) => tipo switch {
    "Normal"  => total,
    "Premium" => total * 0.90m,
    "Vip"     => total * 0.80m,
    _         => throw new ArgumentException()
};
```

## La solución

Encapsular cada algoritmo en una clase con una interfaz común:

```csharp
public interface IEstrategiaDescuento { decimal Aplicar(decimal total); }

public class DescuentoNormal  : IEstrategiaDescuento { public decimal Aplicar(decimal t) => t; }
public class DescuentoPremium : IEstrategiaDescuento { public decimal Aplicar(decimal t) => t * 0.90m; }
public class DescuentoVip     : IEstrategiaDescuento { public decimal Aplicar(decimal t) => t * 0.80m; }

// Contexto — no sabe qué estrategia usa
public class CalculadoraDescuento(IEstrategiaDescuento estrategia) {
    public decimal Calcular(decimal total) => estrategia.Aplicar(total);
}

// Agregar DescuentoBlackFriday no toca nada existente
var calc = new CalculadoraDescuento(new DescuentoPremium());
Console.WriteLine(calc.Calcular(1000)); // 900
```

## Con delegates (para lógica simple)

```csharp
public class CalculadoraDescuento(Func<decimal, decimal> estrategia) {
    public decimal Calcular(decimal total) => estrategia(total);
}

var calcVip = new CalculadoraDescuento(total => total * 0.80m);
```

## Casos típicos

- **Exportadores**: `CsvExportador`, `JsonExportador`, `PdfExportador` → todos implementan `IExportador`
- **Pagos**: `PagoConTarjeta`, `PagoTransferencia`, `PagoMercadoPago` → todos implementan `IEstrategiaPago`
- **Ordenamiento**: `OrdenarPorPrecio`, `OrdenarPorNombre` → todos implementan `IEstrategiaOrdenamiento`
- **Impuestos**: `ImpuestoArgentina`, `ImpuestoUruguay`, `ImpuestoExento`

## Cuándo usarlo / no usarlo

Usarlo: varios algoritmos intercambiables, `if/switch` repetidos, cambio de comportamiento en runtime, testeo aislado.  
No usarlo: una sola variante, el `if` es simple y local, crear clases empeora la legibilidad.

# Parte 2: Visitor

## El problema

Cuando hay una jerarquía estable de tipos y muchas operaciones externas que se necesitan sobre ella:

```csharp
// ✗ — Parrafo termina conociendo HTML, Markdown, accesibilidad, estadísticas...
public class Parrafo { 
    public string ToHtml() { ... }
    public string ToMarkdown() { ... }
    public int ContarPalabras() { ... }
}
```

## La solución: double dispatch

```csharp
// 1. Interfaz del visitante — un método por tipo
public interface IDocumentoVisitor {
    void Visit(Titulo t);
    void Visit(Parrafo p);
    void Visit(Imagen i);
}

// 2. Clases del modelo: solo exponen Accept
public abstract class Elemento { public abstract void Accept(IDocumentoVisitor v); }

public class Parrafo : Elemento {
    public string Texto { get; set; } = "";
    public override void Accept(IDocumentoVisitor v) => v.Visit(this); // this es Parrafo
}

// 3. Visitor concreto — opera sobre los tipos
public class HtmlVisitor : IDocumentoVisitor {
    public void Visit(Titulo t)  => Console.WriteLine($"<h{t.Nivel}>{t.Texto}</h{t.Nivel}>");
    public void Visit(Parrafo p) => Console.WriteLine($"<p>{p.Texto}</p>");
    public void Visit(Imagen i)  => Console.WriteLine($"<img src=\"{i.Url}\" alt=\"{i.Alt}\" />");
}

// Uso
var doc = new List<Elemento> { new Titulo { Texto = "Título", Nivel = 1 }, new Parrafo { Texto = "Hola" } };
var visitor = new HtmlVisitor();
foreach (var e in doc) e.Accept(visitor);
```

Para agregar `MarkdownVisitor` o `WordCountVisitor`: solo agregar una clase nueva, sin tocar el modelo.

## Por qué hace falta Accept

Si llamaras `visitor.Visit(elemento)` directamente con `elemento` declarado como la clase base, el compilador elegiría el overload en tiempo de compilación basándose en el tipo base, no en el tipo real. `Accept` es el primer despacho; `visitor.Visit(this)` dentro de cada clase concreta es el segundo — **double dispatch**.

## Visitor vs Pattern Matching moderno

Para jerarquías pequeñas, el pattern matching de C# puede ser más simple:

```csharp
public static string ToMarkdown(Elemento e) => e switch {
    Titulo t => $"{new string('#', t.Nivel)} {t.Texto}",
    Parrafo p => p.Texto,
    Imagen i  => $"![{i.Alt}]({i.Url})",
    _ => throw new NotSupportedException()
};
```

Visitor conviene cuando la operación es grande, hay muchas, necesitás acumular estado, o trabajás con árboles complejos (AST, UI trees).

## Comparación directa

| Aspecto            | Strategy                 | Visitor                              |
| ------------------ | ------------------------ | ------------------------------------ |
| Problema           | Cambiar algoritmo        | Agregar operaciones sobre jerarquía  |
| Pregunta           | ¿Cómo hago esto?         | ¿Qué operación aplico a estos tipos? |
| Bueno para agregar | Nuevas estrategias       | Nuevas operaciones                   |
| Malo para agregar  | —                        | Nuevos tipos visitables              |
| Cliente llama      | `estrategia.Ejecutar()`  | `elemento.Accept(visitor)`           |
| Ejemplos           | Descuentos, pagos, orden | Documentos, AST, UI tree             |
