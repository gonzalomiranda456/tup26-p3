# Anexo: Use de Git y GitHub

## 1. ¿Qué es Git?

**Git** es un *sistema de control de versiones distribuido*. Traducido: es un programa que lleva un historial completo de todos los cambios que ocurren en una carpeta de archivos (típicamente, el código de un proyecto).

Para entender por qué existe, pensemos en el problema que resuelve. Sin Git, cuando varias personas trabajan sobre el mismo código, el flujo habitual termina siendo algo como:

```
proyecto_final.cs
proyecto_final_v2.cs
proyecto_final_v2_corregido.cs
proyecto_final_v2_corregido_juan.cs
proyecto_final_ESTE_SI_ANDA.cs
```

Git elimina ese caos. Guarda cada versión del proyecto como una *foto* (un **commit**) y permite:

- Ver qué cambió, cuándo, y quién lo hizo.
- Volver a cualquier versión anterior.
- Trabajar en paralelo sobre la misma base sin pisarse.
- Combinar el trabajo de varias personas de forma controlada.

### Distribuido

La parte de "distribuido" significa que **cada persona tiene una copia completa del repositorio en su máquina**, con todo el historial. No depende de un servidor central para trabajar. Se puede hacer commits sin internet.

### Conceptos fundamentales

Estos son los términos que hay que internalizar. El resto se construye sobre ellos.

#### Repositorio (repo)

Es una carpeta gestionada por Git. Dentro tiene una subcarpeta oculta `.git/` donde se almacena todo el historial. Cuando decimos "el repositorio", nos referimos al proyecto completo con su historial.

#### Working directory (directorio de trabajo)

Los archivos como los ves en el explorador de archivos: lo que estás editando en este momento.

#### Staging area (área de preparación) / Index

Una zona intermedia donde se *preparan* los cambios antes de guardarlos definitivamente. Sirve para elegir **qué cambios** entran en el próximo commit. Esto permite, por ejemplo, hacer dos commits separados con cambios distintos aunque hayas editado varios archivos a la vez.

#### Commit

Es una *foto* del proyecto en un momento específico. Cada commit tiene:

- Un identificador único (un hash tipo `a3f9b2c...`).
- Un autor y una fecha.
- Un mensaje explicando qué cambió.
- Una referencia al commit anterior (su "padre").

Los commits forman una cadena histórica. Este historial es el corazón de Git.

#### Branch (rama)

Una **rama** es una línea de desarrollo independiente. Técnicamente es solo un puntero movible a un commit. La rama principal tradicionalmente se llama `main` (antes `master`).

Las ramas permiten trabajar en una funcionalidad nueva sin tocar la versión estable. Cuando la funcionalidad está lista, se *fusiona* (merge) con la rama principal.

#### Merge

Combinar los cambios de una rama en otra. Git intenta hacerlo automáticamente; cuando dos ramas modificaron la misma línea de manera distinta, aparece un **conflicto** que hay que resolver a mano.

#### Remote (remoto)

Una copia del repositorio que vive en otro lado — típicamente en un servidor como GitHub. El remoto por defecto se llama `origin`.

#### Push y Pull

- **Push**: subir tus commits locales al remoto.
- **Pull**: traer los commits del remoto a tu repo local (equivale a `fetch` + `merge`).

### El flujo mental

Visualmente, el ciclo básico de trabajo con Git es:

```
Working Directory  ──(git add)──►  Staging Area  ──(git commit)──►  Historial local
                                                                          │
                      (git push)
                        ▼
                    Remoto (GitHub)
```

## 2. ¿Qué es GitHub?

**GitHub** es un servicio web que hospeda repositorios Git. Es un servidor en la nube donde los equipos suben sus repositorios para compartirlos, colaborar y gestionarlos.

### Diferencia clave: Git vs GitHub

Confundir ambos es el error más común al empezar. La distinción es:

|                  | **Git**                   | **GitHub**                                |
|------------------|---------------------------|-------------------------------------------|
| Qué es           | Un programa (herramienta) | Un sitio web (servicio)                   |
| Dónde corre      | En tu computadora         | En los servidores de Microsoft            |
| Quién lo hizo    | Linus Torvalds, 2005      | GitHub Inc. (hoy de Microsoft), 2008      |
| Necesita internet| No                        | Sí                                        |
| Alternativas     | Mercurial, SVN            | GitLab, Bitbucket, Azure DevOps           |

**Git funciona perfectamente sin GitHub.** Podés usar Git en una carpeta local de tu máquina y nunca subir nada a ningún lado. GitHub es un lugar donde *alojar* repositorios Git para que otros los puedan ver y colaborar.

### Qué agrega GitHub sobre Git

GitHub no solo hospeda: añade funcionalidades que no son parte de Git:

- **Interfaz web** para navegar el código y el historial.
- **Pull Requests (PR)**: mecanismo para proponer cambios y revisarlos antes de fusionarlos.
- **Issues**: sistema de tickets para reportar bugs o proponer funcionalidades.
- **Actions**: automatización (ejecutar tests al subir código, por ejemplo).
- **Releases**, **Wiki**, **Projects**, control de permisos, etc.

### Cómo trabajan juntos

El modelo típico es:

1. El código vive en GitHub (el remoto central compartido).
2. Cada desarrollador **clona** el repo a su máquina.
3. Trabaja localmente con Git (ramas, commits).
4. **Sube** sus cambios a GitHub con `push`.
5. Abre un **pull request** para integrar sus cambios a la rama principal.
6. El resto del equipo revisa, comenta y aprueba.
7. Se fusiona el PR.

## 3. Instalación en Windows

Vamos a instalar dos herramientas:

- **Git for Windows**: el programa Git en sí (te da el comando `git`).
- **GitHub CLI**: un programa oficial de GitHub (comando `gh`) que facilita autenticación, crear PRs, clonar repos privados, etc.

### 3.1 Instalar Git for Windows

**Opción A — Instalador gráfico (recomendado la primera vez):**

1. Descargá desde <https://git-scm.com/download/win>. La descarga debería empezar automáticamente.
2. Ejecutá el `.exe` descargado.
3. Durante el asistente, podés aceptar casi todos los valores por defecto. Dos pantallas a tener en cuenta:
   - **Default editor used by Git**: si no conocés Vim, cambialo a **Visual Studio Code** (o **Notepad**). Vim atrapa a muchos principiantes.
   - **Adjusting the name of the initial branch**: elegí **"Override the default branch name" → `main`**. Es el estándar actual.
4. El resto: Next, Next, Install.

**Opción B — Por línea de comandos con winget (Windows 10/11):**

Abrí PowerShell y ejecutá:

```powershell
winget install --id Git.Git -e
```

**Verificar la instalación:**

Cerrá y volvé a abrir la terminal (PowerShell o CMD), y escribí:

```bash
git --version
```

Debería responder algo como `git version 2.47.0.windows.1`.

**Configuración inicial obligatoria:**

Git necesita saber quién sos para registrar los commits. Ejecutá una vez:

```bash
git config --global user.name "Tu Nombre Apellido"
git config --global user.email "tu.email@alumnos.utn.edu.ar"
```

Usá el mismo email que vas a usar en GitHub. Opcionalmente:

```bash
git config --global init.defaultBranch main
git config --global core.autocrlf true
```

(`autocrlf true` evita problemas con saltos de línea entre Windows y otros sistemas.)

### 3.2 Instalar GitHub CLI

**Opción A — Instalador gráfico:**

Descargá el `.msi` desde <https://cli.github.com/> y ejecutalo.

**Opción B — Con winget:**

```powershell
winget install --id GitHub.cli -e
```

**Verificar:**

```bash
gh --version
```

**Autenticarse con tu cuenta de GitHub:**

```bash
gh auth login
```

Te va a hacer una serie de preguntas. Respuestas recomendadas:

- *Where do you use GitHub?* → **GitHub.com**
- *What is your preferred protocol for Git operations?* → **HTTPS**
- *Authenticate Git with your GitHub credentials?* → **Yes**
- *How would you like to authenticate GitHub CLI?* → **Login with a web browser**

Te mostrará un código de un solo uso (tipo `ABCD-1234`), copialo, apretá Enter y se abre el navegador. Pegás el código, aceptás los permisos, y listo. A partir de ahora `git push` no te va a pedir usuario ni contraseña.

## 4. Flujo de trabajo: resolver un ejercicio y hacer un Pull Request

Este es el escenario típico del curso: el profesor tiene un repositorio con ejercicios en GitHub, y cada alumno resuelve su ejercicio en una rama y lo entrega mediante un PR.

Supongamos que el repo del curso es `alejandrodibattista/tup26-p3` y el ejercicio asignado es `tp1-65000`.

### 4.1 Clonar el repositorio (solo la primera vez)

Abrí una terminal en la carpeta donde querés guardar los proyectos (por ejemplo `C:\Users\ale\Dev\`) y clonalo:

```bash
gh repo clone https://github.com/alejandrodibattista/tup26-p3
cd tup26-p3
```

`gh repo clone` es equivalente a `git clone https://github.com/alejandrodibattista/tup26-p3`, pero más corto y maneja la autenticación automáticamente.

### 4.2 Actualizar el repositorio local

**Esto hay que hacerlo siempre antes de empezar a trabajar.** Garantiza que partís de la versión más reciente, no de lo que tenías ayer.

```bash
git switch main
git pull origin main
```

- `git switch main` te posiciona en la rama principal.
- `git pull origin main` trae los commits nuevos del remoto.

Si es la primera vez después de clonar, `pull` no trae nada — ya estás al día. Pero cuando el profesor publique nuevos ejercicios o correcciones, este es el comando que los trae.

### 4.3 Crear una rama para el ejercicio

**Regla de oro: nunca trabajes directamente sobre `main`.** Cada trabajo va en su propia rama.

```bash
git switch -c tp1-65000
```

El flag `-c` (de *create*) crea la rama y te mueve a ella en un solo paso. El nombre de la rama conviene que describa qué hace — convenciones habituales:

- `tp1-65000`
- `feature/login-con-jwt`
- `fix/bug-division-por-cero`

Para verificar en qué rama estás:

```bash
git branch
```

El asterisco (`*`) marca la rama actual.

### 4.4 Resolver el ejercicio

Ahora trabajás normalmente: abrís tu editor (VS Code, Rider, Visual Studio), editás los archivos, escribís el código, lo compilás, lo probás.

Mientras tanto, podés ir viendo qué cambió con:

```bash
git status
```

Te muestra archivos modificados, agregados, eliminados, y cuáles están en el staging area y cuáles no.

```bash
git diff
```

Te muestra exactamente qué líneas cambiaron.

### 4.5 Hacer commits

Cuando tengas una parte funcionando — no hace falta que el ejercicio esté terminado —, guardá un commit. **Commits pequeños y frecuentes son mejores que un commit gigante al final.**

```bash
git add .
git commit -m "Implementar interfaz IComparable en clase Alumno"
```

- `git add .` pone *todos* los archivos modificados en el staging area. Si querés agregar solo uno: `git add Program.cs`.
- `git commit -m "..."` crea el commit con ese mensaje.

**Buenas prácticas para el mensaje de commit:**

- Empezar con un verbo en infinitivo o imperativo: "Agregar…", "Corregir…", "Refactorizar…".
- Que describa *qué* cambia, no *cómo*. El código ya muestra el cómo.
- Menos de 72 caracteres en la primera línea.

Repetí editar → `add` → `commit` tantas veces como haga falta hasta terminar el ejercicio.

### 4.6 Subir la rama a GitHub

Hasta ahora, todos los commits viven solo en tu máquina. Para subirlos:

```bash
git push -u origin tp1-65000
```

El flag `-u` (de *upstream*) establece el vínculo entre tu rama local y la remota. Solo lo necesitás la primera vez. A partir del segundo push en la misma rama, alcanza con `git push`.

### 4.7 Crear el Pull Request

Ahora que la rama está en GitHub, abrimos el PR. Con GitHub CLI es un solo comando:

```bash
gh pr create --base main --head tp1-65000
```

Te va a abrir un editor para escribir el título y la descripción del PR. Un buen PR contiene:

- **Título** claro: "Ejercicio 5 — Interfaces — Alejandro Pérez".
- **Descripción** con:
  - Qué resuelve el PR.
  - Decisiones de diseño relevantes.
  - Cómo probarlo (si aplica).
  - Cualquier duda que quieras que el profesor mire.

Ejemplo:

```markdown
## Qué hace
Resuelve el ejercicio 5. Implementa `IComparable<Alumno>` ordenando por 
promedio descendente, y `IEquatable<Alumno>` comparando por DNI.

## Decisiones
- Usé `record class` en vez de `class` para aprovechar la igualdad estructural 
  automática, pero sobrescribí `Equals` para usar solo el DNI.
- El método `CompareTo` devuelve 0 cuando los promedios son iguales, 
  sin desempatar por nombre.

## Dudas
¿Está bien que `IEquatable` y la igualdad de `record` convivan, o debería 
volver a `class` común?
```

Alternativamente, sin CLI: `gh` te va a dar una URL; o entrás a GitHub en el navegador, tu rama aparecerá con un botón amarillo "Compare & pull request".

### 4.8 Revisión y ciclo de comentarios

El profesor revisa el PR en GitHub. Puede:

- **Aprobarlo** y fusionarlo.
- **Pedir cambios** con comentarios sobre líneas específicas.

Si pide cambios, el ciclo es:

1. Editás en tu máquina (en la misma rama).
2. `git add .` → `git commit -m "Ajustar CompareTo según revisión"`.
3. `git push` (ya no hace falta `-u`).
4. El PR se actualiza automáticamente con los commits nuevos.

Cuando el profesor aprueba y mergea, tu rama se integra a `main`. Ese es el final del ciclo.

### 4.9 Limpieza después del merge

Una vez que el PR fue fusionado, la rama ya no hace falta:

```bash
git switch main
git pull origin main
git branch -d tp1-65000
```

Y listo — estás limpio y listo para el próximo ejercicio.

## 5. Resumen del ciclo en 10 comandos

```bash
# Una sola vez
git clone <url>

# Al empezar a trabajar en algo nuevo
git switch main
git pull origin main
git switch -c mi-rama

# Mientras trabajás
git status
git diff
git add .
git commit -m "mensaje"

# Para entregar
git push -u origin mi-rama
gh pr create

# Después del merge
git switch main
git pull origin main
git branch -d mi-rama
```

Con estos comandos se resuelve el 90% del trabajo diario. El resto (resolver conflictos, reescribir historia, cherry-pick, rebase, etc.) viene cuando haya necesidad real de aprenderlo.

## 6. Errores típicos a evitar

- **Trabajar directamente en `main`.** Siempre crear una rama.
- **Olvidarse de `git pull` antes de empezar.** Terminás con conflictos evitables.
- **Hacer un único commit gigante al final.** Perdés la capacidad de deshacer cambios parciales.
- **Commits con mensajes inútiles** (`"cambios"`, `"asdf"`, `"fix"`). En seis meses no vas a entender qué hiciste.
- **Subir archivos que no corresponden** (`bin/`, `obj/`, `.vs/`, `node_modules/`). Para eso existe `.gitignore`. Para proyectos .NET, GitHub genera uno automáticamente al crear el repo.
- **Hacer `git push --force`** sin entender qué hace. Puede borrar el trabajo de otros del servidor.

## 7. Circuito con GitHub Desktop

**GitHub Desktop** es la aplicación oficial de GitHub para trabajar con repositorios Git desde una interfaz gráfica. Sirve para clonar repositorios, crear ramas, revisar cambios, hacer commits, subir la rama al remoto y abrir pull requests sin usar la terminal todo el tiempo.

### 7.1 Instalar GitHub Desktop

1. Entrá a <https://desktop.github.com/>.
2. Descargá el instalador para Windows.
3. Ejecutalo y seguí el asistente.
4. Iniciá sesión con tu cuenta de GitHub.

GitHub Desktop usa Git por debajo, así que conviene tener Git for Windows instalado igualmente. No reemplaza Git: lo simplifica con una interfaz visual.

### 7.2 Circuito de trabajo con GitHub Desktop

1. Abrí GitHub Desktop e iniciá sesión.
2. Elegí **Clone a repository from the Internet** y clona el repositorio del curso.
3. Antes de empezar, asegurate de estar en `main` y hacé **Fetch origin** o **Pull origin** para actualizarte.
4. Creá una rama nueva con **Branch > New Branch**.
5. Editá el código en tu editor habitual.
6. Volvé a GitHub Desktop y revisá los archivos modificados.
7. Seleccioná los cambios que querés incluir, escribí el mensaje del commit y hacé clic en **Commit to <rama>**.
8. Hacé clic en **Push origin** para subir la rama a GitHub.
9. Usá **Create Pull Request** o **View on GitHub** para abrir el PR.
10. Si te piden cambios, repetí el ciclo: editar, commit y push.
11. Cuando el PR se fusione, hacé **Pull origin**, volvé a `main` y borrá la rama si ya no la necesitás.

### 7.3 Equivalencia con la terminal

- **Clone** = `git clone`
- **New Branch** = `git switch -c`
- **Commit** = `git add` + `git commit`
- **Push origin** = `git push`
- **Pull origin** = `git pull`
- **Create Pull Request** = abrir el PR en GitHub
