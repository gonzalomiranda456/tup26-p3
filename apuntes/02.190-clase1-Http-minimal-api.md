## 1. El problema: dos programas que necesitan comunicarse

Hasta ahora todos los programas que escribiste corrían solos. Un programa leía datos, los procesaba y los mostraba. Todo en el mismo proceso, en la misma computadora.

Pero la mayoría del software real funciona diferente. Hay un programa que **guarda y procesa datos** (el backend), y otro programa que **los muestra al usuario** (el frontend). Corren en máquinas distintas, y necesitan hablar entre sí.

La pregunta es: ¿cómo?

El mecanismo más simple posible sería este:

```
Programa A:  "dame los productos"
Programa B:  [lista de productos]
```

Un programa pregunta. El otro responde. Eso es todo.

Pero para que dos programas puedan comunicarse sin importar qué lenguaje usan, qué sistema operativo corren, o dónde están en el mundo, necesitan ponerse de acuerdo en **un protocolo**: un conjunto de reglas sobre cómo se hacen las preguntas y cómo se dan las respuestas.

Ese protocolo se llama **HTTP**.

---

## 2. HTTP: el contrato de la comunicación

HTTP (*HyperText Transfer Protocol*) es el protocolo que hace funcionar la web. Cuando abrís un browser y entrás a cualquier sitio, tu browser está mandando mensajes HTTP y recibiendo respuestas HTTP. Todo el tiempo.

Pero HTTP no es solo para browsers. Es el protocolo estándar para que **cualquier programa le hable a cualquier otro programa** a través de una red. Por eso se usa para las APIs.

El modelo es siempre el mismo:

```
Cliente  ──── request ────►  Servidor
Cliente  ◄─── response ────  Servidor
```

- El **cliente** es el que hace la pregunta. Puede ser un browser, una app móvil, otro servidor, o una herramienta de testing.
- El **servidor** es el que escucha, procesa la pregunta y manda la respuesta.
- Un **request** es la pregunta.
- Un **response** es la respuesta.

Dos reglas importantes de HTTP:

1. **Siempre inicia el cliente.** El servidor nunca manda mensajes por su cuenta — solo responde cuando le preguntan.
2. **Cada request es independiente.** El servidor no recuerda los requests anteriores. Cada vez que el cliente pregunta, el servidor lo trata como si fuera la primera vez.

Esta segunda regla tiene una consecuencia importante que vamos a explorar más adelante.

---

## 3. El mensaje de ida: el request

Un request HTTP tiene tres partes:

```
[VERBO] [URL]
[cabeceras opcionales]

[cuerpo opcional]
```

### El verbo — qué querés hacer

El verbo le dice al servidor **qué tipo de operación** querés realizar. Hay varios, pero en el 90% de las APIs se usan cuatro:

| Verbo | Significado | Ejemplo |
|-------|-------------|---------|
| `GET` | Quiero **leer** algo | Darme la lista de tareas |
| `POST` | Quiero **crear** algo nuevo | Agregar una tarea nueva |
| `PUT` | Quiero **reemplazar** algo existente | Actualizar una tarea completa |
| `DELETE` | Quiero **eliminar** algo | Borrar una tarea |

> Un `GET` no debería modificar nada en el servidor. Es solo lectura. Esto es una convención importante: si alguien hace GET dos veces seguidas, debería obtener el mismo resultado.

### La URL — sobre qué lo querés hacer

La URL identifica **el recurso** sobre el que operás. Por convención, las URLs de una API nombran recursos en plural:

```
/tareas          → la colección de tareas
/tareas/5        → la tarea con id 5
/tareas/5/notas  → las notas de la tarea con id 5
```

Combinando verbo y URL, el significado queda claro sin necesitar más explicación:

```
GET    /tareas       → dame todas las tareas
GET    /tareas/5     → dame la tarea 5
POST   /tareas       → creá una tarea nueva
PUT    /tareas/5     → reemplazá la tarea 5
DELETE /tareas/5     → eliminá la tarea 5
```

Esto es lo que se llama una API **REST** (o RESTful): los recursos son sustantivos en la URL, y los verbos HTTP expresan la operación.

### El cuerpo — los datos que mandás

Algunos requests necesitan mandar datos. Por ejemplo, cuando creás una tarea nueva, tenés que decirle al servidor cuál es el texto de esa tarea. Esos datos van en el **cuerpo** del request.

Los requests `GET` y `DELETE` no tienen cuerpo — solo están preguntando o pidiendo que se borre algo, no están mandando datos.

Los requests `POST` y `PUT` sí tienen cuerpo — están mandando los datos de lo que se va a crear o modificar.

---

## 4. El mensaje de vuelta: el response

Un response HTTP también tiene tres partes:

```
[CÓDIGO DE ESTADO]
[cabeceras opcionales]

[cuerpo opcional]
```

### El código de estado — cómo salió

El código de estado es un número de tres dígitos que le dice al cliente si todo salió bien o qué falló. Están agrupados por centena:

| Rango | Significado | Ejemplo |
|-------|-------------|---------|
| `2xx` | **Éxito** | Todo salió bien |
| `4xx` | **Error del cliente** | La pregunta estaba mal hecha |
| `5xx` | **Error del servidor** | El servidor tuvo un problema |

Los más usados en APIs:

| Código | Nombre | Cuándo usarlo |
|--------|--------|---------------|
| `200 OK` | Éxito general | GET, PUT exitosos |
| `201 Created` | Recurso creado | POST exitoso que crea algo |
| `204 No Content` | Éxito sin cuerpo | DELETE exitoso |
| `400 Bad Request` | Datos inválidos | El cliente mandó datos mal formados |
| `404 Not Found` | No encontrado | El recurso pedido no existe |
| `500 Internal Server Error` | Error del servidor | Algo explotó adentro |

> El código de estado no es decorativo. Un cliente bien escrito lo usa para decidir qué hacer: si recibe `201`, sabe que puede mostrar "tarea creada"; si recibe `404`, sabe que puede mostrar "no encontrado". Si el servidor siempre devuelve `200` aunque haya un error, el cliente no puede distinguir el éxito del fracaso.

### El cuerpo — los datos que devuelve

El cuerpo de la respuesta contiene los datos que el servidor devuelve. Para un `GET /tareas` sería la lista de tareas. Para un `POST /tareas` exitoso sería la tarea recién creada (con su id asignado).

---

## 5. JSON: el formato de los datos

El cuerpo de los requests y responses necesita un formato estándar que cualquier programa pueda entender. En las APIs modernas ese formato es **JSON** (*JavaScript Object Notation*).

JSON tiene cuatro construcciones básicas:

```json
"texto"          // string — siempre comillas dobles
42               // número
true / false     // booleano
null             // ausencia de valor
```

Con esas cuatro piezas se arman dos estructuras:

```json
{ "clave": valor, "otraClave": valor }   // objeto
[ valor, valor, valor ]                  // array
```

Y se pueden anidar:

```json
{
  "id": 1,
  "texto": "Estudiar para el parcial",
  "completada": false,
  "etiquetas": ["estudios", "urgente"]
}
```

C# y .NET tienen soporte nativo para JSON con `System.Text.Json`. Minimal API lo usa automáticamente: si un handler devuelve un objeto C#, lo serializa a JSON. Si un handler recibe un cuerpo JSON, lo deserializa al tipo C# correspondiente.

## 6. De HTTP a Minimal API

Ya entendemos el modelo: el cliente manda un verbo + URL (y a veces un cuerpo), el servidor responde con un código + (a veces) un cuerpo.

**Minimal API es la forma más directa de implementar ese modelo en C#.** La idea es simple: declarás qué URL y qué verbo maneja cada función, y esa función es el handler.

```csharp
app.MapGet("/tareas", () => { /* handler para GET /tareas */ });
app.MapPost("/tareas", (Tarea t) => { /* handler para POST /tareas */ });
app.MapDelete("/tareas/{id}", (int id) => { /* handler para DELETE /tareas/5 */ });
```

El mapa es directo:

```
GET /tareas      →  app.MapGet("/tareas", handler)
POST /tareas     →  app.MapPost("/tareas", handler)
DELETE /tareas/5 →  app.MapDelete("/tareas/{id}", handler)
```

Nada más. No hay clases, no hay atributos, no hay herencia. Una ruta, una función.

## 7. El proyecto

### Estructura

Un proyecto Minimal API es un proyecto de consola con el paquete de ASP.NET Core. La forma más rápida de crearlo:

```bash
dotnet new web -n TareasApi
cd TareasApi
```

Esto genera tres archivos:

```
TareasApi/
├── TareasApi.csproj   ← referencia al SDK web
├── appsettings.json   ← configuración (no lo tocamos hoy)
└── Program.cs         ← toda la aplicación
```

El `Program.cs` generado es:

```csharp
var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

app.MapGet("/", () => "Hello World!");

app.Run();
```

Cuatro líneas. Ejecutalo con `dotnet run` y visitá `http://localhost:5000` en el browser.

### Las tres líneas que importan

```csharp
var builder = WebApplication.CreateBuilder(args);
```
Configura el servidor: carga settings, prepara el sistema de logging, registra servicios. Por ahora no le prestamos atención — `builder` es la fase de configuración *antes* de que el servidor empiece a escuchar.

```csharp
var app = builder.Build();
```
Construye la aplicación a partir de la configuración. A partir de acá agregamos los endpoints.

```csharp
app.Run();
```
Arranca el servidor y empieza a escuchar requests. Esta línea bloquea — el programa no pasa de acá hasta que lo detenés con Ctrl+C.

Entre `Build()` y `Run()` es donde vive toda la definición de los endpoints.

## 8. Los primeros endpoints

Vamos a construir la API de tareas de a pasos. Empezamos sin base de datos — los datos viven en una lista en memoria.

### Paso 1 — una lista en memoria

```csharp
var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

// Lista en memoria — los datos desaparecen al reiniciar
var tareas = new List<string> { "Estudiar HTTP", "Practicar Minimal API" };

app.MapGet("/tareas", () => tareas);

app.Run();
```

Ejecutá con `dotnet run` y abrí `http://localhost:5000/tareas` en el browser. Vas a ver:

```json
["Estudiar HTTP","Practicar Minimal API"]
```

Minimal API tomó la `List<string>`, la serializó a JSON y la devolvió con `200 OK`. Automático.

### Paso 2 — un modelo de datos

Una string no alcanza. Una tarea necesita id, texto y estado. Definimos un `record`:

```csharp
record Tarea(int Id, string Texto, bool Completada);
```

Actualizamos la lista y el endpoint:

```csharp
var tareas = new List<Tarea> {
    new(1, "Estudiar HTTP",        false),
    new(2, "Practicar Minimal API", false),
};

app.MapGet("/tareas", () => tareas);
```

Resultado:

```json
[
  { "id": 1, "texto": "Estudiar HTTP",        "completada": false },
  { "id": 2, "texto": "Practicar Minimal API", "completada": false }
]
```

### Paso 3 — obtener una tarea por id

Cuando el cliente pide `GET /tareas/1`, necesitamos extraer el `1` de la URL. Eso es un **parámetro de ruta**: se declara con llaves en la URL y aparece como parámetro del handler con el mismo nombre.

```csharp
app.MapGet("/tareas/{id}", (int id) => {
    var tarea = tareas.FirstOrDefault(t => t.Id == id);
    return tarea;
});
```

Si pedís `GET /tareas/1` recibís:

```json
{ "id": 1, "texto": "Estudiar HTTP", "completada": false }
```

Si pedís `GET /tareas/99` recibís... `null` serializado como respuesta vacía con `200 OK`. Eso está mal — si algo no existe, el código correcto es `404`. Hay que mejorar el handler:

```csharp
app.MapGet("/tareas/{id}", (int id) => {
    var tarea = tareas.FirstOrDefault(t => t.Id == id);

    if (tarea is null)
        return Results.NotFound();

    return Results.Ok(tarea);
});
```

`Results` es la clase que Minimal API provee para construir responses con el código de estado correcto:

| Método | Código | Cuándo usarlo |
|--------|--------|---------------|
| `Results.Ok(dato)` | `200` | GET o PUT exitoso con datos |
| `Results.Created(url, dato)` | `201` | POST exitoso |
| `Results.NoContent()` | `204` | DELETE exitoso |
| `Results.NotFound()` | `404` | El recurso no existe |
| `Results.BadRequest(msg)` | `400` | Datos inválidos del cliente |

> Cuando el handler devuelve un objeto directamente (como `return tareas`), Minimal API usa `200 OK` por defecto. Para cualquier otro código, usamos `Results`.

## 9. Probando la API

El browser solo puede hacer `GET`. Para probar `POST`, `PUT` y `DELETE` necesitás una herramienta cliente HTTP.

Las opciones más simples:

- **REST Client** (extensión de VS Code): escribís los requests en un archivo `.http`, los ejecutás con un click.
- **Thunder Client** (extensión de VS Code): interfaz gráfica similar a Postman.
- **curl**: línea de comandos, viene instalado en todos los sistemas.

### Con REST Client (recomendado para la cursada)

Creá un archivo `requests.http` en la carpeta del proyecto:

```http
### Listar todas las tareas
GET http://localhost:5000/tareas

###

### Obtener una tarea por id
GET http://localhost:5000/tareas/1

###

### Obtener una tarea que no existe
GET http://localhost:5000/tareas/99
```

Cada `###` separa los requests. Haciendo click en "Send Request" sobre cada uno ves la respuesta completa: código, cabeceras y cuerpo.

## 10. Recibir datos: POST con cuerpo

Hasta acá el cliente solo pedía datos. Ahora el cliente también va a **mandar** datos: el texto de la nueva tarea.

Los datos van en el **cuerpo del request** como JSON. Minimal API los deserializa automáticamente si declarás el parámetro con el tipo correspondiente:

```csharp
app.MapPost("/tareas", (Tarea nueva) => {
    var id     = tareas.Count == 0 ? 1 : tareas.Max(t => t.Id) + 1;
    var tarea  = nueva with { Id = id };   // record: copia con Id asignado

    tareas.Add(tarea);

    return Results.Created($"/tareas/{tarea.Id}", tarea);
});
```

Tres cosas para notar:

**El parámetro `Tarea nueva`** — Minimal API ve que `Tarea` es un tipo complejo (no un `int`, no un `string`) y asume que viene del cuerpo del request. Lo deserializa del JSON automáticamente.

**`nueva with { Id = id }`** — `Tarea` es un `record`, que es inmutable. Para "cambiarle" una propiedad creamos una copia con el campo modificado.

**`Results.Created`** — recibe dos parámetros: la URL donde se puede encontrar el recurso creado (va en el header `Location` de la respuesta) y el recurso en sí (va en el cuerpo). El código de estado es `201`.

### El request de prueba

En el archivo `.http`:

```http
### Crear una tarea nueva
POST http://localhost:5000/tareas
Content-Type: application/json

{
  "id": 0,
  "texto": "Hacer el parcial",
  "completada": false
}
```

La línea `Content-Type: application/json` es obligatoria — le dice al servidor en qué formato vienen los datos. Sin esa cabecera, Minimal API no intenta deserializar el cuerpo.

Response esperado:

```
HTTP/1.1 201 Created
Location: /tareas/3

{
  "id": 3,
  "texto": "Hacer el parcial",
  "completada": false
}
```

### El modelo completo hasta acá

```csharp
var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

var tareas = new List<Tarea> {
    new(1, "Estudiar HTTP",        false),
    new(2, "Practicar Minimal API", false),
};

app.MapGet("/tareas", () => tareas);

app.MapGet("/tareas/{id}", (int id) => {
    var tarea = tareas.FirstOrDefault(t => t.Id == id);
    return tarea is null ? Results.NotFound() : Results.Ok(tarea);
});

app.MapPost("/tareas", (Tarea nueva) => {
    var id    = tareas.Count == 0 ? 1 : tareas.Max(t => t.Id) + 1;
    var tarea = nueva with { Id = id };
    tareas.Add(tarea);
    return Results.Created($"/tareas/{tarea.Id}", tarea);
});

app.Run();

record Tarea(int Id, string Texto, bool Completada);
```

Son 20 líneas. Tenés una API con tres endpoints que funciona. Podés probar cada uno desde el archivo `.http`.

## 11. Resumen

El modelo mental que se llevás de esta clase:

```
HTTP es un contrato:
  request  = verbo + URL + (cuerpo opcional)
  response = código + (cuerpo opcional)

Los verbos expresan la intención:
  GET     → leer     (sin modificar nada)
  POST    → crear
  PUT     → reemplazar
  DELETE  → eliminar

Los códigos de estado comunican el resultado:
  2xx → éxito      (200 OK, 201 Created, 204 No Content)
  4xx → error del cliente   (400 Bad Request, 404 Not Found)
  5xx → error del servidor  (500 Internal Server Error)

Minimal API traduce HTTP a C# de forma directa:
  app.MapGet("/tareas", handler)        → GET /tareas
  app.MapGet("/tareas/{id}", handler)   → GET /tareas/:id
  app.MapPost("/tareas", handler)       → POST /tareas

  Parámetros de ruta:  {id} en la URL → int id en el handler
  Cuerpo del request:  tipo complejo como parámetro → deserializado automáticamente
  Respuestas:          Results.Ok(), Results.Created(), Results.NotFound(), etc.
```

**Lo que le falta a esta API:**

Los datos viven en una lista en memoria. Si reiniciás el servidor, todo se pierde. En la próxima clase conectamos EF Core + SQLite para que la información persista.

## Ejercicio para la casa

Partiendo del código de esta clase, agregá los endpoints que faltan para tener un CRUD completo:

1. `PUT /tareas/{id}` — reemplaza una tarea existente. Si no existe, devuelve `404`. Si existe, devuelve `200` con la tarea actualizada.
2. `DELETE /tareas/{id}` — elimina una tarea. Si no existe, devuelve `404`. Si existe, devuelve `204`.

Tip: el patrón es el mismo que `GET /tareas/{id}` — buscás por id, si no encontrás devolvés `NotFound()`, si encontrás operás y devolvés el resultado apropiado.
