Parte I — El problema que resuelve

1. Cómo nace una dependencia

Toda clase no trivial usa otras clases. Un ProcesadorPedidos necesita guardar pedidos en una base, mandarle un mail al cliente, registrar un log, cobrar con un gateway. Cada uno de esos verbos vive en otra clase. Eso es lo deseable: cada clase con un solo motivo de cambio (SRP). Pero crea una pregunta operativa inmediata:

¿Quién construye a quién?

La forma ingenua es construir las cosas adentro:

public class ProcesadorPedidos {
    private readonly EnviadorEmails        emails  = new EnviadorEmails();
    private readonly RepositorioPedidosSql pedidos = new RepositorioPedidosSql();
    private readonly GatewayMercadoPago    cobros  = new GatewayMercadoPago();

    public void Procesar(Pedido p) { /* usar las dependencias */ }
}

Compila, corre, parece razonable. Pero está roto en seis dimensiones distintas y todavía no nos dimos cuenta.

2. El daño del new

Cuando una clase escribe new para obtener sus dependencias, ocurren todas estas cosas a la vez:

Acoplamiento concreto. ProcesadorPedidos ya no depende de "algo que envía mails": depende exactamente de EnviadorEmails. Si mañana hace falta usar Sendgrid en producción y SMTP en desarrollo, hay que tocar ProcesadorPedidos. La clase que más cuidado debería recibir está atada a la implementación que más cambia.

Imposibilidad de testear. Para correr un test unitario de Procesar, hay que tener un servidor SMTP, una base SQL y una conexión a Mercado Pago. Eso no es un test unitario, es un viaje en taxi.

Configuración filtrada. EnviadorEmails necesita host, puerto, usuario y contraseña. ¿De dónde los saca? Si ProcesadorPedidos los pasa, entonces ProcesadorPedidos los conoce — y ahora también es responsable de la configuración SMTP. Si EnviadorEmails los lee de un archivo, entonces EnviadorEmails tiene una dependencia oculta con el sistema de archivos. Cualquiera de las dos opciones es peor que la situación inicial.

Duplicación silenciosa de recursos. Si tres clases distintas hacen new ConexionDb(), hay tres conexiones donde podía haber un pool de una. El programa funciona pero consume recursos como si la base fuera infinita.

Orden de construcción cristalizado. Si mañana RepositorioPedidosSql necesita un ILogger, hay que cambiar todas las clases que lo construyen. El árbol de dependencias está repetido en cada hoja en lugar de declarado en un solo lugar.

Decisiones políticas mezcladas con lógica. "Usar SMTP", "usar SQL Server", "usar Mercado Pago" son decisiones de configuración del sistema. Que vivan dentro de la clase de negocio significa que la clase de negocio está tomando decisiones que no le corresponden.

El new no es ilegal — sigue siendo perfectamente válido para crear value objects, DTOs, excepciones y cosas pequeñas que no tienen identidad ni recursos. El problema es usarlo para resolver el grafo de dependencias de la aplicación.

3. Inyección por constructor: mover el problema

El primer movimiento correcto es declarar las dependencias en el constructor en lugar de fabricarlas:

public class ProcesadorPedidos {
    private readonly IEnviadorEmails      emails;
    private readonly IRepositorioPedidos  pedidos;
    private readonly IGatewayCobros       cobros;

    public ProcesadorPedidos(
        IEnviadorEmails emails,
        IRepositorioPedidos pedidos,
        IGatewayCobros cobros) {

        this.emails  = emails;
        this.pedidos = pedidos;
        this.cobros  = cobros;
    }

    public void Procesar(Pedido p) { /* usar las dependencias */ }
}

(En C# 12+ con primary constructors, el boilerplate se reduce a una línea, pero el principio es el mismo.)

Tres cosas mejoran instantáneamente:

  1. Las dependencias son explícitas. Mirando el constructor sé exactamente qué necesita esta clase para funcionar.
  2. Son intercambiables. Cualquier implementación de IEnviadorEmails sirve — incluyendo un fake en tests.
  3. No las construye la clase. Se las pasan ya armadas.

Pero el problema no se resuelve, se traslada. Ahora alguien afuera tiene que construir un IEnviadorEmails, un IRepositorioPedidos, un IGatewayCobros, y pasárselos al ProcesadorPedidos. Y ese alguien necesita, a su vez, construir las dependencias de cada uno de ellos. La pregunta sigue en pie:

¿Quién construye a quién?

4. El composition root

La respuesta clásica: toda la composición ocurre en un único lugar, en el borde del programa, antes de que empiece la lógica de negocio. Eso se llama el composition root. En una app de consola, es las primeras líneas de Main. En ASP.NET Core, es Program.cs antes del app.Run().

Una versión manual del composition root para nuestro ejemplo:

// Program.cs

// Hojas del árbol: dependencias sin dependencias.
var configSmtp     = new ConfigSmtp("smtp.gmail.com", 587, "user", "pass");
var connStringSql  = "Server=...;Database=...;";

// Nivel 1: dependencias que sólo usan hojas.
IEnviadorEmails     emails  = new EnviadorEmailsSmtp(configSmtp);
IRepositorioPedidos pedidos = new RepositorioPedidosSql(connStringSql);
IGatewayCobros      cobros  = new GatewayMercadoPago("api-key-...");

// Nivel 2: la clase que orquesta todo.
var procesador = new ProcesadorPedidos(emails, pedidos, cobros);

// Recién ahora arranca la aplicación.
procesador.Procesar(pedido);

Esto es honesto: las decisiones de configuración están en un solo archivo, las clases de negocio no saben nada de SMTP ni de SQL, y testear ProcesadorPedidos con dobles es trivial. Para una app chica, hacerlo a mano es perfectamente razonable.

El problema aparece con el tamaño. Cuando hay 50 servicios, 200 clases de dominio, 30 controllers HTTP y cada uno tiene su grafo de dependencias, el composition root manual se vuelve:

  • Tedioso. Decenas de líneas de cableado donde cada new exige saber el orden topológico.
  • Frágil. Reordenar declaraciones rompe el build. Olvidar pasar una dependencia es un null reference en runtime.
  • Repetitivo. Si una clase aparece en cinco grafos distintos, está construida cinco veces (o hay que crear y guardar variables intermedias para reutilizarla).
  • Sin política de lifetime. ¿La conexión SQL es una sola instancia compartida o una por operación? El código manual no responde sin que pensemos caso por caso.

Acá es donde aparece la idea de delegar la mecánica del cableado a un componente reutilizable. Ese componente es el contenedor de dependencias, y IServiceProvider es su cara estandarizada en .NET.

Parte II — Construcción de un contenedor desde cero

Antes de mostrar la API real, vale la pena construir uno propio. Sirve para entender qué está haciendo el contenedor por debajo y, sobre todo, qué no es magia.

5. Un registro naive

La idea más simple posible: un diccionario que mapea "tipo pedido" a "función que lo construye".

public class ContenedorMinimo {
    private readonly Dictionary<Type, Func<object>> fabricas = new();

    public void Registrar<T>(Func<T> fabrica) where T : class
        => fabricas[typeof(T)] = () => fabrica();

    public T Resolver<T>() where T : class
        => (T)fabricas[typeof(T)]();
}

Uso:

var c = new ContenedorMinimo();

c.Registrar<IEnviadorEmails>(() => new EnviadorEmailsSmtp(configSmtp));
c.Registrar<IRepositorioPedidos>(() => new RepositorioPedidosSql(connStringSql));
c.Registrar<IGatewayCobros>(() => new GatewayMercadoPago("api-key-..."));

c.Registrar<ProcesadorPedidos>(() => new ProcesadorPedidos(
    c.Resolver<IEnviadorEmails>(),
    c.Resolver<IRepositorioPedidos>(),
    c.Resolver<IGatewayCobros>()));

var procesador = c.Resolver<ProcesadorPedidos>();

Esto ya es un service locator. Resolvió un par de cosas:

  • Las decisiones de construcción están centralizadas.
  • Las funciones de fábrica describen "cómo construir", no "construime ahora".
  • Pedir un ProcesadorPedidos ya no requiere conocer su grafo de dependencias.

Pero la función de fábrica del ProcesadorPedidos sigue siendo manual: tuve que listar sus tres dependencias y escribir tres Resolver<T>() a mano. Si una clase tiene diez dependencias, son diez líneas. Si una de ellas cambia, hay que tocar la función de fábrica. La repetición se redujo pero no desapareció.

6. Resolución automática por reflexión

El siguiente movimiento elimina la fábrica manual: el contenedor mira el constructor de la clase pedida y, para cada parámetro, se llama recursivamente.

public class ContenedorReflexivo {
    private readonly Dictionary<Type, Type> registro = new();

    public void Registrar<TInterfaz, TImpl>()
        where TImpl : TInterfaz
        => registro[typeof(TInterfaz)] = typeof(TImpl);

    public T Resolver<T>() => (T)Resolver(typeof(T));

    private object Resolver(Type tipo) {
        // Si está registrado como interfaz, busco la implementación.
        var tipoConcreto = registro.TryGetValue(tipo, out var impl) ? impl : tipo;

        // Asumo un solo constructor público (simplificación).
        var ctor = tipoConcreto.GetConstructors().Single();
        var parametros = ctor.GetParameters();

        // Resuelvo cada parámetro recursivamente.
        var argumentos = parametros
            .Select(p => Resolver(p.ParameterType))
            .ToArray();

        return ctor.Invoke(argumentos);
    }
}

Uso:

var c = new ContenedorReflexivo();
c.Registrar<IEnviadorEmails, EnviadorEmailsSmtp>();
c.Registrar<IRepositorioPedidos, RepositorioPedidosSql>();
c.Registrar<IGatewayCobros, GatewayMercadoPago>();

var procesador = c.Resolver<ProcesadorPedidos>();
// ↑ El contenedor mira el constructor de ProcesadorPedidos, ve que pide
//   tres interfaces, busca la implementación de cada una, las construye
//   recursivamente, y arma todo el árbol.

Esto ya es esencialmente lo que hace Microsoft.Extensions.DependencyInjection. Falta el manejo de configuración (cómo pasarle el connString al RepositorioPedidosSql), el manejo de lifetimes, y un montón de detalles de robustez. Pero la idea central está completa: el contenedor se gana la vida resolviendo grafos de dependencias por reflexión, a partir de un registro declarativo.

7. Lifetimes: por qué la duración importa

Imaginemos que RepositorioPedidosSql abre una conexión SQL en su constructor. ¿Cuántas instancias deberían existir?

  • Una por cada vez que alguien la pide. Una conexión nueva por cada llamada — derrocha conexiones, mata el pool de la base.
  • Una sola para toda la aplicación. La conexión queda viva todo el tiempo — pero no es thread-safe, varios requests concurrentes la pisan.
  • Una por cada operación de negocio. Una nueva conexión por request HTTP — esa request la usa todo el tiempo que dura, y al final se libera.

La tercera es la respuesta correcta para un repositorio. La primera puede ser correcta para un objeto sin estado. La segunda puede ser correcta para una caché en memoria.

El lifetime es una decisión que tiene que vivir en el composition root, no en la clase. La clase no sabe (ni tiene por qué saber) si es invocada por una request HTTP, por un job de fondo, o por un test. Su única obligación es funcionar correctamente para un uso. Decidir cuántas instancias hay y cuándo morir es problema del contenedor.

Tres lifetimes cubren prácticamente todos los casos:

        ┌─────────────────────────────────────────────────────────────┐
        │                                                             │
        │  Transient   →   nueva instancia cada vez que se pide       │
        │  Scoped      →   una instancia por scope                    │
        │  Singleton   →   una instancia para toda la app             │
        │                                                             │
        └─────────────────────────────────────────────────────────────┘

"Scope" es un concepto deliberadamente abstracto: un scope es un período de tiempo durante el cual ciertos servicios viven juntos y se liberan juntos. En ASP.NET Core, cada request HTTP es un scope. En una app de consola, vos creás los scopes manualmente. La utilidad es: "una instancia por unidad de trabajo, no importa cuántas veces la pidan adentro".

Con eso ya tenemos todos los conceptos. Pasamos a la API real.

Parte III — IServiceProvider real

8. La abstracción canónica

IServiceProvider está definida en System desde .NET Framework 1.0. Tiene un único método:

public interface IServiceProvider {
    object? GetService(Type serviceType);
}

Eso es todo. Le pedís un tipo, te devuelve una instancia (o null). La interfaz es deliberadamente minimalista: cualquier cosa que sepa producir objetos a partir de un Type puede implementarla.

Lo que cambia entre frameworks es la fábrica de IServiceProvider: cómo se configura, qué lifetimes soporta, cómo maneja scopes. La implementación oficial moderna vive en el paquete Microsoft.Extensions.DependencyInjection, y es la que usan ASP.NET Core, los Generic Host, los Worker Services, MAUI y básicamente todo lo nuevo de .NET. Cuando alguien dice "el contenedor de DI de .NET" se refiere a esa.

Esa implementación se usa en dos fases bien diferenciadas:

        ┌───────────────────────────┐
        │  Fase de configuración    │   IServiceCollection
        │  (registro de servicios)  │
        └─────────────┬─────────────┘
                      │
                      ▼ BuildServiceProvider()
                      │
        ┌─────────────┴─────────────┐
        │  Fase de ejecución        │   IServiceProvider
        │  (resolución de servicios)│
        └───────────────────────────┘

Antes del Build: registrás. Después del Build: resolvés. No se mezclan.

9. IServiceCollection: la fase de configuración

IServiceCollection es, conceptualmente, una List<ServiceDescriptor>. Cada descriptor dice tres cosas:

  • Qué tipo de servicio se está registrando (típicamente una interfaz).
  • Cómo construirlo (un tipo concreto, una instancia ya creada, o una factory).
  • Con qué lifetime.

La API expone tres métodos primarios, uno por lifetime, con varias sobrecargas:

var services = new ServiceCollection();

// Forma 1: por tipos. El contenedor instancia TImpl por reflexión.
services.AddScoped<IRepositorioPedidos, RepositorioPedidosSql>();

// Forma 2: factory que recibe el provider y devuelve la instancia.
//          Útil cuando hace falta configuración o lógica.
services.AddSingleton<IEnviadorEmails>(sp =>
{
    var config = sp.GetRequiredService<IConfiguration>();
    var smtp   = config.GetSection("Smtp").Get<ConfigSmtp>()!;
    return new EnviadorEmailsSmtp(smtp);
});

// Forma 3: instancia ya construida. Lifetime forzosamente singleton.
var clock = new ClockSistema();
services.AddSingleton<IClock>(clock);

// Forma 4: tipo concreto sin interfaz. Permitido pero menos común.
services.AddTransient<CalculadoraImpuestos>();

Detrás de escena, todas estas llamadas terminan agregando un ServiceDescriptor a la lista. La lista preserva orden y permite duplicados — el último gana en la mayoría de los casos, pero hay APIs como TryAddSingleton que sólo agregan si el servicio no estaba registrado, útil para configuración por convención.

Las extensiones de los frameworks aprovechan esto para exponer registros agrupados:

services.AddDbContext<MiContexto>(opt => opt.UseSqlServer(connString));
services.AddHttpClient<IClienteApi, ClienteApiHttp>();
services.AddAuthentication().AddJwtBearer();
services.AddControllers();

Cada uno de esos métodos registra internamente decenas de servicios relacionados. La fachada esconde un grafo de configuración complejo detrás de una llamada legible.

10. BuildServiceProvider: del registro al provider

Cuando se llama services.BuildServiceProvider(), ocurren varias cosas:

  1. La colección de descriptors queda congelada: el provider resultante no se ve afectado por modificaciones posteriores a la IServiceCollection.
  2. Se construye una caché interna que indexa los descriptors por tipo de servicio.
  3. Se prepara la maquinaria de scopes: cada scope mantiene su propia caché de instancias scoped.
  4. Se valida — si las opciones lo piden — que no haya servicios scoped resueltos como dependencia de servicios singleton (captive dependencies).
var provider = services.BuildServiceProvider(new ServiceProviderOptions {
    ValidateOnBuild = true, // Falla en Build si hay un servicio que no se puede resolver.
    ValidateScopes  = true, // Detecta captive dependencies en runtime.
});

En ASP.NET Core no se llama BuildServiceProvider directamente — lo hace el host por dentro durante el app.Build(). Llamarlo manualmente en Program.cs para resolver algo a mano antes del Run está permitido pero suele ser un olor: indica que la configuración no está siendo resuelta a través del flujo normal.

11. Resolución: GetService vs GetRequiredService

IServiceProvider tiene un método: GetService(Type). Sobre él, las extensiones del paquete agregan azúcar:

// Genérica: evita el cast.
T? GetService<T>();

// No-nullable: tira excepción si el servicio no está registrado.
T  GetRequiredService<T>();

// Para servicios registrados múltiples veces.
IEnumerable<T> GetServices<T>();

La diferencia entre GetService y GetRequiredService no es estética: uno devuelve null si el servicio no está registrado, el otro tira InvalidOperationException. Para servicios obligatorios (la mayoría), GetRequiredService es lo correcto — el null reference en el lugar equivocado, dos llamadas más adelante, es mucho peor que una excepción explícita acá.

Cuando se resuelve un servicio:

  • El provider busca el descriptor.
  • Si la instancia ya está cacheada (singleton, o scoped dentro del mismo scope), la devuelve.
  • Si no, mira el constructor de la implementación, resuelve cada parámetro recursivamente, instancia y cachea según el lifetime.
  • Si el tipo implementa IDisposable o IAsyncDisposable, el provider lo registra para liberarlo cuando el scope (o la app, para singletons) termine.

Ese último punto es importante: el contenedor maneja el ciclo de vida completo de los objetos que crea, incluyendo el dispose. Si vos pediste un servicio, no le hagas dispose vos — lo va a hacer el contenedor en el momento correcto. Si lo construiste vos y lo pasaste con AddSingleton(instancia), en cambio, el contenedor no le hace dispose: la documentación oficial pide explícitamente que el dueño de la instancia se ocupe de liberarla.

Parte IV — Lifetimes en profundidad

12. Transient

services.AddTransient<ICalculadoraImpuestos, CalculadoraImpuestos>();

Cada llamada a GetRequiredService<ICalculadoraImpuestos>() retorna una instancia nueva. Cada vez que un servicio recibe ICalculadoraImpuestos por constructor, recibe una instancia nueva. Si dentro de la misma request HTTP se pide cinco veces, hay cinco instancias.

Cuándo: servicios livianos, sin estado, sin recursos. Calculadoras, validadores, formateadores, mappers. Lo barato y desechable.

Cuándo no: cualquier cosa que abra una conexión, mantenga una caché, sostenga un buffer. Crear y descartar 50 conexiones SQL por request es una mala idea.

Cuidado con IDisposable: los servicios transient que implementan IDisposable quedan retenidos por el scope que los pidió hasta que ese scope termine. Si un singleton resuelve transient IDisposable repetidamente, esos transients se acumulan en el provider raíz hasta que la app termine — equivalente a un memory leak. Para servicios disposable, Scoped o Singleton son casi siempre mejor opción.

13. Singleton

services.AddSingleton<IClock, ClockSistema>();

Una sola instancia para toda la vida de la aplicación. Se construye la primera vez que se pide (o, con ValidateOnBuild, en el Build) y se reutiliza siempre.

Cuándo: servicios sin estado mutable o con estado seguro para concurrencia. Configuración. Caches en memoria con su propia sincronización. Loggers, factories, clientes HTTP de larga vida (HttpClient con IHttpClientFactory), reloj del sistema, generadores de IDs.

Cuándo no: cualquier cosa con estado mutable que no sea thread-safe. Un singleton en una app web es invocado desde decenas de requests concurrentes; si tiene un Dictionary sin lock, el bug es inevitable. Tampoco para servicios que dependan de servicios scoped (ver punto 15).

El detalle del HttpClient: durante años, el consejo fue "usá un solo HttpClient para toda la app". Eso era correcto pero no completo: un HttpClient singleton no respeta cambios de DNS y agota sockets. La solución moderna es IHttpClientFactory — singleton — que internamente maneja un pool de HttpMessageHandler con tiempo de vida controlado:

services.AddHttpClient<IClienteApi, ClienteApiHttp>(c =>
{
    c.BaseAddress = new Uri("https://api.example.com/");
    c.Timeout     = TimeSpan.FromSeconds(30);
});

ClienteApiHttp recibe un HttpClient cuya creación está optimizada por el factory. El consumidor no se entera del manejo de pools.

14. Scoped

services.AddScoped<IRepositorioPedidos, RepositorioPedidosSql>();

Una instancia por scope. En ASP.NET Core, cada request HTTP es un scope automático. Dentro de la misma request, todos los servicios que reciban IRepositorioPedidos reciben la misma instancia.

Esto es lo que hace que el patrón Unit of Work funcione naturalmente: un DbContext registrado como scoped es compartido por el repositorio, el servicio de aplicación y el manejador de transacción dentro de la misma request, pero es completamente independiente entre requests distintas.

Cuándo: prácticamente cualquier servicio que toque infraestructura por request. Repositorios. DbContext. Unit of Work. Servicios de aplicación que orquestan operaciones.

Cuándo no: en código que no tiene un scope natural. Una librería pura, una static class de utilidades, un singleton.

Fuera de ASP.NET Core: los scopes existen igual, pero hay que crearlos manualmente. Eso es lo que muestra la sección 18.

15. Captive dependencies: el bug clásico

Reglón de oro: un servicio nunca debe depender de otro de menor lifetime.

Si un singleton recibe un scoped en su constructor, ocurre lo siguiente: el contenedor resuelve el scoped una vez, en el momento de construir el singleton, y lo guarda dentro. A partir de ahí, el "scoped" deja de cambiar entre requests — quedó capturado por el singleton. Bug clásico, casi imposible de detectar a ojo:

public class ServicioCacheador {                 // SINGLETON
    private readonly IRepositorioPedidos repo;   // SCOPED — ¡problema!

    public ServicioCacheador(IRepositorioPedidos repo) => this.repo = repo;
    // repo es siempre la misma instancia, capturada en el primer uso.
}

La jerarquía de "puede depender de quién" es:

    ┌────────────────────────────────────────────────────────────────────┐
    │                                                                    │
    │   Singleton  ←  puede depender de  ←  Singleton                    │
    │   Scoped     ←  puede depender de  ←  Singleton, Scoped            │
    │   Transient  ←  puede depender de  ←  Singleton, Scoped, Transient │
    │                                                                    │
    └────────────────────────────────────────────────────────────────────┘

Leyendo: un singleton sólo puede depender de singletons; un scoped puede depender de singletons y scopeds; un transient puede depender de cualquier cosa.

Parte V — Patrones de uso

16. Constructor injection (la forma canónica)

El 95% del uso de DI en una aplicación bien diseñada se ve así:

public class ProcesadorPedidos {
    private readonly IEnviadorEmails          emails;
    private readonly IRepositorioPedidos      pedidos;
    private readonly IGatewayCobros           cobros;
    private readonly ILogger<ProcesadorPedidos> log;

    public ProcesadorPedidos(
        IEnviadorEmails emails,
        IRepositorioPedidos pedidos,
        IGatewayCobros cobros,
        ILogger<ProcesadorPedidos> log) {
        this.emails  = emails;
        this.pedidos = pedidos;
        this.cobros  = cobros;
        this.log     = log;
    }

    public void Procesar(Pedido p) {
        log.LogInformation("Procesando pedido {Id}", p.Id);
        cobros.Cobrar(p);
        pedidos.Guardar(p);
        emails.EnviarConfirmacion(p);
    }
}

Tres reglas:

  1. Las dependencias van en el constructor. Nunca como propiedades públicas, nunca asignadas después.
  2. Se guardan como readonly. Una vez asignadas, no cambian.
  3. El constructor no hace nada más. Asignaciones, opcionalmente algún ?? para valores por defecto, nada más. Sin queries a base, sin lecturas de archivo, sin lógica.

La regla de oro derivada: el constructor de un servicio debe ser barato. Si tarda 200ms en correr, alguien hizo algo mal.

Con C# 12+, primary constructors hacen esto más conciso:

public class ProcesadorPedidos(IEnviadorEmails emails, IRepositorioPedidos pedidos, IGatewayCobros cobros, ILogger<ProcesadorPedidos> log) {
    public void Procesar(Pedido p) {
        log.LogInformation("Procesando pedido {Id}", p.Id);
        cobros.Cobrar(p);
        pedidos.Guardar(p);
        emails.EnviarConfirmacion(p);
    }
}

Mismo comportamiento, menos boilerplate.

17. Factories cuando el tipo depende del runtime

El constructor injection asume que sabés en tiempo de compilación qué dependencias necesitás. A veces no es así: dependés de un valor que sólo existe en runtime para decidir qué instancia construir.

Ejemplo: un procesador de pagos que recibe el método de pago como string ("tarjeta", "transferencia", "efectivo") y necesita una estrategia distinta para cada uno.

Mal: meter la decisión en el procesador.

public class ProcesadorPagos {
    public void Procesar(Pago pago) {
        if      (pago.Metodo == "tarjeta")       /* ... */
        else if (pago.Metodo == "transferencia") /* ... */
        // Más métodos = más ifs = el OCP llora.
    }
}

Bien: factory que resuelve la implementación correcta.

public interface IEstrategiaPago { 
    void Procesar(Pago pago); 
}

public class EstrategiaTarjeta       : IEstrategiaPago { /* ... */ }
public class EstrategiaTransferencia : IEstrategiaPago { /* ... */ }
public class EstrategiaEfectivo      : IEstrategiaPago { /* ... */ }

public interface IFactoryEstrategiaPago {
    IEstrategiaPago Para(string metodo);
}

public class FactoryEstrategiaPago(IServiceProvider sp) : IFactoryEstrategiaPago {
    public IEstrategiaPago Para(string metodo) => metodo switch {
        "tarjeta"       => sp.GetRequiredService<EstrategiaTarjeta>(),
        "transferencia" => sp.GetRequiredService<EstrategiaTransferencia>(),
        "efectivo"      => sp.GetRequiredService<EstrategiaEfectivo>(),
        _ => throw new ArgumentException($"Método de pago desconocido: {metodo}")
    };
}

Registro:

services.AddScoped<EstrategiaTarjeta>();
services.AddScoped<EstrategiaTransferencia>();
services.AddScoped<EstrategiaEfectivo>();
services.AddScoped<IFactoryEstrategiaPago, FactoryEstrategiaPago>();

Uso:

public class ProcesadorPagos(IFactoryEstrategiaPago factory) {
    public void Procesar(Pago pago) => factory.Para(pago.Metodo).Procesar(pago);
}

Esto es el caso legítimo de inyectar IServiceProvider: dentro de una factory cuya razón de existir es decidir qué instancia construir basándose en datos de runtime. La clase de negocio (ProcesadorPagos) no toca el provider — sólo la factory lo hace, y la factory es infraestructura.

Una variante más tipada cuando los tipos están cerrados: registrar todas las estrategias bajo la misma interfaz y usar IEnumerable:

services.AddScoped<IEstrategiaPago, EstrategiaTarjeta>();
services.AddScoped<IEstrategiaPago, EstrategiaTransferencia>();
services.AddScoped<IEstrategiaPago, EstrategiaEfectivo>();

public class FactoryEstrategiaPago(IEnumerable<IEstrategiaPago> estrategias) : IFactoryEstrategiaPago {
    public IEstrategiaPago Para(string metodo) => estrategias.Single(e => e.Soporta(metodo));
}

Acá ni siquiera hace falta el provider: el contenedor inyecta directamente todas las implementaciones registradas.

18. IServiceScopeFactory en background services

Un BackgroundService corre fuera de cualquier request HTTP — vive todo el tiempo que vive la aplicación. Por lo tanto, el BackgroundService mismo es un singleton. Pero típicamente necesita usar servicios scoped (un DbContext, un repositorio).

Si lo intenta directamente:

public class ServicioReintentos(IRepositorioPedidos repo) : BackgroundService {
    // ¡Captive dependency! repo queda capturado para siempre.
}

Lo correcto es inyectar IServiceScopeFactory y crear un scope manual por cada iteración del loop:

public class ServicioReintentos(IServiceScopeFactory scopeFactory, ILogger<ServicioReintentos> log) : BackgroundService {
    
    protected override async Task ExecuteAsync(CancellationToken ct) {
        while (!ct.IsCancellationRequested) {
            using var scope = scopeFactory.CreateScope();
            var repo = scope.ServiceProvider.GetRequiredService<IRepositorioPedidos>();

            try {
                var pendientes = await repo.PedidosFallidosAsync(ct);
                foreach (var p in pendientes)
                    await ReintentarAsync(p, scope.ServiceProvider, ct);
            } catch (Exception ex) {
                log.LogError(ex, "Error en ciclo de reintentos");
            }

            await Task.Delay(TimeSpan.FromMinutes(5), ct);
        }
    }
}

El using sobre el scope garantiza que al salir del bloque se llame Dispose() en todos los servicios scoped/transient-disposable creados durante esa iteración. La próxima iteración crea un scope nuevo, con un DbContext fresco, sin leaks.

Esa es la versión correcta del patrón "ejecutar lógica que necesita scope desde un singleton". IServiceScopeFactory es la única dependencia "tipo provider" que se inyecta directamente, y sólo en casos infraestructurales como este.

19. IOptions<T>: configuración como dependencia

Un caso particular de inyección que aparece en casi toda app moderna. Imaginemos EnviadorEmailsSmtp que necesita host, puerto, usuario, contraseña.

Forma manual:

services.AddSingleton<IEnviadorEmails>(sp =>
{
    var config = sp.GetRequiredService<IConfiguration>();
    var smtp = new ConfigSmtp(
        Host:     config["Smtp:Host"]!,
        Puerto:   int.Parse(config["Smtp:Puerto"]!),
        Usuario:  config["Smtp:Usuario"]!,
        Password: config["Smtp:Password"]!);
    return new EnviadorEmailsSmtp(smtp);
});

Funciona pero es repetitivo y frágil (strings, casts, ifs por nullable).

Con IOptions<T>:

// 1. Definir la clase de configuración.
public class ConfigSmtp {
    public string Host     { get; set; } = "";
    public int Puerto      { get; set; }
    public string Usuario  { get; set; } = "";
    public string Password { get; set; } = "";
}

// 2. Registrar binding desde appsettings.json:
//    {
//      "Smtp": { "Host": "...", "Puerto": 587, "Usuario": "...", "Password": "..." }
//    }
services.Configure<ConfigSmtp>(builder.Configuration.GetSection("Smtp"));

// 3. La clase consume IOptions<T>:
public class EnviadorEmailsSmtp(IOptions<ConfigSmtp> opt) : IEnviadorEmails {
    private readonly ConfigSmtp cfg = opt.Value;

    public Task EnviarAsync(Email e, CancellationToken ct) {
        // usar cfg.Host, cfg.Puerto, etc.
    }
}

IOptions<T> es una variante registrada como singleton: Value se calcula una vez y se cachea. Hay variantes:

  • IOptionsSnapshot<T> — scoped, releé el binding por scope. Útil para configuración que cambia y debe verse en la próxima request.
  • IOptionsMonitor<T> — singleton con notificaciones cuando la configuración cambia en runtime.

Para casos simples, IOptions<T> alcanza.

Parte VI — Anti-patrones

20. Service Locator

El antipatrón fundamental: en lugar de declarar las dependencias en el constructor, la clase recibe IServiceProvider y resuelve a demanda.

// ✗ Mal.
public class ProcesadorPedidos(IServiceProvider sp) {
    public void Procesar(Pedido p) {
        var repo    = sp.GetRequiredService<IRepositorioPedidos>();
        var cobros  = sp.GetRequiredService<IGatewayCobros>();
        var emails  = sp.GetRequiredService<IEnviadorEmails>();
        // ...
    }
}

Tres problemas, en orden de gravedad:

Las dependencias quedan ocultas. Mirando el constructor, esta clase parece depender solamente de "el sistema". Para saber qué usa de verdad, hay que leer el cuerpo de cada método. Si una dependencia se agrega o se saca, no hay nada en la firma de la clase que cambie. La explicitud que ganamos con constructor injection se perdió.

Los tests se complican. Un test unitario tiene que mockear un IServiceProvider completo, no las tres dependencias reales. Existen helpers que lo facilitan, pero el ruido de configuración por test crece.

El acoplamiento al contenedor se filtra. La clase ya no es portable a otros lugares — depende de la abstracción del contenedor en sí, no sólo de los servicios. Si mañana el código se mueve a una librería que no tiene contenedor (una azure function, una unidad de procesamiento que recibe sus dependencias por constructor en código manual), hay que reescribirla.

La regla operativa: IServiceProvider no aparece en código de aplicación. Aparece en código de infraestructura (factories como la del punto 17, scope factories como las del punto 18, middleware, filtros, frameworks). El usuario común del contenedor no lo ve.

21. Capturar el provider raíz

Una variante particularmente nociva del Service Locator: guardar una referencia estática al provider raíz para usarla desde cualquier parte del código.

// ✗ Mal.
public static class ServiceContainer {
    public static IServiceProvider Provider { get; set; } = null!;
}

// En Program.cs:
ServiceContainer.Provider = app.Services;

// En cualquier clase del proyecto:
var repo = ServiceContainer.Provider.GetRequiredService<IRepositorioPedidos>();

Aparte de los problemas del Service Locator común, este patrón agrega:

  • Un singleton global que se vuelve dependencia oculta de todo el proyecto.
  • Resolución desde el scope raíz, lo que rompe los scopes de request: pedir un servicio scoped desde ServiceContainer.Provider cuando estás dentro de una request HTTP devuelve una instancia que no comparte scope con la request, generando inconsistencias sutiles (ej: dos DbContext distintos en la misma operación).
  • Imposible de testear: el estado global persiste entre tests.

Cuando aparezca este patrón en una code review, eliminarlo sin contemplaciones es la respuesta correcta. Reemplazarlo por constructor injection en cada caso requiere más trabajo, pero ese trabajo es exactamente lo que estaba evitando.

Parte VII — Síntesis

22. Filosofía de diseño en una página

IServiceProvider materializa varios principios al mismo tiempo. Vale la pena enumerarlos juntos:

Inversión de dependencias (DIP). Las clases dependen de abstracciones — interfaces — y el contenedor inyecta las implementaciones concretas en el composition root. La clase de negocio no decide qué IRepositorioPedidos usar; el contenedor lo decide al armar el grafo.

Inversión de control. La clase no busca lo que necesita; le es entregado. La clase no controla cuándo se construye, cómo se construye, cuántas veces existe ni cuándo se libera. Todo eso es política del contenedor. La clase sólo se concentra en su responsabilidad.

Separación de configuración y ejecución. La fase IServiceCollection es declarativa: dice qué hay disponible. La fase IServiceProvider es ejecutiva: produce instancias. No se mezclan. Esto hace que la configuración del sistema sea una unidad legible, separada del runtime.

Composición sobre herencia. El patrón natural en DI es recibir colaboradores y delegar, no extender clases base. Decoradores, factories, adapters, observers — todos los patrones que componen comportamiento — encajan limpiamente en un contenedor.

Cierre y apertura (OCP). Agregar una implementación nueva no requiere tocar las que existen: registrarla en el contenedor alcanza. Cambiar de EnviadorEmailsSmtp a EnviadorEmailsSendgrid es una línea de configuración, no una refactorización.

Política de lifetimes en un solo lugar. Decidir si algo es transient, scoped o singleton es una decisión de configuración, no una decisión de diseño de la clase. La clase corre igual con cualquiera de los tres; el contenedor decide cuál tiene sentido para esta aplicación.

La síntesis en una frase:

El contenedor de DI es la respuesta de la plataforma a una pregunta vieja: "¿quién construye a quién?". La respuesta es: nadie en particular dentro del grafo, todos en el composition root, y cada cosa el tiempo que corresponda.

23. Checklist operativa

Para registrar:

  • ¿La clase necesita una conexión, una caché, un DbContext? → Scoped.
  • ¿La clase no tiene estado mutable y es cara de construir? → Singleton.
  • ¿La clase es liviana, sin recursos, sin estado? → Transient.
  • ¿La clase mantiene estado mutable compartido entre threads? → Singleton sólo si está blindada con sincronización; en duda, Scoped.
  • ¿El servicio implementa IDisposable? → Evitar Transient.

Para usar:

  • ¿La clase es de negocio o aplicación? → Constructor injection con interfaces.
  • ¿Necesita decidir qué construir según un valor de runtime? → Factory que recibe IServiceProvider.
  • ¿Es un BackgroundService que necesita servicios scoped? → IServiceScopeFactory y using por iteración.
  • ¿Necesita configuración del appsettings.json? → IOptions<T> con services.Configure<T>().
  • ¿Está pensando en hacer sp.GetRequiredService<...>() desde adentro de una clase de negocio? → No.

Para diagnosticar:

  • ¿Falla con "cannot consume scoped service from singleton"? → Captive dependency. Convertir el singleton en scoped, o usar IServiceScopeFactory.
  • ¿Las instancias scoped no se comparten dentro de una operación? → Está pidiéndolas desde el provider raíz en lugar del de la request. Verificar dónde se está creando el scope.
  • ¿Memoria que crece sin parar y nunca se libera? → Servicios transient IDisposable resueltos desde el provider raíz. Cambiar lifetime o crear scopes manuales.
  • ¿Build falla en desarrollo pero no en producción? → ValidateOnBuild está activo en Development y detecta servicios mal registrados que en producción explotan después.

                │                                                │
                │   1. ¿Cómo se construye cada cosa?             │
                │   2. ¿Cuánto tiempo tiene que vivir?           │

IServiceProvider no es magia: es un diccionario sofisticado que sabe construir grafos de objetos respetando políticas de duración. Entender que es exactamente eso — y que la API "real" de .NET no es muy distinta del ContenedorReflexivo de la sección 6, sólo más robusto y con lifetimes — desmitifica casi todo lo que parece complejo cuando se la encuentra por primera vez.

24. Receta 1 — Minimal API: inyectar servicios en endpoints sin tocar el provider

Cuándo usar esta receta

Cuando estás armando una API HTTP y querés que los handlers reciban servicios automáticamente como parámetros, sin hacer app.Services.GetRequiredService(...) adentro de cada endpoint.

Objetivo

Lograr este estilo:

app.MapPost("/pedidos", async (
    CrearPedidoDto dto,
    IProcesadorPedidos procesador,
    ILogger<Program> log) =>
{
    var id = await procesador.CrearAsync(dto);
    log.LogInformation("Pedido creado: {Id}", id);
    return Results.Created($"/pedidos/{id}", new { id });
});

El endpoint no crea dependencias, no resuelve el provider y no sabe si atrás hay SQL, JSON o una API externa.

Paso 1 — registrar servicios antes del Build

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddScoped<IRepositorioPedidos, RepositorioPedidosEf>();
builder.Services.AddScoped<IProcesadorPedidos, ProcesadorPedidos>();
builder.Services.AddLogging();

var app = builder.Build();

Regla práctica:

  • AddScoped para repositorios y servicios de aplicación usados por request.
  • AddSingleton para servicios globales sin estado mutable peligroso.
  • AddTransient para objetos chicos, livianos y sin recursos.

Si no sabés cuál elegir para un servicio que toca base de datos, la respuesta casi siempre es scoped.

Paso 2 — escribir el servicio con constructor injection

public interface IProcesadorPedidos {
    Task<int> CrearAsync(CrearPedidoDto dto);
}

public class ProcesadorPedidos : IProcesadorPedidos {
    private readonly IRepositorioPedidos repo;
    private readonly ILogger<ProcesadorPedidos> log;

    public ProcesadorPedidos(
        IRepositorioPedidos repo,
        ILogger<ProcesadorPedidos> log) {
        this.repo = repo;
        this.log = log;
    }

    public async Task<int> CrearAsync(CrearPedidoDto dto) {
        var pedido = new Pedido { Cliente = dto.Cliente, Total = dto.Total };
        await repo.GuardarAsync(pedido);
        log.LogInformation("Pedido persistido para {Cliente}", dto.Cliente);
        return pedido.Id;
    }
}

Indicaciones prácticas:

  • Si una clase necesita IServiceProvider, sospechá.
  • Si una clase de negocio hace new SqlConnection(...), sospechá más fuerte.
  • Si el constructor empieza a tener 8 dependencias, no metas el provider: revisá responsabilidades.

Paso 3 — pedir el servicio directamente en el handler

app.MapPost("/pedidos", async (
    CrearPedidoDto dto,
    IProcesadorPedidos procesador) =>
{
    var id = await procesador.CrearAsync(dto);
    return Results.Created($"/pedidos/{id}", new { id });
});

Minimal API hace este trabajo por vos:

  • toma el body y lo bindea a CrearPedidoDto
  • toma IProcesadorPedidos del contenedor
  • ejecuta el handler con ambas cosas listas

Paso 4 — usar app.Services sólo en arranque, no dentro de endpoints

Caso legítimo: querés correr una inicialización al arrancar.

using (var scope = app.Services.CreateScope()) {
    var repo = scope.ServiceProvider.GetRequiredService<IRepositorioPedidos>();
    await repo.SemillarDatosAsync();
}

Eso está bien porque ocurre en el borde de la app, no en la lógica del endpoint.

Lo que no conviene hacer

app.MapPost("/pedidos", async (CrearPedidoDto dto) => {
    var procesador = app.Services.GetRequiredService<IProcesadorPedidos>();
    var id = await procesador.CrearAsync(dto);
    return Results.Created($"/pedidos/{id}", new { id });
});

Funciona, pero te compra problemas:

  • el handler oculta sus dependencias
  • los tests quedan más incómodos
  • el endpoint queda atado al contenedor concreto

Checklist rápida

  • ¿El endpoint recibe interfaces/servicios como parámetros? → bien.
  • ¿Las implementaciones se registran antes de builder.Build()? → bien.
  • ¿Usás app.Services sólo para inicialización de arranque? → bien.
  • ¿Resolviste servicios adentro del handler sin necesidad? → mal olor.

25. Receta 2 — EF Core: registrar DbContext, inicializar la base y consumirlo bien

Cuándo usar esta receta

Cuando querés usar EF Core en una app ASP.NET Core o Minimal API y necesitás saber:

  • cómo registrar el contexto
  • cómo crear la base al arrancar
  • cómo consumir el contexto sin romper los lifetimes

Objetivo

Tener esta cadena completa funcionando:

  1. registrar DbContext
  2. crear la base si hace falta
  3. inyectar el contexto o un repositorio en handlers/servicios
  4. evitar errores típicos de lifetime

Paso 1 — registrar el contexto como scoped

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddDbContext<TareasContext>(options =>
    options.UseSqlite("Data Source=tareas.db"));

No hace falta poner AddScoped<TareasContext>(): AddDbContext ya lo registra como scoped, que es exactamente lo que suele necesitarse.

Paso 2 — si querés una capa intermedia, registrar el repositorio

builder.Services.AddScoped<IRepositorioTareas, RepositorioTareasEf>();

Implementación:

public class RepositorioTareasEf : IRepositorioTareas {
    private readonly TareasContext db;

    public RepositorioTareasEf(TareasContext db) {
        this.db = db;
    }

    public Task<List<Tarea>> ListarAsync() => db.Tareas.ToListAsync();

    public async Task AgregarAsync(Tarea tarea) {
        db.Tareas.Add(tarea);
        await db.SaveChangesAsync();
    }
}

Receta mental:

  • DbContext → infraestructura
  • repositorio → opcional pero útil para desacoplar
  • servicio de aplicación → usa el repositorio
  • handler/controller → usa el servicio

Paso 3 — crear la base al arrancar usando un scope manual

var app = builder.Build();

using (var scope = app.Services.CreateScope()) {
    var db = scope.ServiceProvider.GetRequiredService<TareasContext>();
    db.Database.EnsureCreated();
}

Esto es importante porque TareasContext es scoped. Si lo pedís desde el provider raíz sin scope, empezás a jugar en modo difícil.

Paso 4 — consumir el contexto o el repositorio por inyección

Opción A: el handler recibe el contexto directo.

app.MapGet("/tareas", async (TareasContext db) =>
    await db.Tareas.AsNoTracking().ToListAsync());

Opción B: el handler recibe un repositorio.

app.MapGet("/tareas", async (IRepositorioTareas repo) =>
    await repo.ListarAsync());

Ambas son válidas. Regla práctica:

  • proyecto pequeño, demo, TP corto → inyectar DbContext directo puede estar bien
  • proyecto con lógica de negocio más seria → preferir servicio o repositorio

Paso 5 — elegir bien el lifetime de lo que usa DbContext

Si una clase recibe TareasContext, esa clase no debería ser singleton.

// ✓ bien
builder.Services.AddScoped<IRepositorioTareas, RepositorioTareasEf>();

// ✗ mal
builder.Services.AddSingleton<IRepositorioTareas, RepositorioTareasEf>();

Si registrás el repositorio como singleton, estás capturando un servicio scoped dentro de uno más largo. El contenedor te lo va a reprochar — con razón.

Lo que no conviene hacer

Mal 1: construir el contexto a mano dentro del handler.

app.MapGet("/tareas", async () => {
    using var db = new TareasContext();
    return await db.Tareas.ToListAsync();
});

Para una demo mínima puede "andar", pero estás evitando el contenedor, duplicando configuración y perdiendo coherencia con el resto de la app.

Mal 2: guardar un DbContext en una static o singleton.

Eso casi siempre termina en bugs de concurrencia, estados mezclados o errores de tracking.

Checklist rápida

  • ¿Registraste el contexto con AddDbContext? → bien.
  • ¿La inicialización de base ocurre dentro de CreateScope()? → bien.
  • ¿Los servicios que usan DbContext son scoped? → bien.
  • ¿Estás haciendo new MiContexto() en cualquier lado? → revisar.

26. Receta 3 — BackgroundService: usar servicios scoped sin romper lifetimes

Cuándo usar esta receta

Cuando tenés un worker, un cron interno o un BackgroundService que corre cada cierto tiempo y necesita usar repositorios, DbContext o cualquier otro servicio scoped.

Problema típico

Un BackgroundService vive toda la app, o sea: es singleton. Entonces esto está mal:

public class WorkerMails : BackgroundService {
    private readonly TareasContext db;

    public WorkerMails(TareasContext db) {
        this.db = db;
    }
}

TareasContext es scoped. El worker lo captura para siempre. Lifetime roto, bug servido.

Receta correcta

Paso 1: inyectar IServiceScopeFactory.

public class WorkerMails : BackgroundService {
    private readonly IServiceScopeFactory scopeFactory;
    private readonly ILogger<WorkerMails> log;

    public WorkerMails(
        IServiceScopeFactory scopeFactory,
        ILogger<WorkerMails> log) {
        this.scopeFactory = scopeFactory;
        this.log = log;
    }

    protected override async Task ExecuteAsync(CancellationToken ct) {
        while (!ct.IsCancellationRequested) {
            using var scope = scopeFactory.CreateScope();
            var repo = scope.ServiceProvider.GetRequiredService<IRepositorioTareas>();

            try {
                await repo.ProcesarPendientesAsync(ct);
            } catch (Exception ex) {
                log.LogError(ex, "Error procesando pendientes");
            }

            await Task.Delay(TimeSpan.FromMinutes(1), ct);
        }
    }
}

Paso 2: registrar el worker y los servicios normales.

builder.Services.AddScoped<IRepositorioTareas, RepositorioTareasEf>();
builder.Services.AddHostedService<WorkerMails>();

Regla operativa

Cada iteración importante del worker debería abrir su propio scope si va a usar servicios scoped.

Pensalo así:

  • request HTTP → el framework crea un scope
  • worker/background → vos creás el scope

Lo que sí y lo que no

Sí: usar scopeFactory.CreateScope() cuando el trabajo necesita repositorios, contextos o servicios scoped.

Sí: resolver servicios scoped dentro del bloque using.

No: guardar el servicio scoped en un campo del worker.

No: reutilizar el mismo scope para toda la vida del proceso.

Checklist rápida

  • ¿El worker inyecta IServiceScopeFactory? → bien.
  • ¿Cada ciclo crea y libera un scope? → bien.
  • ¿Resolviste el repositorio dentro del scope? → bien.
  • ¿Guardaste un DbContext en un campo del worker? → mal.

Estas recetas no reemplazan la explicación conceptual anterior: la aterrizan. Si recordás una sola regla práctica después de leer este apunte, que sea esta:

En el código de aplicación normal, pedí dependencias por constructor o por parámetros del framework. IServiceProvider dejalo para infraestructura, factories y scopes manuales.
