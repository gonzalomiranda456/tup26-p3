# Model Context Protocol (MCP) con el SDK Oficial de C#/.NET

El SDK oficial de .NET para el Model Context Protocol (MCP) es desarrollado y mantenido en colaboración por Microsoft, Anthropic y la comunidad de código abierto (construido sobre la base de la librería fundacional `McpDotNet`). 

Permite conectar modelos de lenguaje (LLMs) con tus herramientas, bases de datos y recursos en el ecosistema de .NET de manera estandarizada y segura.

---

## 1. Paquetes de NuGet Oficiales

El SDK se divide en tres piezas según las necesidades de tu arquitectura:

1. **`ModelContextProtocol.Core`**: Contiene las interfaces y modelos de datos mínimos del protocolo. Ideal si solo estás construyendo un cliente liviano.
2. **`ModelContextProtocol`**: Paquete principal que incluye la integración con Hosting genérico de .NET e Inyección de Dependencias (`IServiceCollection`). Recomendado para servidores basados en consola (`stdio`).
3. **`ModelContextProtocol.AspNetCore`**: Diseñado específicamente para alojar servidores MCP sobre HTTP, integrando el transporte oficial en ASP.NET Core de forma nativa.

---

## 2. Cómo Definir Herramientas (Tools) con Atributos

A diferencia del enfoque manual donde interpretamos el JSON en un switch, el SDK oficial utiliza **Reflexión** y **Atributos** para mapear automáticamente métodos de C# a herramientas del protocolo MCP.

```csharp
using System.ComponentModel;
using ModelContextProtocol.Server; // Contiene los atributos del servidor

[McpServerToolType] // 1. Marca la clase como contenedora de herramientas
public class MisHerramientas {
    [McpServerTool] // 2. Registra este método como una herramienta MCP
    [Description("Suma dos números enteros y retorna el resultado.")] // 3. ¡Esencial para que el LLM sepa qué hace!
    public int Sumar(int a, int b) {
        return a + b;
    }
}
```

> [!IMPORTANT]
> El atributo `[Description]` de `System.ComponentModel` es crítico. El SDK lo utiliza para generar la descripción en el JSON de descubrimiento que lee el LLM. Si omites la descripción, el modelo de IA no sabrá cuándo usar tu herramienta.

## 3. Implementación del Servidor HTTP

Para exponer tus herramientas en la web (usando la especificación oficial de transporte HTTP de MCP), utilizamos el paquete `ModelContextProtocol.AspNetCore` dentro de un archivo base minimalista (*file-based*):

```csharp
#:sdk Microsoft.NET.Sdk.Web
#:package ModelContextProtocol.AspNetCore --prerelease

using System.ComponentModel;
using Microsoft.AspNetCore.Builder;
using Microsoft.Extensions.DependencyInjection;
using ModelContextProtocol.Server;

var builder = WebApplication.CreateBuilder(args);
builder.WebHost.UseUrls("http://localhost:5003");

// Registrar el motor MCP y vincular la clase de herramientas
builder.Services.AddMcpServer()
    .WithHttpServerTransport()
    .WithTool<MisHerramientas>();

var app = builder.Build();

// Mapea automáticamente los endpoints del estándar (/sse, /messages, etc.)
app.MapMcpEndpoints();

await app.RunAsync();

// Clase contenedora de herramientas
public class MisHerramientas {
    [McpServerTool]
    [Description("Saluda a un usuario por su nombre.")]
    public string Saludar(string nombre) {
        return $"Hola, {nombre}! (desde el SDK oficial de Microsoft)";
    }
}
```

## 4. Implementación del Cliente MCP

Para consumir el servidor HTTP recién creado, utilizamos un cliente oficial configurado con el transporte HTTP de cliente (`HttpClientTransport`):

```csharp
#:sdk Microsoft.NET.Sdk
#:package ModelContextProtocol --prerelease

using System;
using System.Threading.Tasks;
using ModelContextProtocol.Client;
using ModelContextProtocol.Core.Transport; // Contiene HttpClientTransport y opciones

Console.WriteLine("=== CLIENTE MCP OFICIAL DE .NET ===");

// 1. Configuramos el transporte HTTP apuntando al endpoint del servidor
var opcionesTransporte = new HttpClientTransportOptions {
    Endpoint = new Uri("http://localhost:5003/mcp")
};
var transporte = new HttpClientTransport(opcionesTransporte);

// 2. Creamos y conectamos el cliente
await using var cliente = await McpClient.CreateAsync(transporte);
Console.WriteLine("✓ Conexión establecida e inicializada con el servidor.");

// 3. Consultamos la lista de herramientas disponibles en el servidor
var respuestaTools = await cliente.ListToolsAsync();
Console.WriteLine("\nHerramientas descubiertas en el servidor:");
foreach (var tool in respuestaTools.Tools) {
    Console.WriteLine($"- Nombre: {tool.Name} | Descripción: {tool.Description}");
}

// 4. Invocamos la herramienta 'Saludar'
Console.WriteLine("\nInvocando herramienta 'Saludar'...");
var resultado = await cliente.CallToolAsync("Saludar", new { nombre = "Ada Lovelace" });

// 5. Mostramos el texto devuelto
Console.WriteLine($"Respuesta del servidor: {resultado.Content[0].Text}");
```

## 5. Cuándo usar Stdio vs. HTTP

En el SDK oficial, la elección del canal de comunicación (*transport*) se realiza con un solo método de configuración en el builder del servidor:

* **`.WithStdioServerTransport()`**: El servidor habla a través de la consola (`stdin/stdout`). Es la opción estándar si vas a conectar el servidor a un cliente local como **Claude Desktop**, **Cursor IDE** o **Windsurf**, que inician tu servidor de C# como un proceso secundario y leen su salida directamente.
* **`.WithHttpServerTransport()`**: El servidor expone una API web. Adecuado si tu servidor MCP correrá en la nube (como Azure, AWS o Docker) para dar servicio a múltiples agentes de forma remota.
