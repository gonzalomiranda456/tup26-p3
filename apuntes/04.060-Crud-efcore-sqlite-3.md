## ¿Qué vamos a construir?

Una agenda de contactos de línea de comandos donde cada contacto puede tener:

- Un **nombre**.
- Una **categoría** (Familia, Trabajo, Amigos...).
- Varias **direcciones**, donde una dirección puede ser un teléfono o un email.

Nos enfocamos a propósito en consola: sin formularios, sin UI, sin frameworks web. Queremos que toda la atención vaya al **modelado de datos y al acceso a la base**. Una vez que entendés EF Core en consola, usarlo en MAUI, ASP.NET o lo que sea es *exactamente el mismo código* — cambia lo de alrededor.

El objetivo de esta clase es entender **qué es un ORM y cómo trabaja con relaciones**. Para eso elegimos un modelo con tres tablas relacionadas: es lo suficientemente simple para entenderlo de un vistazo, y lo suficientemente rico para ver aparecer las relaciones 1:N y N:1, las foreign keys, la carga de datos relacionados, y el borrado en cascada.

Lo que queremos entender al final:

- Qué es un ORM y cuándo conviene usarlo.
- Cómo se traduce una clase C# en una tabla SQL.
- Cómo se traducen las relaciones entre clases en foreign keys.
- Qué es un `DbContext` y por qué su ciclo de vida importa.
- Cómo funciona el *change tracking*.
- Cómo se cargan datos relacionados con `Include`.
- Dónde está el límite entre "dejá que EF Core decida" y "necesito pensar el SQL".

> **Una aclaración importante**. En esta clase NO vamos a usar migraciones. Vamos a dejar que EF Core cree la base automáticamente con `EnsureCreated()`. Las migraciones son una pieza fundamental cuando un proyecto evoluciona, pero introducirlas hoy agrega ruido que nos aleja del objetivo. Las veremos en la próxima clase, con toda la atención que se merecen.

## Parte 1 — Primeros principios

Antes de tocar código, necesitás tener claros algunos conceptos. Si te los saltás, vas a terminar copiando código sin entender qué hace.

### 1.1 — El problema del impedance mismatch

Tu programa piensa en **objetos**: un `Contacto` con propiedades, referencias a otros objetos, listas, herencia. Tu base de datos piensa en **tablas**: filas y columnas, claves primarias, foreign keys, sin polimorfismo.

Los dos modelos no coinciden. Este desajuste tiene un nombre clásico: *object-relational impedance mismatch*. Si lo resolvés a mano, vas a terminar escribiendo decenas de funciones aburridas como:

```csharp
var cmd = new SqliteCommand("SELECT Id, Nombre FROM Contactos WHERE Id = @id");
cmd.Parameters.AddWithValue("@id", id);
using var reader = cmd.ExecuteReader();
if (reader.Read()) {
    var contacto = new Contacto {
        Id = reader.GetInt32(0),
        Nombre = reader.GetString(1),
    };
    // ahora otra query para traer las direcciones...
    // y otra para la categoría...
    return contacto;
}
```

Multiplicá eso por cada tabla, cada operación, cada cambio de schema, cada join. Cuando agregás relaciones (como en este ejercicio) el problema se amplifica: tenés que hacer varias queries y unir los datos a mano.

Un **ORM** (Object-Relational Mapper) automatiza esa traducción. Vos trabajás con objetos; la librería traduce a SQL por vos, y sabe cómo seguir las referencias entre ellos.

### 1.2 — Qué es Entity Framework Core

EF Core es el ORM oficial de Microsoft para .NET. Hace cuatro cosas principales:

1. **Mapea clases a tablas.** Una clase `Contacto` se convierte en la tabla `Contactos`. Sus propiedades se vuelven columnas. EF Core infiere los tipos, las claves, los nombres.
2. **Traduce LINQ a SQL.** Escribís `db.Contactos.Where(c => c.Nombre.StartsWith("A")).ToList()` y EF Core genera el `SELECT ... WHERE Nombre LIKE 'A%'` por vos.
3. **Entiende relaciones.** Si `Contacto` tiene una lista de `Direccion`, EF Core sabe generar la foreign key, sabe hacer el JOIN cuando las pedís juntas, y sabe borrarlas en cascada cuando borrás el contacto.
4. **Rastrea cambios.** Cargás un contacto, le agregás una dirección, llamás `SaveChanges()`, y EF Core genera el `INSERT` correcto — sabiendo cuál es el `ContactoId` que corresponde.

EF Core 10 salió en noviembre 2025 y es release LTS soportado hasta noviembre 2028. Corre sobre .NET 10.

### 1.3 — Por qué SQLite

SQLite es un motor de base de datos **embebido**: toda la base vive en un único archivo. No hay servidor, no hay conexiones de red, no hay cuentas de usuario.

Ventajas para nuestro caso:

- **Cero configuración.** No hay que instalar nada. El paquete NuGet ya trae el motor.
- **Portable.** El archivo `.db` lo podés copiar, mover, versionar, compartir.
- **Multiplataforma.** Windows, Linux, macOS, sin cambios.
- **Real.** No es una simulación — es un motor SQL completo, usado en producción por Android, iOS, Firefox, y básicamente toda la industria.

Desventajas (que no nos afectan en este ejercicio):

- Mala concurrencia de escritura: si varios procesos escriben a la vez, se traban.
- No soporta stored procedures ni triggers avanzados.
- No escala horizontalmente.

Para una app de escritorio o una agenda personal, estas desventajas no existen. Para una app web con 10000 usuarios escribiendo al mismo tiempo, cambiás a PostgreSQL o SQL Server **sin tocar el código de EF Core** — solo el provider. Ese es parte del valor del ORM.

### 1.4 — Code First

Hay dos formas de empezar un proyecto con EF Core:

- **Database First**: ya existe una base de datos y vos generás clases C# a partir de ella. Típico cuando heredás un sistema legacy.
- **Code First**: escribís las clases en C# y EF Core genera la base. Es el enfoque moderno por defecto.

Nosotros usamos **Code First**. Nuestras clases C# son la fuente de verdad; la base se deriva de ellas.

### 1.5 — Las relaciones, traducidas

Esta es la parte central de la clase. Pensemos cómo se traducen las relaciones entre objetos a tablas SQL.

**Relación N:1 (muchos contactos pertenecen a una categoría):**

En objetos:
```csharp
class Contacto  { Categoria Categoria; }
class Categoria { string Nombre; }
```

En tablas:
```
Contactos(Id, Nombre, CategoriaId → Categorias.Id)
Categorias(Id, Nombre)
```

La clase `Contacto` guarda una *referencia* a `Categoria`. La tabla `Contactos` guarda una *foreign key* a la fila de `Categorias`.

**Relación 1:N (un contacto tiene muchas direcciones):**

En objetos:
```csharp
class Contacto  { List<Direccion> Direcciones; }
class Direccion { Contacto Contacto; }
```

En tablas:
```
Contactos(Id, Nombre)
Direcciones(Id, Valor, ContactoId → Contactos.Id)
```

La lista vive del lado de `Contacto` en el modelo de objetos. En el modelo de tablas, la foreign key vive del lado "muchos" — es decir, cada `Direccion` recuerda a qué `Contacto` pertenece. Esa es la forma natural de expresarlo en SQL: en una base relacional no hay "listas", solo relaciones.

EF Core se encarga de hacer esta traducción por vos. Cuando cargás un contacto y pedís sus direcciones, hace internamente un `JOIN` o un `SELECT` aparte, junta los resultados, y te devuelve un objeto `Contacto` con la lista de `Direccion` ya poblada.

### 1.6 — El DbContext: puerta de entrada a todo

Todo lo que hagas con EF Core pasa por un objeto `DbContext`. Es la "unidad de trabajo" (*unit of work*) — representa una sesión con la base.

Un `DbContext`:

- Abre y cierra la conexión a la base.
- Contiene las `DbSet<T>`, que son las colecciones consultables (una por cada tabla).
- Rastrea los cambios que hacés sobre las entidades cargadas.
- Ejecuta las operaciones pendientes cuando llamás `SaveChanges()`.

Regla importantísima: **el `DbContext` no es thread-safe y es barato de crear**. Creás uno, lo usás para una unidad de trabajo (una operación del menú), lo desechás. No lo compartas entre hilos. No lo hagas `static`. No lo tengas vivo horas.

Usaremos `using` para que se desechen solos.

### Por qué separamos en tres capas

Podríamos poner todo en `Program.cs`. Sería más corto. Pero quiero que practiquemos separación de responsabilidades desde el día uno:

| Capa         | Responsabilidad                           | Qué NO debe hacer                 |
|--------------|-------------------------------------------|-----------------------------------|
| **Models**   | Representar los datos del dominio         | Saber de SQL, consola, o EF Core  |
| **Data**     | Configurar EF Core y el mapeo a la base   | Tener lógica de negocio           |
| **Services** | Operaciones CRUD y reglas de negocio      | Leer de `Console` ni imprimir     |
| **Program**  | Menú y entrada/salida por consola         | Hablar directo con la base        |

Esta separación no es un capricho académico. Es lo que te permite, mañana, reutilizar `ContactoService` desde una app MAUI o una Web API sin cambiar una línea. El CRUD no "sabe" que hoy lo llama una consola.

### Decisiones de diseño explícitas

| Decisión                                       | Por qué                                                                |
|------------------------------------------------|---|
| SQLite con archivo local `agenda.db`           | Cero infra para el ejercicio, portable                                 |
| `EnsureCreated()` en lugar de migraciones      | Nos enfocamos en el ORM, no en la evolución del schema                 |
| Tabla única `Direcciones` con enum `Tipo`      | Evita duplicar estructura "teléfono/email" en dos tablas casi idénticas|
| `DbContext` nuevo por operación                | Respeta el ciclo de vida correcto, evita memory leaks                  |
| `Id` autoincremental en todas las tablas       | Convención de EF Core, cero configuración                              |
| Operaciones síncronas                          | Simplicidad pedagógica; async se verá cuando haga falta                |
| Service sin `Console.WriteLine`                | La lógica no depende de la UI; es testeable                            |

## Parte 4 — Los modelos (el corazón del diseño)

Acá está la parte más importante del tutorial. **Cómo modelás las clases determina cómo se ven las tablas y cómo fluye todo el resto del código.**

### 4.1 — El enum `TipoDireccion`

Archivo: `Models/TipoDireccion.cs`

```csharp
namespace AgendaEF.Models;

public enum TipoDireccion {
    Telefono = 1,
    Email    = 2,
}
```

**Qué está pasando:**

- Un enum es la forma más segura de representar un conjunto cerrado de valores. El compilador te obliga a usar solo los valores permitidos; no podés tipear `"telefono"` con minúscula y romper todo.
- Los valores numéricos (`= 1`, `= 2`) los pongo explícitos: así, si mañana alguien agrega un valor en el medio de la lista, los que ya existen no cambian su número. Esto importa porque **EF Core guarda el enum en la base como número por defecto**. Si los valores cambian, la base queda inconsistente.
- No uso 0. Es buena costumbre: el 0 es el valor por defecto de cualquier enum, y que represente un estado "sin asignar" ayuda a detectar bugs.

### 4.2 — La entidad `Categoria`

Archivo: `Models/Categoria.cs`

```csharp
using System.ComponentModel.DataAnnotations;

namespace AgendaEF.Models;

public class Categoria {
    public int Id { get; set; }

    [Required]
    [MaxLength(50)]
    public string Nombre { get; set; } = string.Empty;

    // Navegación inversa: una categoría puede tener muchos contactos
    public List<Contacto> Contactos { get; set; } = new();
}
```

**Puntos clave:**

- **`Id`** sin atributos. EF Core usa convención: una propiedad llamada `Id` se asume clave primaria. En SQLite, un `int` PK se auto-incrementa.
- **`[Required]`** en `Nombre` → se traduce a `NOT NULL`.
- **`[MaxLength(50)]`** documenta la intención y se respeta en providers estrictos.
- **`List<Contacto> Contactos`** es una **propiedad de navegación**. No es una columna en la tabla — es la forma que tiene el modelo de objetos de expresar "una categoría conoce a todos los contactos que la usan". EF Core la usa para hacer joins cuando pedís categoría + contactos juntos.
- **`= new()`** inicializa la lista vacía. Evita `NullReferenceException` cuando accedés a una categoría recién creada antes de `SaveChanges()`.

### 4.3 — La entidad `Contacto`

Archivo: `Models/Contacto.cs`

```csharp
using System.ComponentModel.DataAnnotations;

namespace AgendaEF.Models;

public class Contacto {
    public int Id { get; set; }

    [Required]
    [MaxLength(100)]
    public string Nombre       { get; set; } = string.Empty;

    public DateTime CreadoEn   { get; set; } = DateTime.UtcNow;

    // Relación N:1 con Categoria
    public int CategoriaId     { get; set; }
    public Categoria Categoria { get; set; } = null!;

    // Relación 1:N con Direccion
    public List<Direccion> Direcciones { get; set; } = new();
}
```

Este es el centro de gravedad del modelo. Miralo despacio.

**La relación con Categoria:**

Tenemos dos propiedades que representan lo mismo desde ángulos distintos:

- `CategoriaId` es la **foreign key** — el número que en la tabla `Contactos` apunta a una fila de `Categorias`. Es un dato crudo.
- `Categoria` es la **propiedad de navegación** — el objeto `Categoria` cargado en memoria. Lo usás cuando querés acceder a datos de la categoría desde el contacto (ej: `contacto.Categoria.Nombre`).

¿Por qué tener las dos? Porque resuelven problemas distintos:

- Si solo querés *asignar* una categoría (ej: "este contacto ahora es de Trabajo, cuyo id es 2"), con `CategoriaId = 2` alcanza. No necesitás traer el objeto `Categoria` de la base.
- Si querés *leer* datos de la categoría, usás la propiedad de navegación.

El patrón "FK + navegación" es una convención fuerte en EF Core. Aprendelo bien. El nombre `CategoriaId` es clave: EF Core por convención busca una propiedad con ese nombre para mapearla como la FK de la navegación `Categoria`.

**El `= null!` en `Categoria`:**

Es una promesa al compilador: "sé que esta propiedad parece que podría ser null, pero en la práctica EF Core la va a poblar cuando cargue el contacto, así que ignorá el warning". Es un truco común con propiedades de navegación no-opcionales. Como alternativa podrías hacer la propiedad nullable (`Categoria?`) pero eso implica chequear null en todos lados del código.

**La relación con Direcciones:**

Del lado del `Contacto`, la relación 1:N se expresa como una `List<Direccion>`. Del lado de la `Direccion` (que vemos a continuación) vive la foreign key `ContactoId`. EF Core ve este par y entiende "ajá, uno a muchos".

### 4.4 — La entidad `Direccion`

Archivo: `Models/Direccion.cs`

```csharp
using System.ComponentModel.DataAnnotations;

namespace AgendaEF.Models;

public class Direccion {
    public int Id             { get; set; }

    [Required]
    public TipoDireccion Tipo { get; set; }

    [Required]
    [MaxLength(150)]
    public string Valor       { get; set; } = string.Empty;

    // Relación N:1 con Contacto
    public int ContactoId     { get; set; }
    public Contacto Contacto  { get; set; } = null!;
}
```

**Qué observar:**

- **`Tipo`** es nuestro enum. EF Core lo guardará como `INTEGER` (los valores `1` o `2`). Simple y eficiente.
- **`Valor`** es el string que contiene el número de teléfono o el email. Un solo campo para ambos casos. El tipo del contenido lo indica `Tipo`.
- Otra vez el par **`ContactoId` + `Contacto`**, pero esta vez del lado "muchos" de la relación. La FK vive acá porque en una base relacional, **la FK siempre vive en el lado "muchos"** de una relación 1:N. Es un hecho inescapable del modelo relacional.

### 4.5 — Diagrama mental

Pongamos todo junto:

```
 ┌──────────────┐
 │ Categoria    │       ┌───────────────┐       ┌──────────────┐
 │──────────────│       │ Contacto      │       │ Direccion    │
 │ Id (PK)      │──────<│ Id (PK)       │──────<│ Id (PK)      │
 │ Nombre       │       │ Nombre        │       │ Tipo         │
 └──────────────┘       │ CreadoEn      │       │ Valor        │
     1 categoría        │ CategoriaId FK│       │ ContactoId FK│
    muchos contactos    └───────────────┘       └──────────────┘
                         1 contacto
                      muchas direcciones
```

Las flechas `──────<` indican "uno a muchos": el lado del `<` es el "muchos". Leelo así:

- *Una categoría* tiene *muchos contactos*.
- *Un contacto* tiene *muchas direcciones*.

Los tres `Id` son claves primarias autoincrementales. Las dos FK viven del lado "muchos" de cada relación. No hay más configuración que esto.

## Parte 5 — El DbContext

Archivo: `Data/AgendaContext.cs`

```csharp
using AgendaEF.Models;
using Microsoft.EntityFrameworkCore;

namespace AgendaEF.Data;

public class AgendaContext : DbContext {
    public DbSet<Contacto>  Contactos   => Set<Contacto>();
    public DbSet<Categoria> Categorias  => Set<Categoria>();
    public DbSet<Direccion> Direcciones => Set<Direccion>();

    protected override void OnConfiguring(DbContextOptionsBuilder options) {
        string path = Path.Combine(AppContext.BaseDirectory, "agenda.db");
        options.UseSqlite($"Data Source={path}");
    }

    protected override void OnModelCreating(ModelBuilder modelBuilder) {
        // Índice para búsquedas rápidas por nombre de contacto
        modelBuilder.Entity<Contacto>()
            .HasIndex(c => c.Nombre);

        // Cada categoría tiene un nombre único
        modelBuilder.Entity<Categoria>()
            .HasIndex(c => c.Nombre)
            .IsUnique();
    }

    /// <summary>
    /// Crea la base si no existe, con las tablas según el modelo.
    /// Llamar una vez al arrancar la app.
    /// </summary>
    public static void InicializarBase() {
        using var db = new AgendaContext();
        db.Database.EnsureCreated();

        // Semilla: si no hay categorías, crear las típicas
        if (!db.Categorias.Any()) {
            db.Categorias.AddRange(
                new Categoria { Nombre = "Familia" },
                new Categoria { Nombre = "Trabajo" },
                new Categoria { Nombre = "Amigos"  },
                new Categoria { Nombre = "Otros"   }
            );
            db.SaveChanges();
        }
    }
}
```

**Qué está pasando:**

- **Tres `DbSet`**, uno por tabla. EF Core necesita verlos para saber qué clases son entidades. Si solo declarás `Contactos` pero usás `Categoria`, EF Core la descubre igual siguiendo las propiedades de navegación — pero es buena práctica exponer todas explícitamente.
- **`OnConfiguring`** configura la conexión. Para SQLite, la connection string es simplemente `Data Source=ruta/al/archivo`.
- **`OnModelCreating`** es donde se configura el modelo con la **Fluent API**. Acá pusimos dos índices: uno común sobre `Contacto.Nombre` (vamos a buscar por nombre seguido), y uno único sobre `Categoria.Nombre` (no queremos dos categorías llamadas "Trabajo").
- **`EnsureCreated()`** mira el modelo actual y, si la base no existe, la crea con todas las tablas, índices y foreign keys. Si ya existe, no hace nada.
- **La semilla inicial** (las cuatro categorías típicas) es un agregado mío: queremos que la app tenga categorías disponibles desde el primer arranque. Se ejecuta una sola vez gracias al check `if (!db.Categorias.Any())`.

> **¿Por qué `EnsureCreated()` y no migraciones?**
>
> `EnsureCreated()` es perfecto para el caso que nos ocupa: queremos que la base aparezca automáticamente para poder enfocarnos en el ORM. Pero tiene un límite grave: **no soporta cambios al modelo después de la primera ejecución**. Si mañana agregás una propiedad a `Contacto`, `EnsureCreated()` ve que la base existe y se queda tranquilo — aunque le falte la columna nueva. El programa después va a explotar.
>
> Para manejar cambios al modelo sin perder datos, se usan **migraciones**: una forma de aplicar cambios incrementales y versionados a la base. Ese es un tema en sí mismo, y merece una clase aparte. Hoy nos enfocamos en entender el ORM.
>
> **Regla práctica para este ejercicio**: si cambiás el modelo, borrá el archivo `agenda.db` y dejá que el programa lo cree de nuevo. Vas a perder los datos de prueba — no hay problema, son de prueba.

### Fluent API vs Data Annotations

Hay dos formas de configurar el modelo:

- **Data Annotations**: atributos en la entidad (`[Required]`, `[MaxLength]`). Simples, viven junto a la propiedad.
- **Fluent API**: código en `OnModelCreating` usando `modelBuilder`. Más expresivo, puede hacer cosas que los atributos no (como índices únicos, relaciones complejas, configuración específica del provider).

**Regla**: lo básico va en atributos, lo avanzado en Fluent API. Coexisten sin problema; si hay conflicto, gana Fluent API.

## Parte 6 — El servicio CRUD

Archivo: `Services/ContactoService.cs`

Acá viven las operaciones de negocio. Una por cada operación del menú. Es un bloque largo; lo explicamos por partes debajo.

```csharp
using AgendaEF.Data;
using AgendaEF.Models;
using Microsoft.EntityFrameworkCore;

namespace AgendaEF.Services;

public class ContactoService {
    // ─── Categorías ──────────────────────────────────────

    public List<Categoria> ListarCategorias() {
        using var db = new AgendaContext();

        return db.Categorias
            .AsNoTracking()
            .OrderBy(c => c.Nombre)
            .ToList();
    }

    // ─── Contactos: Create ───────────────────────────────

    public Contacto Agregar(string nombre, int categoriaId, List<Direccion> direcciones) {
        using var db = new AgendaContext();

        var contacto = new Contacto {
            Nombre = nombre.Trim(),
            CategoriaId = categoriaId,
            Direcciones = direcciones,
        };

        db.Contactos.Add(contacto);
        db.SaveChanges();

        return contacto;
    }

    // ─── Contactos: Read ─────────────────────────────────

    public List<Contacto> ListarTodos() {
        using var db = new AgendaContext();

        return db.Contactos
            .AsNoTracking()
            .Include(c => c.Categoria)
            .Include(c => c.Direcciones)
            .OrderBy(c => c.Nombre)
            .ToList();
    }

    public Contacto? BuscarPorId(int id) {
        using var db = new AgendaContext();

        return db.Contactos
            .AsNoTracking()
            .Include(c => c.Categoria)
            .Include(c => c.Direcciones)
            .FirstOrDefault(c => c.Id == id);
    }

    public List<Contacto> BuscarPorNombre(string termino) {
        using var db = new AgendaContext();

        string t = termino.Trim();
        return db.Contactos
            .AsNoTracking()
            .Include(c => c.Categoria)
            .Include(c => c.Direcciones)
            .Where(c => EF.Functions.Like(c.Nombre, $"%{t}%"))
            .OrderBy(c => c.Nombre)
            .ToList();
    }

    // ─── Contactos: Update ───────────────────────────────

    public bool Actualizar(int id, string nombre, int categoriaId) {
        using var db = new AgendaContext();

        var contacto = db.Contactos.Find(id);
        if (contacto is null) {
            return false;
        }

        contacto.Nombre = nombre.Trim();
        contacto.CategoriaId = categoriaId;

        db.SaveChanges();
        return true;
    }

    // ─── Contactos: Delete ───────────────────────────────

    public bool Eliminar(int id) {
        using var db = new AgendaContext();

        var contacto = db.Contactos
            .Include(c => c.Direcciones)
            .FirstOrDefault(c => c.Id == id);

        if (contacto is null) {
            return false;
        }

        db.Contactos.Remove(contacto);
        db.SaveChanges();
        return true;
    }

    // ─── Direcciones ─────────────────────────────────────

    public Direccion AgregarDireccion(int contactoId, TipoDireccion tipo, string valor) {
        using var db = new AgendaContext();

        var direccion = new Direccion {
            ContactoId = contactoId,
            Tipo = tipo,
            Valor = valor.Trim(),
        };

        db.Direcciones.Add(direccion);
        db.SaveChanges();

        return direccion;
    }

    public bool EliminarDireccion(int direccionId) {
        using var db = new AgendaContext();

        var direccion = db.Direcciones.Find(direccionId);
        if (direccion is null) {
            return false;
        }

        db.Direcciones.Remove(direccion);
        db.SaveChanges();
        return true;
    }
}
```

Hay varias decisiones clave acá. Vamos por partes.

### 6.1 — El patrón `using var db = new AgendaContext()`

Cada método abre un `DbContext` nuevo y lo desecha al terminar. Esto es el patrón *short-lived context*. Te garantiza tres cosas:

1. La conexión a la base se cierra limpiamente.
2. El change tracker se libera (no acumula entidades en memoria entre operaciones).
3. Dos operaciones no se contaminan entre sí.

En apps grandes con inyección de dependencias, esto se configura automáticamente. En una consola simple, lo hacemos a mano.

### 6.2 — `AsNoTracking()` en las lecturas

Cuando cargás una entidad con EF Core, por defecto el context la **trackea** — registra su estado para detectar cambios después. Si solo querés leer (no modificar), esto es desperdicio: consume memoria y CPU para nada.

`.AsNoTracking()` le dice a EF Core "esto es solo lectura, no te molestes en trackear". Resultado: consultas más rápidas, menos memoria. **Usalo siempre en lecturas puras**. Es de los cambios con más impacto en performance que podés hacer.

En los métodos `Actualizar` y `Eliminar` *no* usamos `AsNoTracking`, porque justamente necesitamos que EF Core trackee los cambios para generar el `UPDATE` / `DELETE` correcto.

### 6.3 — `.Include()`: la clave para cargar relaciones

Esta es la nueva estrella del tutorial:

```csharp
db.Contactos
  .Include(c => c.Categoria)
  .Include(c => c.Direcciones)
```

**Sin `Include`**, si cargás un contacto y accedés a `contacto.Categoria`, te va a venir `null`. EF Core por defecto **no trae las relaciones** — las cargás explícitamente. Esto es *eager loading*: decís por adelantado qué relaciones querés.

El SQL que genera EF Core con estos `Include` es un único JOIN grande, algo como:

```sql
SELECT c.Id, c.Nombre, c.CategoriaId,
       cat.Id, cat.Nombre,
       d.Id, d.Tipo, d.Valor, d.ContactoId
FROM Contactos c
LEFT JOIN Categorias cat ON cat.Id = c.CategoriaId
LEFT JOIN Direcciones d ON d.ContactoId = c.Id
ORDER BY c.Nombre
```

EF Core después arma los objetos a partir de estas filas: agrupa las direcciones por contacto y las mete en la lista correspondiente. Esa es parte de la magia.

**¿Por qué `Include` y no cargar automáticamente todo?** Porque traer relaciones es caro. Si un contacto tiene 100 direcciones y cargás 1000 contactos, cargar automáticamente todas las direcciones te baja la base entera. EF Core te obliga a pedir explícitamente lo que querés: es un default conservador y correcto.

**El problema N+1**: si en lugar de `Include` hicieras esto:

```csharp
foreach (var c in db.Contactos.ToList())          // 1 query
    Console.WriteLine(c.Categoria?.Nombre);       // ¿otra por cada contacto?
```

... sin `Include`, `c.Categoria` sería null (y no harías nada útil). Pero si tuvieras *lazy loading* activado (que no activamos), cada acceso a `c.Categoria` dispararía una query nueva. 1000 contactos = 1001 queries. Ese es el famoso problema N+1, y es una de las razones más comunes de que una app con ORM sea lenta. `Include` lo evita: una sola query.

### 6.4 — `Find` vs `FirstOrDefault`

- **`Find(id)`**: busca por clave primaria. Primero revisa el change tracker (si la entidad ya está cargada en memoria, la devuelve sin ir a la base). Solo funciona con la PK. No soporta `Include`.
- **`FirstOrDefault(c => ...)`**: busca con cualquier condición LINQ. Siempre va a la base. Acepta `Include`.

Por eso en `Eliminar` usamos `FirstOrDefault` con `Include`: necesitamos que EF Core cargue las direcciones junto al contacto para poder borrarlas en cascada. En `Actualizar`, como solo vamos a tocar propiedades del contacto y no las direcciones, `Find` alcanza y es más rápido.

### 6.5 — Borrado en cascada

Cuando borramos un contacto en `Eliminar`, ¿qué pasa con sus direcciones? Quedarían huérfanas apuntando a un contacto que no existe. Eso violaría la integridad referencial.

EF Core detecta automáticamente la relación 1:N entre `Contacto` y `Direccion` (por las propiedades de navegación y la FK `ContactoId` *requerida*), y configura **cascade delete** por defecto. Es decir: cuando borrás el contacto, sus direcciones se borran también.

Por eso cargamos el contacto con sus direcciones (`Include`) antes de eliminarlo — para que EF Core las trackee y sepa que tiene que borrarlas en memoria. Hacerlo explícito es más claro y más portable entre providers.

La categoría, en cambio, *no* se borra cuando borramos un contacto — una categoría puede estar compartida por muchos contactos. Eso también lo infiere EF Core de la relación N:1.

### 6.6 — Agregar direcciones en dos momentos distintos

Miren cómo tratamos `Direcciones` en dos escenarios:

**Al crear el contacto** (`Agregar`), pasamos la lista de direcciones ya construida:

```csharp
var contacto = new Contacto {
    Nombre = nombre,
    CategoriaId = categoriaId,
    Direcciones = direcciones,  // ya tiene la lista adentro
};
db.Contactos.Add(contacto);
db.SaveChanges();
```

EF Core ve el contacto con su lista, hace un `INSERT` al contacto, y después `INSERT`s a cada dirección usando el `Id` recién generado del contacto. Todo en una sola transacción.

**Después, cuando querés agregar una dirección a un contacto existente** (`AgregarDireccion`), pasamos el `ContactoId`:

```csharp
var direccion = new Direccion {
    ContactoId = contactoId,  // apuntamos al contacto existente
    Tipo = tipo,
    Valor = valor,
};
db.Direcciones.Add(direccion);
db.SaveChanges();
```

No necesitamos cargar el contacto entero. Con saber su id, podemos crear una nueva fila en `Direcciones` apuntando a él. Esta es la magia de separar FK y navegación que vimos en la sección 4.3.

### Ejercicio mental: ver el SQL generado

Una de las críticas más frecuentes a los ORMs es que *ocultan el SQL que generan* — y cuando la consulta es lenta, no tenés idea por qué.

EF Core te permite ver el SQL. Agregá esto al `OnConfiguring` del `AgendaContext`:

```csharp
options.UseSqlite($"Data Source={path}")
       .LogTo(Console.WriteLine, Microsoft.Extensions.Logging.LogLevel.Information);
```

Cosas que descubrís mirando el SQL:

- El `Include(c => c.Categoria)` se traduce como `LEFT JOIN Categorias`.
- El `Include(c => c.Direcciones)` trae las direcciones en la misma query.
- El `Where` se traduce como `WHERE`, no como filtrado en memoria.
- Los índices que definimos en `OnModelCreating` aparecen al crear la base.

Este superpoder — ver el SQL en vivo — es la mejor forma de usar un ORM sin volverte dependiente de él.

## Parte 9 — Decisiones de diseño aplicables a otros CRUDs

Cerramos con los principios transferibles. Estos son los que deberías llevarte a *cualquier* app de acceso a datos que construyas en el futuro.

### 9.1 — El DbContext tiene ciclo de vida corto

Un `DbContext` es como una conversación con la base. Abrís la conversación, hacés lo tuyo, cortás. No la dejás abierta todo el programa. Razones concretas:

- **Memory leaks**: el change tracker acumula entidades a medida que consultás. Un contexto largo crece indefinidamente.
- **Datos stale**: si otro proceso modifica la base, tu contexto largo sigue viendo la versión vieja.
- **Transacciones confusas**: cada `SaveChanges` es una transacción. Un contexto largo con muchos saves puede dejar la base en estados inconsistentes si algo falla.

Regla operativa: **un `DbContext` por unidad de trabajo** (una operación del menú, una request web, una pantalla de MAUI).

### 9.2 — Separá lectura y escritura mentalmente

Las operaciones de lectura (Read) son fundamentalmente distintas de las de escritura (Create, Update, Delete). Tratarlas igual es error común.

- **Lecturas**: usá `AsNoTracking`. Son idempotentes (podés repetirlas sin consecuencias). No requieren transacción.
- **Escrituras**: sin `AsNoTracking` (necesitás tracking para que `SaveChanges` sepa qué hacer). No son idempotentes. Pueden requerir transacción si involucran varias tablas.

Esta distinción es el germen del patrón **CQRS** (Command Query Responsibility Segregation), que separa explícitamente el modelo de lectura del de escritura. Para una agenda no hace falta llegar a CQRS, pero la mentalidad sí.

### 9.3 — FK + propiedad de navegación: el patrón cardinal

Cuando modelás una relación, casi siempre querés ambas:

```csharp
public int CategoriaId { get; set; }         // la FK cruda
public Categoria Categoria { get; set; }     // la navegación
```

Razones:

- **Asignar es barato**: si solo querés decir "este contacto ahora es Trabajo (id 2)", `CategoriaId = 2` y listo. No traés el objeto `Categoria` entero de la base.
- **Leer es cómodo**: `contacto.Categoria.Nombre` en lugar de hacer queries a mano.
- **EF Core los relaciona automáticamente** por convención de nombre (`CategoriaId` ↔ `Categoria`).

Aprendelo bien. Lo vas a usar en cada relación que modeles.

### 9.4 — `Include` explícito es la forma correcta

EF Core por defecto **no carga las relaciones automáticamente**. Esto es una decisión deliberada: cargar todo siempre es insostenible en apps con muchos datos. Vos decidís qué traer en cada query, con `Include`.

Consecuencia: si accedés a `contacto.Categoria` sin haber hecho `Include`, obtenés `null`. No es un bug del ORM; es vos olvidándote. Si te encontrás siempre olvidándote, considerá poner `Include` en el service — pero siempre explícito, nunca "cargá todo siempre por las dudas".

### 9.5 — Pensá el modelo relacional, no solo el de objetos

En el modelo de objetos, `Contacto` tiene una `List<Direccion>`. En el modelo relacional, la FK vive del lado de `Direccion`. Son formas distintas de expresar la misma idea.

Cuando diseñes entidades, hacelo pensando en ambos modelos al mismo tiempo. Un buen ejercicio: antes de escribir la clase, dibujá el diagrama de tablas. Si el diagrama es raro, probablemente el modelo de objetos también lo va a ser.

Pregunta de chequeo: *¿dónde vive la foreign key?* La respuesta es siempre "en el lado muchos" de una relación 1:N, o "en una tabla intermedia" en una relación N:N. Si no está clara, el modelo no está claro.

### 9.6 — Convención sobre configuración, pero conocé la configuración

EF Core hace muchísimo por convención: el nombre de la tabla viene del `DbSet`, la clave primaria de `Id`, las relaciones se infieren por pares `FooId` + `Foo`. Esto es genial para empezar rápido.

Pero hay un peligro: creer que la convención es todo. Tarde o temprano vas a necesitar Fluent API para índices únicos, restricciones complejas, nombres no convencionales, configuración de cascada distinta. Invertí en aprenderla.

### 9.7 — Mirá el SQL generado

Lo dijimos en la Parte 8 pero merece repetirse como principio: **nunca uses un ORM sin saber qué SQL está generando**.

- Te volvés mejor en SQL viendo ejemplos reales de traducción de LINQ.
- Detectás problemas de performance temprano (como el famoso N+1).
- Cuando el ORM genera algo subóptimo, podés saltarlo con SQL crudo (`FromSqlRaw`, `ExecuteSqlRaw`).

El ORM es una comodidad, no un reemplazo del conocimiento de bases de datos.

### 9.8 — Validación en capas

Hay tres lugares donde validar que un contacto tiene nombre:

1. **En la UI** (antes de llamar al service): feedback inmediato al usuario.
2. **En el service** (regla de negocio): garantiza que aunque la UI falle, las reglas se cumplan.
3. **En la base** (constraint `NOT NULL`): última línea de defensa, garantía a nivel de storage.

Parece redundante, pero cada una protege de un escenario distinto. *Defense in depth*.

## Cierre

Lo que acabamos de construir son unas 300 líneas de C#. Una agenda con relaciones entre tres tablas, persistente, con CRUD completo. Hace 15 años, hacer esto a mano requería decenas de queries SQL, mapeo manual, y una tarde entera de trabajo. EF Core lo convirtió en un par de horas y algunas decenas de líneas de código real de acceso a datos.

Pero — el caveat pedagógico de siempre — la baja barrera de entrada es un riesgo si no entendés lo que pasa por debajo. Un alumno que copia y pega EF Core sin entender `DbContext`, tracking, `Include`, FK + navegación, puede escribir código que funciona *hasta que no funciona*. Y cuando se rompa, va a estar ciego.

Por eso insistimos en el *por qué*: por qué `AsNoTracking`, por qué `Include`, por qué contextos cortos, por qué tanto FK como navegación, por qué mirar el SQL. El cómo lo vas a recordar cinco minutos; el por qué te sirve para los próximos diez años.

Y lo más importante: los principios no son específicos de EF Core. El impedance mismatch, el unit of work, las relaciones 1:N con FK del lado muchos, el problema N+1, la separación lectura/escritura — todo esto vale con Dapper, con SQLAlchemy, con Hibernate, con cualquier ORM que uses en los próximos 20 años. Los frameworks pasan; los principios quedan.

*Estamos agregando complejidad adicional que no contribuye al entendimiento del tema* — la misma regla de siempre. Acá no agregamos async, ni migraciones, ni inyección de dependencias, ni repositorios, ni unit of work patterns. No porque estén mal, sino porque para lo que estamos aprendiendo hoy, son ruido. Cuando el proyecto crezca y el ruido se vuelva necesario, vas a saber cuándo introducir cada pieza — y por qué.

**En la próxima clase**: migraciones. Cómo evolucionar el modelo sin perder datos. Cómo versionar los cambios al schema. Por qué `EnsureCreated()` no alcanza en producción.
