# Patrón Builder en C#

El **patrón Builder** sirve para construir objetos complejos **paso a paso**, separando el proceso de construcción del objeto final.

La idea central es esta:

```csharp
var objeto = new Builder()
    .ConAlgo()
    .ConOtraCosa()
    .ConConfiguracionEspecial()
    .Build();
```

En vez de tener un constructor monstruoso como este:

```csharp
var auto = new Auto("Toyota", "Corolla", 2024, true, false, 4, "Nafta", "Automática", true, true);
```

Que es básicamente una ensalada rusa con parámetros.

---

## 1. El problema que resuelve

Supongamos que queremos crear una clase `Pizza`.

```csharp
public class Pizza
{
    public string Masa { get; }
    public string Salsa { get; }
    public string Queso { get; }
    public bool TieneJamon { get; }
    public bool TieneAceitunas { get; }
    public bool TieneMorron { get; }

    public Pizza(
        string masa,
        string salsa,
        string queso,
        bool tieneJamon,
        bool tieneAceitunas,
        bool tieneMorron)
    {
        Masa = masa;
        Salsa = salsa;
        Queso = queso;
        TieneJamon = tieneJamon;
        TieneAceitunas = tieneAceitunas;
        TieneMorron = tieneMorron;
    }
}
```

Uso:

```csharp
var pizza = new Pizza(
    "fina",
    "tomate",
    "mozzarella",
    true,
    false,
    true
);
```

El problema aparece enseguida:

```csharp
true, false, true
```

¿Qué es cada cosa?

¿Jamón? ¿Aceitunas? ¿Morrones? ¿Ananá? ¿La AFIP?

El constructor funciona, pero la intención del código queda escondida.

---

## 2. Primer principio: construir no siempre es inicializar

Cuando un objeto es simple, un constructor alcanza:

```csharp
public class Persona
{
    public Persona(string nombre, string apellido)
    {
        Nombre = nombre;
        Apellido = apellido;
    }

    public string Nombre { get; }
    public string Apellido { get; }
}
```

Uso:

```csharp
var persona = new Persona("Ana", "García");
```

Perfecto.

Pero hay objetos cuya creación implica varias decisiones:

```text
qué propiedades son obligatorias
qué propiedades son opcionales
qué combinaciones son válidas
qué valores por defecto usar
en qué orden preparar las partes
qué validaciones aplicar antes de crear
```

Ahí la construcción empieza a ser una responsabilidad propia.

El patrón Builder aparece cuando construir el objeto **merece su propio objeto**.

---

## 3. Filosofía de diseño

La filosofía del Builder es:

> Separar la lógica de construcción del objeto construido.

El objeto final debería representar el resultado.

El builder debería representar el proceso para llegar a ese resultado.

Ejemplo conceptual:

```text
Pedido        = objeto final
PedidoBuilder = proceso de armado del pedido
```

La clase `Pedido` no debería necesariamente saber todas las formas posibles de construirse.

El `PedidoBuilder` sí.

---

## 4. Solución básica

Primero, el objeto final:

```csharp
public class Pizza
{
    public string Masa { get; }
    public string Salsa { get; }
    public string Queso { get; }
    public bool TieneJamon { get; }
    public bool TieneAceitunas { get; }
    public bool TieneMorron { get; }

    public Pizza(
        string masa,
        string salsa,
        string queso,
        bool tieneJamon,
        bool tieneAceitunas,
        bool tieneMorron)
    {
        Masa = masa;
        Salsa = salsa;
        Queso = queso;
        TieneJamon = tieneJamon;
        TieneAceitunas = tieneAceitunas;
        TieneMorron = tieneMorron;
    }

    public override string ToString()
    {
        return $"Pizza de masa {Masa}, salsa {Salsa}, queso {Queso}";
    }
}
```

Ahora el builder:

```csharp
public class PizzaBuilder
{
    private string _masa = "media masa";
    private string _salsa = "tomate";
    private string _queso = "mozzarella";
    private bool _tieneJamon;
    private bool _tieneAceitunas;
    private bool _tieneMorron;

    public PizzaBuilder ConMasa(string masa)
    {
        _masa = masa;
        return this;
    }

    public PizzaBuilder ConSalsa(string salsa)
    {
        _salsa = salsa;
        return this;
    }

    public PizzaBuilder ConQueso(string queso)
    {
        _queso = queso;
        return this;
    }

    public PizzaBuilder ConJamon()
    {
        _tieneJamon = true;
        return this;
    }

    public PizzaBuilder ConAceitunas()
    {
        _tieneAceitunas = true;
        return this;
    }

    public PizzaBuilder ConMorron()
    {
        _tieneMorron = true;
        return this;
    }

    public Pizza Build()
    {
        return new Pizza(
            _masa,
            _salsa,
            _queso,
            _tieneJamon,
            _tieneAceitunas,
            _tieneMorron
        );
    }
}
```

Uso:

```csharp
Pizza pizza = new PizzaBuilder()
    .ConMasa("fina")
    .ConSalsa("tomate")
    .ConQueso("mozzarella")
    .ConJamon()
    .ConMorron()
    .Build();
```

Ahora el código se lee como una frase.

Eso no es cosmética: es diseño.

## 5. Qué mejora Builder

### Antes

```csharp
var pizza = new Pizza("fina", "tomate", "mozzarella", true, false, true);
```

Problemas:

```text
parámetros poco expresivos
muchos argumentos
booleanos ambiguos
difícil de leer
difícil de mantener
fácil equivocarse en el orden
```

### Después

```csharp
var pizza = new PizzaBuilder()
    .ConMasa("fina")
    .ConSalsa("tomate")
    .ConQueso("mozzarella")
    .ConJamon()
    .ConMorron()
    .Build();
```

Ventajas:

```text
intención explícita
orden más flexible
valores por defecto
validación centralizada
construcción progresiva
menor cantidad de constructores
```

## 6. Constructor telescópico

El Builder suele aparecer para evitar el problema del **constructor telescópico**.

Ejemplo:

```csharp
public class Reporte
{
    public Reporte(string titulo) { }
    public Reporte(string titulo, string autor) { }
    public Reporte(string titulo, string autor, bool incluirFecha) { }
    public Reporte(string titulo, string autor, bool incluirFecha, bool incluirIndice) { }
    public Reporte(string titulo, string autor, bool incluirFecha, bool incluirIndice, bool incluirGraficos) { }
}
```

Esto escala horrible.

Cada nuevo parámetro opcional puede generar otra sobrecarga.

El Builder evita eso:

```csharp
var reporte = new ReporteBuilder()
    .ConTitulo("Ventas 2026")
    .ConAutor("Alejandro")
    .IncluirFecha()
    .IncluirIndice()
    .IncluirGraficos()
    .Build();
```

Más claro. Menos sufrimiento. La computadora también agradece, aunque no lo diga.

## 7. Estructura general del patrón

Objeto final:

```csharp
public class Producto
{
    public string ParteA { get; }
    public string ParteB { get; }
    public string ParteC { get; }

    public Producto(string parteA, string parteB, string parteC)
    {
        ParteA = parteA;
        ParteB = parteB;
        ParteC = parteC;
    }
}
```

Builder:

```csharp
public class ProductoBuilder
{
    private string _parteA = "";
    private string _parteB = "";
    private string _parteC = "";

    public ProductoBuilder ConParteA(string valor)
    {
        _parteA = valor;
        return this;
    }

    public ProductoBuilder ConParteB(string valor)
    {
        _parteB = valor;
        return this;
    }

    public ProductoBuilder ConParteC(string valor)
    {
        _parteC = valor;
        return this;
    }

    public Producto Build()
    {
        return new Producto(_parteA, _parteB, _parteC);
    }
}
```

Uso:

```csharp
Producto producto = new ProductoBuilder()
    .ConParteA("A")
    .ConParteB("B")
    .ConParteC("C")
    .Build();
```

## 8. Builder fluido

En C#, el Builder suele implementarse con una interfaz **fluent**.

Eso significa que cada método devuelve el propio builder:

```csharp
return this;
```

Ejemplo:

```csharp
public PizzaBuilder ConJamon()
{
    _tieneJamon = true;
    return this;
}
```

Gracias a eso podés encadenar llamadas:

```csharp
new PizzaBuilder()
    .ConJamon()
    .ConMorron()
    .Build();
```

Este estilo se llama **Fluent Interface**.

No es obligatorio para Builder, pero en C# es muy común.

## 9. Caso de uso 1: construir un pedido

Objeto final:

```csharp
public class Pedido
{
    public string Cliente { get; }
    public List<string> Productos { get; }
    public string MetodoPago { get; }
    public string DireccionEntrega { get; }
    public bool EsExpress { get; }

    public Pedido(
        string cliente,
        List<string> productos,
        string metodoPago,
        string direccionEntrega,
        bool esExpress)
    {
        Cliente = cliente;
        Productos = productos;
        MetodoPago = metodoPago;
        DireccionEntrega = direccionEntrega;
        EsExpress = esExpress;
    }
}
```

Builder:

```csharp
public class PedidoBuilder
{
    private string _cliente = "";
    private readonly List<string> _productos = new();
    private string _metodoPago = "Efectivo";
    private string _direccionEntrega = "";
    private bool _esExpress;

    public PedidoBuilder ParaCliente(string cliente)
    {
        _cliente = cliente;
        return this;
    }

    public PedidoBuilder AgregarProducto(string producto)
    {
        _productos.Add(producto);
        return this;
    }

    public PedidoBuilder ConMetodoPago(string metodoPago)
    {
        _metodoPago = metodoPago;
        return this;
    }

    public PedidoBuilder ConDireccionEntrega(string direccion)
    {
        _direccionEntrega = direccion;
        return this;
    }

    public PedidoBuilder Express()
    {
        _esExpress = true;
        return this;
    }

    public Pedido Build()
    {
        if (string.IsNullOrWhiteSpace(_cliente))
            throw new InvalidOperationException("El cliente es obligatorio.");

        if (_productos.Count == 0)
            throw new InvalidOperationException("El pedido debe tener al menos un producto.");

        return new Pedido(
            _cliente,
            new List<string>(_productos),
            _metodoPago,
            _direccionEntrega,
            _esExpress
        );
    }
}
```

Uso:

```csharp
Pedido pedido = new PedidoBuilder()
    .ParaCliente("Ana")
    .AgregarProducto("Notebook")
    .AgregarProducto("Mouse")
    .ConMetodoPago("Tarjeta")
    .ConDireccionEntrega("San Miguel de Tucumán")
    .Express()
    .Build();
```

Fijate que la validación queda en `Build()`.

Eso tiene sentido: el objeto se valida cuando se intenta cerrar su construcción.

## 10. Caso de uso 2: construir consultas SQL

Este es muy típico.

Sin Builder:

```csharp
string sql = "SELECT Nombre, Email FROM Clientes WHERE Activo = 1 ORDER BY Nombre";
```

Si empieza a crecer dinámicamente, se vuelve incómodo.

Builder:

```csharp
public class SqlQueryBuilder
{
    private readonly List<string> _select = new();
    private string _from = "";
    private readonly List<string> _where = new();
    private string _orderBy = "";

    public SqlQueryBuilder Select(string columna)
    {
        _select.Add(columna);
        return this;
    }

    public SqlQueryBuilder From(string tabla)
    {
        _from = tabla;
        return this;
    }

    public SqlQueryBuilder Where(string condicion)
    {
        _where.Add(condicion);
        return this;
    }

    public SqlQueryBuilder OrderBy(string columna)
    {
        _orderBy = columna;
        return this;
    }

    public string Build()
    {
        if (_select.Count == 0)
            throw new InvalidOperationException("Debe indicar al menos una columna.");

        if (string.IsNullOrWhiteSpace(_from))
            throw new InvalidOperationException("Debe indicar la tabla.");

        string sql = $"SELECT {string.Join(", ", _select)} FROM {_from}";

        if (_where.Count > 0)
            sql += $" WHERE {string.Join(" AND ", _where)}";

        if (!string.IsNullOrWhiteSpace(_orderBy))
            sql += $" ORDER BY {_orderBy}";

        return sql;
    }
}
```

Uso:

```csharp
string sql = new SqlQueryBuilder()
    .Select("Nombre")
    .Select("Email")
    .From("Clientes")
    .Where("Activo = 1")
    .Where("Edad >= 18")
    .OrderBy("Nombre")
    .Build();

Console.WriteLine(sql);
```

Salida:

```sql
SELECT Nombre, Email FROM Clientes WHERE Activo = 1 AND Edad >= 18 ORDER BY Nombre
```

Advertencia seria: para SQL real no armes queries con strings crudos si hay entrada de usuario. Usá parámetros. El Builder puede ayudar, pero no hace magia anti-inyección SQL. No es un cura sanador.

## 11. Caso de uso 3: construir HTML

```csharp
public class HtmlBuilder
{
    private readonly List<string> _items = new();

    public HtmlBuilder H1(string texto)
    {
        _items.Add($"<h1>{texto}</h1>");
        return this;
    }

    public HtmlBuilder P(string texto)
    {
        _items.Add($"<p>{texto}</p>");
        return this;
    }

    public HtmlBuilder Button(string texto)
    {
        _items.Add($"<button>{texto}</button>");
        return this;
    }

    public string Build()
    {
        return string.Join(Environment.NewLine, _items);
    }
}
```

Uso:

```csharp
string html = new HtmlBuilder()
    .H1("Agenda")
    .P("Listado de contactos")
    .Button("Agregar")
    .Build();
```

Salida:

```html
<h1>Agenda</h1>
<p>Listado de contactos</p>
<button>Agregar</button>
```

Acá el Builder permite expresar estructura de manera progresiva.

## 12. Caso de uso 4: configuración de objetos

Supongamos una clase `ServidorConfig`.

```csharp
public class ServidorConfig
{
    public string Host { get; }
    public int Puerto { get; }
    public bool UsarHttps { get; }
    public int TimeoutSegundos { get; }

    public ServidorConfig(
        string host,
        int puerto,
        bool usarHttps,
        int timeoutSegundos)
    {
        Host = host;
        Puerto = puerto;
        UsarHttps = usarHttps;
        TimeoutSegundos = timeoutSegundos;
    }
}
```

Builder:

```csharp
public class ServidorConfigBuilder
{
    private string _host = "localhost";
    private int _puerto = 80;
    private bool _usarHttps;
    private int _timeoutSegundos = 30;

    public ServidorConfigBuilder ConHost(string host)
    {
        _host = host;
        return this;
    }

    public ServidorConfigBuilder ConPuerto(int puerto)
    {
        _puerto = puerto;
        return this;
    }

    public ServidorConfigBuilder UsarHttps()
    {
        _usarHttps = true;
        _puerto = 443;
        return this;
    }

    public ServidorConfigBuilder ConTimeout(int segundos)
    {
        _timeoutSegundos = segundos;
        return this;
    }

    public ServidorConfig Build()
    {
        if (_puerto <= 0 || _puerto > 65535)
            throw new InvalidOperationException("Puerto inválido.");

        return new ServidorConfig(
            _host,
            _puerto,
            _usarHttps,
            _timeoutSegundos
        );
    }
}
```

Uso:

```csharp
ServidorConfig config = new ServidorConfigBuilder()
    .ConHost("api.midominio.com")
    .UsarHttps()
    .ConTimeout(60)
    .Build();
```

Observá algo importante:

```csharp
.UsarHttps()
```

no solo cambia `UsarHttps`.

También puede ajustar el puerto por defecto a `443`.

Esa es lógica de construcción. Buen lugar para el Builder.

## 13. Caso de uso 5: objetos inmutables

Builder encaja muy bien con objetos inmutables.

Objeto final:

```csharp
public class Usuario
{
    public string Nombre { get; }
    public string Email { get; }
    public string Rol { get; }
    public bool Activo { get; }

    public Usuario(string nombre, string email, string rol, bool activo)
    {
        Nombre = nombre;
        Email = email;
        Rol = rol;
        Activo = activo;
    }
}
```

Builder:

```csharp
public class UsuarioBuilder
{
    private string _nombre = "";
    private string _email = "";
    private string _rol = "Usuario";
    private bool _activo = true;

    public UsuarioBuilder ConNombre(string nombre)
    {
        _nombre = nombre;
        return this;
    }

    public UsuarioBuilder ConEmail(string email)
    {
        _email = email;
        return this;
    }

    public UsuarioBuilder ComoAdmin()
    {
        _rol = "Admin";
        return this;
    }

    public UsuarioBuilder Inactivo()
    {
        _activo = false;
        return this;
    }

    public Usuario Build()
    {
        if (string.IsNullOrWhiteSpace(_nombre))
            throw new InvalidOperationException("El nombre es obligatorio.");

        if (string.IsNullOrWhiteSpace(_email))
            throw new InvalidOperationException("El email es obligatorio.");

        return new Usuario(_nombre, _email, _rol, _activo);
    }
}
```

Uso:

```csharp
Usuario admin = new UsuarioBuilder()
    .ConNombre("Laura")
    .ConEmail("laura@example.com")
    .ComoAdmin()
    .Build();
```

El objeto final no tiene setters.

Una vez creado, queda consistente.

## 16. Builder con records en C#

En C# moderno, muchas veces podés resolver parte del problema con `record` y expresión `with`.

```csharp
public record Usuario(
    string Nombre,
    string Email,
    string Rol,
    bool Activo
);
```

Uso:

```csharp
var usuarioBase = new Usuario(
    Nombre: "Usuario Test",
    Email: "test@example.com",
    Rol: "Usuario",
    Activo: true
);

var admin = usuarioBase with
{
    Rol = "Admin"
};
```

Esto es muy elegante.

Entonces, ¿Builder sigue teniendo sentido?

Sí, cuando necesitás:

```text
validaciones complejas
valores por defecto elaborados
construcción paso a paso
colecciones internas
métodos expresivos
reglas de negocio de armado
evitar constructores enormes
```

Si solo necesitás clonar cambiando dos propiedades, `record` + `with` puede ser suficiente.

No hay que usar Builder para abrir una puerta que ya estaba abierta.

## 17. Builder clásico con Director

En el patrón original de GoF, aparece una figura llamada **Director**.

El Director conoce una receta de construcción.

Ejemplo:

```csharp
public interface IComputadoraBuilder
{
    void AgregarCpu();
    void AgregarRam();
    void AgregarDisco();
    Computadora Build();
}
```

Producto:

```csharp
public class Computadora
{
    public string Cpu { get; set; } = "";
    public string Ram { get; set; } = "";
    public string Disco { get; set; } = "";

    public override string ToString()
    {
        return $"{Cpu}, {Ram}, {Disco}";
    }
}
```

Builder concreto:

```csharp
public class ComputadoraGamerBuilder : IComputadoraBuilder
{
    private readonly Computadora _computadora = new();

    public void AgregarCpu()
    {
        _computadora.Cpu = "Intel i9";
    }

    public void AgregarRam()
    {
        _computadora.Ram = "32GB";
    }

    public void AgregarDisco()
    {
        _computadora.Disco = "SSD 2TB";
    }

    public Computadora Build()
    {
        return _computadora;
    }
}
```

Director:

```csharp
public class DirectorComputadora
{
    public Computadora Construir(IComputadoraBuilder builder)
    {
        builder.AgregarCpu();
        builder.AgregarRam();
        builder.AgregarDisco();

        return builder.Build();
    }
}
```

Uso:

```csharp
var director = new DirectorComputadora();

Computadora pc = director.Construir(
    new ComputadoraGamerBuilder()
);
```

Este enfoque sirve cuando querés que el mismo proceso construya distintas variantes.

Por ejemplo:

```text
ComputadoraGamerBuilder
ComputadoraOficinaBuilder
ServidorBuilder
```

Mismo algoritmo general, distintos detalles.

En C# moderno, muchas veces se usa más el Builder fluido que el Builder con Director.

## 18. Builder fluido vs Builder clásico

### Builder clásico

Tiene pasos separados y a veces un Director.

```csharp
builder.AgregarCpu();
builder.AgregarRam();
builder.AgregarDisco();
var pc = builder.Build();
```

### Builder fluido

Encadena métodos.

```csharp
var pc = new ComputadoraBuilder()
    .ConCpu("Intel i9")
    .ConRam("32GB")
    .ConDisco("SSD 2TB")
    .Build();
```

En aplicaciones C#, el fluido suele ser más cómodo y legible.

## 19. Ejemplo con `StringBuilder`

`.NET` tiene un ejemplo real muy famoso: `StringBuilder`.

```csharp
var sb = new StringBuilder();

sb.AppendLine("Hola");
sb.AppendLine("Mundo");

string texto = sb.ToString();
```

`StringBuilder` no es exactamente el patrón Builder GoF puro, pero comparte la idea:

```text
construir progresivamente un resultado final
```

Resultado:

```csharp
string texto = sb.ToString();
```

Acá el objeto final es el `string`.

## 20. Ejemplo real en ASP.NET Core

Esto también se ve mucho:

```csharp
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddControllers();
builder.Services.AddEndpointsApiExplorer();
builder.Services.AddSwaggerGen();

var app = builder.Build();

app.UseHttpsRedirection();
app.UseAuthorization();

app.MapControllers();

app.Run();
```

El objeto `builder` junta configuración.

Después:

```csharp
var app = builder.Build();
```

crea la aplicación final.

Esa forma de trabajar está muy alineada con la idea Builder.

## 21. Relación con SOLID

### SRP

El objeto final representa datos/comportamiento del dominio.

El Builder se ocupa de construirlo.

```text
Usuario        → representa usuario
UsuarioBuilder → sabe construir usuarios
```

Cada uno con su responsabilidad.

### OCP

Podés agregar nuevas formas de construcción sin tocar demasiado el objeto final.

Ejemplo:

```csharp
PedidoBuilder
PedidoExpressBuilder
PedidoMayoristaBuilder
```

O métodos nuevos:

```csharp
.ConEnvioExpress()
.ConDescuentoMayorista()
```

### DIP

Podés depender de abstracciones si tenés builders intercambiables:

```csharp
public interface IPedidoBuilder
{
    IPedidoBuilder ParaCliente(string cliente);
    IPedidoBuilder AgregarProducto(string producto);
    Pedido Build();
}
```

Aunque en builders fluidos concretos, no siempre hace falta interfaz.

No metas interfaz por deporte. Después hay que mantenerla. Y no hay medalla.

## 22. Builder vs Factory Method

Se parecen porque ambos crean objetos.

Pero resuelven problemas distintos.

### Factory

Sirve cuando querés delegar **qué objeto concreto crear**.

```csharp
INotificador notificador = NotificadorFactory.Crear("email");
```

La factory decide:

```text
email → EmailNotificador
sms   → SmsNotificador
push  → PushNotificador
```

### Builder

Sirve cuando querés controlar **cómo se construye un objeto complejo paso a paso**.

```csharp
var pedido = new PedidoBuilder()
    .ParaCliente("Ana")
    .AgregarProducto("Mouse")
    .ConMetodoPago("Tarjeta")
    .Build();
```

Resumen:

```text
Factory: decide cuál crear.
Builder: decide cómo armarlo.
```

## 23. Builder vs Object Initializer

En C# podés hacer:

```csharp
var usuario = new Usuario
{
    Nombre = "Ana",
    Email = "ana@example.com",
    Rol = "Admin",
    Activo = true
};
```

Esto es simple y legible.

Entonces, ¿para qué Builder?

Porque el object initializer requiere setters públicos:

```csharp
public string Nombre { get; set; }
```

Y no centraliza validaciones complejas.

Builder puede producir un objeto inmutable:

```csharp
public string Nombre { get; }
```

Y validar en `Build()`.

Object initializer está bien para objetos simples.

Builder está mejor para objetos con reglas.

## 24. Builder vs parámetros nombrados

C# permite esto:

```csharp
var pizza = new Pizza(
    masa: "fina",
    salsa: "tomate",
    queso: "mozzarella",
    tieneJamon: true,
    tieneAceitunas: false,
    tieneMorron: true
);
```

Esto mejora mucho la legibilidad.

Entonces Builder no siempre es necesario.

Builder gana cuando:

```text
hay muchos opcionales
hay colecciones
hay defaults
hay validaciones
hay pasos
hay combinaciones
hay recetas
hay construcción progresiva
```

Si solo querés aclarar argumentos, los parámetros nombrados pueden alcanzar.

## 25. Builder vs Opciones/Options Pattern

En .NET es común usar opciones:

```csharp
public class EmailOptions
{
    public string Host { get; set; } = "";
    public int Port { get; set; } = 25;
    public bool UseSsl { get; set; }
}
```

Uso:

```csharp
var options = new EmailOptions
{
    Host = "smtp.example.com",
    Port = 587,
    UseSsl = true
};
```

Eso está perfecto para configuración simple.

Pero si configurar implica reglas:

```text
si UseSsl entonces puerto default 465
si proveedor Gmail entonces host smtp.gmail.com
si modo producción entonces timeout mayor
```

un Builder puede expresar mejor esas decisiones.

## 26. Builder con validación fuerte

Ejemplo:

```csharp
public class CuentaBancaria
{
    public string Titular { get; }
    public decimal SaldoInicial { get; }
    public string Moneda { get; }

    public CuentaBancaria(string titular, decimal saldoInicial, string moneda)
    {
        Titular = titular;
        SaldoInicial = saldoInicial;
        Moneda = moneda;
    }
}
```

Builder:

```csharp
public class CuentaBancariaBuilder
{
    private string _titular = "";
    private decimal _saldoInicial;
    private string _moneda = "ARS";

    public CuentaBancariaBuilder ConTitular(string titular)
    {
        _titular = titular;
        return this;
    }

    public CuentaBancariaBuilder ConSaldoInicial(decimal saldo)
    {
        _saldoInicial = saldo;
        return this;
    }

    public CuentaBancariaBuilder EnDolares()
    {
        _moneda = "USD";
        return this;
    }

    public CuentaBancaria Build()
    {
        if (string.IsNullOrWhiteSpace(_titular))
            throw new InvalidOperationException("El titular es obligatorio.");

        if (_saldoInicial < 0)
            throw new InvalidOperationException("El saldo inicial no puede ser negativo.");

        return new CuentaBancaria(
            _titular,
            _saldoInicial,
            _moneda
        );
    }
}
```

Uso:

```csharp
CuentaBancaria cuenta = new CuentaBancariaBuilder()
    .ConTitular("Ana")
    .ConSaldoInicial(10000)
    .EnDolares()
    .Build();
```

## 27. Builder con pasos obligatorios

Una debilidad del Builder simple es que permite esto:

```csharp
var pedido = new PedidoBuilder()
    .Build();
```

Y recién falla en runtime.

Una alternativa es obligar datos mínimos en el constructor del builder:

```csharp
public class PedidoBuilder
{
    private readonly string _cliente;
    private readonly List<string> _productos = new();
    private string _metodoPago = "Efectivo";

    public PedidoBuilder(string cliente)
    {
        if (string.IsNullOrWhiteSpace(cliente))
            throw new ArgumentException("El cliente es obligatorio.");

        _cliente = cliente;
    }

    public PedidoBuilder AgregarProducto(string producto)
    {
        _productos.Add(producto);
        return this;
    }

    public PedidoBuilder ConMetodoPago(string metodoPago)
    {
        _metodoPago = metodoPago;
        return this;
    }

    public Pedido Build()
    {
        if (_productos.Count == 0)
            throw new InvalidOperationException("Debe agregar al menos un producto.");

        return new Pedido(
            _cliente,
            new List<string>(_productos),
            _metodoPago,
            "",
            false
        );
    }
}
```

Uso:

```csharp
Pedido pedido = new PedidoBuilder("Ana")
    .AgregarProducto("Mouse")
    .ConMetodoPago("Tarjeta")
    .Build();
```

Así reducís estados inválidos.

## 28. Step Builder

Existe una variante más estricta llamada **Step Builder**.

Sirve para forzar el orden de construcción usando interfaces.

Ejemplo: obligar a indicar primero cliente, luego producto, luego permitir construir.

```csharp
public interface IPedidoClienteStep
{
    IPedidoProductoStep ParaCliente(string cliente);
}

public interface IPedidoProductoStep
{
    IPedidoBuildStep AgregarProducto(string producto);
}

public interface IPedidoBuildStep
{
    IPedidoBuildStep AgregarProducto(string producto);
    IPedidoBuildStep ConMetodoPago(string metodoPago);
    Pedido Build();
}
```

Implementación:

```csharp
public class PedidoStepBuilder :
    IPedidoClienteStep,
    IPedidoProductoStep,
    IPedidoBuildStep
{
    private string _cliente = "";
    private readonly List<string> _productos = new();
    private string _metodoPago = "Efectivo";

    private PedidoStepBuilder()
    {
    }

    public static IPedidoClienteStep Crear()
    {
        return new PedidoStepBuilder();
    }

    public IPedidoProductoStep ParaCliente(string cliente)
    {
        _cliente = cliente;
        return this;
    }

    public IPedidoBuildStep AgregarProducto(string producto)
    {
        _productos.Add(producto);
        return this;
    }

    public IPedidoBuildStep ConMetodoPago(string metodoPago)
    {
        _metodoPago = metodoPago;
        return this;
    }

    public Pedido Build()
    {
        return new Pedido(
            _cliente,
            new List<string>(_productos),
            _metodoPago,
            "",
            false
        );
    }
}
```

Uso:

```csharp
Pedido pedido = PedidoStepBuilder.Crear()
    .ParaCliente("Ana")
    .AgregarProducto("Mouse")
    .ConMetodoPago("Tarjeta")
    .Build();
```

Ahora no podés llamar `Build()` antes de agregar un producto.

El compilador te frena.

Esto es potente, pero más verboso.

Usalo cuando la validez del orden sea importante. Si no, es demasiado protocolo para hacer una pizza.

## 29. Cuándo usar Builder

Usalo cuando:

```text
el constructor tiene demasiados parámetros
hay muchos parámetros opcionales
hay booleanos ambiguos
el objeto requiere validación antes de crearse
hay valores por defecto importantes
el objeto se construye en varios pasos
hay colecciones internas
querés objetos finales inmutables
querés mejorar legibilidad en tests
hay recetas o variantes de construcción
```

---

## 30. Cuándo no usar Builder

No lo uses cuando:

```text
el objeto tiene dos o tres propiedades simples
un constructor claro alcanza
un object initializer alcanza
los parámetros nombrados alcanzan
no hay reglas de construcción
el builder solo copia propiedades sin aportar valor
```

Ejemplo donde Builder sería exceso:

```csharp
public record Punto(int X, int Y);
```

Esto ya está perfecto:

```csharp
var punto = new Punto(10, 20);
```

No hace falta:

```csharp
var punto = new PuntoBuilder()
    .ConX(10)
    .ConY(20)
    .Build();
```

Eso no es diseño. Es burocracia con llaves.

## 33. Resumen brutalmente útil

El Builder responde esta pregunta:

> ¿Cómo construyo un objeto complejo de forma clara, flexible y segura, sin llenar el constructor de parámetros confusos?

Respuesta:

> Uso un objeto auxiliar que acumula decisiones de construcción y al final produce el objeto válido con `Build()`.

Código mínimo:

```csharp
public class ProductoBuilder
{
    private string _nombre = "";
    private decimal _precio;

    public ProductoBuilder ConNombre(string nombre)
    {
        _nombre = nombre;
        return this;
    }

    public ProductoBuilder ConPrecio(decimal precio)
    {
        _precio = precio;
        return this;
    }

    public Producto Build()
    {
        return new Producto(_nombre, _precio);
    }
}
```

Uso:

```csharp
Producto producto = new ProductoBuilder()
    .ConNombre("Teclado")
    .ConPrecio(50000)
    .Build();
```

La esencia:

```text
Builder = construcción paso a paso + intención explícita + objeto final consistente
```

El objeto final no debería ser una víctima de constructores kilométricos. Para eso existe Builder.
