# Guía práctica de patrones en .NET para aplicar SRP

## Introducción — el problema que resuelven

Toda aplicación útil termina creciendo: lee archivos, valida datos, guarda en una base de datos, llama APIs externas, muestra información en pantalla, registra logs, envía notificaciones y maneja errores. Al principio parece cómodo poner todo junto en una sola clase. Después esa clase empieza a cambiar por cualquier motivo: cambió la base de datos, cambió el formato JSON, cambió la UI, cambió una regla de negocio, cambió el proveedor de email.

Ese es el problema que ataca el **Single Responsibility Principle** —SRP—:

> Una clase debería tener una sola razón para cambiar.

El principio no dice que una clase tenga un solo método, ni que sea microscópica. Dice que su responsabilidad conceptual debe ser clara. Cuando una clase cambia por motivos distintos, probablemente tiene responsabilidades mezcladas.

Tomemos un ejemplo típico: una agenda de contactos que guarda datos en JSON y permite agregar contactos.

```csharp
// ── Solución mezclada — demasiadas responsabilidades ─────────────────────

public sealed class AgendaService {
    private readonly string path = "contactos.json";

    public void Agregar(Contacto contacto) {
        List<Contacto> contactos;

        if (File.Exists(path)) {
            string json = File.ReadAllText(path);
            contactos = JsonSerializer.Deserialize<List<Contacto>>(json) ?? [];
        }
        else {
            contactos = [];
        }

        if (string.IsNullOrWhiteSpace(contacto.Nombre))
            throw new InvalidOperationException("El nombre es obligatorio.");

        contactos.Add(contacto);

        string nuevoJson = JsonSerializer.Serialize(contactos, new JsonSerializerOptions {
            WriteIndented = true
        });

        File.WriteAllText(path, nuevoJson);
    }
}
```

Funciona. Pero la clase está haciendo de todo:

```text
AgendaService
    ├─► sabe dónde está el archivo
    ├─► sabe leer archivos
    ├─► sabe parsear JSON
    ├─► sabe validar contactos
    ├─► sabe agregar contactos
    └─► sabe guardar archivos
```

La pregunta de diseño no es “¿funciona?”. La pregunta correcta es:

> ¿Por cuántas razones distintas puede cambiar esta clase?

La respuesta es: demasiadas.

Una versión más limpia separa responsabilidades:

```csharp
// ── Solución separada — cada clase con una razón de cambio ───────────────

public sealed class AgendaService {
    private readonly IContactoRepository repository;
    private readonly IContactoValidator validator;

    public AgendaService(IContactoRepository repository, IContactoValidator validator) {
        this.repository = repository;
        this.validator  = validator;
    }

    public void Agregar(Contacto contacto) {
        validator.Validar(contacto);

        List<Contacto> contactos = repository.ObtenerTodos();
        contactos.Add(contacto);
        repository.GuardarTodos(contactos);
    }
}
```

Ahora el servicio expresa el caso de uso. No sabe si los datos vienen de JSON, SQLite, SQL Server, memoria o una API. Ese detalle vive en otra clase.

Ese es el papel de los patrones: **dar formas conocidas de separar responsabilidades sin inventar una arquitectura nueva cada martes**.

## 1. Filosofía de diseño

### 1.1 SRP no significa “una clase, una línea”

SRP no exige clases absurdamente pequeñas. Exige clases con una responsabilidad coherente.

```text
Mal criterio:
    "Esta clase tiene muchos métodos, entonces viola SRP."

Mejor criterio:
    "Esta clase puede cambiar por motivos distintos, entonces probablemente viola SRP."
```

Una clase puede tener varios métodos si todos pertenecen a la misma responsabilidad.

Por ejemplo, un repositorio puede tener:

```csharp
ObtenerTodos()
ObtenerPorId(int id)
Guardar(Contacto contacto)
Eliminar(int id)
```

Eso sigue siendo una responsabilidad: persistir contactos.

En cambio, si además tiene:

```csharp
ValidarEmail()
RenderizarHtml()
EnviarWhatsapp()
CalcularDeuda()
```

ahí ya no es una clase. Es un placard antes de que llegue visita.

### 1.2 Los patrones separan razones de cambio

Cada patrón ayuda a cortar una responsabilidad distinta:

| Problema recurrente                                     | Patrón típico        |
|---------------------------------------------------------|-----------------------|
| Cambia la forma de guardar datos                        | Repository            |
| Cambia la transacción o unidad de persistencia          | Unit of Work          |
| Cambia el algoritmo                                     | Strategy              |
| Cambia la forma de crear objetos                        | Factory               |
| Cambia una API externa                                  | Adapter               |
| Se agregan comportamientos transversales                | Decorator             |
| Se quiere simplificar un subsistema                     | Facade                |
| Se quiere representar una acción                        | Command               |
| Se quiere separar UI de estado y acciones               | MVVM                  |
| Se quiere reaccionar a eventos sin acoplar              | Observer / Events     |
| Se quiere separar configuración                         | Options Pattern       |
| Se quiere separar lógica HTTP transversal               | Middleware / Filters  |
| Se quiere separar modelo interno de contrato externo    | DTO + Mapper          |

### 1.3 .NET favorece composición sobre herencia profunda

En .NET moderno se ve mucho esta forma:

```csharp
public sealed class ContactoService {
    private readonly IContactoRepository repository;
    private readonly INotificador notificador;
    private readonly ILogger<ContactoService> logger;

    public ContactoService(
        IContactoRepository repository,
        INotificador notificador,
        ILogger<ContactoService> logger) {
        this.repository = repository;
        this.notificador = notificador;
        this.logger = logger;
    }
}
```

La clase no crea todo lo que necesita. Lo recibe. Eso permite reemplazar piezas sin tocar el caso de uso.

```text
ContactoService
    ├─► usa IContactoRepository
    ├─► usa INotificador
    └─► usa ILogger<T>
```

Esa arquitectura es SRP llevado al armado de objetos.

## 2. Dependency Injection — separar uso de creación

### 2.1 Qué problema resuelve

Sin Dependency Injection, una clase suele crear sus propias dependencias:

```csharp
public sealed class ContactoService {
    private readonly JsonContactoRepository repository = new("contactos.json");

    public void Agregar(Contacto contacto) {
        List<Contacto> contactos = repository.ObtenerTodos();
        contactos.Add(contacto);
        repository.GuardarTodos(contactos);
    }
}
```

Problema: `ContactoService` queda acoplado a `JsonContactoRepository`. Si mañana se usa SQLite, hay que modificar el servicio.

Con DI, la clase recibe sus dependencias:

```csharp
public sealed class ContactoService {
    private readonly IContactoRepository repository;

    public ContactoService(IContactoRepository repository) {
        this.repository = repository;
    }

    public void Agregar(Contacto contacto) {
        List<Contacto> contactos = repository.ObtenerTodos();
        contactos.Add(contacto);
        repository.GuardarTodos(contactos);
    }
}
```

Ahora el servicio no sabe qué implementación concreta usa.

### 2.2 Registro en .NET

En ASP.NET Core, Worker Services, MAUI y otras plantillas modernas, el contenedor de DI ya viene integrado.

```csharp
builder.Services.AddSingleton<IContactoRepository>(
    new JsonContactoRepository("contactos.json")
);

builder.Services.AddScoped<ContactoService>();
```

Uso:

```csharp
app.MapPost("/contactos", (Contacto contacto, ContactoService service) => {
    service.Agregar(contacto);
    return Results.Ok();
});
```

### 2.3 SRP aplicado

```text
ContactoService        → ejecuta el caso de uso
JsonContactoRepository → persiste contactos en JSON
DI Container           → construye y conecta objetos
```

La responsabilidad de construir objetos sale del servicio y queda en la composición de la aplicación.

## 3. Repository — separar dominio de persistencia

### 3.1 Qué problema resuelve

El patrón Repository separa la lógica de negocio del mecanismo concreto de almacenamiento.

```text
Servicio / caso de uso
    ↓
Repositorio
    ↓
Archivo / base de datos / API / memoria
```

Contrato:

```csharp
public interface IContactoRepository {
    List<Contacto> ObtenerTodos();
    void GuardarTodos(List<Contacto> contactos);
}
```

Implementación en JSON:

```csharp
public sealed class JsonContactoRepository : IContactoRepository {
    private readonly string path;

    public JsonContactoRepository(string path) {
        this.path = path;
    }

    public List<Contacto> ObtenerTodos() {
        if (!File.Exists(path))
            return [];

        string json = File.ReadAllText(path);
        return JsonSerializer.Deserialize<List<Contacto>>(json) ?? [];
    }

    public void GuardarTodos(List<Contacto> contactos) {
        string json = JsonSerializer.Serialize(contactos, new JsonSerializerOptions {
            WriteIndented = true
        });

        File.WriteAllText(path, json);
    }
}
```

Servicio:

```csharp
public sealed class ContactoService {
    private readonly IContactoRepository repository;

    public ContactoService(IContactoRepository repository) {
        this.repository = repository;
    }

    public void Agregar(Contacto contacto) {
        List<Contacto> contactos = repository.ObtenerTodos();
        contactos.Add(contacto);
        repository.GuardarTodos(contactos);
    }
}
```

### 3.2 Versión con memoria para pruebas

```csharp
public sealed class InMemoryContactoRepository : IContactoRepository {
    private readonly List<Contacto> contactosInternos = [];

    public List<Contacto> ObtenerTodos() {
        return contactosInternos.ToList();
    }

    public void GuardarTodos(List<Contacto> contactos) {
        contactosInternos.Clear();
        contactosInternos.AddRange(contactos);
    }
}
```

El servicio no cambia. Solo cambia la implementación del repositorio.

### 3.3 SRP aplicado

```text
ContactoService             → reglas del caso de uso
JsonContactoRepository      → persistencia en archivo JSON
InMemoryContactoRepository  → persistencia en memoria
```

Una razón de cambio por clase.

## 4. Unit of Work — separar cambios de confirmación

### 4.1 Qué problema resuelve

Unit of Work agrupa operaciones y las confirma como una unidad.

En EF Core, `DbContext` cumple en gran medida el rol de Repository + Unit of Work:

```csharp
context.Contactos.Add(contacto);
context.Contactos.Remove(contactoViejo);
context.Contactos.Update(contactoEditado);

await context.SaveChangesAsync();
```

Hasta que se llama a `SaveChangesAsync()`, EF Core solo está registrando cambios.

```text
DbContext
    ├─► rastrea entidades
    ├─► detecta cambios
    ├─► agrupa operaciones
    └─► confirma con SaveChangesAsync()
```

### 4.2 Ejemplo de servicio

```csharp
public sealed class ContactoService {
    private readonly AgendaDbContext context;

    public ContactoService(AgendaDbContext context) {
        this.context = context;
    }

    public async Task AgregarAsync(Contacto contacto) {
        context.Contactos.Add(contacto);
        await context.SaveChangesAsync();
    }
}
```

### 4.3 SRP aplicado

```text
ContactoService → decide qué operación de negocio hacer
DbContext       → rastrea y confirma cambios
Base de datos   → persiste físicamente
```

El servicio no arma SQL, no abre conexiones y no maneja manualmente el estado de cada entidad. Eso queda encapsulado en EF Core.

## 5. Strategy — separar algoritmos intercambiables

### 5.1 Qué problema resuelve

Cuando una clase tiene muchos `if` o `switch` para elegir comportamientos, suele aparecer una responsabilidad escondida: seleccionar y ejecutar algoritmos variables.

Ejemplo mezclado:

```csharp
public sealed class ExportacionService {
    public void Exportar(IEnumerable<Contacto> contactos, string formato, string path) {
        if (formato == "csv") {
            File.WriteAllLines(path, contactos.Select(c => $"{c.Apellido},{c.Nombre}"));
            return;
        }

        if (formato == "json") {
            string json = JsonSerializer.Serialize(contactos);
            File.WriteAllText(path, json);
            return;
        }

        throw new InvalidOperationException("Formato no soportado.");
    }
}
```

Cada nuevo formato modifica `ExportacionService`.

### 5.2 Aplicando Strategy

```csharp
public interface IExportadorContactos {
    string Formato { get; }
    void Exportar(IEnumerable<Contacto> contactos, string path);
}
```

```csharp
public sealed class CsvExportadorContactos : IExportadorContactos {
    public string Formato => "csv";

    public void Exportar(IEnumerable<Contacto> contactos, string path) {
        File.WriteAllLines(path, contactos.Select(c => $"{c.Apellido},{c.Nombre}"));
    }
}
```

```csharp
public sealed class JsonExportadorContactos : IExportadorContactos {
    public string Formato => "json";

    public void Exportar(IEnumerable<Contacto> contactos, string path) {
        string json = JsonSerializer.Serialize(contactos, new JsonSerializerOptions {
            WriteIndented = true
        });

        File.WriteAllText(path, json);
    }
}
```

Servicio:

```csharp
public sealed class ExportacionService {
    private readonly IEnumerable<IExportadorContactos> exportadores;

    public ExportacionService(IEnumerable<IExportadorContactos> exportadores) {
        this.exportadores = exportadores;
    }

    public void Exportar(string formato, IEnumerable<Contacto> contactos, string path) {
        IExportadorContactos exportador = exportadores
            .Single(e => e.Formato == formato);

        exportador.Exportar(contactos, path);
    }
}
```

Registro:

```csharp
builder.Services.AddSingleton<IExportadorContactos, CsvExportadorContactos>();
builder.Services.AddSingleton<IExportadorContactos, JsonExportadorContactos>();
```

### 5.3 SRP aplicado

```text
CsvExportadorContactos   → exporta CSV
JsonExportadorContactos  → exporta JSON
ExportacionService       → coordina la exportación
```

Agregar XML ya no obliga a tocar el servicio. Se agrega una estrategia nueva.

## 6. Factory — separar creación de objetos

### 6.1 Qué problema resuelve

A veces el problema no es ejecutar un comportamiento, sino decidir qué objeto concreto crear.

```csharp
public interface IExportadorFactory {
    IExportadorContactos Crear(string formato);
}
```

```csharp
public sealed class ExportadorFactory : IExportadorFactory {
    private readonly IEnumerable<IExportadorContactos> exportadores;

    public ExportadorFactory(IEnumerable<IExportadorContactos> exportadores) {
        this.exportadores = exportadores;
    }

    public IExportadorContactos Crear(string formato) {
        return exportadores.Single(e => e.Formato == formato);
    }
}
```

Servicio:

```csharp
public sealed class ExportacionService {
    private readonly IExportadorFactory factory;

    public ExportacionService(IExportadorFactory factory) {
        this.factory = factory;
    }

    public void Exportar(string formato, IEnumerable<Contacto> contactos, string path) {
        IExportadorContactos exportador = factory.Crear(formato);
        exportador.Exportar(contactos, path);
    }
}
```

### 6.2 Factories comunes en .NET

```csharp
IHttpClientFactory
ILoggerFactory
IDbContextFactory<TContext>
```

Estos nombres no son casualidad. Separan la creación/configuración de objetos del código que los usa.

### 6.3 SRP aplicado

```text
ExportadorFactory     → selecciona/crea el exportador adecuado
ExportacionService    → ejecuta el caso de uso
Exportador concreto   → produce un formato específico
```

## 7. Adapter — separar tu modelo de una API externa

### 7.1 Qué problema resuelve

Cuando una librería externa tiene una interfaz que no coincide con la de tu aplicación, el Adapter traduce entre ambas.

Tu aplicación quiere esto:

```csharp
public interface INotificador {
    Task EnviarAsync(string destino, string mensaje);
}
```

Pero un SDK externo ofrece esto:

```csharp
public sealed class WhatsAppSdkClient {
    public Task SendTextAsync(string phoneNumber, string body) {
        // SDK externo
        return Task.CompletedTask;
    }
}
```

Adapter:

```csharp
public sealed class WhatsAppNotificador : INotificador {
    private readonly WhatsAppSdkClient client;

    public WhatsAppNotificador(WhatsAppSdkClient client) {
        this.client = client;
    }

    public Task EnviarAsync(string destino, string mensaje) {
        return client.SendTextAsync(destino, mensaje);
    }
}
```

Servicio:

```csharp
public sealed class ContactoService {
    private readonly INotificador notificador;

    public ContactoService(INotificador notificador) {
        this.notificador = notificador;
    }

    public Task AvisarAltaAsync(Contacto contacto) {
        return notificador.EnviarAsync(
            contacto.Telefono,
            $"Contacto creado: {contacto.Nombre}"
        );
    }
}
```

### 7.2 SRP aplicado

```text
ContactoService       → decide cuándo notificar
INotificador          → contrato propio de la aplicación
WhatsAppNotificador   → adapta el SDK externo
WhatsAppSdkClient     → API concreta del proveedor
```

La aplicación no queda contaminada por los nombres, modelos y rarezas del proveedor externo. Todo SDK trae su zoológico; mejor dejarlo en una jaula.

## 8. Decorator — agregar comportamiento sin modificar la clase original

### 8.1 Qué problema resuelve

Decorator permite envolver un objeto con otro que implementa la misma interfaz y agrega comportamiento antes o después.

Contrato:

```csharp
public interface IContactoRepository {
    List<Contacto> ObtenerTodos();
    void GuardarTodos(List<Contacto> contactos);
}
```

Repositorio real:

```csharp
public sealed class JsonContactoRepository : IContactoRepository {
    public List<Contacto> ObtenerTodos() {
        return [];
    }

    public void GuardarTodos(List<Contacto> contactos) {
        // guardar en JSON
    }
}
```

Decorator con logging:

```csharp
public sealed class LoggingContactoRepository : IContactoRepository {
    private readonly IContactoRepository inner;
    private readonly ILogger<LoggingContactoRepository> logger;

    public LoggingContactoRepository(
        IContactoRepository inner,
        ILogger<LoggingContactoRepository> logger) {
        this.inner = inner;
        this.logger = logger;
    }

    public List<Contacto> ObtenerTodos() {
        logger.LogInformation("Leyendo contactos.");
        return inner.ObtenerTodos();
    }

    public void GuardarTodos(List<Contacto> contactos) {
        logger.LogInformation("Guardando {Cantidad} contactos.", contactos.Count);
        inner.GuardarTodos(contactos);
    }
}
```

### 8.2 Decorator en `Stream`

La familia `Stream` usa esta idea de forma muy clara:

```text
FileStream
    ↓
BufferedStream
    ↓
GZipStream
    ↓
CryptoStream
```

Cada capa agrega una responsabilidad:

```text
FileStream      → archivo
BufferedStream  → buffer
GZipStream      → compresión
CryptoStream    → cifrado
```

### 8.3 SRP aplicado

```text
JsonContactoRepository     → persistencia
LoggingContactoRepository  → logging
```

En vez de ensuciar el repositorio con logs, cache, métricas y retry, se agregan capas especializadas.

## 9. Facade — simplificar un subsistema

### 9.1 Qué problema resuelve

Facade ofrece una API simple sobre un conjunto de objetos internos.

```csharp
public sealed class AgendaExporter {
    private readonly IContactoRepository repository;
    private readonly IExportadorFactory factory;

    public AgendaExporter(
        IContactoRepository repository,
        IExportadorFactory factory) {
        this.repository = repository;
        this.factory = factory;
    }

    public void ExportarAgenda(string formato, string path) {
        List<Contacto> contactos = repository.ObtenerTodos();
        IExportadorContactos exportador = factory.Crear(formato);
        exportador.Exportar(contactos, path);
    }
}
```

Uso:

```csharp
exporter.ExportarAgenda("csv", "contactos.csv");
```

El consumidor no necesita conocer repositorios, factories ni estrategias.

### 9.2 SRP aplicado

```text
AgendaExporter       → coordina el caso de uso completo
Repository           → obtiene datos
Factory              → selecciona exportador
Exportador concreto  → genera archivo
```

Una Facade debe simplificar. Si empieza a acumular toda la lógica de la aplicación, dejó de ser fachada y pasó a ser baldío municipal.

## 10. Command — separar acciones ejecutables

### 10.1 Qué problema resuelve

Command encapsula una acción como objeto. Es muy común en WPF, MAUI, Avalonia, WinUI y arquitecturas MVVM.

```csharp
public interface ICommand {
    bool CanExecute(object? parameter);
    void Execute(object? parameter);
    event EventHandler? CanExecuteChanged;
}
```

Ejemplo:

```csharp
public sealed class GuardarContactoCommand : ICommand {
    private readonly ContactoService service;

    public GuardarContactoCommand(ContactoService service) {
        this.service = service;
    }

    public bool CanExecute(object? parameter) {
        return parameter is Contacto contacto &&
               !string.IsNullOrWhiteSpace(contacto.Nombre);
    }

    public void Execute(object? parameter) {
        if (parameter is Contacto contacto)
            service.Agregar(contacto);
    }

    public event EventHandler? CanExecuteChanged;
}
```

### 10.2 SRP aplicado

```text
Vista / botón          → dispara la acción
Command                → representa la acción ejecutable
ContactoService        → ejecuta la regla de negocio
```

El botón no guarda contactos. El ViewModel no tiene por qué conocer detalles de persistencia. El comando representa la intención.

## 11. MVVM — separar UI, estado e interacción

### 11.1 Qué problema resuelve

MVVM es muy usado en .NET desktop/mobile: WPF, MAUI, Avalonia, WinUI.

Separa la aplicación así:

```text
View
    ↓ binding
ViewModel
    ↓ usa
Service
    ↓ usa
Repository
```

Responsabilidades:

| Pieza      | Responsabilidad                 |
|------------|---------------------------------|
| View       | Dibujar la interfaz             |
| ViewModel  | Estado y acciones de la pantalla|
| Model      | Datos y reglas del dominio      |
| Service    | Casos de uso                    |
| Repository | Persistencia                    |

### 11.2 Ejemplo mínimo

```csharp
public sealed class ContactoViewModel {
    private readonly ContactoService service;

    public string Nombre { get; set; } = "";
    public string Apellido { get; set; } = "";

    public ICommand GuardarCommand { get; }

    public ContactoViewModel(ContactoService service) {
        this.service = service;
        GuardarCommand = new GuardarContactoCommand(this.service);
    }
}
```

### 11.3 SRP aplicado

```text
View             → presentación
ViewModel        → estado de pantalla y comandos
Command          → acción ejecutable
Service          → caso de uso
Repository       → datos
```

Esto evita meter consultas SQL, parseo JSON y validaciones dentro del code-behind. El code-behind cargado es humedad en los cimientos.

## 12. Observer / Events — separar productor de consumidores

### 12.1 Qué problema resuelve

Observer permite que un objeto publique eventos sin saber quién los escucha.

En .NET aparece naturalmente con:

```csharp
event
EventHandler
IObservable<T>
```

Ejemplo:

```csharp
public sealed class ContactoService {
    public event EventHandler<Contacto>? ContactoAgregado;

    public void Agregar(Contacto contacto) {
        // guardar contacto...

        ContactoAgregado?.Invoke(this, contacto);
    }
}
```

Suscriptores:

```csharp
service.ContactoAgregado += (_, contacto) => {
    Console.WriteLine($"Nuevo contacto: {contacto.Nombre}");
};

service.ContactoAgregado += (_, contacto) => {
    // actualizar pantalla
};
```

### 12.2 SRP aplicado

```text
ContactoService  → agrega contacto y publica evento
Logger           → registra evento
UI               → refresca pantalla
```

El servicio no necesita conocer al logger ni a la UI. Solo publica algo que ocurrió.

## 13. Options Pattern — separar configuración de lógica

### 13.1 Qué problema resuelve

En .NET, la configuración suele venir de `appsettings.json`, variables de entorno, secretos o parámetros externos.

Mal enfoque:

```csharp
public sealed class JsonContactoRepository {
    private readonly IConfiguration configuration;

    public JsonContactoRepository(IConfiguration configuration) {
        this.configuration = configuration;
    }

    public List<Contacto> ObtenerTodos() {
        string path = configuration["Agenda:ArchivoContactos"]!;
        // leer archivo...
        return [];
    }
}
```

El repositorio queda parseando configuración por todos lados.

Con Options Pattern:

```csharp
public sealed class AgendaOptions {
    public string ArchivoContactos { get; set; } = "contactos.json";
}
```

Registro:

```csharp
builder.Services.Configure<AgendaOptions>(
    builder.Configuration.GetSection("Agenda")
);
```

Uso:

```csharp
public sealed class JsonContactoRepository : IContactoRepository {
    private readonly AgendaOptions options;

    public JsonContactoRepository(IOptions<AgendaOptions> options) {
        this.options = options.Value;
    }

    public List<Contacto> ObtenerTodos() {
        string path = options.ArchivoContactos;
        // leer archivo...
        return [];
    }
}
```

### 13.2 SRP aplicado

```text
appsettings.json          → valores externos
AgendaOptions             → modelo tipado de configuración
Options infrastructure    → enlaza configuración
Repository / Service      → usa configuración ya modelada
```

El servicio usa configuración. No la busca con strings mágicos en cada esquina.

## 14. Middleware Pipeline — separar responsabilidades HTTP

### 14.1 Qué problema resuelve

ASP.NET Core procesa requests usando un pipeline de middlewares.

```text
Request
    ↓
Exception Handling
    ↓
Routing
    ↓
Authentication
    ↓
Authorization
    ↓
Endpoint
```

Ejemplo:

```csharp
app.UseExceptionHandler("/error");
app.UseRouting();
app.UseAuthentication();
app.UseAuthorization();

app.MapControllers();
```

Cada middleware tiene una responsabilidad concreta.

### 14.2 Middleware propio

```csharp
public sealed class RequestLoggingMiddleware {
    private readonly RequestDelegate next;
    private readonly ILogger<RequestLoggingMiddleware> logger;

    public RequestLoggingMiddleware(
        RequestDelegate next,
        ILogger<RequestLoggingMiddleware> logger) {
        this.next = next;
        this.logger = logger;
    }

    public async Task InvokeAsync(HttpContext context) {
        logger.LogInformation("Request: {Method} {Path}",
            context.Request.Method,
            context.Request.Path);

        await next(context);
    }
}
```

Registro:

```csharp
app.UseMiddleware<RequestLoggingMiddleware>();
```

### 14.3 SRP aplicado

```text
RequestLoggingMiddleware → logging HTTP
AuthenticationMiddleware → identidad
AuthorizationMiddleware  → permisos
Endpoint                 → caso concreto
```

El endpoint no debería ocuparse de autenticar, autorizar, loguear, manejar errores globales y además resolver negocio. Para eso está el pipeline.

## 15. Filters — separar lógica transversal de controladores

### 15.1 Qué problema resuelve

Los filtros de ASP.NET Core permiten sacar lógica transversal de los controladores:

```text
AuthorizationFilter
ActionFilter
ExceptionFilter
ResultFilter
EndpointFilter
```

Ejemplo:

```csharp
public sealed class LogActionFilter : IActionFilter {
    private readonly ILogger<LogActionFilter> logger;

    public LogActionFilter(ILogger<LogActionFilter> logger) {
        this.logger = logger;
    }

    public void OnActionExecuting(ActionExecutingContext context) {
        logger.LogInformation("Ejecutando action.");
    }

    public void OnActionExecuted(ActionExecutedContext context) {
        logger.LogInformation("Action ejecutada.");
    }
}
```

Controlador limpio:

```csharp
[ServiceFilter(typeof(LogActionFilter))]
public sealed class ContactosController : ControllerBase {
    private readonly ContactoService service;

    public ContactosController(ContactoService service) {
        this.service = service;
    }

    [HttpPost("contactos")]
    public IActionResult Crear(Contacto contacto) {
        service.Agregar(contacto);
        return Ok();
    }
}
```

### 15.2 SRP aplicado

```text
Controller   → traduce HTTP a caso de uso
Filter       → logging, validación, errores, autorización
Service      → reglas del caso de uso
```

El controlador no termina siendo repositorio, validador, logger, mapper y psicólogo familiar.

## 16. DTO + Mapper — separar contrato externo de modelo interno

### 16.1 Qué problema resuelve

Una entidad del dominio o de EF Core no siempre debería exponerse directamente por una API.

Entidad:

```csharp
public sealed class Contacto {
    public int Id { get; set; }
    public string Nombre { get; set; } = "";
    public string Apellido { get; set; } = "";
    public string Email { get; set; } = "";
}
```

DTO:

```csharp
public sealed class ContactoDto {
    public int Id { get; set; }
    public string NombreCompleto { get; set; } = "";
    public string Email { get; set; } = "";
}
```

Mapper:

```csharp
public static class ContactoMapper {
    public static ContactoDto ToDto(Contacto contacto) {
        return new ContactoDto {
            Id = contacto.Id,
            NombreCompleto = $"{contacto.Nombre} {contacto.Apellido}",
            Email = contacto.Email
        };
    }
}
```

Uso:

```csharp
List<ContactoDto> resultado = contactos
    .Select(ContactoMapper.ToDto)
    .ToList();
```

### 16.2 SRP aplicado

```text
Contacto       → modelo interno
ContactoDto    → contrato externo
ContactoMapper → conversión entre ambos
```

Si cambia la API pública, no necesariamente cambia el dominio. Si cambia el dominio, no necesariamente cambia el contrato externo.

## 17. Contraste completo — de clase mezclada a diseño separado

Problema: agregar un contacto, validarlo, guardarlo y notificar.

### Versión mezclada

```csharp
public sealed class AgendaService {
    public async Task AgregarAsync(Contacto contacto) {
        if (string.IsNullOrWhiteSpace(contacto.Nombre))
            throw new InvalidOperationException("El nombre es obligatorio.");

        string path = "contactos.json";

        List<Contacto> contactos;
        if (File.Exists(path)) {
            string jsonActual = await File.ReadAllTextAsync(path);
            contactos = JsonSerializer.Deserialize<List<Contacto>>(jsonActual) ?? [];
        }
        else {
            contactos = [];
        }

        contactos.Add(contacto);

        string jsonNuevo = JsonSerializer.Serialize(contactos, new JsonSerializerOptions {
            WriteIndented = true
        });

        await File.WriteAllTextAsync(path, jsonNuevo);

        using HttpClient client = new HttpClient();
        await client.PostAsJsonAsync("https://api.notificador.com/messages", new {
            contacto.Telefono,
            Mensaje = $"Contacto creado: {contacto.Nombre}"
        });
    }
}
```

Responsabilidades mezcladas:

```text
AgendaService
    ├─► validación
    ├─► lectura de archivo
    ├─► serialización JSON
    ├─► regla de negocio
    ├─► escritura de archivo
    ├─► creación de HttpClient
    └─► llamada a API externa
```

### Versión separada

Contratos:

```csharp
public interface IContactoValidator {
    void Validar(Contacto contacto);
}

public interface IContactoRepository {
    Task<List<Contacto>> ObtenerTodosAsync();
    Task GuardarTodosAsync(List<Contacto> contactos);
}

public interface INotificador {
    Task EnviarAsync(string destino, string mensaje);
}
```

Servicio:

```csharp
public sealed class AgendaService {
    private readonly IContactoValidator validator;
    private readonly IContactoRepository repository;
    private readonly INotificador notificador;

    public AgendaService(
        IContactoValidator validator,
        IContactoRepository repository,
        INotificador notificador) {
        this.validator = validator;
        this.repository = repository;
        this.notificador = notificador;
    }

    public async Task AgregarAsync(Contacto contacto) {
        validator.Validar(contacto);

        List<Contacto> contactos = await repository.ObtenerTodosAsync();
        contactos.Add(contacto);
        await repository.GuardarTodosAsync(contactos);

        await notificador.EnviarAsync(
            contacto.Telefono,
            $"Contacto creado: {contacto.Nombre}"
        );
    }
}
```

Mapa:

```text
AgendaService
    ├─► IContactoValidator    → validación
    ├─► IContactoRepository   → persistencia
    └─► INotificador          → notificación
```

El servicio todavía coordina varias cosas, pero su responsabilidad es clara: **ejecutar el caso de uso “agregar contacto”**. Los detalles especializados viven afuera.

## 18. Mapa completo de patrones por responsabilidad

```text
Construcción y dependencias
    ├─► Dependency Injection     — inyectar dependencias
    └─► Factory                  — crear/seleccionar objetos

Persistencia
    ├─► Repository               — abstraer acceso a datos
    └─► Unit of Work             — agrupar y confirmar cambios

Comportamiento variable
    ├─► Strategy                 — algoritmos intercambiables
    └─► Command                  — acciones ejecutables

Integración y extensibilidad
    ├─► Adapter                  — adaptar APIs externas
    └─► Decorator                — agregar comportamiento envolviendo

Presentación e interacción
    ├─► MVVM                     — separar UI, estado y acciones
    └─► DTO + Mapper             — separar modelo interno de contrato externo

Aplicaciones web
    ├─► Middleware Pipeline      — responsabilidades HTTP encadenadas
    └─► Filters                  — lógica transversal de controllers/endpoints

Coordinación
    ├─► Facade                   — simplificar subsistemas
    └─► Observer / Events        — publicar eventos sin conocer consumidores

Configuración
    └─► Options Pattern          — configuración tipada y separada
```

## 19. Reglas prácticas

### 19.1 Detectar responsabilidades por razones de cambio

Preguntas útiles:

```text
¿Esta clase cambiaría si cambia la base de datos?
¿Cambiaría si cambia la UI?
¿Cambiaría si cambia el proveedor externo?
¿Cambiaría si cambia una regla de validación?
¿Cambiaría si cambia el formato de salida?
¿Cambiaría si cambia la configuración?
```

Si varias respuestas son “sí”, hay responsabilidades mezcladas.

### 19.2 No crear abstracciones antes de tiempo

SRP no significa llenar todo de interfaces por deporte.

```text
Mala señal:
    IContactoServiceFactoryProviderManager

Buena señal:
    IContactoRepository
    INotificador
    IExportadorContactos
```

La abstracción debe responder a una variación real o razonablemente probable.

### 19.3 Preferir nombres que revelen responsabilidad

Buenos nombres:

```text
JsonContactoRepository
ContactoValidator
CsvExportadorContactos
WhatsAppNotificador
AgendaOptions
ContactoMapper
```

Malos nombres:

```text
ContactoManager
AgendaHelper
Utils
GeneralService
ProcessData
```

`Manager` y `Helper` muchas veces significan “no sé qué responsabilidad tiene esto, pero compila”.

### 19.4 Separar orquestación de detalle

Un servicio puede coordinar dependencias:

```csharp
validator.Validar(contacto);
repository.Guardar(contacto);
notificador.Enviar(...);
```

Eso no viola SRP si su responsabilidad es el caso de uso.

Lo problemático sería que el mismo servicio implemente internamente validación, persistencia, serialización, HTTP y logging.

### 19.5 Usar los patrones como bisturí, no como decoración

Los patrones no son medallas. Son herramientas.

```text
¿Tengo muchos formatos de exportación?      → Strategy
¿Tengo una API externa incómoda?            → Adapter
¿Quiero agregar logging/cache/retry?        → Decorator
¿La creación de objetos ensucia el servicio?→ Factory
¿Estoy mezclando negocio y datos?           → Repository
¿La pantalla tiene demasiada lógica?        → MVVM / Command
```

## Resumen

```text
SRP
    └─► Una clase, una razón clara para cambiar

Patrones clásicos en .NET para aplicarlo:
    ├── Dependency Injection  → separar creación de uso
    ├── Repository            → separar persistencia de negocio
    ├── Unit of Work          → separar cambios de confirmación
    ├── Strategy              → separar algoritmos intercambiables
    ├── Factory               → separar creación de objetos
    ├── Adapter               → separar API propia de API externa
    ├── Decorator             → agregar comportamiento sin modificar
    ├── Facade                → simplificar subsistemas
    ├── Command               → encapsular acciones
    ├── MVVM                  → separar UI, estado y acciones
    ├── Observer / Events     → desacoplar eventos y reacciones
    ├── Options Pattern       → separar configuración
    ├── Middleware Pipeline   → separar responsabilidades HTTP
    ├── Filters               → separar lógica transversal web
    └── DTO + Mapper          → separar contrato externo de modelo interno

Regla final:
    └─► Si una clase cambia por varios motivos distintos,
        probablemente necesita que alguna responsabilidad salga de ahí.
```

La idea no es diseñar con más clases porque sí. La idea es que cada clase tenga un trabajo reconocible, testeable y reemplazable. El buen diseño no hace ruido: simplemente deja de romperse por cualquier pavada.
