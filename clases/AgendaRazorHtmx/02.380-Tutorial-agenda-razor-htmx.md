# Agenda CRUD con Razor Pages, HTMX y CSS propio

> Construimos una agenda de contactos con CRUD completo usando Razor Pages con handlers inline (`@functions`), partials reales, un DTO como `record`, persistencia con **Entity Framework Core + SQLite** y un único `ContactStore` para el acceso a datos. La interfaz se arma con **CSS propio** (sin framework) y **HTMX**: una pequeña librería JavaScript que extiende el comportamiento de las páginas HTML de forma declarativa, devolviendo fragmentos de HTML en lugar de JSON. Aprovechamos características más avanzadas de HTMX: eventos personalizados disparados desde el servidor con el header `HX-Trigger`, `hx-include` para enviar el token antiforgery, y selección de la lista con radios navegables por teclado. La base de ejemplo se siembra solo si arranca vacía, para no duplicar contactos al reiniciar.

## 1. Qué vamos a construir

Una agenda de contactos con las operaciones de un CRUD: listar, buscar, ver detalle, crear, editar y borrar. La pantalla es de tipo **maestro-detalle**: a la izquierda la lista de contactos, a la derecha un panel que cambia según lo que estés haciendo (ver, crear o editar).

Lo distintivo de esta arquitectura es que **no escribimos JavaScript propio para la interacción**. HTMX, que es una pequeña librería JavaScript cargada en la página, lee atributos HTML como `hx-get`, `hx-post`, `hx-trigger`, `hx-target` y `hx-swap` para decidir qué evento disparar, qué método HTTP invocar y qué parte del HTML reemplazar. El servidor responde con fragmentos ya renderizados por Razor. No hay API JSON, no hay framework de frontend, no hay estado duplicado en el navegador.

A diferencia de versiones más simples, acá usamos cosas que valen la pena conocer:

- **Persistencia real** con EF Core + SQLite (los datos sobreviven al reinicio).
- **Protección antiforgery activada**, con el token viajando por HTMX vía `hx-include`.
- **Eventos personalizados de HTMX** disparados desde el servidor (header `HX-Trigger`) para que la lista se refresque sola cuando algo cambia.
- **Selección por radios** navegable con teclado, accesible.
- **CSS propio** con variables, diseño responsive y soporte de accesibilidad (`focus-visible`, `prefers-reduced-motion`, `forced-colors`).

La estructura de carpetas queda así:

```txt
AgendaRazorHtmx/
├── Program.cs                 <- top-level + modelo, DTO, EF Core y ContactStore
├── Pages/
│   ├── Index.cshtml           <- pantalla principal + handlers inline
│   ├── _ContactList.cshtml    <- lista (panel maestro)
│   ├── _ContactDetail.cshtml  <- detalle (solo lectura)
│   ├── _ContactForm.cshtml    <- alta y edición (un solo formulario)
│   └── Shared/
│       └── _Layout.cshtml     <- plantilla base: HTMX + CSS propio
├── wwwroot/
│   ├── css/
│   │   └── site.css           <- estilos propios (sin framework)
│   └── js/
│       └── htmx.min.js        <- htmx servido localmente (sin CDN)
└── agenda.db                  <- base SQLite (se crea sola al arrancar)
```

> **Por qué todo el código C# vive en `Program.cs`:** para una app chica, mantener el modelo, el DTO, el `DbContext` y el `ContactStore` en un solo archivo deja ver el sistema completo de un vistazo, sin saltar entre carpetas. No es una regla general —un proyecto grande conviene separarlo en capas—, pero acá la cercanía ayuda a aprender. Cada tipo igual tiene una responsabilidad clara.

## 2. Cómo fluye una interacción

Antes de escribir código conviene tener el modelo mental claro. Cuando el usuario hace algo (un clic, escribir en el buscador), el recorrido es siempre el mismo:

```txt
Navegador
   │  HTMX dispara un GET o POST
   ▼
Index.cshtml  (un handler: OnGetDetail, OnPostSave, etc.)
   │  el handler delega
   ▼
ContactStore   (acceso a datos)
   │
   ▼
EF Core  →  SQLite (agenda.db)
```

Y la respuesta vuelve convertida en HTML:

```txt
SQLite → ContactStore → Handler → Partial Razor → HTML
   │
   ▼
HTMX recibe ese HTML y reemplaza solo una parte de la página
```

> **La clave de HTMX:** el servidor no devuelve datos crudos (JSON) que el navegador tenga que interpretar. Devuelve **HTML ya armado**, y HTMX simplemente lo inserta en el lugar indicado. Por eso no hace falta JavaScript: la lógica de "qué se ve" vive en el servidor, en los partials de Razor.

## 4. Por qué Razor Pages y los handlers inline

Razor Pages genera HTML en el servidor. Una página es un archivo `.cshtml` que mezcla HTML con C#, y la directiva `@page` al inicio la convierte en una URL accesible:

```razor
@page

<h1>Hola Razor</h1>
```

Normalmente la lógica de una página Razor vive en un archivo aparte (`Index.cshtml.cs`, el *code-behind*). En este tutorial usamos un enfoque alternativo: **handlers inline** dentro del mismo `.cshtml`, en un bloque `@functions`.

```razor
@functions {
    public IActionResult OnGetDetail(int id) {
        // Se ejecuta con: GET /?handler=Detail&id=1
    }
}
```

La convención de nombres de Razor traduce automáticamente la URL al método:

| Petición HTTP                  | Método que se ejecuta      |
|--------------------------------|----------------------------|
| `GET /`                        | `OnGet()`                  |
| `GET /?handler=Detail&id=1`    | `OnGetDetail(int id)`      |
| `POST /?handler=Save`          | `OnPostSave(...)`          |

El verbo (`GET`/`POST`) y el parámetro `handler` arman el nombre del método. Los parámetros restantes de la URL (`id=1`) se asignan a los argumentos del método por nombre: eso es **model binding**.

> **Por qué handlers inline en este tutorial:** mantenerlos en el `.cshtml` deja toda la lógica de la pantalla en un solo archivo, fácil de leer de corrido. En un proyecto grande conviene el code-behind separado, pero para aprender el flujo completo de un vistazo, lo inline es más claro. Es una decisión pedagógica, no una recomendación general.

## 5. Por qué CSS propio y no un framework

En esta versión **no usamos PicoCSS ni Bootstrap ni Tailwind**: escribimos un `wwwroot/css/site.css` propio. La razón es que la pantalla maestro-detalle tiene requisitos concretos —dos paneles con scroll independiente, lista seleccionable con radios, altura completa de la ventana— que es más limpio resolver con un puñado de reglas a medida que peleando contra los defaults de un framework.

El CSS se apoya en tres ideas:

```css
:root {
    --color-surface:  #ffffff;
    --color-selected: #dbeafe;
    --color-focus:    #2563eb;
    --radius: 8px;
    --space-4: 1rem;
    /* ... */
}
```

- **Variables CSS** (`--color-*`, `--space-*`) para mantener consistencia y poder ajustar el tema en un solo lugar.
- **Layout con Grid** para el maestro-detalle (`grid-template-columns: 320px 1fr`) y scroll contenido en cada panel.
- **Accesibilidad de fábrica:** estilos de `:focus-visible`, una utilidad `.visually-hidden`, y bloques `@media (prefers-reduced-motion)` y `@media (forced-colors)` para respetar las preferencias del sistema.

El archivo completo está en [wwwroot/css/site.css](wwwroot/css/site.css). No hace falta memorizarlo; lo importante es entender **qué hace cada clase** cuando aparezca en los partials (`.panel`, `.contact-item`, `.detail-grid`, `.actions`, `.primary`, `.danger`, `.muted`, `.visually-hidden`).

> **Por qué CSS propio y no un framework:** un framework "sin clases" como Pico es genial para formularios simples, pero acá necesitábamos control fino sobre el grid de dos columnas, el scroll de cada panel y el resaltado de la fila seleccionada. Con ~300 líneas de CSS propio obtenemos exactamente la interfaz que queremos, sin dependencias externas ni clases que pelean entre sí. El trade-off es que lo escribimos nosotros, pero queda totalmente bajo nuestro control.

## 6. Por qué HTMX y qué características usamos

HTMX es una pequeña librería JavaScript que amplía el comportamiento de HTML de manera declarativa. En vez de programar manualmente los clics, los fetch o el reemplazo del DOM, agregás atributos al markup para decirle al navegador qué evento dispara la petición, qué método HTTP usar y qué fragmento HTML reemplazar. Un ejemplo:

```html
<button
    hx-get="/?handler=Detail&id=2"
    hx-target="#detail"
    hx-swap="innerHTML">
    Ver detalle
</button>
```

Al hacer clic, HTMX hace tres cosas: pide por `GET` a esa URL, recibe el HTML que devuelve el servidor, y mete ese HTML dentro de `#detail`. La lógica está declarada en HTML y HTMX, por debajo, ejecuta el comportamiento con JavaScript.

Estos son los atributos y técnicas que usamos en el tutorial —van bastante más allá de lo básico:

| Atributo / técnica          | Para qué sirve                                                                |
|-----------------------------|-------------------------------------------------------------------------------|
| `hx-get` / `hx-post`        | Hacer una petición GET o POST                                                 |
| `hx-target`                 | A qué elemento aplicar la respuesta                                           |
| `hx-swap`                   | Cómo insertar el HTML (`innerHTML` reemplaza el contenido)                    |
| `hx-trigger`                | Qué evento dispara la petición, con modificadores como `delay:250ms`          |
| `hx-include`                | Enviar campos extra junto con la petición (el token antiforgery, el buscador) |
| `hx-confirm`                | Mostrar un cuadro de confirmación antes de enviar                             |
| **Eventos personalizados**  | `hx-trigger="contactsChanged from:body"` escucha un evento de la app          |
| **Header `HX-Trigger`**     | El servidor dispara ese evento devolviendo un header en la respuesta          |

Las dos últimas filas son la parte interesante. En vez de que cada acción tenga que reconstruir media pantalla, el servidor puede **avisar** "los contactos cambiaron" devolviendo un header `HX-Trigger: contactsChanged`. La lista, que está escuchando ese evento, se refresca sola. Lo vemos en detalle en la [sección 14](#14-la-técnica-clave-refrescar-dos-paneles-con-un-evento).

> **Por qué HTML parcial y no JSON:** en una SPA tradicional, el servidor manda JSON y el navegador tiene código JavaScript que lo convierte en HTML. Eso duplica la lógica de presentación en dos lados. Con HTMX, el servidor manda el HTML ya armado por Razor y el navegador solo lo coloca. Una sola fuente de verdad para la presentación, cero JavaScript que mantener.

## 7. El modelo y el DTO de entrada

Todo el código C# de la app vive en `Program.cs`, después de las instrucciones de arranque, dentro de `namespace AgendaRazorHtmx`. Empecemos por el modelo y el DTO.

El **modelo** representa cómo vive un contacto **dentro** de la aplicación; es también la entidad que EF Core mapea a una tabla:

```csharp
public class Contact {
    public int Id { get; set; }
    public string Nombre   { get; set; } = "";
    public string Apellido { get; set; } = "";
    public string Telefono { get; set; } = "";
    public string Email    { get; set; } = "";
    public string Notas    { get; set; } = "";
}
```

El **DTO de entrada** define qué datos llegan desde un formulario. Usamos un `record` porque es inmutable y se declara en una línea:

```csharp
public record ContactInput(
    int Id,
    string Nombre,
    string Apellido,
    string Telefono,
    string Email,
    string Notas
);
```

> **Por qué un solo DTO para crear y editar (y no dos):** acá usamos el `Id` como señal. Si `Id == 0`, es un contacto nuevo; si trae un `Id` real, es una edición. El formulario es el mismo en ambos casos y manda un campo oculto `Id` (`0` al crear). Esto simplifica la pantalla: un único `_ContactForm`, un único handler `OnPostSave`. El `ContactStore.Save` decide internamente si inserta o actualiza según el `Id`.
>
> **Por qué no usar `Contact` directamente en los formularios:** el modelo es lo que vive adentro y lo que EF Core persiste; el DTO es la frontera de lo que aceptamos de afuera. Separarlos evita que el formulario pueda tocar propiedades que no debería y deja explícito qué campos son válidos como entrada.

También tenemos un pequeño **view model** para la lista, que combina los contactos con cuál está seleccionado (para resaltarlo):

```csharp
public record ContactListViewModel(List<Contact> Contacts, int? SelectedId);
```

## 8. El acceso a datos: EF Core, SQLite y ContactStore

El acceso a datos lo concentra la clase `ContactStore`. A diferencia de guardar en una `List` en memoria, acá los datos persisten en un archivo SQLite (`agenda.db`) a través de **Entity Framework Core**.

Primero, el `DbContext` —el puente entre las clases C# y la base:

```csharp
public class AgendaDbContext(DbContextOptions<AgendaDbContext> options) : DbContext(options) {
    public DbSet<Contact> Contacts => Set<Contact>();
}
```

Y el `ContactStore`, que ofrece las operaciones que la página necesita:

```csharp
public class ContactStore(IDbContextFactory<AgendaDbContext> dbContextFactory) {
    public List<Contact> GetAll(string? search = null) {
        using AgendaDbContext dbContext = dbContextFactory.CreateDbContext();
        IQueryable<Contact> query = dbContext.Contacts
            .OrderBy(contact => contact.Apellido)
            .ThenBy(contact => contact.Nombre);

        if (!string.IsNullOrWhiteSpace(search)) {
            string term = search.Trim();

            query = query.Where(contact =>
                EF.Functions.Like(contact.Nombre, $"%{term}%")
                || EF.Functions.Like(contact.Apellido, $"%{term}%")
                || EF.Functions.Like(contact.Telefono, $"%{term}%")
                || EF.Functions.Like(contact.Email, $"%{term}%")
                || EF.Functions.Like(contact.Notas, $"%{term}%")
            );
        }

        return query.ToList();
    }

    public Contact? GetById(int id) {
        using AgendaDbContext dbContext = dbContextFactory.CreateDbContext();
        return dbContext.Contacts.SingleOrDefault(contact => contact.Id == id);
    }

    public Contact Save(ContactInput input) {
        using AgendaDbContext dbContext = dbContextFactory.CreateDbContext();
        Contact contact;

        if (input.Id == 0) {
            contact = new Contact();
            dbContext.Contacts.Add(contact);
        }
        else {
            contact = dbContext.Contacts.Single(contact => contact.Id == input.Id);
        }

        contact.Nombre = input.Nombre;
        contact.Apellido = input.Apellido;
        contact.Telefono = input.Telefono;
        contact.Email = input.Email;
        contact.Notas = input.Notas;

        dbContext.SaveChanges();
        return contact;
    }

    public void Delete(int id) {
        using AgendaDbContext dbContext = dbContextFactory.CreateDbContext();
        Contact? contact = dbContext.Contacts.SingleOrDefault(c => c.Id == id);

        if (contact is null) {
            return;
        }

        dbContext.Contacts.Remove(contact);
        dbContext.SaveChanges();
    }
}
```

Hay varias decisiones que conviene entender:

> **`GetAll` unifica listar y buscar.** Si `search` es nulo o vacío, devuelve todo ordenado por apellido; si trae texto, filtra con `EF.Functions.Like` (la traducción de `LIKE` de SQL a EF Core). Así la página tiene un solo camino para poblar la lista, con o sin filtro.
>
> **`Save` decide insertar o actualizar según el `Id`.** Con `Id == 0` crea un `Contact` nuevo y lo agrega; con un `Id` real lo busca y le copia los campos. Una sola operación para las dos intenciones, en línea con el DTO único.
>
> **Por qué `IDbContextFactory` y un `using` por operación.** En vez de inyectar un `DbContext` compartido, inyectamos una *fábrica* y creamos un contexto efímero en cada método (`using` lo libera al terminar). Esto evita problemas de concurrencia y de estado acumulado entre operaciones, y encaja bien con peticiones cortas como las de HTMX.

## 10. Program.cs: dependencias y arranque

Acá registramos los servicios en el contenedor de inyección de dependencias, configuramos el pipeline y creamos/sembramos la base al iniciar.

El esqueleto de `Program.cs` (las instrucciones de arranque van arriba; las clases de las secciones 7–9 van debajo, dentro de `namespace AgendaRazorHtmx`):

```csharp
using AgendaRazorHtmx;
using Microsoft.EntityFrameworkCore;

var builder = WebApplication.CreateBuilder(args);
string connectionString = "Data Source=./agenda.db";

builder.Services.AddRazorPages();
builder.Services.AddDbContextFactory<AgendaDbContext>(options => options.UseSqlite(connectionString));
builder.Services.AddScoped<ContactStore>();

var app = builder.Build();

using (IServiceScope scope = app.Services.CreateScope()) {
    IDbContextFactory<AgendaDbContext> dbContextFactory =
        scope.ServiceProvider.GetRequiredService<IDbContextFactory<AgendaDbContext>>();

    using AgendaDbContext dbContext = dbContextFactory.CreateDbContext();
    dbContext.Database.EnsureCreated();
    AgendaDbSeeder.Seed(dbContext);
}

app.UseHttpsRedirection();
app.UseStaticFiles();
app.UseRouting();

app.MapRazorPages();

app.Run();
```

Decisiones a entender acá:

> **Antiforgery queda ACTIVADO.** Usamos `AddRazorPages()` tal cual, sin desactivar nada. ASP.NET exige un token en cada `POST` para protegerse de ataques CSRF, y eso es lo correcto. La clave es que **HTMX sí puede mandar ese token**: cada formulario incluye `@Html.AntiForgeryToken()` (que genera un input oculto) y los botones que hacen `POST` usan `hx-include="closest form"` para enviarlo junto con la petición. Así obtenemos la protección sin renunciar a HTMX. Lo vemos en los partials (sección 13).
>
> **`AddDbContextFactory` en vez de `AddDbContext`.** Registramos una *fábrica* de contextos porque el `ContactStore` crea un contexto efímero por operación (sección 8). Es el patrón recomendado cuando no querés un contexto por request sino contextos de vida corta y bien acotada.
>
> **`AddScoped<ContactStore>`:** una instancia por petición HTTP. El store no guarda estado propio (cada método abre y cierra su contexto), así que un objeto fresco por request es lo correcto.
>
> **`EnsureCreated()` + `Seed(...)` al arrancar:** creamos el archivo SQLite y su tabla si no existen, y sembramos los datos de ejemplo. `EnsureCreated` es suficiente para un tutorial; en un proyecto real con esquema cambiante se usan **migraciones** de EF Core.

## 11. El layout

El layout es la plantilla HTML que envuelve a todas las páginas: el `<head>` con los estilos y el script de HTMX, y la estructura base.

Primero descargamos HTMX a un archivo local (en vez de depender de una CDN), así la app funciona también sin internet:

```bash
curl -fsSL https://unpkg.com/htmx.org@2.0.4/dist/htmx.min.js -o wwwroot/js/htmx.min.js
```

Creá `Pages/Shared/_Layout.cshtml`:

```razor
<!DOCTYPE html>
<html lang="es">
<head>
    <meta charset="utf-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1.0" />
    <title>@ViewData["Title"] - Agenda Razor HTMX</title>
    <link rel="icon" href="data:," />
    <script src="/js/htmx.min.js"></script>
    <link rel="stylesheet" href="/css/site.css" />
</head>
<body>
    <main class="container">
        @RenderBody()
    </main>
</body>
</html>
```

> **Por qué HTMX local y no CDN:** `<script src="/js/htmx.min.js">` apunta al archivo que descargamos a `wwwroot/js/`. Lo sirve `app.UseStaticFiles()` (sección 10). Frente a una CDN, esto hace que la app funcione sin internet, no dependa de un servicio externo y cargue siempre la misma versión. El archivo queda versionado en el repo (no está en `.gitignore`).
>
> **`<link rel="icon" href="data:," />`** evita el pedido automático de `/favicon.ico` (y el 404 en la consola) sin tener que crear un ícono.
>
> **`@RenderBody()`** es donde se inserta el contenido de cada página.

## 12. La página principal con handlers inline

Esta es la pantalla central: el HTML del maestro-detalle más todos los handlers en un bloque `@functions`.

Creá `Pages/Index.cshtml`:

```razor
@page
@using AgendaRazorHtmx
@using Microsoft.AspNetCore.Mvc
@inject ContactStore ContactStore

@{
    ViewData["Title"] = "Agenda";
    Layout = "_Layout";
}

<h1 id="agenda-title">Agenda</h1>

<div class="agenda">
    <section class="panel master-panel" aria-labelledby="agenda-title">
        <button
            class="primary"
            hx-get="/?handler=New"
            hx-target="#detail"
            hx-swap="innerHTML">
            Nuevo contacto
        </button>

        <hr />

        <form id="search-form">
            <label class="visually-hidden" for="search">Buscar contactos</label>
            <input
                id="search"
                name="search"
                type="search"
                placeholder="Buscar por nombre, teléfono o email"
                autocomplete="off"
                hx-get="/?handler=List"
                hx-trigger="input changed delay:250ms, search"
                hx-target="#master"
                hx-swap="innerHTML" />
        </form>

        <div
            id="master"
            class="list-scroll"
            hx-get="/?handler=List"
            hx-trigger="load, contactsChanged from:body"
            hx-include="#search-form"
            hx-swap="innerHTML">
        </div>
    </section>

    <section class="panel">
        <div id="detail" aria-live="polite" aria-atomic="true">
            <p>Seleccioná un contacto.</p>
        </div>
    </section>
</div>

@functions {
    public IActionResult OnGetList(string? search, int? selectedId) {
        return Partial("_ContactList", new ContactListViewModel(ContactStore.GetAll(search), selectedId));
    }

    public IActionResult OnGetNew() {
        Response.Headers["HX-Trigger"] = "contactsChanged";
        return Partial("_ContactForm", new ContactInput(0, "", "", "", "", ""));
    }

    public IActionResult OnGetEmpty() {
        return Content("<p>Seleccioná un contacto.</p>", "text/html");
    }

    public IActionResult OnGetDetail(int id) {
        Contact? contact = ContactStore.GetById(id);

        if (contact is null) {
            return NotFound();
        }

        return Partial("_ContactDetail", contact);
    }

    public IActionResult OnGetEdit(int id) {
        Contact? contact = ContactStore.GetById(id);

        if (contact is null) {
            return NotFound();
        }

        Response.Headers["HX-Trigger"] = "contactsChanged";
        return Partial("_ContactForm", ToInput(contact));
    }

    public IActionResult OnPostSave(ContactInput input) {
        Contact savedContact = ContactStore.Save(input);

        Response.Headers["HX-Trigger"] = "contactsChanged";
        return Partial("_ContactDetail", savedContact);
    }

    public IActionResult OnPostDelete(int id) {
        ContactStore.Delete(id);

        Response.Headers["HX-Trigger"] = "contactsChanged";
        return Content("<p>Contacto eliminado.</p>", "text/html");
    }

    private static ContactInput ToInput(Contact contact) {
        return new ContactInput(
            contact.Id,
            contact.Nombre,
            contact.Apellido,
            contact.Telefono,
            contact.Email,
            contact.Notas
        );
    }
}
```

Esta página concentra el HTML de la pantalla, la inyección del store con `@inject`, y los handlers que responden a HTMX. Lo que **no** contiene es igual de importante: no tiene el modelo, ni el DTO, ni el acceso a datos. Todo eso vive en `Program.cs`.

Los dos paneles tienen `id` para que HTMX sepa a cuál apuntar: **`#master`** (la lista) y **`#detail`** (ver/crear/editar).

> **La lista se carga y se refresca con eventos.** El `<div id="master">` arranca vacío y trae `hx-trigger="load, contactsChanged from:body"`: se llena al cargar la página (`load`) y vuelve a pedir la lista cada vez que en el `<body>` se dispara el evento `contactsChanged`. ¿Quién dispara ese evento? El servidor, cuando guardás o borrás (más sobre esto en la sección 14).
>
> **El buscador usa un trigger con modificadores.** `hx-trigger="input changed delay:250ms, search"` significa: dispará cuando el valor *cambie* (`input changed`), pero esperá 250 ms desde la última tecla (`delay:250ms`, el clásico *debounce*), y también ante el evento `search` (cuando el usuario limpia el campo con la "x"). El `hx-include="#search-form"` del `#master` hace que el texto del buscador viaje también en los refrescos automáticos de la lista.
>
> **Por qué no hay `OnGet` que arme la lista:** la lista la trae `OnGetList` (con `search` nulo devuelve todo), igual que cualquier búsqueda. Un solo handler sirve para el primer render y para los filtros posteriores.
>
> **`OnGetEmpty` y `OnPostDelete` devuelven `Content(...)` y no un partial:** son fragmentos triviales (un `<p>`), así que se devuelven como HTML directo en vez de crear un archivo de partial para una sola línea.

## 13. Los partials

Los partials son fragmentos de HTML reutilizables. Cada handler devuelve uno (o un `Content` simple), y HTMX lo inserta en la página.

### Lista (el panel maestro)

Creá `Pages/_ContactList.cshtml`:

```razor
@using AgendaRazorHtmx
@model ContactListViewModel

<form class="contact-list">
    <fieldset class="contact-group">
        <legend class="visually-hidden">Contactos</legend>

        @if (Model.Contacts.Count == 0) {
            <p class="muted">No hay contactos para esa búsqueda.</p>
        } else {
            foreach (Contact contact in Model.Contacts) {
                string inputId = $"contact-{contact.Id}";

                <div class="contact-option">
                    <input
                        id="@inputId"
                        class="contact-radio"
                        type="radio"
                        name="selectedContact"
                        value="@contact.Id"
                        checked="@(Model.SelectedId == contact.Id)"
                        hx-get="/?handler=Detail&id=@contact.Id"
                        hx-trigger="change"
                        hx-target="#detail"
                        hx-swap="innerHTML" />

                    <label class="contact-item" for="@inputId">
                        <strong>@contact.Apellido, @contact.Nombre</strong>
                        <br />
                        <span class="muted">@contact.Telefono</span>
                    </label>
                </div>
            }
        }
    </fieldset>
</form>
```

> **Por qué radios y no enlaces:** cada contacto es un `<input type="radio">` (visualmente oculto) con su `<label>` estilado como tarjeta. Esto da, gratis, **navegación por teclado**: con las flechas te movés entre contactos como en cualquier grupo de radios, y al cambiar la selección (`hx-trigger="change"`) HTMX pide el detalle. El `checked="@(Model.SelectedId == contact.Id)"` resalta el contacto activo. El `<fieldset>` con `<legend>` (oculto con `.visually-hidden`) agrupa los radios de forma accesible.

### Detalle (solo lectura)

Creá `Pages/_ContactDetail.cshtml`:

```razor
@using AgendaRazorHtmx
@model Contact

@{
    Contact contact = Model;
}

<div class="detail-view">
    <div class="detail-header">
        <div>
            <h2>@contact.Nombre @contact.Apellido</h2>
            <p class="muted">@contact.Telefono</p>
        </div>
    </div>

    <dl class="detail-grid">
        <div><dt>Nombre</dt><dd>@contact.Nombre</dd></div>
        <div><dt>Apellido</dt><dd>@contact.Apellido</dd></div>
        <div>
            <dt>Teléfono</dt>
            <dd>@(string.IsNullOrWhiteSpace(contact.Telefono) ? "Sin teléfono" : contact.Telefono)</dd>
        </div>
        <div>
            <dt>Email</dt>
            <dd>@(string.IsNullOrWhiteSpace(contact.Email) ? "Sin email" : contact.Email)</dd>
        </div>
        <div class="detail-notes">
            <dt>Notas</dt>
            <dd>@(string.IsNullOrWhiteSpace(contact.Notas) ? "Sin notas" : contact.Notas)</dd>
        </div>
    </dl>

    <form class="actions" hx-target="#detail" hx-swap="innerHTML">
        @Html.AntiForgeryToken()

        <button
            type="button"
            class="primary"
            hx-get="/?handler=Edit&id=@contact.Id">
            Editar
        </button>

        <button
            type="button"
            class="danger"
            hx-post="/?handler=Delete&id=@contact.Id"
            hx-include="closest form"
            hx-confirm="¿Eliminar este contacto?">
            Eliminar
        </button>
    </form>
</div>
```

> **El detalle usa `@model Contact` directo:** no necesita envoltorio, le basta el contacto. Los campos vacíos muestran un texto de cortesía ("Sin teléfono", etc.) en vez de quedar en blanco.
>
> **El token antiforgery viaja con `hx-include="closest form"`:** el botón "Eliminar" hace un `POST` y necesita el token. Está dentro de un `<form>` que tiene `@Html.AntiForgeryToken()`, y `hx-include="closest form"` le dice a HTMX que incluya los campos de ese formulario (entre ellos, el token oculto) en la petición. Por eso el `POST` pasa la validación antiforgery sin desactivar nada.
>
> **`hx-confirm`** muestra un cuadro de confirmación del navegador antes de enviar el borrado.

### Alta y edición (un solo formulario)

Creá `Pages/_ContactForm.cshtml`:

```razor
@using AgendaRazorHtmx
@model ContactInput

<h2>
    @(Model.Id == 0 ? "Nuevo contacto" : $"Editar {Model.Nombre} {Model.Apellido}")
</h2>

<form
    hx-post="/?handler=Save"
    hx-target="#detail"
    hx-swap="innerHTML">

    @Html.AntiForgeryToken()

    <input type="hidden" name="Id" value="@Model.Id" />

    <label for="Nombre">Nombre <span class="required-mark" aria-hidden="true">*</span></label>
    <input id="Nombre" name="Nombre" value="@Model.Nombre" autocomplete="given-name" required autofocus />

    <label for="Apellido">Apellido <span class="required-mark" aria-hidden="true">*</span></label>
    <input id="Apellido" name="Apellido" value="@Model.Apellido" autocomplete="family-name" required />

    <label for="Telefono">Teléfono</label>
    <input id="Telefono" name="Telefono" value="@Model.Telefono" autocomplete="tel" inputmode="tel" />

    <label for="Email">Email</label>
    <input id="Email" name="Email" type="email" value="@Model.Email" autocomplete="email" inputmode="email" />

    <label for="Notas">Notas</label>
    <textarea id="Notas" name="Notas">@Model.Notas</textarea>

    <div class="actions">
        <button class="primary" type="submit">Guardar</button>

        @if (Model.Id == 0) {
            <button type="button"
                    hx-get="/?handler=Empty"
                    hx-target="#detail"
                    hx-swap="innerHTML">
                Cancelar
            </button>
        } else {
            <button type="button"
                    hx-get="/?handler=Detail&id=@Model.Id"
                    hx-target="#detail"
                    hx-swap="innerHTML">
                Cancelar
            </button>
        }
    </div>
</form>
```

> **Un solo formulario para crear y editar.** El campo oculto `Id` decide: `0` al crear, el id real al editar (lo prepara `OnGetEdit` con el helper `ToInput`). El título y el botón Cancelar se adaptan con un `@if`. Al enviar, `OnPostSave` recibe el `ContactInput` por model binding y `ContactStore.Save` inserta o actualiza según el `Id`.
>
> **Mejoras de formulario:** `required` y `autofocus` en Nombre, `autocomplete` e `inputmode` apropiados en cada campo (para que el teclado del móvil y el autocompletado del navegador ayuden), y el asterisco `*` marcando los obligatorios.
>
> **El formulario no incluye ningún refresco de la lista.** Quien limpia el resaltado de selección al abrir el formulario son los handlers `OnGetNew` y `OnGetEdit`, que disparan el evento `contactsChanged` (lo vemos en la sección 14). Así el formulario solo se ocupa de sí mismo y el refresco de `#master` pasa siempre por el mismo mecanismo.

## 14. La técnica clave: refrescar dos paneles con un evento

Hay un problema clásico en este tipo de pantallas: cuando **guardás** o **borrás**, cambian *dos* zonas a la vez. Al guardar, el `#detail` muestra el contacto y la lista de `#master` puede haber cambiado (nombre nuevo, orden distinto, un contacto más). Al borrar, el `#detail` muestra "Contacto eliminado" y el contacto debe desaparecer de `#master`.

Pero una petición HTMX normal apunta a **un solo** elemento (`hx-target`). ¿Cómo actualizamos el otro?

La solución que usamos son los **eventos personalizados disparados desde el servidor**. Funciona en dos partes:

**1. El `#master` escucha un evento** ([Index.cshtml](Pages/Index.cshtml)):

```html
<div id="master"
     hx-get="/?handler=List"
     hx-trigger="load, contactsChanged from:body"
     hx-include="#search-form"
     hx-swap="innerHTML">
</div>
```

El `contactsChanged from:body` significa: "cuando en el `<body>` se dispare un evento llamado `contactsChanged`, volvé a pedir la lista".

**2. El servidor dispara ese evento** devolviendo un header en la respuesta ([Index.cshtml](Pages/Index.cshtml)):

```csharp
public IActionResult OnPostSave(ContactInput input) {
    Contact savedContact = ContactStore.Save(input);

    Response.Headers["HX-Trigger"] = "contactsChanged";   // ← avisa "la lista cambió"
    return Partial("_ContactDetail", savedContact);
}
```

Cuando HTMX recibe una respuesta con el header `HX-Trigger: contactsChanged`, **dispara ese evento en el navegador**. El `#master`, que lo está escuchando, se refresca solo. Así, con una respuesta que apunta a `#detail`, también logramos actualizar `#master` —sin escribir JavaScript y sin que las plantillas se conozcan entre sí.

Los cuatro handlers que tocan la pantalla principal usan el mismo recurso, cada uno por su motivo:

| Handler        | Por qué dispara `contactsChanged`                                 |
|----------------|-------------------------------------------------------------------|
| `OnPostSave`   | El contacto pudo cambiar de nombre, de orden, o ser nuevo         |
| `OnPostDelete` | El contacto desaparece de la lista                                |
| `OnGetNew`     | Limpia el resaltado de selección al abrir el formulario de alta   |
| `OnGetEdit`    | Limpia el resaltado de selección al abrir el formulario de edición|

> **Un solo mecanismo para refrescar la lista.** Antes, el formulario incluía un `<div>` con `hx-trigger="load"` que recargaba `#master` por su cuenta. Al mover ese refresco al evento `contactsChanged`, todo el que necesita actualizar la lista lo hace de la misma forma: seteando el header en el servidor. El `#master` es el único que sabe *cómo* se pide la lista; los demás solo avisan que cambió.

> **Por qué esto es más limpio que incrustar la lista en cada respuesta.** Una alternativa es el *out-of-band swap* (`hx-swap-oob`): devolver, dentro de la misma respuesta, un segundo fragmento marcado para reemplazar `#master`. Funciona, pero acopla las plantillas (cada respuesta de guardar/borrar tendría que incluir la lista). Con `HX-Trigger`, cada handler devuelve solo lo suyo y "avisa" del cambio; el `#master` se encarga de refrescarse. El costo es una petición HTTP extra (la acción + el refresco de la lista), imperceptible en esta app.

## 15. Los flujos completos

Vale la pena seguir dos flujos de punta a punta para fijar cómo encajan las piezas.

### Flujo de alta

```txt
Clic en "Nuevo contacto"
   │  hx-get /?handler=New
   ▼
OnGetNew()  →  setea HX-Trigger: contactsChanged  +  devuelve _ContactForm (Id = 0)
   │  HTMX mete el formulario en #detail  Y  dispara contactsChanged
   │  (#master se refresca y se limpia el resaltado de selección)
   ▼
El usuario completa el formulario y envía
   │  hx-post /?handler=Save  (el binding arma ContactInput; hx-include manda el token)
   ▼
OnPostSave(input)  →  ContactStore.Save(input)  (Id == 0 ⇒ inserta)
   │  setea header HX-Trigger: contactsChanged
   │  devuelve _ContactDetail del nuevo contacto
   ▼
HTMX mete el detalle en #detail  Y  dispara contactsChanged
   ▼
#master (que escucha contactsChanged) vuelve a pedir la lista → aparece el nuevo contacto
```

### Flujo de borrado

```txt
El usuario ve un detalle y hace clic en "Eliminar"
   │  hx-confirm pide confirmación
   │  hx-post /?handler=Delete&id=2  (hx-include manda el token)
   ▼
OnPostDelete(2)  →  ContactStore.Delete(2)
   │  setea header HX-Trigger: contactsChanged
   │  devuelve Content("<p>Contacto eliminado.</p>")
   ▼
HTMX pone ese mensaje en #detail  Y  dispara contactsChanged
   ▼
#master vuelve a pedir la lista → el contacto ya no está
```

Abrí en el navegador la URL que muestre la consola (por ejemplo `https://localhost:7xxx`). Probá este recorrido para ejercitar todo:

| `AgendaDbContext`      | Puente EF Core ↔ SQLite                                    |
| `ContactStore`         | Acceso a datos (listar/buscar, ver, guardar, borrar)       |
| `AgendaDbSeeder`       | Siembra los datos de ejemplo solo si la base está vacía    |

### Los handlers de un vistazo

```csharp
public IActionResult OnGetList(string? search, int? selectedId); // listar y filtrar
public IActionResult OnGetNew();                                 // formulario de alta
public IActionResult OnGetEmpty();                               // panel vacío (cancelar alta)
public IActionResult OnGetDetail(int id);                        // ver
public IActionResult OnGetEdit(int id);                          // formulario de edición
public IActionResult OnPostSave(ContactInput input);             // crear o actualizar
public IActionResult OnPostDelete(int id);                       // borrar
```

### Características de HTMX que usamos

- `hx-get` / `hx-post`, `hx-target`, `hx-swap`.
- `hx-trigger` con modificadores: `delay:250ms`, eventos `load`, `change`, `search`.
- **Eventos personalizados** (`contactsChanged from:body`) disparados desde el servidor con el header **`HX-Trigger`**.
- `hx-include` para enviar el token antiforgery (`closest form`) y el buscador (`#search-form`).
- `hx-confirm` para confirmar el borrado.
- Selección por **radios** con `hx-trigger="change"`, navegable por teclado.
